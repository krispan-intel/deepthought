2. Technical Invention Disclosure

2.1 Problem Statement

The Linux fair scheduler must make placement decisions on extremely hot paths under strict determinism, bounded runtime, and lock-ordering rules, yet the most valuable microarchitectural evidence for placement on Intel systems arrives asynchronously through PEBS PMI/NMI paths. A credible scheduler optimization therefore has to solve a combined correctness, safety, and abuse problem. First, it must define exactly when a PEBS sample belongs to a task phase that the scheduler actually cares about, across preemption, migration, idle transitions, signal delivery, sched-out latency, and late PMIs. Second, it must do so without dereferencing unstable task pointers or relying on pid/tid values that are vulnerable to reuse. Third, it must capture evidence in NMI context with fixed cost, no allocation, no unbounded loops, and no scheduler-lock interaction. Fourth, it must publish a task hint that rq-hot readers can consume in O(1) without cross-rq dependencies, extra locks, or side effects that indirectly perturb fairness. Fifth, it must be abuse-bounded in a multi-tenant kernel: a malicious task must not be able to convert PEBS telemetry into a covert channel, a priority surrogate, or a general-purpose readback interface for other tenants' behavior. Existing mechanisms do not solve this complete problem. perf_event_task_sched_in/out can gate PMU collection, but they do not create a scheduler-visible correctness model for late sample attribution or a secure publication contract. Existing scheduler signals such as util_est, PELT, uclamp, and EAS are synchronous scheduler-native signals; they do not address asynchronous NMI evidence, sample loss, or precise-event correctness. Prior performance-counter-guided scheduling literature often assumes periodic polling, broader policy freedom, or direct performance feedback loops, but it does not define the kernel-grade invariants needed to safely consume PEBS evidence in CFS without breaking determinism or fairness.

2.2 Prior-Art Gap

The prior-art gap is the absence of an end-to-end kernel control plane with all of the following concrete invariants at once. Existing perf sched-in/sched-out machinery answers only when PMU counting is enabled; it does not provide an application ABI for declaring bounded scheduler-relevant phases, does not bind late PEBS samples to a CPU-local sched window journal, and does not publish a scheduler-safe hint with neutral-on-drop semantics. Existing per-task scheduler signals such as util_est, uclamp, and EAS are fundamentally different: they derive from runnable time, capacity, or energy models under scheduler control, not from asynchronous PEBS PMIs that can arrive after a task has been preempted or migrated. Prior counter-guided scheduling research generally uses global epochs, periodic readers, or direct heuristic adaptation, and therefore omits the kernel-hard problems that matter here: stable lifetime cookies instead of pid reuse, same-CPU time validation instead of synchronized clocks, NMI-safe fixed-width tuple capture instead of variable record parsing, a shard-stable single writer for per-task publication, and an expressly forbidden effect surface that limits actuation to deterministic placement tie-breaks. The invention fills that gap with a joint time-triggered and event-triggered mechanism: user software marks phase boundaries, the scheduler defines the only valid collection windows under rq lock, PEBS provides precise microarchitectural evidence inside those windows, and the scheduler consumes only a coarse confidence-gated hint under a neutral-by-default contract. Another gap is security architecture. Prior art rarely defines exact privilege gates, accounting points, or non-readback rules. This invention does: user space never gains raw PMC or PEBS readback through this interface, CR4.PCE-based RDPMC exposure is not required, unprivileged hint state is not exported, and all observable effect is reduced to a capped tie-break among already-equivalent legal CPUs with quotas and ignore rules for RT/DL, isolated CPUs, cpuset and NUMA restrictions, high loss, and low confidence.

2.3 Proposed Invention

The invention introduces a kernel ABI and data path named here as sched_pebs_phase. The ABI is a single control syscall, sys_sched_pebs_phase_ctl(op, arg, size), or equivalent prctl-backed interface, with four mandatory operations: CREATE, BEGIN, END, and DESTROY. CREATE takes a fixed-format struct such as {version, flags, event_profile, timeout_us, arm_budget_us_per_sec, sample_budget_per_sec, min_confidence_q8, max_loss_q8, max_hint_divergences_per_sec} and returns a kernel-issued 64-bit token_id. Ownership is per thread. Default inheritance is none; BEGIN/END apply only to the current thread and only one active token is allowed, so nested phases return -EBUSY. If flags request clone inheritance and the child is created with CLONE_THREAD under identical credentials, the child receives a fresh token object with copied policy but a new token_id and new task cookie in REGISTERED state, never ACTIVE, so attribution remains per-thread and cannot bleed across task lifetimes. execve() always auto-ENDs and DESTROYs active tokens by default; signal delivery merely causes normal preemption behavior and does not itself end the phase; task exit forces END plus destruction. CREATE may fail with -EPERM if CAP_PERFMON or cgroup/LSM policy denies use, -EOPNOTSUPP if the CPU lacks the required PEBS format, and -EINVAL for invalid profiles or limits. BEGIN transitions REGISTERED to ACTIVE_PENDING_ARM; if the task is currently running in user mode and budgets allow, the next sched-in arms PEBS. END transitions ACTIVE_PENDING_ARM, ARMED, SUSPENDED, TIMED_OUT, or EXHAUSTED back to REGISTERED and neutralizes the hint epoch if needed. DESTROY transitions any state to DEAD and removes the cookie mapping after grace-period teardown. Internally, each eligible task owns a refcounted sched_pebs_ctx carrying a never-reused 64-bit cookie allocated from a global atomic counter, a monotonically increasing window_generation, current token policy, aggregate counters, and a seqcount-protected hint snapshot. A global xarray keyed by cookie points to sched_pebs_ctx and is visible under RCU. The scheduler writes CPU-local window records under rq lock on each sched-in and sched-out. Each per-CPU journal contains one current open window and a small fixed ring of recently closed windows; each record stores {cookie, generation, start_clk, end_clk, event_profile, flags}. On sched-in of a fair-class task with an ACTIVE token, user-mode execution, valid affinity, valid security policy, and remaining budget, the kernel increments ctx->window_generation, writes a new open window in that CPU journal with start_clk=local_clock(), end_clk=U64_MAX, generation=ctx->window_generation, cookie=ctx->cookie, then enables the precreated per-CPU PEBS event for the chosen allowlisted event profile. On sched-out, preemption, migration away, kernel-mode transition, timeout, or budget exhaustion, the same CPU under rq lock closes the window by setting end_clk=local_clock(), marks the open slot closed, pushes it into the fixed closed-window ring, and disables PEBS for that task window. That design is the attribution anchor: a PEBS sample is accepted only if its same-CPU timestamp falls within exactly one CPU-local journal window and that window resolves to a live cookie context. The NMI path never dereferences task_struct and never trusts pid/tid. It copies only a fixed-size tuple from the PEBS record and PMU context into a preallocated per-CPU ring, for example {cpu_id, sample_clk, event_profile, latency_bucket, weight_bucket, data_src_class, exact_ip_bit, user_mode_bit, overflow_flags}. No raw IP, branch stack, virtual address, physical address, or callchain is exported into the scheduler path. The tuple is enqueued using atomic reservation plus store-release commit; if the ring is full, the tuple is dropped and the drop counter increments, which later lowers loss/confidence and can force neutral. A deferred shard worker drains tuples, validates each by cpu_id and sample_clk against the corresponding CPU-local window journal, resolves the journal's cookie through the RCU xarray, acquires a sched_pebs_ctx reference with refcount_inc_not_zero, and updates per-task aggregates. If a sample cannot be mapped to exactly one closed or open window, if the cookie no longer resolves, if the task has exited, if timeout/budget rules invalidate the epoch, or if loss is too high, the sample is ignored and the task remains neutral. To guarantee a single writer for the published hint, all cookies are hashed to a stable attribution shard, and only that shard's worker may publish the final snapshot for that cookie. Publication is via seqcount into a compact snapshot such as {epoch, class, confidence_q8, loss_q8, preferred_llc_id, preferred_nid, last_valid_gen, flags}. Readers in the scheduler use read_seqcount_begin/read_seqcount_retry once; if the retry fails or static key is off, they treat the hint as neutral. The hint classes are deliberately coarse. An initial allowlist can classify MEM_REMOTE when PEBS data_src shows remote DRAM and long latency dominates, MEM_LOCAL when long-latency local memory dominates, CORE_STEADY when precise retired instructions are high and memory-latency samples are sparse, otherwise NEUTRAL. The scheduler contract is explicit and narrow: these hints may only break ties among candidate CPUs that are already legal and equivalent under existing policy. They may not alter vruntime, NICE, load weight, bandwidth, util_est, PELT, uclamp, slice length, wakeup preemption, or runnable list ordering.

Exact scheduler touchpoints and proof sketch:
1. select_task_rq_fair(): after normal affinity, cpuset, capacity, NUMA, asym-packing, and existing wake_affine/EAS logic identifies legal candidate CPU(s), call sched_pebs_tiebreak(prev_cpu, target_cpu, p, hint) only when the candidates are otherwise equal under current policy. No additional CPU scans are introduced; the helper compares at most the already-available candidate CPUs.
2. wake_affine(): when existing wake_affine logic has not already produced a strict winner, and only for fair-class tasks, the helper may prefer keeping a MEM_REMOTE or MEM_LOCAL task on a CPU sharing the previously validated LLC or NUMA node, or may keep a CORE_STEADY task on the current CPU if already equivalent. The helper cannot flip a decision when cpumask legality, NUMA policy, or capacity differ.
3. find_idlest_cpu() optional path: only when existing logic already found equal-idle candidates and a compile-time/static-key option is enabled. The helper compares the current best and challenger only; it never expands the search.

Bounded-cost read contract:
- static_branch_unlikely(&sched_pebs_key) gates the entire feature.
- Reader touches only p->sched_pebs_hint and topology constants already used by the scheduler.
- Reader takes no locks, performs no allocation, performs no remote journal reads, and performs at most one seqcount retry before neutral fallback.
- No cross-rq dependency is introduced because the hint belongs to the waking task and is read locally under existing rq-path context.

Forbidden effect surface:
- never change vruntime
- never change weight or priority
- never change slice length or bandwidth
- never reorder runnable entities
- never force migration of a currently running task
- never override cpuset, affinity, isolation, NUMA mempolicy, or scheduler class rules
- never read back raw PEBS data to unprivileged software

2.5 Implementation Plan

1. Define ABI and state machine. - Add uapi struct sched_pebs_phase_cfg { u16 version; u16 flags; u16 event_profile; u16 reserved0; u32 timeout_us; u32 arm_budget_us_per_sec; u32 sample_budget_per_sec; u16 min_confidence_q8; u16 max_loss_q8; u16 max_hint_divergences_per_sec; u16 reserved1; u64 token_id; }. - Implement ops CREATE, BEGIN, END, DESTROY. - Error codes: -EPERM (missing CAP_PERFMON or policy), -EOPNOTSUPP (PEBS profile unsupported), -EINVAL (bad config), -EBUSY (nested/active token), -EDQUOT (budget exhausted), -ETIME (phase timed out), -ESTALE (invalid token state), -ESRCH (token absent for current task). - States: DEAD -> REGISTERED -> ACTIVE_PENDING_ARM -> ARMED -> SUSPENDED -> REGISTERED, with side branches to EXHAUSTED and TIMED_OUT; DESTROY returns to DEAD from any state. - Inheritance rules: default none; optional CLONE_THREAD inheritance clones policy but creates new token_id and new cookie in REGISTERED state; no inheritance across fork without CLONE_THREAD; exec auto-destroys by default; exit always tears down. 2. Add task lifetime identity and teardown-safe context. - Introduce struct sched_pebs_ctx with fields: refcount_t refs, struct task_struct *task, u64 cookie, u32 window_generation, token policy, seqcount_t hint_seq, hint snapshot, aggregate counters, per-epoch quotas, and last_hint_divergence counters. - Allocate cookie from atomic64_inc_return(); never reuse within practical kernel lifetime. - Store ctx in an RCU-protected xarray keyed by cookie. - NMI path never stores task_struct pointers. Deferred attribution resolves cookie->ctx under RCU and uses refcount_inc_not_zero() on ctx. If task-specific fields are needed beyond ctx mirrors, the worker may obtain get_task_struct() only in process context after ctx->task is checked under RCU and task->usage can be incremented nonzero; release with put_task_struct(). - Exit path: force END, close any open window, clear ctx->task, remove xarray entry, synchronize_rcu(), drop task-held reference, and let late tuples neutralize because cookie no longer resolves. 3. Implement CPU-local window journal and exact accounting points. - Per CPU, maintain one open slot plus a small fixed ring of recently closed windows. Each slot stores {cookie, generation, start_clk, end_clk, llc_id, nid, event_profile, flags}. - On sched-in under the destination rq lock: if task has ACTIVE token, is fair-class, returning to user mode, and budgets allow, increment ctx->window_generation, write open slot with start_clk=local_clock(), end_clk=U64_MAX, generation=ctx->window_generation, and enable the precreated PEBS event. - On sched-out/preemption/migration/kernel-entry/timeout/budget exhaustion under same rq lock: set end_clk=local_clock(), move slot to closed ring, disable PEBS. - Arm-budget accounting point is actual user-mode armed runtime only, measured as end_clk-start_clk at close; sleep time and blocked time do not consume arm budget. - Sample-budget accounting point is accepted PEBS tuples per token per replenishment epoch, not raw PMU overflows. - Replenish budgets on a coarse one-second epoch using per-UID and per-cgroup token buckets; if either bucket is empty, sched-in refuses to arm and the hint remains neutral. 4. Constrain PEBS profile and NMI tuple. - Allowlist only profiles with precise_ip>=2 and fixed metadata support. Initial profiles: a. MEM_LAT_128: MEM_TRANS_RETIRED.LOAD_LATENCY_GT_128 with weight + data_src. b. MEM_LAT_256: MEM_TRANS_RETIRED.LOAD_LATENCY_GT_256 with weight + data_src. c. INST_PREC: INST_RETIRED.PREC_DIST as denominator/profile support. - Reject any profile lacking required PEBS fields or exact user-mode attribution. - NMI copies only a fixed-size tuple such as {u16 cpu; u16 event_profile; u16 latency_bucket; u16 weight_bucket; u16 data_src_class; u16 flags; u64 sample_clk}. Raw IP, data address, physical address, branch records, or callchains are never stored in the scheduler pipeline. - Ring enqueue uses atomic head reservation and store-release commit. If backlog cap is exceeded, increment loss counters and drop. 5. Implement deferred attribution and publication. - Drain per-CPU rings in bounded batches into cookie-hash shards so each cookie has one stable writer. - Validation rule for every tuple: find exactly one same-CPU journal window whose [start_clk, end_clk) contains sample_clk; if none or multiple, drop; resolve that window's cookie through xarray; if ctx missing, drop; if token state invalid, timed out, or generation stale, drop. - Aggregate accepted samples into coarse counters. 4. Risks & Mitigations

  • Risk: late PMI arrival after sched-out or migration could misattribute a sample. Mitigation: validate by same-CPU sample timestamp against a CPU-local open/closed window journal and accept only unique interval matches; otherwise drop neutral.
  • Risk: pid reuse or stale task_struct pointers could corrupt attribution. Mitigation: use a dedicated 64-bit lifetime cookie and refcounted sched_pebs_ctx resolved through RCU; never queue raw task pointers from NMI.
  • Risk: scheduler hot-path overhead could regress wakeup latency. Mitigation: static-key gate the entire feature, keep reader O(1), compare only already-known candidate CPUs, and fall back to neutral on any seqcount retry or gating failure.
  • Risk: hint-driven placement could indirectly perturb fairness or create starvation. Mitigation: forbid any effect on vruntime, weight, slice, preemption, or runnable ordering; allow only tie-break use among already-equivalent legal CPUs; cap hint-driven divergences per epoch.
  • Risk: malicious tenants could use the mechanism as a covert channel. Mitigation: deny raw readback, expose only privileged aggregate counters, gate CREATE with CAP_PERFMON plus cgroup/LSM policy, and rate-limit both arming time and hint-driven divergences.
  • Risk: unsupported or heterogeneous Intel CPUs may lack required PEBS metadata. Mitigation: enforce an explicit event-profile allowlist and require precise_ip and metadata support; unsupported systems remain permanently neutral.
  • Risk: PEBS loss or NMI backlog under heavy load may reduce hint quality. Mitigation: track accepted versus dropped tuples, quantize loss_q8, and force neutral whenever loss exceeds threshold.
  • Risk: application misuse of BEGIN/END could create noisy or excessively long phases. Mitigation: enforce timeout_us, single active token per thread, -EBUSY on nesting, and auto-destroy on exec/exit.
  • Risk: excessive journal retention or deferred backlog could increase memory footprint. Mitigation: use fixed-size per-CPU journals and rings with bounded shard workers and explicit drop-to-neutral semantics.
  • Risk: prior-art overlap could be alleged as a combination of perf gating and scheduling heuristics. Mitigation: emphasize and claim the specific invariants that prior art lacks in combination: CPU-local window journals, lifetime cookies, fixed-size low-entropy NMI tuples, single-writer seqcount publication, and forbidden scheduler effect surface.
