2. Technical Invention Disclosure

2.1 Problem Statement

Linux fair-class wakeup placement treats KVM vCPU threads as ordinary fair tasks and therefore relies on generic host-side indicators such as wake_affine, idle preference, runnable depth, util_est, uclamp, asym packing, and NUMA policy. On x86 with EPT/TDP-MMU, that misses a distinct class of migration cost: a vCPU may have just enjoyed several quiet windows on the same EPT root with stable 2 MiB or 1 GiB translations, or it may have just exited a churn-heavy interval caused by dirty-log write-protect activation, memslot invalidation, root replacement, large-page split/coalesce activity, or remote SPTE zap batches. Those states materially change the cost of moving the vCPU across an LLC boundary because TLB residency, page-walk cache warmth, and LLC-contained page-table lines differ sharply across those phases. Existing scheduler inputs cannot see that MMU-specific signal in a bounded, lock-safe form compatible with the wakeup hot path, so the kernel may migrate a vCPU at exactly the moment when preserving same-LLC placement would avoid disproportionate translation rewarm cost and tail-latency spikes.

2.2 Prior-Art Gap

Known scheduler mechanisms solve different problems. sched_migration_cost is a static host-wide knob; it cannot express that one specific vCPU has just switched EPT roots or suffered a dirty-log write-protect burst while another task has not. wake_affine compares current wakeup locality and generic load but has no visibility into second-level translation stability, large EPT leaf continuity, or MMU recovery windows. SD_SHARE_PKG_RESOURCES and related domain flags encode topology, not task-specific translation state. NUMA balancing operates on host memory-placement timescales and does not model microsecond-to-millisecond windows of EPT churn. util_est and uclamp encode compute demand and clamping policy, not the translation rewarm tax of moving a vCPU after EPT disruption. Generic cache-hotness reasoning is also insufficient because EPT stability and invalidation activity are not equivalent to ordinary data-cache warmth; a vCPU can appear CPU-light yet still carry unusually high page-walk and TLB recovery cost. The gap is therefore a missing bridge from x86 KVM-managed EPT behavior to a bounded, scheduler-hot-path-safe wakeup hint with explicit validity semantics.

2.3 Proposed Invention

The invention adds a per-task Wakeup Locality Hint object for eligible KVM vCPU threads. KVM privately accumulates a bounded translation-pressure summary per vCPU using event classes already visible at serialized x86 MMU paths: root switch, dirty-log write-protect batch, memslot invalidation batch, large-page split batch, remote zap batch epoch, and quiet windows with stable roots. At KVM_RUN and VM-exit boundaries, KVM publishes a fixed-size hint record containing: enabled bit, state, stability class, bounded penalty, jiffies expiry, observed epochs for tracing, and a maximum of four preferred CPUs from the vCPU's current LLC and capacity class. CFS consults that record only inside select_task_rq_fair() after baseline placement has already produced a legal target, and only to decide whether an already-considered alternate candidate such as prev_cpu or wake_cpu should win a near-tie. The hint therefore expresses a concrete event-triggered and time-triggered contract: EPT churn or stability in KVM causes transient locality preference, but the preference expires automatically, carries its own validity, and never becomes a correctness dependency.

2.5 Implementation Plan

1. Add a generic scheduler-owned hook type, struct sched_wakeup_locality_hint, plus a task_struct pointer field behind a static key, with cacheline alignment and RCU lifetime management. 2. Add per-CPU topology keys sched_llc_id and sched_capacity_class, updated during topology rebuild and hotplug, with READ_ONCE-safe accessors for scheduler hot paths and producer-side publish filtering. 3. Add a small x86 KVM vCPU-local accumulator for EPT locality state: local_pressure, quiet_windows, last_root_epoch_seen, last_remote_epoch_seen, and dominant leaf-class observation. 4. Instrument only already-serialized x86 MMU commit points to update the accumulator with the defined event weights; remote MMU work records only a coarse once-per-batch epoch advance, not per-SPTE task-visible writes. 5. Implement publish_hint() at KVM_RUN/VM-exit boundaries: fold pending local and remote batch signals, choose STABLE vs RECOVERY vs REVOKED, compute jiffies expiry, cap penalty_pct to {0,4,8}, and build preferred_cpu[<=4] from the allowed mask filtered by same llc_id and capacity class. 6. Add a fair-class wakeup helper invoked at the end of select_task_rq_fair() that performs the bounded seqcount snapshot and applies the strict near-tie rule only to already-considered candidate CPUs. 7. Add tracepoints and counters for publish state changes, fallback reasons, candidate reorders, expired-token hits, and recovered-vs-stable wakeups so the mechanism can be tuned and disabled safely. 8. Gate the entire feature behind a config option and static key, default off, with clear documentation that the hint is advisory only and not a security or correctness primitive.

2.6 Validation Plan

Workloads: (a) KVM guests under dirty-log write-protect enable/disable cycles representative of live migration precopy; (b) memslot invalidation and ballooning/memory-hotplug stress; (c) huge-page split/coalesce stress in TDP-MMU guests; (d) interrupt-driven network or storage appliance VMs with short wake/sleep intervals; (e) mixed-host contention on multi-LLC Xeon and hybrid x86 topologies. Metrics: cross-LLC wakeup migrations per vCPU, p95/p99 guest wakeup-to-run latency, remote TLB shootdown count, MMU root switches, SPTE zap batch count, page-walk/TLB miss counters, host scheduler wakeup cost, and total host CPU overhead of the feature. Acceptance criteria: at least 10% reduction in cross-LLC wakeup migrations for churn-heavy vCPUs, at least 5% improvement in p99 guest wakeup latency for at least one target workload class, no measurable regression beyond noise for workloads where the hint remains absent or invalid, and less than 0.3% host CPU overhead when the feature is enabled but non-triggering. Additional invariants to verify: no change in CPU choice when the hint is absent/expired/invalid, no violation of cpuset/cgroup affinity or sched_core constraints, no use from RT/DL/periodic-balance paths, and publication frequency remaining within the stated per-jiffy bound under storm conditions.

4. Risks & Mitigations

  • Risk: stale topology data during CPU hotplug could leave a dead CPU in preferred_cpu[]. Mitigation: build preferred_cpu[] from active and allowed CPUs only, filter again at read time with cpu_active_mask and affinity checks, and revoke the hint if preferred_cnt collapses to zero.
  • Risk: hint influence could accidentally override stronger scheduler policy. Mitigation: apply the helper only after baseline placement, only on fair-class wakeups, only to already-considered candidates, and only within the explicit near-tie thresholds; otherwise return base_cpu unchanged.
  • Risk: seqcount misuse on the writer side could make the design unsound. Mitigation: require a defined publish_lock wrapper around every write_seqcount_begin/end pair and forbid bare writer-side seqcount updates.
  • Risk: event storms could turn the feature into a contention source. Mitigation: use vCPU-local saturating accumulators, represent remote work as one coarse batch epoch, and cap publish frequency to once per jiffy per vCPU except on class transition.
  • Risk: side-channel or policy abuse concerns if the hint is treated as authoritative. Mitigation: keep the state coarse, expiry short, feature default-off, no guest or userspace export, and document that the hint is advisory only and never used for security, isolation, or correctness decisions.
  • Risk: 32-bit torn reads of expiry and generations. Mitigation: include all multibyte fields in the seqcount snapshot and rely on RCU for lifetime, not on naturally atomic wide loads.
