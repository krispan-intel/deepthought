2. Technical Invention Disclosure

2.1 Problem Statement

Linux blk-mq exposes queue depth, completion counts, and request latency, but it does not natively know whether completion execution for a given hardware queue is suffering remote-cache, remote-DRAM, or HITM-heavy memory behavior on the CPU that is actually servicing that completion path. Existing PEBS and perf facilities can sample these effects, yet they are architected as profiling tools and not as a kernel control input with correctness boundaries. The missing mechanism is a kernel-owned path that: (a) consumes only validated PEBS DS records rather than relying on interrupted RIP assumptions, (b) maps each record in NMI context to a stable tuple `{region_uid, source_cookie, hctx_cookie, cpu, pmu_epoch, ctx_seq, data_src_class}` without leaking attribution across nested interrupts, threaded handlers, softirq/NAPI callbacks, poll loops, migration, or CPU hotplug, (c) defers any blk-mq and IRQ resolution until process context and revalidates against live generations before acting, (d) applies only to truly dedicated managed completion IRQs that satisfy irq core rules, and (e) disables itself whenever PEBS quality is insufficient because of multiplexing, throttling, lost records, unsupported `data_src` bits, or external policy conflicts. Without that contract, PMU-driven IRQ steering is too error-prone for production kernels.

2.2 Prior-Art Gap

Prior art falls into four buckets, none of which provides the claimed mechanism. First, perf, PEBS, perf c2c, BPF, tracepoints, and kallsyms-based tools collect samples and can reveal memory-source locality, but they do not define an in-kernel actuation contract with validated DS-record capture, generation-safe blk source identities, or managed-IRQ constrained transactions. Second, irqbalance, manual `/proc/irq/*/smp_affinity` control, and driver-specific NVMe or MSI-X affinity logic steer interrupts using topology and load information, not a kernel-owned PEBS tuple that is revalidated against live blk-mq state before every decision. Third, static keys, tracepoints, BTF-based attribution, or generic code annotations can identify code regions, but they do not define a compile-time `region_uid` scheme with collision handling, module-build scoping, auditability, and NMI-safe scope stacks that survive nesting and hotplug. Fourth, research on interrupt steering and cache-aware placement generally teaches feedback control or topology heuristics, not the particular combination of `PERF_SAMPLE_DATA_SRC` classification, `(cpu, pmu_epoch, ctx_seq)` state isolation, tombstone-safe deferred revalidation, OFF/DEGRADED/ON signal gating, and ownership-aware managed-affinity transactions. The novel boundary is therefore the inseparable contract of tuple capture at validated PEBS drain, deferred revalidation against live blk/IRQ generations, quality-gated eligibility, and rollback-safe managed affinity control.

2.3 Proposed Invention

The invention has six coordinated mechanisms. 1. Validated PEBS drain capture and bounded `data_src_class` mapping. A hook at the x86 PEBS DS-record completion path consumes only records already accepted by perf as precise and non-corrupt. The hook reads `perf_sample_data::data_src` and classifies it into a bounded class set: `LOCAL_CORE_CACHE` for `PERF_MEM_LVL_L1`, `PERF_MEM_LVL_L2`, or `PERF_MEM_LVL_LFB` hits; `LLC_LOCAL` for `PERF_MEM_LVL_L3` hits without remote indication; `LOCAL_DRAM` for `PERF_MEM_LVL_LOC_RAM`; `REMOTE_CACHE` for `PERF_MEM_LVL_REM_CCE1` or `PERF_MEM_LVL_REM_CCE2`; `REMOTE_DRAM` for `PERF_MEM_LVL_REM_RAM1`, `PERF_MEM_LVL_REM_RAM2`, or RAM plus `PERF_MEM_REMOTE_REMOTE`; `HITM_LOCAL` or `HITM_REMOTE` when `PERF_MEM_SNOOP_HITM` is set, split by remote indication; `PMEM_OR_CXL` for `PERF_MEM_LVL_PMEM`; and `UNKNOWN` otherwise. If the PMU model lacks remote or HITM qualifiers, the mapping collapses to `COARSE_CACHE` or `COARSE_RAM`, the quality state cannot rise above DEGRADED, and actuation is disabled. 2. Compile-time `region_uid` registry. Completion-path annotations use a macro such as `DECLARE_BLK_PEBS_REGION(name, path_type)` that emits a constant descriptor into a dedicated linker section. Each descriptor carries `{module_build_id64, source_file_crc32, local_region_index16 from __COUNTER__, path_type8, start_ip, end_ip}`. At boot or module load, the kernel sorts descriptors, rejects duplicate tuples before hashing, and computes a 128-bit `region_uid = SipHash(build_id || source_file_crc32 || local_region_index || path_type)`. Because module build ID is part of the key, IDs intentionally change across kernel or module rebuilds; there is no cross-upgrade ABI promise, which prevents stale persistent mappings from being reused after upgrades. The registry is read-only, emitted into the build manifest, and exported only for audit via a read-only kernel view, never as a hot-path control plane. 3. Generation-safe blk completion-source registry. Each steerable `blk_mq_hw_ctx` registers a source entry `{source_cookie64, hctx_cookie64, irq, request_queue cookie, tagset cookie, src_gen64, baseline_managed_mask, driver_hint_mask, policy_owner}`. `src_gen64` increments on queue-map rebuild, controller reset, suspend/resume, IRQ re-request, namespace/controller rebind, or any blk-mq topology change that can invalidate prior source identity. Before teardown the entry is tombstoned and ownership is released; deferred aggregation discards any sample whose `{source_cookie, hctx_cookie, src_gen}` no longer resolves to the live registry entry. 4. Formal no-cross-context-leakage invariant. Each CPU maintains an NMI-safe scope stack. Push points are placed around hardirq completion wrappers, threaded IRQ completion wrappers, softirq/NAPI completion batches, and poll completion batches. Every push increments a per-CPU 64-bit `ctx_seq` and stores `{region_uid, source_cookie, hctx_cookie, cpu, pmu_epoch snapshot, ctx_seq, scope_type}`. Hardirq and softirq paths are naturally pinned to the current CPU; threaded IRQ and poll batches execute under `migrate_disable()` for the annotated batch so the `(cpu, ctx_seq)` pair remains stable. Nested interrupts simply push another entry and restore the parent on pop. A PEBS sample may be attributed only if the same-CPU top-of-stack entry is live at drain time and later revalidates against the stored `pmu_epoch`, `ctx_seq`, and source generation. Otherwise the sample becomes unattributed and cannot influence control. 5. `pmu_epoch` and quality-state gating. Each CPU has a 64-bit `pmu_epoch` that increments on PEBS event enable or disable, DS buffer reset or reprogramming, perf multiplex schedule-in or schedule-out of the controlling event, PMU throttling resets, CPU hotplug online or offline affecting the event, and any loss condition that can change sample interpretability. The controller maintains OFF, DEGRADED, and ON states per decision window. OFF applies when PEBS or `data_src` is unavailable, when no eligible managed vectors exist, when lost-sample ratio exceeds 5%, when event residency under multiplexing falls below 80%, when any eligible CPU lacks installed per-CPU state, or when epoch churn occurs inside the window. DEGRADED applies when only coarse cache-versus-RAM classification is available, when lost-sample ratio is between 1% and 5%, when residency is between 80% and 95%, when online CPU coverage is incomplete but above 50%, or when a recent ownership conflict occurred; DEGRADED is observe-only. ON requires full remote or HITM qualifiers, less than 1% lost samples, greater than 95% event residency, all eligible CPUs online and instrumented, and no policy conflict or hotplug transition in the last three windows. `pmu_epoch` and `ctx_seq` are both `u64`; if either counter ever approaches wrap, the feature forces OFF and requires reinitialization, eliminating ABA aliasing. 6. Eligibility rules, managed-mask constrained transaction, and rollback. A completion IRQ is eligible for actuation only if `irq_desc` has exactly one action, `IRQF_SHARED` is absent, the vector maps one-to-one to one blk-mq data-queue `hctx`, the queue is not an admin queue, the path is not polled-only or NAPI-only, the descriptor is affinity-managed, the irq chip supports affinity changes, the descriptor is not per-CPU or otherwise immutable, and the driver has not declared internal completion steering ownership. Runtime detection uses the live IRQ descriptor, blk-mq source registry, queue type, and driver capability flags. The candidate target mask is computed as the intersection of the managed baseline affinity in `irq_data->common->affinity`, the housekeeping mask, the effective cpuset mask, the online CPU mask, any driver-provided hint mask, and a NUMA candidate mask derived from sustained remote-DRAM or remote-HITM evidence for that `hctx`. If the intersection is empty, actuation is rejected. The only steady-state retarget primitive is `irq_set_affinity_and_hint()`, invoked from process context by a controller workqueue. `disable_irq()` is never used for routine retarget because the invention relies on established IRQ affinity APIs rather than implicit vector remap. `synchronize_irq()` is reserved for teardown or ownership release after the controller has stopped issuing updates and after locks taken by the handler are dropped; it is never called from the same IRQ thread or from hot-path contexts. Ownership states are `NONE`, `DRIVER`, `PEBS`, and `EXTERNAL`. `PEBS` may claim ownership only after the vector has remained stable for multiple ON windows. Any external affinity change detected by notifier or effective-mask mismatch makes `EXTERNAL` win immediately, restores the baseline managed mask, increments `policy_epoch`, and places the vector in a 5-minute hold-down. Rollback also occurs if p99 completion latency regresses by more than 5% for 3 consecutive 10-second windows, throughput drops by more than 2% for 3 windows, eligibility is lost, or quality falls below ON. One retarget per vector per 60 seconds bounds oscillation.

2.4 Architecture

The architecture is a narrow bridge from Intel PEBS evidence to blk-mq locality control.

Control-flow diagram:

boot or module load
  -> load compile-time region descriptors
  -> reject duplicate descriptor tuples
  -> derive audited `region_uid` values
  -> register steerable `blk_mq_hw_ctx` sources with `{source_cookie, hctx_cookie, irq, src_gen}`

completion path enter
  -> hardirq / threaded IRQ / softirq-NAPI / poll wrapper
  -> push `{region_uid, source_cookie, hctx_cookie, cpu, pmu_epoch, ++ctx_seq}` onto same-CPU scope stack

NMI PEBS drain
  -> perf validates DS record
  -> controller hook reads `data_src`
  -> map to bounded `data_src_class`
  -> snapshot top-of-stack on same CPU
  -> append minimal tuple to per-CPU ring

process-context worker
  -> revalidate `source_cookie`, `hctx_cookie`, `src_gen`, `pmu_epoch`, `ctx_seq`
  -> aggregate by `hctx` and decision window
  -> evaluate OFF / DEGRADED / ON state machine
  -> if ON and IRQ is eligible:
       target = managed baseline
                & housekeeping
                & cpuset effective mask
                & online mask
                & driver hint
                & NUMA candidate
       -> `irq_set_affinity_and_hint(irq, target)`
       -> dwell timer and rollback monitor

completion path exit
  -> pop scope entry
  -> restore parent entry if nested

Key invariants:
- No sample crosses CPUs, epochs, or context frames.
- No torn-down `hctx` can be acted upon because deferred aggregation revalidates `src_gen` and tombstone state.
- No vector is moved unless it is dedicated, managed, and ownership-clean.
- No debug or reporting interface is part of the control plane; audit views are read-only.

2.5 Implementation Plan

1. Add a Kconfig-gated x86-only controller that depends on Intel PEBS support and `PERF_SAMPLE_DATA_SRC` availability. Keep the default mode observe-only. 2. In the x86 perf PEBS DS-record drain path, after perf has validated the record, add a minimal hook that reads `perf_sample_data::data_src`, snapshots the per-CPU scope stack top, stamps the current `pmu_epoch`, and appends a fixed-size tuple to an NMI-safe per-CPU ring buffer. The hook exports no raw IP, virtual address, or physical address. 3. Introduce `include/linux/blk_pebs.h` and a blk-mq controller implementation that provides region macros, per-CPU scope stack helpers, and source-registry APIs. Add opt-in wrappers for completion scopes in blk-mq aware drivers and transports: hardirq handler wrappers, threaded IRQ wrappers, softirq/NAPI completion batch wrappers, and poll batch wrappers. For threaded IRQ and poll batches, use `migrate_disable()` only around the annotated batch so the attribution key remains CPU-stable without pinning unrelated work. 4. Implement the compile-time region registry loader and collision checker. Persist no region mapping across reboot or module rebuild; derive `region_uid` fresh each boot or module load from the descriptor tuple. Expose a read-only audit manifest and fail closed if collisions are detected. 5. Implement the generation-safe source registry keyed by live `request_queue` and `blk_mq_hw_ctx` lifetime. Increment `src_gen` on queue-map rebuilds, controller reset, suspend/resume, IRQ re-request, and hotplug-driven queue topology changes. Tie per-CPU `pmu_epoch` increments to PEBS event enable/disable, DS reprogramming, perf multiplex schedule transitions, throttling resets, and CPU online/offline events. 6. Implement the aggregation worker, quality state machine, and ownership-aware affinity transaction engine. Use `irq_set_affinity_and_hint()` for steady-state movement, reserve `synchronize_irq()` for teardown after ownership release, and attach an affinity notifier or equivalent effective-mask watcher so external changes win immediately. Rate-limit to one retarget per 60 seconds and enforce a 5-minute hold-down after rollback or external override. 7. Enable actuation only for data queues that satisfy the dedicated managed-IRQ rules. Explicitly exclude shared vectors, admin queues, poll-only queues, NAPI-only completions, and drivers that declare internal steering ownership. All excluded paths still may contribute observe-only attribution if annotated. 8. Land the feature in stages: first observe-only attribution and audit views, then quality-state gating and regression monitors, then managed-IRQ actuation for a limited set of opt-in storage drivers such as PCIe NVMe and virtual block transports that can prove one-vector-to-one-hctx completion ownership.

2.6 Validation Plan

1. Correctness and concurrency validation. Build kernel tests that force nested hardirq, threaded IRQ, softirq/NAPI, and poll completion sequences and verify the invariant that a sample is attributed only when same-CPU stack liveness, `pmu_epoch`, `ctx_seq`, and `src_gen` all match. Include CPU hotplug during sampling, queue reset during aggregation, suspend/resume, and IRQ re-request cases. 2. Negative eligibility validation. Confirm that shared IRQs, admin queues, polled mode, NAPI-only completion paths, non-managed vectors, per-CPU IRQs, immutable affinity domains, and drivers with internal steering ownership all remain observe-only and never enter the actuation state. 3. PMU quality validation. Force perf multiplexing, PEBS throttling, lost-record injection, and model variations with missing remote or HITM bits. Verify OFF and DEGRADED transitions occur at the declared thresholds and that no actuation occurs until ON requirements are satisfied for three consecutive windows. 4. Policy coexistence validation. Run with irqbalance, cpuset isolation, nohz_full housekeeping constraints, and manual affinity changes. Verify target masks are always the intersection of managed baseline, housekeeping, cpuset, online, driver hint, and NUMA candidate masks; verify external changes immediately revoke `PEBS` ownership and trigger rollback plus hold-down. 5. Performance validation. On dual-socket NVMe, NVMe-oF, and virtio-blk systems, compare baseline versus observe-only versus actuation modes using mixed read and write workloads. Measure p50, p99, and p99.9 completion latency, IOPS, CPU cycles per I/O, remote-DRAM sample share, and remote-HITM sample share. Success criteria are reduced remote/HITM share and improved or neutral p99 latency without violating rollback thresholds. 6. Safety validation. Confirm no raw IP or address data leaves the PEBS drain hook, confirm `synchronize_irq()` is invoked only from process-context teardown paths, and confirm counter wrap protection forces OFF rather than permitting aliasing.

4. Risks & Mitigations

  • SKU variability risk: some Intel CPUs expose only coarse `data_src` qualifiers. Mitigation: map to `COARSE_CACHE` or `COARSE_RAM`, cap quality at DEGRADED, and forbid actuation on those models.
  • Attribution coverage risk: not every completion path is naturally wrapped. Mitigation: require explicit driver or transport opt-in for hardirq, threaded IRQ, softirq/NAPI, and poll batches, and treat uncovered paths as unattributed rather than guessed.
  • Concurrency and hotplug risk: stale mappings can arise during queue reset, migration, or CPU offline events. Mitigation: use per-CPU `pmu_epoch`, per-CPU `ctx_seq`, per-source `src_gen`, tombstones, and fail-closed OFF transitions during hotplug churn.
  • IRQ policy conflict risk: irqbalance, cpuset changes, or driver resets may concurrently alter affinity. Mitigation: use managed-mask intersection, effective-mask monitoring, explicit ownership states, immediate external-owner precedence, rollback, and a 5-minute hold-down.
  • Oscillation risk: workloads may change faster than the controller can stabilize. Mitigation: require multiple ON windows before claim, permit at most one retarget per vector per 60 seconds, and roll back on sustained regression instead of on single-window noise.
  • Latency overhead risk in NMI context: PEBS drain hooks must stay extremely small. Mitigation: write only a fixed-size tuple to preallocated per-CPU storage, export no raw addresses, and defer all blk/IRQ resolution and policy work to a process-context worker.
  • Eligibility misclassification risk: shared vectors or internally steered drivers could be moved incorrectly. Mitigation: require single-action non-shared descriptors, one-to-one `hctx` mapping, managed affinity, and explicit driver capability flags before actuation.
