1. Executive Summary

One-Liner

Use an immutable, epoch-pinned APIC/LLC/NUMA locality contract and a per-mm single-flight flush publication state machine to give x86 TLB shootdowns a stable target snapshot with hierarchical APIC-domain acknowledgment, bound first-pass `lru_add_drain_all()` fan-out behind deterministic convergence rules, and rank only policy-eligible PMem/CXL demotion targets with decaying refault suppression.
Novelty: The invention is not generic topology awareness and it is not a restatement of existing `mmu_gather` batching. The new element is a shared control-plane abstraction: an immutable locality contract, rebuilt only on hotplu
Feasibility: Every part maps to existing Linux and x86 machinery. The locality contract can be built from CPUID/APIC topology already parsed by x86, keyed primarily by `X86_FEATURE_EXTAPIC` plu

2. Technical Invention Disclosure

2.1 Problem Statement

Large x86 Linux systems pay three related memory-management costs whose causes are topologically local but whose synchronization scope is often unnecessarily broad. First, reclaim needs up-to-date LRU state. Today a reclaim event that is effectively local to one node can still trigger broad `lru_add_drain_all()` fan-out, causing cross-socket IPIs and shared-cache contention even when the useful residue is concentrated near the allocating CPUs. On DRAM plus PMem/CXL systems, this overhead appears exactly when reclaim pressure is already high. Second, x86 remote TLB invalidation is correctness-critical and latency-sensitive. Existing Linux batching through `tlb_gather_mmu()` and `tlb_finish_mmu()` reduces the number of flush calls, but the actual remote shootdown still fundamentally uses flat target enumeration and per-call metadata. The hard problems are unchanged: page-table free safety, CPUs entering or leaving an mm during the flush window, lazy TLB state, `kthread_use_mm()` borrowers, and CPU hotplug. A performance optimization that does not define a stable target-snapshot contract and an ordering rule for late-arriving CPUs risks missed invalidations or hung waiters. Third, tiered-memory demotion already respects mempolicy, cpusets, and memory tiers, but among policy-eligible slow tiers it still lacks a topology contract that says which destination is near in the same sense used for reclaim and shootdown. Without that, PMem/CXL demotion can oscillate into a technically allowed but topologically poor target, causing fast refaults and wasted media bandwidth. The invention addresses all three through one shared control-plane object while preserving Linux's existing reclaim, TLB, and policy semantics.

2.2 Prior-Art Gap

The closest Linux mechanisms solve adjacent pieces but not the disclosed bridge. 1. `mmu_gather` / `tlb_finish_mmu` batch page-table tear-down work, but their scope is a caller-local invalidation episode. They do not create a persistent, per-mm published descriptor in `mm_context_t` that another CPU can observe during `switch_mm_irqs_off()` and use to self-flush if it joined the mm after publication. 2. Current x86 shootdown uses cpumask-based targeting and waiting, with `flush_tlb_info` lifetime tied to the current flush call. That is different from an immutable, RCU-retired publication object with explicit state transitions, merge rules, epoch pinning, and hierarchical acknowledgment over a fixed APIC-domain membership snapshot. 3. Generic tree barriers or per-node counters are not enough. The novelty is specifically that the hierarchy is derived from x86 APIC/x2APIC topology, each CPU belongs to exactly one leaf domain in the pinned epoch, and the same pinned contract is consumed by reclaim and demotion under deterministic quality gates. 4. Existing memory-tier demotion orders eligible nodes by tier relationships and distance, but it does not share a hotplug-safe APIC/LLC/NUMA contract with shootdown and reclaim, nor does it keep a per-(source,destination) decaying refault suppression score tied to that same epoch. 5. Existing reclaim drain behavior does not provide a measurable quality gate and bounded convergence rule that says: one local-first attempt is allowed only if topology completeness and historical efficacy are good, and otherwise the kernel must escalate within a fixed number of tries to node/global draining. The patent boundary is therefore the epoch-pinned locality contract plus the per-mm publication/late-arrival/hotplug rules, with reclaim and demotion as dependent applications of that single abstraction.

2.3 Proposed Invention

The invention introduces two new kernel-visible control-plane objects and strict rules for how they are used. A. Epoch-pinned locality contract - An immutable locality contract is built under CPU-hotplug serialization and published with RCU. - Each contract carries: `epoch`, APIC mode flags, `cpu_online_snapshot`, `cpu_to_leaf_domain[]`, `leaf_domain_cpumasks[]`, parent aggregation masks, LLC/die/package/NUMA identifiers, a bounded distance matrix for policy-eligible demotion ranking, and quality metrics. - A CPU belongs to exactly one leaf domain in a contract epoch. Parent domains are fixed aggregations of those leaves; no CPU can migrate between domains within the lifetime of a contract. - Preferred construction order is: x2APIC logical cluster when `X86_FEATURE_EXTAPIC` is present and Linux is actually running in x2APIC logical destination mode; otherwise LLC sibling group, then die/package, then NUMA node as a conservative fallback. - When topology changes, a new contract with `epoch + 1` is built; old contracts remain valid for already-published flushes until RCU retirement. B. Per-mm single-flight publication object - A per-mm pointer stored in `mm_context_t` references the currently published flush object, if any. - The flush object contains: `flush_seq`, pinned `epoch`, immutable target cpumask snapshot, immutable leaf expected-count array, flush class (`range`, `fullmm`, or forced-global fallback), address range, page-size class, PCID/INVPCID mode bits, `freed_tables` flag, and state. - States are explicit: 1. `IDLE`: no published object. 2. `PUBLISHED`: object is visible via `smp_store_release()`, compatible callers may merge, late-arriving CPUs may observe the pending sequence. 3. `SEALED`: sender froze the object with `cmpxchg`; no further merge is allowed; IPIs or self-flush obligations are now defined by the sealed snapshot. 4. `COMPLETED`: every CPU in the sealed target snapshot has either acknowledged after invalidation or has been completed by the CPU-hotplug dead-state rule. 5. `RETIRED`: object detached from `mm_context_t` and awaiting RCU free. - Who may transition: - The first caller for a given mm publishes `IDLE -> PUBLISHED`. - Concurrent callers may merge only while state is `PUBLISHED`, the pinned epoch matches, and the requests are merge-compatible. - The sender performing the actual remote invalidation seals `PUBLISHED -> SEALED`. - The initiator or designated completion path sets `SEALED -> COMPLETED` after observing all required acknowledgments. - The final holder retires `COMPLETED -> RETIRED` and drops the RCU reference. - Memory ordering: - Publisher fully initializes the object, then uses `smp_store_release(pub_ptr, obj)`. - Consumers use `smp_load_acquire(pub_ptr)` before reading fields. - Remote CPUs execute local invalidation first and only then release-store their leaf completion or per-cpu completion sequence. - Waiters acquire-load completion state before permitting `tlb_finish_mmu()` to free page tables. C. Stable target-snapshot contract and late-arrival rule - The sealed target mask is `mm_cpumask(mm) & cpu_online_snapshot` captured while holding `cpus_read_lock()` during publication/sealing. - That sealed mask is authoritative for CPUs already associated with the mm at seal time, even if they later go lazy or context switch away. - A CPU that enters the mm after publication is handled by generation, not by racing to edit the target mask. In `switch_mm_irqs_off()` and any `kthread_use_mm()`-style entry path, the CPU compares the mm's published `flush_seq` with its per-mm last-observed sequence. If behind, it must perform the required local invalidation before returning to user mode or before using borrowed page tables, then store the observed sequence. - This rule makes the snapshot stable while still preventing missed flushes for late joiners. D. Hotplug rule - Snapshot construction and sealing happen under `cpus_read_lock()` so the online set cannot change during target definition. - If a CPU was in the sealed snapshot and goes offline before sending an acknowledgment, the flush does not treat `cpu_online == false` as equivalent to completion. Instead, the CPU-hotplug teardown path records completion only after the target CPU has crossed the architecture-defined dead boundary where it can no longer execute the mm without coming back online and passing through the entry rule again. That hotplug-dead completion is counted against the sealed snapshot, so waiters cannot hang and cannot falsely complete early. - Epoch rebuild due to hotplug never mutates an in-flight flush; the flush keeps using its pinned contract epoch. E. Hierarchical acknowledgment - For each flush, expected counts are precomputed as the intersection of the sealed target mask with each leaf-domain membership mask in the pinned epoch. - Remote CPUs acknowledge only their leaf domain. Parents aggregate fixed child counts. This prevents undercounting or overcounting because the domain map is immutable for the flush. - If x2APIC logical-cluster delivery is enabled at runtime, the initiator may reduce IPI fan-out by sending per-leaf or per-cluster. If logical delivery is unavailable, disabled, or administratively turned off, Linux uses the current flat send path but still waits on the same sealed snapshot and leaf counts. F. Coalescing rules - Merge-compatible range flushes must satisfy all of the following: same mm; same pinned epoch; both are user-space, non-global invalidations; neither requires immediate page-table freeing before completion; neither crosses a hugepage split/collapse or page-size transition; both share the same PCID/INVPCID semantics already accepted by current x86 code paths. - A `fullmm` request absorbs a pending range request for the same mm by sealing the object as `fullmm`. - Mandatory escalation to the existing full/flat path occurs for any kernel/global mapping flush, any request requiring a global TLB semantic change, any hugepage split/collapse ambiguity, any incompatible PCID mode, any epoch mismatch, any publication already sealed with incompatible fields, or any case where page-table pages could be freed before all targets complete. - The invention never changes the local x86 invalidation primitive: Linux still uses `INVPCID` when available and appropriate, otherwise the current CR3/context-flush behavior. G. Quality-gated reclaim and demotion consumers - Reclaim may use local-first drain only if: topology completeness is 100% for online CPUs; no contract rebuild occurred in the last 1000 ms; historical local-first efficacy for the pgdat is at least 90% over the last 128 sampled attempts; and the last sampled residual pages after forced escalation stayed below 8% of scanned pages. - Reclaim convergence rule: at most one local-first drain attempt is allowed per reclaim episode. If after one `shrink_node()` pass reclaimed pages are below 25% of target, or if 200 microseconds elapse without sufficient progress, the kernel escalates immediately to existing node/global drain behavior. A second local-first attempt is forbidden until a new reclaim episode. - Demotion eligibility order is strict: mempolicy and cpuset constraints first, then memcg charge legality, then existing memory-tier eligibility, then topology ranking among the remaining nodes, then refault suppression as a ranking penalty only. - Refault suppression is per `(src_nid,dst_nid)` pair, based on sampled demotion provenance and workingset refaults. Penalty increases only when sampled pages refault within two workingset windows after demotion; penalty halves every 10 seconds, clears on topology epoch change or 60 seconds of inactivity, and cannot eliminate the last policy-eligible destination. If all eligible destinations are penalized, the kernel still picks the lowest-penalty eligible node to avoid starvation. H. Optional measurement assistance - Optional PMU/LBR freeze support such as `X86_FEATURE_AMD_LBR_PMC_FREEZE` may be used to gather cleaner calibration samples for locality quality, persistent-memory access cost, and refault attribution. These counters only influence whether an optimization remains enabled; they never gate correctness.

2.4 Architecture

Control-plane objects

1. `apic_locality_contract` (RCU, immutable, epoch E)
   - APIC mode flags
   - online CPU snapshot
   - cpu -> leaf-domain map
   - leaf-domain -> cpumask map
   - parent aggregation map
   - LLC/die/package/NUMA IDs
   - demotion distance matrix
   - quality metrics

2. `mm_context_t::published_flush`
   - `flush_seq`
   - pinned `epoch`
   - sealed target cpumask
   - leaf expected counts
   - flush class and range
   - PCID/INVPCID mode bits
   - `freed_tables`
   - state: `IDLE/PUBLISHED/SEALED/COMPLETED/RETIRED`

ASCII flow

`flush_tlb_mm_range(mm, start, end, flags)`
    -> acquire `cpus_read_lock()`
    -> snapshot `mm_cpumask(mm) & online_mask`
    -> bind snapshot to locality contract epoch E
    -> publish per-mm object (`PUBLISHED`)
    -> compatible callers may widen range or escalate to `fullmm`
    -> sender seals object (`SEALED`)
    -> send IPIs by x2APIC logical cluster if runtime-enabled
       else send current flat cpumask IPIs
    -> remote CPU:
         load-acquire published object
         execute local invalidate (`INVPCID` or current CR3 path)
         release-store leaf-domain completion / observed seq
    -> parent/root aggregate fixed leaf counts
    -> initiator observes all expected completions (`COMPLETED`)
    -> `tlb_finish_mmu()` may now free page-table pages
    -> detach and RCU-retire object (`RETIRED`)

Late-arrival path

`switch_mm_irqs_off()` or borrowed-mm entry
    -> load mm published seq
    -> if CPU last-seen seq < published seq
          self-invalidate locally before running with mm
          store observed seq

Hotplug path

CPU offline teardown
    -> cross CPU-dead boundary
    -> record completion for any sealed snapshot that still expects this CPU
    -> wake waiters

Reclaim consumer

reclaim on pgdat N
    -> consult locality contract quality gate
    -> optional local-first `lru_add_drain_all()` scope = leaf/package near N
    -> if insufficient progress after one pass or 200 us
          escalate to existing node/global drain

Demotion consumer

eligible destinations = policy filter first
    -> rank by locality contract distance
    -> add refault penalty by `(src,dst)` pair
    -> choose best remaining node
    -> if all penalized, choose lowest-penalty eligible node

The same epoch-pinned contract feeds all three consumers, but only the TLB path is allowed to affect correctness-critical completion. Reclaim and demotion are advisory users with mandatory fallback.

2.5 Implementation Plan

1. Add the locality-contract builder on x86 topology change boundaries. - Build a new immutable contract when CPU hotplug or APIC topology discovery changes. - Publish via RCU after full initialization. - Record whether runtime mode is actually x2APIC logical destination mode; do not infer that from CPUID alone. - Store quality-gate counters in a separate mutable statistics block so the contract object itself remains immutable. 2. Extend `mm_context_t` with publication metadata. - Add a per-mm published pointer, monotonically increasing `flush_seq`, and per-mm enable/disable flags. - Keep publication state arch-owned so the novelty boundary is explicit and separate from temporary `flush_tlb_info` lifetime. 3. Wire the shootdown path. - In `flush_tlb_mm_range()` and friends, create or merge a publication object under the compatibility rules. - Seal before sending any remote invalidation. - Teach the completion path to count leaf-domain acknowledgments against the sealed snapshot. - Ensure `tlb_finish_mmu()` waits only on `COMPLETED` or direct fallback completion before freeing page tables. 4. Add late-arrival self-flush and borrowed-mm handling. - In `switch_mm_irqs_off()` and borrowed-mm entry/exit paths, compare the mm published sequence with the CPU's last-observed sequence. - Force local invalidation on mismatch before executing the mm. - Keep the rule independent of whether the CPU was in the original sealed target snapshot. 5. Add hotplug completion integration. - Hold `cpus_read_lock()` across snapshot construction and sealing. - On CPU teardown, record completion only after the target CPU is past the dead boundary and cannot continue to execute the mm. - Do not mutate the sealed target mask or expected counts of an in-flight flush. 6. Add reclaim local-first gate. - Hook the first-pass drain decision near reclaim entry where `lru_add_drain_all()` is currently considered. - Keep the feature off unless topology completeness, stability window, and sampled efficacy thresholds are met. - Enforce immediate escalation after one failed local-first pass. 7. Add demotion ranking. - In memory-tier demotion selection, retain existing mempolicy/cpuset/memcg filters exactly as today. - Only after policy filtering, apply locality ranking and decaying refault penalty. - Store sampled provenance in optional metadata such as `page_ext` or other existing sampling infrastructure; keep it probabilistic to control overhead. 8. Add observability and self-disable. - Export tracepoints for publication state transitions, seal conflicts, late self-flushes, hotplug completions, reclaim escalations, and demotion penalties. - Automatically disable hierarchical mode for a cooldown period if late self-flush rate exceeds 2% over 10,000 flushes, if hotplug rebuild frequency exceeds one rebuild per second, or if local-first reclaim efficacy drops below threshold. 9. Apply persistent-memory calibration. - Use existing application performance tooling and optional PMU/LBR freeze support to characterize whether APIC-locality predictions correlate with PMem/CXL access behavior on the target platform. - Treat those measurements as enable/disable hints only.

1. Ordering and safety invariants - Prove and test invariant A: a remote CPU's acknowledgment is not visible until its local invalidation has completed. - Prove and test invariant B: any CPU that begins executing an mm after the object is published must either be present in the sealed snapshot or perform the late self-flush before use. - Prove and test invariant C: page-table pages are not freed until every CPU from the sealed snapshot has acknowledged or has been completed by the CPU-dead hotplug rule. 2. Concurrency stress - Exercise concurrent `mprotect`, `munmap`, THP split/collapse, `fork`/`exit`, and `kthread_use_mm()` while forcing high shootdown rates. - Run with lockdep, KCSAN, and CPU-hotplug churn to verify no missed flushes, no hung waiters, and no use-after-free during RCU retirement. 3. Semantic coverage matrix - Validate range flush, full-mm flush, PCID-enabled flush, INVPCID-disabled fallback, kernel/global flush fallback, and hugepage split escalation. - Validate CPUs entering and leaving `mm_cpumask(mm)` during the flush window, including lazy TLB transitions. 4. Hotplug-specific testing - Offline CPUs while a sealed snapshot is in flight. - Confirm that waiters complete only after CPU-dead completion, not merely because the online bit changed. - Confirm old epoch objects remain valid until retirement while new flushes use the new epoch. 5. Reclaim convergence testing - On single-node pressure and multi-socket pressure, measure whether the first-pass local drain reduces IPIs without stalling reclaim. - Verify the deterministic fallback: no reclaim episode may attempt more than one local-first pass; by the second pass behavior must be identical to existing node/global draining. 6. Demotion correctness testing - Build workloads with strict mempolicy, cpuset, and memcg constraints; confirm the ranking layer never violates them. - On DRAM plus PMem/CXL, measure demotion throughput, refault rate, and starvation behavior. - Verify suppression decay, clear, and last-destination override rules. 7. Persistent-memory-oriented workload evaluation - Use PMem/CXL databases, filesystems, and analytics traces to evaluate p99 latency, shootdown tail, reclaim IPI count, and refault oscillation. - Use optional PMU/LBR freeze support only to improve sample cleanliness and confirm that correctness remains unchanged when counters are unavailable. 8. A/B comparison against current Linux - Compare against stock flat shootdown plus existing `mmu_gather` batching, current `lru_add_drain_all()` behavior, and stock demotion ranking. - Key metrics: median and p99 shootdown latency, number of remote IPIs per flush, reclaim completion time, pages scanned per reclaimed page, demotion success rate, refaults per GiB demoted, and feature auto-disable frequency.

4. Risks & Mitigations

  • Risk: prior-art overlap with generic tree barriers or existing `mmu_gather` batching. Mitigation: claim the arch-specific per-mm publication object in `mm_context_t`, the late-arrival self-flush rule, the hotplug-dead completion rule, and the epoch-pinned APIC-domain membership snapshot rather than generic batching alone.
  • Risk: high topology churn from CPU hotplug or unstable firmware topology. Mitigation: rebuild immutable contracts by epoch, pin each flush to one epoch, and auto-disable optimization paths when rebuild frequency exceeds threshold.
  • Risk: false completion or missed completion during hotplug. Mitigation: never equate `cpu_online == false` with completion; count a departing CPU only after CPU-dead teardown guarantees it cannot resume the mm without re-entry and self-flush.
  • Risk: unsafe coalescing across incompatible flush semantics. Mitigation: define narrow merge compatibility and mandatory escalation for global mappings, hugepage transitions, incompatible PCID classes, or any case that could free page tables before completion.
  • Risk: reclaim local-first mode could delay convergence. Mitigation: require explicit quality metrics and bound the optimization to one attempt per reclaim episode with immediate escalation on insufficient progress or elapsed time.
  • Risk: demotion suppression could starve an otherwise valid destination. Mitigation: make suppression a ranking penalty only, decay it on a fixed schedule, clear it on epoch change or inactivity, and always permit the lowest-penalty eligible destination when alternatives are exhausted.
  • Risk: extra metadata or acknowledgment counters could add cacheline contention. Mitigation: align leaf counters per cacheline, keep the contract immutable, and sample demotion provenance rather than annotating every page movement.
  • Risk: machine-specific locality assumptions may not correlate with persistent-memory behavior. Mitigation: keep PMU/LBR-assisted calibration optional and disable the optimization when measured efficacy does not exceed thresholds.
