2. Technical Invention Disclosure

2.1 Problem Statement

Linux drivers repeatedly pay a correctness and latency penalty when multiple software producers mutate shared hardware-facing state. In register-bank paths, parent MFD or PMIC drivers expose overlapping bits to multiple clients, so immediate regmap_update_bits style traffic serializes callers and still emits redundant bus transactions even when several writes collapse to one final shadow value. In DMA-ring paths, multiple producers must reserve slots, initialize descriptors, make those descriptors DMA-visible, publish software metadata, ring a doorbell, and later reclaim slots without reusing entries the device can still fetch or prefetch. Existing drivers often solve these as separate ad hoc patterns. That leaves subtle bugs around torn metadata publication, ring wrap-around, stale observations after wrap, missed doorbells, owner stalls, and incorrect use of writel, readl, dma_wmb, or dma_sync APIs. The invention unifies both mutation surfaces under a generation-sealed commit-frontier model with an explicit hardware-visible completion contract.

2.2 Prior-Art Gap

Closest known art falls into four buckets: regcache or regmap delayed write-back for register devices; multi-producer descriptor rings that rely on per-descriptor ready bits and tail-doorbell batching; generic TSX lock-elision techniques; and per-driver doorbell coalescing heuristics. None of those disclose a driver-local abstraction in which logical reservation order is decoupled from publish order by a separate immutable commit-record plane, then rejoined by a contiguous-frontier algorithm that returns a precise generation token to each caller. The void evidence is directionally consistent with that gap: there is sparse adjacency between x86 feature-gated fast paths and PMIC register enable semantics, between matrix-style summary structures and driver publication logic, and between driver commit logic and hypervisor-aware x86 capability control. Those voids support a cross-domain invention that bridges register-bank commit coalescing, bitmap-assisted frontier advancement, and virtualization-aware TSX gating.

2.3 Proposed Invention

Each driver instance owns a commit-frontier object with two layers. Layer one is a mutation staging plane: a producer reserves a logical sequence range, fills either shadow register state or DMA descriptors, and computes a per-range generation tag. Layer two is a commit-record plane: once the staged payload is fully initialized and made device-visible, the producer writes a compact commit record and publishes a single release-ordered commit_word. A single combiner, called the frontier owner, examines commit records in reservation order, advances the maximal contiguous ready frontier, performs the required hardware publication step such as a regmap bus write or MMIO tail doorbell, and then marks the corresponding generation tokens complete. For DMA rings, descriptors carry a generation or phase tag and a valid or owner field written last so the device cannot legally consume a partially initialized slot even if it speculatively prefetches cachelines. For register-bank drivers, multiple overlapping bit updates collapse into one immutable sealed generation before the bus transaction is issued. The same token API exposes completion when and only when the committed generation is hardware-visible under the driver's contract.

2.5 Implementation Plan

Phase 1 implements the generic commit-frontier library and the non-TSX path for one coherent DMA ring driver. Phase 2 adds ready-bitmap frontier advancement, generation tokens, and reclaim-tag validation. Phase 3 adds a register-bank adapter above regmap or bus-write helpers so overlapping bit updates collapse into sealed generations. Phase 4 adds non-coherent DMA support, IOMMU-specific validation, and owner handoff paths for sleepable publication. Phase 5 adds the optional RTM reservation fast path behind x86 feature and policy gates.

2.6 Validation Plan

Validate both correctness and speed. Correctness validation uses lockdep, KCSAN, fault injection, wraparound stress, forced owner preemption, CPU hotplug, non-coherent DMA emulation, and virtualization scenarios with RTM masked off. Performance validation measures lock hold time, MMIO doorbell count, bus write count, tail latency, throughput, and token completion latency versus a legacy direct path. The key acceptance criterion is unchanged externally visible semantics with lower bus or MMIO traffic and lower producer contention.

4. Risks & Mitigations

  • Risk: prior art may cover parts of the DMA-ring submechanics. Mitigation: claim the separate commit-record plane, contiguous-frontier bitmap algorithm, and generation-token completion semantics as the primary combination.
  • Risk: a single owner can become a throughput bottleneck. Mitigation: owner work is limited to frontier advancement and one publication action per contiguous batch; producers remain lock-free after publication, and owner rescue plus handoff prevents stalls.
  • Risk: devices without per-descriptor validity may weaken the descriptor-safety story. Mitigation: use shadow staging and copy the fully sealed batch to hardware-visible memory only when complete, then doorbell once.
  • Risk: non-coherent DMA paths are easy to implement incorrectly. Mitigation: make sync-for-device and sync-for-cpu placement explicit in the adapter API and forbid driver-specific shortcuts.
  • Risk: RTM may be unavailable or unreliable in guests or on hardened hosts. Mitigation: keep RTM non-essential and disable it by default unless the platform policy explicitly permits it.
  • Risk: cacheline sharing between adjacent commit records may hurt scalability. Mitigation: offer a 64-byte padded record mode for high-contention deployments without changing publication rules.
  • Risk: wrap-pad records consume logical space. Mitigation: they simplify correctness materially and can be optimized away only for hardware proven to support split contiguous segments safely.
