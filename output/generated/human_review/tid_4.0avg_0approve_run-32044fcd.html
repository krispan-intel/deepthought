2. Technical Invention Disclosure

2.1 Problem Statement

Linux expedited membarrier for an mm-scoped command must guarantee that every CPU that could execute user code under the target mm after the issuer's request has passed through the required remote ordering point before the syscall returns. The hard cases are not the common case; they are concurrent issuers, CPUs in lazy-TLB state, tasks migrating between CPUs during snapshot and dispatch, CPUs entering or leaving the mm while the issuer is sending IPIs, CPUs sitting in idle or guest mode, and CPUs being offlined. Existing descriptions are often underspecified because they talk about a send mask but do not define an exact snapshot moment, a precise membership rule aligned with Linux residency semantics, or an authoritative publish point that proves a remote CPU has completed the required action. On x86 there is a second, separable problem: APIC send cost matters on large systems, but x2APIC addressing and virtualization behavior are too platform-dependent to be trusted as the correctness proof. The invention therefore addresses both dimensions at once: a software-verifiable completion contract with exact membership invariants, and an APIC-aware transport planner whose only effect is performance.

2.2 Prior-Art Gap

Current Linux expedited membarrier already performs important pieces of the job: it supports private expedited commands, derives targets from mm residency state such as mm_cpumask, uses generic SMP call-function or IPI rendezvous machinery, and returns only after that generic rendezvous path reports completion. Generic Linux infrastructure also already includes per-CPU state, global command sequencing, and many fanout optimizations. The minimal novel delta is therefore not 'use a sequence' or 'use APIC topology'; those would be obvious. The gap is an explicit, claimable contract that: (1) defines moment S using a writer-serialized mm residency seqcount covering the exact membership transitions that determine whether a CPU still participates; (2) freezes target set T at S but closes post-S migration races by requiring entry-side and exit-side catch-up against mm->mb_pending_seq; (3) defines completion solely by a post-barrier store-release of per-CPU obs_seq and an issuer-side load-acquire test obs_seq >= r; and (4) treats any topology-aware x2APIC fanout as a safety-gated, non-authoritative transport layer. Existing APIC and smp_call_function prior art teaches ways to send interrupts, and existing Linux membarrier teaches expedited synchronization, but not this combined snapshot-bound proof obligation with explicit residency invariants and non-authoritative APIC optimization.

2.3 Proposed Invention

The proposed system is an x86_64 embodiment for MEMBARRIER_CMD_PRIVATE_EXPEDITED-class operations, optionally including PRIVATE_EXPEDITED_SYNC_CORE, with these concrete elements. 1. Request reservation and publication. The issuer reserves a unique 64-bit global sequence r from atomic64_t mb_global_seq. It then performs smp_store_release(&mm->mb_pending_seq, r). That store-release is the publication point that later entrants and leavers compare against. 2. Residency state and writer serialization. For an mm that has registered for private expedited commands, mm_context_t gains a raw spinlock mb_residency_lock and a seqcount mb_residency_seq. Writer-side transitions that change whether a CPU belongs to the mm-scoped residency set execute under that lock and seqcount: switch_mm into the mm, switch_mm away from the mm, enter_lazy_tlb for the mm, leave_mm for the mm, and the residency-clearing side of exit_mmap/teardown. The write-side critical section updates both the membership bit in mm_cpumask(mm) and the per-CPU residency mode for that mm (ACTIVE, LAZY, or NONE). Because per-CPU obs_seq storage exists for every online CPU from bringup to CPU_DEAD, readers cannot observe 'CPU in mask but ack storage uninitialized'. 3. Moment S and target set T. The issuer holds cpus_read_lock across snapshot, dispatch, and wait. It then repeats: begin = read_seqcount_begin(&mm->mb_residency_seq); copy T = mm_cpumask(mm) & cpu_online_mask; optionally snapshot per-CPU ACTIVE/LAZY tags for debugging or policy; retry if read_seqcount_retry says a writer raced. The successful seqcount read defines moment S. A CPU belongs to frozen set T iff at S it is marked ACTIVE or LAZY for that mm and therefore could resume a thread of that mm without first passing through a membership transition that would run the catch-up rule. This aligns T with the same residency notion Linux already uses for mm-scoped shootdown participation. 4. Remote action and publish point. For each cpu in T, the transport layer delivers a non-blocking notification. The remote action is exactly the existing membarrier command-specific action required by Linux for that command: for PRIVATE_EXPEDITED the existing full memory barrier path; for PRIVATE_EXPEDITED_SYNC_CORE the existing sync-core path before return to userspace. The handler must not sleep, allocate, or take sleeping locks. Immediately after the command-specific action, the CPU performs smp_store_release(&per_cpu(obs_seq, cpu), max(current_obs_seq, r)). This release is the only authoritative completion evidence. 5. Entry-side and exit-side catch-up. To avoid missing CPUs that join after S or leave before their bit is cleared, the invention adds two local rules. On switch_mm into the mm, or on the return-to-user path for a task running that mm, the CPU performs seq = smp_load_acquire(&mm->mb_pending_seq); if seq > obs_seq then it executes the same command-specific action locally and store-releases obs_seq = seq before userspace continues. Symmetrically, before a CPU clears its ACTIVE or LAZY membership for that mm in switch_mm away, leave_mm, or teardown, it again compares mm->mb_pending_seq against obs_seq and performs the same catch-up if needed before clearing the bit under mb_residency_seq. Therefore no CPU can resume userspace under the mm after S without a barrier at or after r, and no CPU can disappear from membership in a way that strands the waiter. 6. Completion predicate and concurrency rule. The issuer waits only on the frozen set T. For each cpu in T it repeatedly performs smp_load_acquire(&per_cpu(obs_seq, cpu)) and completes when every value is >= r. Equality is not required; >= naturally handles multiple issuers and coalescing. If a CPU processes a later request r2, publishing obs_seq = r2 also satisfies all earlier requests r1 <= r2. This is why the design uses 64-bit monotonic sequences and scopes the x86 embodiment to x86_64 atomic64_t platforms; 32-bit torn sequence handling is intentionally excluded. 7. Missed-IPI handling without APIC proof. The issuer never trusts APIC delivery-status bits, IRR/ISR state, or multicast completion. If a CPU in T is still lagging, the issuer recomputes a laggard subset L = { cpu in T | load_acquire(obs_seq[cpu]) < r } and resends notifications to L using the safe generic per-CPU path, typically physical unicast or the existing smp_call_function_single/many fallback. Forward progress depends only on eventual obs_seq publication or membership-transition catch-up, not on APIC transport metadata. 8. Hotplug, idle, guest mode. cpus_read_lock prevents a target CPU from completing the CPUHP offline transition while it is in T, so the waiter cannot lose track of a target due to hot-unplug. If an implementation wants shorter read-lock hold time, a CPUHP offline callback may perform the same catch-up and publish obs_seq before the CPU is removed from cpu_online_mask, but that is an optimization, not the primary proof. NOHZ idle CPUs are still safe because an IPI wakes them or the catch-up rule fires before they re-enter user mode. Guest-mode CPUs are safe because completion is not inferred from APIC state; the design simply waits for obs_seq. Under virtualization, the transport planner disables logical multicast and falls back to generic authoritative sends. 9. APIC-topology-aware planner. The send-side planner is purely an optimization. It always executes the issuer locally when self is in T. For remaining targets it defaults to x2APIC physical unicasts. It may batch a subset by full-cluster logical delivery only when all of the following hold: x2APIC is enabled; CPUID leaf 0x1F or 0xB yields a stable x2APIC cluster decomposition; the targeted subset equals an exact, full online cluster mask; topology generation has not changed since boot; the system is not running under a hypervisor-paravirtual APIC mode; interrupt remapping or other destination-rewriting facilities are absent or explicitly known not to rewrite x2APIC logical cluster semantics; and a kernel safety knob allows the mode. If any condition fails, the planner falls back to physical unicasts or existing generic fanout. APIC optimization can only reduce sends; it cannot alter the completion proof.

2.4 Architecture

Core objects:
- global atomic64_t mb_global_seq
- per-mm atomic64_t mb_pending_seq
- per-mm raw_spinlock_t mb_residency_lock + seqcount_t mb_residency_seq
- existing mm_cpumask(mm)
- per-CPU atomic64_t obs_seq

Control flow:

Issuer CPU
  reserve r = atomic64_inc_return(mb_global_seq)
  smp_store_release(mm->mb_pending_seq, r)
  cpus_read_lock()
  do {
      begin = read_seqcount_begin(mm->mb_residency_seq)
      T = mm_cpumask(mm) & cpu_online_mask
  } while (read_seqcount_retry(mm->mb_residency_seq, begin))   // defines moment S
  plan fanout(T, cpuid_topology, x2apic_mode, platform_gates)
  dispatch notifications to T
  while (exists cpu in T with smp_load_acquire(obs_seq[cpu]) < r) {
      L = laggards(T, r)
      resend L with authoritative unicast/generic path
  }
  cpus_read_unlock()
  return to caller

Remote/target CPU path
  on IPI, switch_mm into mm, switch_mm away from mm, leave_mm, or return-to-user under mm:
      seq = smp_load_acquire(mm->mb_pending_seq)
      if (seq > obs_seq_local) {
          existing membarrier remote action for command
          smp_store_release(obs_seq_local, seq)
      }

Writer-side residency transitions under mb_residency_lock + mb_residency_seq:
  switch_mm into mm      -> update per-CPU mode ACTIVE, set mm_cpumask bit
  enter_lazy_tlb(mm)     -> update per-CPU mode LAZY, keep/set mm_cpumask bit
  switch_mm away/leave   -> catch up if pending_seq > obs_seq, then clear or retag bit
  exit_mmap/teardown     -> clear membership only after catch-up rule

Key invariants:
- T is a frozen set at moment S.
- No CPU may return to userspace under mm with obs_seq < mm->mb_pending_seq.
- No CPU may clear mm residency with obs_seq < mm->mb_pending_seq.
- APIC transport is advisory; obs_seq is authoritative.

2.5 Implementation Plan

1. Add opt-in state to mm_context_t for tasks that register for private expedited membarrier: mb_pending_seq, mb_residency_lock, and mb_residency_seq. Keep the feature disabled for unregistered mms so ordinary context-switch paths do not pay the extra serialization cost. 2. Introduce a small per-CPU structure containing obs_seq and a residency-mode tag used only for registered mms. Initialize it at CPU bringup and keep it valid until CPU_DEAD. 3. Instrument the residency transition sites already relevant to mm ownership: switch_mm_irqs_off, enter_lazy_tlb, leave_mm, and the teardown path that removes the mm from runnable residency. Under mb_residency_lock plus seqcount, update mm_cpumask and the per-CPU ACTIVE/LAZY/NONE tag atomically from the reader's perspective. Before clearing residency, run the exit-side catch-up rule if mm->mb_pending_seq exceeds local obs_seq. 4. Implement membarrier_expedited_snapshot(mm, cmd) that reserves r, store-releases mm->mb_pending_seq, takes the seqcount-stable snapshot T under cpus_read_lock, and records whether self is in T. This helper becomes the front door for the new protocol. 5. Implement the non-blocking remote handler wrapper. It must invoke the pre-existing command-specific membarrier action and then store-release obs_seq. Do not add sleeping locks, allocations, or debug control-plane dependencies inside the handler. 6. Add the entry-side catch-up check to the switch_mm/return-to-user path for registered mms so post-S entrants cannot run userspace with stale obs_seq. The check should be a fast compare against mm->mb_pending_seq and should short-circuit in the common case where no request is pending. 7. Build the x86 transport planner. The planner should consult boot-time topology parsed from CPUID 0x1F or 0xB, x2APIC enablement, hypervisor presence, interrupt-remapping status, and a topology-generation stamp. It should always support self-local execution and physical unicasts, and only enable full-cluster logical sends when the exact gating rules are satisfied. 8. Add a laggard resend loop that uses obs_seq as the sole truth source. If any CPU in T still has obs_seq < r after the first pass, resend to just that laggard set through the authoritative generic path. Ignore APIC delivery-status bits completely. 9. Scope the initial implementation to x86_64. State explicitly that 32-bit x86 is excluded from the claimed embodiment to avoid torn 64-bit sequence handling and because the APIC-topology optimization depends on deployment assumptions that are effectively x86_64-only in production.

2.6 Validation Plan

1. LKMM proof sketch. Model the issuer's smp_store_release(mm->mb_pending_seq, r), the target CPU's command-specific barrier followed by smp_store_release(obs_seq, r), and the issuer's smp_load_acquire(obs_seq) wait loop. Show that once the issuer observes obs_seq >= r, the target CPU has necessarily executed the required barrier action before that observation. On x86 TSO this reduces to ordinary loads/stores plus compiler barriers, but the release/acquire annotations make the proof explicit and portable. 2. Membership-race proof. Prove four invariants: (I1) a CPU in T at S cannot disappear without either handling the notification or performing exit-side catch-up before clearing residency; (I2) a CPU not in T at S cannot return to userspace under the mm without entry-side catch-up if mm->mb_pending_seq > obs_seq; (I3) seqcount retry prevents readers from observing torn membership transitions; and (I4) cpus_read_lock prevents hotplug from removing a target CPU from cpu_online_mask during snapshot/wait. 3. Concurrency stress. Run multi-issuer tests that create overlapping expedited membarrier requests on the same mm and on different mms, with forced task migration, rapid switch_mm churn, and back-to-back requests. Verify completion uses obs_seq >= r rather than equality and that coalesced later acknowledgments satisfy earlier requests. 4. CPU-state stress. Exercise NOHZ idle, deep idle exit, guest mode under KVM, and CPU hotplug. Confirm that logical multicast is automatically disabled under virtualization or interrupt-remapped environments and that cpus_read_lock or CPUHP catch-up prevents indefinite waits. 5. Transport-fallback tests. Inject failures or artificial delays into the initial APIC planner choice, including intentionally disabling full-cluster logical delivery after the first send. Verify that laggard resend through physical unicast or generic call-function paths still completes correctly because obs_seq is the only authority. 6. Performance evaluation. Benchmark private expedited membarrier latency and APIC write count on high-core-count x86_64 servers for sparse, medium-density, and full-cluster target sets. Compare local-only, physical-unicast-only, and gated logical-cluster modes. Measure not just median latency but p99/p999 tail latency and total send operations. 7. Regression checks. Confirm no change in semantics for unregistered mms, no sleeping or debugfs/procfs dependencies in the handler, and no dependency on APIC delivery-status bits. Confirm that disabling the APIC planner still leaves the snapshot-bound completion contract intact.

4. Risks & Mitigations

  • Risk: prior-art overlap with existing Linux expedited membarrier and smp_call_function fanout. Mitigation: keep independent claims centered on the snapshot-bound completion contract, entry/exit catch-up invariants, and post-barrier obs_seq publication; treat APIC fanout as a dependent optional embodiment.
  • Risk: added context-switch overhead from writer-side residency serialization. Mitigation: enable the extra lock and seqcount only for mms that explicitly register for private expedited membarrier, and keep the common no-pending-request fast path to a single compare against mm->mb_pending_seq.
  • Risk: topology-driven logical delivery behaves differently under virtualization or interrupt remapping. Mitigation: default to physical unicast, disable logical clustering when X86_FEATURE_HYPERVISOR or remapping ambiguity is present, and never use transport metadata as completion evidence.
  • Risk: subtle race bugs around lazy-TLB state, migration, and teardown. Mitigation: serialize all membership-affecting transitions under the same seqcount domain, require catch-up before residency clear or user return, and validate with LKMM plus migration/hotplug stress tests.
  • Risk: indefinite wait on CPUs in idle, guest mode, or hotplug transitions. Mitigation: hold cpus_read_lock across snapshot/send/wait as the primary guarantee, optionally add CPUHP offline catch-up for shorter lock hold, and use obs_seq-based laggard resend instead of trusting a one-shot IPI.
