2. Technical Invention Disclosure

2.1 Problem Statement

Linux high-throughput device drivers expose a real but dangerous optimization surface spanning hardware-visible and software-visible knobs. In the live-switchable scope of this disclosure, the allowed runtime classes are limited to predeclared numeric fields such as irq_coalesce_usecs, irq_coalesce_frames, tx completion batch threshold, refill_low_wm, descriptor prefetch distance, poll_budget_cap, queue_cpu_mask_id, irq_vector_affinity_id, CQE compression or moderation mode bits expressly marked online-safe by the driver, and service_program_id only when that ID selects among precompiled const micro-op tables interpreted by fixed driver code rather than swapping code or function pointers. A larger surface may be declared but only as REQUIRES_FULL_REINIT, including ring_depth, queue_count, MSI-X vector count, descriptor_layout_id, queue object topology, DMA mapping format, and any field that requires allocation, free, queue creation, or reopen. The kernel problem is that these knobs trade throughput, latency proxy, cache locality, interrupt rate, and energy, but the datapaths that would benefit from adaptation run in IRQ, softirq/NAPI, or blk-mq contexts where sleeping, energy reads, scoring, and partial reprogramming are unsafe. Naive adaptation risks mixed-epoch fast-path execution, stale CPU-mask references, races with ethtool or reset paths, incorrect posted-MMIO ordering, broken DMA ownership transitions, and oscillation caused by package-scoped energy noise.

2.2 Prior-Art Gap

Existing RAPL and powercap facilities primarily measure or cap power; they do not define a driver-safe actuation boundary, do not solve device/package attribution limits, and do not provide safe degradation when RAPL is unavailable or untrustworthy. Existing DIM-style adaptive interrupt moderation, devfreq-like governors, and queue-affinity tuners may run from timers or workqueues, but they still synthesize settings online, usually target one or two knobs, and generally lack: a precise ring/DMA IR; build-time equality-saturation search over coupled hardware/software parameters; explicit proof obligations covering DMA ownership, descriptor bounds, barrier placement, and IRQ-context constraints; immutable build-generated finite variant tables; per-field metadata for visibility and quiesce requirements; a single serialization point shared with all existing driver reconfiguration paths; and a formal PREPARE->QUIESCE->PROGRAM->VERIFY->RESUME->PUBLISH protocol with release/acquire fast-path publication. That boundary, not mere workqueue execution, is the inventive wall.

    • Switching protocol order is internally inconsistent: it says PREPARE->QUIESCE->PROGRAM->VERIFY->PUBLISH->RESUME (and elsewhere mentions RESUME before/after PUBLISH). You must define a single canonical order and justify it with a concrete invariant: no fast-path reader can observe a variant index that refers to partially-programmed HW state. Typically this means RESUME (reenable IRQ/NAPI/queues) must happen after PUBLISH only if PUBLISH is purely about exposing already-safe state; otherwise PUBLISH must occur after RESUME plus a confirmed quiesce boundary for readers.
    • The publication model is underspecified and risks mixed-epoch access. Saying “smp_store_release active variant index” is not enough unless you also specify what readers load, in what contexts (NAPI/IRQ/softirq), and what constitutes an epoch boundary (e.g., per-queue NAPI poll loop, per-xmit). If any reader caches pointers/derived parameters, you need either (a) a single immutable table with index-only reads and no pointer chasing, or (b) RCU (or seqlock) with explicit lifetime rules. Spell out exactly what data is protected by the mutex vs. what is lockless, and prove no UAF / torn read.
    • Quiesce requirements are too hand-wavy across subsystems. For netdev, you must specify the exact sequence (e.g., rtnl_lock? netif_device_detach? napi_disable per NAPI instance? synchronize_net? disable_irq/synchronize_irq?) and what it guarantees for TX and RX paths. For blk-mq, “blk_mq_quiesce_queue” vs “blk_mq_freeze_queue” have different semantics; you must state which is required per knob class and ensure no in-flight request observes reprogrammed doorbells/register layouts.
    • Rollback semantics are not defined at the hardware level. If PROGRAM partially succeeds then VERIFY fails, you need an explicit revert-to-known-good sequence that is safe under the same quiesce model, including re-validating device state (ring pointers, DMA mappings, interrupt moderation registers). Without a specified ‘golden’ baseline and idempotent programming rules, rollback can be unsafe or impossible on real devices.

Also define sampling windowing and overflow handling for energy counters so the guardrail cannot oscillate due to wraparound or noisy attribution.
