2. Technical Invention Disclosure

2.1 Problem Statement

Intel AMX makes context handoff materially more expensive when a task's saved architectural state contains non-init XTILEDATA, because preserving and restoring tile payload is substantially heavier than ordinary vector state and is often coupled to matrix-heavy cache footprints. Baseline Linux CFS understands generic locality, idle CPU selection, load, capacity, and cache-hot heuristics, but it does not have a scheduler-safe, low-overhead indicator of the narrower fact that matters here: the task's last authoritative saved software fpstate actually contained XTILEDATA, and therefore future migration or cross-LLC wake placement is likely to incur extra xstate restore and locality cost. Permission-oriented signals such as XCR0 bits, AMX enablement, or XFD state are too broad because many tasks are AMX-eligible yet never materialize tile state. Conversely, reading live registers or forcing XSAVE from the scheduler would be architecturally unsafe and too expensive. The problem is therefore to expose a precise AMX handoff-cost hint to CFS using only xstate images that the kernel has already legally materialized, and then to consume that hint only as a bounded tie-breaker that does not override feasibility, fairness, isolation, or capacity rules.

2.2 Prior-Art Gap

The gap over prior art is the conjunction of five elements that existing Linux and x86 mechanisms do not provide together. First, existing scheduler heuristics such as wake_affine, select_idle_sibling, cache-hot checks, last_cpu preference, util_est, and generic load balancing reason about recency and load, not about whether a saved architectural image actually contains XTILEDATA. Second, x86 AMX permission and trap machinery such as XCR0/XSS enablement, arch_prctl permission state, and XFD describe eligibility or first-use policy, not authoritative saved payload presence; they cannot safely distinguish a task that is AMX-enabled but tile-unused from one whose last saved image truly contains XTILEDATA. Third, existing FPU bookkeeping and lazy/optimized save machinery are correctness and performance mechanisms for state management, not scheduler inputs with explicit invalidation, isolation scoping, and hot-path bounds. Fourth, XSAVEOPT and XSAVES reduce save/restore work but do not teach the scheduler when to prefer same-LLC placement or how to do so without parsing live state. Fifth, no known prior art combines a saved-image-derived XTILEDATA bit, a masked and quantized XTILECFG bucket, a formally defined isolation-domain token, a per-rq decaying history keyed by that token, and a hard-capped near-tie-only LLC-local CFS bias with deterministic fallback. This is why the proposal is not an obvious variant of cache-hot or recent-CPU bias: it uses a different underlying fact, gathered at different kernel hook points, with different correctness and security constraints.

2.3 Proposed Invention

The invention introduces two new scheduler data structures and updates them only on existing xstate transition paths. Each task gains an `amx_sched_meta` record stored in `task_struct->thread` adjacent to `thread.fpu`, containing at least: `flags` (`VALID`, `COSTLY`, `BUCKET_VALID`, `TRUSTED_DOMAIN`), `bucket` (2 bits plus valid flag), `iso_token` (64-bit hashed isolation-domain token), `iso_gen` (generation for domain changes), and `publish_seq` or equivalent release-published versioning. Each runqueue gains a small `amx_rq_hist` record containing `seqcount_t seq`, `u64 iso_token`, `u32 last_epoch`, `u8 bucket`, and `u8 strength`; this is about one cacheline fraction per CPU and is local to the owning CPU. Primary signal derivation is tightly constrained. The scheduler never inspects live tile registers and never forces XSAVE. Instead, metadata is refreshed only after the kernel has already materialized a stable software fpstate for correctness reasons. There are three writer classes. (1) Save-path writers: immediately after the x86 save helper has stored a task's user xstate into `task->thread.fpu.fpstate` during context-switch or other existing save points, while the task's state is serialized and before the metadata is published. (2) Import-path writers: immediately after a successful ptrace/regset write, sigreturn restore, or other xstate import path has validated user input, handled standard versus compacted format, and normalized the result into the task's software fpstate. (3) Reset/invalidation writers: execve, any helper that reinitializes fpstate, AMX permission disable, XFD/XCOMP/XCR0 policy changes affecting user AMX state, and explicit xstate-clearing paths. Fork/clone is handled conservatively: the child starts with `VALID=0`, `COSTLY=0`, and `BUCKET_VALID=0` even if fpstate bytes were copied, so a parent's historical hint is never inherited as a scheduler fact until the child itself reaches a legal save or import point. At those save/import hooks, the code reads the normalized software XSAVE header's `xfeatures` bitmap, not a raw user buffer and not merely `xcomp_bv`. `xcomp_bv` is consulted only by the existing import validator to determine whether incoming state was in compacted format and to normalize offsets; the scheduler hint is derived afterward from the normalized header. The primary bit is `XTILEDATA_PRESENT = header.xfeatures & XFEATURE_MASK_XTILEDATA`, additionally masked by the task's currently permitted user xfeature mask so that stale pre-disable bits cannot survive an AMX permission change. If the bit is absent, `COSTLY` is cleared. If present, `COSTLY` is set. Optional bucket derivation occurs only when `header.xfeatures` also contains `XFEATURE_MASK_XTILECFG` and the XTILECFG payload has already passed the kernel's normal validation rules. Bucketization is deliberately coarse to reduce fingerprinting. The implementation reads only validated XTILECFG fields needed for broad footprint classes: whether a non-init palette is selected, the number of active tiles (rows[i] and colsb[i] both non-zero), and aggregate configured tile bytes `sum(rows[i] * colsb[i])`. It then quantizes those into a tiny tuple such as `{palette_nonzero, active_tiles_bin in [0-2,3-8], bytes_bin in [<=256,257-1024,>1024]}` and hashes that tuple with a per-boot secret before truncating to a 4-way bucket. Raw rows/colsb arrays are never retained in scheduler metadata, the bucket label changes every boot, and `BUCKET_VALID` is clear if any field is unavailable or validation failed. This keeps the information content intentionally low while still distinguishing small, medium, and large AMX footprints. The isolation-domain token is defined formally as `iso_token = siphash_3u64(core_sched_cookie_or_0, effective_cpuset_partition_seq, cpu_cgroup_css_serial_or_0, boot_secret_domain)`. The token is computed only in slow paths where those identities are already stable under existing cgroup/cpuset/core-sched synchronization, then cached in the task metadata. Hot scheduler paths never walk cgroup state; they only read the cached token with `READ_ONCE()` or load-acquire semantics. Any change to effective core-sched cookie, cpuset partition, or CPU cgroup membership increments `iso_gen`, clears `TRUSTED_DOMAIN`, `VALID`, and `BUCKET_VALID`, and republishes the metadata. Because runqueue history requires exact `iso_token` equality and short expiry, old-domain history becomes unusable immediately after a token change and ages out without any cross-domain reuse. Full shared-history matching is enabled only when a trusted domain can be formed, i.e., when core scheduling is active and the token incorporates a non-zero effective cookie plus stable cpuset/cgroup identities. Otherwise the design falls back to per-task same-LLC bias only, with no shared per-rq history at all. Per-rq decaying history is implemented in a way that is safe under rq-locking and cheap under contention. `rq->amx_rq_hist` is written only by the CPU that owns that runqueue, under `rq->lock`, and only on events that already touch scheduling state locally: a costly fair-class task switching out, being dequeued after a minimum runtime slice, or completing a migration onto that CPU. There are no remote writes on wakeup, so there is no cross-CPU cacheline bouncing from the mechanism itself. Decay is mostly read-side, not write-side: the history stores `last_epoch` and a small saturating `strength` counter, and readers compute `effective_strength = max(0, strength - floor((now_epoch - last_epoch) / half_life_epochs))`. A practical configuration is a 2 ms half-life with 8 ms maximum lifetime. Readers in wakeup placement and periodic balance use `read_seqcount_begin/retry` around lockless reads of the remote CPU's history; if a read races, the hint is simply treated as absent. Because the history is only advisory, the correctness rule is fail-open to baseline CFS, never stall or spin. Tie-breaker math is explicit and bounded. In `select_task_rq_fair`, the mechanism is entered only if a static key is enabled, the task is fair-class, `VALID && COSTLY` is true, and baseline CFS has already produced a feasible candidate set after cpuset, affinity, uclamp, asym-capacity, core-sched, SMT, and wake-affine checks. Let `best_base` be the baseline best score already computed by CFS. Only candidate CPUs in the same LLC as the task's current LLC or previous CPU's LLC, and only those with `base_delta <= near_tie`, are eligible. A concrete bound is `near_tie = max(8, capacity_orig_of(best_cpu) / 64)` on the standard 1024 util scale. The AMX bonus is then capped at 6 util points total: `+4` for same LLC as the task's last CPU, `+2` if the candidate runqueue's trusted-domain history matches both `iso_token` and `bucket` with non-zero effective strength. The total bonus is always less than the near-tie window, so it cannot overturn a clearly better baseline placement. If any precondition fails, or if no same-LLC candidate remains within the near-tie window, the scheduler returns the unmodified baseline choice. Under asymmetric capacity the bonus is applied only after `fits_capacity()` and capacity-pressure rules succeed; under SMT/core-sched it is applied only among candidates already admissible under existing cookie checks. In periodic or active load balance, the design never invents a new imbalance metric. It is used only after the existing balancer has already determined that migration is permitted and multiple destination CPUs are otherwise near-equivalent within the same LLC scheduling domain. The same bonus cap is then used only to choose among those already-legal destinations. It cannot block a required migration, cannot pull a task into a forbidden CPU set, and cannot move RT/DL tasks because the mechanism is fair-class-only. Ordering and race rules are explicit. Writer paths first complete the fpstate save/import/reset, then derive the AMX metadata, then publish metadata with store-release. Scheduler readers use load-acquire or `READ_ONCE()` on the published fields and never dereference the fpstate itself. Therefore the scheduler cannot observe `COSTLY=1` before the corresponding saved image exists, and it cannot race a live FPU state transition because it consumes only cached metadata. This directly addresses the materialization/stability requirement. The side-channel model is similarly explicit. The shared-history path is restricted to trusted domains only. Bucket cardinality is intentionally tiny, bucket labels are per-boot keyed, history is keyed by hashed domain token, history expires quickly, and untrusted tasks get at most their own same-LLC bias or no bonus at all. Because one isolation domain never matches another domain's token, scheduler placement differences caused by one tenant's AMX footprint are not reused for another tenant. For stronger deployments, the bucket/history subfeature can be disabled and the invention still retains the core novelty of XTILEDATA-derived same-task LLC bias. Hot-path cost is bounded. With the static key disabled, non-AMX workloads pay one predicted-false branch. With the key enabled but the task not `VALID && COSTLY`, the extra work is a few cached loads and branches. Only for costly tasks already being considered within an LLC domain does the scheduler perform O(1) seqcount reads of tiny per-rq history records for candidate CPUs it was already scanning. The invention therefore converts real x86 xstate facts into a scheduler hint with explicit correctness, fairness, and isolation bounds instead of a speculative heuristic.

2. The XTILEDATA fact comes from normalized header.xfeatures after a legal save/import.

2.5 Implementation Plan

1. Add `struct amx_sched_meta` to x86 task state near `thread.fpu` and add `struct amx_rq_hist` plus a static key to scheduler state. 2. Instrument x86 FPU/xstate save helpers to refresh AMX metadata immediately after software fpstate materialization; instrument xstate import/regset/sigreturn paths to refresh metadata only after validation and normalization succeed. 3. Implement reset/invalidation hooks for execve, fpstate reinit, AMX enable/disable policy changes, ptrace/regset replacement, sigreturn restore, and fork/clone child initialization; add slow-path token recomputation hooks for core-sched cookie changes, cpuset partition changes, and CPU cgroup migration. 4. Implement XTILECFG quantization using only validated fields, 4-way boot-keyed hashing, and explicit `BUCKET_VALID` publication; never store raw XTILECFG arrays in scheduler metadata. 5. Implement per-rq history update under `rq->lock` on costly fair-task switch-out/dequeue/migration-in, using seqcount-protected lockless reads and read-side time decay; do not write remote runqueues from wakeup placement. 6. Modify `select_task_rq_fair` and same-LLC load-balance destination choice so the AMX bonus is considered only after baseline feasibility checks and only within the explicit near-tie window; cap the total bonus below the near-tie threshold. 7. Gate the entire feature behind a static key and x86 feature checks so systems without AMX, or systems where operators disable the feature, retain exact baseline behavior. 8. Expose tracepoints or schedstat counters for `amx_costly_set`, `amx_bucket_valid`, `amx_bias_applied`, `amx_bias_skipped_no_domain`, and `amx_history_match` to support evaluation without exposing raw per-task bucket values to user space.

2.6 Validation Plan

The evaluation plan is kernel-realistic and measures both benefit and cost. 1. Correctness validation: - Unit-test the metadata transitions for save, import, execve, fork/clone, ptrace xstate write, sigreturn restore, AMX permission disable, cpuset move, CPU cgroup attach, and core-sched cookie change. - Verify that `COSTLY` tracks only materialized `header.xfeatures & XTILEDATA` and is cleared on every invalidation event. - Verify that compacted XSAVES user buffers do not affect scheduling until the import path normalizes them. 2. Hot-path overhead: - Measure instruction count and branch count in `select_task_rq_fair` with static key off, with key on but non-costly tasks, and with costly tasks inside an LLC domain. - Measure rq-lock hold time deltas and seqcount retry rates on large Xeon systems. - Confirm there are no remote wakeup writes to other CPUs' history records. 3. Performance benefit: - Run AMX-heavy GEMM/BF16 and matrix-factorization workloads, including mixed AMX and non-AMX colocations. - Measure cross-LLC migration count, XTILEDATA restore count, throughput, tail latency, and LLC miss rate. - Compare baseline CFS, same-task-only LLC bias, and full trusted-domain history mode. 4. Fairness and scheduler health: - Measure vruntime skew, runnable latency, starvation indicators, and load-balance convergence under mixed workloads. - Stress asymmetric-capacity, SMT/core-sched, and cpuset-isolated configurations to ensure the feature never overrides existing admissibility checks. 5. Security and leakage: - In multi-tenant experiments, verify that tasks in different tokens never receive history matches from one another, that bucket labels change after reboot, and that disabling trusted-domain formation eliminates shared-history effects. Success criteria are: statistically significant reduction in cross-LLC migrations and AMX restore-heavy handoffs for AMX-costly workloads, no measurable regression for non-AMX workloads with static key off or on, bounded rq-lock overhead, and no cross-token history reuse.

4. Risks & Mitigations

  • Risk: incorrect interpretation of XSAVE/XSAVES layout could create false positives. Mitigation: derive the scheduling signal only from the normalized software fpstate after the existing import/save code has handled compacted-vs-standard format, and consult `header.xfeatures` rather than raw user offsets.
  • Risk: stale metadata could survive task lifecycle or isolation changes. Mitigation: use exhaustive invalidation on execve, fork/clone child start, ptrace/regset replacement, sigreturn restore, AMX policy changes, fpstate resets, cpuset/cgroup migration, and core-sched cookie changes, with release-published metadata and generation increments.
  • Risk: per-rq history could add contention or cacheline bouncing. Mitigation: make writes local-only under `rq->lock`, avoid remote writes on wakeup, use static-key gating, and use read-side decay plus seqcount lockless reads.
  • Risk: fairness regressions from an oversized bias. Mitigation: apply the feature only inside the baseline feasible set, only within an explicit near-tie window, and cap the total bonus below that window with immediate fallback to baseline CFS otherwise.
  • Risk: scheduler-visible differences could leak cross-tenant information. Mitigation: restrict shared history to trusted domains only, hash the isolation token, use a tiny boot-keyed bucket space, expire history quickly, and allow operators to disable bucket/history mode while retaining same-task LLC bias only.
  • Risk: bucketization could be either too coarse to help or too fine to be safe. Mitigation: start with a fixed 4-way bucket based on validated XTILECFG aggregate classes, measure benefit in AMX-heavy workloads, and keep raw XTILECFG detail out of scheduler metadata.
