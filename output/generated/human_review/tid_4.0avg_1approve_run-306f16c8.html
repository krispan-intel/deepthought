2. Technical Invention Disclosure

2.1 Problem Statement

High-throughput network workloads (100Gbps+ NICs, DPDK-bypass kernel paths, cloud-native microservices) can drive CPU package power beyond RAPL PL1 sustained limits, triggering coarse-grained hardware power throttling (PL2 clamp or thermal throttle) that causes unpredictable tail latency spikes of 10–100ms. Current Linux NAPI has no feedback path from the power domain to packet processing scheduling decisions: netdev_budget (default 300) is a static sysctl with no runtime adaptation. Cloud providers running thousands of heterogeneous network workloads per host need per-socket energy budget enforcement for network I/O to support (a) carbon-aware scheduling SLOs, (b) power capping for rack-level capacity planning, and (c) predictable tail latency under thermal constraints—all without resorting to the blunt instrument of hardware throttling. The impedance mismatch between thermal sensor update rates (~1–100ms) and packet arrival rates (nanoseconds at line rate) means the governor must operate at the NAPI poll scheduling granularity (per-poll-cycle, ~microseconds), not per-packet, making NAPI budget adjustment the correct control surface rather than per-packet steering. Additionally, the control hierarchy must be explicitly defined: CPU frequency governors (intel_pstate, schedutil) manage P-states and set the overall package power envelope; this NAPI budget governor operates strictly as a subordinate leaf consumer that reads RAPL energy headroom—defined as PL1_limit minus current_package_power_EMA—and adjusts only the network scheduling budget, never competing with or overriding frequency governor decisions. This separation prevents the feedback instability where P-state transitions would be misattributed to NAPI activity: the governor's error signal is (energy_headroom - network_energy_target), not raw package power, isolating it from non-network power variations.

2.2 Prior-Art Gap

A thorough prior art search across kernel subsystems, industry solutions, and userspace power management confirms no existing mechanism bridges RAPL energy domains to NAPI scheduling: (1) DIM (net/core/dim/) adjusts interrupt coalescing parameters based on throughput/latency profiles but has zero power-domain inputs. (2) The powercap/RAPL framework (drivers/powercap/intel_rapl_common.c) exposes MSR_PKG_ENERGY_STATUS to userspace via sysfs but provides no in-kernel notification, callback, or subscription API for other kernel consumers. (3) CPU frequency governors (cpufreq/intel_pstate) manage P-states globally but do not interact with NAPI scheduling or per-netdev budgets; crucially, our governor does not duplicate their role—it consumes the energy headroom they establish. (4) Intel DPDK's RTE_FLOW API provides hardware offload classification and RSS steering but operates entirely in userspace with polled PMDs—no kernel energy feedback loop exists. (5) Cilium's eBPF/XDP datapath and Facebook's Katran load balancer perform topology-aware packet steering via bpf_redirect_map() but make no power-domain queries. (6) Intel Resource Director Technology (RDT, via resctrl) provides LLC and memory bandwidth partitioning per-CLOSID but does not integrate with network scheduling or NAPI budgets. (7) The thermal_zone framework provides trip-point notifications but these are coarse (critical/hot/passive) and not designed for continuous proportional control of network subsystem parameters. (8) Userspace power management daemons—TLP (thinkpad-specific runtime PM), laptop-mode-tools (disk/VM writeback tuning), tuned (static sysctl profile application), and systemd power-profiles-daemon (platform_profile switching)—none modulate NAPI budgets or per-netdev scheduling parameters; tuned's 'network-throughput' profile sets static sysctl values (netdev_budget, tcp_rmem) but performs no closed-loop energy feedback. (9) Vendor NIC drivers (e1000e, igb, ixgbe, ice, mlx5) implement EEE advertisement and static tx_lpi_timer configuration via ethtool but none modulate EEE timers based on CPU power state, RAPL readings, or frequency governor transitions. (10) The LAN88XX PHY driver and LRELPA_10_1PAIR register definitions (identified in void analysis) define extended page-space and pair configurations for Microchip/SMSC Ethernet PHYs but provide no power-feedback hooks—these represent unexplored register spaces where energy-aware timer modulation could be implemented on supporting hardware. No prior art implements a closed-loop controller translating continuous RAPL energy-rate measurements into NAPI budget scaling factors with formally specified controller gains and stability guarantees.

2.3 Proposed Invention

A new kernel governor module (net/core/energy_napi_gov.c) implementing a formally specified closed-loop energy-to-NAPI-budget controller with the following concrete design: (1) ENERGY SAMPLING KTHREAD: One kthread per populated CPU socket, pinned to a housekeeping core on that socket via cpu_to_node() affinity. Each kthread reads MSR_PKG_ENERGY_STATUS via rdmsrl_safe() at a configurable interval T_sample (default 1ms, range 100μs–10ms). The raw 32-bit energy counter (unit: 2^{-ESU} joules, where ESU is from MSR_RAPL_POWER_UNIT bits 12:8, typically ESU=14 giving ~61nJ resolution) is converted to instantaneous power via first-difference divided by T_sample, then smoothed with an exponential moving average (EMA) with decay factor α=0.125 (equivalent to ~8-sample window): P_ema[n] = α·P_raw[n] + (1-α)·P_ema[n-1]. CRITICAL NUMA DESIGN: Each kthread reads ONLY its local socket's RAPL MSR and produces a scaling factor consumed ONLY by NAPI instances whose associated NIC interrupts are affined to cores on that same socket (determined via irq_get_effective_affinity_mask() at NAPI enable time). Cross-socket coherency is eliminated by design—no NAPI instance ever reads a remote socket's seqcount. If a NIC's IRQ affinity spans multiple sockets, the governor uses the scaling factor of the socket hosting the majority of the IRQ's target cores, with a sysfs-visible per-netdev attribute showing the governing socket. (2) PI CONTROLLER WITH CONCRETE PARAMETERS: The controller computes error signal e[n] = (energy_headroom - P_network_target), where energy_headroom = PL1_limit - P_ema[n] (PL1 read from MSR_PKG_POWER_LIMIT bits 14:0 scaled by power unit), and P_network_target is a configurable per-socket parameter (default: 15% of PL1, range 5–50% via sysfs). The PI output u[n] = Kp·e[n] + Ki·Σe[k] with SPECIFIED GAINS: Kp=0.4 (dimensionless scaling factor per watt of error), Ki=0.05 (integral gain per sample period, providing ~20-sample integral time constant). Anti-windup: the integral accumulator is clamped to [−2.0, +2.0] to prevent saturation during sustained overload. The scaling factor sf = clamp(1.0 + u[n], sf_min, sf_max) where sf_min=0.1 (floor: never reduce NAPI budget below 10% of base) and sf_max=2.0 (ceiling: allow up to 2x base budget when energy headroom is abundant). HYSTERESIS DEADBAND: When |e[n]| < deadband (default: 5% of P_network_target, i.e., ±0.75W for a 100W TDP socket at 15% target), the controller holds its previous output—no update occurs. This prevents hunting during steady-state operation. STATE TRANSITIONS require N_confirm consecutive samples (default: 4) outside the deadband before the controller resumes active adjustment, providing anti-flapping protection with a ~4ms response latency at default T_sample. STABILITY ANALYSIS: For the discrete-time PI with T_sample=1ms, the open-loop crossover frequency is approximately Kp/(2π·T_sample) ≈ 64Hz, well below the Nyquist frequency of 500Hz. The phase margin is >60° for plant time constants >10ms (typical for thermal/power domains), ensuring no oscillatory behavior for workload transitions between idle and saturation. Bursty traffic (sub-millisecond bursts) is smoothed by the EMA filter before reaching the controller. (3) SEQCOUNT-PROTECTED PUBLICATION: The scaling factor is written by the kthread into a per-socket struct energy_napi_state using write_seqcount_begin()/write_seqcount_end() (include/linux/seqlock.h). NAPI poll completion paths (net/core/dev.c:napi_poll()) read via read_seqcount_begin()/read_seqcount_retry() to obtain the scaling factor locklessly. The adjusted budget is computed as: adjusted_budget = clamp(base_budget * sf, MIN_NAPI_BUDGET, netdev_budget). This adds exactly one seqcount read (2 memory barriers, ~5ns on x86) to the NAPI poll hot path—no MSR reads, no locks, no syscalls. (4) CONTROL HIERARCHY INTEGRATION: The governor is explicitly a leaf consumer in the power management hierarchy. It does NOT set P-states, does NOT write RAPL power limits, and does NOT interact with intel_pstate or schedutil. It reads PL1 and current energy as inputs and adjusts only NAPI budgets as output. If intel_pstate changes the P-state (altering package power), this appears as a change in energy_headroom which the governor correctly interprets—a P-state increase that raises package power reduces headroom, causing the governor to conservatively reduce NAPI budget, which is the correct behavior (the CPU is already power-constrained by frequency scaling). A P-state decrease increases headroom, allowing the governor to expand NAPI budget. There is no spurious attribution because the error signal is headroom-relative, not absolute power. (5) OPTIONAL EEE TIMER MODULATION EXTENSION: An optional sub-module (net/core/energy_napi_eee.c, compile-time CONFIG_ENERGY_NAPI_EEE) extends the governor to modulate tx_lpi_timer on supported PHYs. HARDWARE CAPABILITY DETECTION: At registration time, the governor queries phy_device->drv->flags for a new PHYDEV_CAP_RUNTIME_LPI_TIMER flag (defined by this invention as BIT(4) in include/linux/phy.h:phy_driver.flags). PHY drivers that support runtime tx_lpi_timer changes without IEEE 802.3az link renegotiation set this flag; initially, Intel igc (I225/I226) and Realtek r8169 (RTL8125B+) are candidates based on datasheet review showing separated LPI timer registers from auto-negotiation state machines. The governor modulates tx_lpi_timer within bounds [PHY_MIN_LPI_TIMER_US, PHY_MAX_LPI_TIMER_US] (default [20, 400] μs) using the same scaling factor, with an independent hysteresis deadband of ±10% to prevent PHY state machine churn. PER-NETDEV GRANULARITY: The EEE sub-module maintains a per-netdev energy attribution estimate using the NIC's hardware packet/byte counters (via ndo_get_stats64) to compute per-netdev power contribution as a fraction of socket total, enabling differential LPI aggressiveness—a saturated link gets aggressive LPI while an idle link on the same socket remains unaffected. ETHTOOL CONFLICT RESOLUTION: A new ethtool flag ETHTOOL_EEE_GOV_MANAGED (set via 'ethtool --set-eee dev gov-managed on') delegates tx_lpi_timer control to the governor. When set, userspace ethtool --set-eee writes to tx_lpi_timer are rejected with -EBUSY and a netdev_warn() message. When not set (default), the governor does not touch that netdev's EEE timers. The per-interface sysfs attribute /sys/class/net/<dev>/energy_gov/eee_managed provides read/write access to this flag. This explicit opt-in model eliminates ambiguity about the precedence hierarchy.

2.4 Architecture

The system comprises four architectural layers with strict NUMA-local data flow:

[Layer 1 – RAPL Sampling] Per-socket kthreads (one per populated socket, pinned to housekeeping cores) read local MSR_PKG_ENERGY_STATUS at 1ms intervals via rdmsrl_safe(). Raw energy deltas are converted to power and smoothed via EMA (α=0.125). Each kthread also reads PL1 from MSR_PKG_POWER_LIMIT at initialization and on RAPL power limit change notifications (if available via platform interrupt, otherwise re-read every 1000 samples ≈ 1s).

[Layer 2 – PI Controller] Each per-socket kthread runs an independent PI controller instance. Error signal: e = (PL1 × network_fraction_target) − P_ema. Controller output: u = Kp·e + Ki·∫e (Kp=0.4, Ki=0.05, integral clamped to [−2.0, +2.0]). Deadband: |e| < 5% of target → hold previous output. Anti-flapping: 4 consecutive samples outside deadband before resuming. Output scaling factor: sf = clamp(1.0 + u, 0.1, 2.0).

[Layer 3 – Seqcount Publication & NAPI Consumption] The scaling factor is published via per-socket seqcount (write_seqcount_begin/end). NAPI poll paths on napi_complete_done() read sf via read_seqcount_begin/retry. Socket affinity is resolved at NAPI enable time via irq_get_effective_affinity_mask() → cpu_to_node(). Adjusted budget = base_budget × sf, clamped to [MIN_NAPI_BUDGET, netdev_budget]. Cross-socket reads never occur.

[Layer 4 – Optional EEE Extension] If CONFIG_ENERGY_NAPI_EEE is enabled, a per-netdev EEE modulator consumes the same scaling factor and adjusts tx_lpi_timer on PHYs with PHYDEV_CAP_RUNTIME_LPI_TIMER capability. Per-netdev power attribution is estimated via hardware packet/byte counters. Ethtool conflict is resolved via explicit ETHTOOL_EEE_GOV_MANAGED opt-in flag.

Control hierarchy: intel_pstate/schedutil → sets P-states → determines package power envelope → RAPL-NAPI-governor reads residual headroom → adjusts NAPI budgets. The governor is a pure consumer of energy headroom, never a setter of power policy.

NUMA data flow guarantee: kthread[socket N] → seqcount[socket N] → NAPI instances on cores in socket N. No cross-socket memory access on the hot path. For NICs with IRQ affinity spanning sockets, the majority-socket rule applies with explicit sysfs visibility.

4. Risks & Mitigations

  • RISK: PI controller oscillation under bursty traffic patterns with sub-millisecond burst periods. MITIGATION: EMA pre-filter (α=0.125, ~8-sample smoothing window) attenuates high-frequency power transients before they reach the controller. Deadband (±5% of target) prevents controller response to noise-level fluctuations. Anti-flapping (4-sample confirm) prevents rapid output reversals. Phase margin analysis shows >60° stability margin for plant time constants >10ms. Configurable Kp/Ki via sysfs allows runtime tuning for specific workload profiles.
  • RISK: MSR read latency (~200ns for rdmsrl on modern x86) in kthread adds jitter to sampling interval. MITIGATION: kthread uses hrtimer-based scheduling (not msleep) for precise T_sample intervals. MSR read is <0.1% of the 1ms sampling period. Even with ±10% jitter, the PI controller's integral term provides steady-state accuracy and the EMA filter absorbs timing variations.
  • RISK: RAPL energy counter 32-bit wraparound (at 100W, the counter wraps every ~2621 seconds with ESU=14). MITIGATION: At 1ms sampling interval, consecutive reads differ by at most ~6.1μJ (100W × 1ms), far below the 32-bit range (~262kJ). Standard unsigned subtraction handles wraparound correctly. A safety check flags any delta exceeding 10× expected maximum as a missed sample and discards it.
  • RISK: Package-level RAPL conflates network and non-network power, causing the governor to react to CPU-intensive non-network workloads. MITIGATION: The energy headroom formulation (PL1 - P_ema) means the governor conservatively reduces NAPI budget when total package power is high, regardless of source—this is the correct behavior for power capping. For workloads where attribution matters, the per-netdev byte-counter-based power estimation in the EEE extension provides a coarse but actionable per-device signal. Future extension: Intel PECI or per-domain RAPL (if exposed for uncore/IO) could provide finer attribution.
  • RISK: Limited PHY hardware support for runtime tx_lpi_timer changes without link renegotiation. MITIGATION: The EEE extension is optional (CONFIG_ENERGY_NAPI_EEE) and requires explicit PHYDEV_CAP_RUNTIME_LPI_TIMER capability advertisement from PHY drivers. Initial targets are Intel igc (I225/I226) and Realtek r8169 (RTL8125B+) based on datasheet review. PHYs without this capability are simply skipped—the core NAPI budget governor functions independently of EEE support. The invention's primary value (NAPI budget scaling) requires zero PHY hardware dependencies.
  • RISK: Interaction with existing CPU frequency governors (intel_pstate, schedutil) causing feedback loops. MITIGATION: Architecturally eliminated by design—the governor reads RAPL counters and PL1 as inputs but never writes power limits or P-state requests. It operates as a leaf consumer of energy headroom established by the frequency governor. P-state changes appear as headroom changes that the governor correctly responds to (higher P-state → less headroom → more conservative NAPI budget, which is appropriate). No circular dependency exists because the NAPI budget change does not feed back into cpufreq governor inputs (cpufreq responds to CPU utilization, not network packet rates).
  • RISK: Ethtool/userspace conflict for EEE timer control. MITIGATION: Explicit ETHTOOL_EEE_GOV_MANAGED opt-in flag per netdev. Default is unmanaged (userspace retains control). When managed, ethtool writes return -EBUSY with netdev_warn(). Sysfs per-device attribute provides visibility. No implicit override—administrator must explicitly delegate EEE control to the governor.

⚠ Fatal Flaw: The core assumption that PHY EEE tx_lpi_timer can be adjusted at runtime without triggering link renegotiation is not supported by the vast majority of Ethernet PHYs in production (Intel I210/I211, Realtek RTL8111, Marvell 88E1512). The proposed PHYDEV_F_RUNTIME_LPI capability bit requires hardware support that does not exist—most PHYs require writing to MDIO registers that trigger autonegotiation restart. Without per-PHY hardware validation data, this design cannot be implemented.

  • The invention assumes PHY hardware supports runtime tx_lpi_timer adjustment without link renegotiation, but existing drivers (e1000e, igb, ixgbe, r8169) all write EEE registers during link setup and do not expose runtime reconfiguration. You must either (a) provide MDIO register sequences for specific PHYs proving renegotiation-free updates, or (b) redesign to accept brief link interruptions and specify maximum renegotiation frequency bounds.
  • The PI controller stability analysis is missing. With 500ms–2s sampling (Ts), a Ki=0.05 integral gain can wind up significantly during long low-power periods. You claim anti-windup clamping at ±50µs but do not prove this prevents oscillation when workload shifts from idle to line-rate in <100ms. Provide z-domain pole analysis or empirical step-response data showing <10% overshoot.
  • The interaction between delayed_work RAPL sampling and phylib's existing phy_ethtool_set_eee() is underspecified. phy_ethtool_set_eee() takes rtnl_lock; your governor's delayed_work callback must either (a) use rtnl_trylock and defer on contention, or (b) introduce a phylib-internal mutex. The current proposal does not specify locking hierarchy, risking ABBA deadlock with userspace ethtool calls.
  • Socket-local partitioning assumes netdev-to-NUMA-node affinity is static, but XPS (Transmit Packet Steering) and IRQ affinity can change at runtime via /proc/irq/*/smp_affinity.
