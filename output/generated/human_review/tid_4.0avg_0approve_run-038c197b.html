1. Executive Summary

One-Liner

A scheduler-owned, per-CPU kernel perf_event uses precise x86 PEBS mem-load-latency sampling to emit bounded stall tuples containing {rq_seq, task_cookie, exec_epoch, stall_class, weight} into an NMI-safe SPSC ring, and the fair scheduler drains a hard-capped number of tuples under the local rq->lock, accepts only tuples whose execution-instance key still matches the current or just-dequeued fair task on that runqueue, publishes a saturating and decaying per-task stall debt only at switch boundaries, and uses that debt only to apply tightly clamped earlier-resched and wakeup-preemption bias without modifying vruntime, vlag, weights, eligibility, parent cfs_rq shares, or group bandwidth accounting.
Novelty: The novelty is not merely 'using PEBS for scheduling'; it is the specific scheduler-safe attribution and publication contract that makes PEBS evidence legally and technically distinct from observability, cache heuristics
Feasibility: The design is feasible on production x86 Linux because it reuses existing perf/x86 PEBS plumbing, confines all PMI/NMI work to allocation-free per-CPU writes, performs no rq lockin

2. Technical Invention Disclosure

2.1 Problem Statement

Linux fair scheduling primarily reasons over vruntime, vlag, weight, runnable time, latency_nice, and fixed granularity thresholds. Those signals describe how long an entity has run or waited, but they do not directly represent that the currently running task is repeatedly incurring precise long-latency memory stalls while still nominally consuming CPU. Intel x86 PEBS can expose that evidence, but PEBS arrives in PMI/NMI context, where taking rq->lock, allocating memory, walking task lists, or mutating scheduler state is unsafe. A deployable invention therefore needs a kernel-owned PMU ownership model that coexists with perf, an attribution key that survives migration/exec/exit, an NMI-safe ring with explicit memory ordering, a bounded stall classification, and scheduler actuation rules that provably do not alter eligibility, vruntime monotonicity, cgroup shares, or starvation properties. Existing scheduler signals such as util_avg/util_est/PELT, PSI, iowait boosting, or generic cache-pressure heuristics do not solve that specific bridge problem because they are time-smeared or blocking-oriented signals rather than precise execution-instance stall evidence.

2.2 Prior-Art Gap

The unfilled gap is the absence of a kernel-safe, claimable bridge from PEBS sample evidence to fair-scheduler decisions with auditable attribution and bounded semantics. Existing PEBS- or perf-based research on stall-aware scheduling typically treats PEBS as a generic profiling input, global ranking signal, or migration heuristic, but does not define a Linux rq-local execution-instance contract with stale-sample rejection across switch, migration, execve, exit, throttling, and idle transitions. Cache- and memory-pressure schedulers generally use MPKI, LLC occupancy, PSI, or heuristic counters; they do not rely on precise PEBS load-latency tuples tied to a single current execution instance. Linux itself already has PSI, iowait boost, util_avg, util_est, and EAS, but those signals estimate utilization, blocked time, or energy/capacity, not instruction-precise backend memory stall behavior of the thread currently owning the CPU. The novelty boundary is therefore: not generic seqlock or ring mechanics; not 'use PEBS to bias scheduler' in the abstract; but specifically the execution-instance attribution key, NMI-safe tuple emission, stale-drop invariant, and switch-boundary debt publication that preserve EEVDF/CFS fairness contracts and group scheduling invariants.

2.3 Proposed Invention

The invention adds an optional scheduler feature, enabled only by root via sysctl and implemented as a kernel-internal per-CPU perf_event, that samples only precise user-mode PEBS retired memory-load latency events. Each CPU maintains: (a) a 64-bit rq_seq incremented under rq->lock whenever a leaf fair task instance becomes the current task on that rq; (b) a published run-instance record containing rq_seq, task_cookie, exec_epoch, and a valid bit; (c) an SPSC ring of fixed size, for example 256 entries, with producer-owned head and consumer-owned tail; and (d) per-task fields pebs_cookie, pebs_exec_epoch, pebs_last_rq_seq, pebs_debt, and pebs_debt_epoch. At fork, pebs_cookie is assigned from a global atomic64 counter. On successful execve, pebs_exec_epoch increments. On each fair switch-in under rq->lock, the scheduler updates rq clock, increments rq_seq, stores that rq_seq in p->pebs_last_rq_seq, and publishes {rq_seq, pebs_cookie, pebs_exec_epoch, valid=1}. On switch-out to idle or non-fair class, valid is cleared with release ordering. The PEBS PMI/NMI handler never locks the runqueue and never dereferences arbitrary tasks; it reads the published run-instance with acquire semantics, classifies the PEBS payload into a bounded stall_class and weight, and appends a tuple if and only if the published key is valid and the event is precise and non-throttled. A tuple contains exactly: rq_seq (64b), task_cookie (64b), exec_epoch (32b), stall_class (8b), weight (8b), flags (8b), and a reserved byte, padded to an aligned fixed-size entry. Preferred stall classes are: class 0 ignore; class 1 moderate memory stall, such as latency 192-255 cycles; class 2 severe local-memory stall, such as latency 256-511 cycles with local DRAM data source; class 3 severe remote-memory stall, such as latency >=512 cycles or remote cache/remote DRAM data source. The scheduler drains tuples only under the same CPU's rq->lock in fair-class paths, with a hard budget Bctx per switch path, for example 8, and an optional pressure-relief budget Btick, for example 2, in task_tick_fair only when occupancy exceeds a watermark. A tuple is accepted only if it matches the current or just-dequeued leaf fair task on that rq by all three key fields: rq_seq equals p->pebs_last_rq_seq, task_cookie equals p->pebs_cookie, and exec_epoch equals p->pebs_exec_epoch. All other tuples are dropped as stale. Accepted tuples add saturating quanta to pebs_debt, for example weights 1, 2, and 4 with Dmax=8. Debt decays by one quantum at each switch boundary in which the task ran without an accepted tuple, and additionally by one quantum per scheduler period equivalent while the task is runnable but not current, so debt is self-clearing. The only actuation is bounded real-time bias against the debt-bearing current task: (1) earlier resched eligibility by reducing uninterrupted residency by at most Delta_slice_max = min(base_slice/8, sysctl_sched_min_granularity/2) after the scheduler's normal slice computation; and (2) a reduced wakeup non-preemption margin against that task by at most Delta_wake_max = sysctl_sched_wakeup_granularity/4 after latency_nice and baseline wakeup-preempt calculations. No modification is made to vruntime, vlag, se->deadline, load weight, parent cfs_rq accounting, bandwidth quotas, or entity eligibility. Group scheduling is preserved because the debt is stored only on leaf tasks and is consulted only after the ordinary parent group entity has already won service under normal CFS/EEVDF rules. If a cfs_rq is throttled, if the task is no longer a leaf fair entity, or if the CPU-local PEBS event is inactive due to PMU conflicts, the mechanism neither accumulates new debt nor acts on existing debt beyond passive decay.

2.4 Architecture

Control flow and data ownership:

userspace admin
    |
    v
sysctl/cgroup root policy -> scheduler feature enable
    |
    v
per-CPU kernel perf_event (PEBS mem-load-latency, precise_ip>=2, pinned-opportunistic)
    |
    | PMI/NMI on precise sample
    v
NMI producer on same CPU
    1. smp_load_acquire(published_run_instance)
    2. classify PEBS payload -> stall_class, weight
    3. if invalid/multiplexed/throttled/guest/kernel: drop
    4. write tuple into per-CPU ring slot
    5. smp_store_release(head)

per-CPU published_run_instance under rq->lock
    set_next_task_fair()/switch-in:
        rq_seq++
        p->pebs_last_rq_seq = rq_seq
        publish {rq_seq, pebs_cookie, pebs_exec_epoch, valid=1}
    switch-out to idle/non-fair:
        publish {valid=0}

fair scheduler consumer under local rq->lock
    put_prev_task_fair() and optional watermark path in task_tick_fair()
        head = smp_load_acquire(ring.head)
        while tail != head and budget > 0:
            read tuple fields with READ_ONCE
            if tuple matches current/prev leaf fair task instance:
                pebs_debt = min(Dmax, pebs_debt + weight)
            else:
                drop as stale
            tail++
        smp_store_release(ring.tail, tail)
        decay debt and publish effective_debt only at switch boundary

scheduler actuation (no fairness math mutation)
    baseline CFS/EEVDF pick and eligibility unchanged
    if current task has effective_debt > 0:
        earlier resched by <= Delta_slice_max
        easier wakeup preemption against current by <= Delta_wake_max

Key invariants:
- PMI/NMI never touches rq->lock or task lists.
- Consumer never dereferences remote rq or remote task state.
- Accepted tuples must match {rq_seq, task_cookie, exec_epoch}.
- Eligibility, vruntime, vlag, weights, cgroup shares, and bandwidth enforcement are unchanged.
- On migration, execve, exit, idle transition, or throttling, stale tuples fail the key match and are discarded.

2.5 Implementation Plan

1. PMU ownership and gating: add CONFIG_SCHED_PEBS_BIAS plus a root-only sysctl, such as kernel.sched_pebs_bias=0/1, and an optional root-only cgroup knob to allow opt-in only for selected service classes. Instantiate one kernel-internal perf_event per CPU using the existing perf/x86 PMU infrastructure. The event is configured as precise user-mode mem-load latency PEBS, equivalent to the facility behind perf mem-loads with ldlat threshold, precise_ip>=2, exclude_kernel=1, exclude_hv=1, exclude_guest=1. The ownership policy is opportunistic: user-requested pinned or exclusive perf events win; if the scheduler event cannot remain continuously scheduled, or if perf reports multiplexing/throttling, the CPU-local sampler marks itself inactive and contributes no actuation until rearmed. No userspace RDPMC or debugfs control plane is required. 2. Data structures: extend task_struct with pebs_cookie (u64), pebs_exec_epoch (u32), pebs_last_rq_seq (u64), pebs_debt (u8), pebs_flags, and timestamp/epoch for decay. Extend rq with pebs_rq_seq (u64), published_run_instance, ring buffer, and counters for accepted, stale, dropped_full, dropped_inactive, and throttled samples. Use fixed-size aligned ring entries and a power-of-two ring size so indexing is a mask. 3. Publication and lifecycle: assign pebs_cookie at fork from atomic64_inc_return. Increment pebs_exec_epoch on successful execve before return to userspace. In the fair-class switch-in path under rq->lock, after rq clock update and before the task is exposed as current, increment pebs_rq_seq, copy it to p->pebs_last_rq_seq, and publish {rq_seq, pebs_cookie, pebs_exec_epoch, valid=1} with release semantics. On switch-out to idle or non-fair class, clear valid with release semantics. There is no runtime reset of pebs_rq_seq; if wrap were ever approached, the feature disables on that CPU, which is practically unreachable with 64 bits. On migration, the source CPU's queued tuples become stale because the destination switch-in assigns a new rq_seq; source tuples cannot match destination p->pebs_last_rq_seq. 4. NMI-safe ring contract: producer-owned head and consumer-owned tail are u32. Producer sequence: tail = smp_load_acquire(&ring->tail); if head - tail == RING_SIZE, increment dropped_full and return; WRITE_ONCE each field of entries[head & mask]; then smp_store_release(&ring->head, head + 1). Consumer sequence under rq->lock: head = smp_load_acquire(&ring->head); read entries[tail & mask] with READ_ONCE field loads until budget exhausted; after consumption, smp_store_release(&ring->tail, new_tail). Because only the producer writes head and only the consumer writes tail, and head is advanced only after the slot is fully written, torn reads are avoided without additional locking. 5. Stall classification: use only PEBS samples with valid precise_ip and valid latency/data-source payload. Ignore sub-threshold latency. Map accepted samples into 3 debt-producing classes: class 1 weight 1 for moderate latency; class 2 weight 2 for severe local-memory latency; class 3 weight 4 for severe remote-memory or very high latency. Because debt is a penalty to the sampled task rather than a reward, workloads cannot gain CPU by fabricating stalls; at worst they self-penalize. If PEBS payload lacks sufficient precision or data source on a given model, that CPU stays in inactive mode rather than approximating with low-confidence counters. 6. Drain points and overhead bounds: drain in put_prev_task_fair() with hard budget Bctx=8 entries per context-switch path; optionally drain in task_tick_fair() only if ring occupancy exceeds 75 percent, with Btick=2 and no immediate actuation except updating pending debt. Worst-case incremental work per switch is O(Bctx) on local cache-resident data: one acquire load of head, up to 8 fixed-size slot reads, up to 8 three-field key comparisons, and up to 8 saturating adds/decays; there is no unbounded scan. Worst-case incremental work per tick is O(Btick) only on watermark breach. A 256-entry ring with 32-byte entries costs about 8 KiB per CPU. 7. Scheduler actuation and fairness proof: keep baseline CFS/EEVDF entity selection, vruntime/vlag evolution, and bandwidth accounting unchanged. Apply only two bounded penalties against the current debt-bearing leaf task: earlier resched by Delta_slice_max=min(base_slice/8, sysctl_sched_min_granularity/2) and reduced wakeup non-preemption margin by Delta_wake_max=sysctl_sched_wakeup_granularity/4 after latency_nice is already incorporated. Define effective_debt as saturating at Dmax=8 and decaying by one quantum per switch boundary without a matching tuple. Because the invention never enlarges the eligible set, never changes virtual time, and can only subtract bounded real-time slack from the current task, any continuously runnable entity remains subject to the baseline starvation proof plus at most X=Delta_slice_max+Delta_wake_max additional real-time skew on any single next-dispatch opportunity. That preserves starvation safety while still reducing harmful residency of repeatedly stalled tasks. 8. cgroup and throttling semantics: store debt only on leaf tasks, never on group entities. Parent cfs_rq selection and weights remain untouched, so cross-cgroup CPU share distribution is unchanged. If a cfs_rq is bandwidth-throttled, debt is frozen for actuation and only decays; tuples associated with pre-throttle rq_seq values are dropped as stale. When the task is re-enqueued after throttling, it receives a new rq_seq before any future tuples can be accepted. 9. Security and virtualization: do not expose raw PEBS records, IPs, addresses, or data sources to unprivileged tasks. The scheduler consumes only coarse stall classes and drop counters. Disable by default in guests unless the vPMU explicitly advertises precise PEBS virtualization support and the admin opts in; otherwise use exclude_guest=1. On host vCPU threads, the feature may remain enabled only for host userspace execution; guest-mode samples are excluded. If PEBS delivery is trapped, masked, or model-blacklisted due to errata, the CPU sampler remains inactive.

1. Correctness of attribution: instrument fork, execve, migration, sleep/wakeup, exit, idle entry, and cfs bandwidth throttle/unthrottle paths to verify that tuples from an older rq_seq, older exec_epoch, or source CPU never survive stale-drop checks. Microarchitectural coverage: validate on at least two Intel server generations with PEBS mem-load latency support, and separately on unsupported or partially supported systems to show correct fail-open disablement. 4. Risks & Mitigations

  • Risk: conflict with user perf_event ownership or PMU multiplexing. Mitigation: implement as an opportunistic kernel-internal perf_event with lower ownership priority than pinned or exclusive user events; if continuously scheduled precise PEBS is unavailable, mark the CPU sampler inactive and decay debt to zero rather than using multiplexed samples for actuation.
  • Risk: NMI-context corruption or torn ring reads. Mitigation: use a strict SPSC ring with producer-only head, consumer-only tail, fixed-size aligned entries, release/acquire ordering, drop-on-full behavior, and no dynamic memory allocation, rq locking, or task lookup in PMI/NMI context.
  • Risk: misattribution across migration, execve, exit, dequeue, or idle transition. Mitigation: require a three-field execution-instance key match {rq_seq, task_cookie, exec_epoch} and clear valid publication on idle or non-fair switch-out; stale tuples are silently dropped.
  • Risk: starvation or fairness regression. Mitigation: never alter vruntime, vlag, eligibility, load weight, parent cfs_rq shares, or bandwidth quotas; cap earlier-resched and wakeup bias by Delta_slice_max and Delta_wake_max; saturate and decay debt; and restrict actuation to leaf tasks only.
  • Risk: cgroup hierarchy distortion. Mitigation: consult debt only after the group entity has already been selected by normal parent-level scheduling; do not store or propagate debt on non-leaf sched_entity objects; freeze actuation while cfs bandwidth throttling is active.
  • Risk: side-channel or abuse concerns from exposing PEBS-derived information. Mitigation: expose no raw PEBS addresses or IPs to unprivileged code, aggregate only coarse stall classes for scheduler use, gate feature enablement to root, optionally gate by root-controlled cgroup policy, and default to exclude_guest/exclude_hv operation.
  • Risk: unsupported or errata-affected Intel models. Mitigation: require an allowlist of PEBS formats that provide precise latency and data-source payloads with known-correct behavior; blacklist problematic models; fail open to baseline scheduling when support is ambiguous.
  • Risk: scheduler lock hold inflation from ring drain. Mitigation: drain only a hard-bounded number of tuples per context-switch path and a smaller optional number per tick on watermark breach, making overhead proportional to configured budgets rather than to sample backlog.

  • Make the producer side correct under nested/re-entrant PMIs: either (a) prove PMU NMI handler cannot re-enter for this event on this CPU, or (b) explicitly drop samples when in NMI (per-cpu in_nmi depth guard), or (c) implement MPSC-safe ring reservation (atomic fetch-add) and document ordering/barriers.

  • Handle migration and remote wakeups precisely: ensure any tuple produced on CPU X cannot be consumed after the task migrates to CPU Y in a way that still passes your match predicate. This usually requires bumping the rq sequence on every switch/dequeue and including cpu in the key, plus rejecting if current->cpu != tuple->cpu at consume time.
