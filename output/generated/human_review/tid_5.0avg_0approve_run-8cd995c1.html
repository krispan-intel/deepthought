2. Technical Invention Disclosure

2.1 Problem Statement

Modern x86 Linux servers routinely map tens to hundreds of MSI/MSI-X vectors per device across multiple sockets, LLC domains, housekeeping CPUs, and isolated CPUs. The kernel needs a deterministic hardirq-path view of per-vector facts and a safe runtime mechanism for moving affinity as load shifts, but current choices are poor. Reading mutable irq_desc or driver-private state before locking is either unsafe or too expensive at interrupt entry; pushing policy into user-space daemons races CPU hotplug, managed interrupt rules, irqchip restrictions, and explicit administrator overrides; and existing affinity APIs accept CPU masks directly, so the policy engine itself must know transient legality details it does not own. The technical problem is therefore to create a low-overhead x86 interrupt steering mechanism that: 1) gives hardirq entry a constant-time, immutable metadata snapshot; 2) lets policy consume stable telemetry without dereferencing unstable kernel objects; 3) prevents the policy layer from naming CPUs directly or indirectly; 4) arbitrates correctly against procfs, sysfs wrapper, and in-kernel affinity writers; 5) remains correct under CPU online/offline and vector migration; and 6) confirms that a requested move actually completed on x86 before allowing further moves, so high-rate interrupts do not thrash when effective affinity lags requested affinity.

2.2 Prior-Art Gap

Existing approaches miss the required combination of publication discipline, sealed policy output, and in-kernel confirmation. irqbalance and similar daemons observe counters after the fact and write cpumasks through procfs, which races hotplug and cannot see managed-IRQ legality or x86 vector-move completion atomically. irq_set_affinity_hint() is advisory and still exposes CPU identities to the policy side. Generic BPF telemetry or tracepoint approaches can observe interrupt behavior, but they do not provide a kernel-enforced lease ABI that forbids cpumask expression, and they do not tie lease ownership to all affinity write sites with epochs. Prior art around snapshot/revalidate is generic concurrency hygiene, not the invention; the inventive gap is the x86-specific publication of immutable vector-slot metadata plus epoch-fenced, kernel-decoded lease arbitration and post-commit confirmation against effective affinity and vector cleanup. No known approach combines: exact vector-slot attachment for both MSI/MSI-X and IO-APIC interrupts; a verifier-bounded classifier and scorer that cannot hold unstable references; kernel-owned non-singleton CPU classes; event-driven dirty-IRQ selection instead of O(N_irqs) scans; and x86 move-complete confirmation before new policy actions are admitted.

2.3 Proposed Invention

The invention introduces a two-stage, x86-specific affinity lease bridge. Stage 1 is publication and telemetry. When x86 assigns or retargets a vector, the vector allocator builds an immutable irq_vector_meta containing irq number, source type (MSI/MSI-X or IO-APIC), managed flag, trigger mode, current package/NUMA/LLC identifiers, kernel-owned class identifier, admin-override state, and a telemetry index. For MSI/MSI-X, publication occurs after the destination CPU/vector pair is chosen and before the new vector becomes dispatch-visible; on retarget, a fresh snapshot is published into the new CPU's vector slot with rcu_assign_pointer(), the old slot is marked retired, and the old metadata is freed only after x86 vector cleanup retires the old mapping. For IO-APIC lines, publication occurs when the routed GSI/pin is bound to an x86 vector; the metadata differs only in source-type fields, because the local APIC vector dispatch path is still the fast-path anchor. Hardirq entry executes a static-key-gated fixed-offset load from the vector slot, may run an optional BPF_PROG_TYPE_X86_IRQ_CLASSIFY program with no helpers, no maps, no loops, maximum 128 instructions, and return value restricted to keep-current or a small class override 0..63, then updates a per-CPU telemetry page under raw seqcount. Telemetry captures hardirq-safe facts only: interrupt count, last TSC, delta-TSC EWMA accumulator, max delta-TSC, current CPU id, current vector class id, last confirmed target domain id, and dirty-threshold counters. No rq locks, irq_desc pointers, or sleeping locks are touched. Stage 2 is scoring, decoding, and commit. A per-NUMA lease kthread drains dirty IRQ ids from lockless per-node llist queues. An IRQ becomes dirty only on threshold crossings, confirmation state changes, hotplug/class-version changes, or external affinity writes; this avoids O(N_irqs) scanning. The kthread snapshots telemetry with read_seqcount_retry(), derives slow-changing facts such as NUMA distance, LLC domain, housekeeping eligibility, cpuset-effective permission, pending-confirm age, and recent move failures, and passes a stable bpf_irq_lease_ctx to a non-sleepable BPF_PROG_TYPE_IRQ_LEASE scorer. The scorer is restricted to ctx reads plus bpf_ktime_get_ns(), no maps, no kptrs, no tail calls, no helper that can sleep, stack <= 256 bytes, bounded loops disabled, and instruction budget <= 4096. It returns a packed 64-bit lease word with exact layout: bits 0..2 action enum {0 HOLD, 1 RESELECT, 2 PREFER_LOCAL, 3 RELAX_LOCALITY, 4 SUSPEND_AUTO}; bits 3..5 locality domain enum {0 ANY, 1 NUMA, 2 LLC, 3 PACKAGE, 4 HOUSEKEEPING, 5 PERF_CLASS, 6 ADMIN_POOL, 7 CURRENT_ONLY}; bits 6..11 class selector 0..63 referencing a kernel-owned class table; bits 12..19 min-residency in 250 microsecond quanta; bits 20..27 cooldown in 250 microsecond quanta; bits 28..31 urgency 0..15; bits 32..39 window selector 0..255; bits 40..47 hysteresis 0..255; bits 48..63 reserved and required to be zero. The decoder never interprets any field as a CPU id or cpumask. Kernel-owned class ids are built only from fixed, non-user-defined equivalence families such as housekeeping class, ITMT performance bin, package-local performance bin, and non-isolated service pool; class construction rejects per-CPU classes and any class with fewer than two online CPUs. If hotplug shrinks a class below two CPUs, resolution returns empty and the lease decodes to HOLD rather than a singleton target. Before commit, the kernel resolves the lease into a candidate mask by intersecting the requested domain/class with cpu_online_mask, housekeeping and isolation eligibility, cpuset-effective permission, managed-IRQ restrictions, explicit admin pin masks, and the current irqchip-visible legal set. Revalidation is two-phase: the scorer sees a snapshot stamped with affinity_epoch and topology_version, then the commit path rechecks those values under desc lock just before calling the normal affinity path. Every external write to /proc/irq/*/smp_affinity, /proc/irq/*/smp_affinity_list, any sysfs wrapper, and any in-kernel caller of irq_set_affinity() or irq_force_affinity() funnels through an epoch bump helper that increments desc->lease.affinity_epoch, records owner = EXTERNAL or owner = KERNEL_CALLER, cancels any pending confirm, and optionally sets lease_mode = MANUAL until explicitly returned to AUTO. The lease engine therefore never overwrites a newer external decision. Commit success is not defined as 'irq_set_affinity returned 0'. Instead, the IRQ enters PENDING_CONFIRM with requested_seq, requested_epoch, and requested_mask_digest. x86 vector migration then advances a move-complete generation only after old-vector cleanup retires the previous vector mapping. A move is confirmed only when both conditions hold: 1) the x86 move-complete generation for the IRQ is at least requested_seq, and 2) the current effective affinity, read after commit, intersects the resolved lease mask and contains no CPU now outside the legalized domain/class. If confirmation does not arrive before confirm_deadline_ns, the engine records a lag fault, extends cooldown, and suppresses further moves for that IRQ. This makes effective-affinity lag a first-class state rather than a source of oscillation. Hotplug is handled by CPUHP hooks and topology versioning. A CPUHP_AP_IRQ_LEASE_PREPARE_OFFLINE hook runs before final vector teardown for the outgoing CPU, bumps a global topology_version, invalidates any pending lease whose resolved set included that CPU, and enqueues affected IRQs for rescore; existing x86 IRQ migration logic moves vectors off the CPU, while the lease engine treats the old score as stale. A CPUHP_AP_IRQ_LEASE_ONLINE hook rebuilds the kernel-owned class table, increments class_version, and wakes the local lease kthread. Because both score snapshots and commits carry topology_version and class_version, a CPU going offline between score and commit causes the pre-commit revalidation to abort before any new affinity request is issued. If a CPU goes offline after commit but before confirmation, the epoch or topology version mismatch causes the pending-confirm to retire as stale and the IRQ is re-enqueued after the core hotplug migration completes. The design is therefore hotplug-correct without sleeping or polling in hardirq context. Security and containment are integral. Only privileged code in the initial user namespace with CAP_BPF and CAP_SYS_ADMIN may load or attach the classifier or scorer. There is no unprivileged attachment, no per-container attachment, and no direct exposure of raw per-interrupt timestamp streams; the scorer sees only aggregated windowed telemetry and stable topology labels. Per-IRQ token buckets and per-node global token buckets cap commit frequency, one pending confirm per IRQ is permitted at a time, and scorer execution is event-driven, so a malicious program cannot turn the mechanism into a high-rate affinity-flipping DoS engine. The disabled-path target on Xeon is near-zero overhead because static_key and static_call patch out both classifier and telemetry hooks; the enabled telemetry-only target is one metadata pointer load plus cache-hot per-CPU stores, with a design target below roughly 35 cycles at hardirq entry on recent Xeon cores.

2.4 Architecture

Major components: 1) x86 vector publication path; 2) hardirq fast-path metadata load and telemetry update; 3) event-driven dirty-IRQ queueing; 4) per-NUMA lease kthread running bounded BPF scoring; 5) kernel legality decoder and epoch arbiter; 6) x86 move-complete confirmer; 7) CPUHP hotplug revalidator.

ASCII control flow:
vector assign / retarget
  -> build immutable irq_vector_meta
  -> rcu_assign_pointer(percpu_vector_slot->meta, new_meta)
  -> old_meta retired after x86 vector cleanup

interrupt entry on CPU X
  -> static_branch_unlikely(irq_lease_enabled)
  -> meta = rcu_dereference(slot->meta)
  -> optional classify(meta_only_ctx) return class_override
  -> update per_cpu telemetry page under raw seqcount
  -> if threshold crossed and dirty bit was clear: enqueue irq on per-node llist
  -> normal handler dispatch unchanged

per-NUMA lease kthread wake
  -> drain dirty IRQ ids and pending-confirm timers
  -> seqcount snapshot telemetry + read stable topology state
  -> build bpf_irq_lease_ctx {irq, epochs, topology_version, class_version, managed, source_type, counts, delta_tsc stats, last_move_age, pending_confirm, legal_domain_sizes}
  -> run BPF_PROG_TYPE_IRQ_LEASE scorer
  -> decode 64-bit lease word
  -> resolve domain/class to candidate CPUs
  -> pre-commit revalidate under desc lock against epochs, online mask, admin owner, managed restrictions, isolation, cpuset, irqchip-visible legality
  -> call normal affinity path
  -> set PENDING_CONFIRM(requested_seq, requested_epoch, deadline)

x86 vector move completion
  -> old vector cleanup retires previous mapping
  -> move-complete generation advanced
  -> confirmer checks effective affinity now matches legal resolved set
  -> mark CONFIRMED or LAGGED and requeue with extended cooldown

CPU hotplug
  -> CPUHP offline hook bumps topology_version, cancels stale pending confirms, requeues affected IRQs
  -> core migration moves interrupts off dying CPU
  -> CPUHP online hook rebuilds class table, bumps class_version, wakes local kthread

Exact legality sources used during resolution and revalidation: cpu_online_mask; genericirq managed-affinity state via irqd_affinity_is_managed() and related irq_data flags; effective affinity via irq_data_get_effective_affinity_mask(); existing affinity commit path via irq_set_affinity()/irq_force_affinity(); housekeeping and isolation policy from kernel housekeeping masks and nohz_full / isolcpus state; cpuset-effective permission from the task-independent cpuset effective CPU masks exported to kernel core code; explicit admin pin mask and lease_mode; and final irqchip narrowing learned from effective affinity after the request. The topology/class system is versioned: each rebuild increments class_version, each external affinity write increments affinity_epoch, and both are carried through score, commit, and confirm.

2.5 Implementation Plan

1) Add struct irq_vector_meta and per-IRQ lease state. The meta contains immutable fields only: irq, source_type, managed flag, trigger type, package_id, numa_id, llc_id, kernel_owned_class_id, telemetry_index, admin_owner_snapshot, and publication generation. The lease state contains affinity_epoch, topology_version_seen, class_version_seen, owner, lease_mode, requested_seq, confirmed_seq, pending_confirm flag, token buckets, last_confirmed_mask_digest, last_move_ns, and fault counters. 2) Wire x86 publication points. In the x86 vector allocation / retarget path, allocate a new irq_vector_meta, populate it after the destination CPU/vector is known, and publish it to the destination vector slot with rcu_assign_pointer(). For retarget, publish new before exposing the slot, mark old slot retired, and retire old metadata with call_rcu() only after the x86 vector cleanup path signals the previous mapping is dead. For MSI/MSI-X, do this at the point the vector destination is committed; for IO-APIC, do it when the line's route entry is bound to a local APIC vector. 3) Implement hardirq telemetry. Add a static-key-patched hook in x86 vector entry that dereferences slot->meta, optionally executes the no-helper classifier, and updates a per-CPU telemetry page indexed by telemetry_index. The page is single-writer from local hardirq, guarded by raw seqcount, and stores interrupt count, last_tsc, delta_tsc EWMA accumulator, max delta, dirty threshold state, and last confirmed locality. No mutable irq_desc reads, no memory allocation, no maps, and no locks other than local seqcount writer semantics are added. 4) Add dirty-IRQ selection. Each IRQ has a dirty bit. Threshold crossings in hardirq, confirmation failures, class-version changes, hotplug events, and external affinity writes attempt a lockless set-and-enqueue onto a per-NUMA llist. The lease kthread waits on queue non-empty or timeout. Periodic fallback sweep is limited to IRQs already in PENDING_CONFIRM, not all IRQs, so steady-state work remains event-driven. 5) Introduce BPF_PROG_TYPE_IRQ_LEASE and optional BPF_PROG_TYPE_X86_IRQ_CLASSIFY. Classifier contract: meta-only ctx, no helpers, no maps, no loops, max 128 insns, return keep-current or class override 0..63. Scorer contract: bpf_irq_lease_ctx only, bpf_ktime_get_ns() allowed, no maps, no kptrs, no tail calls, no sleepable helpers, max 4096 insns, stack <= 256 bytes, return 64-bit lease word. Verifier checks reserved bits are zero along all exits and that no program path synthesizes out-of-range action/domain/class values. 6) Define kernel-owned class table lifecycle. Classes are built only from predefined families: housekeeping service pool, package-local ITMT performance bins, package-local non-isolated workers, and optional admin-defined service pools that must each contain at least two online CPUs. The table is rebuilt on CPUHP online/offline and relevant housekeeping/cpuset topology changes. Each rebuild increments class_version. Resolution uses class id plus domain enum, never a raw CPU number. If a class becomes singleton or empty, decode returns invalid and the engine converts the request to HOLD. 7) Add arbitration and epoch plumbing. Funnel every affinity-changing path through a common helper that bumps affinity_epoch. Sources include procfs affinity writes, any sysfs wrapper, irqbalance acting through procfs, and in-kernel callers of irq_set_affinity() / irq_force_affinity(). The helper records the writer type. External writers set lease_mode = MANUAL by default and cancel any older pending confirm. AUTO mode can be restored by an explicit control write. Pre-commit revalidation under desc lock aborts if affinity_epoch, topology_version, class_version, or owner changed since score time. 8) Add legality resolution and commit. Decode the lease word, build a candidate set from domain/class, intersect with online, managed, effective, isolation, housekeeping, cpuset, and admin constraints, and skip commit if empty or if min-residency / cooldown prevents action. Use per-IRQ token bucket for local thrash control and per-node global token bucket to bound aggregate commit rate and lock contention. Only one outstanding PENDING_CONFIRM is allowed per IRQ. 9) Add x86 confirmation path. Extend the x86 vector migration backend so old-vector cleanup advances a move-complete sequence for the IRQ. A move becomes CONFIRMED only when move_complete_seq >= requested_seq and effective affinity now conforms to the legalized resolved set. If confirmation lags, mark the IRQ LAGGED, stretch cooldown, and suppress additional moves until either confirmed or aged out. 10) Add CPUHP hooks. Offline hook executes before final CPU teardown to bump topology_version, cancel stale pending confirms, and requeue affected IRQs. Online hook rebuilds class tables, bumps class_version, and wakes the relevant lease kthread. Revalidation guarantees that a CPU disappearing between score and commit cannot be selected by a stale lease. 11) Expose observability and admin control. Provide read-only stats for epochs, confirmations, lag faults, and class versions; provide explicit AUTO vs MANUAL control so operators can pin an IRQ and know the lease engine will not override it until re-enabled.

2.6 Validation Plan

1) Correctness validation: unit-test lease decoding so all reserved-bit, out-of-range, and singleton-class cases resolve to reject or HOLD; add KUnit tests for epoch races, hotplug invalidation, and owner precedence; test that external writes during PENDING_CONFIRM cancel the pending lease without rollback writes. 2) x86 move-confirm validation: instrument vector retarget and cleanup to verify requested_seq, move_complete_seq, and confirmed_seq ordering; run stress with high interrupt rates and delayed vector cleanup to prove the engine does not issue a second move while effective affinity still lags. 3) Hotplug validation: repeatedly online/offline CPUs during sustained MSI-X traffic, verify topology_version mismatch aborts stale commits, confirm affected IRQs are rescored only after hotplug migration stabilizes, and verify no stale class id resolves to a singleton CPU. 4) Managed-IRQ and irqchip validation: test managed NVMe/NIC vectors, IO-APIC lines, and hierarchical irqchip configurations; verify candidate masks are reduced by effective affinity when hardware narrows them and that the scorer never learns or emits forbidden CPUs directly. 5) Performance validation: compare baseline, compiled-but-disabled, telemetry-only, and full lease modes on Xeon. Target compiled-but-disabled overhead near zero via static patching, telemetry-only added hardirq cost below roughly 35 cycles median, and no measurable ACK/EOI reordering. Measure that dirty-queue wakeups scale with active IRQs rather than total IRQ count. 6) Security validation: confirm only init-namespace privileged loaders can attach; verifier rejects maps, helper misuse, pointer capture, and out-of-range lease fields; token-bucket caps prevent affinity-flip DoS; telemetry exported to BPF is aggregated window data rather than per-event raw timestamps. 7) Workload validation: run multi-socket NIC RSS, NVMe queue, and accelerator interrupt workloads with isolcpus / nohz_full / housekeeping mixes; measure p99 interrupt service latency, LLC-local hit rate, socket-to-socket traffic, and number of manual affinity exceptions versus irqbalance and static pinning baselines.

4. Risks & Mitigations

  • Risk: x86 vector-move completion may vary across irqchip and vector-domain implementations. Mitigation: define confirmation in terms of a small new x86 move-complete generation hook plus effective-affinity convergence, and fall back to HOLD when a backend cannot provide completion semantics rather than pretending the move completed.
  • Risk: reviewers may argue the design is just snapshot-and-revalidate plus BPF policy. Mitigation: emphasize that novelty is the sealed lease ABI, non-singleton class system, epoch fencing tied to all affinity write sites, and x86 post-move confirmation, none of which exists in daemon-based cpumask steering.
  • Risk: BPF could become a covert channel or DoS vector. Mitigation: init-namespace privileged attachment only, no maps or pointer-bearing helpers, aggregated telemetry only, bounded instructions, one outstanding pending confirm per IRQ, and token-bucket commit caps.
  • Risk: topology classes might accidentally collapse to single CPUs under hotplug, reintroducing indirect CPU naming. Mitigation: class-table rebuild rejects singleton classes, resolution of an undersized class returns invalid, and decode degrades to HOLD instead of choosing the remaining CPU.
  • Risk: event-driven queues could still overload under bursty traffic. Mitigation: use per-IRQ dirty bits to coalesce enqueue operations, per-node llists to localize wakeups, and periodic sweeps only for PENDING_CONFIRM IRQs rather than global scans.
  • Risk: operator expectations can be violated if external writes are silently overridden. Mitigation: treat all external affinity writes as owner-changing epoch events, default them to MANUAL mode with explicit AUTO re-enable, and expose observability counters for cancelled pending leases and owner transitions.
