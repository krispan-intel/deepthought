2. Technical Invention Disclosure

2.1 Problem Statement

In KVM's x86 TDP MMU, 2MB and 1GB EPT leaves improve EPT walk depth, TLB reach, and guest execution efficiency, but the same ranges can become pathological when host-side MM churn repeatedly invalidates or reclassifies backing pages. Reclaim, migration, KSM merges or breaks, NUMA balancing, unmap or remap, mprotect transitions, dirty logging transitions, and memslot changes can cause a range to be promoted to a huge EPT leaf and then immediately split or zapped again. The result is repeated EPT construction work, more remote INVEPT or TLB invalidation, longer time in MMU critical sections, and unstable tail latency on multicore and Hyper-Threaded systems. Existing correctness machinery removes stale SPTEs, but it does not provide a bounded, per-range, guest-physical contract that says: this range is currently unstable for huge-leaf promotion, so continue mapping with 4KB leaves for a limited interval even if contiguity and alignment momentarily reappear.

2.2 Prior-Art Gap

Existing KVM mechanisms such as notifier-driven invalidation, memslot generations, SPTE race handling, NX hugepage controls, dirty-log-triggered splitting, and remote flush logic are correctness mechanisms, not promotion-stability publication mechanisms. They answer whether a mapping is still legal, not whether a range is a poor candidate for immediate 2MB or 1GB reuse. Likewise, host-side THP or collapse heuristics are HVA- and host-page-centric rather than guest-range- and EPT-level-centric, and they do not carry churn provenance from Linux MM into KVM's huge-leaf decision points. The gap is especially clear under multicore invalidation storms: the kernel can correctly split or zap, yet nothing prevents every CPU from racing to re-promote the same unstable range. No close prior art surfaced for the combined design of generic ignore-safe mmu_notifier provenance hints, bounded per-range guest-frame lease state, a per-VM revocation publication sequence, and a fairness-bounded hold-down policy that remains purely advisory and never substitutes for correctness fences.

2.3 Proposed Invention

The invention introduces an additive advisory layer between Linux MM churn events and KVM x86 EPT hugepage promotion. First, struct mmu_notifier_range.flags is extended with high-order advisory provenance bits, for example MMU_NOTIFY_HINT_CHURN_RECLAIM = BIT(16), MMU_NOTIFY_HINT_CHURN_MIGRATION = BIT(17), MMU_NOTIFY_HINT_CHURN_KSM = BIT(18), MMU_NOTIFY_HINT_CHURN_NUMA = BIT(19), MMU_NOTIFY_HINT_CHURN_UNMAP = BIT(20), and MMU_NOTIFY_HINT_CHURN_MPROTECT = BIT(21). These bits are optional, default to zero, do not alter callback ordering, do not change blockability semantics, and are ignored by subscribers that do not understand them. Second, each VM owns a fixed-memory lease table keyed by {slot_id, memslot_generation, aligned_gfn_base, level}, where level is 2MB or 1GB. An exemplary implementation uses 64 shards and 4096 total entries, with each shard implemented as a fixed 4-way set-associative array under a shard-local raw spinlock and seqcount. Each entry stores provenance_mask, active_depth for nested start/end tracking, last_revocation_seq, cooldown_deadline, token_count, last_refill_time, and stable_epoch count. Third, KVM maintains atomic64_t ept_revocation_seq plus coarse per-level fallback fences. On invalidate_range_start, KVM resolves the affected memslot/GFN intervals using the existing memslot snapshot path under SRCU, updates the exact lease entries when the span is small enough, or updates the coarse per-VM fence when the range is too wide or collisions prevent safe insertion. After metadata is updated, KVM publishes ept_revocation_seq with release semantics. Promotion paths load the sequence with acquire semantics before choosing 2MB or 1GB target level. This sequence is not a correctness generation; it is a visibility barrier for advisory instability metadata. Fourth, the state machine is explicit and bounded: start events increment active_depth and zero promotion tokens, end events decrement active_depth and extend cooldown_deadline, overlapping or nested events OR provenance bits and take the maximum deadline, and successful stable epochs refill tokens up to a fixed cap. Example defaults are a maximum post-end cooldown of 2 ms for 2MB and 8 ms for 1GB, token cap of 2 for 2MB and 1 for 1GB, refill of one token after 10 ms or four clean aging epochs, and a mandatory probe promotion after a longer stable interval so hugepage promotion cannot be starved indefinitely. If an invalidation spans too many intervals, only a bounded number of exact entries are touched and the mechanism falls back to a coarse per-VM level fence. Thus notifier cost remains O(1) with fixed memory. NUMA-awareness is optional and memory-safe: NUMA provenance may be sampled only from PFNs that are already referenced or pinned by the existing collapse path; invalid, special, device, DAX, or otherwise unstable PFNs are skipped, and the embodiment can omit NUMA sampling entirely.

Concurrency and safety contract: notifier writers never take kvm->mmu_lock or slots_lock; they take only shard-local raw spinlocks and publish via seqcount plus release-store of ept_revocation_seq. Promotion readers never take shard locks; they use seqcount retry and READ_ONCE on fixed entries, so they cannot observe torn advisory state. Because entries are fixed and never freed until VM teardown, there is no use-after-free on the advisory read path. Memslot updates are handled by including memslot_generation in the lease key and ignoring entries whose generation no longer matches the current slot snapshot.

2.5 Implementation Plan

1. Generic MM API extension: add advisory high-bit definitions to mmu_notifier_range.flags and document them as ignore-safe provenance only. Default is zero. Existing subscribers continue to receive the same event ordering and may ignore the new bits. Expected producers are reclaim, migration, KSM, unmap or remap, NUMA balancing, and protection-change paths that already invoke notifier invalidation. 2. Per-VM data structures: add struct kvm_ept_lease_table in struct kvm_arch with 64 shards, fixed entry arrays, one seqcount and one raw spinlock per shard, atomic64_t ept_revocation_seq, and two coarse fences for 2MB and 1GB. Each lease entry contains slot_id, memslot_generation, level, aligned_gfn_base, provenance_mask, active_depth, last_revocation_seq, cooldown_deadline_ns, token_count, last_refill_ns, and stable_epochs. Collision semantics are precise: same-key updates merge by OR-ing provenance, saturating active_depth, and taking the max deadline; different-key collisions evict the oldest inactive entry only; if all ways are active, update the coarse fence instead of spinning or allocating. 3. Notifier path algorithm: in invalidate_range_start, use the existing KVM memslot snapshot path under SRCU to translate the HVA span to slot/GFN spans. For each level, touch at most a fixed exact-track budget, for example 8 aligned intervals total; if the span exceeds that budget, write the coarse level fence only. Under the shard lock, fill or merge the entry, set active_depth++, set token_count=0, extend deadline according to provenance class, record last_revocation_seq = next_seq, then publish the new sequence with smp_store_release. In invalidate_range_end, resolve the same keys when possible, decrement active_depth, and set cooldown_deadline to max(existing, now + post_end_holdoff). Nested or overlapping start/end pairs are handled by active_depth and deadline=max(deadline, new_deadline). If an end event cannot find the exact entry because of prior coarse fallback or safe eviction, only the coarse fence is updated; correctness ordering is unchanged because the advisory plane never controls zap or flush. 4. TDP MMU integration points: consult the lease before any large-leaf installation in the x86 TDP MMU map path, including the target-level decision in the page-fault map flow, the direct 1GB install branch, and any background or deferred collapse path that upgrades 4KB or 2MB mappings into a larger EPT leaf. The read sequence is: seq = smp_load_acquire(&ept_revocation_seq); read matching exact entry and coarse fence using seqcount retry; if memslot_generation mismatches, ignore the entry; if active_depth > 0 or cooldown_deadline is in the future, cap target level at 4KB; if the deadline passed and token_count > 0, allow one huge-leaf probe and decrement a token; otherwise stay at 4KB. Fast page fault remains unchanged because it upgrades permissions on an existing SPTE rather than creating a new huge SPTE. Aging paths remain unchanged except that clean aging epochs may increment stable_epochs and refill a token once the bounded stability threshold is met. This avoids permanently biasing the system to 4KB mappings. 5. Fairness and progress: the hold-down policy is explicitly bounded. Maximum cooldown is capped per level. Token refill is deterministic. A mandatory probe promotion is forced after a longer stable interval, for example 50 ms without matching revocation or 64 clean 4KB faults, so continuous low-grade churn cannot starve hugepage promotion forever. Background collapse retries are also capped per interval; a denied promotion records next-eligible time so the collapse path does not thrash on every pass. Because denied promotions fail early on the advisory check, mmu_lock contention does not grow without bound. 6. Correctness boundary: the advisory plane never suppresses or delays any required split, zap, generation bump, permission downgrade, dirty-log-induced split, NX-hugepage restriction, memslot update, SPTE race resolution, remote TLB flush, or INVEPT. Those paths run exactly as today. If the lease table is absent, stale, full, or inconsistent, the only effect is that KVM falls back to 4KB mappings; stale mappings still cannot persist beyond existing correctness fences. 7. Teardown and lifetime: allocate the lease table when the VM is created or when TDP MMU is enabled. On VM teardown, unregister the notifier first, then wait for SRCU/RCU grace, then free the advisory table. Because readers only inspect fixed entries and writers never free entries in-place, there is no use-after-free window. 8. Optional NUMA embodiment: if the implementation uses NUMA locality as part of its stability score, it may inspect only PFNs that the collapse path already holds by stable references. Invalid or special PFNs are skipped, and the feature can be compiled out or disabled without affecting the rest of the invention.

2.6 Validation Plan

Validate in four layers. First, API and semantics validation: prove that default-zero flags yield byte-for-byte equivalent behavior for notifiers that do not opt in, and add overlap and nesting tests where invalidate_range_start/end arrive in nested and partially overlapping order with mixed provenance bits. Second, concurrency validation: stress with many vCPUs, concurrent memslot updates, reclaim, migration, and dirty logging while asserting that notifier code never takes kvm->mmu_lock or slots_lock, that seqcount readers never observe torn keys, and that teardown after notifier unregister plus SRCU/RCU grace prevents stale access. Third, correctness validation: run KVM selftests and targeted tests for dirty-log toggles, memslot move/delete, mprotect downgrades, NX hugepage restrictions, fast page faults, page aging, remote flushes, and nested invalidation storms to confirm that stale mappings do not survive beyond existing correctness fences. Fourth, performance and fairness validation: compare baseline KVM against the invention on workloads with reclaim, KSM, live migration, NUMA balancing, and userfaultfd movement; measure huge-leaf install rate, split/re-promote oscillation count, remote INVEPT/TLB activity, mmu_lock hold time, vCPU exit rate, and guest p99 latency. Include a hostile low-grade churn test to confirm that capped cooldown plus token refill and mandatory probe allow hugepage promotion to recover instead of being starved forever.

4. Risks & Mitigations

  • Overly coarse suppression under very wide invalidations: mitigate by exact tracking for small spans, separate 2MB and 1GB keys, and coarse-fence fallback only when range width or collisions exceed a fixed budget.
  • Potential starvation of hugepage promotion: mitigate by fixed maximum cooldown, deterministic token refill, and mandatory probe promotion after a bounded stable interval.
  • Locking or teardown bugs between notifier context and KVM MMU paths: mitigate by never taking kvm->mmu_lock or slots_lock in notifier callbacks, using fixed entries with shard-local raw spinlocks plus seqcount publication, and freeing only after notifier unregister and SRCU/RCU grace.
  • Incorrect dependence on advisory state for correctness: mitigate by formally constraining the mechanism to deny or defer huge-leaf install only; all split, zap, permission, memslot, dirty-log, SPTE-race, and remote flush paths remain mandatory and unchanged.
  • NUMA-sampling safety and device-PFN corner cases: mitigate by limiting sampling to pages already referenced by the existing path, skipping invalid or special PFNs, and making NUMA scoring optional rather than foundational.
  • Prior-art overlap with generic split-thrash heuristics: mitigate by claiming the specific combination of notifier provenance bits, bounded guest-range lease state, revocation publication sequence, and fairness-bounded advisory gating tied to x86 EPT large-leaf decisions.
