2. Technical Invention Disclosure

2.1 Problem Statement

On large x86 Linux systems, the initiating CPU often spends a significant fraction of TLB shootdown time deriving and re-deriving the destination set rather than waiting for remote cores to execute the actual invalidation. The baseline path repeatedly walks a wide cpumask, touches many zero words, and may repartition the same target CPUs multiple times during a burst of flushes against the same mm. This overhead grows with CONFIG_NR_CPUS and is particularly visible on Xeon systems using x2APIC routing, where the send backend can benefit from cluster-local grouping but current software typically recomputes that grouping from scratch for each flush. AVX-512-capable cores can classify 512-bit chunks of a temporary mask image much faster than repeated scalar next-bit scanning, but the Linux kernel cannot simply spray SIMD into arbitrary synchronization paths because many flush sites run in atomic or IRQ-constrained contexts. The problem is therefore to exploit x86 microarchitectural width without weakening the exact TLB correctness contract under CPU hotplug, task migration, and nested invalidation activity.

2.2 Prior-Art Gap

Existing prior art is fragmented. Linux cpumask and bitmap helpers provide correct scalar iteration but do not define a TLB-shootdown-specific capture point, descriptor schema, or bounded reuse rule. Existing x86 APIC backends provide correct send and completion behavior but do not expose a kernel contract for precompiled routing buckets tied to the currently active backend. Intel AVX/AVX-512 optimization manuals teach wide zero detection and chunk classification in user-space or general software terms, but they do not show how to integrate those instructions into a Linux synchronization path that is constrained by mm invalidation serialization, cpus_read_lock(), kernel FPU safety rules, and hotplug-stable APIC routing. Prior art also does not specify a reentrancy-safe non-sleeping storage model for a compiled shootdown descriptor, a memory-ordering boundary that prevents accidental future publication bugs, or a precise statement of why CPUs that start running an mm after capture are not required members of the captured set. Without those kernel-specific constraints, a generic 'snapshot bitmap then iterate' approach is both obvious and incomplete.

2.3 Proposed Invention

Introduce an x86 TLB Shootdown Sparse Destination Descriptor (SDD) layer with two compile backends and one strict correctness boundary. 1. ENTRY CONTRACT. The helper may be entered only after the caller has already established the invalidation using the same existing serialization that makes today's flush_tlb_mm_range()-style shootdown correct, such as the current page-table/invalidation critical section used by munmap/mprotect(), tlb_gather_mmu teardown, or mmu_notifier invalidation. The helper then acquires cpus_read_lock() before reading cpu_online_mask or any APIC-routing state. No wake suppression or deferred delivery is in scope. 2. EXACT CAPTURE OF S. Under that boundary, the helper computes a single immutable target set S = mm_cpumask(mm) & cpu_online_mask & ~cpumask_of(this_cpu). There is no invented arch_tlb_ipi_eligible_mask; self-exclusion is the only extra predicate, because the local CPU already performs its local flush via the unchanged baseline path. Over-inclusion is permitted: a CPU that migrated away or still lingers in mm_cpumask(mm) may receive an unnecessary IPI, which is already acceptable today. Under-inclusion is prevented by the existing x86 mm TLB generation rules: a CPU that begins running the mm only after S is captured must observe the post-invalidation generation in switch_mm and self-flush before user execution, so it does not have to be in S. cpus_read_lock() prevents online/offline transitions from changing cpu_online_mask or routing membership during capture and dispatch. 3. CONCRETE SDD SCHEMA. For CONFIG_NR_CPUS = N, let W = BITS_TO_LONGS(N) and C = DIV_ROUND_UP(N, 16). Each SDD slot contains: hdr { u16 words_total=W; u16 nr_spans; u16 nr_buckets; u16 nr_targets; u8 backend_mode; u8 flags; u16 local_cpu; u16 depth; u16 reserved; u32 nr_cpu_ids_snapshot; }, spans[W] where each span is { u16 first_word; u16 word_count; u16 bucket_index; u16 value_offset; }, values[W] as packed non-zero unsigned long words, and buckets[C] where each bucket is { u32 routing_key; u16 first_span; u16 span_count; }. Maximum spans are W, maximum packed values are W, maximum buckets are C. Memory footprint per slot is 16*W + 8*C + 16 bytes on x86_64, which is about 3088 bytes when N=4096. 4. REENTRANCY-SAFE NON-SLEEPING STORAGE. Storage is a per-CPU ring of exactly two preallocated SDD slots. The outermost flush on a CPU reserves slot 0; one nested flush on the same CPU may reserve slot 1; any further nesting, any failed reservation, or any schema-capacity anomaly causes an immediate one-shot fallback to the legacy cpumask walk. There is no retry loop and no sleeping allocation. Because the descriptor is consumed only on the same CPU and within the same call chain, it is passed by stack pointer to lower helpers and is never published to other CPUs. Ordinary program order is therefore sufficient; if a future implementation publishes the descriptor outside the stack/ring scope, it must use release-store on publish and acquire-load on consume, but that publication model is outside this disclosure. 5. TWO COMPILE BACKENDS. The scalar backend is always available and walks the copied words once to emit only non-zero runs. The optional AVX-512 backend is entered only when the CPU reports AVX-512 support and irq_fpu_usable() is true, with !in_nmi(), !irqs_disabled(), and the compile occurring before any IPI send. It executes kernel_fpu_begin(), loads eight 64-bit mask words per ZMM register, derives a non-zero lane mask using AVX-512 compare/test plus k-mask instructions, skips all-zero 512-bit chunks in one step, and emits spans for consecutive non-zero lanes where lane i maps to word index base_word + i. It then executes kernel_fpu_end() before dispatch. If any AVX-512 precondition fails, compilation falls back to the scalar backend with no semantic change. 6. OPTIONAL PUBLIC-KEY BUCKETIZATION. Bucketization is enabled only when the active APIC send backend is x2APIC cluster delivery. The routing key is the public cluster portion of the logical APIC destination value already used by the backend, i.e. logical_dest_id >> 4 for 16-CPU clusters. APIC ID and logical-destination mapping are fixed for a CPU while it is online, and cpus_read_lock() prevents online/offline state changes during the descriptor lifetime, so the key is topology-stable for the duration of use. For xAPIC physical, x2APIC physical, UV, or any backend that lacks an exported stable public routing key, bucketization is disabled and the SDD is iterated only by spans. Bucketization is a pure partition of S: it never adds or removes CPUs and never changes the final completion mask. 7. DISPATCH, COMPLETION, AND REUSE. Dispatch iterates the SDD spans, optionally grouped by bucket, reconstructs only the needed temporary mask slices, and invokes the same existing x86 remote-call/IPI send helper and the same completion wait/ack mechanism used by the baseline implementation. The wait mask is exactly S. Ordering among buckets is not semantically relevant because completion already waits for all targeted CPUs. The SDD may be reused only for another flush against the same mm while both the original invalidation serialization and the same cpus_read_lock() critical section are still held on the same initiator CPU; dropping either boundary invalidates the descriptor immediately and forces recompilation or legacy fallback. 8. DENSITY AND PROGRESS RULES. If nr_spans is close to W, if nr_targets exceeds a tunable dense threshold such as half of online CPUs, or if the measured compile cost exceeds a small fixed budget, the helper bypasses SDD dispatch and uses the legacy walk directly because the sparse representation offers no advantage. This guarantees the new path cannot increase latency without bound under dense masks or contention: every failure mode adds only O(1) slot checks plus, at worst, one linear pass over the local copied words before falling back to the already-correct baseline.

2.4 Architecture

Authoritative correctness boundary:
- Caller has already advanced the mm invalidation/TLB generation under the same existing serialization that makes today's shootdown correct.
- cpus_read_lock() is acquired before reading cpu_online_mask or routing state and is held until the final send decision is fixed.
- Local CPU flush remains on the unchanged baseline path; S only covers remote CPUs.
- Descriptor lifetime ends before either the invalidation serialization or cpus_read_lock() is dropped.

Concrete schema and bounds:
- hdr: words_total, nr_spans, nr_buckets, nr_targets, backend_mode, flags, local_cpu, depth, nr_cpu_ids_snapshot.
- span entry: first_word, word_count, bucket_index, value_offset.
- bucket entry: routing_key, first_span, span_count.
- values[]: packed non-zero unsigned long words copied from S.
- Max spans = BITS_TO_LONGS(CONFIG_NR_CPUS).
- Max buckets = DIV_ROUND_UP(CONFIG_NR_CPUS, 16) for x2APIC cluster mode; 0 otherwise.

Supported backend modes:
- xAPIC physical: supported, no bucketization.
- x2APIC physical: supported, no bucketization.
- x2APIC cluster: supported, bucketization by public logical destination cluster key.
- UV or other vendor-specific logical modes: dispatch supported through legacy semantics, but SDD bucketization disabled unless the backend exports a stable public routing key.

Memory-ordering rule:
- Current design is same-CPU stack/ring consumption only, so there is no cross-CPU publication and no extra barrier requirement beyond ordinary program order.
- Any future attempt to queue or RCU-publish the descriptor is a different embodiment and must add release/acquire publication semantics.

2.5 Implementation Plan

1. Add a new helper pair in the x86 TLB fanout path, e.g. build_tlb_sdd() and dispatch_tlb_sdd(), with lockdep assertions that the caller still holds the required invalidation serialization and that cpus_read_lock() is held from capture through dispatch decision. 2. Define struct tlb_sdd_hdr, struct tlb_sdd_span, struct tlb_sdd_bucket, and struct tlb_sdd_slot exactly as bounded above. Allocate DEFINE_PER_CPU(struct tlb_sdd_slot[2], tlb_sdd_ring). Add a tiny per-CPU depth counter or slot-owner bits so recursion beyond two levels immediately fails closed to legacy. 3. Implement capture_sdd_mask(mm, slot): copy mm_cpumask(mm) intersected with cpu_online_mask minus the local CPU into a local words array once, count nr_targets and non-zero words, and decide early whether the set is too dense to benefit. Do not read mm_cpumask(mm) or cpu_online_mask again after this capture. 4. Implement build_tlb_sdd_scalar(slot): one pass over the copied words that emits contiguous non-zero word runs into spans[] and packs only the non-zero values into values[]. Record nr_spans, nr_targets, and density statistics. 5. Implement build_tlb_sdd_avx512(slot) under a static branch enabled only on AVX-512-capable kernels. Entry predicates are CPU feature present, irq_fpu_usable(), !in_nmi(), !irqs_disabled(), and descriptor building occurring before any remote send. Inside kernel_fpu_begin()/kernel_fpu_end(), process the copied mask eight 64-bit words at a time, derive a non-zero lane mask, and emit spans by translating lane bits to word indices. On any predicate failure, exception, or debug disable, tail-call the scalar builder. 6. Implement bucketize_tlb_sdd(slot): read the active APIC backend mode once while cpus_read_lock() is held. If and only if the backend is x2APIC cluster delivery, derive routing_key = logical_dest_id >> 4 and group spans into buckets[] without changing membership. Otherwise set nr_buckets = 0. 7. Implement dispatch_tlb_sdd(slot, data): iterate buckets when present, otherwise spans directly; reconstruct the temporary mask slice for each send and invoke the existing send_IPI_mask()/smp_call_function_many_cond() path unchanged. The completion mask remains the exact captured S. No change is made to ack accounting or remote handler behavior. 8. Wire burst-local reuse to the same initiator call chain only. Reuse is allowed only if the same CPU still owns the slot, the same invalidation serialization scope is still held, cpus_read_lock() has not been dropped, and the mm identity has not changed. Any drop of those conditions destroys the descriptor immediately. 9. Add tracepoints and counters for: slot busy fallback, dense-mask fallback, AVX-512 eligible/used, bucketized sends, and average nr_spans. These are observability aids only and not control-plane inputs. 10. Keep the legacy cpumask walk bit-for-bit available and selectable at each failure point so worst-case progress remains identical to current kernels.

2.6 Validation Plan

1. Correctness under migration: stress mprotect()/munmap()/MADV_DONTNEED against threads that continuously migrate across CPUs while validating that no stale translations survive. Instrument mm TLB generation checks to confirm that post-capture entrants self-flush rather than relying on inclusion in S. 2. CPU hotplug safety: run a hotplug torture loop that onlines/offlines CPUs while issuing flush bursts from parallel processes. Verify with lockdep and tracing that cpus_read_lock() spans capture through dispatch decision, that bucket routing keys never change within the critical section, and that fallback handles backend instability. 3. Nested flush behavior: inject recursion through notifier, reclaim, and fault paths so one CPU can experience outer and inner flushes. Confirm slot 0 then slot 1 usage, and confirm depth-3 recursion falls directly to legacy with no corruption and no livelock. 4. Context gating for AVX-512: add assertions that the AVX-512 compile backend is never entered in NMI, never entered when irq_fpu_usable() is false, and always returns through kernel_fpu_end(). Run hardirq-heavy and preempt-disabled scenarios to ensure scalar or legacy fallback is taken. 5. Backend coverage: test xAPIC physical, x2APIC physical, x2APIC cluster, and one non-cluster vendor backend. Verify that bucketization is enabled only in x2APIC cluster mode and that the final completion mask is identical to the baseline. 6. Performance characterization: compare baseline versus scalar SDD versus AVX-512 SDD on representative Xeon systems such as Ice Lake and Sapphire Rapids. Measure initiator cycles, shootdown tail latency, nr_spans, and reuse benefit for single flushes and 4-8 flush bursts. The target outcome is roughly 15-25% lower initiator cycles for sparse single flushes and 35-60% lower initiator-side scan cost across short bursts, with less than 2% regression on dense all-CPU masks due to density fallback. 7. Abuse testing: deliberately force slot exhaustion, disable AVX-512 at runtime, and simulate dense masks to prove that the fallback path always completes and that no condition can loop indefinitely or defer flush completion.

4. Risks & Mitigations

  • Risk: AVX-512 in the kernel can violate context rules or inflate save/restore cost. Mitigation: AVX-512 is limited to the compile sub-step, guarded by irq_fpu_usable() and !in_nmi(), wrapped in kernel_fpu_begin()/kernel_fpu_end(), and never required for correctness because scalar and legacy fallbacks remain available.
  • Risk: Nested flushes on one CPU could overwrite per-CPU scratch. Mitigation: use a fixed two-slot per-CPU ring with explicit ownership/depth tracking and immediate legacy fallback at depth greater than two or on reservation failure.
  • Risk: A captured set could miss CPUs due to migration or hotplug. Mitigation: rely only on the existing authoritative invalidation serialization plus cpus_read_lock(); CPUs joining after capture self-flush via existing TLB generation rules, and hotplug transitions are blocked from changing online membership during descriptor use.
  • Risk: Bucketization could depend on unstable or private topology state. Mitigation: enable it only for x2APIC cluster mode using the public logical-destination cluster key; disable it for physical and vendor-specific backends unless they export an equally stable public key.
  • Risk: Sparse compilation may be slower on dense masks. Mitigation: count non-zero words and targets during capture, apply an early dense threshold, and jump directly to the legacy walk when the sparse representation offers no benefit.
  • Risk: Future developers might publish the descriptor across CPUs without barriers. Mitigation: document that the current embodiment is same-CPU stack/ring consumption only and explicitly require release/acquire publication semantics for any future queued or RCU-published embodiment.
