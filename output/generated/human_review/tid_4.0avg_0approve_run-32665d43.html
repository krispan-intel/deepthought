2. Technical Invention Disclosure

2.1 Problem Statement

On x86 virtualization hosts, the expensive part of an EPT generation transition is often not only the invalidation, write-protect, split, or A/D harvest event itself, but the transient fan-out created when fair-class vCPU threads keep migrating across additional CPUs and LLC domains before the active generation converges. Existing Linux scheduler signals understand runnable load, wake locality, legal masks, capacity fit, and NUMA history, but they do not know that a particular vCPU thread is temporarily lagging relative to a current EPT-maintenance window. KVM can infer that pressure, but today that knowledge lives in VM-centric state that the scheduler cannot safely consult under `rq->lock`. The missing piece is a lock-safe, hotplug-safe, fairness-safe, per-task publication contract that exposes only a bounded transient cache-scope preference without creating scheduler-to-KVM callbacks or correctness coupling.

2.2 Prior-Art Gap

Current mechanisms are adjacent but not equivalent. KVM remote-TLB or flush-target tracking is reactive and records CPUs that have already run a vCPU; it does not proactively bias placement to avoid enlarging that set. Scheduler locality heuristics such as wake-affine, LLC-aware balancing, NUMA balancing, and cache-hot scoring operate on generic locality history and not on EPT-generation maintenance state. Affinity masks, cpusets, isolcpus, and housekeeping are static legality controls and cannot express a microsecond-scale convergence window. None of these mechanisms defines a per-task, scheduler-readable, lockless snapshot derived from EPT generation transitions and consumed only as a decaying tie-break at named fair-scheduler decision points while preserving the invariant that the legal CPU mask is never expanded.

2.3 Proposed Invention

Introduce an internal kernel-only bridge in which each KVM-managed x86 vCPU task may own an optional `sched_ept_scope_hint` object. KVM publishes a fully materialized snapshot into that object only at vCPU-safe points, never from scheduler context. The snapshot contains only POD scalar state: an opaque `ept_generation_cookie`, event flags, topology generation, up to two stable LLC-domain identifiers with representative anchor CPUs, a preferred NUMA node fallback, `published_ns`, `expires_ns`, and a small bounded `max_bonus`. The scheduler reads only that task-local object via `rcu_dereference()` plus `seqcount_latch_t`, performs constant-time checks first, and only then intersects the hinted scope with the already legal candidate mask. The resulting effect is a strictly monotonic, short-lived tie-break bonus used only for fair-class wakeup placement and small-imbalance migration choice. The bias never creates a legal destination that was not already legal, never blocks mandatory migration, and becomes inert automatically on expiry, sleep, topology mismatch, hotplug remap failure, task exit, or teardown.

2.4 Architecture

Data path: `EPT/MMU event -> per-vCPU snapshot synthesis in KVM -> seqcount-latch publish into task-local hint -> scheduler tie-break helper -> legal-mask intersection -> bounded bonus`.

Hint storage and concurrency contract:
- `task_struct` holds only `struct sched_ept_scope_hint __rcu *ept_hint`.
- `sched_ept_scope_hint` is separately allocated from a dedicated SLAB cache and marked `__cacheline_aligned_in_smp` so KVM updates do not share a hot cacheline with unrelated scheduler fields.
- The object contains `seqcount_latch_t seq` and two banks of `struct sched_ept_scope_snapshot`; no snapshot bank contains pointers to `kvm`, `vcpu`, MMU objects, or topology objects.
- Each snapshot bank contains: `version`, `flags`, `ept_generation_cookie`, `event_bitmap`, `topology_gen`, `primary_domain_id`, `secondary_domain_id`, `primary_anchor_cpu`, `secondary_anchor_cpu`, `preferred_nid`, `max_bonus`, `ttl_ns`, `published_ns`, and `expires_ns`.
- RCU protects outer-object lifetime only; `seqcount_latch_t` protects payload coherence. Writers fill the inactive bank with `WRITE_ONCE()` scalar stores and then flip the latch; readers under `rq->lock` use `read_seqcount_latch_begin()` and retry at most twice. The scheduler therefore observes either the old complete bank or the new complete bank, never torn timestamps or mixed domain IDs.

Producer rules:
- Only KVM code running in the owning vCPU task context may publish or refresh the hint.
- Safe publish points are after local observation of an EPT generation rollover, after an INVEPT-causing epoch is committed, on dirty-log or write-protect entry/exit, and after A/D harvest or equivalent housekeeping that indicates current locality pressure.
- Primary anchor is the LLC domain of the current CPU at publish time.
- Secondary anchor is the most recent distinct LLC domain seen for the same vCPU within the same short event window; otherwise it is invalid.
- Preferred NUMA node is the node of the primary anchor and is used only as a fallback when LLC remap is unavailable.

Scheduler consumer and exact decision sites:
- The scheduler never consults VM-global `ept_hint_seq`, never dereferences `vcpu->kvm`, and never walks EPT or MMU state under `rq->lock`; it reads only `p->ept_hint`.
- The helper is called only from fair-class wakeup placement in `select_task_rq_fair()`, the near-tie wake-affine choice, and fair small-imbalance migration tie-break logic.
- The helper first performs constant-time gates: static key enabled, `p->sched_class == &fair_sched_class`, non-null `p->ept_hint`, `ACTIVE` set and `SECURITY_INERT` clear, and `ktime_get_mono_fast_ns() < expires_ns`.
- Only after those gates pass does it resolve LLC masks.
- Legality order is fixed: start from the scheduler's ordinary legal candidates; intersect with `p->cpus_ptr`; intersect with cpuset/cgroup and isolation restrictions; intersect with core-scheduling cookie and SMT legality; preserve existing capacity, asym-capacity, thermal-pressure, EAS, and uclamp filters; only then intersect with CPUs in the primary or secondary LLC domains, or with the preferred NUMA node fallback.
- Invariant: the post-hint candidate set is always a subset of the pre-hint legal candidate set and never a superset.

Hotplug and topology handling:
- Each anchor stores both a stable LLC `domain_id` and a representative `anchor_cpu`.
- If `snapshot.topology_gen` matches the current scheduler topology generation, the stored `domain_id` is used directly.
- If it does not match, the scheduler remaps via the representative CPU only if `cpu_online(anchor_cpu)`; if remap fails, it falls back to `preferred_nid`; if that also fails, the hint is inert.
- Even after remap, the scope remains bounded to at most two LLC domains.

Decay and scoring:
- Default `ttl_ns` is 128000, with an implementation range bounded to 64000-250000 ns.
- `remaining = expires_ns - now_ns`; `bonus = floor(max_bonus * remaining / ttl_ns)` clamped to `0..max_bonus`.
- `max_bonus` defaults to 2, primary LLC gets `bonus`, secondary LLC gets `max(bonus - 1, 0)`, and NUMA fallback gets 1 only when LLC mapping is unavailable.
- Because the bonus is small, monotonic, and non-sticky across sleep, it can only resolve near ties and cannot dominate load, capacity, or utilization metrics or create pathological ping-pong.

Teardown and security:
- On task exit or vCPU teardown, KVM publishes an inert snapshot or clears `p->ept_hint`, then frees the object via `call_rcu()`.
- VM destruction never leaves dangling scheduler references because the scheduler never points into VM memory.
- No user ABI is added; optional observability exposes only sanitized counters and domain IDs.

2.5 Implementation Plan

Phase 1: define `struct sched_ept_scope_hint` and `struct sched_ept_scope_snapshot`, add `struct sched_ept_scope_hint __rcu *ept_hint` to `task_struct`, and create a dedicated cacheline-aligned SLAB cache. Phase 2: implement KVM-side snapshot synthesis at existing EPT/MMU safe points. Compute the opaque generation cookie, event bitmap, primary anchor, optional secondary anchor, preferred NUMA node, topology generation, and absolute expiry before publication. Phase 3: implement `sched_read_ept_scope_snapshot()` as an RCU + `seqcount_latch_t` read helper that returns an inert result unless the task is fair-class and the snapshot is coherent, active, and unexpired. Phase 4: integrate a single tie-break helper into `select_task_rq_fair()`, the wake-affine near-tie path, and fair small-imbalance migration choice code. Apply the helper only after normal legality, capacity, and core-scheduling filters, and assert that the hinted mask is a subset of the preexisting legal mask. Phase 5: add topology-generation tracking, representative-CPU remap logic, `cpu_online()` validation, and preferred-NUMA fallback for hotplug and partitioning scenarios. Phase 6: guard the entire feature behind a static key so the disabled case is effectively zero-cost, and add benchmark-only tracepoints or counters for publish events, live-hint reads, remap success, bonus application, CPUs-touched, and LLCs-touched. Phase 7: review and prove invariants: no non-fair-class use, no scheduler dereference of VM/vCPU state, no legal-mask expansion, no bypass of cpuset/isolation/core-scheduling restrictions, and no inhibition of mandatory or asym-capacity-driven migration.

2.6 Validation Plan

Benchmark matrix: dirty-logging and live-migration stress with many fair-class vCPU threads; A/D-harvest-heavy memory pressure workloads; multi-socket and multi-LLC servers; hosts with CPU hotplug and topology rebuild events; hosts with core scheduling enabled; and NOHZ/full-dynticks systems. Primary metrics: (1) unique CPUs touched during one active EPT-generation window, defined as the count of physical CPUs that execute a vCPU thread carrying the same live `ept_generation_cookie` before that cookie expires or all lagging vCPUs converge; (2) distinct LLC domains touched during the same window; (3) remote invalidation or shootdown fan-out, measured as INVEPT-related remote work or equivalent flush-target counts; (4) guest-visible stall time during dirty-log, write-protect, and EPT invalidation epochs; (5) wakeup-to-run latency and p95/p99 latency of lagging vCPU threads; (6) scheduler overhead in cycles and branch misses on `select_task_rq_fair()`; and (7) migration count and cross-LLC migration rate during active hint windows. Instrumentation: KVM tracepoints at snapshot publish sites carrying only opaque generation cookie and event flags; scheduler tracepoints indicating whether a live hint was read, whether topology remap succeeded, and whether a tie-break bonus was applied; benchmark-build aggregation of per-generation CPU and LLC sets; and host/guest latency probes to correlate fan-out with guest stalls. Acceptance criteria: statistically significant reduction in median and p99 CPUs-touched and LLCs-touched per active EPT-generation window; reduction in remote invalidation fan-out or equivalent work; no measurable regression in scheduler legality or forced-migration behavior; near-zero overhead when the static key is off; bounded minimal overhead when on; and no policy violations under affinity, cpuset, isolcpus, housekeeping, uclamp, asym-capacity, hotplug, or core-scheduling constraints. Security evaluation: verify the feature remains kernel-internal with no new ioctl, procfs, sysfs, or guest ABI; test whether the transient bias changes cross-VM co-location measurably; and confirm that security-sensitive configurations can mark hints inert without affecting scheduler correctness.

4. Risks & Mitigations

  • Risk: torn reads or mixed-field freshness bugs under `rq->lock`. Mitigation: use RCU only for object lifetime and `seqcount_latch_t` for payload coherence; store only POD scalars in the snapshot; retry reads a bounded number of times.
  • Risk: scheduler fast-path cost regression. Mitigation: use a static key, fair-class gate, non-null pointer gate, active-flag gate, and `ktime_get_mono_fast_ns()` expiry gate before any LLC or NUMA mapping work.
  • Risk: anchor CPUs go offline or topology changes invalidate stored scopes. Mitigation: store both stable domain IDs and representative CPUs, compare topology generation, require `cpu_online()` on remap, and fall back to preferred NUMA node or inert behavior.
  • Risk: bias interferes with load balance, asym-capacity, thermal pressure, EAS, or uclamp. Mitigation: apply only after ordinary legality and capacity filters, cap `max_bonus` at a small value, and bypass the mechanism for mandatory migration and decisive capacity mismatches.
  • Risk: false sharing or cacheline bounce from KVM updates. Mitigation: keep only one pointer in `task_struct` and place the mutable hint object in a dedicated cacheline-aligned allocation.
  • Risk: ping-pong across LLC domains. Mitigation: bound the scope to at most two anchors, use a short TTL, use monotonic decay, and do not maintain scheduler-written sticky history.
  • Risk: cross-VM contention or timing-channel amplification. Mitigation: keep the feature kernel-internal, short-lived, fair-class-only, subordinate to cpusets/isolation/core scheduling, and allow security policy to force `SECURITY_INERT` behavior.
  • Risk: teardown or ownership bugs on task exit or VM destruction. Mitigation: publish only in owning vCPU task context, clear `p->ept_hint` before freeing related vCPU objects, and retire the hint object with `call_rcu()`.
  • Risk: prior-art overlap with known scheduler locality or KVM TLB-tracking mechanisms. Mitigation: claim the specific combination of per-task published EPT-derived snapshot state, RCU plus latch publication contract, bounded two-anchor scope, explicit expiry, and named tie-break-only scheduler decision changes.
