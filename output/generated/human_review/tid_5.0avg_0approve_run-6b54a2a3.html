2. Technical Invention Disclosure

2.1 Problem Statement

Guest mapping of host files through virtio-fs DAX or other FS-DAX-backed VM memory typically falls back to 4 KiB EPT because safe 2 MiB EPT installation requires more than GPA alignment. For a specific PMD-sized file-offset window, the host must know that: (1) the window is entirely inside i_size; (2) it maps one 2 MiB-aligned, physically contiguous PFN run; (3) the EPT memory type is uniform across the run; (4) reflink, COW, dax-layout conversion, long-term pinning, dirty-log constraints, badblock remap, or sharing semantics will not make the run unstable mid-window; and (5) any concurrent truncate, punch-hole, collapse-range, insert-range, remap, or media-failure path cannot recycle those PFNs until all stale huge EPT leaves are gone. Existing KVM huge-leaf admission sees HVA and PFN state too late and too indirectly; existing filesystem mutation code knows extent truth but cannot express it to KVM in a lockless, PMD-granular, machine-checkable form or wait on a KVM completion object before old PFNs are reused.

2.2 Prior-Art Gap

Current KVM invalidation sequencing, including mmu_invalidate_seq and mmu_notifier retry rules, is keyed to mm and HVA invalidation, not to address_space plus PMD file windows. It tells KVM when to retry after address invalidation, but it does not tell KVM whether a given file-offset PMD window is extent-contiguous, partial-EOF, reflink-unsafe, memtype-mixed, or temporarily unstable because a filesystem transaction is shifting offsets. Likewise, DAX PMD fault paths can opportunistically map large pages when alignment is obvious, but they do not provide a reusable, per-window contract consumable by KVM across guest mappings. Most importantly, prior art does not provide a filesystem-visible retire cookie whose meaning is: every KVM instance mapping this file range has zapped any covering huge EPT leaf and completed the remote invalidation required before those old PFNs may be reassigned. That is the topological void between filesystem truth and x86 EPT datapath optimization.

2.3 Proposed Invention

Introduce a filesystem-owned auxiliary object per DAX-capable address_space, referred to here as fsdax_ept_state, containing two PMD-granule xarrays keyed by pmd_index = pgoff >> (PMD_SHIFT - PAGE_SHIFT). The first xarray stores immutable, RCU-published contract records for windows that are currently eligible candidates for 2 MiB EPT. The second xarray stores deny-state records for windows that are temporarily unstable or permanently no-huge. KVM consumes these objects only at huge-leaf admission sites for guest-backed file mappings. A contract record contains: contract_seq, stable_epoch, base_pgoff, pfn0, npages=512, memtype_class, share_class, perm_mask, and blocker_bits. Valid contract publication means the filesystem has proven, for that exact 2 MiB file window, that the file range is fully populated by a single aligned extent run, lies wholly within i_size, resolves to a contiguous DAX PFN run, has one EPT-safe cacheability class, and is not currently subject to reflink/COW or other blockers captured in blocker_bits. A deny record contains: deny_state in {UNSTABLE_TEMP, NOHUGE_PERM}, reason code, deny_seq, and optional expiry or age metadata. The deny ledger is authoritative only in the negative direction. For any mutation that can change PFN ownership or PMD eligibility, the filesystem performs: publish deny state with release ordering; increment topology_epoch; enumerate overlapping VMAs from mapping->i_mmap; request KVM retire for those HVA ranges and obtain a retire cookie; quarantine the old PFN run; complete extent mutation; wait on the retire cookie in sleepable context after dropping blocking FS locks; then either republish a new VALID contract or leave the window absent or NOHUGE_PERM. KVM may install a 2 MiB EPT leaf only if the deny ledger does not veto, a VALID contract is present, and all existing KVM-side checks still succeed.

2.4 Architecture

Core objects:
1. fsdax_ept_state (anchored in the DAX inode or address_space private extension)
2. contract_xa: xarray keyed by PMD file window
3. deny_xa: xarray keyed by PMD file window
4. retire_cookie: refcounted completion object spanning all overlapping KVM mappings discovered through mapping->i_mmap

Proposed record layout:
- struct dax_ept_contract {
    u64 contract_seq;
    u64 stable_epoch;
    pgoff_t base_pgoff;
    u64 pfn0;
    u16 npages;
    u8 state;          // VALID or NOHUGE_PERM
    u8 memtype_class;  // WB, WC, UC_MINUS, UNKNOWN, implementation-defined EPT-safe class
    u8 share_class;    // EXCLUSIVE_STABLE, SHARED_STABLE, REQUIRES_4K
    u8 perm_mask;      // guest-visible R/W/X envelope
    u32 blocker_bits;  // REFLINK, COW, MIXED_EXTENT, PARTIAL_EOF, LONGTERM_PIN, DIRTYLOG, HWPOISON, LAYOUT_BREAK
    struct rcu_head rcu;
  }
- struct dax_ept_deny {
    u64 deny_seq;
    u8 deny_state;     // UNSTABLE_TEMP or NOHUGE_PERM
    u8 reason;
    u16 age;
    struct rcu_head rcu;
  }

KVM admission rule:
- deny_xa says UNSTABLE_TEMP or NOHUGE_PERM => force 4 KiB path
- else contract_xa must contain VALID
- then KVM still performs existing checks: memslot alignment, dirty logging constraints, current PFN run still matches, EPT memtype still legal, and retry rules under current invalidate sequence
- only then may a 2 MiB EPT leaf be installed

Lifetime rules:
- records are immutable after publish; writers replace entire record, never mutate in place
- publication uses xa_store plus RCU pointer replacement; freeing uses kfree_rcu
- inode eviction or unmount marks fsdax_ept_state dead, swaps out xarray roots, denies future huge promotion, and defers freeing until both RCU grace and outstanding retire_cookie references drain

2.5 Implementation Plan

Phase 1: add fsdax_ept_state and PMD-window keying in the iomap DAX path, with publication helpers that can compute VALID versus ABSENT/NOHUGE_PERM from extent metadata, i_size, dax capability, and memtype class. Phase 2: wire deny publication into every PFN-changing path for DAX-capable filesystems: truncate, hole punch, collapse or insert range, reflink/remap exchange, dax layout transitions, and badblock remap. Build the per-inode PFN quarantine list so old runs cannot be recycled early. Phase 3: implement retire orchestration. Walk mapping->i_mmap to resolve overlapping VMAs, aggregate subranges by mm or KVM instance, create retire_cookie objects, and signal completion only after huge-leaf zap plus remote invalidation finish. Explicitly forbid waiting while holding blocking filesystem transaction locks. Phase 4: patch the KVM huge-EPT admission sites for file-backed mappings so they consume deny_xa first, then contract_xa, then existing KVM checks. The KVM code path must contain a mechanical 'deny only' assertion for deny_xa and must retain all existing retry and correctness checks. Phase 5: add bounded reclaim for cold entries, RCU-safe teardown, and statistics: promotions attempted, promotions denied by contract absence, promotions denied by UNSTABLE, retire wait latency, and quarantine backlog. Phase 6: optimize only after correctness is proven, for example by caching explicit NOHUGE_PERM entries for hot partial-EOF windows and batching retire requests for adjacent PMD windows.

2.6 Validation Plan

Correctness validation: - Prove the invariant that stale, missing, or evicted metadata cannot create a wrong huge mapping because the default action is denial and the existing KVM zap plus retry logic still applies. - Add race tests where KVM repeatedly faults a PMD window while another thread performs truncate, hole punch, reflink, collapse-range, insert-range, and badblock remap on the same file. - Verify that old PFNs are never returned to the allocator until retire_cookie completion. Performance validation: - Measure 2 MiB EPT hit rate, EPT walk reductions, and guest-visible p95 or p99 latency for DAX-backed file access inside VMs. - Compare baseline late PFN-only promotion against contract-guided admission under churn-heavy workloads. - Measure memory overhead of contract_xa and deny_xa under sparse files and random access with reclaim enabled. Concrete race timeline to demonstrate benefit: 1. A PMD window is VALID and KVM promotes it to a 2 MiB EPT leaf. 2. A host thread punches a 4 KiB hole in the middle of that window. 3. Revised design publishes UNSTABLE before extent change and requests retire; any concurrent KVM refault sees deny_xa=UNSTABLE and maps 4 KiB only. 4. Existing KVM zap and flush retire the stale huge leaf; retire_cookie completes. 5. The filesystem commits the new split extent layout. No new VALID contract is published because the window is now mixed. 6. KVM remains on 4 KiB entries until a later mutation recreates a full eligible run. This prevents promote-zap oscillation that current KVM logic alone does not suppress, because current logic can retry after each invalidation but still lacks a filesystem-authored temporary veto tied to in-flight extent churn. Regression validation: - Confirm no behavior change for non-DAX files, anonymous memory, or filesystems that do not publish fsdax_ept_state. - Confirm that when KVM ignores the new ABI, correctness remains identical to baseline and only the optimization is lost.

The method of claim 1, wherein contracts and deny records are stored in xarrays keyed by PMD file window, published as immutable RCU-protected records, reclaimed with grace-period deferral, and detached on inode eviction or unmount so lockless KVM readers either observe a fully formed record or no record.

A system comprising a DAX-capable filesystem, an address_space-scoped PMD contract store, a deny-only churn ledger, and a KVM retire-fence provider, wherein the system suppresses promote-zap oscillation during truncate, punch-hole, collapse-range, insert-range, reflink, remap, layout-break, or media-remap events while preserving baseline correctness when contract metadata is absent or stale.

4. Risks & Mitigations

  • Risk: deadlock between filesystem mutation locks and KVM MMU locks. Mitigation: prohibit retire_cookie waiting while holding i_rwsem exclusive or extent transaction locks; quarantine PFNs first, then wait in sleepable post-commit context.
  • Risk: memory growth from per-window metadata on sparse files. Mitigation: lazy allocation, absence-as-deny semantics, capped per-inode entry counts, and eviction of cold NOHUGE_PERM and VALID entries under pressure.
  • Risk: a KVM site accidentally treats ledger state as positive proof. Mitigation: centralize admission in one helper with a hard-coded sequence of deny check -> contract check -> existing KVM checks, and add assertions or code review gates that prevent bypass.
  • Risk: incomplete mutation coverage leaves stale windows marked stable. Mitigation: route every PFN-changing DAX mutation through shared prechange and postchange helpers and include collapse-range, insert-range, reflink, remap, layout conversion, and badblock paths in the mandatory hook set.
  • Risk: memtype classification drift across devices or architectures. Mitigation: scope the invention to x86 EPT-capable systems and require UNKNOWN or mixed memtype to deny huge promotion.
  • Risk: retire latency reduces mutation throughput. Mitigation: quarantine PFNs and wait after dropping blocking locks, batch adjacent PMD windows into one retire_cookie, and keep baseline 4 KiB mapping available during churn.
