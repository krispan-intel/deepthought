2. Technical Invention Disclosure

2.1 Problem Statement

Intel TDX guests, and similar confidential-computing guests on x86, can require an architectural page-accept step before a private page is safely CPU-accessible. If that accept step executes inline in do_anonymous_page(), do_wp_page(), file-backed private folio instantiation, swapin, migration target allocation, or compaction replacement, it injects unbounded latency into mmap_lock-, page-table-lock-, and reclaim-sensitive paths and creates severe p99/p999 stalls under bursty private-memory demand. Linux also lacks a mechanism that safely pre-produces accepted private capacity while preserving strict domain semantics: accepted private pages must never become shared-eligible by accident, private-page cost must be attributed to the eventual owning lruvec only when ownership is stable, and all control signals must remain safe across memcg offlining, node hotplug, swapoff, zswap/zram paths, THP split/merge, migration, compaction, and heavy file-cache pressure. The problem is therefore not only reducing acceptance latency, but doing so with exact attribution, bounded hot-path overhead, and reclaim behavior that cannot regress correctness or forward progress.

2.2 Prior-Art Gap

Known approaches such as first-touch acceptance, one-time boot sweeps, per-CPU confidential-page caches, or ad hoc pre-accept pools do not provide the required end-to-end contract. They do not define a page state that is simultaneously isolated from buddy and PCP yet reclaim-safe during acceptance; they do not specify the first stable ownership point for anon faults, COW, file-cache insertion, swapin, migration, and compaction; they do not solve double-charge and miss-charge hazards for THP assembly, THP split, folio migration, or memcg reparenting; they do not expose an arch-neutral interface whose timed region excludes zeroing, memcg charge, page-fault overhead, TLB shootdowns, and unrelated encryption work; they do not bound timing overhead with explicit sampling and clock-source rules; and they do not define a reclaim formula with numerical caps, hysteresis, and mandatory fallback conditions that preserve sc->may_swap, sc->may_writepage, memcg low/min, PSI, workingset refault distance, MGLRU aging, and forward progress. Existing work also does not define an RCU/percpu lifetime model for per-lruvec advisory state during memcg teardown and node hot-remove.

2.3 Proposed Invention

Introduce an Accepted-Private Reservoir, or APR, per NUMA node. A sleepable background acceptor running only on housekeeping CPUs isolates order-0 private pages from the node's unaccepted-private free supply, marks them ISOLATED_INFLIGHT, performs the architecture-specific private-page accept primitive, and enqueues them into ACCEPTED_FREE_PRIVATE batches. Each batch contains 64 order-0 pages, a generation number, a sampled-or-estimated total accept time, a per-slot published bitmap, and no persistent folio metadata. When a page is popped from APR, the allocator receives an ephemeral accept token {batch_id, generation, slot_id, pages=1, ns_share_q10}. That token stays in allocation-local state until the first stable ownership point is reached. The first stable ownership point is defined as the earliest point at which the eventual memcg and node lruvec are fixed and the folio is committed to become visible, but before first PTE installation or folio_add_lru() visibility. At that point the kernel calls apr_publish_credit(token, lruvec), which uses test_and_set_bit(slot_id, batch->published) to guarantee exactly-once publication. If allocation aborts before that point, the folio is returned to APR or to private free supply with the slot still unpublished. Synchronous fallback uses the same arch-neutral accept API, wraps the result in a synthetic one-slot token, and publishes immediately after successful memcg charge. Reclaim then reads only a short-window, decaying ACCEPT-debt signal attached to the destination lruvec and uses it as a strictly capped additive anon-scan bonus; it never rewrites refault distance, aging generations, reclaim_idx, memcg protection, or writeback/swap rules. APR drains only to private-domain free supply, and any private-to-shared reuse still requires an explicit domain conversion path.

4. Risks & Mitigations

  • Risk: batch metadata could be published twice during retries or unusual error paths. Mitigation: per-slot test_and_set publication bitmap, generation checks, and debug assertions on every publish attempt.
  • Risk: timing reads could add overhead or create a high-resolution side channel. Mitigation: ktime_get_mono_fast_ns() only on sampled batches, rate-limited token bucket, coarse quantization, and policy knobs to disable timing while retaining APR correctness.
  • Risk: reclaim bias could over-favor anon and hurt file-cache recovery. Mitigation: purely additive bias, 12.5 percent cap, no reduction of baseline file scan, hysteresis, and forced zero after ineffective reclaim rounds.
  • Risk: memcg teardown or node hotplug could UAF per-lruvec state. Mitigation: RCU-protected pointers, DEAD state, redirect-and-drain offline sequence, grace-period-delayed free, and zero-debt interpretation for missing state.
  • Risk: swapin ownership may be unrecoverable for zswap, zram, swapoff, or stale slot reuse. Mitigation: defer once to the consuming fault path, then drop only the advisory signal and never corrupt memcg ownership.
  • Risk: THP or large-folio paths could reintroduce hidden persistent metadata. Mitigation: restrict APR storage to order-0 pages and use only stack-local compound-credit aggregation for large-folio assembly.
  • Risk: background acceptance could interfere with latency-sensitive workloads. Mitigation: pin acceptors to housekeeping CPUs, low-priority scheduling, duty-cycle caps, and node-local fill thresholds.
  • Risk: partial backend failures could leak pages across domain boundaries. Mitigation: partial accept returns accepted_pages explicitly, only successfully accepted pages enter ACCEPTED_FREE_PRIVATE, and every failure path returns pages to private-only supply.
