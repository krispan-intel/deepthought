2. Technical Invention Disclosure

2.1 Problem Statement

eBPF programs now sit on hot synchronization paths in Linux: map value ownership transfer, reference publication, queue and ring index advancement, state handoff between process and interrupt context, and coordination between BPF programs and surrounding kernel subsystems. On x86, these paths are sensitive not only to atomic instruction choice but also to microarchitectural details such as LOCKed operation behavior, split-lock penalties, CMPXCHG16B availability, TSO ordering assumptions, and mitigation/runtime instrumentation state. Existing choices are too coarse. Always routing through helpers is safe but adds call overhead and obscures architecture-specific ordering opportunities. Letting the x86 JIT opportunistically inline raw atomics from local instruction context is also insufficient because it does not carry forward verifier-proven facts about address provenance, object lifetime, alignment, context legality, or tooling requirements, and it provides no formal mechanism to prevent later JIT passes or runtime patching from splitting, probing, or otherwise perturbing an ordering-critical sequence. The result is either unnecessary overhead or a risk of backend-specific semantic drift under PREEMPT_RT, KCSAN/KASAN/KCOV, ftrace/trampoline attachment, split-lock detection, and evolving mitigation policy.

2.2 Prior-Art Gap

Prior art covers individual pieces but not their claimed combination or boundaries. Existing verifier logic already proves safety for many pointer classes and rejects invalid memory access, but it does not externalize those proofs into an immutable per-program synchronization contract table with a normative Linux primitive-family identifier and explicit fallback policy. Existing x86 BPF JIT logic already chooses instruction encodings based on opcode width and CPU capabilities, but it does not resolve synchronization sites through a one-time admissibility table tied to stable, non-revocable x86 features, nor does it carry a no-motion/no-splitting region constraint through every JIT and late-patching phase. Existing kernel atomics and LKMM primitives define semantics, but they are not used as named contract targets inside a verifier-to-JIT ABI. Existing alternatives/static_call/ftrace/text_poke mechanisms patch kernel text, but they do not provide a BPF-specific contract that says a given synchronization site must remain contiguous, non-probed, and semantically identical whether realized inline or through a thunk/helper. Existing helper side-effect tracking also does not specify call-equivalence across inline, out-of-line, and instrumented paths for PREEMPT_RT, NMI, IRQ, RCU, and sanitizer configurations.

2.3 Proposed Invention

1. Synchronization Contract Vector schema. Each verified program receives an immutable sidecar table in bpf_prog_aux, for example: struct bpf_sync_contract_entry { u32 insn_idx; u16 region_len_insns; u8 linux_primitive_id; u8 width_code; u8 memory_order; u8 addr_class; u8 align_log2; u8 lifetime_class; u8 ctx_mask_allowed; u8 ctx_mask_required; u8 rcu_requirement; u8 proof_mask; u8 admissibility_class; u8 fallback_kind; s16 fixed_field_off; u32 btf_type_id; u32 map_type_id; }; The key field is linux_primitive_id, a normative identifier naming the intended Linux primitive family rather than a generic optimization hint. Example values are SCV_PRIM_ATOMIC_FETCH_ADD, SCV_PRIM_ATOMIC_CMPXCHG, SCV_PRIM_ATOMIC_XCHG, SCV_PRIM_SMP_LOAD_ACQUIRE, SCV_PRIM_SMP_STORE_RELEASE, SCV_PRIM_SMP_MB, and optional SCV_PRIM_ARCH_CMPXCHG128 only when the kernel already exposes a named 128-bit arch primitive. memory_order is one of RELAXED, ACQUIRE, RELEASE, ACQ_REL, FULL. addr_class is one of MAP_VALUE_GLOBAL, MAP_VALUE_PERCPU, TRUSTED_BTF_OBJ, KPTR_FIELD, or OTHER_REJECTED. lifetime_class distinguishes refcounted object, map-owned stable storage, RCU-protected object field, and unsupported transient storage. proof_mask encodes verifier-proven facts such as PROOF_NONFAULTING, PROOF_NATURAL_ALIGN, PROOF_CACHELINE_CONTAINED, PROOF_CONST_OFFSET, PROOF_TRUSTED_BASE, PROOF_NO_SLEEP, and PROOF_RCU_HELD. 2. How verifier computes eligibility. The verifier identifies synchronization-relevant sites from existing BPF_ATOMIC instructions, from verifier-recognized helper/kfunc call sites that already denote Linux atomic or barrier semantics, and from any future pseudo-insn lowered before final verification. For each site it derives fields from concrete verifier state, not free-form inference: - Address class comes from register base type: PTR_TO_MAP_VALUE or PTR_TO_MAP_VALUE_ADJ with a fixed non-null base may become MAP_VALUE_GLOBAL; PTR_TO_BTF_ID or PTR_TO_BTF_ID_OR_NULL may become TRUSTED_BTF_OBJ only if the kfunc prototype and BTF metadata mark the pointee as trusted kernel memory and the path proves non-null; PERCPU storage is marked MAP_VALUE_PERCPU and is ineligible for global-atomic primitive families in the initial version. - Alignment comes from reg->off, var_off/tnum low-bit proof, object layout alignment, and fixed field offset. Inline eligibility requires natural alignment for the primitive width. For LOCKed widths that may trigger split-lock or cacheline-straddle penalties, the verifier additionally proves cacheline containment when object layout and constant offset allow it; otherwise the site is marked inline-ineligible and forced to fallback. - Lifetime and RCU requirements come from pointer provenance and program attach type. A field reachable only under rcu_read_lock() receives RCU_REQUIRED. A map value with stable lifetime under map refcounting receives MAP_OWNED_STABLE. A transient pointer or unresolved dynptr slice is rejected or routed to helper if a legal helper exists. - Context masks are derived from program type and attach point: TASK, SOFTIRQ, HARDIRQ, NMI, PREEMPT_RT, and SLEEPABLE_FORBIDDEN. A site that may execute in NMI cannot resolve to an instrumented thunk that is not NMI-safe. - Width is limited to 1, 2, 4, and 8 bytes for generic atomic families; 16 bytes is permitted only for a named arch primitive family when CX16 is admissible and 16-byte alignment is proven. If a synchronization operation is semantically legal in BPF but inline proofs fail, the verifier does not reject solely for optimization reasons; it emits an SCV entry with admissibility_class forcing fallback. The verifier rejects only when the underlying operation itself has no legal kernel-semantic realization for that pointer/type/width/context combination. 3. One-time x86 admissibility resolution rule. After verification, and only after x86 alternatives and system-wide capability intersection are final, x86_bpf_resolve_sync_contracts() evaluates each entry once using: ADMISSIBLE(entry) = primitive_supported(entry.linux_primitive_id, entry.width_code) && proofs_satisfied(entry.proof_mask) && context_supported(entry.ctx_mask_allowed, prog_attach_context) && tooling_policy_supported(entry) && stable_feature_whitelist_satisfied(entry) && mitigation_safe_template_exists(entry); The stable_feature_whitelist is intentionally narrow and testable. It includes only architectural TSO assumptions plus globally stable x86 capability bits whose availability is not expected to be revoked by post-load mitigations or per-task toggles, such as CX8 and, when system-wide present, CX16. It explicitly excludes RTM/HLE and any feature whose usability is known to change under microcode or mitigation policy. The resolver uses cpu_feature_enabled()/system-wide capability intersection after alternatives patching, not speculative per-task state, and only for whitelist members. Because templates are restricted to ordinary LOCK/CMPXCHG/MFENCE/load/store sequences whose correctness does not depend on spectre-family toggles, IBRS/IBPB/STIBP/SSBD/MDS/TAA/BHI/retbleed/SLS mitigations do not invalidate the load-time choice. If a future kernel build changes the whitelist, newly loaded programs see the new table; existing programs remain correct because no admitted template relied on revocable features. 4. Exact inline-template versus thunk/helper contract. Resolution outcome is stored per entry as RES_INLINE_TEMPLATE, RES_INSTRUMENTED_THUNK, RES_NOINSTR_THUNK, or RES_EXISTING_HELPER. Selection order is deterministic: first admissible inline template, else admissible thunk, else existing helper, else verifier rejection only if no legal semantic realization exists. - Inline template means a contiguous byte sequence emitted as one JIT region and documented as equivalent to the named Linux primitive family for atomicity, tearing, and ordering. - Instrumented thunk means a fixed out-of-line kernel symbol compiled to call the same Linux primitive family through normal kernel APIs so that KCSAN/KASAN/KCOV/lockdep behavior matches existing kernel practice. This path is allowed only when ctx_mask excludes NMI and any other context that forbids such instrumentation. - Noinstr thunk means a dedicated notrace/noinstr/non-sleeping thunk for NMI/hardirq legality. It never sleeps, never takes mutexes, never calls back into BPF, and performs only the named atomic/LKMM primitive plus minimal register marshaling. - Existing helper is the final semantics-preserving path when instrumentation policy, width, provenance, or context prevent both inline and thunk realization. All three realizations are call-equivalent by construction: they have the same memory-order guarantee under LKMM, the same atomicity/tear-free behavior for the admitted width, the same fault behavior (none on admitted trusted memory), the same sleepability prohibition, and the same RCU/preemption preconditions as recorded in the SCV entry. 5. Non-movable JIT region enforcement. The verifier tags the source instruction and copies region metadata into bpf_insn_aux_data.sync_region_id plus a JIT-side jit_ctx.sync_regions[] table carrying [start_insn, end_insn, resolution, forbid_probe, forbid_patch_inside]. During x86 emission, emit_sync_region() outputs the full region contiguously and returns byte offsets recorded in the final prog aux metadata. Every pass must honor the region: constant blinding may only use a pre-approved blinded template variant for the whole region, not split immediates inside it; prologue/epilogue insertion occurs only outside region boundaries; direct-tail-call patch sites are placed outside sync regions; exception-table or NOP sled insertion cannot occur inside a region; kprobe/ftrace/fprobe registration rejects interior addresses; and any future text_poke-style rewrite of JIT text must target only designated patch slots outside sync regions. This makes the barrier concrete even if current x86 BPF JIT performs limited classic motion, because the hazard addressed is not just algebraic reordering but later splitting, probing, or patch-site insertion around an ordering-critical sequence. 6. Lifetime, ownership, and RCU rules. The SCV table is allocated during program finalization, stored as aux->sync_contracts, marked immutable before JIT emission, and owned by bpf_prog_aux. Executing JIT text does not dereference SCV memory; it uses fully resolved inline code or fixed thunk/helper call targets. The SCV sidecar is therefore metadata for load-time resolution and optional diagnostics only. It is freed together with bpf_prog_aux in the same deferred program teardown path and under the same grace-period discipline used for JIT text and fprobe/trampoline detachment, e.g. bpf_prog_put() -> deferred free -> RCU/tasks-trace grace period -> free JIT image and SCV sidecar. Because no runtime instruction embeds a direct SCV pointer, there is no fast-path UAF vector even during program replacement. 7. Alignment and rejection rules. For inline atomic families, natural alignment is mandatory. width 1 always passes alignment; width 2 requires 2-byte proof; width 4 requires 4-byte proof; width 8 requires 8-byte proof; width 16 requires 16-byte proof plus CX16 admissibility and a named 128-bit primitive family. If natural alignment or constant-offset proof is missing but a helper/thunk exists, fallback is mandatory and explicit. If the operation would violate the named primitive family's legality assumptions altogether, such as a 16-byte cmpxchg on an object class without 16-byte alignment guarantee or on an unsupported address class, the verifier rejects the program rather than silently weakening semantics.

2.4 Architecture

The system is intentionally split into portable verifier facts and x86-only realization:

Verifier / Load-Time Path
-------------------------
[BPF insn stream]
      |
      v
[Verifier sync-site recognizer]
      |
      +--> computes base type, non-faulting proof, fixed offset, width, alignment,
      |    cacheline containment, context mask, RCU requirement, lifetime class
      v
[Immutable SCV in bpf_prog_aux]
      |
      +--> entry[0..N-1]: {linux_primitive_id, memory_order, addr_class,
      |                    align_log2, proof_mask, ctx_mask, rcu_requirement,
      |                    admissibility_class, fallback_kind}
      v
[x86 admissibility resolver]
      |
      +--> consults stable-feature whitelist: TSO baseline, CX8, optional CX16
      +--> ignores revocable features: RTM/HLE/etc.
      +--> checks mitigation-safe template catalog
      v
[Per-entry resolution]
  |            |                 |
  |            |                 +--> RES_EXISTING_HELPER
  |            +--------------------> RES_NOINSTR_THUNK
  +---------------------------------> RES_INLINE_TEMPLATE / RES_INSTRUMENTED_THUNK

JIT / Runtime Path
------------------
[jit_ctx.sync_regions[]]
      |
      +--> region boundaries copied into bpf_insn_aux_data and final image metadata
      +--> forbid_probe + forbid_patch_inside flags
      v
[emit_sync_region()]
  | contiguous template only |
  | no interior blinding     |
  | no prologue splice       |
  | no tail-call patch slot  |
  | no extable/nop insert    |
      v
[Final JIT text]
  | fast path never dereferences SCV |
  | inline or thunk/helper target only |
      v
[Deferred teardown]
  --> detach trampolines/fprobes --> RCU/tasks-trace grace period --> free JIT image + SCV

Representative primitive mapping table:
- SCV_PRIM_SMP_LOAD_ACQUIRE   -> x86 plain load + acquire-preserving template equivalent to smp_load_acquire()
- SCV_PRIM_SMP_STORE_RELEASE  -> x86 release-preserving template equivalent to smp_store_release()
- SCV_PRIM_ATOMIC_FETCH_ADD   -> x86 LOCK XADD template or thunk/helper equivalent to atomic_fetch_add_*()
- SCV_PRIM_ATOMIC_CMPXCHG     -> x86 LOCK CMPXCHG/CMPXCHG16B template or equivalent thunk/helper
- SCV_PRIM_SMP_MB             -> x86 full barrier template equivalent to smp_mb()

The novelty-bearing bridge is the SCV plus resolution machinery, not any single opcode.

2.5 Implementation Plan

1. Verifier changes. Add a synchronization-site recognition pass in kernel/bpf/verifier.c after pointer/type stabilization and before final subprog/JIT decisions. Extend bpf_insn_aux_data with sync_region_id and sync_contract_index. Emit struct bpf_sync_contract_table into bpf_prog_aux, mark it immutable once verification succeeds, and plumb through verifier-derived facts: base register type, constant offset, tnum alignment proof, object/BTF alignment, map type, lifetime class, and attach-context mask. 2. Core metadata and lifetime plumbing. Add aux->sync_contracts, aux->nr_sync_contracts, and final-image byte-offset metadata for sync regions in kernel/bpf/core.c. Free SCV sidecar from the same deferred path that frees JIT text, after the grace period already used for safe program disposal and trampoline detachment. 3. x86 admissibility resolver. In arch/x86/net/bpf_jit_comp.c add a template catalog keyed by linux_primitive_id, width, memory_order, and context class. Add x86_bpf_resolve_sync_contracts() that runs after alternatives/capability intersection are settled and before final emission. The resolver must consult only the stable-feature whitelist (TSO baseline, CX8, optional CX16) and reject any template that depends on revocable features or dynamic policy. 4. Non-movable region machinery. Represent each contracted site as a region object in jit_ctx.sync_regions[]. Update every relevant emission phase so the region is contiguous and sealed: hardening/constant blinding chooses whole-region variants only; prologue/epilogue insertion is outside region bounds; any tail-call immediate patch slots are emitted outside the region; no debug NOP island or exception metadata is inserted inside the region. Expose region byte ranges to probe-registration code so kprobe/fprobe/ftrace reject interior addresses. 5. Thunk library. Implement two families of out-of-line symbols in arch/x86 or kernel/bpf: instrumented thunks that call ordinary Linux atomic APIs for task/softirq-safe contexts, and notrace/noinstr thunks for NMI/hardirq-safe contexts. Both families must be non-sleeping, must not recurse into BPF, and must preserve the SCV-defined RCU and preemption assumptions. The thunk ABI passes data pointer, width/order selector, and value operands in fixed registers; no SCV pointer is passed. 6. Deterministic fallback semantics. Encode per-entry resolution in readonly aux metadata for observability and debugging. Expose debug output through bpftool or tracepoints as contract index -> primitive id -> inline/thunk/helper resolution -> reason code, so portability is explicit and deterministic rather than implicit. 7. Selftests. Add BPF selftests covering map-value atomics, trusted BTF object fields, RCU-required object access, PREEMPT_RT-safe attach types, NMI attach types, sanitizer-enabled kernels, split-lock-sensitive offsets, and CX16/no-CX16 systems. Each test should assert not just functional correctness but the expected resolution mode and region-sealing behavior.

2.6 Validation Plan

1. LKMM equivalence tests. For every primitive family and memory order, run litmus-style producer/consumer and compare-exchange tests comparing inline template, instrumented thunk, noinst thunk, and helper fallback against the corresponding Linux primitive semantics. Include acquire/release/full-barrier publication cases and verify no weaker outcome appears. 2. Context legality matrix. Exercise task, softirq, hardirq, NMI, and PREEMPT_RT kernels. Confirm that sites whose ctx_mask includes NMI never resolve to instrumented thunks, and that no fallback path sleeps or takes incompatible locks. 3. Verifier proof coverage. Create tests that vary pointer class, nullability, constant offset, tnum alignment, BTF field alignment, per-cpu/global map classes, and RCU-held/not-held states. Confirm inline eligibility only when every required proof bit is present, fallback when semantics remain legal but proofs are insufficient, and verifier rejection only when no legal realization exists. 4. Stability under patching/instrumentation. Enable KCSAN, KASAN, KCOV, ftrace, and BPF trampolines. Verify region byte ranges are reported as non-probeable, that thunk/helper targets remain semantically identical under instrumentation, and that no interior patch site exists within sealed regions. Confirm that alternatives/static_call/text_poke activity outside the JIT image does not change contract resolution. 5. Microarchitectural corner cases. Run split-lock-sensitive tests by placing fields near cacheline boundaries, confirm inline rejection or fallback when cacheline containment is not proven, verify CX16-gated 16-byte contracts only on systems with system-wide CX16, and confirm no contract uses RTM/HLE even when present. 6. Lifetime and teardown. Stress load/unload, program replacement, map lifetime churn, and fprobe attachment/detachment while synchronization-heavy programs execute. Confirm JIT code never dereferences freed SCV metadata and that deferred freeing after grace period prevents use-after-free. 7. Performance validation. Measure hot-path latency and throughput for map-value synchronization workloads against helper-only baseline on representative x86 servers. Attribute improvement only to admitted inline templates; fallback paths should match existing semantics and near-existing overhead.

4. Risks & Mitigations

  • Risk: prior-art challenge that the design is an obvious combination of verifier facts and JIT templates. Mitigation: anchor claims on the exact SCV schema, the normative primitive-family identifier, the stable-admissibility table, and the non-movable region enforcement across late-patching/probing phases.
  • Risk: feature-stability arguments may be attacked as vague. Mitigation: define an explicit whitelist limited to TSO baseline, CX8, and optional system-wide CX16, and expressly exclude RTM/HLE and any revocable or mitigation-sensitive feature.
  • Risk: instrumentation and PREEMPT_RT semantics may diverge between inline and fallback paths. Mitigation: separate instrumented and noinst thunk families, encode context legality in SCV, and require call-equivalence to named Linux primitives as a contractual invariant.
  • Risk: alignment and split-lock corner cases could undermine correctness or performance claims. Mitigation: make natural alignment and, where configured, cacheline-containment proof mandatory for inline LOCKed templates; otherwise force deterministic fallback or reject if no legal primitive exists.
  • Risk: lifetime bugs if JIT text references SCV metadata. Mitigation: prohibit runtime dereference of SCV entries from JIT code and free SCV only in the same deferred teardown path as JIT text after grace period.
  • Risk: overbreadth around unsupported pointer classes. Mitigation: limit admissible address classes to verifier-proven non-faulting kernel-owned memory such as stable map values and trusted BTF object fields, and explicitly exclude user, packet, stack, and unresolved dynptr memory.
