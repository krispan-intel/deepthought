2. Technical Invention Disclosure

2.1 Problem Statement

Linux mutex and rwsem optimistic spinning is primarily driven by scheduler and owner-state signals such as owner-on-CPU, need_resched, and existing optimistic-spin queue logic. That logic is useful when the owner is about to release, but it is blind to a different failure mode that appears on modern Intel x86 server systems: the lockword load itself becomes repeatedly expensive because the cache line carrying the lockword is bouncing across sockets, is persistently remote in the coherence fabric, or is contending with other sharers and memory hierarchy pressure. In those cases, the kernel may legally spin yet lose throughput, tail latency, and energy because the polling load is itself a high-latency microarchitectural operation. Existing Linux heuristics do not consume precise per-lockword load-address evidence from the processor. Existing tools such as perf and lockstat can observe contention, but they do not safely translate sampled lockword load addresses into a live in-kernel policy input that can be consulted in the mutex/rwsem midpath with a hard bound on added work. The technical problem is therefore to turn Intel PEBS precise retired-load evidence into a kernel-safe, lifetime-safe, low-overhead, per-lock control signal that reduces wasteful spinning for specific locks while preserving Linux lock correctness, ABI-visible structure layout, and all existing sleep/wake fallback behavior.

2.2 Prior-Art Gap

Prior art falls into three buckets and leaves a material gap. First, observability tools such as perf, PEBS-based postmortem analysis, and lockstat are diagnostic; they report contention or latency but do not form a production control plane that is consumed directly in mutex/rwsem optimistic-spin decisions. Second, existing Linux mutex and rwsem spinning heuristics are owner- and scheduler-centric; they do not use precise sampled lockword data addresses to discriminate locks whose polling load is persistently microarchitecturally expensive. Third, generic adaptive locks and hash/bucket specialization schemes do not disclose a Linux-safe address-to-object identity layer with explicit generation management, tombstoning, and quarantine that lets asynchronous samples be attributed to the correct live lock object without modifying struct mutex or struct rw_semaphore. The revised disclosure therefore differentiates on concrete mechanisms absent from the art: an allowlisted PEBS format gate that refuses samples unless PERF_SAMPLE_ADDR is known to represent the retired load's data linear address; a generation-safe registration registry for kernel lockwords only; a static-key-gated fixed-capacity lookup usable from non-sleeping optimistic-spin code; and a stable RCU-lifetime hint object whose packed policy word is updated in place, avoiding callback churn. The claims are expressly narrowed away from user-futex lockwords, futex dedicated-domain transitions, call_rcu callback sharding, and other committee-flagged side mechanisms. That narrowing strengthens non-obviousness by centering the inventive core on generation-safe address-to-lock identity plus direct midpath consumption in existing Linux mutex/rwsem logic.

2.3 Proposed Invention

Introduce a Lockword Evidence and Spin Hint subsystem (LESH) for Linux on supported Intel x86 PEBS-capable systems. 1. Scope and supported objects. The claimed embodiment covers kernel mutex and rwsem lockwords only. Registration APIs accept only kernel addresses associated with known mutex/rwsem polling words, for example a mutex owner field or rwsem count/owner field used by the existing optimistic-spin logic. User-space futex addresses are not accepted in the claimed embodiment; any future futex extension is optional, outside the independent claims, and would require mmu_notifier-backed invalidation and a separate transition-state proof. 2. Opt-in generation-safe registration. Registration creates an external metadata object, not an in-structure field, containing: canonical lockword address; lock class enum {mutex,rwsem}; byte width and alignment; a monotonically increasing generation number; a 64-bit lock_cookie unique within the boot; module or call-site fingerprint; state {ACTIVE,TOMBSTONED,QUARANTINED}; registration timestamp; and a pointer to a stable hint object. Registration inserts the metadata into a fixed-capacity hash table keyed by canonical address. Unregistration marks the entry TOMBSTONED, removes it from new policy formation, and places it in QUARANTINED for at least two RCU grace periods plus a time quarantine window so deferred PEBS records cannot be rebound to a newly allocated object at the same address. Re-registration of the same address is denied until quarantine expiry. This is the generation-safe address-to-object identity core. 3. PEBS admissibility gate. The subsystem enables only on Intel PMU models listed in an in-kernel allowlist where the selected precise retired-load event is known, on that model, to populate perf sample address data with the retired load's data linear address. perf_event_attr is configured with precise_ip=3, sample_period set by a boot/sysctl knob, sample_type containing at least PERF_SAMPLE_IP, PERF_SAMPLE_ADDR, PERF_SAMPLE_TID, PERF_SAMPLE_TIME, PERF_SAMPLE_CPU, and PERF_SAMPLE_IDENTIFIER, and optionally PERF_SAMPLE_WEIGHT and PERF_SAMPLE_DATA_SRC when the chosen event provides latency or memory-source qualifiers. exclude_user=1 and exclude_guest=1 are mandatory in the claimed embodiment so only kernel-host samples are consumed. If PERF_SAMPLE_ADDR is absent, zero, non-canonical after LA57/LAM normalization, outside kernel canonical address space, or inconsistent with data_src/mem_op=load when PERF_SAMPLE_DATA_SRC is present, the sample is dropped. If the PMU model, virtualization state, PEBS format, or perf driver cannot prove that sample.addr is a retired-load data address, the feature hard-disables rather than degrading to IP-only attribution. 4. Fast NMI collection. The PEBS overflow/NMI path performs only bounded copying into a preallocated per-CPU ring buffer. Each ring record is fixed-size and contains: canonicalized sampled_addr, cpu, timestamp, event identifier, optional latency weight, and a compact validity bitmap. The NMI path does not allocate, resolve hashes, take sleeping locks, or publish policy. 5. Deferred translation and aggregation. A worker thread drains per-CPU rings, resolves sampled_addr against the ACTIVE registry under RCU, and updates a per-lock evidence accumulator. Accumulators maintain a sliding window including sample_count, weighted_cost_sum, last_window_start, and suppression_score. An update contributes only if the sample matches the registered lock class and passes the validity bitmap. The scoring function is conservative: a lock is marked expensive only after a minimum number of samples in a window and hysteresis relative to a global baseline of lockword polling cost on that machine. Thus a single noisy sample cannot suppress spinning. 6. Stable RCU-lifetime hint object. Each registered lock owns one stable hint object allocated at registration time and freed only after unregister plus grace period. The object contains a packed policy_word updated atomically in place, for example {mode, spin_budget_cap, epoch, reason_code}. mode is one of INHERIT_DEFAULT, TIGHTEN_BUDGET, or BYPASS_SPIN. spin_budget_cap is a small cap on loop iterations or elapsed cycles relative to the existing midpath. The worker updates policy_word with store-release semantics only when thresholds and hysteresis justify a state change. Because the pointer to the hint object is stable, policy updates do not create repeated call_rcu churn; RCU protects lifetime, while atomic publication protects content ordering. 7. Direct mutex/rwsem consumption. Existing optimistic-spin call sites invoke a helper such as lesh_should_spin(lockword_addr, class, default_budget) near the start of mutex_optimistic_spin() and rwsem_optimistic_spin()/rwsem_spin_on_owner(), before entering or extending the owner-spin loop. The helper is guarded by a static_key_false so the zero-registration case remains essentially the current kernel fast path. When enabled, the helper performs a bounded RCU read-side lookup in the fixed-capacity address table, reads the stable hint object's packed policy_word with READ_ONCE, and either (a) leaves the existing logic unchanged, (b) tightens the spin budget, or (c) bypasses the optimistic-spin loop and falls through to the existing sleep/slowpath. The helper never allocates, never sleeps, and never changes mutex/rwsem correctness; it only gates optional spinning. 8. Security gating. Because the evidence source is address-sensitive, the feature is privileged. Enabling the PEBS control plane requires a boot parameter or sysctl writable only by CAP_PERFMON or CAP_SYS_ADMIN in the initial namespace. Raw sampled addresses are not exposed through tracepoints or debugfs. User space cannot register objects, cannot query per-lock hints, and cannot influence policy except indirectly through ordinary workload behavior against kernel locks already chosen by kernel code for registration.

2.4 Architecture

The architecture has five cooperating planes.

A. Registration plane. Kernel code paths that own eligible mutex/rwsem objects call lesh_register_*() and lesh_unregister_*(). The registration plane canonicalizes the kernel address, checks bucket-capacity limits, allocates metadata and one stable hint object from a mempool, publishes the entry into a fixed-capacity RCU-readable hash table, and maintains ACTIVE/TOMBSTONED/QUARANTINED state. Admission control rejects registrations that would exceed a configured bucket-depth bound, preserving lookup cost.

B. Evidence acquisition plane. A per-CPU perf_event is created only on allowlisted Intel x86 models and only with precise retired-load events that are known to return a valid data linear address in PERF_SAMPLE_ADDR. The subsystem requires sample_type bits for address, time, cpu, identifier, and IP, optionally weight/data_src. exclude_user=1 and exclude_guest=1 are mandatory. Unsupported PMU model, guest virtualization, PEBS-off, or format ambiguity forces a hard disable path.

C. Translation plane. NMI handlers push compact records into per-CPU rings. A worker thread drains those rings, rejects invalid samples, canonicalizes addresses with LA57 awareness and tag stripping rules, resolves them against the ACTIVE registry under RCU, and updates evidence accumulators keyed by lock_cookie/generation.

D. Policy plane. Each lock's accumulator computes a suppression_score over a sliding window. Hysteresis yields three states: INHERIT_DEFAULT, TIGHTEN_BUDGET, or BYPASS_SPIN. State is encoded into a packed policy_word inside the stable hint object. Publication is by atomic store to that word; the RCU pointer to the hint object itself is stable for the lock lifetime, so no update-time callback storm is created.

E. Consumption plane. mutex_optimistic_spin() and rwsem_optimistic_spin()/rwsem_spin_on_owner() call a static-key-gated helper. The helper runs only where existing optimistic spinning is already permitted and non-sleeping. It performs one bounded hash lookup and one READ_ONCE of the policy_word. On PREEMPT_RT or builds where optimistic spinning is disabled or materially constrained, the static key remains off and the helper resolves to the existing behavior.

The critical invariants are: no lock layout change; no PEBS processing in the lock-acquire atomic path beyond a bounded hint lookup; no sample-to-object attribution without a live ACTIVE registration; no address reuse without tombstone and quarantine; and no policy action that can create a new correctness state distinct from the existing Linux slowpath.

2.5 Implementation Plan

1. Model/PMU gating. Add an x86-specific allowlist table keyed by family/model/stepping or PMU capability bits for precise retired-load events that are acceptable for lockword sampling. For each supported model, store the raw event selector or perf alias, whether latency weight is available, and whether PERF_SAMPLE_DATA_SRC is expected. During init, verify CPU vendor is Intel, PEBS is enabled, the PMU driver reports precise sampling support, and the system is not in a virtualization mode where data address semantics are uncertain. If any check fails, leave the static key off and report feature-disabled. 2. perf_event configuration. Create one per-CPU kernel perf event with attr.precise_ip=3, attr.sample_period set by tunable, attr.exclude_user=1, attr.exclude_guest=1, attr.sample_type containing PERF_SAMPLE_IP | PERF_SAMPLE_ADDR | PERF_SAMPLE_TID | PERF_SAMPLE_TIME | PERF_SAMPLE_CPU | PERF_SAMPLE_IDENTIFIER, plus PERF_SAMPLE_WEIGHT and PERF_SAMPLE_DATA_SRC when supported. Use pinned/non-inherit kernel events. Register an overflow handler that copies only fixed-size records into a preallocated per-CPU ring. 3. Address normalization and validity. Implement lesh_canonicalize_addr() that strips any architecturally ignored metadata bits, validates canonical form under 4-level or 5-level paging, and rejects non-kernel addresses in the claimed embodiment. If PERF_SAMPLE_DATA_SRC exists and does not encode a load, reject the sample. If sample.addr is zero or missing, reject the sample. This directly addresses the committee request for exact record-field-level gating. 4. Registry and lifetime. Implement a fixed-capacity hash table with RCU-readable buckets and mempool-backed nodes. Store metadata {addr, class, generation, lock_cookie, state, hint_ptr}. On unregister: set state=TOMBSTONED; remove from ACTIVE matching; advance generation; queue the entry into a quarantine list; after two grace periods and quarantine timeout, recycle the node. Deny reuse of the same address while quarantined. This yields a mechanically checkable address-reuse rule. 5. Hint representation. Allocate one stable hint object per registration. Encode the policy as a single machine word or tightly packed structure updated atomically with store-release semantics. Suggested fields: 2-bit mode, 8-bit spin_cap, 16-bit epoch, 8-bit reason_code, remaining bits reserved for version and confidence. Because the pointer is stable, policy updates are in-place and do not generate repeated call_rcu traffic. 6. Aggregation logic. The worker keeps a per-lock accumulator: sample_count, weighted_cost_sum, moving average, last_transition_jiffies, and hysteresis thresholds. Default transition example: INHERIT_DEFAULT to TIGHTEN_BUDGET when the lock has at least N valid samples in the window and mean weighted cost exceeds baseline by X%; TIGHTEN_BUDGET to BYPASS_SPIN when the score exceeds Y for M consecutive windows; reverse transitions require lower thresholds for K windows. Parameters are tunable but bounded. 7. Midpath hook points. Add hooks in mutex_optimistic_spin() before OSQ/owner-spin loop entry and in rwsem_optimistic_spin() or rwsem_spin_on_owner() before entering the owner-spin polling loop. The helper must be inline, non-allocating, and wrapped by static_branch_unlikely(&lesh_enabled_key). The helper executes under an explicit rcu_read_lock_sched_notrace()/unlock pair or another existing non-sleeping read-side region already valid at that call site. If CONFIG_PREEMPT_RT is enabled or optimistic spinning is disabled for that primitive, compile the helper to 'no opinion' so existing behavior is preserved. 8. Security surface. Enabling/disabling and diagnostics are privileged to CAP_PERFMON or CAP_SYS_ADMIN in the init namespace. Export only aggregate counters such as registered_locks, samples_accepted, samples_dropped_reason, and policy_state_counts. Do not export raw sampled addresses, lockword addresses, or per-lock identifiers to unprivileged users. Add quotas: max_registered_locks, max_updates_per_second per lock, and per-module registration caps to resist adversarial churn. 9. Explicit exclusion of futex embodiment. Remove futex dual-domain mechanisms from the implementation scope of the claimed embodiment. If a future product version adds PEBS-guided futex specialization, it must use a separate design with mmu_notifier invalidation, VMA tombstoning, and a fully proved DEFAULT_ONLY/DUAL/DEDICATED_ONLY state machine; that future work is not required for this disclosure to stand.

2.6 Validation Plan

Validation has to prove correctness, attribution fidelity, bounded overhead, and security. 1. PEBS positive/negative controls. Add x86 selftests that create a kernel workload intentionally polling registered mutex/rwsem lockwords and verify that accepted samples contain non-zero canonical kernel PERF_SAMPLE_ADDR values mapping to the registered lockword. Negative controls force unsupported conditions: missing PERF_SAMPLE_ADDR, guest/virtualized environment, unsupported PMU model, user-only samples, and an event type not known to carry data linear address. Expected result: subsystem hard-disables or drops samples without forming policy. 2. Address-reuse and lifetime tests. Add KUnit tests for register/unregister/quarantine/re-register sequences. Verify that samples arriving after unregister cannot be attributed to a new lock at the same address until quarantine expires. Inject delayed ring records to confirm stale samples are discarded. Verify two-grace-period quarantine invariant. 3. Midpath safety tests. Instrument mutex_optimistic_spin() and rwsem optimistic-spin call sites to confirm the helper does not sleep, allocate, or exceed the configured instruction/latency budget. Measure added cycles in the enabled-but-no-match case and the enabled-with-match case. The target is a bounded constant-time lookup with no observable semantic change when mode=INHERIT_DEFAULT. 4. Functional contention benchmarks. Run kernel microbenchmarks and real workloads that create remote lockword bouncing across sockets. Compare baseline Linux against LESH enabled. Expected outcomes: reduced wasted spin cycles, reduced LLC/coherence traffic for affected locks, and no regression for locks that remain in INHERIT_DEFAULT. 5. PREEMPT_RT and config matrix. Build-test with CONFIG_PREEMPT_RT, CONFIG_MUTEX_SPIN_ON_OWNER on/off, CONFIG_RWSEM_SPIN_ON_OWNER on/off, and different RCU flavors. Expected outcome: helper is compiled out or returns no opinion where optimistic spinning is disabled or inappropriate. 6. Security tests. Verify only privileged users can enable the subsystem. Confirm debugfs/tracefs does not expose raw kernel addresses. Confirm unprivileged namespaces cannot observe hint state or sampled addresses. Run registration-churn stress to verify quotas and mempool bounds prevent memory exhaustion. 7. Prior-art differentiation packet. Prepare side-by-side code-path analysis against current Linux mutex/rwsem spin logic and perf/lockstat behavior, showing that existing code has no generation-safe address-to-lock registry and no direct per-lock PEBS-driven spin suppression input. That packet supports prosecution and committee review.

4. Risks & Mitigations

  • Risk: PERF_SAMPLE_ADDR may not always denote the retired load's data linear address on every Intel PMU model or under virtualization. Mitigation: use an explicit CPU-model allowlist, require precise_ip at the configured level, require PERF_SAMPLE_ADDR presence, optionally cross-check PERF_SAMPLE_DATA_SRC indicates a load, and hard-disable the subsystem when format semantics are uncertain rather than falling back to IP-only attribution.
  • Risk: Address reuse could misattribute late PEBS samples to a different lock object. Mitigation: generation-safe registration with TOMBSTONED and QUARANTINED states, minimum two-grace-period quarantine, reuse denial during quarantine, and worker-side generation/state checks before policy formation.
  • Risk: Publication churn could create excessive call_rcu pressure or ordering bugs. Mitigation: allocate one stable hint object per registration, protect lifetime with RCU, and publish state changes by atomic in-place updates to a packed policy word using release/acquire semantics.
  • Risk: The extra lookup in optimistic-spin code could add unacceptable latency. Mitigation: static_key gating, fixed-capacity bounded hash table, admission control on bucket depth, preallocated nodes, inline helper, and compile-out behavior on PREEMPT_RT or configs without optimistic spinning.
  • Risk: Security or side-channel exposure from address-sensitive sampling. Mitigation: kernel-only sampling via exclude_user=1, exclude_guest=1, privileged enablement through CAP_PERFMON or CAP_SYS_ADMIN in the initial namespace, no export of raw addresses or per-lock identifiers to unprivileged interfaces, and aggregate-only diagnostics.
  • Risk: Registration churn or malicious kernel modules could cause memory pressure or destabilize lookup bounds. Mitigation: mempool-backed fixed maxima, per-module or subsystem quotas, rate-limited policy updates, and rejection once capacity or bucket-depth thresholds are reached.
  • Risk: Prior-art challenges may argue obvious combination of profiling and adaptive spinning. Mitigation: anchor claims on the specific combination of PEBS data-address admissibility rules, generation-safe address-to-object identity with quarantine, and bounded direct consultation in existing mutex/rwsem optimistic-spin code without lock layout changes.
  • Risk: Committee feedback raised user-futex lifetime and dual-domain transition issues. Mitigation: the claimed embodiment excludes user-futex lockwords and futex domain transitions entirely; any future futex embodiment would require mmu_notifier invalidation, VMA-range contracts, and a separate mechanically checkable transition-state design.
