2. Technical Invention Disclosure

2.1 Problem Statement

Linux memory tiering reduces heterogeneous memory to node distance and demotion targets. That flattening loses CXL-specific facts such as HDM decoder interleave width (1/2/4/8/16-way), switch depth (0–4 hops), host-bridge attachment socket, and cross-socket asymmetry. As a result, the demotion path in shrink_folio_list() (mm/vmscan.c) can demote a folio from local DRAM into a technically slower CXL tier that is a poor choice relative to another CXL region with lower latency or fewer switch hops, increasing refaults measured via workingset_refault() and remote reads visible in uncore CHA counters. Additionally, Linux lacks a mechanism to translate x86 PEBS load-latency samples (MSR_PEBS_LD_LAT_THRESHOLD filtered by data_src.mem_lvl == REMOTE_CCE or data_src.mem_snoop == MISS for CXL address ranges) into direct folio promotion decisions. Hot CXL folios remain trapped in high-latency memory until coarse PTE-scanning in task_numa_work() or refault heuristics in workingset_activation() eventually detect them — a latency of seconds to minutes versus the sub-millisecond detection possible with PEBS at 10K sample/s. Furthermore, large folios (order-9 PMD folios = 2MB on x86) that cross multiple HDM interleave stripes (typical stripe width: 256B–4KB granularity, meaning a single 2MB folio spans 512–8192 stripes across N devices) amplify LLC misses due to non-uniform access latency across stripes, increase CXL link queue depth on multi-device interleaves, and degrade tail latency. Current transparent hugepage (THP) policy in hugepage_vma_check(), compaction in compact_zone(), and folio allocation in __alloc_pages() are entirely unaware of CXL stripe boundaries — creating a tension between TLB reach optimization and fabric-local data placement that no existing kernel subsystem resolves.

2.2 Prior-Art Gap

**Topology exposure without mm integration.** HMAT (ACPI 6.5 §5.2.27) and CDAT (CXL 3.1 §8.1.11) expose latency and bandwidth attributes, and the CXL core (drivers/cxl/) knows region and decoder topology via struct cxl_region and struct cxl_decoder. However, Linux mm does not preserve that information as a first-class policy object in struct memory_tier (mm/memory-tiers.c), struct pglist_data, or the demotion graph built by migrate_demote_mapping_update(). The memory_tier framework classifies nodes by abstract latency buckets but does not encode switch depth, interleave width, host-bridge socket affinity, or decoder geometry. **Coarse NUMA balancing without PEBS-CXL bridge.** NUMA balancing (kernel/sched/fair.c: task_numa_work) relies on hinting faults from PTE protection downgrades and scans page tables at configurable intervals (numa_balancing_scan_delay_ms). DAMON (mm/damon/) observes access frequency at region granularity (minimum 4KB, typically larger). Neither converts x86 PEBS load-latency samples — specifically the data_src field encoding memory hierarchy level and snoop result — for CXL-backed PFNs into direct folio-migration policy compatible with existing mm locking rules (folio_lock, lru_lock, migrate_folio). The perf subsystem can record these samples to userspace ring buffers but provides no kernel-internal state machine for mm consumption. **Folio management blind to stripe geometry.** THP (mm/huge_memory.c), compaction (mm/compaction.c), and folio management (mm/filemap.c, mm/page_alloc.c) reason about contiguity, refaults, and generic NUMA locality but not decoder stripe boundaries. The CXL subsystem (drivers/cxl/core/region.c) manages region creation and decoder programming (cxl_decoder_commit) without providing an mm-facing abstraction that classifies PFN ranges by stripe residency. No existing work addresses the interaction of folio order heuristics (THP policy via /sys/kernel/mm/transparent_hugepage, madvise MADV_HUGEPAGE, compaction watermarks, __GFP_COMP semantics) with CXL interleave geometry, nor defines precedence when multiple policies demand conflicting folio orders. **No prior art bridges all three gaps.** CXLMemUring proposes asynchronous CXL pool access but operates at device-driver level without kernel mm integration. TPP (Transparent Page Placement) adds tiered placement heuristics but lacks CXL topology awareness. MEMTIS profiles memory access but targets offline analysis rather than online kernel policy. No published work or kernel patchset combines fabric-topology-derived allocation classes, PEBS-to-mm promotion pipelines, and stripe-aware folio order capping in a unified framework.

2.3 Proposed Invention

Introduce three tightly integrated kernel abstractions that bridge CXL fabric topology into Linux mm policy: **1. Fabric Locality Class (FLC).** An FLC is a kernel object (struct flc, embedded in struct memory_tier) derived from CXL HDM decoder geometry, region membership, host-bridge attachment, x86 socket proximity, and measured latency bands. Each FLC is published with an epoch number (monotonically increasing u64, incremented on any topology change) and a deterministic class_id computed as: ``` class_id = (socket_local_flag << 16) | (switch_depth_inverse << 12) | (interleave_width_log2 << 8) | latency_bucket ``` where socket_local_flag is 1 when the CXL root port's PCIE segment matches the CPU socket's root complex (determined via ACPI _PXM → NUMA node → socket mapping), switch_depth_inverse is (MAX_SWITCH_DEPTH(4) − observed_depth) to prefer direct-attached regions (value 4 for direct, 3 for single-switch, etc.), interleave_width_log2 is ilog2(HDM decoder interleave ways) to favor striped bandwidth (0 for ×1, 1 for ×2, 2 for ×4, 3 for ×8), and latency_bucket is a 4-bit quantized band (0–15) derived from CDAT DSLBIS read latency entries at region-commit time (see §3 for calibration details). FLCs are RCU-published via rcu_assign_pointer() on struct memory_tier, allowing lockless reads in demotion and allocation hot paths. On topology change (CXL region create/destroy via /sys/bus/cxl or hot-plug events), the CXL driver calls flc_topology_update() which: (a) allocates a new FLC set via kmalloc(GFP_KERNEL), (b) populates class_ids from current decoder state, (c) publishes via rcu_assign_pointer(), (d) calls synchronize_rcu() before kfree(old_flc) to ensure no in-flight readers reference stale data. In-flight allocations during the RCU grace period observe either the old or new FLC atomically — both are valid — and the epoch number lets slow paths detect staleness and retry. **2. PEBS-to-MM Promotion Mailbox.** A per-CPU NMI-safe ring buffer (struct pebs_mm_mailbox, 4096 entries × 16 bytes = 64KB per CPU) receives CXL-relevant PEBS samples from the PMU NMI handler. Each entry contains: {pfn: u64, latency_cycles: u16, data_src: u16} extracted from the PEBS record's data_src and latency fields. The NMI handler filters samples where data_src.mem_lvl indicates CXL/remote and latency exceeds a configurable threshold (default: 500 cycles, tunable via /sys/kernel/mm/flc/pebs_latency_threshold_cycles). A per-CPU kthread (flc_pebs_promoter, running at SCHED_IDLE priority) drains the mailbox every flc_pebs_scan_interval_ms (default: 100ms). For each PFN, the kthread: (a) looks up the folio via pfn_to_folio(), (b) checks folio_test_lru() and folio_trylock(), (c) if the folio's current FLC class_id is worse (numerically lower) than the best available DRAM tier, queues it on a per-node promotion list, (d) calls migrate_pages(&promote_list, alloc_promotion_target, ...) with the target node selected by highest class_id. Overflow semantics: when the ring buffer is full, the NMI handler increments a per-CPU overflow counter (exported via debugfs) and drops the sample — acceptable because PEBS is statistical and individual sample loss does not affect convergence. The mailbox uses a lock-free SPSC (single-producer/single-consumer) ring with atomic head/tail indices. The NMI handler is the sole producer; the kthread is the sole consumer. No spinlocks are taken in NMI context. The kthread acquires folio_lock() and lru_lock only outside NMI, following existing mm locking conventions. **3. Stripe Residency Descriptor (SRD) and Folio Order Cap.** For each CXL region, at region-commit time (cxl_region_commit → flc_srd_build), the kernel constructs an SRD table: a flat array of struct srd entries indexed by region-relative stripe index. The PFN-to-SRD-index mapping is: ``` srd_index = (pfn - region_base_pfn) >> stripe_order ``` where stripe_order = ilog2(stripe_granularity >> PAGE_SHIFT). For a 256B stripe granularity with 4KB pages, stripe_order = 0 and every page is its own stripe — but the SRD aggregates into stripe groups of 2^stripe_order pages. For the common case of 256B granularity on a 4-way interleave across a 1TB region, the SRD array contains (1TB / 256B) / 4 = ~1 billion entries — far too large. Therefore, the SRD uses a two-level scheme: a top-level array of SRD group descriptors (one per 2MB = PMD-size region, ~512K entries for 1TB), each containing {device_bitmap: u16, max_safe_order: u8, stripe_latency_penalty_ns: u16}. The device_bitmap encodes which CXL devices are touched by a PMD-aligned folio at this offset; max_safe_order is the largest folio order where all pages land on a single device; stripe_latency_penalty_ns is the measured cross-device penalty. The SRD group array occupies 5 bytes × 512K = 2.5MB per TB of CXL region — acceptable overhead for server-class systems. **Folio order cap integration points.** The max_safe_order from SRD lookup is applied as a folio order ceiling at the following specific call sites: (a) **Demotion path — demote_folio_list() in mm/vmscan.c:** Before calling alloc_demote_folio() which invokes __alloc_pages(), the demotion logic calls flc_max_order(target_nid, folio_pfn(folio)) to obtain the SRD-derived order cap. If the current folio order exceeds the cap, the folio is split via split_folio_to_order(folio, max_safe_order) before migration. (b) **Compaction path — compact_zone() in mm/compaction.c:** In the isolate_migratepages_block() scanner, when a candidate folio resides in a CXL-backed PFN range (detected via flc_is_cxl_pfn(pfn) — a static branch enabled only when CXL regions are active), the migration target order is clamped to flc_max_order(). (c) **Anonymous fault path — do_anonymous_page() and do_huge_pmd_anonymous_page() in mm/memory.c:** When the fault target VMA's backing node is CXL-backed, the THP order decision in thp_vma_allowable_orders() is intersected with the SRD order cap via flc_max_order(nid, fault_pfn). This prevents allocation of PMD-size folios that would span multiple devices. (d) **User-initiated migration — migrate_pages() syscall in mm/migrate.c:** The move_pages_to_node() helper applies flc_max_order() capping identically to path (a). (e) **Explicit exclusion — NUMA balancing in kernel/sched/fair.c:** The NUMA balancing migration path (migrate_misplaced_folio) is NOT capped by SRD because it operates on DRAM-to-DRAM migrations where stripe geometry is irrelevant. This is enforced by checking flc_is_cxl_pfn(target_pfn) before applying the cap. **Cost-benefit decision function.** The folio order cap is not applied unconditionally — a cost-benefit gate determines whether capping (and potential splitting) is worthwhile: ``` should_cap = (stripe_latency_penalty_ns > tlb_miss_cost_ns × (current_order - max_safe_order)) ``` where stripe_latency_penalty_ns is from the SRD group descriptor (calibrated at region-commit time from CDAT DSLBIS read-latency entries for cross-device access; if CDAT is unavailable, derived from HMAT initiator-target latency differences; if neither is available, a conservative static default of 150ns is used — representing typical CXL 2.0 added latency over DDR5), tlb_miss_cost_ns is a per-architecture constant (x86: 7ns for L2 TLB miss + page walk, from Intel optimization manual §2.1.3; configurable via /sys/kernel/mm/flc/tlb_miss_cost_ns for platforms with profiled values), and current_order minus max_safe_order represents the number of additional TLB entries needed after splitting. This threshold is evaluated at region-commit time (not in the hot path): the SRD group descriptor stores a precomputed should_cap boolean for each PMD-aligned group, so the hot-path check is a single byte load from the SRD array — O(1) with no runtime sampling, no side-channel risk, and no allocation-path variability. **RCU lifecycle for SRD.** The SRD table is allocated at cxl_region_commit time and freed at cxl_region_destroy time. During the region's lifetime, the SRD is immutable — no hot-plug updates modify stripe geometry because CXL spec §8.2.9 requires region destruction and re-creation to change interleave settings. The RCU publication is therefore one-shot: rcu_assign_pointer(region→srd, new_srd) at commit, rcu_dereference(region→srd) in hot paths, kfree_rcu(old_srd, rcu) at destroy. Concurrent readers during region destruction observe either the valid SRD or NULL (triggering fallback to uncapped folio order), both safe. There is no concurrent mutation case to synchronize because CXL region reconfiguration inherently serializes through cxl_region_rwsem.

**Data flow:** (1) At region-commit, the CXL driver calls flc_topology_update() which reads decoder geometry and CDAT/HMAT tables, computes FLC class_ids, builds the SRD group array with precomputed should_cap flags, and RCU-publishes both. (2) During steady-state operation, demotion/compaction/fault paths perform O(1) SRD lookups (array index from PFN) and O(1) FLC reads (rcu_dereference on memory_tier) with no locks in the fast path. (3) Concurrently, the PMU NMI handler deposits PEBS samples into per-CPU SPSC rings; kthreads drain these and issue migrate_pages() calls for promotion. (4) On topology change, new FLC/SRD are published via RCU; old structures are freed after a grace period.

**Locking hierarchy:** flc_topology_update() holds cxl_region_rwsem (existing) → mem_tier_lock (new, mutex) for publication. Hot-path readers use rcu_read_lock() only. PEBS kthread acquires folio_lock() → lru_lock in standard mm order. No new lock inversions are introduced.

2.5 Implementation Plan

**Phase 1: FLC Core (weeks 1–4).** (a) Define struct flc in include/linux/memory-tiers.h with class_id, epoch, socket_local, switch_depth, interleave_width, latency_bucket fields. (b) Extend struct memory_tier with struct flc __rcu *flc pointer. (c) Implement flc_topology_update() in mm/memory-tiers.c: enumerate CXL regions via cxl_region_for_each(), read decoder geometry from struct cxl_decoder, parse CDAT DSLBIS entries via existing cxl_cdat_read_table(), compute class_ids, RCU-publish. 4. Risks & Mitigations

  • **Risk: SRD memory overhead for very large CXL regions.** A 4TB CXL region requires ~10MB for the SRD group array (5 bytes × 2M PMD-aligned groups). **Mitigation:** This is acceptable for server-class systems with hundreds of GB of DRAM. For extreme cases, the SRD group size can be increased from PMD (2MB) to PUD (1GB) alignment, reducing the array by 512×, at the cost of coarser order capping. A sysfs tunable (/sys/kernel/mm/flc/srd_group_shift) controls this trade-off.
  • **Risk: PEBS sampling rate may be insufficient for rapidly changing access patterns.** At 10K samples/s with a 100ms drain interval, the mailbox processes ~1000 samples per interval per CPU. For workloads with millions of active CXL folios, convergence to optimal placement may take tens of seconds. **Mitigation:** The scan interval and PEBS threshold are tunable. For latency-critical workloads, operators can increase PEBS frequency (accepting higher PMU overhead) or decrease the scan interval. The system degrades gracefully — slower convergence, not incorrect placement.
  • **Risk: FLC class_id ranking may not reflect actual performance for all workloads.** The deterministic ranking function prioritizes socket locality > switch depth > interleave width > latency, which may not be optimal for bandwidth-bound workloads that prefer wider interleave over lower latency. **Mitigation:** The bit-field positions in the class_id formula are chosen for the common latency-sensitive case. A future extension can add a sysfs-configurable weight vector, but the initial implementation targets the dominant use case. Operators can override tier ordering via existing memory_tier sysfs interfaces.
  • **Risk: Interaction with MGLRU and future mm changes.** MGLRU (Multi-Gen LRU) in mm/vmscan.c has its own folio aging and eviction logic that may conflict with FLC-directed demotion. **Mitigation:** The FLC order cap is applied at the folio allocation/migration boundary, below MGLRU's policy decisions. MGLRU decides *what* to demote; FLC decides *where* and *at what order*. The integration point (demote_folio_list) is shared by both classic LRU and MGLRU paths, ensuring uniform behavior.
  • **Risk: CDAT/HMAT data may be inaccurate or absent on early CXL platforms.** Some CXL 1.1 devices or prototype hardware may not provide CDAT tables, and BIOS HMAT entries may not reflect actual CXL latency. **Mitigation:** The three-tier fallback (CDAT → HMAT → static default of 150ns) ensures the system always has a latency estimate. The static default is conservative (higher than typical DDR latency, lower than worst-case CXL), biasing toward splitting when uncertain — a safe default that trades some TLB efficiency for stripe-correct placement. Boot-time dmesg warnings indicate which latency source was used.
  • **Risk: Security — PEBS mailbox as a side channel.** The per-CPU ring buffer contains PFNs and latency data that could theoretically leak information about memory access patterns. **Mitigation:** The mailbox is kernel-internal with no userspace-readable interface (only aggregate overflow counters via debugfs, not individual samples). The kthread runs in kernel context. No new attack surface is exposed beyond what perf_event_open with CAP_PERFMON already provides.
