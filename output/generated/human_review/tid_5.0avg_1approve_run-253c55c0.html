2. Technical Invention Disclosure

2.1 Problem Statement

Linux currently schedules KVM vCPU threads mostly as ordinary tasks using signals such as last_cpu, wake_affine history, generic cache-hotness, sched_migration_cost, NUMA-fault history, and topology packing. Those signals say little about whether the vCPU has just suffered x86 EPT disruption that will temporarily amplify second-level translation overhead, such as mmu_notifier-driven SPTE invalidation, TDP MMU root or leaf zaps, 2M/1G hugepage splits, dirty-logging write-protect bursts, EPTP changes, or remote-TLB/INVEPT epochs that invalidate paging-structure-cache and related walk locality. The missing signal matters in overcommitted Xeon-class hosts where a vCPU may be migrated across LLC or NUMA boundaries immediately after an EPT churn burst; the vCPU then pays both the usual CPU-cache cold-start cost and a short-lived translation re-warm penalty caused by rebuilding EPT walk locality, refilling TLB/VPID-associated reach, and re-establishing benefit from stable large mappings. The core systems problem is therefore not generic locality, but how to export just enough virtualization-specific, second-level-translation state into scheduler decision points without any scheduler hot-path dereference into KVM, without EPT walks, without trap-based sampling, without hidden O(nr_vcpus) scans on invalidation, and without creating a guest-controllable anti-migration lever.

2.2 Prior-Art Gap

The unresolved gap is a formally bounded contract between KVM x86 MMU internals and CFS placement logic. A practical solution must specify: the exact object that the scheduler reads; a race-free update/read scheme that needs only READ_ONCE/WRITE_ONCE and local barriers; a concrete event taxonomy distinguishing high-impact events from ambiguous low-impact churn; an attribution algorithm that is O(1) or amortized O(1) per invalidation event with no extra full-vCPU scans; a definition of impacted cohort that is precise for targeted events and conservatively bounded for VM-wide epochs; a decay/TTL/rate-limit function that makes the signal transient rather than sticky; cacheline-conscious placement so frequent MMU churn does not bounce task_struct lines across CPUs; and precise scheduler semantics showing that the hint can only alter tie-break ordering among already-legal CPUs. No known prior approach combines all of those properties for second-level translation churn: existing heuristics lack KVM cooperation, and existing KVM telemetry is not shaped for rq-locked scheduler consumption.

2.3 Proposed Invention

The invention introduces a Translation Debt Descriptor (TDD) mirrored into each KVM vCPU thread's task_struct under a static key such as CONFIG_SCHED_KVM_EPT_HINT. The mirrored object is a 16-byte structure stored entirely in task-local memory: seq (u32), debt_q8 (u16), stability_class (u8), flags (u8), source_epoch (u32), and expires_jiffies_or_ns_quanta (u32). KVM maintains the producer-side state in two places: a per-VM scheduler-facing MMU summary in struct kvm or struct kvm_arch containing vm_ept_epoch, vm_debt_bucket, anti_gaming_state, and event-rate token buckets; and a per-vCPU consumption state in struct kvm_vcpu_arch containing last_seen_vm_epoch, last_refresh_ts, last_guest_exit_ts, last_ept_root_gen, and optional direct-target pending bits. Event-to-debt mapping is explicit and saturating: INVEPT all-context or EPTP switch events add +96 debt units, single-context invalidation or memslot-wide zap epochs add +64, TDP leaf/root zap +48, 2M/1G split +48, dirty-log write-protect burst +24, and collapse/stabilization of large SPTEs subtract 32 while raising stability_class. Ambiguous low-impact events may contribute only +8 or be dropped entirely when below threshold. Global invalidations are handled in O(1) by incrementing vm_ept_epoch and updating vm_debt_bucket once; targeted invalidations reuse already-existing vCPU request masks or destination iteration and therefore add no asymptotic scan beyond the correctness path already required by KVM. Each vCPU lazily consumes unseen epoch deltas at existing safe boundaries such as vcpu_put/vcpu_load, vCPU wake/kick paths already touching that vCPU, block/unblock transitions, or KVM_RUN loop boundaries. Only then does the vCPU thread write its own task_struct mirror using WRITE_ONCE, so remote mmu_notifier CPUs do not thrash scheduler-read cachelines. Reader-side semantics are simple: the scheduler performs seq0=READ_ONCE(seq); if odd, treat as neutral; read payload with READ_ONCE; then read seq1 and accept only when seq0==seq1 and even, otherwise treat as neutral. This preserves a zero-dereference hot path into KVM. The hint is strictly transient: effective_debt decays with a 2 ms half-life, expires after 8 ms, is refreshed at most once per 1 ms per vCPU, and is capped so that the scheduler-visible bias never exceeds min(sysctl_sched_migration_cost_ns/8, 150000 ns equivalent). Anti-gaming protection is built in: per-VM epoch bumps are coalesced to one bump per 250 us window, per-vCPU visible debt is token-bucket limited, and if churn remains saturated for more than 32 ms the descriptor is forced to neutral for a 50 ms cooldown. Thread lifetime rules are explicit: the descriptor is zeroed on vCPU thread creation, reset on vCPU teardown, and reset again before any thread is rebound to a different struct kvm_vcpu so stale epochs cannot leak across reuse.

2.4 Architecture

The architecture has three planes. Plane 1 is event capture inside KVM x86 MMU code. Hooks are placed on mmu_notifier invalidate_range_start/end paths that cause SPTE invalidation or write-protect, TDP MMU zap root/leaf operations, hugepage split and collapse bookkeeping, EPTP changes, and remote-TLB/INVEPT-related flush epochs. Each hook classifies the event by expected near-term translation impact and either updates a per-VM epoch/debt accumulator or piggybacks on an already-targeted vCPU request path. Plane 2 is lazy attribution and mirroring. For VM-global events, KVM only updates vm_ept_epoch and vm_debt_bucket; the next time a given vCPU crosses an existing safe boundary, it compares last_seen_vm_epoch against vm_ept_epoch, computes delta debt, discounts it if the vCPU has been idle beyond an active window, assigns a stability_class based on whether the latest net event was disruptive or stabilizing, computes expires time, and mirrors the descriptor into current->task_struct using the seq protocol. For targeted events, the same mirror update may be done immediately when KVM already has the destination vCPU in hand, again without any new scan. Plane 3 is scheduler consumption. Only fair-class scheduling paths consult the descriptor, and only after normal policy pruning has produced a set of legal CPUs. In select_task_rq_fair(), the hint is considered in the wake_affine branch between prev_cpu and the alternative candidate; in select_idle_sibling(), it biases toward an otherwise-idle CPU in the same LLC or recent-translation domain as the vCPU's previous placement; and in the fair wakeup same-domain tie-break, it slightly penalizes cross-LLC choice when load/util differences are already below the cap. The hint never changes cpumask eligibility, never widens a domain, never affects periodic load balancing decisions that must restore convergence, never applies to RT/DL/stop/idle classes, and never changes mandatory migrations, cpuset enforcement, cgroup bandwidth rules, or uclamp min/max ordering. Because the mirrored task_struct field is read-mostly and only self-updated by the vCPU thread at bounded frequency, the design keeps cacheline bouncing low; the hot scheduler line carries only the 16-byte descriptor, while write-intensive counters stay in KVM-owned structures.

2.6 Validation Plan

Validation is split into correctness, complexity, performance, and abuse-resistance. For correctness, use KUnit or kernel selftests around the seq read/write helper, descriptor neutralization, vCPU create/exit/rebind reset, and bypass ordering so mandatory migrations, affinity masks, cpusets, cgroups, and uclamp outcomes are unchanged when the hint is present. For complexity, add tracepoints to count per-event work and prove that VM-global invalidations do not trigger new full-vCPU scans; measure that each invalidation updates only constant-size VM state, and that each vCPU consumes each epoch at most once. For performance, run KVM guests on Intel Xeon hosts with EPT enabled under workloads that generate THP split/collapse, dirty logging, memslot churn, live migration precopy, and host-memory invalidations; compare scheduler placement, cross-LLC migrations, vCPU wake-to-run latency, guest tail latency, host throughput, and PMU-derived page-walk indicators before and after the feature. Measure especially the short window after INVEPT-like bursts, where the mechanism should reduce harmful post-invalidation migrations without changing steady-state balancing. For scheduler safety, verify that the total scheduler-visible bias never exceeds the configured cap, that periodic load balancing still converges under imbalance, and that the worst-case extra non-mandatory migration prevented or induced is bounded to one bias opportunity per TTL window, i.e. at most 125 per second per vCPU with an 8 ms TTL, and lower in practice due to the 1 ms mirror refresh gate. For abuse-resistance, run hostile guests that alternate map/unmap, dirty-log toggles, and memslot churn to force chronic MMU events; verify token-bucket limiting, neutral cooldown activation, and absence of durable self-pinning or unfair isolation from competing tenants. For portability/fallback, confirm the feature auto-disables on non-x86 KVM, shadow paging, or non-EPT paths, and that ambiguous low-impact invalidations either map to negligible debt or no debt at all.

4. Risks & Mitigations

  • Risk: coarse VM-wide epochs may over-attribute debt to long-idle vCPUs. Mitigation: consume global epochs lazily with an active-window discount using last_guest_exit_ts, cap stale-vCPU debt to a fraction of the normal maximum, and expire the descriptor after 8 ms.
  • Risk: frequent remote MMU events could bounce task_struct cachelines. Mitigation: remote invalidation paths update only per-VM counters; task_struct is written primarily by the owning vCPU thread at bounded refresh intervals, and the mirrored descriptor is kept to 16 bytes with aligned placement.
  • Risk: guest-triggered churn could become a host scheduling DoS or anti-migration lever. Mitigation: coalesce epoch bumps to one per 250 us, cap per-vCPU visible debt, enforce per-VM token buckets, and neutralize the hint for a cooldown period when saturation persists.
  • Risk: overlap with existing heuristics could make the feature redundant or destabilizing. Mitigation: restrict use to second-level translation churn not observable from ordinary scheduler history, apply only as a smaller-than-migration-cost tie-break, and bypass all convergence-critical and mandatory-policy paths.
  • Risk: event taxonomy may misclassify low-impact invalidations. Mitigation: start with a conservative allowlist of high-correlation events, treat ambiguous events as low weight or neutral, and expose tracepoints for empirical tuning on Xeon hosts.
  • Risk: stale metadata could leak across vCPU thread exit or reuse. Mitigation: zero the descriptor and reset source_epoch on create, teardown, and rebinding, and require sequence-valid even reads so partially updated state is treated as neutral.
  • Risk: portability concerns outside x86 EPT/TDP MMU. Mitigation: gate the feature behind x86 KVM EPT capability and disable it for shadow paging, non-EPT virtualization, or architectures lacking equivalent invalidation semantics.
