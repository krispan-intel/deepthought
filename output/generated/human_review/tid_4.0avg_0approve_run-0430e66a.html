1. Executive Summary

One-Liner

For iomap-backed buffered reads and mmap faults in a TDX guest, Linux inserts a locked large-folio placeholder into the page cache, accepts each 4 KiB private subpage before any guest touch, fetches bytes from an untrusted auxiliary cache source into shared bounce buffers at fs-verity-block granularity, verifies each block against a ticket cryptographically bound to inode identity plus mapping generation, copies only verified blocks into the accepted folio, and marks the folio Uptodate only after full-range accept, verify, and materialization complete under final invalidate-lock revalidation.

2. Technical Invention Disclosure

2.1 Problem Statement

In a TDX guest, guest-private pages must be accepted before any architecturally visible guest access. Linux file reads and file-backed faults populate the page cache in private memory, while fs-verity and immutable-image deployments increasingly want to reuse bytes from a host-visible or otherwise untrusted auxiliary cache. Naively combining these ideas is unsafe for several independent reasons: a large folio may span EOF, a readahead edge, multiple iomap extent classes, or multiple verity blocks; xarray insertion used for duplicate suppression is not the same thing as publishing usable data; copying untrusted bytes directly into a large private folio before cryptographic verification blurs the boundary between filled and verified; truncation, hole punch, overlayfs copy-up, reflink/COW remap, and fscrypt state changes can invalidate the file identity while a population worker is in flight; and incidental kernel touches such as allocator zeroing, poisoning, debug instrumentation, or memcpy into an unaccepted private page can fault or violate the intended security invariant. The technical problem is therefore to define a Linux-correct producer-side state machine in which a large page-cache folio can be allocated and inserted early for duplicate suppression, but no byte becomes user-visible unless the current inode identity is still valid, every 4 KiB private subpage has been accepted, every fs-verity verification unit covering the effective byte range has verified exactly over the correct length, and the folio is either fully published or fully aborted with no stale attachment left in the mapping.

2.2 Prior-Art Gap

Prior art covers large folios, iomap extent walking, FS-Cache and netfs validation pipelines, fs-verity Merkle-tree verification, overlayfs lower-layer immutability, and x86 TDX page acceptance, but it does not teach the combined rule set required here. Missing from the art is: a backend-agnostic untrusted auxiliary-cache interface whose correctness does not depend on the backend; a concrete on-wire ticket bound to inode identity, verity descriptor digest, block index, valid length, and freshness fields; a distinction between xarray placeholder insertion and actual data publication; explicit per-subpage accept state and per-verity-block verify state on a large folio; an accept-before-any-private-touch ordering that enumerates the touchpoints that must be suppressed or reordered; a precise invalidate, truncate, hole-punch, reflink, and copy-up concurrency model; and an explicit folio-sizing algorithm that refuses any size not exactly representable as a set of full verification units plus at most one well-defined terminal partial block. Generic verify-after-fetch designs may validate data eventually, and generic large-folio page-cache designs may reduce TLB pressure, but neither defines the ticket-bound state machine that allows both properties simultaneously under TDX.

4. Risks & Mitigations

  • Risk: Holding filemap_invalidate_lock_shared() across a cold large-folio fill could increase latency for truncate or invalidation. Mitigation: bound the folio size to at most 2 MiB, use the path only for cold misses on immutable content, and keep immediate fallback on backend delay or verification stalls.
  • Risk: Some kernels or debug configurations may touch folio payload bytes before the explicit TDX acceptance point. Mitigation: gate the path on configurations that can suppress pre-accept payload touches, and otherwise fall back to the baseline verified read path.
  • Risk: Overlayfs, reflink, and fscrypt semantics differ across filesystems. Mitigation: make the auxiliary path opt-in behind a stable_mapping_generation callback and keep the default eligibility gate conservative.
  • Risk: Prior art may argue that this is only a combination of FS-Cache plus fs-verity plus large folios. Mitigation: emphasize the three-unit transfer, verification, and publication split; the ticket-bound inode identity model; and the explicit accept-before-touch plus locked-placeholder publication protocol.

⚠ Fatal Flaw: The design relies on a “mapping invalidation generation / mapping_gen” concept for revalidation, but Linux does not provide a single, universally correct generation counter for all truncate/punch-hole/invalidate/reflink/stacked-fs cases; without a concrete integration point (e.g., invalidate_lock + precise recheck rules), the proposed publication/revalidation step is underspecified and can be racy.

  • Define the exact locking and revalidation contract against truncation/invalidation: which lock(s) are held during final recheck (mapping->invalidate_lock? i_rwsem? folio_lock only?), what state is rechecked (i_size, index still valid, no invalidation in range), and how you guarantee that once Uptodate is set, no concurrent truncate/punch can have made the data stale. “mapping_gen” must map to a real kernel mechanism or be replaced by a correct scheme.
  • Clarify the accept-before-touch invariant end-to-end: folio allocation paths commonly trigger implicit touches (debug pagealloc poisoning, KASAN, memcg charging paths, speculative prefetch/zeroing). You must specify a safe allocation/accept sequence that (a) never touches unaccepted private memory, (b) still satisfies MM invariants, and (c) has a hard fallback/disable knob when those invariants cannot be met on a given config.
