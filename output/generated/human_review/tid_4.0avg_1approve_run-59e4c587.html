2. Technical Invention Disclosure

2.1 Problem Statement

Intel VMX and APIC virtualization let a Linux KVM guest accept interrupts without frequent exits, but several Linux-visible interrupt sources still need a host-side completion point with level semantics. For IOAPIC level pins, irqfd or VFIO INTx resamplers, and other EOI-sensitive routes, completing too early can deassert or resample a source before the guest has actually retired the in-service vector, while completing too late can stall a still-asserted device or trap the system into unnecessary exits. Existing hardware features expose ingredients such as posted interrupts, virtual-APIC register virtualization, x2APIC virtualization, and optional EOI-induced exits, but they do not define a single end-to-end rule for when Linux may fire a source-specific completion action that remains correct across xAPIC, x2APIC, APICv enable/disable transitions, nested VMX, shared vectors, route updates, vCPU migration, or endpoint teardown. The disclosure solves that by tying completion to a host-readable, mode-qualified proof of ISR retirement at a defined VMX coherence boundary and by making that proof usable only together with a delivery-instance ledger, source-lifetime pinning, and transition-safe drain rules.

2.2 Prior-Art Gap

Prior art covers virtual-APIC pages, posted-interrupt descriptors, EOI-exit bitmaps, APIC-access traps, x2APIC MSR virtualization, route updates under SRCU, and generic interrupt coalescing. What is absent is a concrete correctness protocol that simultaneously answers seven questions: which exact delivery instance is awaiting completion, which host-readable state is authoritative for proving that the guest-visible ISR bit for that instance has cleared, which host boundaries are coherent enough to read that state, which Linux completion side effect is legal for the source class involved, which route and endpoint metadata must still match before committing, which loaded-VMCS and nested-VMX invariants must hold so the read belongs to the intended guest context, and how APICv mode transitions drain PIR, IRR, ISR, notification, and pending requests without lost or duplicate delivery. The gap is therefore not another APIC feature or another tuning heuristic, but an exact-once host commit discipline spanning KVM interrupt delivery, APIC state observation, route lifetime, and VMX mode transitions.

2.3 Proposed Invention

The invention introduces a per-vCPU APIC completion ledger plus a mode-qualified reconciliation engine in KVM. When KVM injects or posts a level-semantic interrupt that may require a later Linux-visible completion side effect, the delivery path allocates a delivery-instance record and attempts to arm it as the sole owner of the destination vCPU/vector slot. The record key is {source_cookie, destination_vcpu_id, vector, delivery_seq}; the stored validation tuple additionally includes {route_gen, source_gen, trigger_mode, dest_mode, apic_mode_epoch, nested_level, completion_kind}. source_cookie is a stable source identity obtained from the endpoint that will later receive completion, such as an IOAPIC pin cookie or an irqfd/VFIO INTx resampler cookie. delivery_seq is a monotonic 64-bit per-source sequence that never reuses an in-flight value; wrap is handled by refusing new arm operations if the next value would alias an outstanding record, which in practice requires quiescing and draining long before arithmetic wrap. Each destination vCPU owns (i) vector_owner[256], an array of owner tokens published with release semantics, (ii) vector_waitq[256], a bounded FIFO or intrusive list for same-vector contenders, and (iii) an xarray or rhashtable of live delivery records keyed by owner token. At most one record may be ARMED for a given destination vCPU/vector at a time. If the same source re-asserts while its record is already armed, the delivery is coalesced onto that record; if a different source targets the same vector while another owner is armed, the new record is parked in the vector wait queue and cannot commit until it is later promoted to owner by a new delivery or a retry after the current owner retires. Reconciliation runs only in host VMX code, reads an authoritative ISR-clear proof for the specific APIC mode and nested level, revalidates route and endpoint lifetime under SRCU and stable references, and then performs exactly one source-specific completion side effect by atomically changing the record state from ARMED to COMMITTED. If any proof or validation fails, the record remains pending, is retried, or is retired without side effect according to explicit mismatch rules.

2.4 Architecture

The architecture is a four-stage data path with explicit coherence boundaries.

ASCII overview:

[Source endpoint]
    |  (IOAPIC pin / irqfd resampler / VFIO INTx)
    v
[Delivery classifier]
    - derives source_cookie
    - tags completion_kind
    - snapshots route_gen/source_gen/trigger mode/dest mode
    v
[Per-vCPU ledger]
    vector_owner[vector] -> owner_token
    live_records[owner_token] = {state, source_cookie, route tuple, mode epoch, nested level}
    vector_waitq[vector] -> queued contenders for shared vectors
    v
[Guest delivery path]
    - software IRR injection, APICv virtual delivery, or posted interrupt
    v
==== VMX coherence boundary ====
    VM exit OR pre-entry slow path while intended loaded VMCS is current
    v
[Authoritative ISR-clear observer]
    - non-APICv/xAPIC or x2APIC emulation: KVM LAPIC shadow ISR after exit
    - APICv virtual-APIC mode: vAPIC page or synchronized KVM shadow after VM-exit serialization
    - nested L2: observe the state corresponding to vmcs02-owned guest APIC context before switching back to vmcs01
    v
[SRCU + lifetime revalidation]
    - compare route_gen/source_gen/trigger/dest/mask
    - confirm endpoint ref still alive and not DYING
    v
[Commit map]
    - IOAPIC level -> EOI notifier path
    - irqfd/VFIO INTx -> resample callback/eventfd signal
    - MSI/MSI-X -> no completion side effect

The observer is intentionally decoupled from policy. A selective EOI-exit bitmap, posted-interrupt notification, or preemption timer may mark a candidate for early reevaluation, but no such event by itself proves retirement. The proof is always the mode-qualified ISR-cleared observation plus ledger ownership and revalidation.

2.5 Implementation Plan

1. Extend KVM interrupt routing for completion-capable sources so each eligible route exposes a stable source_cookie, source_gen, completion_kind, and ref/notifier hooks. 2. Add per-vCPU ledger state: vector_owner[256], vector_waitq[256], live record store, apic_mode_epoch, and transition bookkeeping for posted interrupts. 3. Instrument the delivery path for IOAPIC level and irqfd/VFIO INTx sources so arming a record happens before delivery publication and coalesces or queues correctly on shared vectors. 4. Add host-side reconciliation hooks in the post-VM-exit path and optional pre-entry slow path, with strict predicates that the intended VMCS context is current and that the record's nested_level and apic_mode_epoch match. 5. Implement authoritative ISR observers for software LAPIC, APICv virtual-APIC, and nested L1/L2 contexts, including any required synchronization from the vAPIC page into a KVM shadow before reading. 6. Implement SRCU revalidation and stable-reference retirement rules, then the source-specific completion map for IOAPIC level and irqfd/VFIO INTx. 7. Implement APICv transition handling that quiesces posted interrupts, performs PIR-to-IRR synchronization, rewrites controls, rechecks for arrivals during transition, and retries when necessary. 8. Add diagnostics tracepoints for record creation, proof success/failure, commit, retire, route mismatch, and transition retries; these support validation but are not required by the claims.

2.6 Validation Plan

Validation covers correctness first and performance second. Correctness tests include: (a) single-vCPU IOAPIC level interrupt retirement with and without selective EOI exits; (b) same-vector shared-source scenarios proving that only the owner_token's source receives completion; (c) concurrent route changes, masking, and teardown while armed records exist, verifying no use-after-free and no completion against stale route generations; (d) nested VMX tests with L2-targeted interrupts proving that vmcs02-scoped records do not commit from vmcs01-visible state; (e) APICv enable/disable, posted-interrupt enable/disable, x2APIC virtualization toggles, and EOI-bitmap updates under load, verifying no lost or double-injected interrupts and correct PIR -> IRR transfer ordering; and (f) migration or CPU-preemption scenarios showing commit executes only with the intended loaded VMCS current. Performance tests measure reduced always-on exit pressure versus unconditional EOI trapping on representative Linux workloads, including interrupt-heavy network/storage virtualization and host-resampler workloads, while optional PMU or trace data are used only to quantify benefits rather than define correctness.

4. Risks & Mitigations

  • Risk: stale or incoherent reads of hardware-updated APIC state. Mitigation: bind every proof read to a VMX coherence boundary and, for APICv, require explicit synchronization from the vAPIC page to a host shadow when direct exit-time coherence is not guaranteed.
  • Risk: double completion on shared vectors or repeated exits. Mitigation: exactly-once gating through owner_token plus atomic ARMED -> COMMITTING transition and queued contenders for same-vector nonowners.
  • Risk: use-after-free or semantic misdirection after route churn. Mitigation: stable endpoint references or teardown notifiers plus route_gen and source_gen comparison under SRCU before commit.
  • Risk: lost or duplicate interrupts during APICv mode changes. Mitigation: transition_in_progress gating, suppress-notification or equivalent quiesce, mandatory PIR -> IRR synchronization, barriers around control rewrites, and retry on arrivals during transition.
  • Risk: nested VMX commits against the wrong guest context. Mitigation: per-record nested_level and vmcs_epoch checks, with vmcs12 excluded as an authoritative proof source.
  • Risk: unbounded completion latency if no helpful exits occur. Mitigation: allow optional selective EOI-exit trapping, timer-based reevaluation, or forced exits as latency policy accelerators without changing the correctness criterion.
