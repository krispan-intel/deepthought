2. Technical Invention Disclosure

2.1 Problem Statement

On x86 servers with CXL Type-3 memory, Linux package-idle selection and package-power mitigation are largely driven by CPU-local or package-local signals, while important CXL activity is discovered asynchronously in probe, event-log servicing, poison list retrieval/clear, sanitize, health polling, region reconfiguration, reset recovery, and hotplug workqueues. This creates three concrete gaps. 1. `intel_idle` / cpuidle cannot cheaply determine whether a package-local CXL host path is in a fresh non-quiescent interval without calling into CXL code or taking sleeping locks, both of which are unacceptable in the idle path. 2. When a package is actively power-limited, background CXL mailbox traffic continues to consume package power even though much of that traffic is best-effort maintenance rather than urgent RAS or recovery. 3. Existing kernel mechanisms do not provide a topology-safe way to bind a given CXL memdev path to exactly one x86 package only when the mapping is trustworthy, nor a rule for what happens when the path is behind a switch, a shared I/O die, a virtualized root complex, or ambiguous ACPI proximity. The practical failure case is a multi-socket platform where a Type-3 memdev hangs from the root complex nearest package 1. A background poison-list walk or event-log refresh on that path may keep consuming package 1 power during PL1/PL2 assertion while `intel_idle` on package 1 has no lockless evidence that the domain is active. Simultaneously, socket 0 should not be penalized, and urgent CXL RAS traffic must still progress. The invention solves precisely that scoped problem.

2.2 Prior-Art Gap

The topological void is the absence of a per-host-domain kernel contract that spans CXL producer state, x86 package power-pressure, and package-scope idle arbitration. 1. Contrast to `powercap` / Intel RAPL control loops: RAPL can program or observe package power limits, but it cannot identify which serialized CXL mailbox command on which root-complex path is safe to defer. Failure case: package 1 enters active power limitation while an event-log scan and poison-list walk are queued on one CXL mailbox; RAPL may lower available power, but it cannot express 'defer only the best-effort scan for this host domain while still allowing fatal error recovery.' 2. Contrast to `pm_qos` and cpuidle governors: latency QoS is global or per-CPU policy, not a root-complex-scoped transient derived from asynchronous CXL activity. Failure case: using `pm_qos` to protect one CXL path would over-constrain unrelated CPUs or both sockets, and governors still lack a lockless producer snapshot tied to that host domain. 3. Contrast to workqueue power-efficient mode and freezer behavior: those mechanisms alter worker placement or suspend semantics, not per-mailbox class arbitration under package power pressure. Failure case: power-efficient workqueues still execute best-effort CXL maintenance during PL1 assertion, and freezer semantics are suspend-oriented rather than domain-local power arbitration. 4. Contrast to runtime PM autosuspend: runtime PM reasons about device power state, not class-aware serialization of mailbox traffic while the CXL memory device remains online and serving memory. Failure case: a Type-3 memory device backing active regions cannot be autosuspended, yet the kernel still needs to defer only low-value mailbox maintenance during package power pressure. 5. Contrast to generic mailbox prioritization: existing serialized mailbox locking does not incorporate root-complex->package binding, cached package power-limit-active state, or claimable fairness guarantees for urgent versus deferred maintenance. The non-obviousness therefore lies in combining a root-complex-anchored topology binding, a cached and epoch-stamped package power-pressure signal, a lockless self-expiring activity snapshot for package-idle readers, and a bounded-deferral mailbox scheduler with explicit ambiguity fallback.

2.3 Proposed Invention

The invention introduces a new in-kernel object, `struct cxl_host_power_domain`, and a narrow contract surface between the CXL core and x86 package power/idle logic. A. Exact contract surface - Domain identity: `segment`, `root_bus`, `host_bridge_uid`, and `root_port_bdf` (or host-bridge key for RCiEP paths) form the stable host-domain key. - Binding result: `package_id`, `package_cpumask`, `bind_confidence` in `{STRONG, WEAK, AMBIGUOUS, UNKNOWN}`, and `topology_epoch`. - Cached snapshot fields published under seqcount: `active`, `active_expires_ns`, `package_power_limit_active`, `power_sample_ns`, `power_epoch`, `valid`, and `bind_confidence`. - Scheduling API exported to CXL producers: `enqueue_best_effort(cmd, deadline_ns)` and `enqueue_urgent(cmd, deadline_ns)`. - Read API exported to x86 idle logic: `cxl_hpd_read_snapshot(domain, &snap)` returning a lockless copy suitable for `intel_idle` package-state pruning. B. Deterministic topology binding algorithm 1. For each CXL memdev, walk PCI ancestry to the nearest upstream root port; if the device is RCiEP or the path lacks a root port, use the owning PCI host bridge as the anchor. 2. If the memdev is behind a CXL switch, all downstream ports that converge on the same upstream root port / host bridge share the same host-power-domain key. 3. Resolve candidate package ownership in order: (i) host bridge or root-port ACPI `_PXM`; (ii) SRAT/HMAT proximity of the attached memory target intersected with online CPU packages; (iii) unique package inferred from the online CPUs associated with that root-complex locality. 4. If the resolution yields exactly one online package, set `bind_confidence=STRONG` and bind. If multiple packages remain plausible, if a shared I/O die makes ownership non-unique, if virtualization hides trustworthy locality, or if ACPI proximity is missing/contradictory, set `bind_confidence=AMBIGUOUS` or `UNKNOWN` and do not enable package-coupled idle gating or package-power-driven deferral. 5. CPU hotplug, ACPI reconfiguration, PCI removal, or reprobe increments `topology_epoch`, RCU-swaps the domain pointer on affected memdevs, and re-runs the binding algorithm. C. Package-power-pressure backend - The backend samples the package-level power-limitation status in `IA32_PACKAGE_THERM_STATUS` on one housekeeping CPU selected from the bound `package_cpumask` (lowest-numbered online housekeeping CPU for that package). - Validity is correlated with enabled package power limits from `MSR_PKG_POWER_LIMIT`; if package limits are disabled or the package is unbound, `valid=false`. - Sampling is event-driven: delayed work is armed only while a domain has deferred best-effort work, while `package_power_limit_active` remains asserted, or immediately after a package-limit reprogramming event. There is no always-on periodic polling and no wakeup on isolated CPUs. - Sampling cadence is capped at once every 2 ms while active. Clearing `package_power_limit_active` requires two consecutive inactive samples separated by at least 500 us to avoid oscillation livelock. Assertion is immediate on the first active sample. - Staleness rule: if `now - power_sample_ns > 4 ms`, the cached power signal is treated as invalid and maintenance deferral fails open, i.e. best-effort work is no longer deferred solely on that stale sample. D. Lockless package-idle gating - CXL producers increment an atomic domain-active counter on entry to mailbox-using or topology-mutating work and decrement it on exit. - On the first increment from zero, the writer publishes `active=true` and `active_expires_ns=now + hold_ns`, where `hold_ns` is a conservative class constant (for example 250 us for mailbox follow-up and 2 ms for reset-recovery work), not a computed device-specific exit-latency budget. - While activity continues, writers refresh `active_expires_ns`; when the counter reaches zero, writers publish `active=false`. - `intel_idle` only consults this snapshot for candidate package-scope idle states. If `bind_confidence!=STRONG`, package-coupled gating is disabled. If `active=true`, or if a previously asserted active snapshot has gone stale without a clearing update, the package-deep state is pruned for that bound package only. Core-only idle states are untouched. E. Mailbox arbiter and fairness guarantees - All mailbox-using CXL maintenance calls are funneled through a per-domain scheduler layered above the per-device mailbox mutex. - Urgent lane: fatal event handling, recovery-critical state fetches, and RAS-required commands use `enqueue_urgent()` and bypass queued best-effort work. - Best-effort lane: health refresh, background event-log scan, background poison-list walk, and similar maintenance use `enqueue_best_effort()` with explicit deadlines. Default class bounds are 10 ms for short recovery-follow-up metadata refresh, 25 ms for poison/background scrub maintenance, and 100 ms for telemetry/event scanning. - Dispatch order is `urgent > aged_best_effort > best_effort`, where a best-effort command whose deadline expires is promoted to `aged_best_effort`. - Quantitative invariant: a newly arrived urgent command never sits behind queued best-effort commands and can be delayed only by the single mailbox command already in flight. When package pressure is inactive and best-effort work exists, the arbiter grants at least one best-effort command or 2 ms of best-effort service in every 10 ms clear window. This gives a claimable starvation bound without sacrificing urgent priority. F. Context and lifecycle rules - `cxl_mbox_send_cmd()` / `cxl_internal_send_cmd()` remain callable only from sleepable process or workqueue context. Submission sites reached from notifier, AER, or other atomic contexts may only queue work into the urgent lane. - The mailbox scheduler enforces `WARN_ON_ONCE(in_atomic() || irqs_disabled())` on direct submission paths. - Device removal and suspend freeze the domain, call `cancel_delayed_work_sync()` on power-refresh and drain workers, RCU-detach memdev->domain pointers, wait for outstanding refs, and only then free the domain. Resume and reprobe rebuild the mapping and increment `topology_epoch`.

2.4 Architecture

The architecture is a three-bridge path: `CXL producers -> cxl_host_power_domain -> x86 package idle/power consumers`.

1. Producer side
- CXL call paths that ultimately funnel into `cxl_mbox_send_cmd()` or `cxl_internal_send_cmd()` are classified as urgent or best-effort before they contend on the mailbox mutex.
- Representative sites include identify/init refresh, event-record retrieval, poison list get/clear, health/status queries, sanitize progress, and reset recovery.
- Atomic/notifier call sites never touch the mailbox directly; they raise a bit and queue urgent work.

2. Host-power-domain core
- One `cxl_host_power_domain` exists per unique root-complex/host-bridge key.
- Data members: identity tuple, bound package metadata, `seqcount_t snapshot_seq`, atomic active counter, `power_epoch`, `topology_epoch`, `refcount_t`, RCU-protected memdev links, `delayed_work power_refresh_work`, `delayed_work be_drain_work`, and scheduler queues.
- Lock ordering is fixed: `bind_mutex -> sched_lock -> (drop sched_lock) -> device mailbox mutex`. The mailbox mutex is never held while taking `bind_mutex`, and seqcount write-side publication happens while holding only the scheduler state lock.

3. x86 package consumer side
- A per-package backend chooses one housekeeping CPU to read package power-limitation status and publish `package_power_limit_active` plus `power_epoch` into all strongly bound domains for that package.
- `intel_idle` performs a seqcount read of the snapshot, checks `bind_confidence`, and prunes only candidate package-scope states for the affected package.

4. Failure semantics
- Strong bind + fresh active snapshot -> prune package-deep idle.
- Strong bind + fresh power-limit-active snapshot -> defer best-effort mailbox work until deadline or clear window.
- Stale or invalid power signal -> fail open for maintenance deferral.
- Ambiguous/untrusted topology -> disable package-coupled gating and package-coupled power deferral; local urgent-vs-best-effort queueing may still be used, but no package-specific idle or power claim is made.

5. Suspend/hotplug/reconfiguration
- On suspend or PCI removal, the domain enters `FROZEN`, queued delayed work is synchronously cancelled, memdev links are RCU-cleared, and any pending best-effort queue is either drained before suspend or replayed after resume according to command class.
- On CPU hotplug or ACPI topology change, the package housekeeping CPU and `package_cpumask` are recalculated under `bind_mutex`, `topology_epoch` increments, and the bind confidence may degrade to `AMBIGUOUS`, which immediately disables idle gating and package-pressure deferral.

2.5 Implementation Plan

Phase 1 - Contract and domain object - Add `drivers/cxl/core/host_power_domain.c` and a matching internal header that defines `struct cxl_host_power_domain`, snapshot fields, bind-confidence enum, and the APIs `enqueue_best_effort(cmd, deadline_ns)`, `enqueue_urgent(cmd, deadline_ns)`, and `cxl_hpd_read_snapshot()`. - Add RCU/refcount lifetime handling and a per-domain seqcount snapshot. Phase 2 - Topology binding - During memdev discovery and reprobe, walk PCI ancestry to determine the root port / host bridge anchor. - Resolve package ownership using `_PXM`, SRAT/HMAT, and online-package intersection in the exact priority order described above. - Store `bind_confidence`, `package_id`, `package_cpumask`, and `topology_epoch`. Phase 3 - Package power backend - Add an x86 helper that schedules per-package delayed work on a housekeeping CPU to read `IA32_PACKAGE_THERM_STATUS` and correlate validity with `MSR_PKG_POWER_LIMIT` enable state. - Publish `package_power_limit_active`, `power_sample_ns`, and `power_epoch` into strongly bound domains; implement the 2 ms cadence, 500 us clear debounce, and 4 ms staleness rule. Phase 4 - Mailbox scheduler interposition - Funnel current CXL mailbox submission sites through the domain scheduler above the per-device mailbox mutex. - Audit and classify the current major mailbox users: identify/init refresh, event-log retrieval, poison get/clear, health/status, sanitize, and reset recovery. - Add direct-path assertions so any process/workqueue caller may submit, but atomic/notifier/AER callers must queue urgent work only. Phase 5 - Idle integration - Add a read-only helper callable from `intel_idle` package-state selection to fetch the seqcount snapshot for the package-local domains. - Only package-scope candidate states are pruned; core-only states and unrelated sockets remain untouched. Phase 6 - Lifecycle hardening - Implement suspend, resume, PCI removal, and CPU hotplug handling with `cancel_delayed_work_sync()`, RCU pointer swaps, refcount drains, and `topology_epoch` increments. - Add tracepoints for domain binding changes, power-pressure transitions, deadline promotion, urgent bypass, and idle-state pruning decisions.

4. Risks & Mitigations

  • Risk: topology false positives on shared I/O dies or virtualized CXL paths. Mitigation: require unique-package resolution for `STRONG` binding and fall back to `UNKNOWN` otherwise.
  • Risk: power-limit signal oscillation causes repeated defer/run transitions. Mitigation: immediate assert, debounced clear using two inactive samples separated by at least 500 us, and epoch tracking.
  • Risk: urgent storms starve best-effort maintenance. Mitigation: deadline promotion to aged-best-effort plus a guaranteed best-effort slice after pressure clears.
  • Risk: direct mailbox submission from atomic context deadlocks or sleeps illegally. Mitigation: queue-only atomic shims and `WARN_ON_ONCE(in_atomic() || irqs_disabled())` at the direct submission funnel.
  • Risk: hot-remove or suspend races with queued delayed work. Mitigation: `cancel_delayed_work_sync()`, RCU pointer swaps, refcount drains, and topology epochs before free.
  • Risk: idle-path regressions from over-broad gating. Mitigation: consult the snapshot only for package-scope idle candidates and only for domains with `STRONG` binding confidence.
