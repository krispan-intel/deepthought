2. Technical Invention Disclosure

2.1 Problem Statement

Heavy AVX2 and especially AVX-512 regions on Intel x86 can trigger hardware-managed frequency ratio limits that outlive the last vector instruction. Linux power management today largely reacts to demand rather than to an explicit statement from the hardware that a lower ceiling is temporarily in force. As a result, `schedutil` can keep requesting a turbo frequency that the hardware will not honor, especially when a scalar task wakes on the SMT sibling of a core that just executed AVX-512. The result is request overshoot, oscillation between requested and attainable frequency, wasted package power, and unfair latency for scalar follow-on work. The scope of the hardware limit is also not per task and not reliably per logical CPU. On current Intel server parts the AVX license or attenuation effect is at least shared by the physical core across SMT siblings, and future telemetry sources may report a wider coupling domain such as die or package. A correct kernel design therefore cannot infer pressure from task state, PMU sampling, or XSTATE ownership alone. A second problem appears in cpuidle. Immediately after a clipped AVX-512 phase, the core may re-enter idle just before scalar cleanup or sibling work returns. Existing idle governors may pick a deep C-state because they see only generic idle-history signals, not a recent vector-induced cooldown shared by the physical core. That creates avoidable exit-latency penalties at exactly the moment the core is recovering from a hardware-enforced vector downclock event.

2.2 Prior-Art Gap

Intel optimization manuals describe that AVX and AVX-512 execution can reduce attainable frequency, and some Intel parts expose raw `perf_limit_reasons` or `MSR_CORE_PERF_LIMIT_REASONS` style diagnostics. That is still not a Linux control abstraction. Those interfaces are vendor-specific, expose raw reason bits rather than a scheduler-unit scalar, and do not define how the signal should be keyed to physical cores, aggregated to `cpufreq` policies, shaped with hysteresis and cooldown, or consumed without double counting. `intel_pstate` diagnostic plumbing may surface the bits, but it does not create a generic kernel signal with explicit domain semantics, deterministic decay, or a formal contract for cross-subsystem consumption. Existing Linux scheduler mechanisms also miss the target. PELT utilization, `util_est`, `uclamp`, and frequency invariance model runnable demand; thermal pressure models a generic capacity loss; none of them represents a hardware-declared AVX-specific ratio cap. PMU- or task-based inference is both noisy and semantically wrong because the hardware restriction is shared and can persist after the causative instructions retire. On the idle side, `menu` bases prediction on observed idle durations and interactivity correction, `TEO` ranks states from timer-event hit and miss history, and `intel_idle` demotion tables are static heuristics. None uses an explicit current-status AVX limit bit, none keys cooldown to the shared physical-core domain, and none applies a bounded ranking-only bias after PM QoS, target residency, exit latency, and next-timer checks. The gap is therefore a topology-correct, formally bounded, vendor-neutral kernel abstraction for explicit AVX ratio limiting that can safely feed both `schedutil` and cpuidle without changing existing eligibility semantics.

2.3 Proposed Invention

The invention adds a generic kernel abstraction named here a vector-limit provider, with an x86 Intel implementation. The provider is enabled only on models where two conditions are true at boot: `(1)` a current-status telemetry source exists for active core frequency limiting due to vector-license or transition attenuation, using `MSR_CORE_PERF_LIMIT_REASONS` or the exact model-specific equivalent already used by Intel `perf_limit_reasons` reporting, and `(2)` the platform exposes the corresponding scalar, AVX2, and AVX-512 ratio ceilings through CPUID-model tables or model-specific ratio-offset registers. Only the current-status half of the telemetry is used for control; sticky logged bits are ignored because they indicate historical rather than current limiting. If the model cannot document both semantics and scope, the feature is disabled rather than guessed. The provider computes a stable sharing key for each online CPU as `(scope, package_id, die_id, cluster_id, core_type, core_id, policy_cookie)`. For present AVX license telemetry the default `scope` is `CORE`, so both SMT siblings share one domain object. If a future Intel source explicitly reports a wider coupling scope, the implementation can select `DIE`, `PACKAGE`, or `POLICY`; if scope is unknown, no domain object is created. Domain membership is rebuilt on CPU hotplug and on `cpufreq` policy rebuild under `cpus_read_lock()` plus a provider mutex, and the resulting tables are published to readers with RCU. A `cpufreq` policy stores an RCU pointer to the set of intersecting domains, and policy aggregation is strictly `max(domain_pressure)` because a shared policy cannot satisfy a request above its most constrained member. Scalar math is explicit and testable. Let `Rscalar` be the boot-time scalar ratio baseline for the policy, and let `Ravx2` and `Ravx512` be the corresponding AVX-capped ratios. At each refresh, the telemetry selects `Ractive` from `{Rscalar, Ravx2, Ravx512}` according to whether no vector limit, AVX2 limit, or AVX-512 limit is currently asserted. The instantaneous normalized severity is `raw_q10 = clamp(((Rscalar - Ractive) * 1024) / max(1, Rscalar), 0, 1024)`. The exported scheduler-unit pressure is `pressure = DIV_ROUND_CLOSEST(raw_q10_eff * SCHED_CAPACITY_SCALE, 1024)`. A per-domain finite-state machine shapes oscillatory telemetry into a bounded signal. Parameters are explicit: `N_on = 2`, `N_off = 2`, `sample_period_ns = max(250000, transition_delay_ns / 4)`, `min_clip_ns = 200000`, `min_cooldown_ns = 200000`, and `max_cooldown_ns = 2000000`. On an active sample, the state enters `CLIPPED`, `peak_q10 = max(peak_q10, raw_q10)`, `cooldown_span_ns = clamp(min_cooldown_ns + peak_q10 * 1800000 / 1024, min_cooldown_ns, max_cooldown_ns)`, and `cooldown_until_ns = now + min_clip_ns + cooldown_span_ns`. After `N_off` inactive samples, the state becomes `COOLDOWN`, and the effective severity is `raw_q10_eff(now) = DIV_ROUND_UP(peak_q10 * max(cooldown_until_ns - now, 0), cooldown_span_ns)`. If no new active telemetry arrives, `cooldown_until_ns` is never extended, `max(cooldown_until_ns - now, 0)` decreases monotonically under monotonic time, and therefore the exported pressure cannot be prolonged by scheduler or governor behavior. Only new active telemetry may raise the signal. The generic interface contract is intentionally narrow and vendor-neutral. The x86 code lives in `arch/x86/kernel/cpu/vector_limit.c` with declarations in `include/linux/vector_limit.h`. It exports `arch_scale_vector_pressure(policy_cpu)` returning `0 .. SCHED_CAPACITY_SCALE` and `cpuidle_vector_rank_bias_ns(cpu, predicted_ns, now_ns)` returning a bounded nanosecond bias. Updates are owned by the arch provider and happen only from non-sleeping local-CPU refresh points such as `schedutil` update context, scheduler tick, or idle exit; readers never touch hardware. Runtime topology pointers are RCU-protected. Multi-field state snapshots use `seqcount_latch_t`, and `cooldown_until_ns` is stored in `atomic64_t` with monotonic-max update semantics so cpuidle readers cannot observe torn 64-bit timestamps even on 32-bit or weakly ordered architectures. Consume-exactly-once correctness is explicit. `schedutil` applies the pressure only where it computes the capacity term supplied to `map_util_freq()`: `effective_max = max(1, arch_scale_cpu_capacity(cpu) - arch_scale_thermal_pressure(cpu) - arch_scale_vector_pressure(policy_cpu))`. The signal is not injected into `arch_scale_cpu_capacity()`, `arch_scale_freq_capacity()`, PELT, `uclamp`, EAS placement, or thermal-pressure accounting. RT and DL frequency updates that already flow through `schedutil` see the same single `effective_max`, but there is no second composition point elsewhere, so double counting is structurally prevented. The cpuidle embodiment reuses the same domain state without changing eligibility. `now_ns` comes from `ktime_get_mono_fast_ns()`, which is monotonic and immune to wall-clock adjustments. `remaining_ns = now_ns < cooldown_until_ns ? cooldown_until_ns - now_ns : 0`. `predicted_ns` is clamped to non-negative `u64`. The ranking-only bias is `bias_ns = min(predicted_ns >> 2, min(remaining_ns, 250000))`, and `rank_predicted_ns = predicted_ns - bias_ns`. This guarantees no underflow, no negative predicted time, and a hard cap of both 25 percent of the governor's own prediction and 250 microseconds absolute. `menu` and `TEO` call the helper only after their normal PM QoS, target residency, exit latency, and next-timer filters have already produced the eligible state set. The bias is then used only inside the ranking score. A state that is ineligible before the helper remains ineligible after the helper, so the invention cannot bypass existing latency or QoS constraints indirectly. Suspend and resume behavior is defined. Because `ktime_get_mono_fast_ns()` does not move backward under wall-clock adjustment, stored deadlines remain ordered during runtime. Because AVX ratio limiting cannot legitimately survive system suspend, a PM notifier clears all domains on suspend and resume, preventing stale cooldown timestamps after long sleep intervals.

2.4 Architecture

ASCII control flow:

x86 boot / hotplug
  -> detect AVX ratio tables + current perf-limit telemetry semantics
  -> build `vector_limit_domain` objects keyed by `(scope, package, die, cluster, core_type, core_id, policy_cookie)`
  -> publish domain tables with RCU

local CPU refresh path (`schedutil` update / tick / idle exit)
  -> rate-limit by `sample_period_ns`
  -> read current vector-limit reason MSR on the local CPU only
  -> classify mode: `NONE | AVX2 | AVX512`
  -> compute `raw_q10`
  -> update FSM snapshot under `seqcount_latch`
  -> update `atomic64_t cooldown_until_ns`

policy read path (`schedutil`)
  -> RCU lookup intersecting domains for `policy->related_cpus`
  -> `policy_pressure = max(effective_pressure(domain, now))`
  -> `effective_max = base_capacity - thermal_pressure - policy_pressure`
  -> `map_util_freq(util, effective_max)`

idle read path (`menu` / `TEO`)
  -> read cached domain snapshot only
  -> `bias_ns = cpuidle_vector_rank_bias_ns(cpu, predicted_ns, now_ns)`
  -> rank among already-eligible states using `predicted_ns - bias_ns`

Pseudocode:

for each online CPU c:
    if !telemetry_supported(c) or !ratio_table_supported(c):
        continue
    scope = detect_scope(c)   // CORE by default; DIE/PACKAGE/POLICY only if explicitly reported
    if scope == UNKNOWN:
        continue
    key = make_key(scope,
                   topology_physical_package_id(c),
                   topology_die_id(c),
                   topology_cluster_id(c),
                   x86_core_type(c),
                   topology_core_id(c),
                   cpufreq_policy_cookie(c))
    domain = lookup_or_create(key)
    cpumask_set_cpu(c, domain->cpus)

for each cpufreq policy p:
    p->vector_domains = { d | cpumask_intersects(d->cpus, p->related_cpus) }

refresh_domain(cpu, now_ns):
    if now_ns < this_cpu_read(next_sample_ns):
        return
    mode = read_current_vector_limit_mode(cpu)
    raw_q10 = ratio_to_q10(mode)
    write_seqcount_latch_begin(&domain->seq)
    update_fsm(domain, mode, raw_q10, now_ns)
    write_seqcount_latch_end(&domain->seq)
    this_cpu_write(next_sample_ns, now_ns + sample_period_ns)

policy_pressure(policy, now_ns):
    maxp = 0
    rcu_read_lock()
    for each domain d in policy->vector_domains:
        maxp = max(maxp, effective_pressure(d, now_ns))
    rcu_read_unlock()
    return maxp

Locking contract:
- runtime read paths take no scheduler locks and no cpuidle governor locks;
- runtime read paths use only RCU, seqcount retry, and atomic64 reads;
- topology rebuild uses `cpus_read_lock()` plus a provider mutex and then RCU swap;
- cpuidle never performs MSR reads, remote calls, or scheduler lock acquisition.

2.5 Implementation Plan

1. Add `include/linux/vector_limit.h` with generic stubs and two exported helpers: `arch_scale_vector_pressure(int cpu)` and `cpuidle_vector_rank_bias_ns(int cpu, u64 predicted_ns, u64 now_ns)`. Keep vendor reason codes private to x86. 2. Implement the Intel provider in `arch/x86/kernel/cpu/vector_limit.c`. Parse boot-time ratio ceilings from documented CPUID or model tables, map supported models to the exact current-status performance-limit reason bits, and refuse enablement on models where either ratio data or scope semantics are ambiguous. Reuse only documented current-status bits for control; leave sticky logged bits to diagnostics. 3. Create `vector_limit_domain` objects and policy-domain attachment tables. Build and rebuild them in CPU hotplug and cpufreq policy callbacks using `cpus_read_lock()` plus a provider mutex, publish the new tables with RCU, and include `core_type` in the key so heterogeneous packages cannot alias P-core and E-core identifiers. 4. Implement the non-sleeping refresh path. Reads happen only on the local CPU, rate-limited by `sample_period_ns`, and write the state machine under `seqcount_latch_t` while storing `cooldown_until_ns` in `atomic64_t`. Use `ktime_get_mono_fast_ns()` for all deadlines. Register a PM notifier that clears all domains on suspend and resume. 5. Integrate with `schedutil` at the single composition point where the governor computes the `max` capacity term that is passed to `map_util_freq()`. Do not modify PELT, `uclamp`, EAS, thermal pressure, or invariance paths. That preserves consume-exactly-once semantics. 6. Integrate with `menu_select()` and `teo_select()` only after each governor has formed the eligible state set from PM QoS, target residency, exit latency, and next-timer constraints. Feed `rank_predicted_ns` only into the ranking score. Do not alter eligibility checks, demotion tables, or `intel_idle` state-disable logic. 7. Add tracing and observability as diagnostics only, for example tracepoints that expose mode transitions, exported pressure, and cooldown horizon. Do not use debugfs or tracing interfaces as the control plane. The runtime control plane remains the internal provider API. 8. Add KUnit and hotplug tests for ratio math, state-machine monotonicity, RCU rebuild correctness, and cpuidle bias bounds. Add selftests or trace-based benchmarks for mixed scalar/AVX2/AVX-512 workloads on SMT-on, SMT-off, shared-policy, and heterogeneous-topology systems.

2.6 Validation Plan

The validation plan proves both feasibility and boundedness. A. Arithmetic and state-machine correctness: - Unit-test `raw_q10`, `pressure`, `cooldown_span_ns`, and `bias_ns` for edge cases including `Rscalar = 1`, `raw_q10 = 0`, `raw_q10 = 1024`, oscillatory `AVX512 <-> NONE` telemetry, and very small or zero `predicted_ns`. - Prove `0 <= pressure <= SCHED_CAPACITY_SCALE`, `0 <= bias_ns <= min(predicted_ns / 4, 250000)`, and `rank_predicted_ns >= 0`. - Prove monotonicity: if no new active telemetry is observed, `cooldown_until_ns` is not extended and exported pressure is non-increasing with time. B. Concurrency and topology correctness: - Stress CPU hotplug while repeatedly rebuilding policy attachments and verify that RCU readers in `schedutil`, `menu`, and `TEO` never dereference freed domain objects. - Run lockdep and KCSAN with forced refresh traffic to prove that cpuidle paths do not take scheduler locks and that scheduler paths do not take cpuidle governor locks. - Exercise 32-bit and weakly ordered configurations or litmus-style tests to verify that `atomic64_t cooldown_until_ns` cannot be read torn. C. Functional performance measurements: - Benchmarks with AVX-512 bursts followed by scalar wakeups on the SMT sibling; mixed AVX2 and scalar threads inside one shared `cpufreq` policy; and package-level throughput under bursty vector phases. - Compare baseline Linux versus the invention on frequency-request overshoot, actual attained frequency, package power, energy per transaction, and p95/p99 wakeup latency. - For cpuidle, compare state-selection distributions and wake latencies with `menu` and `TEO`, confirming that the bias only shifts ranking among already-eligible states and does not systematically suppress deep idle under common scalar workloads. D. Platform and lifecycle coverage: - Test on Intel servers with AVX2 only, AVX-512 capable SKUs, SMT enabled and disabled, policies spanning multiple cores, and heterogeneous topologies where `core_type` differs. - Validate suspend/resume by asserting that all cooldown state is cleared and no stale bias survives across sleep transitions. Success criteria are: fewer futile max-frequency requests during vector cooldown, lower package energy during mixed vector/scalar phases, no regressions in non-AVX workloads, no lock-order issues, and a mathematically bounded cpuidle bias that never overrides PM QoS or latency eligibility.

4. Risks & Mitigations

  • Risk: Intel model differences may expose different performance-limit reason encodings or ratio-table semantics. Mitigation: enable only from an explicit model whitelist with documented bit definitions and ratio sources; disable the provider on ambiguity instead of attempting heuristic decoding.
  • Risk: runtime MSR reads could add overhead on scheduler paths. Mitigation: refresh only on the local CPU, rate-limit reads with `sample_period_ns`, guard the entire feature with a static key, and make cpuidle consume only cached state rather than hardware reads.
  • Risk: incorrect scope detection could mis-aggregate pressure across CPUs. Mitigation: make `CORE` the default only for telemetry explicitly documented as core-shared, include `core_type` in the domain key for heterogeneous packages, support wider scopes only when firmware or hardware enumerates them, and disable the feature when scope is unknown.
  • Risk: the cpuidle bias could steer too aggressively away from deep states. Mitigation: cap `bias_ns` to both 25 percent of the governor's own prediction and 250 microseconds absolute, apply it only after existing eligibility filters, and expose tracing so platform owners can validate that deep idle remains reachable under non-AVX workloads.
  • Risk: stale cooldown state after suspend or CPU hotplug could create false pressure. Mitigation: clear all domains on suspend and resume, rebuild domain tables on hotplug and policy changes under RCU publication rules, and keep reader paths lock-free so no stale pointer survives grace-period retirement.
  • Risk: interaction with thermal pressure or frequency invariance could accidentally double count capacity loss. Mitigation: define a single composition point inside `schedutil` and prohibit feeding vector pressure into any other scheduler or cpufreq capacity pipeline.
