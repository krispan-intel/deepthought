2. Technical Invention Disclosure

2.1 Problem Statement

TDX guests can boot or hot-add with large ranges of private memory still in the unaccepted state. When Linux decides that a task should allocate or migrate onto destination node D, the buddy allocator may report nominal free capacity on D that is unusable until the guest performs TDX page acceptance. Purely reactive acceptance pushes this cost onto anonymous fault service, folio migration completion, and allocator slowpaths, which compounds the very NUMA-placement latency Linux is trying to reduce and creates outsized p95 and p99 stalls on confidential-VM workloads.

2.2 Prior-Art Gap

Existing TDX handling is predominantly demand-driven: pages are accepted only when an allocation or fault path finally needs them. Existing AutoNUMA and migration code observe remote hint faults, preferred-node changes, and migration outcomes, but do not transform those signals into a bounded per-node forecast of accepted-capacity demand. Existing buddy allocators also do not expose allocator-visible accepted-versus-unaccepted free inventory that lets fast paths prefer already accepted memory while preserving order and migratetype semantics. The missing combination is a topology-weighted, telemetry-derived accepted-capacity estimator coupled to a race-safe free-page-only acceptance worker and an allocator representation that preserves normal Linux free-page behavior once acceptance is complete.

2.3 Proposed Invention

The kernel adds a per-node acceptance controller that periodically computes an accepted-capacity deficit for each NUMA node from recent AutoNUMA and migration observations. When node D shows a positive deficit and pressure gates permit work, a node-local acceptance worker removes a bounded batch of free private-unaccepted buddy blocks from D under the zone lock, marks them isolated for acceptance, drops the lock, executes the x86 TDX page-accept helper on the physical range, optionally clears the pages if zero-on-accept is not architecturally guaranteed, reacquires the zone lock, and reinserts the exact same blocks into accepted buddy lists for the same order and migratetype. The allocator fast path and PCP refill logic always prefer accepted inventory; unaccepted inventory never appears on PCP lists. A synchronous fallback exists only on the blocking allocation slowpath and only when the caller is allowed to sleep and is not in reclaim-, compaction-, or emergency-allocation context. The result is an acceptance-state prefetch mechanism that is topology-aware, pressure-aware, and buddy-visible.

2.4 Architecture

Control flow: AutoNUMA hint faults plus migration outcomes plus preferred-node flips plus local accepted-inventory misses feed a per-node credit estimator; the estimator applies x86 topology weighting and reclaim hysteresis; a per-node ktdx_acceptd worker spends credits by isolating free-unaccepted buddy blocks from zone free_area lists; the worker executes the TDX page-accept operation outside the zone lock; accepted blocks are put back into the same node and migratetype as ordinary buddy pages; rmqueue and PCP paths see only accepted pages unless a strictly bounded blocking slowpath fallback is invoked. ASCII view: telemetry collectors -> per-node credit buckets -> pressure gates -> isolate free-unaccepted block -> TDX accept helper -> optional zero-before-publish -> accepted free_area and PCP refill.

2.5 Implementation Plan

Phase 1: add tracepoints and counters to task_numa_fault, task_numa_placement, and migrate_misplaced_folio to accumulate per-destination-node demand and accepted-inventory miss signals. Phase 2: extend the buddy allocator representation in the zone free_area path so free blocks can reside on accepted or unaccepted lists while preserving existing order and migratetype semantics; modify rmqueue_pcplist, __rmqueue_smallest, move_to_free_list, and __free_one_page so PCP paths see only accepted inventory and merge rules enforce matching acceptance class. Phase 3: add a per-node kthread or kthread_worker based acceptance daemon that isolates free-unaccepted blocks under zone->lock, executes the TDX accept helper outside the lock, optionally zeroes the block, and republishes it as accepted. Phase 4: wire reclaim, compaction, PSI, online_pages, offline_pages, alloc_contig_range, CMA, and hugetlb reservation gates into the credit clamp and freeze logic. Phase 5: add the bounded synchronous fallback to the preferred-node blocking allocation slowpath, with explicit lockdep assertions and context checks that reject atomic, reclaim, compaction, IRQ, and swap paths.

2.6 Validation Plan

Compare the invention against reactive TDX acceptance on confidential-VM kernels using synthetic and mixed workloads: anonymous page-fault storms, AutoNUMA migration workloads, memory hot-add bursts, and real latency-sensitive services. Primary metrics are p50, p95, p99, and max latency for anonymous faults, migrate_misplaced_folio completion, and __alloc_pages_slowpath duration on the preferred node. Secondary metrics are accepted-versus-unaccepted free inventory per node, remote allocation rate, accepted-credit overshoot, high-order fragmentation, compaction success, and CPU time consumed by the acceptance worker. Correctness validation includes lockdep, no-accept-in-atomic trace assertions, memory hotplug online and offline stress, alloc_contig_range and CMA contention tests, hugetlb reservation tests, memcg charge and reclaim tests, and fault injection that freezes a node mid-accept to prove generation-based drain behavior. Security validation confirms that only free private pages are accepted, shared pages never enter the worker path, and zero-before-publish occurs when zero-on-accept is unavailable.

3. Patent Claims

  1. 1. A method in a Linux confidential-virtual-machine guest for reducing memory-allocation and NUMA-migration latency, comprising: maintaining, for each NUMA node, separate accepted and unaccepted free-page inventories within buddy-allocator order and migratetype classes; deriving a per-node accepted-capacity credit from recent NUMA-balancing and page-migration telemetry; isolating only free private-unaccepted pages from an unaccepted inventory when the credit is positive; performing a TDX page-accept operation on the isolated pages outside the zone lock in a sleepable context; and reinserting the pages into an accepted inventory of the same node, order, and migratetype.
  2. 2. The method of claim 1, wherein the per-node accepted-capacity credit is computed from a temporally windowed combination of remote hint faults, migration successes, preferred-node transitions, and accepted-inventory misses, and wherein each contribution is weighted by x86 NUMA topology distance and by runnable residency on the destination node.
  3. 3. The method of claim 1, wherein per-cpu page lists are restricted to accepted pages and unaccepted pages remain exclusively on zone-lock-protected free lists, such that allocator fast paths consume accepted memory first without introducing a new pageblock migratetype.
  4. 4. The method of claim 1, wherein synchronous acceptance is permitted only in a blocking allocation slowpath that is not atomic, not interrupt context, not reclaim, not compaction, not swap I/O, and not emergency memory-allocation context, and wherein the synchronous acceptance is limited to a bounded aligned batch and a single retry attempt.
  5. 5. The method of claim 1, further comprising freeze-and-drain coordination with memory hotplug, contiguous allocation, CMA isolation, and hugetlb reservation, wherein pages in isolate, CMA, hugetlb-reserved, or offline-pending ranges are excluded from background acceptance and any isolated pages are drained before node offlining proceeds.

4. Risks & Mitigations

  • Risk: false-positive NUMA demand could accept too much memory on a node. Mitigation: short sampling windows, EWMA decay, topology weighting, residency weighting, and hard credit clamps tied to reclaim and watermark state.
  • Risk: accepted and unaccepted class separation could hinder coalescing. Mitigation: preserve merge rules by class, opportunistically accept aligned neighbors only when credits permit, and never promise large-page readiness.
  • Risk: hotplug or alloc_contig_range could race with a background worker. Mitigation: freeze generation, isolate count drain, and explicit exclusion of isolate, CMA, hugetlb-reserved, and offline-pending ranges.
  • Risk: accidental invocation from fragile contexts could violate allocator forward progress. Mitigation: keep all ordinary acceptance in a sleepable worker and gate the fallback with explicit blocking-context checks, PF_MEMALLOC exclusion, and no-accept trace assertions.
  • Risk: accepted pages could expose undefined contents if zero-on-accept is absent. Mitigation: perform zero-before-publish in the worker when architectural guarantees are unavailable.
  • Risk: CPU overhead of the worker could offset latency gains. Mitigation: bound batch size, limit worker CPU share per node, and stop work immediately when pressure gates or quiet-window decay reduce credits to zero.
