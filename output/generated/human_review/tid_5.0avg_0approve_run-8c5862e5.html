2. Technical Invention Disclosure

2.1 Problem Statement

High-PPS virtio-net workloads often recycle packet buffers and ring-adjacent metadata inside a small set of guest-physical regions, and that locality is frequently visible first in QEMU userspace datapath or vhost-user before KVM chooses where to spend hugepage-recovery effort. On x86 TDP-MMU, fault-time mapping may legitimately leave such regions as 4 KiB SPTEs because backing is temporarily fragmented, attributes are mixed, dirty logging or write-protect tracking is active, memslots are churning, or the range simply was not revisited by any later recovery path at the right time. When those transient blockers clear, KVM has no VM-scoped ABI for the VM owner to say 'retry this 2 MiB guest-frame chunk first.' The result is avoidable second-level page-walk cost, lower EPT TLB reach, and higher CPU-per-packet on the networking vCPUs even though the hot region is well known at the device datapath layer.

2.2 Prior-Art Gap

Existing knobs such as madvise(MADV_HUGEPAGE), MADV_COLLAPSE, host THP, hugetlb, or generic prefetch/readahead advice operate on host virtual address ranges or host backing policy rather than VM-scoped guest-frame chunks. Existing KVM TDP-MMU large-page promotion logic already knows how to install a huge SPTE when all normal conditions align, but it does not expose a VM-fd UAPI that lets userspace submit bounded, generation-aware, guest-frame hints saying which 2 MiB chunks are queue-hot and should be retried first. Prior art also lacks the specific correctness contract here: the hint is keyed to GFN space, carries no PFNs, does not change permissions or A/D handling, is admitted in O(1) time into a bounded per-VM structure, and is discarded whenever memslot topology, mmu invalidation state, dirty logging, page tracking, nested virtualization mode, or confidential-guest mode make the hint stale or inapplicable.

2.3 Proposed Invention

A queue-aware userspace observer aggregates completion-side datapath reuse into 2 MiB guest-frame chunks and computes a stability_score for each chunk from repeated reuse across epochs, queue IDs, and recency. The observer submits only the chunk identifiers and scores through the VM fd. KVM admits each chunk in O(1) time into a bounded per-VM structure consisting of an xarray or equivalent hash-index keyed by the tuple (gfn_base_2m, current_vm_generation_domain) for deduplication, plus a fixed number of priority-band FIFO lists for scheduling. Duplicate submissions overwrite the existing record's stability_score, bucket, and last-seen timestamp instead of creating a second record. Non-duplicate submissions are accepted only while per-VM and per-kuid or cgroup quotas permit; otherwise the ioctl returns EAGAIN. A single per-VM delayed worker drains the highest non-empty priority bucket subject to a strict batch budget, for example up to 16 chunks or 50 microseconds per run. For each chunk, the worker revalidates current memslot coverage, gates, and sequence numbers, then invokes the existing x86 TDP-MMU hugepage-recovery path against the already-populated 512-page span. A 2 MiB EPT leaf is installed only if normal KVM rules already allow it: compatible host backing, uniform SPTE attributes and memtype, no tracking modes that require 4 KiB granularity, notifier safety, stable MMU generation, and no mode exclusions. Failed attempts record a compact failure class and drive exponential backoff on that exact chunk so repeated futile retries do not starve other MMU work.

2.4 Architecture

In the primary embodiment, QEMU or a vhost-user backend observes descriptor completion batches, resolves them to GPA and then GFN using authoritative device-mapping state, folds the addresses into 2 MiB chunk IDs, and maintains an epoch-local stability table. Submission is off the datapath critical path and uses the VM fd only. Inside KVM, KVM_X86_SUBMIT_GFN_RANGE_HINTS performs constant-time syntactic validation, quota accounting, and deduplicating enqueue into the per-VM xarray plus priority buckets. No deep MMU walk occurs on submission. The per-VM delayed worker runs independently of vCPU threads. It first finds the current memslot for each candidate under slots synchronization, never dereferencing a stored raw memslot pointer. It precomputes or reuses any transient bookkeeping outside mmu_lock. When it enters the x86 TDP-MMU collapse path, it holds only the same locks already required by that path, performs no memory allocation while holding mmu_lock or tdp iter locks, and aborts the chunk if allocation would be required. Immediately before installing a huge SPTE it re-reads mmu_invalidate_seq to avoid racing mmu_notifier invalidations. If installation succeeds, existing KVM flush machinery is used; on Intel EPT this means the normal single-context INVEPT or equivalent existing remote-TLB flush path, and no GPA-range INVEPT capability is assumed. The feature is explicitly TDP-MMU only, is valid for classic EPT and optionally AMD NPT, is ignored or rejected under shadow MMU, and is outside scope for Intel TDX or any confidential mode using SEPT-like semantics.

2.6 Validation Plan

Benchmark baseline KVM against hint-enabled KVM on classic Intel EPT hosts using multiqueue virtio-net or vhost-user workloads with strong buffer reuse, including PPS-saturated forwarding and packet-recycling cases. Measure host CPU cycles per packet, PMU indicators of second-level translation cost such as walk duration and dTLB walk activity, throughput, and tail latency. Compare against host-HVA advice alone, such as madvise(MADV_HUGEPAGE), to show that VM-fd-scoped GFN prioritization yields additional benefit. The differentiating experimental claim is that, at the same offered PPS, the hint-enabled system reduces second-level walk-related PMU activity and lowers host CPU per packet on queue-hot workloads because targeted 2 MiB EPT leaves appear sooner on the hot chunks. Separately validate that under dirty logging, page tracking, nested virtualization, memslot churn, mmu_notifier activity, or TDX-like confidential modes, the feature degrades to baseline behavior by rejecting or clearing hints with bounded overhead and no semantic change to dirty accounting, A/D handling, permissions, or guest-visible behavior. Also record queue depth, stale-drop rate, backoff rate, and lack of added latency on the vCPU fault path, and confirm that no per-range success notification is exposed that would let a tenant infer host hugepage availability in fine detail.

4. Risks & Mitigations

  • Risk: Hot chunks are still not promotable because host backing is fragmented or SPTE attributes are non-uniform. Mitigation: the hint is advisory only, failed chunks record failure classes, and per-chunk exponential backoff prevents repeated futile work.
  • Risk: Memslot replacement, KVM_SET_USER_MEMORY_REGION changes, or mmu_notifier invalidations race the queue. Mitigation: hint records never keep raw memslot pointers, consume-time lookup is repeated under slots synchronization, and mmu_invalidate_seq is re-checked immediately before any huge-SPTE install.
  • Risk: Hint storms become a denial-of-service vector. Mitigation: explicit max_ranges_per_call, max_queued_chunks_per_vm, per-kuid or cgroup accounting, token-bucket throttling, duplicate overwrite semantics, and fixed worker budgets guarantee bounded submit cost and bounded worker CPU.
  • Risk: The interface becomes a side channel for host hugepage availability. Mitigation: the ioctl reports only submit-time syntax, policy, or freshness errors; it does not synchronously reveal per-range promotion success, host PFNs, or detailed backing state, and any optional counters are coarse, VM-local, and rate-limited.
  • Risk: Tracking modes or alternate MMU modes require 4 KiB semantics. Mitigation: dirty logging, page tracking, write-protect tracking, shadow MMU, nested-virt exclusions, and confidential-guest modes cause explicit reject, ignore, or queue-clear behavior rather than semantic change.
  • Risk: Rebuild storms or repeated stale hints under heavy churn reduce value. Mitigation: hint_epoch bumps clear queued work in bulk, stale-drop counters can temporarily disable the feature for the VM until the next stable epoch, and userspace can refresh only after KVM_X86_GET_GFN_HINT_STATE reports a new epoch.
  • Risk: A/D granularity or EPT invalidation semantics are accidentally changed. Mitigation: the invention explicitly preserves existing A/D and remote-TLB flush behavior, assumes only the existing single-context or equivalent invalidation path, and does not claim or require GPA-range INVEPT.
