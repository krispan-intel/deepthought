2. Technical Invention Disclosure

2.1 Problem Statement

Linux scheduling, including sched_ext policies, increasingly needs short-timescale task and CPU suitability hints on hybrid x86 systems, but the scheduler fast path cannot safely compose those hints in real time. CPU-side inputs such as Intel HFI, ITMT, thermal pressure, and arch_scale_cpu_capacity() describe only part of the placement problem. Task-side inputs such as L2 miss pressure, LLC streaming behavior, DTLB walk intensity, and hugepage sensitivity are visible through PMU sampling or memory-side evidence, but PMU overflow handlers may execute in NMI context and cannot take sleeping locks, allocate memory, walk scheduler topology, or mutate unbounded pid-keyed structures. sched_ext eBPF policies also cannot be trusted to reconstruct legality from racing cpuset, cpu_active, and sched_domain state on their own. The technical problem is therefore to provide sched_ext with a single-read, verifier-safe, low-latency decision object that combines x86 capacity and bounded task-phase evidence, remains correct across cpuset churn, CPU hotplug, topology rebuild, and object lifetime races, and guarantees forward progress when the hint becomes stale or invalid.

2.2 Prior-Art Gap

Existing interfaces each expose fragments but not the claimed kernel contract. arch_scale_cpu_capacity(), ITMT, Intel HFI, thermal pressure, and cpufreq data provide scalar CPU desirability but no per-task immutable lease generation, no expiry, no legal-placement cookie, and no mandatory fallback enforcement. sched_domains and LLC IDs provide topology but not a committed shortlist bound to a specific task generation. EAS, util_est, and uclamp express utilization or clamp policy, not PMU-derived phase compatibility or hugepage/cache pressure classes. Generic seqcount, RCU, or snapshot exposure mechanisms still require the BPF policy to fetch multiple mutable objects, reason about cross-source consistency, retry on races, and self-police legality. Perf-based phase-detection work generally stops at observability or classifier output and does not define a verifier-safe sched_ext ABI with opaque generation cookies, fixed-size descriptors, release/acquire publication, kernel-side legality recheck, and non-bypassable default legal dispatch. The non-obvious combination is precisely this: immutable legal-placement envelope plus expiry plus must-use commit helper, with PMU mining, LLC locality, and hugepage affinity as bounded dependent hints rather than the core claim.

2.3 Proposed Invention

Provide a Committed x86 Lease-Generation Snapshot Bridge with five bounded stages. Stage 1 - CAPTURE (NMI-safe, strictly bounded): Configure a small event set per sampled CPU, for example INST_RETIRED.ANY, CPU_CLK_UNHALTED.THREAD or cycles, an L2 miss event such as MEM_LOAD_RETIRED.L2_MISS or L2_RQSTS.MISS, and DTLB_LOAD_MISSES.WALK_COMPLETED. Sampling runs in overflow mode with period >= 500k retired instructions and an absolute cap of 2000 interrupts per second per CPU. The overflow handler writes a 32-byte record into a fixed per-CPU append-only ring of 4096 records: { task_cookie:u64, pid:u32, cpu:u16, event_mask:u16, ts_mono_ns:u64, l2_bucket:u8, llc_bucket:u8, dtlb_bucket:u8, ipc_bucket:u8, flags:u8 }. The handler performs no allocation, no blocking, no pid-hash mutation, no topology lookup, and no scheduler-facing state update. On full ring, it drops the sample and increments a per-CPU drop counter. Stage 2 - FOLD, CLASSIFY, AND BOUNDED MINING (deferred context only): A per-node or per-housekeeping kthread drains at most 256 records per wake and updates a fixed-capacity 4-way set-associative phase table keyed by task_cookie. The table size is static, for example 16384 entries globally or per node, so memory growth is bounded. Each entry stores exponentially decayed counters for l2_mpki, llc_mpki, dtlb_mpki, ipc, sample_count, last_seen_ns, last_llc_id, and a hugepage-evidence accumulator derived from sampled DTLB walk intensity plus whether the task recently executed in a THP-backed VMA or hugetlb-backed region observed by a separate bounded memory notifier. Every 100 ms, the worker computes updated bucket cutpoints from sliding histograms per hybrid class using fixed percentiles such as P50, P80, and P95; the resulting mining_seq is monotonic and versioned. The worker maps each active entry into stable enums: phase_class in { unknown, compute_dense, l2_bound, llc_streaming, dtlb_bound, mixed } and hugepage_affinity in { unknown, neutral, thp_beneficial, hugepage_strong }. This is scheduler-oriented data mining, but it is bounded, versioned, and entirely off the scheduler hot path. Stage 3 - BUILD AUTHORITATIVE LEGAL-PLACEMENT ENVELOPE: For each candidate task, the worker derives an internal legal mask using current authoritative state: legal_mask = p->cpus_ptr intersection cpu_active_mask intersection topology-selected span. The topology-selected span may be the task's current LLC domain, hybrid-capacity-preferred domain, or NUMA-local subset, but the result is always intersected with the task's current cpuset allowance and active CPUs. The worker ranks at most eight candidate CPUs using a score composed of normalized capacity, hybrid-class compatibility, LLC locality, L2-miss penalty, and hugepage affinity. The full cpumask remains kernel-internal; the BPF-visible object contains only a fixed-size shortlist and metadata. Stage 4 - PUBLISH IMMUTABLE GENERATION: The worker allocates an RCU-managed immutable object from a dedicated cache: kernel object = { rcu_head, full legal_mask, desc_v1 }. desc_v1 contains fixed-size fields only: abi_version:u16, len:u16, generation:u32, flags:u32, task_cookie:u64, lease_cookie:u64, publish_mono_ns:u64, expiry_mono_ns:u64, cpuset_seq:u32, hotplug_seq:u32, topology_seq:u32, mining_seq:u32, llc_id:u16, numa_id:u16, hybrid_class:u8, phase_class:u8, hugepage_affinity:u8, candidate_cnt:u8, preferred_cpu:s16, fallback_cpu_hint:s16, candidate_cpu[8]:u16, score_q10[8]:u16, perf_cap_q10[8]:u16, energy_cap_q10[8]:u16, l2_bucket[8]:u8, llc_bucket[8]:u8, dtlb_bucket[8]:u8. No variable-sized arrays are exposed in v1. After full initialization, the writer performs smp_store_release() on the task's published lease pointer. Old objects are reclaimed with call_rcu() after grace period. Stage 5 - READ AND COMMIT THROUGH VERIFIER-SAFE KFUNCS: The sched_ext BPF policy obtains the current task_cookie from existing task-cookie support and calls only fixed-ABI helpers such as scx_bpf_lease_read_v1(task_cookie, &out), scx_bpf_lease_commit_v1(task_cookie, lease_cookie, chosen_cpu, dsq_id, flags), and scx_bpf_lease_default_v1(task_cookie, dsq_id, flags). scx_bpf_lease_read_v1 enters rcu_read_lock(), performs an acquire load of the published pointer, copies the immutable fixed-size desc into BPF stack memory, and returns. scx_bpf_lease_commit_v1 revalidates that the task_cookie matches the task in the active sched_ext hook frame, the abi_version and len are supported, the lease_cookie matches, expiry_mono_ns has not passed, chosen_cpu is active, and chosen_cpu remains legal for the task under current cpuset and topology state. If validation succeeds, the helper commits placement. If any check fails, the helper must perform default legal dispatch: first prefer the current CPU's local DSQ if legal and runnable, otherwise invoke the kernel's default select_cpu path, and if no specific legal CPU is selected, enqueue on the global DSQ. There is no retry loop on a failed lease generation, so forward progress is preserved even if leases expire continuously.

2.4 Architecture

The architecture is a publish-once, validate-on-commit bridge between x86 microarchitectural sensing and sched_ext placement.

ASCII overview:

PMU overflow/NMI capture
    -> per-CPU fixed sample rings
    -> fold/mining kthread
    -> immutable lease builder
    -> smp_store_release(task->lease_ptr)
    -> sched_ext eBPF scx_bpf_lease_read_v1()
    -> BPF policy chooses one candidate from shortlist
    -> scx_bpf_lease_commit_v1()
       -> validate cookie + expiry + current legality
       -> either commit requested CPU
       -> or force default legal dispatch

Internal object layout:
- struct scx_lease_gen_kern_v1
  - rcu_head rcu
  - cpumask legal_mask   (kernel-only; never exposed directly to BPF)
  - struct scx_lease_desc_v1 desc

BPF-visible struct scx_lease_desc_v1 fields and units:
- abi_version:u16 = 1
- len:u16 = sizeof(desc_v1)
- generation:u32 = per-task monotonically increasing lease generation
- flags:u32 = validity and optional-hint flags
- task_cookie:u64 = stable task identity used by kfuncs
- lease_cookie:u64 = opaque SipHash(task_cookie, generation, seqs, boot_nonce)
- publish_mono_ns:u64 = CLOCK_MONOTONIC timestamp in ns at publication
- expiry_mono_ns:u64 = absolute CLOCK_MONOTONIC deadline in ns
- cpuset_seq:u32 = seq bumped on cpuset-authority change
- hotplug_seq:u32 = seq bumped on CPU online/offline authority change
- topology_seq:u32 = seq bumped on sched-domain or LLC topology rebuild
- mining_seq:u32 = seq of histogram-derived cutpoints used for classification
- llc_id:u16, numa_id:u16 = placement locality identifiers
- hybrid_class:u8 = 0 unknown, 1 performance, 2 efficient, 3 SMT-shared
- phase_class:u8 = 0 unknown, 1 compute_dense, 2 l2_bound, 3 llc_streaming, 4 dtlb_bound, 5 mixed
- hugepage_affinity:u8 = 0 unknown, 1 neutral, 2 thp_beneficial, 3 hugepage_strong
- candidate_cnt:u8 = 0..8
- preferred_cpu:s16 = best candidate or -1
- fallback_cpu_hint:s16 = safe hint for low-cost fallback or -1
- candidate_cpu[8]:u16 = ranked CPU IDs
- score_q10[8]:u16 = normalized rank score 0..1024
- perf_cap_q10[8]:u16 = normalized performance capacity 0..1024
- energy_cap_q10[8]:u16 = normalized energy efficiency 0..1024
- l2_bucket[8], llc_bucket[8], dtlb_bucket[8]:u8 = classifier buckets used for transparency and debugging

Memory-ordering contract:
- Writers fully initialize desc and legal_mask, then publish the pointer with smp_store_release().
- Readers use rcu_read_lock() plus rcu_dereference()/smp_load_acquire() before copying desc.
- Because desc_v1 is fixed-size and immutable after publication, readers need no retry loop and cannot observe torn variable-length fields. Variable-sized cpumask exposure is intentionally excluded from v1 to preserve the single-snapshot property.

Lifetime and stale-value rules:
- Published lease objects are RCU-reclaimed only after pointer replacement and grace period.
- On task exit, the task's lease pointer is nulled with release semantics and the last object is freed via RCU.
- BPF cannot safely reuse cached descriptors across generations because commit requires exact task_cookie plus lease_cookie match; any descriptor from an old generation, old seq tuple, or expired deadline is rejected and mapped to fallback.
- Unknown abi_version or unknown enum values are treated as unsupported and force fallback rather than heuristic interpretation.

2.5 Implementation Plan

1. Kernel lease core: Add built-in sched_ext lease support in kernel/sched/ext so kfunc availability does not race with module unload. If modular packaging is required, pin active BPF links and refuse provider unload until all lease kfunc users detach, then wait for an RCU grace period before unregistering. 2. Capture path: Add per-CPU ring buffers and perf overflow handlers. Enforce a static event set, minimum sample period, and hard interrupt-rate cap. If PMU availability is missing on a CPU or the cap is exceeded, mark optional phase hints unavailable and continue publishing capacity-only leases. 3. Bounded fold/mining path: Implement the fixed-capacity phase table and histogram mining worker. Eviction is bounded and explicit: overwrite only entries idle beyond a threshold such as 50 ms; otherwise drop and count. No unbounded hash tables, no per-sample allocation, and no sleeping locks in overflow context. 4. Authority and serialization: Maintain seq counters under existing serialized update paths rather than from the scheduler hot path. cpuset_seq is incremented only while cpuset authority is being updated under cpuset serialization; hotplug_seq is incremented only during CPU hotplug state transitions; topology_seq is incremented only during sched-domain or LLC topology rebuild serialization. The lease builder may run under cpus_read_lock() plus RCU to observe a stable snapshot, but it never runs under rq->lock and never takes cpuset or topology locks in any sched_ext fast path. At commit time, legality is authoritative in-kernel: the helper checks cpu_active(chosen_cpu), current task allowance, and current sequence tuple before placement. 5. rq-lock interaction rule: Restrict scx_bpf_lease_read_v1() and scx_bpf_lease_commit_v1() to non-rq-locked sched_ext hooks such as select_cpu and pre-dispatch enqueue paths. The verifier rejects these kfuncs from hooks that execute under rq->lock. The commit helper itself does not take cpuset_mutex, sched topology locks, or CPU hotplug write-side locks; it uses lockless sequence checks and current legality tests only, preventing lock inversion with rq->lock. 6. Mandatory fallback semantics: Make lease commit non-bypassable for lease-aware tasks. Once a task is marked lease-aware, direct BPF dispatch without either scx_bpf_lease_commit_v1() or scx_bpf_lease_default_v1() is rejected by the sched_ext core. Error mapping is deterministic: ESTALE, EEXPIRED, EINVALCPU, ECOOKIE, EABI, or EILLEGAL all trigger default legal dispatch, not silent failure. Default legal dispatch means local DSQ if current CPU is legal; else core default select_cpu result; else global DSQ. Because the helper never retries a bad lease generation, the task is always enqueued somewhere legal in O(1) helper invocations. 7. Hugepage and LLC dependent feature: Add an optional memory notifier that sets a bounded hugepage evidence bit when the task runs in THP or hugetlb-backed VMAs. The lease builder combines that with DTLB-walk pressure and LLC locality to bias ranking toward CPUs within the LLC domain already associated with the task's recent large-page working set. If the feature is unavailable, hugepage_affinity remains neutral and the independent claim still stands. 8. BPF policy contract: The reference BPF scheduler reads the descriptor, checks abi_version == 1, ignores any optional field whose corresponding flag is absent, selects a CPU from candidate_cpu[0..candidate_cnt-1], and commits. If candidate_cnt == 0, abi_version is unknown, or any enum value is outside the known range, the BPF program must invoke scx_bpf_lease_default_v1(). This ensures older policies fail safe on newer kernels.

2.6 Validation Plan

Correctness validation: - Race tests that repeatedly toggle cpusets, offline/online CPUs, and rebuild sched domains while tasks continuously obtain and commit leases; assert no illegal CPU placement, no use-after-free, and no missing enqueue. - Memory-model tests using stress and KCSAN-style instrumentation to confirm that readers never observe partially initialized desc_v1 fields and never require retry loops. - Lifetime tests that exit tasks, detach BPF programs, and churn leases while RCU grace periods are delayed. Performance validation: - Measure hot-path overhead of read plus commit against a sched_ext baseline; target no PMU reads and no topology walks in the hot path, with helper cost stable regardless of topology size because candidate lists are fixed-size. - Bound telemetry cost by configuration: <= 2000 PMU interrupts/sec/CPU, 32-byte records, 4096-record per-CPU rings, max 256 drained samples per worker wake. Demonstrate worst-case CPU overhead remains below a configurable budget such as 1 percent per sampled CPU; when exceeded, the kernel auto-degrades to capacity-only leases. Workload validation: - Hybrid-core latency workloads: hackbench, netperf, nginx, memcached, and database request traces to show reduced tail latency and better CPU-class matching. - Memory-bound and hugepage-sensitive workloads: THP-enabled RocksDB, Redis, STREAM-style memory tests, and an HPC or astrophysical simulation representative of FLASH-like hugepage usage to show fewer harmful migrations across LLC domains and reduced DTLB/L2 miss penalties. - Compare against baselines using only arch_scale_cpu_capacity(), only HFI/ITMT, only topology/LLC affinity, and no lease mechanism. Fallback validation: - Inject stale cookies, expired timestamps, unknown abi versions, illegal CPUs, and outdated sequence tuples; verify the task always reaches a legal DSQ and never livelocks even when every lease attempt fails. Compatibility validation: - Verify enum forward compatibility by running an older BPF policy against a kernel that emits a new optional enum value; expected behavior is deterministic fallback, not mis-scheduling.

4. Risks & Mitigations

  • Risk: PMU event availability and semantics vary across x86 generations. Mitigation: define a minimal supported event matrix, publish capability flags in lease flags, and degrade to capacity-only leases when a requested event set is unavailable.
  • Risk: Sampling overhead could erase scheduler gains. Mitigation: enforce minimum sampling periods, hard per-CPU interrupt-rate caps, bounded ring sizes, bounded drain counts, and automatic temporary disable of optional phase mining when overhead budget is exceeded.
  • Risk: Frequent cpuset or hotplug churn can make leases expire before use. Mitigation: short explicit expiries, exact sequence cookies, O(1) fallback with no retry loop, and capacity-only ranking when sequence churn is high.
  • Risk: Misclassification of memory-bound tasks could bias placement incorrectly. Mitigation: use coarse stable enums, versioned mining cutpoints, optional hint flags, and default-safe behavior when evidence confidence is low or unknown.
  • Risk: Prior-art overlap from exposing capacity or perf data to scheduling policy. Mitigation: keep the independent claim narrowly focused on immutable lease generation plus authoritative kernel commit and fallback semantics, and position PMU mining, LLC locality, and hugepage affinity as dependent claims.
