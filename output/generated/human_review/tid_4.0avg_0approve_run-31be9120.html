2. Technical Invention Disclosure

2.1 Problem Statement

AMX-heavy Linux guests running on KVM incur host-side penalties that existing virtualization interfaces do not address cleanly: vCPU migration across LLC domains destroys locality for matrix phases, and second-level page mappings may stay fragmented even when the guest is repeatedly touching dense, tile-shaped working sets that would benefit from stable huge mappings. The host, however, must preserve scheduler fairness, affinity, cpusets, cgroup policy, core scheduling, dirty logging, NUMA and THP rules, and MMU invalidation correctness. It must not read guest memory, fault, allocate, take guest FPU ownership, or perform XSAVE/XRSTOR under rq locks or TDP MMU fast paths. It also must not let malformed xstate, drifting CPU models, or migration retries silently change the guest-visible AMX ABI. The missing system problem is therefore a kernel-safe way to expose AMX-aware optimization signals to the host scheduler and MMU while freezing migration semantics and keeping hot paths side-effect free.

2.2 Prior-Art Gap

Existing QEMU CPU models, KVM CPUID filtering, and XSAVE migration payloads can make source and destination hosts broadly compatible, but they do not create a kernel-enforced sealed contract that freezes exactly which AMX-related CPUID 0x7 bits, CPUID 0xD layout, XCR0 bits, IA32_XSS values, IA32_XFD policy, and tile-geometry maxima are allowed after the VM begins execution. Existing paravirtual scheduler hints communicate preemption or lock state, not a bounded LLC-residency preference derived from matrix-phase structure. Existing KVM/TDP hugepage heuristics rely on fault density, A/D aging, or generic access-range behavior and do not consume an immutable, generation-tagged, AMX-specific geometry descriptor extracted from validated guest xstate. Existing migration checks also do not define a one-way restore-failure latch that prevents repeated retries with incompatible AMX state. The inventive gap is the combination of sealing, bounded host-normalized hinting, xstate-fresh geometry extraction, and generation-safe MMU use.

2.3 Proposed Invention

The invention adds an optional KVM capability, for example `KVM_CAP_X86_AMX_ACE`, with a concrete VM ABI. Userspace first configures the conventional CPU model, then submits `KVM_X86_VM_SET_ACE` carrying a normalized `struct kvm_ace_v1` that includes: AMX-related CPUID 0x7 bits; a CPUID 0xD layout hash for XTILECFG/XTILEDATA sizing and offsets; allowed XCR0 bits; allowed IA32_XSS bits; initial IA32_XFD mask and permitted XFD toggles; and maximum legal AMX geometry such as palette ID bound, tiles bound, rows-per-tile bound, bytes-per-row bound, and total tile-bytes bound. `KVM_X86_VM_SEAL_ACE` transitions the VM from OPEN to SEALED and computes a `normalized_hash`. After seal, KVM rejects any userspace ioctl, restore payload, or runtime control write that would alter those frozen elements. Migration compatibility is defined precisely as: same ACE ABI version, same `normalized_hash`, destination host supports every sealed feature, CPUID 0xD layout matches, and restored guest AMX xstate validates against the sealed maxima. If restore fails, a one-way `ACE_RESTORE_FAILED` latch is set so later restore or run attempts fail deterministically until VM reset or recreation. The scheduler plane introduces one bound 64-byte guest phase-hint record per vCPU via `KVM_X86_VCPU_BIND_PHASE_HINT`. KVM snapshots that GPA only at legal sleepable run boundaries using existing guest-memory helpers. The hint carries no guest addresses; it contains only version, phase epoch, phase class, requested locality score, and requested TTL. KVM clamps those fields to host bounds, derives a host-owned `kvm_ace_residency_token` with a sequence number, current LLC ID, bounded locality score, and expiration in a small number of scheduler slices, and publishes it with WRITE_ONCE. Scheduler hot paths only read the token and may use it as a tiebreaker within already-allowed CPUs; they never override affinity, cpuset, cgroup, core-scheduling, or fairness rules. The MMU plane introduces an immutable `kvm_amx_geom_desc` published under RCU. Geometry extraction is legal only when `ace_xstate_fresh` is set, meaning KVM already has an up-to-date saved guest xstate image in the vCPU's existing xstate buffer due to an existing save/sync path such as successful `KVM_SET_XSAVE`, a migration-save/`KVM_GET_XSAVE` path after KVM's normal guest-state synchronization, or a `KVM_RUN` return path that has already materialized guest xstate to host memory. If the bit is clear, extraction is a strict no-op. The descriptor stores tile geometry plus generation tags: `ace_gen`, `xstate_seq`, `memslots_gen`, `mmu_inval_seq`, and a TDP root role identifier. TDP MMU readers compare all tags; any mismatch causes an immediate no-op. To avoid overlap with generic fault-density schemes, the MMU side also uses a concrete bounded evidence structure: a fixed direct-mapped array of 32 per-vCPU `ace_fault_bucket` entries keyed by 2 MB-aligned GFN base. Each bucket records bounded counts for read/write/execute EPT violations, an A/D-young bit when available, a saturating score, and `last_jiffies`. Buckets are allocated once at vCPU creation, updated in amortized O(1), and replaced only by same-key overwrite or lowest-score eviction; no per-fault allocation occurs. A geometry descriptor can only raise hugepage admissibility if existing KVM promotion rules already pass, the current bucket score crosses threshold, the geometry footprint fits the candidate 2 MB window, the descriptor is unexpired, and all generation tags match. Thus geometry acts only as a host-side admissibility constraint, not as an address selector.

Host-owned structures:
- `struct kvm_ace_v1 { u32 abi_version; u32 flags; u64 amx_cpuid7_mask; u64 cpuid_0d_layout_hash; u64 xcr0_allowed_mask; u64 xss_allowed_mask; u64 xfd_initial_mask; u64 xfd_toggle_mask; u16 max_palette; u16 max_tiles; u16 max_rows_per_tile; u16 max_bytes_per_row; u32 max_total_tile_bytes; u8 normalized_hash[32]; u64 ace_gen; }`
- `struct kvm_ace_residency_token { u32 seq; u16 llc_id; u8 locality_score; u8 ttl_slices; u64 expires_jiffies; }`
- `struct kvm_amx_geom_desc { u64 ace_gen; u64 xstate_seq; u64 memslots_gen; u64 mmu_inval_seq; u32 tdp_root_role_id; u16 palette; u16 tiles; u16 rows; u16 bytes_per_row; u32 total_tile_bytes; u8 confidence; u8 expires_in_quanta; }`
- `struct ace_fault_bucket { u64 gfn_2m_base; u16 faults_r; u16 faults_w; u8 faults_x; u8 score; u32 last_jiffies; u8 ad_young; u8 valid; }[32]` per vCPU.

State machine and invariants:
1. ACE is mutable only in OPEN state. `KVM_X86_VM_SEAL_ACE` moves it to SEALED before first vCPU run.
2. After SEALED, KVM rejects modifications to AMX-related CPUID 0x7/0xD exposure, XCR0 policy, IA32_XSS policy, IA32_XFD policy, or ACE geometry maxima.
3. `ace_xstate_fresh` is set only on paths that already leave a synchronized guest xstate image in the existing KVM xstate buffer, and each such event increments `xstate_seq`. If the bit is not set, no geometry descriptor is built.
4. Descriptor publication snapshots `ace_gen`, `xstate_seq`, `memslots_gen`, `mmu_inval_seq`, and `tdp_root_role_id`. MMU readers require exact match across all of them and otherwise do nothing.
5. The residency token never contains guest addresses and expires quickly. The geometry descriptor never leaves the kernel by default. Existing KVM hugepage, dirty logging, and memory pressure rules remain the primary gate; ACE only adds a secondary bias signal.

2.5 Implementation Plan

1. UAPI and capability negotiation: - Add `KVM_CAP_X86_AMX_ACE`, `KVM_X86_VM_SET_ACE`, `KVM_X86_VM_GET_ACE`, `KVM_X86_VM_SEAL_ACE`, and `KVM_X86_VCPU_BIND_PHASE_HINT` in `include/uapi/linux/kvm.h` and document them in the KVM API docs. - QEMU flow: build normal x86 CPU model first, query capability, submit ACE, read back normalized ACE/hash, include that hash in migration metadata, and seal before first `KVM_RUN`. 2. VM seal enforcement and compatibility predicate: - Store ACE in `struct kvm` with state OPEN/SEALED/RESTORE_FAILED. - At seal time, reject illegal or host-inconsistent combinations: AMX feature bits not supported by host, CPUID 0xD layout inconsistent with xfeatures, XCR0 bits outside host support, IA32_XSS supervisor bits outside policy, or IA32_XFD masks enabling unsupported xfeatures. - Hook post-seal enforcement into the existing CPUID/xstate/MSR control paths so AMX-related `KVM_SET_CPUID2`, xstate restore, and relevant control-register/MSR writes cannot widen or alter the sealed envelope. - Compatibility test on migration restore: `src.abi_version == dst.abi_version && src.normalized_hash == dst.normalized_hash && dst_host_caps superset src_caps && restored_xstate validates under src maxima`. On failure, set `RESTORE_FAILED` and return the same terminal error on every subsequent restore/run until reset. 3. Phase-hint sampling and scheduler token: - Bind exactly one 64-byte guest GPA per vCPU. Snapshot it only at `KVM_RUN` boundaries where guest-memory access may sleep. - Normalize fields: clamp locality score to a small range such as 0..15, clamp TTL to a small number such as 1..4 scheduler quanta, ignore unknown versions, and hash only `(vm_id, vcpu_id, phase_epoch)` into the token sequence. - Scheduler integration lives in existing wake-placement or task-selection hooks and uses the token only as an LLC-locality tiebreaker inside allowed CPU masks. 4. Saved-xstate freshness contract and geometry extraction: - Maintain `ace_xstate_fresh` and `xstate_seq` per vCPU. - Set them only when KVM already has a synchronized guest xstate image in the existing save buffer: after successful `KVM_SET_XSAVE`, after migration-save/`KVM_GET_XSAVE` style paths that already invoke the normal KVM guest-state synchronization, and after a `KVM_RUN` return path that has already copied AMX state into the buffer. - Clear or invalidate on vCPU reset, failed restore, ACE reseal, or any operation that changes the control policy relevant to AMX xfeatures. - Extract palette, tile count, max rows, bytes per row, and total tile bytes from that saved buffer only in the vCPU thread, validate them against the sealed ACE maxima, and publish a new RCU descriptor if valid; otherwise publish nothing. 5. MMU evidence and deterministic decay: - Add 32 fixed `ace_fault_bucket` entries per vCPU in the TDP MMU path, keyed by 2 MB-aligned GFN base, allocated once at vCPU creation. - On each EPT violation relevant to a hugepage candidate, update the matching bucket in O(1): same-key saturating increment or lowest-score replacement. Record read/write/execute class, `last_jiffies`, and A/D-young when present. - Decay lazily with no timers: `delta = min(MAX_SHIFT, (jiffies - last_jiffies) >> ACE_DECAY_SHIFT); score = score >> delta; last_jiffies = jiffies;`. Without EPT A/D, use only violation-derived increments plus the same lazy decay. - Promotion bias is applied only if current KVM/TDP hugepage checks already pass, bucket score is above threshold, descriptor TTL is live, geometry fits within the currently faulting 2 MB-aligned range, and all descriptor generation tags match current `memslots_gen`, `mmu_inval_seq`, and root role. Any mismatch is an immediate no-op; there is no partial application. 6. Security and observability discipline: - Keep ACE descriptors and geometry internal to KVM by default: no debugfs, perf, or tracepoint export of raw geometry or token contents. - If observability is needed, expose only coarse aggregated counters behind an explicit admin-only opt-in. - Ensure the hint contains no guest GPAs, and geometry cannot force promotion on an ineligible region; it can only raise confidence for a region already surfaced by ordinary KVM fault handling.

4. Risks & Mitigations

  • Risk: post-seal CPU-model drift could create a split-brain AMX ABI between userspace and kernel; mitigation: seal ACE only after normal CPU-model setup, store a normalized ACE hash in KVM, and reject all AMX-related CPUID/xstate/MSR mutations after sealing.
  • Risk: stale geometry could bias MMU decisions after guest phase changes; mitigation: publish geometry only when `ace_xstate_fresh` is set, tag each descriptor with `xstate_seq` plus MMU generations, give it a short expiry, and require exact tag match or immediate no-op.
  • Risk: a malicious guest could induce excessive host work through EPT fault storms; mitigation: use a fixed 32-bucket per-vCPU evidence table, saturating counters, same-key overwrite or lowest-score eviction, and no per-fault allocation or global scans.
  • Risk: hosts without EPT A/D may degrade signal quality or need expensive maintenance; mitigation: define a deterministic lazy-decay policy based on `jiffies`, apply it only on local bucket access, and avoid timers entirely.
  • Risk: hints or geometry could amplify cross-VM side channels through predictable hugepage behavior; mitigation: prohibit guest addresses in hints, keep raw token/geometry internal to KVM by default, and let ACE only raise admissibility when ordinary KVM hugepage gates already pass.
  • Risk: restore loops on incompatible AMX state could cause operational churn; mitigation: add the one-way `RESTORE_FAILED` latch so repeated attempts fail immediately and consistently until VM reset or recreation.
  • Risk: scheduler fairness could regress if locality bias is too strong; mitigation: bound token lifetime and weight tightly and apply it only as an LLC-locality preference inside existing scheduler legality and fairness rules.
