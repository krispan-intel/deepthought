2. Technical Invention Disclosure

2.1 Problem Statement

CXL host bridges, switches, ports, and Type-3 devices expose low-rate but latency-sensitive control-plane interrupts such as mailbox completion, event-log notification, RAS signaling, media-status change, poison-related notifications, and topology-change events. On multisocket x86 servers, the default CPU chosen for these MSI or MSI-X interrupts can be remote from the CXL root port, decoder, or memory expander, creating unnecessary cross-socket traffic and longer mailbox or RAS tail latency. The placement decision also ages poorly: decoder rebalance, memory-region reconfiguration, CPU hotplug, node hotplug, hot-remove recovery, or switch-path changes can invalidate an initially good CPU target. A simple in-place retarget is not enough. On x86 systems with interrupt remapping, software may observe an interrupt after irq_set_affinity and even after synchronize_irq, yet still be unable to prove whether that interrupt corresponds to the pre-switch route or the post-switch route, because synchronize_irq drains executing handlers, not the visibility of backend remap-table invalidation or device-side policy writes that selected the old MSI-X entry. That ambiguity blocks correctness-sensitive policies such as stale-route suppression, locality accounting, and bounded batching. In parallel, some CXL event sources expose durable event logs or sticky status, so servicing every interrupt immediately wastes CPU under correctable or advisory storms, while other sources such as mailbox completion and fatal errors are not paceable at all. A kernel-resident mechanism is needed that improves x86-locality for CXL interrupts, makes route generation observable during retarget, coalesces stale-lane late arrivals without unbounded workqueue amplification, respects IRQF_ONESHOT and sleepability rules, degrades safely when vectors or remap hooks are unavailable, and never fights administrator affinity control.

2.2 Prior-Art Gap

Linux already provides pci_alloc_irq_vectors_affinity, managed IRQ placement, irq_set_affinity_hint, irq_set_affinity, synchronize_irq, request_threaded_irq, CPU-hotplug migration, and backend-specific interrupt-remap logic in Intel VT-d and AMD IOMMU code. CXL already provides event logs, sticky event-status indications, and mailbox or register-level interrupt policy control. The missing element is an end-to-end CXL-specific handoff protocol that makes route generation observable and commits only when both host and device routing are known to have crossed the handoff boundary. The precise failure mode left unsolved by the prior art is this: with one virq, software may mask, synchronize, update affinity, and unmask, but any later interrupt remains ambiguous because it may have been generated before the device observed the new policy or before the IOMMU’s remap or interrupt-entry cache observed the new target, and the same virq identity is used on both sides of the transition. Synchronize_irq is insufficient because it waits only for already-running handlers on that virq; it does not prove that queued invalidation for Intel VT-d interrupt-remap entries, AMD interrupt-remap-table visibility, or any posted write that changes the selected MSI-X entry has completed. Standard interrupt moderation also does not solve the CXL-specific distinction between durable event logs, which can be drained in bounded windows, and non-durable sources, which must bypass pacing entirely. No existing Linux mechanism combines dual-lane route identity, x86 remap-visibility commit, stale-lane coalescing, CXL-durable severity taxonomy, vector-pressure fallback, and explicit administrator-ownership handoff in one coherent protocol.

2.3 Proposed Invention

The invention introduces a per-source cxl_irq_source control object and a dual-generation handoff protocol. Representative state: struct cxl_irq_lane { int virq; u16 msix_entry; cpumask_var_t target_mask; u8 slot_id; }; enum cxl_route_state { OLD_ACTIVE, SWITCHING, NEW_ACTIVE }; enum cxl_batch_state { BYPASS, WINDOW_OPEN, FORCE_DRAIN }; enum cxl_override_state { DRIVER_MANAGED, ADMIN_OWNED }; struct cxl_irq_source { raw_spinlock_t route_lock; struct mutex ctrl_mutex; struct cxl_irq_lane lane[2]; u8 active_slot; u8 pending_slot; u32 generation; enum cxl_route_state route_state; enum cxl_batch_state batch_state; enum cxl_override_state override_state; atomic_t drain_pending; atomic_t urgent_pending; struct hrtimer reopen_timer; ktime_t reopen_deadline; unsigned int max_defer_us; bool single_slot_fallback; bool source_has_durable_log; bool source_is_urgent_only; struct work_struct drain_work; struct irq_affinity_notify affinity_notify; }; Required route-state invariants: 1. Exactly one slot is authoritative for normal servicing at any instant, identified by active_slot. 2. route_state equals OLD_ACTIVE or NEW_ACTIVE only when active_slot and device policy agree. 3. During SWITCHING, the inactive slot may receive interrupts, but those interrupts are never allowed to perform device mailbox or log-drain work directly; they may only coalesce a deferred drain through drain_pending. 4. The new slot becomes authoritative only after both host-side interrupt-remap visibility and device-side event-policy acknowledgement complete. 5. If single_slot_fallback is true, the driver performs ordinary affinity updates but does not claim stale versus current distinguishability. Retarget sequence for a dual-slot source: 1. In sleepable worker context, lock ctrl_mutex. If override_state is ADMIN_OWNED, abort automatic retarget. 2. Select a candidate CPU mask using dev_to_node, the root-port or device NUMA node, cpumask_of_node, housekeeping constraints, and Linux x86 topology that already reflects CPUID 0x1F or 0x0B x2APIC relationships. Prefer a housekeeping CPU local to the CXL root and avoid remote socket spill unless no local online CPU is available. 3. Choose pending_slot as the currently inactive slot. Program only that slot’s host virq affinity through irq_set_affinity on its virq and update irq_set_affinity_hint for observability. 4. Call a backend-defined remap-visibility primitive from the same sleepable context. The minimal new kernel API is arch_irq_remap_sync(struct irq_data *d), or equivalently an irq_domain operation with the same semantics. It returns only after the active interrupt-remap backend has made the updated destination globally visible for that virq. On Intel VT-d, the implementation waits for the queued invalidation and interrupt-entry-cache invalidation associated with the IRTE update to complete. On AMD IOMMU, the implementation waits for completion of the backend’s interrupt-remap-table invalidation sequence for that interrupt. If no backend exposes such a hook, the source is placed in single_slot_fallback and the dual-generation guarantee is disabled. 5. After host remap visibility returns, program the CXL device-side source-to-vector mapping, such as Set Event Interrupt Policy or the source’s interrupt-policy register, so the logical source points to the pending slot’s MSI-X entry. 6. Wait for device acknowledgement or mailbox completion, then acquire route_lock, increment generation, set active_slot to pending_slot, transition route_state to NEW_ACTIVE, and release route_lock. 7. The previously active slot remains registered. Any late arrival there is classified as stale because slot_id no longer equals active_slot. That handler path only atomically sets drain_pending and queues drain_work once. Interrupt servicing rules: - Primary handler fast path reads active_slot, route_state, batch_state, and override flags with READ_ONCE or under route_lock, performs no sleeping operation, issues no mailbox command, and never takes ctrl_mutex. - If the interrupt arrived on a non-authoritative slot, the handler executes: if atomic_cmpxchg(drain_pending, 0, 1) == 0 then queue_work(drain_work); return IRQ_HANDLED. - If the authoritative source is non-durable or urgent-only, the primary handler sets urgent_pending and returns IRQ_WAKE_THREAD. - If the authoritative source is durable and paceable, the primary handler opens or extends a bounded window and returns IRQ_HANDLED without waking the thread, leaving the line unmasked. - The threaded handler or drain_work path is the only context allowed to issue mailbox commands, read event records, inspect overflow bits that require sleepable access, or clear device-side status. Severity-aware batching taxonomy: - Bypass immediately: mailbox completion, fatal and uncorrectable RAS, poison or media conditions that require prompt host action, topology-hot-remove or link-loss indications, and any source without durable state. - Windowable: informational, advisory, warning, and correctable-event classes whose records persist in a CXL device event log until fetched and cleared, or whose source exposes a sticky non-empty status bit. - Progress rule: each windowable source has max_defer_us. The first interrupt opens WINDOW_OPEN. The hrtimer expiration moves the source to FORCE_DRAIN and queues drain_work if no thread is already active. The drain worker reads device event status, fetches records, detects overflow or sticky-nonempty state, and if overflow or saturation is observed it forces BYPASS or FORCE_DRAIN mode until two consecutive clean drains occur. Continuous correctable storms therefore cannot starve drain progress. IRQF_ONESHOT interaction: - Each lane is registered with a primary handler and a threaded handler. - The primary handler returns IRQ_WAKE_THREAD only for urgent or forced-drain service. That is the only case where IRQF_ONESHOT masks the line. - Windowed durable events return IRQ_HANDLED from the primary handler, so the IRQ remains unmasked during the pacing window. - The reopen_timer callback never touches irq_desc state and never unmasks the IRQ. It only flips batch_state to FORCE_DRAIN and queues drain_work if drain_pending was clear. This prevents races that could leave a oneshot IRQ masked or a thread stuck. Allocation and fallback: - Dual-slot mode consumes two MSI-X vectors per revectorable logical source. - If the platform grants fewer than 2 times the number of desired sources, the driver prioritizes sources in this order: fatal or RAS event interrupt, general event-log interrupt, mailbox completion, then lower-value notifications. - Sources that do not receive a pair operate in single-slot mode: ordinary irq_set_affinity plus synchronize_irq is used, but the driver makes no stale-versus-current claim and disables generation-sensitive batching around retarget windows. - If the platform exposes only shared minimum vectors, multiple low-value sources may share one drain worker, but only paired sources receive the generation-tagged handoff guarantee. Affinity ownership boundary: - Default placement uses managed IRQ affinity where supported and publishes hints with irq_set_affinity_hint. - The driver registers irq_affinity_notify on both lanes. Any explicit administrator or userspace-directed affinity change marks override_state as ADMIN_OWNED, cancels pending automatic retarget work, and stops driver-initiated affinity moves. - During ADMIN_OWNED state, CPU hotplug is handled only by generic kernel migration rules required to keep the IRQ on an online CPU; the driver does not pull the IRQ back toward its preferred local CPU until ownership is explicitly returned to DRIVER_MANAGED mode. - This prevents conflict with irqbalance or direct smp_affinity writes while still allowing a policy-controlled automatic mode.

2.4 Architecture

Steady state and handoff model:

[CXL logical source S]
      |
      | device policy selects one MSI-X table entry for S
      v
+----------------------------+
| lane[0] virq / msix entry  |
| lane[1] virq / msix entry  |
+----------------------------+
      |
      | host routes each virq through x86 interrupt-remap backend
      v
[VT-d or AMD IOMMU remap state] --> [x2APIC destination CPU]

Per-source software state:
  active_slot      = 0 or 1
  generation       = monotonically increasing commit id
  route_state      = OLD_ACTIVE | SWITCHING | NEW_ACTIVE
  batch_state      = BYPASS | WINDOW_OPEN | FORCE_DRAIN
  override_state   = DRIVER_MANAGED | ADMIN_OWNED
  drain_pending    = atomic coalescer
  urgent_pending   = atomic urgent marker

Primary-handler decision tree:
  if slot_id != active_slot:
      if drain_pending was 0: queue drain_work
      return IRQ_HANDLED
  if source is urgent-only or batch_state == FORCE_DRAIN:
      set urgent_pending
      return IRQ_WAKE_THREAD
  if source has durable log:
      if batch_state == BYPASS:
          return IRQ_WAKE_THREAD
      open or extend bounded window
      return IRQ_HANDLED
  else:
      return IRQ_WAKE_THREAD

Retarget ordering:
  ctrl_mutex lock
    -> compute target CPU mask from node locality, housekeeping, x2APIC topology
    -> program inactive lane affinity
    -> arch_irq_remap_sync(inactive lane)
    -> program device interrupt policy to inactive MSI-X entry
    -> wait device acknowledgement
    -> route_lock
         route_state = NEW_ACTIVE
         active_slot = pending_slot
         generation++
       unlock
  ctrl_mutex unlock

Formal invariants:
- I1: Only the active slot may initiate immediate servicing; inactive-slot deliveries are always treated as stale late arrivals.
- I2: NEW_ACTIVE is published only after host remap visibility and device acknowledgement, never earlier.
- I3: No lock taken in hardirq or threaded-IRQ context is held across mailbox operations.
- I4: drain_work scheduling is bounded by atomic drain_pending, so stale-lane storms collapse to one outstanding worker per source.
- I5: Admin-owned affinity disables all driver retarget actions except unavoidable kernel migration to an online CPU.
- I6: Batching is enabled only when the source exposes durable event state and a finite max_defer_us; otherwise the source remains immediate-service.

Timer and thread interaction:
  WINDOW_OPEN --timer expiry--> FORCE_DRAIN --queue work or wake thread--> BYPASS or WINDOW_OPEN
  IRQF_ONESHOT masking occurs only on the IRQ_WAKE_THREAD path; timer expiry does not directly mask or unmask the line.

2.5 Implementation Plan

1. Vector provisioning. During CXL port or memdev probe, request MSI-X with PCI_IRQ_MSIX and PCI_IRQ_AFFINITY. For N revectorable sources, attempt to allocate 2N vectors. Build a priority table so fatal or RAS and main event-log sources receive pairs first, with mailbox completion next. Mark remaining sources single_slot_fallback if vector pressure prevents pairing. 2. Lane registration. For each paired source, request_threaded_irq on both virqs with the same source-private data and distinct slot_id values. The primary handler performs only slot classification, atomic flags, timer arm, and optional IRQ_WAKE_THREAD. The thread or workqueue path performs event fetch, clear, and mailbox interaction. 3. x86 locality policy. Compute the preferred CPU mask from dev_to_node, PCI or ACPI PXM, cpumask_of_node, housekeeping_cpumask, and Linux x86 topology helpers that already reflect CPUID-derived package, die, core, and x2APIC relationships. Prefer a non-isolated housekeeping CPU local to the CXL root port or decoder, then same socket, then nearest online fallback. 4. Interrupt-remap completion hook. Add a minimal generic hook, arch_irq_remap_sync or equivalent irq_domain callback, callable only from sleepable worker context with no irq_desc lock held. Intel VT-d implementation waits for completion of the backend’s queued invalidation and interrupt-entry-cache invalidation for the updated IRTE. AMD implementation waits for completion of the interrupt-remap-table invalidation path used after destination updates. Generic code returns -EOPNOTSUPP if absent, which forces single_slot_fallback for that source. 5. Retarget worker. Serialize all lane switching and device-policy writes under ctrl_mutex. Sequence: choose inactive lane, call irq_set_affinity on that virq, call irq_set_affinity_hint, wait arch_irq_remap_sync, issue CXL Set Event Interrupt Policy or equivalent source-policy write to select the inactive MSI-X entry, wait for mailbox or register acknowledgement, then publish active_slot and generation under route_lock. Never hold route_lock across mailbox or arch hook calls. 6. Affinity ownership. Register irq_affinity_notify for each virq. On external affinity change, set override_state to ADMIN_OWNED under route_lock, cancel delayed retarget work, and leave future placement unchanged except for kernel-mandated online-CPU migration. Provide an explicit driver-managed mode toggle to resume automatic placement. 7. Batching engine. Attach a per-source hrtimer and work item. Only sources marked source_has_durable_log participate. Set max_defer_us conservatively, for example 25 to 50 microseconds nominal and never above a hard 250 microsecond cap. The primary handler opens the window. Timer expiry sets FORCE_DRAIN and queues work if drain_pending was clear. The worker fetches CXL event records, checks sticky non-empty and overflow indications, drains records, and if overflow or repeated saturation is present it disables windowing temporarily and forces immediate drains until the storm subsides. 8. Source taxonomy. Classify mailbox completion and fatal or uncorrectable events as source_is_urgent_only. Classify informational, advisory, warning, and correctable log-backed sources as durable and windowable only when the device advertises persistent event records or sticky status semantics. Disable batching for any source that lacks an explicit durable criterion. 9. CPU hotplug and topology changes. Subscribe to CPU hotplug and relevant CXL topology notifications. In DRIVER_MANAGED mode, queue the retarget worker to recompute masks for affected sources. In ADMIN_OWNED mode, do not recompute preferred placement except to survive offlined CPUs. 10. Single-slot behavior. For sources without vector pairs or without a remap-sync hook, use ordinary affinity APIs and optional coalesced drain work, but do not publish any claim that stale and current deliveries are distinguishable. Disable generation-sensitive batching during handoff on those sources.

1. Functional correctness. Add tracepoints logging source id, slot id, generation, route_state transitions, active_slot changes, override_state changes, and whether an interrupt took the stale-lane, window-open, force-drain, or urgent path. Remap-visibility proof. Run on at least three host configurations: Intel VT-d enabled, AMD IOMMU interrupt remapping enabled, and remapping unavailable or disabled. Verify that paired mode is enabled only when the arch hook is supported and that unsupported systems fall back to single-slot mode without claiming route-generation distinction. 3. Inject urgent and windowed events while hrtimer expiries occur close to thread wakeups. Confirm the primary handler wakes the threaded handler only on urgent or forced-drain transitions, that the timer never manipulates irq_desc state, and that no line remains masked after thread completion. 5. Storm and overflow testing. Generate sustained correctable and advisory event storms using platform error injection or synthetic device firmware hooks. Verify bounded CPU usage, max defer never exceeding the configured cap, and automatic bypass when overflow or sticky non-empty conditions persist. 6. Affinity override testing. With irqbalance active and with manual writes to smp_affinity, confirm irq_affinity_notify transitions the source to ADMIN_OWNED, stops driver retargeting, and preserves admin intent across subsequent topology changes except unavoidable online-CPU migration. 7. CPU and topology churn. Repeatedly hotplug CPUs, rebalance nodes, and trigger CXL region or decoder updates while event interrupts fire. Verify retarget workers serialize correctly, old-slot late arrivals remain safe, and no source loses service. 8. Performance evaluation. Measure median and tail latency for mailbox completion and urgent RAS handling, remote-socket interrupt rate, workqueue amplification, and CPU cycles per event on stock Linux versus paired-lane mode. Expected gains are lower remote-interrupt traffic and lower tail latency for urgent local servicing, with reduced CPU overhead during durable correctable-event storms. 9. Negative tests. Force vector scarcity and absence of the remap-sync hook, and confirm the driver enters single-slot fallback, disables generation-sensitive claims, and still services events correctly.

4. Risks & Mitigations

  • Risk: platforms may not expose enough MSI-X vectors for full pairing. Mitigation: rank sources by value, allocate pairs only to the highest-value CXL sources, and place the remainder in explicit single-slot fallback without overclaiming the guarantee.
  • Risk: no generic Linux API currently exposes a remap-visibility barrier with the required semantics. Mitigation: define a minimal sleepable arch_irq_remap_sync hook and gate paired mode on its availability; unsupported backends fall back safely.
  • Risk: stale-lane late arrivals during storms could otherwise create unbounded workqueue churn. Mitigation: use atomic drain_pending coalescing so at most one deferred drain is outstanding per source.
  • Risk: IRQF_ONESHOT plus timer-driven batching could leave an IRQ masked or the service thread stuck. Mitigation: wake the threaded handler only for urgent or forced-drain service, keep timer callbacks out of irq_desc manipulation, and leave windowed interrupts on the non-thread-wakeup path.
  • Risk: durable-log assumptions may be wrong for a given source or firmware implementation. Mitigation: require an explicit durable-state criterion before enabling batching and permanently disable pacing for any source lacking persistent records or sticky state.
  • Risk: user or administrator policy may conflict with automatic locality steering. Mitigation: use managed affinity and affinity notifications, switch to ADMIN_OWNED on external changes, and stop driver-initiated retargeting until ownership is returned.
  • Risk: mailbox latency in the drain path may cause backlog under sustained correctable-event load. Mitigation: bound max defer, force drains on timer expiry, enter bypass mode on overflow or sustained saturation, and keep urgent classes permanently non-batched.
