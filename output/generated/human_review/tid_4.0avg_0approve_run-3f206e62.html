1. Executive Summary

One-Liner

For each L0-owned level-triggered interrupt route, KVM creates at most one outstanding delivery credit per resolved destination only when the interrupt crosses the LAPIC IRR-to-ISR commitment point under an EOI-visible mode, publishes that credit together with the EOI-visibility programming before re-entry, retires it only on an L0-observed matching EOI, and otherwise conservatively reconciles it on any true LAPIC lifetime boundary, route-generation change, or loss of EOI observability so stale EOIs can neither unblock the wrong backend nor strand INTx, remote-IRR, or irqfd resample state.

2. Technical Invention Disclosure

2.1 Problem Statement

KVM on x86 must preserve two safety properties across LAPIC lifetime changes and delivery-mode transitions: first, no pre-boundary pending-delivery artifact may cause a post-boundary injection; second, no post-boundary EOI may retire, resample, deassert, or unmask host-side state that belongs to a pre-boundary or superseded route lifetime. This is hard because Linux virtualization spans software LAPIC state, VMX APICv state, and posted-interrupt state; because only some EOIs are visible to L0; because route destination, vector, and delivery mode can change while an old level interrupt is still logically outstanding; and because nested virtualization can redirect or suppress the observation point. Existing practice often relies on broad reset-time clearing or backend-specific heuristics, but those techniques do not precisely define when a host-side credit becomes outstanding, what exact key a later EOI must match, how retargeting is handled, or what happens when EOI visibility is unavailable.

2.2 Prior-Art Gap

Known art covers pieces of the problem in isolation: local-APIC reset paths clear software state; APICv and posted interrupts accelerate delivery; EOI-exit bitmaps provide selective trapping; IOAPIC remote-IRR and irqfd resample track level-triggered completion; and nested VMX manages L0 or L1 ownership transitions. The gap is the absence of a single contract tying these together around the exact LAPIC service boundary. In particular, prior work does not teach a route-scoped delivery credit that is minted only at the resolved-destination IRR-to-ISR commitment point, tagged with a globally unique route and service identity, armed only when L0 is guaranteed to observe the corresponding EOI, and otherwise reconciled explicitly on visibility loss or route change. That gap is consistent with the retrieved voids around X86_FEATURE_APIC with apic_clear_isr and X86_FEATURE_VMX, which point to an unexplored region between APIC microarchitecture primitives and virtualization lifetime management.

2.3 Proposed Invention

The invention adds a route-scoped delivery-credit layer between KVM irq routing and L0-visible EOI retirement. Each level-triggered host-backed route is given a 64-bit route_id allocated from a per-VM namespace and a 64-bit route_generation incremented on every semantic route change or teardown and reinstall. Each vCPU owns a 64-bit lapic_service_generation incremented on every true LAPIC lifetime boundary. A credit is not created when a source asserts, when irqfd injects, when an MSI is routed, when a PIR bit is set, or when a posted-interrupt notification vector is sent. A credit is created only when delivery reaches the commitment point: the LAPIC service logic for a resolved destination performs the IRR-to-ISR transition for that vector under L0 control. At that moment, if and only if L0 is guaranteed to observe the corresponding EOI in the current mode, KVM allocates an immutable credit object keyed by (route_id, route_generation, destination_vcpu_id, vector, lapic_service_generation, owner_layer=L0), publishes a pointer to that object in the destination vCPU's per-vector armed-credit slot, and programs the EOI-visibility control before the next vmentry. The EOI handler retires only a matching outstanding credit by atomic compare-and-swap, thereby enforcing single retirement. If visibility is absent, lost, or nested away, no EOI-dependent retirement is armed; instead, the affected credits are reconciled explicitly by backend-specific unblock or resample logic. On any true boundary, KVM quiesces the vCPU, synchronously scrubs all pre-boundary pending-delivery machinery that could survive the boundary, increments lapic_service_generation, cancels not-yet-committed deliveries, and reconciles or invalidates any outstanding credits whose future EOI can no longer be trusted.

2.4 Architecture

ASCII overview:

Host level source
  -> KVM irq route { route_id64, route_generation64, trigger=level }
  -> delivery path resolves destination(s)
      -> no credit on assertion, PIR post, or notify-vector send
      -> credit minted only at LAPIC IRR->ISR transition for a concrete destination
          -> immutable credit { route_id, route_generation, destination_vcpu_id, vector, lapic_service_generation, owner_layer=L0, state=OUTSTANDING }
          -> publish per-vCPU armed_credit[vector] via RCU
          -> program EOI-visible mode for that vector before next vmentry

Observed EOI path (L0 visible only)
  -> read armed_credit[vector] with acquire semantics
  -> verify key still matches current route_generation and lapic_service_generation
  -> cmpxchg OUTSTANDING -> RETIRED
  -> invoke backend release: ioapic remote-IRR clear, irqfd resample ack, VFIO INTx unmask, or equivalent

Boundary or visibility-loss path
  -> quiesce vCPU
  -> scrub software LAPIC pending state + APICv state + PI state that could inject stale delivery
  -> increment lapic_service_generation
  -> disarm armed_credit pointers and EOI visibility
  -> reconcile outstanding credits whose EOI is no longer trustworthy

Route-change path
  -> increment route_generation under routing lock
  -> old credits become stale by key mismatch
  -> explicit reconciliation on old route
  -> fresh route may later mint a new credit only if a new IRR->ISR commitment occurs

1. Add per-vCPU state: atomic64_t lapic_service_generation, an array armed_credit[256] of RCU-protected pointers to immutable credit objects, and a small bitmap of vectors whose EOI observability is currently armed by L0. 2. Define the delivery-commitment hook in the LAPIC accept path at the code point that converts pending delivery into in-service state, i.e. the path that sets ISR for the resolved vector and is paired later with apic_clear_isr on EOI. That hook allocates a credit only for level-triggered routes and only when L0 EOI visibility is guaranteed for the current mode. 4. Define the retirement hook in the L0-observed EOI path, including both software EOI emulation and VMX EOI-exit handling, and require cmpxchg-based transition from OUTSTANDING to RETIRED before any backend unblock. 5. Define the boundary detector in KVM LAPIC reset, INIT handling, SIPI-after-INIT activation, IA32_APIC_BASE enable or disable, x2APIC mode change, APICv enable or disable, posted-interrupt descriptor reinitialization, EOI-bitmap ownership change, vCPU load or put that reprograms PI or EOI state, and nested-VMX transitions that change whether L0 or L1 observes EOI. 6. Implement backend reconciliation callbacks for IOAPIC remote-IRR emulation, irqfd resample, and VFIO INTx so an invisible or stale EOI is never required for progress. 7. Integrate route-generation increments into irqfd teardown or reinstall, MSI or IOAPIC redirection updates, delivery-mode changes, destination changes, trigger-mode changes, and source hot-unplug. 8. Add debug and trace visibility: boundary reason, credit issuance, retirement, reconciliation, route-generation bump, and stale-key rejection.

4. Risks & Mitigations

  • Risk: Credit issuance hook placed too early could recreate prior-art heuristics. Mitigation: define and audit the hook strictly at the IRR-to-ISR transition paired with the later ISR-clear path, and reject earlier issuance sites in review tooling.
  • Risk: Route update races could double-create or orphan credits. Mitigation: increment route_generation under the routing write lock before publishing new routing, and require reconciliation of old outstanding credits before reuse.
  • Risk: Weak-memory readers could observe torn tag state. Mitigation: publish immutable credit objects via RCU with release and acquire semantics, and never read mutable multi-field tags locklessly.
  • Risk: Nested virtualization could cause L1-only EOIs to retire L0 credits. Mitigation: include owner_layer in the key and force boundary plus reconciliation whenever L0 visibility is lost or redirected.
  • Risk: Broadcast or lowest-priority delivery could create duplicate retirements. Mitigation: store one credit per concrete committed destination and require cmpxchg on each credit object separately.
  • Risk: Reconciliation could become expensive on large VMs. Mitigation: track only outstanding level-triggered credits and reconcile only affected route or vCPU subsets, keeping cost proportional to k rather than all vectors or all routes.
  • Risk: 64-bit namespace exhaustion is theoretically possible. Mitigation: specify a quiesce and rebuild procedure on impending exhaustion instead of ambiguous wrap comparisons.
