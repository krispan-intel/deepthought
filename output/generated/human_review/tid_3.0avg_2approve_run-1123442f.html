2. Technical Invention Disclosure

2.1 Problem Statement

Linux futex spin-before-park heuristics are tuned for bare metal and cannot observe host-side scheduling delays, nested page-walk costs, or cross-vCPU interference. Under vCPU oversubscription, futex code may spin wastefully when the lock holder is descheduled or may park prematurely when the holder is running but EPT pressure inflates unlock latency. pvqspinlock addresses kernel spinlocks but does not cover userspace futexes, which dominate serialization in database and storage workloads. Orthogonally, KVM's EPT management suffers two structural problems: (a) mmu_lock serializes all EPT root mutations across vCPUs, creating a bottleneck under mmu_notifier-heavy workloads (live migration dirty logging, VFIO DMA map/unmap, transparent hugepage splitting)—note that while tdp_mmu.c introduced read-lock paths for fault handling, write-lock is still required for root allocation, zapping, and mmu_notifier invalidation, and the hold time scales with vCPU count during coordinated flushes; and (b) guest PTI correctness is trusted entirely to the guest kernel, meaning a single kernel bug (e.g., CVE-2017-5754 Meltdown-class transient execution, or implementation errors in PTI page table switching) can silently expose kernel mappings to userspace with no hypervisor-visible signal. No existing mechanism unifies EPT microarchitectural signals (accessed/dirty bits, page-walk metadata) into scheduling policy, reduced-contention root management, and security enforcement in a single coherent framework.

2.2 Prior-Art Gap

pvqspinlock uses vCPU preemption state for kernel spinlocks but does not extend to userspace futexes. Futex adaptive spinning (kernel/futex/futex.h) uses fixed nanosecond budgets without host visibility. EPT accessed/dirty handling is used for KVM memory aging (kvm_age_gfn) and dirty logging (PML), not synchronization policy. No prior art defines a futex-specific EPT sampling layer that translates page heat into scheduling hints. For EPT root management: KVM's current mmu_lock (a spinlock in kvm->mmu_lock) serializes all shadow/TDP page table mutations. The tdp_mmu.c rework introduced read-lock paths for page fault handling but still requires write-lock for root allocation, zapping, and mmu_notifier invalidation. RCU-based approaches to page table management exist in the Linux kernel (e.g., rcu_table_free for process page tables) but have not been applied to EPT roots where INVEPT coordination and VMX EPTP switching add hardware constraints absent from software page tables. Critically, no prior approach uses immutable epoch-versioned snapshots with PD-granularity CoW to reduce mmu_lock hold time while preserving correctness under concurrent mmu_notifier callbacks. For PTI enforcement: Intel SGX EPCM provides hardware-enforced per-page access controls within enclaves, but EPCM operates at enclave granularity with ring-3-only semantics and cannot express kernel/user page-table-level visibility policies across the full guest address space. Nested VMX (L0/L1) EPT switching separates L1-managed L2 address spaces but does not enforce intra-guest privilege separation within a single VM. AMD SEV-SNP RMP provides per-page ownership validation but at host/guest boundary, not kernel/user boundary within the guest. No prior art uses dual-EPTP views with VMFUNC switching to enforce PTI mapping correctness from the hypervisor layer, nor does any prior art address the consistency problem of updating two EPT views atomically under mmu_notifier invalidation.     • The epoch-upgrade-on-EPT-violation mechanism lacks concrete race analysis: when a vCPU checks its per-vCPU generation snapshot against the published generation during fault handling, the published generation can advance between the check and SRCU read-lock acquisition, potentially causing adoption of generation N+2 when the fault occurred under N. The 'generation-pinned SRCU transition' must specify exact memory ordering—whether rcu_dereference() with implied smp_read_barrier_depends() suffices or requires explicit smp_rmb() after generation load—and define the pinning protocol to guarantee the checked generation remains dereferenceable throughout adoption.
    • Memory overhead is claimed 'bounded' but no concrete upper bound or reclamation guarantee is provided. With PD-level (2 MB) CoW, a 64 GB VM has ~32,768 PD entries; if three concurrent generations are live and worst-case invalidation patterns prevent sharing, the metadata overhead scales as 3 × 32,768 × sizeof(ept_pd_entry). You must define maximum memory footprint as a function of guest physical address space size and maximum concurrent generation count, specify the SRCU grace period convergence guarantee that bounds live generations (e.g., 'RECLAIM begins within K SRCU grace periods after DRAIN entry'), and describe the fallback mechanism if memory pressure prevents new epoch allocation.
    • The interaction with transparent huge pages (2 MB/1 GB EPT entries) is completely undefined. When mmu_notifier invalidates a range spanning a transparent huge page, does BUILD-phase construction pre-split the huge SPTE into 4 KB entries in the successor root, or does the new root inherit the huge mapping and defer splitting until a subsequent write fault? If huge pages are pre-split, the 'zero SPTE repopulation' claim is misleading (you populated 512 × 4 KB SPTEs during BUILD). If huge pages are inherited and split on-demand post-adoption, those post-adoption splits require mmu_lock, violating the 'vCPU-side adoption is mmu_lock-free' claim. Specify the huge-page policy and its locking implications.
    • The handling of mmu_notifier callbacks concurrent with ACTIVE-phase roots is critically underspecified and likely violates either performance claims or Linux MM correctness. If mmu_notifier_invalidate_range_start() arrives while an epoch is ACTIVE, you must either (a) immediately zap affected SPTEs in the live ACTIVE root under mmu_lock, which contradicts 'mmu_lock hold time confined to epoch construction,' or (b) defer invalidation visibility until the next epoch BUILD, which violates the Linux MM requirement that invalidation is visible before mmu_notifier_invalidate_range_start() returns. A third option—forcing immediate ACTIVE → DRAIN transition and starting a new BUILD that reflects the invalidation—might be correct, but this must be explicitly specified with proof that no vCPU can fault on stale translations between invalidate_range_start() and the new epoch's PUBLISH.
    • The four-state lifecycle transition conditions and invariants are underspecified. What event triggers BUILD → PUBLISH: is it time-based (periodic epoch tick), threshold-based (N mmu_notifier events accumulated), or on-demand (first EPT violation after invalidation)? If time-based, short epochs waste memory on idle VMs; long epochs delay invalidation visibility. If threshold-based, define the threshold and analyze its interaction with SRCU grace period duration (if grace periods are slower than invalidation arrival rate, generation count grows unbounded). The state machine must specify: (1) the invariant for each state (e.g., 'ACTIVE: exactly one root in ACTIVE state, all vCPUs' generations ≤ ACTIVE generation'), (2) which operations are permitted concurrently in each state (can BUILD and DRAIN overlap?), and (3) the failure-mode abort path if SRCU grace periods stall or memory allocation fails during BUILD.
