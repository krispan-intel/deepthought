2. Technical Invention Disclosure

2.1 Problem Statement

Linux lock slowpaths still rely heavily on static thresholds, scheduler heuristics, and architecture-agnostic pause loops even though contention cost varies sharply by locksite, Xeon generation, socket topology, owner critical-section length, and preemption state. A short spin helps when handoff is imminent; the same spin wastes cycles when the owner is descheduled or stalled behind long-latency retired loads. Existing approaches either profile after the fact, expose PMU data only to userspace tools, or tune individual lock algorithms with generic time-based rules. The practical gap is a kernel-safe mechanism that can observe precise contention cost inside a recognized x86 spin phase, attribute that cost to a stable locksite identity despite KASLR and module churn, and feed back only conservative reductions into existing qspinlock, mutex, and rwsem slowpaths without changing queue semantics, waiter ordering, wake routing, or PREEMPT_RT correctness. The mechanism must also fail closed on unknown CPUs, microcode drift, PMU multiplexing, hypervisors without precise PEBS, hotplug and idle distortion, or any PEBS field mismatch, and it must not let unprivileged perf users bias kernel lock behavior.

2.2 Prior-Art Gap

Prior art falls into four buckets. First, Linux adaptive or optimistic spinning uses owner-running checks, scheduler state, or fixed budgets, but those heuristics cannot tell whether a specific locksite is wasting spin cycles because the owner is stalled on a costly retired load or because handoff is genuinely imminent. Second, lockstat, perf, and similar tools expose contention telemetry to humans or offline autotuners, not to a correctness-bounded in-kernel control plane. Third, PMU-guided tuning literature and comparative studies of performance-monitoring events focus on application analysis or architecture comparison, not on bridging asynchronous PMU samples into kernel lock slowpaths with explicit fairness invariants. Fourth, scalable Unix and HPC code-modernization work uses telemetry to restructure software globally, not to construct a live kernel actuator that is constrained to shorten an existing spin phase and nothing more. None of this art discloses a fail-closed x86 bridge that requires exact PEBS event validation, precise-IP range classification, a CPU-local spin-context cookie, a normalized locksite key, and a monotonic per-primitive actuation matrix.

2.3 Proposed Invention

The invention is a Synchronization Pressure Translation Layer for x86 Linux. At entry to each supported spin phase, the slowpath publishes a per-CPU spin-context cookie `{locksite_key, primitive_id, phase_id, epoch}` using store-release semantics. A kernel-owned pinned Intel PEBS retired-load-latency event, enabled only on an allowlisted family/model/stepping/microcode tuple, produces samples. The NMI or PMI handler accepts a sample only when: the event encoding matches the allowlist entry; `precise_ip >= 2`; the sample is from kernel mode; required fields `IP`, `TIME`, `CPU`, and `WEIGHT` are present and internally consistent; any present `DATA_SRC` field parses consistently with the PEBS format but is ignored for policy; and the sampled IP lands inside a build-time-registered spin-range whose primitive and phase match the current CPU-local spin-context cookie. Accepted samples are appended to a bounded per-CPU ring without allocation or locking. A sleepable aggregator folds those records into per-locksite state, applies minimum sample thresholds, hysteresis, decay, and TTL, and publishes immutable RCU snapshots keyed by `locksite_key`. Each snapshot contains confidence plus primitive-specific caps such as `pending_loop_cap`, `pause_backoff_cap`, `owner_spin_round_cap`, or `osq_round_cap`. Slowpaths consume snapshots only by computing `effective_budget = min(default_budget, snapshot_cap)` for the current spin phase. For qspinlock this can only reduce pre-queue busy looping before queue publication; for mutex and rwsem it can only terminate optimistic spinning earlier and fall through to the existing enqueue or sleep path. The mechanism never infers correctness-critical ownership from cache-coherence metadata, never changes queue insertion, owner handoff, waiter order, or wake routing, and never extends a spin budget beyond stock behavior. On non-Intel, non-PEBS, unknown, or virtualized systems the feature remains disabled and the kernel uses stock heuristics unchanged.

2.4 Architecture

Dataflow: supported slowpath phase entry -> publish per-CPU `spin_context` -> PEBS NMI validates CPU model, microcode, event encoding, required fields, kernel filtering, precise-IP, registered spin-range, and cookie epoch -> append compact record to per-CPU ring -> sleepable aggregator folds records into per-locksite statistics -> publish immutable RCU snapshot -> slowpath reads snapshot and may only clamp the current spin budget. Locksite identity: the synchronous slowpath captures the callsite using a compiler-provided return-IP or explicit site cookie, normalizes it to a relative text offset against the boot-relocated image base, combines it with `primitive_id`, `phase_id`, `lock_class_cookie_or_0`, image UUID or vmlinux build-id, and `module_generation`, and hashes that tuple for table lookup while preserving the full tuple for collision checks. KASLR is neutralized by relative offsets; module unload/reload ABA is blocked by generation numbers and RCU grace periods before reuse; lockdep-off still yields a stable code-site identity. Recognized spin-loop classifier: only exact text intervals emitted by objtool or explicit assembler labels for `native_queued_spin_lock_slowpath`, `mutex_optimistic_spin`, and `rwsem_optimistic_spin` are eligible. Unsupported inlined or alternate implementation paths simply register no interval and therefore cannot generate accepted control samples. Actuation matrix: qspinlock -> allowed knobs are pre-queue pending-loop iteration cap and pause/backoff cap; forbidden transitions are queue publication changes, pending-bit protocol changes, or post-queue fast-path re-entry. Mutex -> allowed knob is owner-spin round cap before `__mutex_add_waiter()`; forbidden transitions are waiter-list ordering changes, wake-order changes, or spinning after waiter insertion. Rwsem -> allowed knob is optimistic owner-spin or OSQ round cap before waiter enqueue or sleep; forbidden transitions are reader/writer handoff changes, post-enqueue fast-path re-entry, or wake-order changes. PREEMPT_RT rule: if a primitive's RT implementation changes semantics or disables its optimistic-spin phase, that primitive's actuator is hard-disabled and the layer remains observational only.

2.5 Implementation Plan

1. Add an x86 PEBS allowlist table keyed by family, model, stepping, and accepted microcode revision range. Each entry stores the exact event select and umask, PEBS record format identifier, permitted programmable counter class, minimum `precise_ip`, required PERF sample fields, latency-field extraction rule from `weight`, and an explicit reject bitmap for malformed samples. 2. Create a kernel-owned pinned perf event group only when the PMU is available, not multiplexed, not already owned by a conflicting kernel user, and not hidden by a hypervisor lacking precise PEBS. If any prerequisite fails, keep the feature static key off. 3. Annotate supported spin loops with start and end labels and emit a `__sptl_spin_ranges` note via objtool or assembler metadata. Register vmlinux ranges at boot and module ranges on load; unregister them on unload after `synchronize_rcu()` and increment `module_generation` so old keys cannot be reused. 4. Insert exact hook points: qspinlock publishes the cookie before entering its pre-queue pending or owner-polling loop and clears it before MCS queue publication; mutex publishes on entry to `mutex_optimistic_spin()` and clears before `__mutex_add_waiter()` or spin exit; rwsem publishes on entry to `rwsem_optimistic_spin()` and clears before waiter enqueue or sleep. 5. Capture uses fixed-size per-CPU records `{ktime, cpu, locksite_key, primitive_id, phase_id, latency_cycles, epoch}`; nested NMI or PMI instances increment a drop counter and never block or retry. 6. Aggregation runs in sleepable context, requires a minimum accepted-sample count, computes confidence and capped budgets, applies decay and TTL, and publishes immutable snapshots with a nonzero minimum floor so actuation only shortens excessive spinning rather than collapsing every spin phase to zero. 7. Slowpaths fetch snapshots with load-acquire and apply only `min(default_budget, snapshot_cap)` while preserving stock owner-running checks, `need_resched`, vCPU-preempted detection, queue insertion, handoff, and wake routing logic. 8. Scheduler and topology safeguards are explicit: qspinlock support is limited to its defined non-RT preempt-disabled phase and never sleeps there; mutex and rwsem actuation is suppressed if their optimistic-spin phase is not active in normal task context. Samples from CPUs in hotplug transition, PMU-throttled state, or idle-suppressed NOHZ periods are excluded from policy updates. 9. Microcode reload, CPU hotplug introducing an unvalidated model, or repeated sample-format mismatches trigger teardown of the event, retirement of all snapshots for the affected domain, and a fail-closed return to stock behavior.

2.6 Validation Plan

Validation has four layers. First, contract tests inject synthetic records and verify fail-closed behavior for unknown family/model/stepping tuples, changed microcode revisions, bad event encodings, missing `weight`, user-mode samples, precise IP outside a registered spin-range, stale module generations, counter multiplexing, and virtualization without precise PEBS. Second, locking-correctness tests use locktorture, mutex and rwsem stress, PREEMPT_RT kernels, CPU hotplug, suspend and idle transitions, and synthetic owner-preemption scenarios to confirm that queue insertion order, waiter order, handoff order, starvation bounds, and lockdep outcomes are identical with the feature enabled versus disabled. Third, performance experiments compare stock Linux against the disclosed layer on multiple Intel server generations and a non-PEBS fallback platform across OLTP, memcached or nginx, kernel builds, HPC-style fine-grained synchronization, and a comparative parallel search workload such as Alpha-Beta pruning, measuring spin cycles, handoff latency, context-switch churn, tail latency, and throughput. Fourth, security and operational tests verify that only the kernel-owned event can feed the layer, that unprivileged perf activity cannot bias snapshots, that a hypervisor disabling precise PEBS forces the static key off, and that enable, disable, and module unload sequences leave no stale locksite entries or ABA collisions.

4. Risks & Mitigations

  • Risk: PEBS skid or misattribution could map a sample to the wrong code region. Mitigation: require `precise_ip >= 2`, reject any IP outside an exact registered spin interval, and require a matching CPU-local spin-context cookie before acceptance.
  • Risk: CPU model drift or microcode updates could silently change PEBS semantics. Mitigation: make microcode revision part of the allowlist tuple, revalidate on boot, hotplug, and microcode reload, and hard-disable on any unknown or changed revision.
  • Risk: Ending spinning earlier could increase sleep or wake churn. Mitigation: apply only confidence-gated caps with hysteresis, decay, TTL, and a nonzero minimum spin floor; never reroute wakeups; and preserve existing scheduler and owner-running eligibility checks.
  • Risk: Module unload or reload could create stale locksite entries or ABA collisions. Mitigation: key entries with image identity plus `module_generation`, unregister ranges with RCU grace periods, and retire snapshots when the generation changes.
  • Risk: PMU ownership conflicts, multiplexing, watchdog constraints, or virtualization could invalidate the telemetry contract. Mitigation: require a kernel-owned pinned event on a validated PMU configuration and disable the feature entirely whenever the contract cannot be maintained.
  • Risk: Unprivileged users could attempt to bias kernel behavior through perf. Mitigation: only kernel-created events feed the layer, user perf sessions cannot program or override the control event, and all snapshots are derived from read-only kernel-owned telemetry.
  • Risk: Vendor lock-in could hinder upstream acceptability. Mitigation: keep the feature optional, strictly additive, and fail-closed; on non-Intel or non-PEBS systems the kernel preserves stock heuristics and ABI behavior with no functional regression.
