2. Technical Invention Disclosure

2.1 Problem Statement

Guests often need to manipulate host-side Linux device drivers that ultimately sleep on slow peripheral buses such as I2C and SPI. A representative example is a virtual PMIC modeled after TPS65218, where boot, brownout recovery, or policy transitions may touch `ENABLE2.GPIO3`, rail-enable bits, reset-control bits, and sequencing registers in quick bursts. Pure KVM MMIO emulation traps every store and cannot safely sleep in the VM-exit path, so the host burns cycles on exits, emulation, and wakeups instead of useful backend work. Naive shared-memory batching is faster but loses the driver-level semantics that matter: register width, masked updates, offset allowlists, ordering, fence behavior, retryability, and side-effect exclusions. The technical problem is therefore to batch guest-authored register intent using x86 EPT transitions while preserving an auditable freeze point, deterministic slot completeness, sleeping-backend safety, and bounded interaction with TDP hugepage promotion.

2.2 Prior-Art Gap

Known approaches each miss a required part of the contract. Classic per-access MMIO trapping is correct but too expensive for register storms. Virtio rings, vhost-user queues, ivshmem, and doorbell-based shared-memory channels batch data movement but do not define a verifiable freeze of ordinary guest RAM before a sleeping Linux backend consumes register intent. KVM dirty logging and dirty ring already use write-protect plus TLB invalidation, but they track page dirtiness for migration and logging rather than validated register operations; they do not encode register offset, width, mask, side-effect policy, guest-visible ABORT/FENCE semantics, or serialized replay into regmap/I2C/SPI backends. Prior art also does not tie a first-write `EPT_VIOLATION_PROT_WRITE` on a single batch aperture to a deterministic slot ABI and to a backend policy table that can safely express `TPS65218_ENABLE2.GPIO3` as a masked update rather than an unstructured guest read-modify-write. Finally, prior art is usually silent on how to confine hugepage disruption to the aperture's containing 2 MiB window and how to bound split quarantine state across memslot movement and promotion churn.

2.3 Proposed Invention

The invention introduces a KVM-managed register-intent aperture with an exact guest ABI, exact host lifecycle, and exact replay contract. 1. Guest-visible memory objects. - One 4 KiB stage page is mapped as ordinary write-back guest RAM through an ordinary memslot. - One separate control page is exposed as always-trapped MMIO and contains command and status registers. - The stage page is initially EPT write-protected in CLOSED state. 2. Slot ABI. - Byte 0..63 of the stage page is a header: `magic`, `abi_version`, `epoch_seq`, `slot_count`, `feature_bits`, and producer-visible status generation. - The remaining space holds 63 cache-line-sized 64-byte slots. - Each slot is little-endian and naturally aligned, with fields `{offset_le16, op_u8, width_u8, flags_le32, value_le64, mask_le64, cookie_le64, seq_le32, crc_le32, reserved_le64, commit_word_le64}`. - A slot is well-formed only if `seq` equals the current epoch, `op/width` are allowed by the provider policy, CRC matches the body, and `commit_word_le64` is written as one aligned 8-byte store encoding `{VALID, epoch_seq, slot_index, crc}`. 3. State machine. - CLOSED: stage GFN is EPT-WP; first eligible guest store traps. - OPEN: KVM has temporarily granted RW on the single stage GFN; guest may fill slots. - CLOSING: guest issued `COMMIT(seq)` or `FENCE(seq)` by MMIO write to the control page; KVM re-write-protects the same GFN, drains remote writable translations, then copies the 4 KiB stage page into kernel memory. - FROZEN: immutable snapshot is queued to a worker; status registers expose `frozen_seq`, `worker_seq`, error bitmap, and terminal outcome. - ABORT: guest may issue `ABORT(seq)` only while OPEN and before freeze completion; KVM re-write-protects the stage GFN, discards the staged epoch, advances `aborted_seq`, and reports ABORTED. 4. Memory ordering and multi-vCPU rules. - Guest software must write all slot body fields, execute `sfence`, perform the final 8-byte aligned store to `commit_word`, execute `sfence`, and only then issue the MMIO `COMMIT` or `FENCE` write. - Multiple vCPUs may write during the same OPEN epoch, but same-slot ownership is first-writer-wins by guest-side atomic claim; non-atomic footer writes, torn writes, same-slot collisions, mismatched sequence numbers, and unsupported widths are ignored for replay and surfaced as per-slot validation errors. - The worker replays slots in deterministic slot-index order, so guest-visible ordering is stable across runs. 5. Backend contract. - Each trusted in-kernel provider registers `kvm_reg_intent_backend_ops` plus a per-offset policy table with entries such as `DIRECT_WRITE`, `UPDATE_BITS`, `BULK_LINEAR`, `NO_BATCH`, `WRITE_ONCE_PER_EPOCH`, `REQUIRE_FENCE_AFTER`, and optional `PRE_DELAY_US` and `POST_DELAY_US`. - Replay is serialized in a worker thread or dedicated kthread, never in the VM-exit path. - Supported backend calls include `regmap_write`, `regmap_update_bits`, `regmap_bulk_write`, `i2c_transfer`, and `spi_sync`; providers declare which are legal. - Auto-retry is permitted only for policy-marked idempotent operations such as masked updates or direct writes to explicitly idempotent registers. Non-idempotent offsets fail terminally and require guest-visible recovery logic. - A `TPS65218_ENABLE2.GPIO3` update is carried as `UPDATE_BITS(offset=ENABLE2, mask=GPIO3_BIT, value=GPIO3_BIT)` and not as a raw guest RMW sequence. 6. Trust and lifecycle. - Only trusted in-kernel providers may call `kvm_ept_aperture_register(kvm, desc)`. - Registration records `stage_gfn`, `control_gfn`, memslot ID and generation, module owner, backend ops, policy table, slot ABI version, and max slot count; KVM takes a module reference and invokes an LSM hook such as `security_kvm_ept_aperture_register()`. - Registration fails closed if the memslot is incompatible, the provider is unauthorized, or the aperture window would exceed the configured per-VM limit. - Unregistration happens on device teardown, VM destroy, memslot delete or move, memslot generation mismatch, or provider unload. If unregistration races with an OPEN epoch, KVM aborts that epoch and reverts future accesses to ordinary MMIO emulation.

EPT leaf policy and hugepage containment.
- Each registered aperture computes a window key `window_gfn = stage_gfn & ~511`, i.e. the 2 MiB EPT window containing the 4 KiB stage page.
- KVM stores aperture state in a per-memslot xarray keyed by `window_gfn`. Each entry is immutable for readers and contains `{reason_bits, memslot_generation, expires_jiffies, provider_id}`.
- `reason_bits` includes `APERTURE_PIN`, plus existing causes such as dirty-log, NX, access tracking, or zap-induced split reasons. Promotion is allowed only when no reason bit remains.
- The hugepage-eligibility lookup occurs under `mmu_lock` in the TDP mapping path after ordinary disqualifiers are known and immediately before large-SPTE installation. This placement prevents accidental creation of a 2 MiB SPTE in a window that must remain split.
- The design does not claim any new EPT encoding; it only ensures that the aperture's own 2 MiB window stays at 4 KiB granularity while other windows remain promotable.

Bounded split quarantine and cleanup.
- One permanent `APERTURE_PIN` entry exists per active aperture window.
- At most one transient cooldown record per aperture window is allowed, used to suppress immediate re-promotion after recent aperture activity.
- Cooldown records are evicted on unregister, memslot move/delete, memslot generation mismatch, or timeout expiry.
- A per-VM cap bounds total aperture windows; registration beyond the cap fails.

Locking and invalidation discipline.
- Readers in the TDP map path do only immutable xarray lookup while holding `mmu_lock`.
- Providers never call back into KVM while holding backend device locks; registration and teardown follow device-config lock -> KVM slots lock -> `mmu_lock` ordering, avoiding AB/BA inversion.
- Memslot updates and teardown invalidate aperture entries using the same lifecycle boundaries as other MMU metadata, so aperture state cannot leak across memslot moves.

Freeze mechanics.
- On `COMMIT` or `FENCE`, KVM marks the epoch CLOSING, walks the TDP roots for the single stage GFN, clears writable permission in every leaf mapping of that GFN, and then calls the existing remote translation drain path, e.g. `kvm_flush_remote_tlbs(kvm)`, which on VMX uses the normal EPT invalidation path before vCPU re-entry.
- COMMIT success is defined only after that drain call returns. Therefore KVM does not snapshot or acknowledge COMMIT while any vCPU can still hold a writable translation for the stage GFN.
- Only after the drain completes does KVM copy the 4 KiB page into a kernel-owned buffer and queue the worker.

2.5 Implementation Plan

Phase 1: Define ABI and registration surface. - Add `kvm_ept_aperture_register()` and `kvm_ept_aperture_unregister()` as internal KVM APIs for trusted in-kernel providers. - Define descriptor fields: `stage_gfn`, `control_gfn`, memslot ID/generation, module owner, policy table pointer, backend ops, ABI version, max slots, and capability flags. - Add an LSM authorization hook and module-refcount ownership checks. Phase 2: Implement state machine and control-page commands. - Add CLOSED/OPEN/CLOSING/FROZEN/ABORTED states and seq accounting. - Implement first-write handling by starting with stage GFN EPT-WP and recognizing eligible `EPT_VIOLATION_PROT_WRITE` on that single GFN. - Implement control-page commands `COMMIT`, `FENCE`, `ABORT`, and `STATUS` as trapped MMIO operations. Phase 3: Implement single-GFN freeze path. - Add a helper that write-protects only the stage GFN across all relevant TDP roots. - After permission clearing, invoke the existing remote TLB drain path and define COMMIT completion as the return point of that drain. - Copy the 4 KiB stage page into a kernel snapshot buffer only after the drain completes. Phase 4: Add validator and replay worker. - Validate header version, epoch sequence, CRC, footer commit word, slot ownership, width legality, alignment, and per-offset policy. - Convert validated slots into serialized backend calls: `regmap_write`, `regmap_update_bits`, `regmap_bulk_write`, `i2c_transfer`, or `spi_sync`. - Add per-slot status, worker error codes, retryability metadata, and terminal outcome publication. Phase 5: Add hugepage-containment policy. - Add per-memslot xarray entries keyed by the 2 MiB aperture window. - Consult the reason bitmap under `mmu_lock` after normal TDP hugepage disqualifiers and before installing a large SPTE. - Reuse the same split-reason accounting style as existing churn causes so aperture logic composes with dirty logging, NX, and zap paths. - Add bounded cooldown and cleanup on memslot generation changes and teardown. Phase 6: Integrate sample providers. - Prototype a regmap-backed PMIC provider with policies for `TPS65218_ENABLE2.GPIO3`, rail-enable registers, and no-batch side-effect registers. - Prototype an I2C-only provider and an SPI provider to prove backend abstraction. - Default failure mode is safe fallback to ordinary MMIO emulation if registration is denied or aperture state becomes invalid.

4. Risks & Mitigations

  • Risk: COMMIT latency rises if remote translation drain is expensive on large vCPU counts. Mitigation: keep the aperture to one stage GFN, batch many register intents per epoch, and expose telemetry so guests choose COMMIT cadence deliberately.
  • Risk: Guest software violates slot-ordering or writes torn footers. Mitigation: require aligned 8-byte commit-word stores, validate CRC and sequence, ignore malformed slots, and publish precise error bits in STATUS.
  • Risk: Same-slot races across vCPUs create ambiguous intent. Mitigation: require guest-side atomic slot claim, define first-writer-wins semantics, replay in slot-index order, and reject colliding malformed slots.
  • Risk: Aperture state could suppress hugepage promotion too broadly. Mitigation: key the policy by the containing 2 MiB window only, bound aperture count per VM, use explicit cooldown expiry, and clean entries on memslot generation changes.
  • Risk: Lock inversion between backend drivers and KVM MMU code. Mitigation: forbid provider callbacks under `mmu_lock`, use immutable lookup in the TDP fast path, and enforce registration/teardown lock ordering.
  • Risk: Auto-retry could duplicate non-idempotent side effects. Mitigation: encode retryability in the per-offset policy table and auto-retry only explicitly idempotent or masked-update operations.
  • Risk: Trust boundary abuse by arbitrary kernel modules. Mitigation: restrict registration to trusted in-kernel providers, hold module references, invoke an LSM authorization hook, and fail closed to ordinary MMIO emulation on denial.
  • Risk: Prior-art challenge that this is just dirty logging repurposed. Mitigation: emphasize claim elements that dirty logging lacks: well-formed semantic slots, control-page COMMIT/FENCE/ABORT, backend policy enforcement, and serialized sleeping-backend replay.
