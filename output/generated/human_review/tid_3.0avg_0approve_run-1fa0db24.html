2. Technical Invention Disclosure

2.1 Problem Statement

On x86 KVM hosts using EPT and the TDP MMU, large amounts of guest memory activity are absorbed by guest page tables, EPT structures, and TLB state instead of producing host-side young/reference signals at folio granularity. Linux reclaim, including MGLRU and memory-tier demotion logic, can therefore under-age the host folios backing actively used guest RAM. The consequence is avoidable refaults, inefficient reclaim, bad fast-tier eviction decisions, and latency spikes, especially on hosts running memory overcommit, tiered DRAM+CXL memory, NUMA-asymmetric memory, or guests using large mappings. The hard problem is not merely seeing that a guest page was accessed; it is turning a bounded subset of EPT observations into a host-MM signal that is race-safe, flush-correct, fair to other workloads, and legal under KVM MMU locking and mmu_notifier rules.

2.2 Prior-Art Gap

Linux page-idle tracking and generic accessed-bit harvesting operate on different semantics and do not define a KVM-specific bridge from second-level translation activity into host folio aging. Broad suggestions to use EPT young/accessed bits for aging also fail to specify the production-critical details that determine correctness and novelty: where in the TDP MMU path the SPTE is sampled; how to avoid O(guest-memory) scans via a bounded watchlist; exactly when clearing an EPT accessed bit is allowed and which invalidation type is needed; how clearing is tied to one EPTP generation so delta meaning survives remote TLB behavior and EPTP switching; how a later GFN-to-HVA-to-PFN-to-folio resolution is protected by memslots SRCU and a kvm->mmu_invalidate_seq retry loop; which translation helper is used and that it may sleep only in deferred worker context; how a stable folio reference is taken; which PFNs and folio classes must be rejected, including !pfn_valid(), reserved PFNs, ZONE_DEVICE/devmap, DAX, special mappings, KSM, hugetlb, unevictable folios, balloon-managed pages, and swapped-out or remapped pages; how huge EPT leaves are excluded or handled without bias; and how any reclaim or tiering effect remains advisory and pressure-gated so it does not distort MGLRU fairness or become an implicit migration engine.

2.3 Proposed Invention

Introduce a per-VM Access-to-Reclaim Bridge (ARB) in KVM x86 TDP MMU. ARB keeps a fixed-size watchlist keyed by 4K GFNs and samples at most a configured budget per epoch. Each watch entry stores at least {gfn, memslot_id, memslot_generation, expected_leaf_level=4K, eptp_generation, source_class, last_seen_epoch, unstable_count, cooldown_until}. Watchlist population is dual-source to avoid bias: one lane is event-driven from newly installed or faulted 4K TDP leaves, and one lane is a low-rate round-robin reservoir over memslots so long-lived resident pages can still be sampled. Each memslot and each NUMA node has a cap, so scan cost and metadata remain O(budget) and bounded. During sampling, ARB walks only watchlist-selected 4K entries through existing TDP leaf iteration and SPTE update machinery. For each candidate, it revalidates memslot generation and leaf size, atomically tests the EPT accessed bit, and optionally clears it only if all of the following are true: the entry still belongs to the same active EPTP generation; the VM has remaining clear budget; no more than one single-context INVEPT will be needed for the current batch; and the kernel's existing remote TLB invalidation path can publish that single-context INVEPT to all vCPUs that can execute the affected EPTP generation before the next epoch. ARB never requests global INVEPT. If more than one flush domain would be needed because of EPTP switching, root churn, nested virtualization, or unsupported CPU behavior, ARB falls back to observe-only mode and leaves the A-bit set. The MMU-lock section produces tokens only, for example {vmid, gfn, memslot_id, memslot_generation, leaf_level=4K, eptp_generation, epoch, timestamp, mode=observe|delta}. No operation that can sleep or fault is permitted while holding MMU locks. In particular, ARB does not translate to PFN, does not touch host folios, and does not call into reclaim or migration code in the SPTE path. A deferred worker consumes tokens in sleepable context. The worker executes the exact safety sequence: (1) `idx = srcu_read_lock(&kvm->srcu)`; (2) `seq = READ_ONCE(kvm->mmu_invalidate_seq)`; (3) `smp_rmb()`; (4) resolve the memslot and check that `token.memslot_generation` still matches; (5) `hva = gfn_to_hva_memslot(slot, token.gfn)` and drop on error HVA; (6) translate through the retry-capable, sleepable KVM path `hva_to_pfn_retry()`; this helper is allowed to sleep or fault, but only here, outside MMU locks; (7) `smp_rmb()` and abort if `mmu_invalidate_retry(kvm, seq)` or memslot-generation recheck says the mapping raced with invalidation, unmap, split, merge, migration, ballooning, or remap; (8) reject non-aging-safe PFNs and folios; (9) convert `pfn_to_page()` to `page_folio()`, take `folio_get()` before dropping the KVM PFN reference, and proceed only if the folio is still LRU-eligible and ordinary memory; (10) apply dedup, cooldown, and pressure gates; (11) call `folio_mark_accessed(folio)` and optionally `memory_tier_advise_folio(folio, MTIER_HINT_HOT)`; (12) `folio_put()` and `kvm_release_pfn_clean()` / equivalent release path. The worker rejects and drops the token, rather than faulting memory in for heat, whenever the mapping is stale or unsuitable. That includes swap-out or refault-needed cases, KSM pages, balloon pages, hugetlb or device/DAX/devmap memory, special mappings, MMIO, unevictable folios, or any folio not on an LRU path. The optional `memory_tier_advise_folio()` hook is explicitly advisory only: it can update a timestamp or demotion-score sideband consulted by existing tier policy, but it cannot call migration directly and cannot override reclaim. Huge EPT leaves are non-goals for direct host aging in the claimed core design. If ARB encounters a 2M or 1G leaf for a watched region, it records leaf-size instability, defers that region, and waits for a natural KVM split caused by existing mechanisms; ARB itself never forces a split merely to harvest heat.

2.5 Implementation Plan

1. Per-VM state and watchlist algorithm. Add ARB state to KVM x86 VM context: fixed-size watchlist, token ring, epoch counter, clear-budget counter, INVEPT-batch accounting keyed by EPTP generation, dedup hash keyed by folio or GFN, and per-memslot / per-node quotas. Implement a two-lane watchlist lifecycle: lane A is event-fed from newly installed or faulted 4K TDP leaves; lane B is a bounded round-robin reservoir over memslot GFNs with a stride so the system does not observe only recently faulted pages. Each entry expires after N cold epochs, on memslot-generation mismatch, on leaf-size instability above threshold, or on repeated translation failure. This keeps scan complexity O(budget) and prevents drift into quasi-global coverage. 2. Sampling path. Reuse existing TDP leaf iteration and SPTE atomic update machinery. Sample only if the current leaf is 4K. If the watched GFN is now covered by a 2M or 1G leaf, set `unstable_count++`, skip heat conversion, and either keep the entry pending for a later natural split or evict it after a small threshold. Do not request a split. This avoids falsely converting one A-bit that covers a large region into per-folio heat. 3. Accessed-bit clear rules. A clear is legal only when the sampled SPTE is a 4K leaf in the expected memslot generation and current EPTP generation, the per-epoch clear budget is not exhausted, and the VM has not already consumed more than one single-context INVEPT for that EPTP generation in the epoch. ARB never coalesces entries from different EPTP generations into one flush batch. If vCPU scheduling or root churn causes a second live generation to appear before the batch flush point, remaining entries for the older generation become observe-only and no extra INVEPT is issued by ARB for them. 4. INVEPT publication. When any A-bit was cleared in a batch, ARB uses KVM's existing remote TLB invalidation machinery to publish a single-context INVEPT for that EPTP generation. The correctness requirement is that every vCPU currently executing or later scheduled with that EPTP generation must observe the cleared state before that same generation is eligible for the next delta epoch. ARB does not require global INVEPT and does not redefine KVM's TLB model; it binds its accounting to the model that already exists. If a platform or VM mode cannot provide that property cheaply or deterministically, ARB degrades to observe-only sampling. 5. Deferred translation contract. Token processing occurs only in worker context with no MMU lock held. The exact sequence is: `idx = srcu_read_lock(&kvm->srcu)`; `seq = READ_ONCE(kvm->mmu_invalidate_seq)`; `smp_rmb()`; locate memslot; validate `token.memslot_generation`; compute HVA with `gfn_to_hva_memslot()`; translate with sleepable `hva_to_pfn_retry()`; `smp_rmb()`; if `mmu_invalidate_retry(kvm, seq)` or memslot generation changed, release any PFN and drop the token; otherwise convert PFN to folio. `hva_to_pfn_retry()` may sleep or fault, which is acceptable only because it runs here. No helper that can sleep is ever called while MMU locks are held. 6. Folio lifetime and filter path. After translation, reject `!pfn_valid(pfn)`, reserved PFNs, device/devmap/ZONE_DEVICE memory, DAX or special mappings, MMIO, hugetlb, KSM, balloon-managed pages, and any folio failing `folio_test_lru()` or marked unevictable. Take `folio_get(folio)` before dropping the KVM PFN reference so the folio remains stable across the advisory call. If the mapping required swap-in, major refault, or other slow recovery to materialize a page, drop the sample; ARB never pages memory in solely to create heat. If the folio changed because of migration, reuse, remap, or invalidation, the seq and generation rechecks force a drop. 7. MM integration and fairness. `folio_mark_accessed()` is used only as an advisory signal. Before calling it, consult per-VM, per-memcg, and per-node token buckets plus a cooldown timestamp so one guest cannot continuously re-promote its entire working set. Permit optional `memory_tier_advise_folio(folio, MTIER_HINT_HOT)` only when node demotion pressure or memory-tier policy says such advice is relevant. That hook can update advisory metadata only; it must not call migration or rebalance directly. Under low pressure or if a memcg has consumed its quota, ARB drops the sample instead of forcing a heat mark. 8. Disablement matrix and graceful degradation. Disable ARB entirely for nested virtualization, confidential VMs, EPT A/D disabled, shadow MMU or non-TDP MMU, and mappings backed by unsupported memory types. In supported VMs, if budgets are exhausted, if EPTP generations churn, if the invalidation sequence races, or if translation/filter checks fail, ARB simply drops tokens or uses observe-only mode. Baseline KVM reclaim behavior is unchanged; the feature cannot make reclaim worse by continuing to emit stale or overly strong signals. 9. Teardown. On VM teardown, memslot deletion, root zap, or feature disable, cancel workers, advance generation numbers, and let queued tokens fail their revalidation checks. No best-effort repair of stale tokens is attempted.

4. Risks & Mitigations

  • Risk: clearing EPT accessed bits can amplify invalidations and erase delta meaning. Mitigation: clear only on 4K leaves, only under explicit clear budget, only when a single-context INVEPT can cover one EPTP generation batch, and otherwise fall back to observe-only mode.
  • Risk: stale GFN-to-folio translation during memslot churn, unmap, ballooning, or migration. Mitigation: process tokens only under memslots SRCU, snapshot `kvm->mmu_invalidate_seq`, use `mmu_invalidate_retry(kvm, seq)` after translation, recheck memslot generation, and drop rather than repair stale tokens.
  • Risk: sleeping or faulting while MMU locks are held. Mitigation: forbid all PFN translation and folio work in the SPTE path; use sleepable `hva_to_pfn_retry()` only in deferred worker context.
  • Risk: unsafe host aging of unsupported memory. Mitigation: reject `!pfn_valid()`, reserved PFNs, devmap/ZONE_DEVICE, DAX, special mappings, MMIO, hugetlb, KSM, balloon pages, unevictable folios, and any folio not on an LRU path; take a stable folio reference before `folio_mark_accessed()`.
  • Risk: huge EPT leaves can misattribute heat and bias reclaim. Mitigation: make 4K leaves the only claimed heat-to-folio path, record leaf-size instability, and never force a hugepage split merely to sample heat.
  • Risk: ARB could distort MGLRU fairness or become a migration engine. Mitigation: keep all outputs advisory only, gate by memcg/node/tier pressure, apply cooldown and quotas, and restrict optional tier advice to metadata updates that existing policy may ignore.
  • Risk: watchlist bias toward recently faulted pages. Mitigation: use a dual-source watchlist with both event-fed and round-robin reservoir entries, enforce per-memslot and per-node caps, and expire cold or unstable entries to preserve O(budget) behavior.
  • Risk: unsupported VM modes break assumptions. Mitigation: explicit disablement for nested virt, confidential VMs, EPT A/D disabled, non-TDP MMU, and unsupported memory types, with graceful fallback to baseline behavior and no reclaim regression.
  • Risk: prior-art overlap with broad EPT-hotness ideas. Mitigation: claim only the narrow bridge contract combining bounded 4K sampling, single-context-INVEPT-per-generation clearing, invalidation-safe deferred translation, strict folio filtering, and advisory-only MM integration.
