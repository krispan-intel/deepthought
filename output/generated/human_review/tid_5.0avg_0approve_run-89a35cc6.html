2. Technical Invention Disclosure

2.1 Problem Statement

Linux exposes powerful but fragmented controls across intel_rapl powercap sysfs, thermal governors, intel_pstate and HWP EPP, cgroups, and eBPF. Existing practice typically relies on static administrator settings or user-space daemons that discover workload changes late, race with kernel subsystems, and update package power constraints one file at a time, which can transiently expose mixed PL1, PL2, and tau values. Existing mechanisms also lack a precise privilege model for cgroup-scoped workload intent, a deterministic exclusivity contract for package ownership, a bounded way to survive cgroup deletion and delegation, and a publication model that lets readers correlate committed tuple, veto reasons, and broker state without cross-file ambiguity. The result is poor ownership semantics, oscillation risk, and weak tenant-awareness precisely where x86 package power control is most sensitive.

2.2 Prior-Art Gap

intel_rapl powercap already serializes individual hardware writes, but it does not define broker ownership, lease expiry, rollback to a prior tuple on commit failure, or a coherent publication snapshot spanning tuple plus reasons plus state. Thermal power_allocator and related governors may clamp power, but they are thermal-first controllers; they do not ingest bounded per-cgroup intent from eBPF, do not expose a package-zone exclusivity lease, and do not preserve a saved baseline tuple for deterministic return to legacy control. schedutil, uclamp, and intel_pstate control frequency selection rather than package RAPL tuples and therefore do not teach a package-scope transaction across PL1, PL2, and tau. Generic BPF hinting patterns are also insufficient: a BPF program writing a generic hint to a map is well-trodden prior art, but a fixed-size kernel-owned per-cgroup descriptor with init_user_ns privilege checks, optional delegated self-targeting, TTL bounds, anti-DoS rate limits, top-K package attribution, and a rollback-verified powercap commit transaction is materially narrower and more concrete. The topological void here is the missing causal bridge between cgroup intent and package power ownership semantics, not mere observability or a generic feedback loop.

2.3 Proposed Invention

The invention introduces a package-local broker for Intel RAPL package domains plus an optional HWP EPP follower. 1. Intent ABI. A sleepable eBPF kfunc accepts a fixed-size intent record: version, flags, urgency in 0..1024, min_share_permille in 0..1000, max_share_permille in 0..1000, burst_permille in 0..1000, ttl_ms in 1..1000, and optional epp_hint in 0..255 or disabled. One active descriptor exists per cgroup, so writes overwrite in place and memory growth is bounded. Arbitrary target cgroup updates require CAP_SYS_ADMIN in init_user_ns plus an LSM hook. Optional delegation is explicit: a parent admin may enable a cgroup v2 subtree flag power_intent.delegate=1, after which a BPF program in that subtree may self-target only its own cgroup and may not modify ancestors or siblings. Each write takes a css reference; on cgroup offlining the descriptor is marked dying and retired after an RCU grace period so the worker never dereferences freed state. 2. Sysfs ABI and exclusivity. For each package zone at /sys/devices/virtual/powercap/intel-rapl:N, the broker adds broker_mode rw 0644, broker_lease_ms rw 0644, broker_state ro 0444, broker_generation ro 0444, broker_writable_mask ro 0444, broker_baseline_tuple ro 0444, and broker_snapshot ro 0444. These files are host sysfs objects, not namespaced; writes require init_user_ns privilege. Writing broker_mode=exclusive captures the current administrator tuple as baseline, probes lock bits and encodability to compute broker_writable_mask, increments topology generation, and enters ARMED. While broker_mode is exclusive and state is ARMED, BROKER_OWNED, FROZEN, or FIRMWARE_VETO, direct user writes to existing constraint_0_power_limit_uw, constraint_0_time_window_us, and constraint_1_power_limit_uw return -EBUSY. Reads from legacy constraint files continue to show actual hardware state. Existing tools therefore remain readable but cannot silently race the broker. No write redirection occurs. 3. Lease model. broker_lease_ms defines a bounded liveness lease, for example 100 to 5000 ms. The kernel worker renews the lease deadline on each successful evaluation cycle, not only on power changes, so exclusivity cannot persist after a wedged worker. If the deadline expires outside suspend, the broker restores broker_baseline_tuple if writable, publishes reason LEASE_EXPIRED, clears exclusivity, and returns the zone to NORMAL. During suspend the state becomes FROZEN and lease expiry is paused until resume. 4. Arbitration inputs and top-K attribution. Every 50 ms by default, clamped to 20 to 200 ms by SKU policy, the broker flushes cgroup_rstat for CPUs in the package, computes per-cgroup package runnable share from scheduler runnable accounting, and maintains a min-heap of the top K=8 cgroups by weighted score. The score is weighted_share = runnable_share multiplied by 0.5 plus urgency divided by 2048 plus burst_permille divided by 2000. Complexity is O(touched_cgroups log K) with a hard cap of 256 active descriptors per package; overflow sets a SATURATED reason bit and lower-ranked descriptors are ignored until decay. The controller also samples package energy delta from RAPL, APERF/MPERF delivery ratio, package thermal headroom, and package power-limit or thermal throttle status. These signals make the attribution materially different from generic utilization control because only the top contributing and recently runnable cgroups influence the tuple. 5. Control-loop math and stability. The broker computes demand_score = clamp(0.50 times topK_runnable plus 0.30 times topK_urgency plus 0.20 times topK_burst, 0, 1). It then computes pl1_target = clamp(P_admin_min plus demand_score times (P_admin_max minus P_admin_min) plus k_h times thermal_headroom_w minus k_t times throttle_penalty_w, P_admin_min, P_admin_max), with k_h bounded so one degree C of headroom cannot create more than 1 W of change and throttle_penalty_w defaulting to 4 W on an asserted thermal or power-limit status bit. pl2_target = clamp(pl1_target plus burst_gain_w, pl1_target, P2_admin_max), where burst_gain_w is derived from topK_burst and capped at 8 W. tau_target is selected from a per-SKU table of encodable RAPL windows by choosing the nearest supported value to clamp(tau_base times (1.25 minus 0.75 times topK_burst plus 0.50 times thermal_pressure), tau_min, tau_max). To avoid chatter the loop applies 1 W hysteresis, step limits of at most 2 W per period for PL1, 4 W per period for PL2, and one encoded step for tau, and resets any integral or history term on topology change or resume. 6. Atomic tuple transaction. The committed set for the package domain is constraint_0_power_limit_uw, constraint_0_time_window_us, and constraint_1_power_limit_uw, plus optional synchronized HWP EPP as a follower value. At enable time the broker defines the atomic writable subset with broker_writable_mask. Unsupported or firmware-locked fields are never partially claimed; if a requested change touches an unwritable field, the entire transaction is vetoed with UNWRITABLE_FIELD and no hardware change is made. Under the existing intel_rapl and powercap serialization point, the broker snapshots the old writable tuple, encodes the new writable tuple, performs writes in a monotone-safe order selected by whether the tuple is tightening or relaxing power, rereads the programmed state, and if any field mismatches or fails to encode, restores the old tuple before releasing the lock. The software-visible atomicity claim is therefore: readers see either the prior tuple or the new verified tuple for the writable subset, never an acknowledged mixed generation. 7. Publication correctness. The broker maintains a double-buffer snapshot protected by a seqcount-latch style publication primitive. Each snapshot contains generation, topology_generation, state, pl1_uw, pl2_uw, tau_us, optional epp, writable_mask, reason_bits, and lease_expiry_boottime_ns. Writers populate the inactive snapshot fully under zone serialization and publish it in one latch flip. broker_snapshot returns the full record in one read. broker_generation is a convenience counter only. Because generation, reasons, and tuple live inside the same published snapshot, sysfs readers can retry until they copy one internally consistent record and will never observe a new generation with stale reason bits. 8. HWP EPP follower and precedence. If Intel HWP is available and the admin enables FOLLOW_EPP, the broker maps aggregated urgency to an EPP target through the existing intel_pstate path only after the RAPL tuple transaction succeeds, using the same generation number in broker_snapshot. Thermal and firmware constraints always have highest precedence, then hardware writable_mask, then administrator min and max envelope, then broker tuple, and finally HWP EPP as an advisory follower. User-space writes regain control only when broker_mode leaves exclusive.

2.4 Architecture

Data path:

[cgroup BPF sleepable program]
        |
        | kfunc: set fixed-size power intent
        v
[per-cgroup kernel descriptor, css-ref tied, TTL bounded]
        |
        | package scorer flushes cgroup_rstat + runnable attribution
        v
[top-K heap per package] + [RAPL energy] + [APERF/MPERF] + [thermal/throttle bits] + [optional HWP EPP state]
        |
        v
[broker controller]
        |
        | validate against baseline envelope + writable mask + lease + state
        v
[transaction builder]
        |
        | old tuple snapshot -> encode -> write in safe order -> verify -> rollback on mismatch
        v
[intel_rapl/powercap serialized commit path]
        |
        +--> [optional intel_pstate HWP EPP follower, same generation]
        |
        v
[seqcount-latch published broker_snapshot]
        |
        +--> broker_snapshot, broker_state, broker_generation
        +--> tracepoints for commit and veto

State machine:

NORMAL -- broker_mode=exclusive --> ARMED
ARMED -- first successful evaluation --> BROKER_OWNED
BROKER_OWNED -- thermal or firmware clamp --> FIRMWARE_VETO
FIRMWARE_VETO -- clamp clears --> BROKER_OWNED
BROKER_OWNED -- suspend_prepare or package hot-remove --> FROZEN
FROZEN -- resume and reprobe --> ARMED
BROKER_OWNED -- lease expiry, no valid worker progress, or admin disable --> NORMAL
ARMED/BROKER_OWNED/FIRMWARE_VETO -- unwritable field or verify failure --> publish veto snapshot, remain in current state unless lease expires

Suspend, hotplug, and lifetime rules:
- Intent TTLs use CLOCK_BOOTTIME, so hints can naturally expire across long suspend intervals.
- On suspend the workqueue is frozen, state becomes FROZEN, and rate-limit timestamps are not used to accumulate credit.
- On resume the broker drops expired intents, reprobes package topology and writable_mask, sets last_eval_ns and last_commit_ns to now, resets controller history, republishes state, and only then allows a fresh control step.
- On CPU hotplug or package rebinding the topology_generation increments and top-K attribution is recomputed from the new CPU set before any new tuple is committed.

2.5 Implementation Plan

1. Add core data structures: broker descriptor, per-package broker state, fixed-size per-cgroup intent record, top-K heap, and published snapshot record. 2. Add the sleepable eBPF kfunc and verifier contract. Reject unknown versions, out-of-range fields, ttl_ms above 1000, more than one update per cgroup per control period, or non-self targeting from delegated subtrees. Wire the LSM hook and init_user_ns capability checks. 3. Attach descriptor lifetime to cgroup lifetime. Use css references on update, RCU-protected lookup during scoring, and offlining callbacks to mark descriptors dying and retire them after grace period. 4. Add package-zone sysfs objects broker_mode, broker_lease_ms, broker_state, broker_generation, broker_writable_mask, broker_baseline_tuple, and broker_snapshot. broker_mode write transitions are serialized under the powercap zone lock. Enabling captures the baseline tuple and computes broker_writable_mask from discovered writable constraints and firmware lock state. 5. Implement the scoring loop on a freezable unbound delayed_work. Every period: flush per-package cgroup statistics, drop expired descriptors, compute top K, read package energy delta, read APERF/MPERF delivery, read thermal and power-limit status, and derive demand_score. Maintain strict complexity bounds by capping active descriptors per package at 256 and K at 8. 6. Implement deterministic control math and clamps. Default update period is 50 ms. Apply 1 W hysteresis, delta limits of 2 W for PL1 and 4 W for PL2 per period, one-step tau hysteresis, and controller reset on resume or topology changes. Choose tau only from the SKU's encodable RAPL windows. 7. Implement the rollback-capable transaction. Under the intel_rapl serialization point: copy old writable tuple; validate that requested fields are a subset of broker_writable_mask; encode new tuple; write in safe order according to whether the broker is tightening or relaxing the envelope; reread hardware; on any mismatch or failure, rewrite the old tuple before unlock and publish a veto snapshot with reason COMMIT_ROLLBACK or VERIFY_FAIL. Only after verify success does the broker increment generation and publish the new snapshot. 8. Implement publication and telemetry. Use a double-buffer seqcount-latch publication model so broker_snapshot returns one coherent record. Add tracepoints for commit, rollback, veto, lease expiry, and state transitions. Include generation and topology_generation in every event. 9. Implement precedence and HWP EPP follower. Thermal and firmware ceilings clamp the computed tuple before transaction build. If FOLLOW_EPP is enabled, derive epp_target from aggregated urgency only after a successful tuple commit and push it through the existing intel_pstate path with the same generation marker. 10. Implement suspend and hotplug handling. On suspend_prepare move to FROZEN, cancel or freeze work, and pause lease countdown. On resume drop expired intents, set last_eval_ns and last_commit_ns to current boottime so a long sleep cannot create a giant immediate step, reprobe writable_mask, and re-enter ARMED. On package hot-remove or zero online CPUs, publish OFFLINE or FROZEN state and stop committing until CPUs return.

4. Risks & Mitigations

  • Risk: reviewers may argue the broker is an obvious wrapper around existing powercap locking. Mitigation: claim the writable-mask transaction with rollback, the -EBUSY exclusivity contract, and the generation-coupled snapshot published at the same serialization point as the distinguishing nucleus.
  • Risk: top-K runnable attribution may be challenged as either obvious or too expensive. Mitigation: bind it to package-local cgroup_rstat plus scheduler runnable share, hard-cap active descriptors, fix K, and emphasize the causal role in package-scope intent arbitration rather than generic accounting.
  • Risk: delegated cgroups could become a privilege escalation path. Mitigation: default deny, require explicit subtree opt-in, restrict delegated mode to self-target only, and route every update through init_user_ns and LSM checks for non-self targets.
  • Risk: long suspend intervals could create a large immediate post-resume power step. Mitigation: use CLOCK_BOOTTIME for TTL expiry, move to FROZEN during suspend, reset controller history and last-eval timestamps on resume, and require a fresh evaluation before any commit.
  • Risk: partial hardware support or firmware lock bits could undermine the atomicity claim. Mitigation: define atomicity only over the discovered writable subset, veto any request touching an unwritable field, and preserve the old tuple on any failed verification.
  • Risk: HWP EPP coupling could be read as overlapping prior art. Mitigation: position EPP as an optional follower derived from the same broker generation after successful RAPL commit, not as the invention's independent center of gravity.
  • Risk: control-loop oscillation across SKUs. Mitigation: use per-SKU tau tables, fixed step limits, 1 W hysteresis, clamp order with thermal and firmware first, and explicit telemetry for saturation and veto causes.
