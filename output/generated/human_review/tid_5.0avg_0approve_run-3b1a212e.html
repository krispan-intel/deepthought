2. Technical Invention Disclosure

2.1 Problem Statement

High-speed Linux networking needs deterministic RX refill and stable TX admission in NAPI and softirq context, but allocation, pinning, page selection, and dma_map or dma_unmap on the fast path introduce jitter and can amplify IOTLB pressure. The problem becomes harder when operators want to source buffers from a non-DRAM NUMA node backed by CXL Type-3 memory, because such memory has different latency and bandwidth behavior, long-term pin eligibility depends on how Linux onlined the capacity, and a naive registration model can exhaust per-queue IOVA budget or silently accept memory classes that should never be posted to a NIC. Existing AF_XDP zero-copy largely models registration as whole-UMEM state, not as a kernel-authored per-chunk readiness contract, so userspace has no precise object showing which chunks are actually pinned, DMA-mapped, owned by a queue generation, or being retired. Any practical design must therefore define exact invocation context, memory-type admission, per-queue cap enforcement, readiness invalidation, lock-safe TX publication, and reset ordering without putting sleepable work into NAPI, without breaking qdisc fairness, and without fighting BQL/DQL.

2.2 Prior-Art Gap

page_pool and shipping drivers recycle pre-mapped pages, but they do not provide a complete policy mechanism that selects only DMA-capable CXL-backed ordinary System RAM on an explicit NUMA node, proves long-term-pin eligibility, and enforces a hard mapped_total budget spanning READY, POSTED, INFLIGHT, and DRAINING unique pages rather than merely bounding a freelist. Existing AF_XDP UMEM registration makes the region registered as a whole and does not publish a read-only per-chunk state table with queue generation, lease cookie, and invalidation semantics, so it cannot distinguish candidate chunks from actually DMA-ready chunks or express READY to DRAINING retirement during reset, munmap, or rebind. RDMA ODP faults pages on demand instead of pre-authoring per-chunk queue readiness; io_uring registered buffers manage CPU-side registration lifetime rather than NIC-postable readiness; and BQL/DQL sees bytes outstanding, not remote CXL latency, READY-supply scarcity, or per-queue IOMMU map pressure. None of those systems define the combined architecture of page_ext-based lease identity, mmu_interval_notifier sequencing, RCU-published per-queue TX directives, and Intel Xeon CXL plus VT-d pressure feedback with explicit clamp points that preserve BQL/DQL authority.

2.3 Proposed Invention

Implement a combined CXL-aware queue optimization subsystem with three coordinated planes: an RX lease broker, a TX queue governor, and an optional AF_XDP readiness UAPI. 1. Control plane and admissible memory classes: - Expose per-RXQ and per-TXQ policy via devlink params or ethtool netlink private params: target_nid, fallback_nid, mem_class, mapped_cap_pages, pin_cap_pages, retire_lowat_pages, tx_desc_floor, tx_desc_ceiling, tx_reclaim_floor, tx_reclaim_ceiling, cxl_pressure_weight, iommu_pressure_weight, ready_supply_weight, and update_interval_us. - mem_class values are DRAM_ONLY, CXL_SYSRAM_ONLY, or DRAM_OR_CXL_SYSRAM. CXL is admitted only if Linux has onlined the capacity as ordinary System RAM in a long-term-pin-eligible zone and the page passes pin_user_pages_fast(FOLL_LONGTERM) plus dma_map_page_attrs on the target NIC. - Explicitly reject ZONE_DEVICE, devdax, fsdax, device-private or device-coherent pages, DAX VMAs, secretmem, CMA-migratable pages, peer-to-peer memory, and any page for which long-term pin or dma_map fails. THP or other compound folios are split to order-0 pages before admission; if split is unavailable, registration fails with EOPNOTSUPP. 2. Exact kernel objects and contexts: - Each netdev that enables the feature owns one cxl_netd kthread_worker bound to the target NUMA node. This worker is the only producer of RX leases and the only publisher of new TX directives. It runs in process context, may sleep, may allocate, may pin pages, may dma_map, and may sample Intel Xeon uncore or VT-d telemetry. - Fast paths never sleep. NAPI RX refill consumes already prepared leases from a per-RXQ single-producer single-consumer ring. TX reads an RCU pointer to a cxl_txq_directive under rcu_read_lock_bh inside ndo_start_xmit and completion code paths. No new sleeping locks are introduced in BH-disabled or IRQ paths. - Control-plane mutation occurs under rtnl_lock, but actual pin or map work is deferred to cxl_netd after RTNL release. Queue reset sequencing uses lock order RTNL -> driver reset mutex -> generation publish -> synchronize_net -> napi_disable or netif_tx_disable -> drain -> unmap or unpin. That ordering avoids inversion with __netif_tx_lock, napi_disable, and existing RX or TX locks. 3. RX lease broker and hard cap invariant: - A lease is one order-0 page DMA mapping plus metadata: dma_addr, page pointer, queue_id, queue_gen, lease_cookie, mem_class, and state. - Each RXQ has a power-of-two SPSC ring of lease descriptors. Producer writes the fully initialized descriptor and then smp_store_release(prod). NAPI consumer reads prod with smp_load_acquire, consumes a lease, posts the descriptor to hardware, and never allocates or dma_maps in refill. - The claimable invariant is per queue on unique mapped pages, not on free-list entries: mapped_pages_total(rxq) = ready_unique_pages + posted_unique_pages + inflight_unique_pages + draining_unique_pages <= mapped_cap_pages(rxq). A page consumes budget at the first INVALID to READY transition and releases budget only after the final DRAINING to INVALID transition and final dma_unmap. - Budget reservation uses atomic_long_try_cmpxchg before pin or dma_map. If reservation fails, no pin and no map occurs. If pin succeeds but dma_map fails, the worker immediately unpins and releases the reserved slot. DRAINING pages still count until the final unmap completes, preventing silent oversubscription under concurrency. - page_pool recycling remains intact: the broker acquires order-0 pages from the page_pool or broker-owned reserve, maps them once, and recycles them through existing page_pool return paths. To avoid stale identity reuse, each mapped page gets a page_ext sidecar struct cxl_page_lease_ext containing lease_cookie, owner_queue_id, generation, state, and a posted_refcnt. page->private is not reused. On recycle or put_page, the driver clears lease_cookie and generation with smp_store_release before returning the page to general page_pool circulation. Cross-CPU recycle paths use smp_load_acquire when reclaiming or validating sidecar state. 4. TX directive ABI and stability model: - The worker publishes one RCU-protected struct cxl_txq_directive per TXQ: struct cxl_txq_directive { u16 abi_version; u16 flags; u32 queue_id; u64 queue_gen; u64 epoch_ns; u16 max_post_desc; u16 max_xmit_more_pkts; u16 wake_comp_thresh; u16 reclaim_batch_desc; u32 ready_pages_hint; u32 mapped_pages_hint; u32 iommu_pressure_ppm; u32 cxl_lat_ns_ewma; }. - Publication cadence is bounded, for example no faster than every 250 microseconds and no slower than every 2 milliseconds while the queue is active. Each update is monotonic-clamped relative to the prior directive: max_post_desc and reclaim_batch_desc change by at most one eighth of ring depth per epoch; max_xmit_more_pkts never drops below 1; wake_comp_thresh never exceeds half the ring. - The clamp point is inside the driver-owned batching loop before descriptor reservation and before doorbell write. It may limit how many descriptors are reserved in one ndo_start_xmit invocation, how long netdev_xmit_more batching may continue before a forced ring, how many completions are reclaimed per clean pass, and the completion count that triggers netif_tx_wake_queue. It never mutates DQL internals, BQL byte accounting, qdisc dequeue policy, watchdog timers, or netdev queue stop/start semantics. - This preserves stability because BQL still decides when bytes outstanding are excessive, while the directive only bounds local burst and wake aggressiveness. The worker cannot command zero progress: floors guarantee at least one packet or minimal descriptor progress per active queue, preventing starvation. Fairness remains with qdisc because dequeue ordering is unchanged; the directive only changes per-call burst depth. 5. Intel Xeon and x86 pressure coupling: - The worker computes each TX directive from three inputs unavailable to BQL alone: READY supply for the paired RX memory domain, VT-d or IOTLB pressure, and Intel Xeon CXL latency or bandwidth pressure. - A concrete telemetry record is: struct cxl_hw_pressure { u32 cxl_rd_bw_mb_s; u32 cxl_lat_ns_ewma; u32 iommu_map_ns_ewma; u32 iommu_inval_qd; u32 llc_miss_bw_mb_s; }. - On Xeon systems, cxl_rd_bw_mb_s and cxl_lat_ns_ewma come from uncore PMU groups for the socket or CXL root complex, and iommu_map_ns_ewma plus iommu_inval_qd come from Intel VT-d driver counters. The worker converts them to a normalized pressure score and computes per-queue directives using EWMA plus floor and ceiling guardrails. On Atom-class or reduced-telemetry x86 systems, the same ABI remains valid but the worker uses VT-d and queue-latency inputs only, with SSE or scalar smoothing instead of AVX2 or AVX-512. SIMD is process-context only under kernel_fpu_begin and never appears in NAPI or IRQ code. - Novel coupling arises because TX bursts are reduced when READY supply on the targeted CXL memory domain is scarce or when VT-d and CXL pressure imply that posting more descriptors would deepen tail latency or mapping contention. BQL cannot infer that from bytes outstanding alone. 6. AF_XDP readiness UAPI: - Introduce a new registrar mode through an extended UMEM registration structure: struct xdp_umem_reg_v2 { u64 addr; u64 len; u32 chunk_size; u32 headroom; u32 flags; u32 abi_version; u32 max_ready_chunks; u32 max_pinned_pages; }. - New flags include XDP_UMEM_F_READYMAP, XDP_UMEM_F_CXL_SYSRAM_PREFERRED, and XDP_UMEM_F_FAIL_ON_FALLBACK. Registration in this mode creates only a candidate chunk universe. It does not imply that every chunk is immediately DMA-ready. - Userspace discovers the readiness mmap layout with getsockopt(SOL_XDP, XDP_MMAP_OFFSETS_EXT, ...), which returns rx_ready_offset, rx_ready_stride, rx_ready_entries, and abi_version. Userspace then maps a read-only region from the AF_XDP socket at XDP_PGOFF_RX_READY. - Each readiness entry is keyed by existing AF_XDP chunk semantics plus kernel identity: struct xdp_rx_ready_desc { u64 chunk_id; u64 lease_cookie; u64 generation; u32 queue_id; u8 state; u8 mem_class; u16 flags; u32 last_error; }. State values are INVALID = 0, DRAINING = 1, READY = 2. generation is 64-bit monotonic with zero reserved as invalid. Wrap is handled by forcing a full queue drain and new cookie epoch before reuse; in practice the width makes wrap non-operational. - Userspace posts only chunks whose readiness entry is READY and whose lease_cookie and generation match the address it is about to place on the FILL ring. The kernel revalidates generation and cookie at post time. The readiness map exports no PFN, IOVA, or physical topology identifiers, limiting side-channel exposure. - Privilege and caps: enabling XDP_UMEM_F_READYMAP requires CAP_NET_ADMIN on the bound netdev and still consumes RLIMIT_MEMLOCK or memcg locked-page budget. Per-queue mapped_cap_pages and max_pinned_pages are enforced regardless of privilege. 7. How readiness differs from classic AF_XDP UMEM: - Classic UMEM registration is effectively whole-region registration. After setup, the region is treated as registered even though runtime faults or invalidations may still happen outside a per-chunk state machine. - In the new mode, registration establishes only a candidate range. The deferred operations are explicit: long-term pin, page-type admission, DMA mapping, queue ownership assignment, and CXL or fallback placement. Only after those steps complete does the kernel publish READY with release ordering. - Before READY, the kernel makes no promise that a chunk can be posted. After READY, the guarantee is that the chunk's backing page is long-term pinned, DMA-mapped to the target NIC, belongs to the named queue generation, and remains postable until the kernel publishes DRAINING or the generation changes. - Failure is observable per chunk via last_error, with reasons such as EOPNOTSUPP for disallowed page type, EAGAIN for cap exhaustion, ESTALE for generation mismatch, and EIO for dma_map failure. This is not mere asynchronous registration plus status; it is a stricter postability contract tied to queue generation and mapped-page accounting. 8. Invalidation, TOCTOU closure, and barriers: - Use mmu_interval_notifier over the UMEM VA range, not classic mmu_notifier, so the kernel can compare invalidate_seq across validation and descriptor post. - READY to DRAINING is triggered by munmap, mremap, process exit, XSK unbind, queue rebind, queue generation bump, memory hot-remove for CXL-backed System RAM, or mprotect changes that remove required access for NIC DMA. Migration-sensitive cases are already blocked by long-term pin eligibility; if a pin becomes invalid through external teardown, the same DRAINING path is used. - The safe boundary is: worker pins page, dma_maps page, writes sidecar metadata and readiness entry, then executes smp_store_release(state = READY). The posting path executes smp_load_acquire on the readiness state, samples the mmu_interval_notifier sequence, verifies generation and lease_cookie, increments posted_refcnt, reserves a descriptor, and executes dma_wmb before the device doorbell. If the notifier sequence changed or the generation mismatched, the kernel rejects the post and decrements posted_refcnt. - Invalidation callback sets state to DRAINING with smp_store_release, bumps the queue generation or invalidate sequence, and prevents new postings. Existing posted or inflight descriptors complete normally because the page remains pinned until posted_refcnt reaches zero. Only then does the retirement worker dma_unmap, unpin, clear page_ext cookie and generation with release ordering, decrement mapped_pages_total, and set state to INVALID. 9. Reset and reconfiguration contract: - Events that force a new generation are netdev down, carrier loss requiring ring reset, FLR or driver reset, RX or TX queue-count change, ring-size change, AF_XDP bind or unbind, and XSK map swap for the queue. - Contract: under RTNL and the driver reset mutex, the subsystem first allocates a new 64-bit queue_gen and publishes a RESET-flagged TX directive and queue metadata via RCU. Then synchronize_net runs to flush readers that may still hold the old pointer. Only after that are napi_disable or netif_tx_disable called for the affected queues. READY entries are transitioned to DRAINING, no new postings are allowed, posted_refcnt is drained, and final unmap or unpin follows. New queue resources are then created and new READY entries carry the new generation. - Removed queues never reuse the old queue_id plus generation pair. Rebound XSK sockets and queue-count changes therefore cannot observe stale READY. A queue generation bump on XSK map swap is mandatory even if the ring size is unchanged. 10. Optional x86 instruction-level acceleration in the worker only: - To tie the invention to x86 microarchitecture best practices without violating atomic-context rules, the multi-queue EWMA and clamp calculations in cxl_netd may use AVX2 or AVX-512 on Xeon, or SSE4.2 on smaller x86 cores, under static feature checks and kernel_fpu_begin. This optimizes telemetry reduction across many queues while leaving the fast path unchanged and safe.

2.5 Implementation Plan

1. Add policy plumbing: - Define devlink or ethtool netlink parameters for mem_class, target_nid, fallback_nid, mapped_cap_pages, pin_cap_pages, retire_lowat_pages, tx_desc_floor and ceiling, reclaim floor and ceiling, telemetry weights, and update interval. - Store policy in netdev private state and schedule cxl_netd after RTNL unlock. 2. Implement RX lease broker: - Add per-RXQ SPSC lease ring, mapped_pages_total accounting, and reserve-before-pin logic. - Add order-0 admission path that validates mem_class, long-term pin eligibility, and dma_map success. - Add page_ext sidecar type cxl_page_lease_ext for lease_cookie, queue_gen, queue_id, state, and posted_refcnt. - Wire NAPI refill to consume READY leases only, with smp_load_acquire on producer index. 3. Implement TX directive path: - Add struct cxl_txq_directive in shared driver-private state and publish with rcu_assign_pointer. - In ndo_start_xmit and completion reclaim path, add rcu_read_lock_bh reads and clamp only max_post_desc, max_xmit_more_pkts, reclaim_batch_desc, and wake_comp_thresh. - Enforce min update interval and monotonic delta limits. 4. Add Intel Xeon and x86 telemetry provider: - Implement a helper that returns cxl_hw_pressure from Xeon uncore PMU sources plus VT-d counters when available. - Add fallback provider using VT-d plus queue latency only for reduced-telemetry x86 systems. - Optionally batch EWMA and clamp calculations with AVX2, AVX-512, or SSE in cxl_netd only. 5. Add AF_XDP readiness ABI: - Extend UMEM registration to xdp_umem_reg_v2 with XDP_UMEM_F_READYMAP. - Add getsockopt offsets and mmap support for the read-only readiness region. - Populate xdp_rx_ready_desc entries from worker and retirement paths. - Return per-chunk last_error values and forbid physical-address disclosure. 6. Implement invalidation and reset handling: - Attach mmu_interval_notifier to READYMAP UMEM ranges. - On notifier invalidation, queue generation bump, netdev down, or XSK map swap, transition READY entries to DRAINING, stop new posts, wait for posted_refcnt to drain, then unmap, unpin, clear page_ext cookie, and publish INVALID. - Use the ordering: publish new generation via RCU, synchronize_net, disable queues, drain, retire, rebuild. 7. Add observability and limits: - Expose counters via ethtool stats or devlink health: ready_pages, draining_pages, mapped_pages_total, cap_hits, longterm_pin_fail, dma_map_fail, stale_cookie_drop, stale_gen_drop, iommu_pressure_ppm, cxl_lat_ns_ewma, and directive update count. - Enforce RLIMIT_MEMLOCK or memcg charging plus per-queue caps. 8. Gate deployment: - Start with one driver integration where page_pool and AF_XDP are already present, then generalize to a helper library for additional drivers.

4. Risks & Mitigations

  • Risk: long-term pin failures or policy surprises on CXL-backed System RAM. Mitigation: admit only pages that pass explicit FOLL_LONGTERM plus dma_map validation, reject unsupported page types up front, and provide deterministic fallback_nid or FAIL_ON_FALLBACK behavior.
  • Risk: stale READY state on recycled page_pool pages. Mitigation: store lease identity in page_ext rather than page->private, clear cookie and generation with release ordering on recycle, and require generation plus cookie match before posting.
  • Risk: TX governor oscillation against BQL or DQL. Mitigation: clamp only driver-owned burst and wake knobs, retain BQL or DQL authority, rate-limit updates, bound step size per epoch, and enforce non-zero floors for active queues.
  • Risk: TOCTOU between readiness validation and descriptor post. Mitigation: use mmu_interval_notifier sequence checks, acquire or release ordering on readiness state, per-entry posted_refcnt, and dma_wmb before doorbell so invalidation blocks only new posts while in-flight descriptors drain safely.
  • Risk: side-channel leakage through the readiness mmap. Mitigation: expose only chunk-local state, queue_id, generation, cookie, and error code; omit PFN, IOVA, NUMA topology, and physical address information; require CAP_NET_ADMIN to enable READYMAP mode.
  • Risk: queue reset races during netdev down, FLR, or XSK map swap. Mitigation: mandate generation bump publication before synchronize_net and queue disable, then perform drain and retirement before enabling new queue state.
  • Risk: telemetry availability differs across x86 platforms. Mitigation: define a stable cxl_hw_pressure ABI with Xeon uncore plus VT-d provider when available and a reduced VT-d plus latency fallback provider elsewhere without changing the queue ABI.
  • Risk: cap undercount when multiple chunks share a page. Mitigation: count budget on unique mapped pages, maintain per-page reference state for chunk membership, and release budget only on final chunk retirement and final dma_unmap.
