2. Technical Invention Disclosure

2.1 Problem Statement

Intel x86 RAPL exposes energy and power limit control at processor-package scope, but Linux virtualization schedules KVM vCPU threads and related helper tasks as ordinary host tasks that can migrate across logical CPUs, cores, and packages. Operators therefore observe only shared package energy even though they need a host-trustworthy answer to three distinct questions: how much of a package's measured energy can be conservatively credited to a VM's actual guest execution; how much additional package energy can be conservatively credited to host-side work performed specifically on behalf of that VM; and under what tightly bounded conditions is it safe to tune the package powercap without fighting hardware enforcement, thermal control, HWP/P-state logic, or an external policy daemon. The challenge is amplified by SMT, turbo, heterogeneous core capacity, multi-socket migration, shared helpers, RAPL counter rollover, unsupported/offline domains, and the fact that package energy includes baseline and uncore components that no honest VM estimator can fully explain from guest CPU time alone.

2.2 Prior-Art Gap

Known RAPL-based VM energy estimation approaches generally infer per-VM energy from userspace-observed CPU time, performance counters, or heuristic proportional models and then tend to force or imply near-complete attribution. They do not crisply claim a host-kernel method that derives guest runtime from actual guest-entry/guest-exit boundaries, preserves physical package identity across vCPU migration, and mandates an explicit residual bucket for baseline, shared-system, idle, and otherwise untracked package energy. They also do not define an enforceable host-assist qualification rule that excludes ambiguous shared helpers by default. This draft further narrows away overlap from the earlier, broader version: there is no guest hint channel, no VM-fd hint cache, no guest memory descriptor path, no guest-triggerable notification mechanism, and no new guest-facing ioctl control plane. The invention is instead limited to host-only KVM/RAPL accounting plus an optional userspace governor that uses the existing powercap ABI.

2.3 Proposed Invention

The invention introduces a conservative package-energy ledger for x86 KVM hosts and an optional confidence-gated package-cap policy loop. 1. Sampling and RAPL semantics A per-package host worker samples only the RAPL PKG domain at a fixed host interval, preferably 250 ms to 1000 ms. For package p it reads the current package energy value and the domain's maximum energy range from the existing Intel RAPL backend. The package energy delta is computed as modular arithmetic: ΔE_pkg[p] = (Ecur[p] - Eprev[p] + Range[p]) mod Range[p]. If the elapsed interval is long enough that more than one wrap could have physically occurred at the administrator-declared maximum package power, the interval is marked ambiguous, confidence is forced to zero for that interval, and no per-VM energy is assigned from that sample. If the package domain is offline, unsupported, or the read fails, the interval is invalid for attribution and the optional controller is frozen. 2. Guest-runtime segmentation from KVM execution boundaries For each vCPU thread, the host stamps a guest slice when the thread enters guest mode and closes that slice when it exits guest mode, is preempted, or otherwise returns to host mode. Each slice records start time, end time, logical CPU, core, and package. Because a task executes on only one logical CPU at a time, a package migration naturally terminates one slice and begins another on the destination package; thus a VM spanning packages accumulates distinct per-package ledgers instead of being flattened into one host-wide estimate. 3. Secure host-assist qualification and tagging Host-assist runtime is counted only for work that is both host-visible and uniquely bindable to one VM. The enforceable rule is a cgroup-v2-based one-VM assist subtree created by host administration in the initial namespace. A VM receives an immutable VM cookie at launch, and an assist cgroup is bound to that cookie by a privileged host operation. Eligible host-assist tasks are: the VM's vCPU thread while outside guest mode; emulator and I/O threads of the corresponding VMM process; and helper threads such as vhost or irq threads only if they are dedicated to that VM and placed in the VM's assist cgroup. Shared helpers, including ksmd, kswapd, ksoftirqd, shared vhost backends, or any task that services multiple VMs or general system work, are never force-attributed and always remain residual. If a task is retagged, migrated between assist cgroups, or observed in an ambiguous binding state, only already-closed slices remain valid and subsequent time is treated as residual until the binding is again unique. This prevents manipulation and double counting. 4. Exact normalization model Attribution uses weighted capacity-time, not raw wall time. For a runtime slice s on physical core c in package p: W(s) = Δt(s) × Cap_core(c) × Freq(c, interval) ÷ ActiveSMT(c, s). Cap_core(c) is the scheduler-visible architectural capacity of that core, normalized to a fixed scale, so hybrid-capacity x86 systems are handled without pretending all cores are equal. Freq(c, interval) is the scheduler frequency-invariance factor for that core during the interval; if that factor is not available, the host may substitute a clamped APERF/MPERF ratio for the same interval, and if neither source is trustworthy the package is marked unsupported for attribution. ActiveSMT(c, s) is the number of simultaneously active SMT siblings on that physical core during the slice, so one active sibling receives full core weight and two active siblings each receive half, preventing logical-thread double inflation. For VM v on package p over interval k: W_guest[v,p,k] = sum of W(s) over that VM's guest slices on package p. W_assist[v,p,k] = sum of W(s) over that VM's qualified host-assist slices on package p. W_vm[v,p,k] = W_guest[v,p,k] + W_assist[v,p,k]. The package denominator is computed at the core level: W_pkg[p,k] = sum over online physical cores in package p of OnlineTime(core,k) × Cap_core(core) × Freq(core,k). Energy is assigned conservatively: E_guest[v,p,k] = ΔE_pkg[p,k] × W_guest[v,p,k] / W_pkg[p,k]. E_assist[v,p,k] = ΔE_pkg[p,k] × W_assist[v,p,k] / W_pkg[p,k]. E_residual[p,k] = ΔE_pkg[p,k] - sum over all VMs of (E_guest + E_assist). If accounting error would make the assigned sum exceed ΔE_pkg, the interval is invalidated, confidence is forced low, and the full interval remains unattributed rather than clipped into a false exact answer. This explicit residual ledger is mandatory, not optional. 5. Multi-socket and migration behavior Each package maintains its own RAPL delta, denominator, confidence score, and VM sub-ledgers. A VM that runs on multiple packages accumulates one ledger entry per package and a VM-total is obtained only by summing package entries after sampling, not by collapsing runtime before attribution. Automatic NUMA balancing, scheduler migration, and vCPU roaming are therefore handled by slice stamping. For policy use, a stricter actuation eligibility rule may be applied: a package-level controller may operate only if the target VM or VM set is dominantly resident on that package over a recent horizon; otherwise the system remains observe-only. 6. Confidence metric and freeze logic Each package interval carries a confidence score C in the range 0 to 100, computed as a deterministic penalty model over observable failure modes, including ambiguous wrap risk, read failures, sampling jitter, high migration churn, missing frequency invariance, assist-tag ambiguity, low-attribution regimes, and external powercap writer detection. Example form: C = 100 - Pwrap - Pread - Pjitter - Pmigrate - Pfreq - Ptag - Plowutil - Pextwrite. The system freezes control eligibility whenever C falls below a threshold such as 80, when residual fraction exceeds an administrator threshold such as 35%, when package thermal or power-limit status is already asserted, when a counter read fails, or when an external actor modifies the powercap between control windows. 7. ABI and trust boundary No guest-visible hint ABI is claimed. No VM-fd hint cache, pollable notification fd, or guest-triggerable event path is required. The kernel claim core ends at internal ledgers and confidence generation. Export may be through host-only tracepoints, debugfs, or another non-guest, host-privileged diagnostic interface. If a production deployment wants a stable operator-facing API, that can be implemented in host userspace by consuming the diagnostic stream and re-exposing it, leaving the upstream kernel claim core independent of a new stable UAPI. 8. Optional powercap control in privileged userspace only The kernel does not write package caps. A separate host userspace governor, running with host privilege and using the existing powercap sysfs interface, may read the confidence and ledger telemetry and adjust only the shared package cap. The governor uses a non-obvious safety invariant: it is permitted to change the cap only when confidence is high, residual is low, package residency is stable, and the measured efficiency metric GuestWeightedWorkPerJoule = sum(W_guest over control window) / sum(ΔE_pkg over control window) remains monotonic across several dwell windows. A cap decrease is allowed only after, for example, three consecutive 10-second windows with C >= 80, residual <= 25%, no external writer, and no worsening of the efficiency metric beyond a 1% tolerance; each decrease is bounded to the smaller of 1 W or 2% of the current cap and may occur no more often than every 30 seconds. A cap increase is allowed only when efficiency or backlog degrades beyond a threshold, or when power-limit/latency symptoms persist, and it is likewise step-bounded. Any external write, thermal event, low-confidence interval, or ambiguous wrap forces a freeze period, e.g. 120 seconds, before control can resume. This creates a bounded, confidence-aware supervisor instead of a generic hysteresis loop.

2.4 Architecture

The architecture is intentionally split between a host-kernel telemetry path and an optional userspace actuation path.

Key trust boundary: guests provide no hints and cannot trigger attribution or actuation. Only the host kernel observes execution boundaries, only host-admin-defined cgroups can qualify assist tasks, and only privileged host userspace may write existing package powercap files.

2.5 Implementation Plan

1. Read-only kernel attribution core Implement a per-package sampler that reads PKG-domain energy and range metadata through the existing Intel RAPL path, computes modular deltas, rejects ambiguous-wrap intervals, and tracks package online/offline state. 2. KVM x86 guest slice instrumentation At KVM x86 guest-entry and guest-exit boundaries, stamp per-vCPU guest slices with time, logical CPU, core, and package. Store closed slices in per-package, per-VM accumulators so migration across packages creates distinct charge records instead of blended runtime. 3. Core-level normalization Derive Cap_core from scheduler architectural capacity and Freq from scheduler frequency invariance, with APERF/MPERF fallback only when reliable. Track active SMT sibling count so slice weight is divided by simultaneous sibling occupancy. 4. Assist-tagging subsystem Define a host-admin-only cgroup v2 binding from a VM cookie to a one-VM assist subtree. Count host-assist runtime only for tasks in that subtree while they are uniquely bound and while the vCPU thread is not in guest mode. Hard-code exclusions for shared kernel workers and ambiguous helpers so they always land in residual. 5. Ledger math and confidence engine For every package interval compute W_guest, W_assist, W_pkg, E_guest, E_assist, and E_residual. Add confidence penalties for wrap ambiguity, read failures, missing frequency invariance, jitter, migration churn, assist-tag ambiguity, low-utilization regimes, and external-writer detection. 6. Export strategy Keep the kernel claim core independent of a new stable UAPI. Export telemetry through host-only diagnostics such as tracepoints or debugfs for development and through a host daemon for production packaging if desired. No guest-facing ioctl, no VM-fd hint cache, and no notification plane are introduced. 7. Optional userspace governor Build a separate privileged daemon that reads the telemetry, aggregates 10-second control windows, detects external writers by comparing the current powercap value against its last commanded value, and applies bounded writes through the existing powercap sysfs files only when the confidence and monotonic-efficiency invariants are satisfied. 8. Deployment modes Support observe-only mode as the default. Enable control mode only when the administrator opts in, specifies min/max package-cap bounds, and accepts the freeze-on-low-confidence policy.

2.6 Validation Plan

1. Package-faithful attribution sanity tests Pin a single VM to one package with all other packages idle, run CPU-bound workloads, and verify that most explainable package energy lands in that VM's guest ledger while a bounded residual remains for package baseline. Repeat with a VM spanning two packages and verify that the per-package ledgers follow actual execution placement. 2. Migration correctness Force periodic vCPU migration across packages and compare slice-stamped attribution against known scheduler placement. Verify that no interval claims energy from a package on which the VM did not execute during that interval. 3. Assist-tag qualification tests Run dedicated QEMU I/O threads and dedicated vhost threads inside the VM assist cgroup and confirm that their host-mode runtime appears in the assist ledger. Then repeat with shared vhost, ksoftirqd, ksmd, and kswapd activity and verify those cycles remain residual rather than being force-assigned. 4. Counter and failure semantics Inject read failures, offline-package transitions, artificially shortened wrap ranges in a test harness, and sample-jitter spikes. Confirm that ambiguous or failed intervals produce low confidence, no false precise attribution, and control freeze behavior. 5. Normalization accuracy under SMT, turbo, and hybrid capacity Compare weighted-capacity attribution against raw wall-time attribution on SMT on/off systems, turbo-enabled vs capped-frequency runs, and heterogeneous-core hosts where scheduler architectural capacity differs. Demonstrate that weighted slices materially reduce error relative to raw runtime. 6. Prior-art boundary validation Benchmark against a Joulemeter/vmPower-style proportional model and show the difference in behavior under baseline-heavy, shared-helper, or migration-heavy conditions: the proposed method should preserve residual instead of forcing a misleading full split. 7. Control-loop safety evaluation In observe-only mode, replay recorded telemetry to confirm that the governor would have remained frozen during thermal assertion, external powercap writes, low-confidence windows, and multi-package instability. In control mode, run bounded package-cap experiments on representative virtualized CPU-bound and mixed I/O workloads, including Linpack-like HPC VMs and consolidation workloads, and verify monotonic efficiency checks, 30-second minimum write spacing, step bounds, and 120-second freeze after anomalies. 8. Security and leakage review Validate that guests cannot read per-VM ledgers directly, cannot spoof assist tags, and cannot induce notification storms because there is no guest-triggerable notification path. Confirm that only host-root or equivalent host-privileged operators can bind assist cgroups and enable actuation.

3. Patent Claims

  1. A computer-implemented method on an x86 Linux host running KVM, comprising: reading a package-scoped RAPL PKG energy counter for each physical processor package over successive intervals; deriving, from host-observed guest-entry and guest-exit boundaries, runtime slices for each virtual machine; stamping each runtime slice with the physical package on which the slice executed; computing a weighted runtime for each slice using physical-core architectural capacity, frequency-invariance information, and simultaneous SMT sibling activity; assigning to each virtual machine, for each package, only a proportional share of the measured package energy corresponding to the weighted runtime of that virtual machine's guest slices; and recording an explicit unattributed residual energy ledger for package energy not assigned to any virtual machine.
  2. The method of claim 1, wherein the weighted runtime further includes host-assist slices only for helper tasks that are uniquely bound to exactly one virtual machine by a host-admin-defined cgroup binding, and excludes shared or ambiguously bound helper tasks from attribution.
  3. The method of claim 1, wherein a vCPU migration between packages is represented by closing a first runtime slice on a first package and opening a second runtime slice on a second package, such that energy attribution is accumulated separately per package and only later aggregated at the virtual-machine level.
  4. The method of claim 1, wherein the package energy delta is computed by modular arithmetic using a maximum energy range associated with the package domain, and an interval is invalidated when multiple counter wraps are physically possible within that interval.
  5. The method of claim 1, wherein the package denominator is computed at a physical-core level rather than a logical-thread level, and the weighted runtime of a slice is divided by the number of simultaneously active SMT siblings on the physical core during the slice.
  6. The method of claim 1, further comprising generating a confidence score based on at least wrap ambiguity, counter read failure, sampling jitter, migration churn, missing frequency-invariance information, assist-tag ambiguity, and low-attribution utilization, and suppressing control eligibility when the confidence score falls below a threshold.

A host power-management system comprising the method of claim 1 and a separate privileged userspace governor that writes only existing package powercap controls, wherein the governor changes a package cap only after multiple consecutive control windows satisfy a confidence threshold, a residual-energy threshold, and a monotonic guest-weighted-work-per-joule condition, and otherwise freezes actuation.

A non-transitory machine-readable medium storing instructions that cause a host system to maintain, for each package, separate guest-energy, host-assist-energy, and residual-energy ledgers for each virtual machine using the method of claim 1 without requiring any guest-supplied energy hints, guest memory descriptors, or guest-triggerable notifications.

4. Risks & Mitigations

  • Kernel upstreamability risk: keep the kernel-side invention telemetry-only and rely on the existing powercap ABI for any actuation in privileged userspace, avoiding a new in-kernel control writer and avoiding a new guest-facing UAPI.
  • Assist-tag ambiguity risk: qualify host-assist time only for tasks uniquely bound to one VM by a host-admin-controlled cgroup-v2 mapping; force all shared or ambiguous helpers into the residual bucket.
  • Prior-art overlap risk: claim the combination of KVM guest-entry/exit segmentation, package-faithful migration stamping, exact weighted-capacity normalization, and mandatory residual ledgering, while explicitly disclaiming guest hint channels and full-attribution heuristics.
  • Normalization accuracy risk on systems lacking reliable frequency invariance: use APERF/MPERF only as an explicit fallback and otherwise mark the package unsupported or low confidence rather than silently reverting to raw wall time.
  • Counter rollover and read-failure risk: use modular delta computation with max-energy-range checks, invalidate ambiguous intervals, and freeze control on any read failure or multi-wrap possibility.
  • Cross-tenant information leakage risk: expose telemetry only to host-privileged contexts, avoid guest-readable notification paths, aggregate at conservative sampling intervals, and make enablement opt-in for operators.
  • Controller conflict risk with thermal management or another power governor: detect external writes to the package cap, treat thermal or power-limit assertion as immediate freeze conditions, and require long dwell times plus bounded step sizes before any new write.
