1. Executive Summary

One-Liner

A dual-plane Linux scheduler extension in which a per-task packed u32 LLC-pressure hint is published only at rq-serialized touch points through a seqcount-protected 64-bit snapshot, while a separate package worker converts wrap-safe x86 RAPL energy deltas, active PL1/PL2 clamp state, and an offline-mined per-SKU sensitivity table into a bounded package capacity_pressure source merged with thermal pressure by max(), so wake-up and balancing code consume only cached acquire-safe state and never read PMU or RAPL telemetry in hot paths.
Novelty: The novelty is a scheduler-native bridge contract, not per-task joule accounting and not generic power management. Plane A defines an rq-safe publication contract for cache interference: a task's last-proven-run interval
Feasibility: The design is implementable with existing x86/Linux primitives: topology_physical_package_id()-based package grouping, cpuhp online/offline callbacks, delayed_work queued on a pack

2. Technical Invention Disclosure

2.1 Problem Statement

Linux scheduling on LLC-sharing Intel systems faces two separate but interacting problems. First, task-level cache interference matters for placement, but the scheduler lacks a lifetime-safe and rq-safe way to consume observed cache-miss behavior in wake-up and balancing code. Second, package-level power limits observed through x86 RAPL reduce effective compute capacity for all CPUs in a package, but RAPL is package-scoped, asynchronous, and vulnerable to wrap, suspend, firmware reset, and hotplug races. Existing scheduler hot paths cannot sleep, cannot take powercap locks, and cannot legitimately perform PMU or RAPL reads. Existing per-CPU capacity reduction mechanisms also do not define how package-scope RAPL evidence should be transformed into a non-thermal capacity signal, how topology changes are handled, or how stale pressure is prevented. The kernel therefore lacks a precise bridge that converts asynchronous microarchitectural telemetry into scheduler-consumable state without violating locking, lifetime, fairness, hotplug, or tracing semantics.

2.2 Prior-Art Gap

Thermal pressure and related hardware pressure mechanisms already reduce effective CPU capacity, but they do not provide a scheduler contract for publishing task-scoped cache interference hints and they generally assume a thermal or firmware source rather than a package-scope RAPL bridge with explicit clamp semantics. uclamp expresses policy intent, not observed cache or power behavior. EAS and schedutil consume capacity signals, but they do not specify how noisy PMU plus RAPL observables become safe scheduler metadata, and using them as a cpufreq control-loop extension would look obvious and high-risk. perf_event can observe misses, but prior art does not define a low-overhead scheduler-specific publication contract in which only the last proven run interval is summarized into a packed hint and published through seqcount-protected state. Likewise, prior art around thermal pressure or power limits does not describe a generic capacity_pressure source that remains semantically distinct from thermal pressure, is published to every CPU in a package under explicit hotplug lifecycle rules, and is driven by a bounded mapping from wrap-safe DeltaE/DeltaT plus active PL1/PL2 state and an offline-mined per-SKU sensitivity table. The gap is therefore the bridge layer itself: publication contract, topology lifecycle, mapping function, and bounded scheduler consumption rules.

2.3 Proposed Invention

The invention introduces a split bridge between x86 microarchitectural telemetry and the Linux scheduler. Plane A - rq-safe LLC-pressure hint publication: 1. Each CPU optionally owns one pinned, non-overflow PMU miss counter chosen from a per-SKU selector table. Preferred events are architectural LLC or longest-latency-cache miss counters; if the SKU lacks a reliable architectural event, the table may select a model-specific L2/LLC proxy that best predicts slowdown on that SKU. 2. The counter is configured once when the feature is enabled. It is never created per task and never armed for sampling interrupts. The scheduler reads the current per-CPU count only at rq-serialized accounting points where the running task is already known: context-switch-out, dequeue-to-sleep, and periodic scheduler tick for the currently running task. If fast counter read is unsupported, multiplexed, virtualized, or otherwise unreliable, the task hint is published as class=unknown with confidence=0. 3. For the just-finished run interval, the writer computes delta_miss and delta_exec_ns, converts to a miss-rate bucket, combines it with runnable co-runner count on the shared LLC domain and an offline-mined per-SKU slowdown table, and produces a packed u32 hint: bits[1:0]=class {0 unknown, 1 low, 2 medium, 3 high}; bits[7:2]=confidence 0..63; bits[15:8]=miss_bucket; bits[23:16]=slowdown_bucket; bits[31:24]=flags {multiplexed, virtualized, reset, aged, table_defaulted, reserved}. 4. Publication uses a seqcount-protected 64-bit snapshot stored in task_struct scheduler state. The low 32 bits contain the packed hint, bits[47:32] contain an epoch that increments on reset/discontinuity, and bits[63:48] contain an age/version field. Writers execute write_seqcount_begin/end around the snapshot store at rq-serialized points. Readers in wake_affine, select_task_rq_fair tie-breaks, and load-balance tie-breaks perform read_seqcount_begin/retry and consume only a fully consistent snapshot. This prevents torn or mismatched reads on 32-bit or weakly ordered systems without adding locks to hot paths. 5. Consumption is bounded. The hint may only break ties when utilization, affinity, and capacity decisions are otherwise within a narrow threshold, for example within 3 percent of each other. It must not modify vruntime, EEVDF lag, deadlines, or entity ordering. Plane B - RAPL-derived package capacity_pressure: 1. CPUs are grouped by topology_physical_package_id(). For each package, the kernel maintains pkg_state {state, topo_gen, leader_cpu, online_mask, last_energy_uj, last_time_ns, sample_epoch, saturation_count, ewma_pressure, confidence, power_ref_table_ptr}. The package state machine is OFFLINE -> DISCOVERED -> BASELINING -> ACTIVE or DEGRADED. OFFLINE means no online CPUs and published pressure=0. DISCOVERED means topology and RAPL capability were found. BASELINING means the worker is collecting the first two good samples after enable, resume, or discontinuity. ACTIVE means pressure may be published. DEGRADED means telemetry exists but confidence is capped or zero because of virtualization, clamp-state absence, missing table, or read error. 2. Leader election is explicit. On each cpuhp online or offline transition, the package callback increments topo_gen, recomputes the stable online_mask, chooses the package leader as the lowest online housekeeping CPU in the package if one exists, otherwise the lowest online CPU, clears pressure on CPUs leaving the package mask, and queues delayed work on the new leader. If the last CPU in the package goes offline, delayed work is canceled and all package pressure is cleared. 3. The worker runs only in process context, default period 20 ms, min 10 ms, max 100 ms. It never executes in scheduler hot paths. It first acquires cpus_read_lock() only long enough to snapshot topo_gen, leader_cpu, and online_mask, then drops the lock before calling powercap or RAPL internals. After reading telemetry it reacquires cpus_read_lock(), verifies topo_gen and leader_cpu are unchanged, and publishes only if the snapshot still matches. If not, it discards the sample, increments sample_epoch, sets confidence=0, and reschedules on the new leader. 4. Wrap-safe power estimation is explicit. Let range_uj be the package domain max_energy_range_uj from powercap. If there is no prior valid sample, if delta_t is outside [5 ms, 200 ms], if suspend or firmware reset is detected, if energy decreases by more than range_uj, or if the powercap domain was re-enabled or re-bound, the worker treats this as a discontinuity: sample_epoch++, ewma_pressure=0, confidence=0, BASELINING state, and no pressure publication beyond zero. Otherwise dE_uj = (E_now >= E_prev) ? (E_now - E_prev) : (E_now + range_uj - E_prev), P_obs_mW = dE_uj * 1000 / delta_t_us. 5. The active package power cap is concrete. If both PL1 and PL2 are visible and enabled, P_cap_mW is the smaller currently enforcing limit; if a clamp-status bit indicates short-term PL2 clamp is active, PL2 wins for that sample. If clamp state is not visible, P_cap_mW is min(enabled limits) and confidence is capped at 32. If no package power limit is readable, package_pressure=0. 6. A per-SKU offline-mined table supplies power_ref_mW and slowdown sensitivity. Table schema: header {magic, version, x86 family-model-stepping mask, table_crc}; active_cpu_buckets[8]; miss_buckets[8]; power_ref_mW[8]; slowdown_permille[8][8]; max_derate_permille[8]. The table is loaded at boot from firmware or a distro-shipped binary under /lib/firmware and is read-only at runtime except for privileged replacement. Validation rejects family-model mismatch or CRC failure. If absent, a conservative built-in default table is used and confidence is capped. 7. The raw capacity-loss scalar is bounded and stable. For the current active_cpu_bucket b, let Pref = power_ref_mW[b]. Compute limit_frac = clamp((Pref - P_cap_mW) / max(Pref, 1), 0, 1). Compute sat_frac = 1 when clamp status is asserted; otherwise sat_frac = clamp((P_obs_mW - 0.97 * P_cap_mW) / max(0.03 * P_cap_mW, 1), 0, 1) only after three consecutive saturated samples with package runnable utilization above 80 percent. Compute table_cap = max_derate_permille[b] / 1000. Compute raw_loss = min(table_cap, limit_frac * sat_frac). Convert to scheduler units raw_pressure = round(raw_loss * SCHED_CAPACITY_SCALE). 8. Smoothing and hysteresis prevent thrash. ewma_pressure uses alpha_up=1/4 and alpha_down=1/16. Publication occurs only when confidence >= 32 and absolute change exceeds 2 percent of SCHED_CAPACITY_SCALE, except during a discontinuity reset, which forces immediate publication of zero. Pressure is cleared only after four consecutive unsaturated samples or any discontinuity. 9. Publication semantics are separate from thermal pressure. The invention introduces or uses a generic capacity_pressure source with source tags {thermal, rapl_pkg}. Effective pressure for a CPU is max(thermal_pressure, rapl_pkg_pressure), not a sum, to avoid double-counting and to preserve existing cpufreq and thermal tooling semantics. Tracing exposes rapl_pkg_pressure separately. 10. Scheduler consumers remain bounded. Effective capacity uses the aggregated pressure through existing capacity-reduction math. Task LLC hints are consumed only as bounded tie-breakers. No cpufreq governor logic, HWP requests, or thermal control loops are modified by the inventive core.

2.5 Implementation Plan

1. Add configuration and capability discovery. - Introduce CONFIG_SCHED_RAPL_PRESSURE and a static key disabled by default. - At init, enumerate packages by topology_physical_package_id(). For each package, allocate pkg_state, register cpuhp callbacks, and probe powercap/intel_rapl package domain for energy_uj, max_energy_range_uj, power limit visibility, and clamp-status support. - Load the read-only per-SKU table by family/model/stepping. Accept only a matching table with valid CRC. If the table is absent or mismatched, install a conservative default and cap confidence. 2. Implement package lifecycle and leader election. - cpuhp online callback: acquire package mutex, recompute online_mask, increment topo_gen, elect leader as first online housekeeping CPU else first online CPU, queue delayed work on leader, and transition to BASELINING. - cpuhp offline callback: increment topo_gen, clear rapl_pkg_pressure on CPUs leaving the package, re-elect leader if any CPU remains, otherwise cancel work and transition to OFFLINE. - Suspend/resume and package rebind paths: mark discontinuity, sample_epoch++, ewma_pressure=0, confidence=0, BASELINING. 3. Implement the worker with explicit lock ordering. - Worker lock order is: (a) cpus_read_lock only to snapshot topo_gen and online_mask, (b) drop cpus_read_lock, (c) perform powercap/RAPL reads, which may take driver mutexes or sleep, (d) reacquire cpus_read_lock to verify topo_gen and publish. - Never call powercap or RAPL reads while holding rq locks, raw spinlocks, or cpus_read_lock across the read. - If topology changed during the read, discard the sample and reschedule without publishing nonzero pressure. 4. Implement wrap-safe DeltaE/DeltaT and pressure mapping. - Maintain last_energy_uj, last_time_ns, valid_sample, sample_epoch, saturation_count. - On each worker iteration, compute discontinuity conditions exactly: no prior sample, dt too small or too large, energy wrap beyond max range, read failure, enable/disable transition, suspend/resume, topology generation change, or powercap domain reset. - Only after two consecutive good samples enter ACTIVE. Then compute P_obs_mW, P_cap_mW, limit_frac, sat_frac, raw_loss, ewma_pressure, and publish rapl_pkg_pressure per CPU in the verified online_mask. - Add tracepoints exposing {package_id, topo_gen, P_obs_mW, P_cap_mW, raw_pressure, ewma_pressure, confidence, reason_flags}. 5. Add a generic capacity_pressure source rather than overloading thermal_pressure. - Extend scheduler capacity-pressure plumbing to track pressure by source and export rapl_pkg_pressure separately. - Effective pressure is max of existing thermal pressure and rapl_pkg_pressure, preserving consumer expectations and avoiding double counting. - Audit EAS, load balancing, and schedutil interaction to confirm they only observe effective capacity and never receive a bogus thermal source label. 6. Implement non-intrusive PMU miss accounting. - Configure one pinned per-CPU miss counter, no sampling period, no overflow interrupt, static-key gated. - Read only on scheduler accounting points for the currently running task. Attribute only the delta since the previous accounting boundary on that CPU. - If perf indicates multiplexing, virtualization, disabled PMU, or failed pinning, force confidence=0 and class=unknown for that interval. - Convert delta_miss and delta_exec to a miss_bucket, combine with runnable LLC-domain co-runner count and the table slowdown_permille to generate the packed u32 hint. 7. Implement the publication contract. - Add to task scheduler state: seqcount_t llc_hint_seq; u64 llc_hint_snapshot. - Writer format: snapshot = ((u64)version << 48) | ((u64)epoch << 32) | packed_hint. - Writers update only at rq-serialized points with write_seqcount_begin/end. - Readers in wake-up and balancing tie-breaks use read_seqcount_begin/retry until a stable snapshot is obtained. If age/version exceeds a configurable staleness window, treat the hint as unknown. 8. Bound scheduler consumption. - Add tie-break helpers that consult the hint only when base policy has already narrowed the choice to near-equal CPUs or domains. - Never change vruntime, lag, or deadline computations. - Apply rapl_pkg_pressure only through effective capacity, not via any cpufreq or HWP actuation. 9. Add observability and admin controls. - Provide read-only debugfs or tracepoints for package state, topo_gen, hint buckets, table version, and confidence statistics; do not expose per-task cache history in a stable ABI by default. - Provide privileged sysfs knobs only for enable, worker period bounds, and table override selection. 10. Upstreaming posture. - Keep the claims and implementation narrowly focused on the scheduler bridge, publication contract, mapping function, and lifecycle correctness rather than on generic power management or frequency control.

2.6 Validation Plan

1. Correctness and concurrency validation. - CPU hotplug torture: repeatedly online/offline CPUs within and across packages while the worker is active; verify no stale rapl_pkg_pressure persists on offlined CPUs and leader migration always converges. - Suspend/resume and firmware-reset simulation: force RAPL counter discontinuities and confirm sample_epoch increments, confidence resets to zero, and published pressure drops to zero until two good samples occur. - Weak-order/torn-read validation: stress seqcount readers and writers with lockdep and kernel concurrency instrumentation to confirm readers never observe mismatched hint payload and epoch. - Virtualization and PMU multiplexing validation: run under guests and under deliberate perf multiplexing; verify class=unknown and pressure=0 fallback. 2. Stability validation. - Use controlled package power limits to create bursty and sustained clamp conditions. Verify the mapping function, EWMA, and hysteresis avoid oscillation in effective capacity and do not cause load-balancer ping-pong. - Sweep worker period from 10 ms to 100 ms and confirm stability envelope and convergence time. 3. Performance validation. - LLC contention suites: STREAM plus SPEC-like memory-sensitive mixes, memcached/Redis plus build jobs, and synthetic co-runner interference tests. Measure p99 latency, throughput, migration rate, and cache-miss slowdown versus baseline. - Power-clamp suites: package-limited CPU stress with and without background cache pressure; compare throughput-per-watt and latency under baseline scheduling versus the invention. - Heterogeneous and HPC-adjacent suites: HPL or HPL-AI style feeder-thread workloads, OpenMP plus GPU offload feeder threads, and MPI communication plus computation mixes. Measure whether CPU feeder threads avoid co-location under cache pressure and whether package clamps are reflected in more stable placement. 4. Fairness and regression validation. - Confirm vruntime, lag, and deadline ordering are unchanged except for tie-break placement decisions. - Measure starvation freedom, wakeup latency, and migration count on vanilla desktop, server, and NO_HZ isolated setups. - Confirm schedutil and thermal tooling still report thermal pressure separately from rapl_pkg_pressure. 5. Table validation. - Validate per-SKU table selection by family/model/stepping and CRC. - Compare default conservative table versus calibrated table on at least three Xeon or Core SKUs; require calibrated tables to outperform default without increasing oscillation or fairness regressions. - Add self-test traces showing power_ref_mW bucket chosen, miss_bucket chosen, and slowdown_permille used for each published hint.

4. Risks & Mitigations

  • Prior-art overlap with thermal pressure or hardware pressure: mitigate by claiming the bridge contract narrowly, using a distinct rapl_pkg capacity_pressure source, and emphasizing seqcount publication, topology lifecycle, and offline-mined mapping rather than generic capacity scaling.
  • Oscillation or negative interaction with cpufreq/HWP: mitigate by keeping the invention out of cpufreq control loops, using max() aggregation rather than additive pressure, applying asymmetric EWMA plus 2 percent hysteresis, and gating publication on sustained clamp evidence.
  • RAPL counter discontinuity, wrap, suspend, or firmware reset causing bogus power estimates: mitigate with explicit discontinuity detection, sample_epoch increments, forced confidence=0, and zero publication until two good samples are observed.
  • CPU hotplug race or stale pressure propagation: mitigate with package topo_gen versioning, explicit leader election, cpuhp callbacks that clear leaving CPUs, and publish-only-after-revalidation semantics.
  • PMU enablement overhead or NMI cost: mitigate by using one pinned per-CPU miss counter with no overflow interrupts, reading only at existing scheduler accounting boundaries, and disabling the feature entirely when fast reads or pinning are unavailable.
  • Virtualization, perf multiplexing, or noisy telemetry corrupting hint quality: mitigate by forcing class=unknown and package_pressure=0 or capping confidence whenever the PMU or RAPL source is not trustworthy.
  • Calibration-table drift or wrong SKU match: mitigate with family-model-stepping match, CRC validation, conservative default tables, privileged override only, and trace visibility into which table and bucket were used.
  • Potential side-channel concerns from exposing per-task cache behavior: mitigate by keeping hints internal to the scheduler, avoiding stable per-task user ABI by default, and exposing only aggregated or privileged diagnostics.
  • Implementation complexity across scheduler, perf, and powercap subsystems: mitigate by keeping subsystem boundaries explicit, performing all cross-subsystem calls in worker or rq-serialized contexts only, and tracing every publication reason for diagnosability.
