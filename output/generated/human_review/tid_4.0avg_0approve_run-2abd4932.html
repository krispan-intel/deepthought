2. Technical Invention Disclosure

2.1 Problem Statement

Linux qspinlock already minimizes lock footprint, but the queued immediate-successor release path still exposes a short sequence of lockword and successor-handoff stores that can generate cacheline bouncing, transient wake/spin windows, and queue-handoff latency out of proportion to the useful work protected by the lock. A generic TSX-on or TSX-off policy is too coarse because RTM availability depends on microcode and BIOS policy, per-CPU feature state, `MSR_IA32_TSX_CTRL`, security mitigations, and whether the compiled kernel is using native or instrumented unlock code. Any kernel design here must preserve the exact semantics of `spin_unlock`, `spin_unlock_irqrestore`, `spin_unlock_irq`, and `spin_unlock_bh`; must tolerate interrupts and NMIs safely; must not alter queue topology or forward-progress guarantees; and must never reuse speculative state after abort. The technical problem is to define a Linux-specific, mechanically checkable unlock-side RTM contract that can be attempted only when a native qspinlock branch is known precisely enough to reproduce the exact baseline final state, while guaranteeing a single bounded attempt and exact fallback to the compiled native path.

2.2 Prior-Art Gap

Known HTM/TLE work typically speculates entire critical sections or elides lock acquisition, which broadens the conflict surface and leaves open questions about interrupt state, lock ownership, queue fairness, and debug-path equivalence. Prior art also often describes generic check-then-update transactional patterns rather than a Linux-native semantic branch with an exact baseline mapping. What is missing is a design that: (1) limits speculation to the unlock-side-only immediate-successor handoff event of native qspinlock; (2) requires an exact revalidation snapshot consisting of full lockword equality and a stable already-linked successor; (3) uses a conservative asynchronous admission plane that tracks runtime TSX availability and abort behavior without polluting the unlock fast path; (4) explicitly excludes paravirt and debug/instrumented unlock implementations; and (5) guarantees that the fallback path is the unchanged baseline helper, not a logically similar reimplementation. That narrower combination materially reduces prior-art overlap and supplies the correctness boundary reviewers asked for.

2.3 Proposed Invention

The invention introduces a per-lock-class control-plane object, `struct tsx_sync_class`, published under RCU and refreshed asynchronously. Each object contains: class identifier, wrapper variant mask (`plain`, `irqrestore`, `irq`, `bh`), a generation number, enable bit, per-CPU commit/abort counters summarized through a conservative EWMA filter, an optional CPU mask, and feature-policy bits derived from runtime TSX status. CPU hotplug and TSX policy changes trigger `tsx_sync_recompute_global_state()`, which updates a global generation and republishes any affected class objects. On the data path, the optimization is only reachable from the native qspinlock immediate-successor unlock helper. The attempt rule is: `if (!static_branch_unlikely(&tsx_sync_any_enabled) || !rcu_dereference(class)->enabled || !this_cpu_read(tsx_sync_cpu_ok)) goto baseline;`. Before XBEGIN, the code captures `val_snapshot = atomic_read(&lock->val)` and `succ_snapshot = READ_ONCE(node->next)`. The capsule is eligible only if: native qspinlock implementation is active; wrapper state matches the entry contract for the unlock variant; `succ_snapshot != NULL`; `(val_snapshot & (_Q_LOCKED_MASK | _Q_PENDING_MASK)) == _Q_LOCKED_VAL`; the branch being executed is the immediate-successor handoff case; and the tail encoded in `val_snapshot & _Q_TAIL_MASK` maps to the same successor identity the baseline helper would use for that branch. Inside the transaction, the code revalidates `atomic_read(&lock->val) == val_snapshot`, `READ_ONCE(node->next) == succ_snapshot`, `READ_ONCE(succ_snapshot->locked) == 0`, and successor identity consistency. It then executes exactly the same logical writes that the selected baseline helper would execute for that branch, using the same helper-derived `val_post` and successor-handoff flag values. No other stores are permitted inside the capsule. On any mismatch, interrupt, NMI, fault, or RTM abort, the processor discards all transactional writes and control immediately jumps to the unchanged baseline helper with the original arguments. There is no retry. Wrapper-specific reenabling of interrupts or bottom halves remains outside the capsule and uses the unchanged wrapper logic, so the only optimized region is the native queued immediate-successor handoff publication itself.

Proof sketch of baseline equivalence: let `S0` be the pre-transaction shared state consisting of the full 32-bit qspinlock word and the relevant successor-node fields for the native immediate-successor branch. Let `B(S0)` be the exact state produced by the current baseline helper for that same branch. The capsule may commit only if the full lockword is still exactly `S0.lockword` and the successor link is still exactly `S0.succ`. The transaction then emits only the stores needed to reach `B(S0)` and commits atomically. Therefore other CPUs can observe either `S0` or `B(S0)`, but no additional intermediate state not already allowed by x86 RTM atomicity. Because the invention is limited to x86 kernels whose baseline helper relies on TSO-compatible release operations, the RTM commit point is at least as strong as the baseline release publication. If the transaction aborts, no transactional store becomes visible and the unchanged baseline helper executes, so the resulting shared state is exactly `B(S0)` for the then-current non-transactional state. This preserves correctness, queue topology, and forward progress while merely removing transient intermediate visibility windows on successful commit.

2.5 Implementation Plan

1. Add a Kconfig option scoped to `X86 && X86_RTM`, with hard exclusions for `CONFIG_PARAVIRT_SPINLOCKS`, `CONFIG_DEBUG_SPINLOCK`, `CONFIG_PROVE_LOCKING`/lockdep, and `CONFIG_KCSAN`; leave the feature default-off. 2. Introduce `struct tsx_sync_class`, a registration API for candidate lock classes, a global static key, per-CPU `tsx_sync_cpu_ok`, and a control-plane worker that applies a conservative EWMA filter to commit/abort telemetry and republishes class objects under RCU. 3. Hook CPU hotplug and TSX-policy refresh points so `tsx_sync_recompute_global_state()` updates the global generation when CPUs come online/offline or TSX policy changes; stale readers remain safe because the final local-CPU gate is rechecked before XBEGIN. 4. Split the native qspinlock immediate-successor unlock code into an unchanged baseline helper and an RTM wrapper helper; the wrapper may only surround that semantic branch and must call the baseline helper directly on any non-attempt or abort so fallback machine code remains the baseline implementation. 5. Implement the exact snapshot and revalidation predicate using symbolic qspinlock macros (`_Q_LOCKED_MASK`, `_Q_PENDING_MASK`, `_Q_TAIL_MASK`) and the current branch's successor-node semantics; if a kernel version cannot expose a precise helper-derived `val_post` or successor identity, the feature is compiled out for that version. 6. Keep wrapper semantics exact by running the capsule before any wrapper-specific reenabling step, leaving `irqrestore`, `irq`, and `bh` post-actions unchanged. 7. Export per-class counters through existing observability hooks so operators can see attempts, commits, abort reasons, and automatic disablement thresholds without affecting the unlock hot path.

2.6 Validation Plan

1. Correctness: prove and test observational equivalence by instrumenting the baseline helper and RTM wrapper to compare `val_post`, successor identity, and wrapper post-actions under stress. 2. Forward progress: verify that every abort path performs at most one failed XBEGIN and then one baseline unlock, with no retry loop, under forced persistent abort, `tsx=off`, `rtm_always_abort`, and microcode-disabled RTM scenarios. 3. Interrupt/NMI safety: run irqstorm, NMI watchdog, perf PMU overflow, and resched interrupt workloads while stressing targeted lock classes; verify no partially released lock state is ever visible and that worst-case unlock latency is bounded by one abort plus the native helper. 4. Build/runtime exclusions: exercise kernels with paravirt spinlocks, lockdep, debug spinlock, and KCSAN enabled to confirm the feature self-disables or is not built. 5. Heterogeneous and hotplug behavior: online/offline RTM-capable and non-capable CPUs, toggle TSX policy where supported, and verify only current-CPU-eligible paths attempt XBEGIN while all others use baseline unlock. 6. Performance: benchmark queued lock handoff latency, LLC misses, lock hold-to-wakeup delay, and tail latency in metadata-heavy microbenchmarks and representative storage/database workloads. 7. Regression safety: compare the optimized kernel against the same kernel with the static key forced off and require identical functional results and no measurable slowdown for untargeted lock classes.

3. Patent Claims

  1. 1. A method in an x86 Linux kernel for optimizing synchronization, comprising: maintaining an RCU-published per-lock-class admission object that reflects runtime RTM eligibility; reaching a native qspinlock immediate-successor unlock branch; capturing a full qspinlock lockword snapshot and an already-linked successor snapshot; performing exactly one RTM attempt; inside the transaction revalidating full lockword equality and successor identity; committing exactly the baseline handoff stores for that branch if the revalidation succeeds; and otherwise executing the unchanged native baseline unlock helper without retry.
  2. 2. The method of claim 1, wherein the RTM eligibility is additionally conditioned on a per-CPU runtime predicate derived from `X86_FEATURE_RTM`, a runtime-always-abort indicator, and current TSX policy, such that heterogeneous systems permit RTM attempts only on currently eligible CPUs and fall back on other CPUs.
  3. 3. The method of claim 1, wherein the admission object is refreshed asynchronously in response to CPU hotplug events and TSX-policy changes, and is republished under RCU with a generation number so hot-path readers do not take locks or perform MSR reads in the unlock path.
  4. 4. The method of claim 1, wherein the revalidation predicate requires equality of the full 32-bit qspinlock word, not merely masked lock bits, and additionally requires that the successor pointer captured before XBEGIN remains identical, non-null, and consistent with the tail encoding used by the native branch.
  5. 5. The method of claim 1, wherein the optimization is limited to native qspinlock implementations and is excluded for paravirtualized queued spinlocks, debug or lockdep-instrumented spinlock implementations, and KCSAN-instrumented unlock paths.
  6. 6. The method of claim 1, wherein wrapper semantics are preserved by executing the RTM capsule before any wrapper-specific interrupt or bottom-half reenabling operation, leaving the post-unlock wrapper logic bit-for-bit identical to the baseline kernel.
  7. 7. The method of claim 1, wherein the committed transactional state is constrained to equal the exact post-state produced by the selected baseline immediate-successor unlock helper for the same snapshot, and the method does not alter queue topology, does not speculate lock acquisition, and does not speculate the protected critical section.
  8. 8. The method of claim 1, wherein a conservative telemetry filter disables the admission object for a lock class after an abort-rate threshold or forward-progress threshold is crossed, and re-enables the lock class only after subsequent asynchronous evaluation, without adding retries to the unlock hot path.

4. Risks & Mitigations

  • Runtime TSX disablement or microcode policy changes can invalidate prior assumptions: mitigate by storing only advisory admission state under RCU and rechecking a local per-CPU eligibility bit immediately before XBEGIN; any stale state only causes a harmless fallback.
  • Persistent abort storms from interrupts, NMIs, or conflict misses can erase benefit: mitigate with a mandatory single-attempt rule, no retry loop, per-class abort telemetry, and conservative automatic disablement.
  • Prior-art overlap with generic HTM/TLE proposals: mitigate by explicitly limiting claims to unlock-side-only native qspinlock immediate-successor handoff, exact baseline-state reproduction, unchanged fallback helper, and no transactional acquisition or full critical-section speculation.
  • Ordering mistakes on non-x86 or instrumented builds could break correctness: mitigate by restricting the invention to x86 kernels whose native helper uses TSO-compatible release semantics and by compile-time exclusion of paravirt/debug/lockdep/KCSAN variants.
  • Kernel-version drift in qspinlock internals could weaken the exactness requirement: mitigate by tying implementation to a semantic branch plus helper-derived post-state; if a future kernel version cannot expose that exact mapping, the feature is disabled for that version rather than approximated.
  • Operator concern about hidden behavior changes: mitigate by making the feature default-off, class-scoped, observable via counters, and reversible via static key and boot/runtime policy controls.
