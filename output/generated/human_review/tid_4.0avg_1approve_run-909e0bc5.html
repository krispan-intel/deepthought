2. Technical Invention Disclosure

2.1 Problem Statement

Static futex bucket hashing and CPU-oblivious reader indication create avoidable remote metadata traffic when the threads contending on a synchronization object migrate across LLC, package, NUMA, or CXL latency domains. On multi-socket x86 systems, especially with asymmetric HMAT/SLIT distances or CXL-attached memory, a private futex can become hot on CPUs that are remote from the bucket lock currently serving it, and a writer waiting behind many readers can pay drain work proportional to CPU count or unbounded reader bypass if new readers continue to enter after writer intent. The kernel challenge is to adapt placement without violating rendezvous reachability, rwsem fairness, existing scheduler legality, PREEMPT_RT constraints, or wake ordering.

2.2 Prior-Art Gap

Legacy futex hashing provides deterministic rendezvous but not an updateable per-key home handle with explicit lifetime, bounded dual-location drain, and a proof-friendly guarantee that concurrent wake paths search at most one active and one draining bucket. Existing wake-affinity logic such as WF_SYNC, wake_affine, wake_wide, and TTWU_QUEUE can bias CPU placement, but it is not driven by a synchronization-object-owned token emitted only after waiter selection and cannot safely claim ownership of lock semantics. Linux percpu_rwsem is also not the same invention: it is a specialized API and implementation optimized around per-CPU reader indication, not a generic rwsem-preserving topology-domain ledger with writer-intent gating, migration-safe per-acquire unlock tokens, fixed domain bounds independent of CPU count, and banked topology-generation handling. The gap is therefore a causal bridge from x86 and CXL locality signals to Linux synchronization objects that is bounded, semantics-preserving, and explicit about legality and lifetime.

2.3 Proposed Invention

The disclosure has two coordinated embodiments. 1. Private futex home directory. A private futex key, derived using the same normalized key material already used for private futex lookup, indexes an RCU-readable home directory entry allocated only on first contention. The home entry contains the normalized key, a stable home_cookie, an active bucket identifier, an optional draining bucket identifier, a topology-generation snapshot, a refcount pinned by enqueued waiters, and a small contention/locality summary. Waiters and wakers first resolve the home entry, not the bucket directly. Rehome never migrates enqueued waiters in place. Instead, under the old bucket lock and home-entry serialization, the kernel publishes a new active bucket, marks the prior bucket as draining, and increments the home_cookie generation. New waiters enqueue only into the new active bucket. Wakes, timeout dequeue, and signal dequeue scan the active bucket and, if non-null, the draining bucket; once the draining bucket no longer contains waiters for that home entry, the draining pointer is cleared and the home entry returns to single-bucket mode. The bound is therefore two buckets per key during rehome. 2. Topology-domain rwsem ledger. Each rw_semaphore optionally embeds a fixed-size ledger with MAX_SYNC_DOMAINS padded counters in each of two banks. Each bank carries a topo_gen, valid slot map version, and reader counters. A writer-pending flag is part of the semaphore and is set with release ordering at writer intent time before the writer may queue for ownership. A reader fastpath is allowed only while writer_pending is clear and the semaphore is ledger-enabled. On success, the reader reserves a task-local token slot, increments the counter for the active bank and the current CPU's compressed locality domain, rechecks writer_pending and bank validity, and on success stores a token containing sem address, sem_cookie, bank id, slot id, topo_gen, and nesting instance. up_read uses that stored token, not current CPU topology, to decrement the exact bank and slot used at acquire time. Thus migration, preemption, and CPU hotplug do not mis-account readers. If topology changes, the semaphore can flip to the other bank with a new topo_gen while the old bank drains, again bounding writer scan to at most two banks. 3. One-shot wake locality token. When a futex wake or rwsem wake has already selected a specific waiter or reader batch according to existing primitive semantics, the code may attach an opaque token to the wake_q element for that task. The token carries only advisory placement data: preferred locality-domain id, emitting CPU, home_cookie or sem_cookie, topo_gen, and a short policy tag. The scheduler may consult it only after legality checks, only for fair-class CPU selection, and only as a tie-break between otherwise legal CPUs. The token is dropped after the wake attempt and never persists in task_struct.

2.5 Implementation Plan

1. Add sync topology compression layer. - Build sync_topology_map from x86 package_id, llc_id, NUMA node, and HMAT/CDAT-derived latency class. - Compress to at most MAX_SYNC_DOMAINS slots; publish with RCU and topo_gen. 2. Implement private futex home directory. - Add home table keyed by normalized private futex key. - Allocate home entries from a slab cache on first contention. - Extend futex wait/wake/timeout/signal paths to resolve the home entry first. - Implement rehome as publish-new/drain-old dual-bucket transition with no waiter migration. - Track preferred domain from recent waiter/waker activity and topology map. 3. Implement wake_q locality token. - Extend wake_q element or a wake_q sidecar with an opaque token structure. - Emit token only after the primitive has selected the waiter(s) to wake. - Consume token only in fair-class CPU selection after legality checks and existing wake-affine heuristics. 4. Implement rwsem ledger. - Add sem_cookie, writer_pending, active_bank, and two banks of padded counters to rwsem under a config guard. - Add a task-local token table allocated with task lifetime, accessed only by current. - Add reader fastpath checks, increment-and-recheck logic, and token recording. - Add writer-intent gating and bounded drain before write ownership. - Add bank rollover on topology change. 5. Fallback and observability. - Expose counters for rehome count, dual-bucket dwell time, rwsem bank rollover count, fastpath fallback reasons, and token ignore reasons. - Disable feature per object or globally when token table overflow, domain compression overflow, or unstable topology is detected repeatedly. 6. PREEMPT_RT and lockdep review. - Keep lock ordering unchanged. - Audit that no new sleeping lock is taken in atomic context and that scheduler interactions remain advisory only.

4. Risks & Mitigations

  • Risk: Home-entry churn or excessive rehome oscillation on phase-changing workloads. Mitigation: hysteresis on rehome triggers, minimum dwell time before a second rehome, and immediate fallback to fixed legacy hashing after repeated oscillation.
  • Risk: rwsem token table overflow in deeply nested reader code. Mitigation: fixed small task-local table with precise overflow fallback to base rwsem for that acquisition; no correctness dependency on token allocation success.
  • Risk: Prior-art challenge against percpu_rwsem. Mitigation: claims require generic rwsem queue preservation, writer-intent gating before ownership, per-acquire migration-safe token binding, and a bounded domain cap independent of CPU count.
  • Risk: Scheduler reviewers reject any hint that looks like a policy override. Mitigation: token is post-selection only, wake_q-scoped, fair-class-only, and ignored unless normal legality and heuristic tie conditions are already satisfied.
  • Risk: PREEMPT_RT incompatibility. Mitigation: no new sleeping locks on hot paths, no new cross-subsystem lock nesting, local-only task token writes, and explicit RT/lockdep validation.
  • Risk: CPU hotplug or HMAT/CDAT churn leaves stale topology references. Mitigation: RCU-published topology maps, topology generations in tokens, and two-bank draining so old-generation tokens remain valid until counters reach zero.
  • Risk: Side-channel concern from exposing fine-grained topology preference. Mitigation: token remains kernel-private, carries only compressed domain ids rather than raw latency values, and is discarded immediately after wake placement.
