2. Technical Invention Disclosure

2.1 Problem Statement

Linux drivers repeatedly encounter the same concurrency asymmetry: many in-kernel producers want to mutate shared hardware intent, but the device-visible side must remain single-writer, ordered, and semantics-correct. In virtio-class drivers, multiple submission paths want to enqueue buffers concurrently, yet the exposed vring must present a legal contiguous publication stream with correct split-ring or packed-ring notification behavior and no stale publication after reset. In regmap-managed drivers, regulator, GPIO, IRQ, thermal, suspend/resume, and fault-recovery paths touch overlapping registers whose bitfields have materially different semantics such as ordinary read/modify/write, write-1-to-clear, read-clear, pulse/toggle, volatile/precious, or reset-sensitive fields. Conventional queue locks, regmap_update_bits loops, or ad hoc batching either serialize too much producer work or leave correctness dependent on fragile path discipline. x86 TSX can reduce contention only for short cache-resident metadata regions, but RTM aborts on interrupts, conflicts, capacity pressure, VM policy, and microcode disablement, so a viable invention must define an exact non-TSX correctness path, an exact publication protocol, and exact reset/epoch rules.

2.2 Prior-Art Gap

Existing art solves fragments, not the full bridge. virtio_ring already ensures that a single queue instance exposes descriptors in order, but it does so by relying on driver-side locking and immediate publication into the vring; it does not define a separate driver-private resolved sequence in which aborted reservations remain as first-class ABORT markers that the publisher can retire while preserving prefix legality. Generic MPSC queues publish software work items but do not define split-ring versus packed-ring device exposure rules, event-index evaluation, wrap-counter handling, DMA visibility ordering, or reset invalidation of pre-published shadow work. regmap/regcache plus single-threaded workqueues serialize access and cache values, but they do not classify field semantics into execution classes, do not require every producer to enter a common monotonic intent log with COMMIT/ABORT states, and do not invalidate queued work by epoch tuple on reset or suspend/resume. HTM and lock-elision art teaches speculative lock avoidance, but not a correctness-independent driver bridge in which TSX accelerates only metadata staging while a release/acquire resolved-prefix publisher enforces device-specific publication legality. The topological-void evidence is useful only at the abstract level that x86 feature-gated staging can be bridged to driver control publication; unrelated TBM, AMX, TPU-factorization, and hypervisor-specific anchors were discarded from the actual claim boundary.

2.3 Proposed Invention

Each participating driver allocates a power-of-two, cacheline-aligned shadow array of slots in normal cacheable RAM and exposes one publication bridge per hardware-visible stream. A slot has the concrete layout `struct dt_slot { u64 seq; u32 epoch; u16 generation; u16 state; u16 op_class; u16 flags; u32 payload_words; u32 abort_code; u64 owner_cookie; u64 retire_cookie; union { struct dt_reg_intent reg; struct dt_vq_intent vq; } u; };` where `state` is one of `FREE`, `RESERVED`, `COMMIT`, `ABORT`, or `PUBLISHED`. Producers obtain `seq = atomic64_fetch_inc_relaxed(&tail_seq)`, derive `slot = table[seq & mask]`, and verify that the slot generation and retire state match the expected reuse epoch before writing payload. If runtime TSX gating passes (`boot_cpu_has(X86_FEATURE_RTM)`, static key enabled, not disabled by microcode/policy, and callsite allowed), the producer may wrap only the metadata reservation/coalescing section in RTM to reduce cacheline ping-pong; otherwise it uses the fallback atomic/spinlock path immediately. The producer writes payload and metadata first, then resolves the slot exactly once with `smp_store_release(&slot->state, COMMIT)` or `smp_store_release(&slot->state, ABORT)`. ABORT codes are explicit and typed, including `ABORT_RTM_CONFLICT`, `ABORT_RTM_CAPACITY`, `ABORT_EPOCH_MISMATCH`, `ABORT_DESC_EXHAUSTED`, `ABORT_CLASS_CONFLICT`, and `ABORT_RESET`. The contiguous-prefix rule is the heart of publication: the publisher starts at `head_seq` and scans upward in order. For the expected sequence number, if the slot tuple `(seq, epoch, generation)` does not match, or the state read by `smp_load_acquire()` is still `RESERVED`, scanning stops and nothing later is exposed. If the state is `ABORT`, the publisher records retirement metadata, advances `head_seq`, and continues without any device-visible action. If the state is `COMMIT`, the publisher executes the payload according to `op_class`, then marks the slot `PUBLISHED` and advances `head_seq`. Because ABORT occupies the missing sequence number in the shadow log, the publisher can retire failed work without creating an externally visible hole. Single-publisher liveness is enforced by a concrete token `struct dt_pub_token { u32 epoch; u32 token_ver; u64 cursor; u64 owner_cookie; u64 lease_deadline; };`. Any producer or helper that notices newly resolved work may attempt `cmpxchg()` from the observed token image to a new image with its `owner_cookie`, same epoch, incremented `token_ver`, and current `cursor = head_seq`. Only the owner may perform device-visible actions. The owner refreshes `lease_deadline` and `cursor` with `WRITE_ONCE()` after each bounded publication batch. A contender may steal only when `ktime_get_mono_fast_ns()` exceeds `lease_deadline`, a second read after `smp_rmb()` shows the same `cursor` and `token_ver`, and a full-image `cmpxchg()` installs a new owner and incremented `token_ver`. The previous owner checks `token_ver` before each publication batch and must stop on mismatch. Slot reuse is ABA-safe because reclamation requires both `published_seq >= slot.seq` and a generation increment before the slot can return to `FREE`; consumers always validate `seq + epoch + generation`, not index alone. In the virtio embodiment, producers never write device-visible vring state directly. They populate only shadow descriptor chains in `dt_vq_intent`, including SG metadata, indirect-table pointer if used, and notification-hint flags. The publisher alone copies committed intents into the real vring. For split rings, it writes descriptor table entries and `avail->ring` heads, then issues `dma_wmb()` or `virtio_wmb(vq->weak_barriers)` before updating `avail->idx`; only after the idx store does it evaluate kick suppression using the same old/new idx comparison model as `virtqueue_kick_prepare()` and then call `virtqueue_notify()` if needed. For packed rings, it writes addr/len/id/next fields first, issues `dma_wmb()/virtio_wmb()`, and writes the final flags/ownership bit containing the current wrap counter last; wrap counter toggling and event-structure evaluation are performed only by the single publisher. In the register-script embodiment, producers write typed register intents rather than raw bus operations. `op_class` is selected from a semantics taxonomy: `RW_MERGE`, `WRITE_LITERAL`, `W1C_FAULT`, `READ_CLEAR_STATUS`, `TOGGLE_PULSE`, `VOLATILE_PRECIOUS`, and `REVALIDATE_ON_EPOCH`. The single executor publishes only a contiguous committed prefix and emits a class-safe script: `RW_MERGE` operations may be coalesced into one shadowed `regmap_update_bits()` group, but `W1C_FAULT`, `READ_CLEAR_STATUS`, `TOGGLE_PULSE`, and `VOLATILE_PRECIOUS` are isolated in fixed order and never merged with unrelated setters. `RW_MERGE` and `WRITE_LITERAL` are idempotent only within the same epoch; `W1C_FAULT`, `READ_CLEAR_STATUS`, and `TOGGLE_PULSE` are explicitly non-idempotent and may not be replayed across epoch changes; only `REVALIDATE_ON_EPOCH` entries may be regenerated through a registered revalidation callback. On an epoch bump, unresolved older-epoch slots are turned into `ABORT_RESET`, and reset-sensitive slots are requeued only through revalidation, not replayed blindly.

2.4 Architecture

ASCII control flow: Producer(s) on any CPU -> reserve monotonic seq with atomic64_fetch_inc -> fill shadow slot(seq, epoch, generation, op_class, payload) -> resolve with smp_store_release(COMMIT or ABORT); contender noticing resolved work -> cmpxchg(dt_pub_token) -> single publisher owner -> scan from head_seq while tuple matches. If slot is RESERVED or tuple-mismatched, scan stops. If slot is ABORT, retire and reclaim only. If slot is COMMIT, dispatch by op_class either to a reg-class executor or to a virtio ring publisher. Split-ring path is descriptor writes plus avail-ring heads, then virtio_wmb/dma_wmb, then avail->idx update, then kick-suppression evaluation and optional notify. Packed-ring path is descriptor fields except final ownership/flags, then virtio_wmb/dma_wmb, then final flags with wrap state, then event-suppression evaluation. Ordering contract: producer writes slot payload before release-store on state; publisher acquire-loads state before acting; publisher release-stores cursor and published_seq before token handoff. Epoch/reset contract: `dt_epoch_bump()` sets reset-pending with release semantics, blocks new publication, invokes `virtqueue_disable_cb()` in the virtio embodiment, acquires or steals the publisher token, performs reset such as `virtio_reset_device()` or register-domain reset handling, increments `global_epoch`, converts unresolved older-epoch slots to `ABORT_RESET`, and only then permits new publication. Producers snapshot `global_epoch` before slot fill and publish `ABORT_EPOCH_MISMATCH` if the epoch changes before resolution. Liveness and fairness: batch quota Q and lease interval L bound owner residency; takeover occurs only after lease expiry plus unchanged token_ver and cursor; hardirq or local-IRQ-disabled producers never livelock on RTM because retry budgets are zero in those contexts and publication can be deferred to a helper. CPU hotplug, NAPI migration, and workqueue migration are handled by token-version invalidation plus lease expiry, so a stalled owner can be replaced without double publication.

2.5 Implementation Plan

Phase 1: add a driver-private `dt_bridge` containing `tail_seq`, `head_seq`, `global_epoch`, `dt_pub_token`, and a power-of-two slot array sized to queue depth or register-intent budget, and implement the slot state machine `FREE -> RESERVED -> COMMIT/ABORT -> PUBLISHED -> FREE` with generation-based reclaim. Phase 2: implement the non-TSX correctness path first using plain atomics plus a short raw spinlock or qspinlock only where slot-reuse metadata must be serialized; use `smp_store_release()` on state resolution and `smp_load_acquire()` in the publisher scan; add explicit ABORT codes and a bounded contiguous-prefix scanner. Phase 3: implement full-image cmpxchg acquisition of `dt_pub_token`, lease refresh, bounded publication batches, takeover on expired lease, and owner-version checks before each publish batch; store `cursor` and `published_seq` with release semantics; add CPU-hotplug and reset hooks that invalidate ownership by incrementing `token_ver`. Phase 4: on `CONFIG_X86`, add an RTM helper enabled only when `boot_cpu_has(X86_FEATURE_RTM)` and a module or static-key policy allows it; set retry budget to 0 in hardirq or local-IRQ-disabled paths, 1 in softirq or NAPI, and at most 2 in task context; map conflict aborts to one retry, capacity/fault/policy aborts directly to fallback, and repeated aborts to temporary per-CPU backoff; perform no MMIO, vring store, notifier, or regmap operation inside RTM. Phase 5: wrap `virtqueue_add_*` style producer sites so they populate `dt_vq_intent` shadow descriptors rather than real vrings; implement split-ring publication as `write desc -> write avail ring heads -> virtio_wmb/dma_wmb -> write avail->idx -> kick_prepare/notify`; implement packed-ring publication as `write fields except ownership flags -> virtio_wmb/dma_wmb -> write final flags with wrap bit -> event suppression evaluation`; keep one ring-format implementation per queue instance. Phase 6: define a per-driver register class table describing mask, class, volatility, preciousness, idempotency, and revalidation policy for each address or field; route regulator, GPIO, IRQ, thermal, and fault paths through intent creation; publish class-safe scripts using `regmap_write()`, `regmap_update_bits()`, or isolated reads as dictated by class. Phase 7: add tracepoints and counters for slot reserve, commit, abort reason, publish batch length, steal events, epoch bumps, kick decisions, RTM attempts and aborts, and contiguous-prefix stall reasons so tuning does not change semantics.

2.6 Validation Plan

1. Correctness and ordering: KUnit tests for slot state transitions, contiguous-prefix stopping on RESERVED or tuple mismatch, ABORT retirement, generation-based ABA prevention, publisher-token takeover, and epoch-bump invalidation; LKMM-style review of release/acquire relations to prove that no later COMMIT can leapfrog an unresolved earlier slot. 2. Virtio compliance: run split-ring and packed-ring tests against virtio-net and virtio-blk with forced descriptor exhaustion, indirect descriptors, event-index suppression, packed wrap transitions, and reset storms; confirm that all device-visible indices are contiguous, no stale pre-reset slot is published, and notifications match `virtqueue_kick_prepare()` semantics. 3. Register semantics: inject mixed `RW_MERGE`, `W1C_FAULT`, `READ_CLEAR_STATUS`, and `TOGGLE_PULSE` workloads into a regmap-backed test driver; verify that mergeable fields coalesce, side-effect classes remain isolated, and reset or suspend-resume causes `ABORT_RESET` or revalidation rather than stale replay. 4. TSX behavior: on x86 systems with RTM enabled, measure reservation latency and abort statistics under contention, interrupt load, and cache pressure; force RTM off via policy, microcode, or VM configuration and show unchanged correctness and bounded fallback progress; confirm that hardirq and NAPI paths do not livelock because retry budgets are zero or one by context. 5. Liveness and failure injection: test CPU hotplug, workqueue migration, stalled-owner simulation, NAPI budget exhaustion, and publisher preemption; verify that takeover occurs only after lease expiry and exact token-image cmpxchg, and that double publication does not occur even when owners race around handoff. 6. Performance: compare against baseline virtqueue lock submission and regmap mutex or update_bits paths on multicore Xeon workloads, reporting throughput, p99 submission latency, bus transaction count, RTM hit rate, and reset recovery time while demonstrating identical externally visible device behavior.

4. Risks & Mitigations

  • Risk: prior-art overlap with generic MPSC queues or virtqueue batching. Mitigation: claim the explicit ABORT-occupies-sequence rule, the exact publisher token with lease, version, and cursor, the seq+epoch+generation tuple, and the ring-format-specific publication ordering rather than generic batching.
  • Risk: TSX disablement on deployed x86 systems reduces upside. Mitigation: make the fallback path the primary correctness path and position RTM as an optional acceleration knob only for cache-resident metadata.
  • Risk: publisher handoff bugs could duplicate or skip publication. Mitigation: require full-image cmpxchg on the token, release/acquire cursor updates, batch-boundary version checks, and generation-based reclaim tests under fault injection.
  • Risk: register taxonomy may be incomplete for a given driver. Mitigation: require an explicit per-driver class table with conservative default classes such as `VOLATILE_PRECIOUS` or `WRITE_LITERAL` and disallow merge optimization until a field is classified.
  • Risk: reset and suspend/resume interactions may be subtle in real drivers. Mitigation: make epoch bump a first-class API invoked from reset, freeze, and restore hooks, with mandatory callback disablement and revalidation hooks for reset-sensitive payloads.
