2. Technical Invention Disclosure

2.1 Problem Statement

Metadata-heavy XDP programs on x86 increasingly combine packet parsing, queue-derived policy, RX hash or VLAN reads, timestamps, data_meta parsing, and bounded flow summarization. Today the generic x86 eBPF JIT must assume only conservative verifier facts and therefore emits address materialization, alignment handling, and guard sequences that are safe across all queues and all attachment instances of the same verified program. That leaves queue-local performance on the table when a NIC driver actually knows a narrower runtime layout for a given RX queue and RX epoch. Existing dispatcher and trampoline mechanisms do not solve this because they fan out among programs or manage attachment lifetime, but they still treat the JIT image as a shared property of bpf_prog and do not provide an attachment-owned, independently versioned specialized image selected from queue-local runtime facts. The observability story is also weak: when multiple specialized images are created and retired live from identical bytecode, RIP-only perf attribution becomes ambiguous, and overflow handlers cannot safely chase attachment pointers. A further gap exists for security analytics and ML-assisted flow inspection: generic BPF maps are not an acceptable model for a driver fast path that must be allocation-free, verifier-irrelevant, and bounded under softirq or threaded-NAPI execution. What is missing is a precise kernel contract that lets drivers publish only BPF-observable queue facts, proves that specialization is a strict subset of verifier permission, attaches variant ownership to the attachment object rather than the shared program, and exports fixed-size numeric observability suitable for flow-based IDS pipelines and pr_fmt-correlated operational debugging.

2.2 Prior-Art Gap

Upstream bpf_dispatcher and multi-program XDP dispatch select among attached programs, but they do not define a queue-epoch descriptor of BPF-observable layout facts, do not perform a verifier-side admissibility intersection for specialization facts, and do not give each attachment its own bounded specialized-image cache. Existing trampoline and bpf_link lifetime mechanisms manage attach and detach safely, but they do not provide queue-indexed variant ownership or generation-tracked JIT images independent of shared bpf_prog state. Existing per-program JIT images are shared across attachments and cannot model independent queue-local specialization without either mutating shared state or cloning whole programs. Perf attribution and symbol lookup solutions infer execution from RIP, symbol names, or address ranges; they do not provide an NMI-safe numeric tag written at execution time and readable without dereferencing freed objects. Fixed-size sketches and BPF maps are known generally, but the specific combination here is narrower: a driver-side per-CPU synopsis plane updated outside BPF map semantics, with a compile-time probe cap, little-endian fixed record ABI, queue-epoch correlation, and explicit ML-feature fields derived from the same validated descriptor contract. The novelty boundary is therefore the combined queue-epoch descriptor, verifier clamp, attachment-owned variant cache and dispatch point, numeric attribution plane, and bounded synopsis ABI.

2.3 Proposed Invention

The invention introduces a concrete kernel contract. 1. Topology Descriptor ABI and invariants. A driver publishes a queue-local immutable descriptor with canonical facts derived only from xdp_buff and xdp_rxq_info state that is already BPF-observable or directly constrains helper-legal pointer movement. The in-kernel canonical layout is: struct xdp_topology_desc_v1 { u16 abi_version; u16 len; u32 flags; u32 queue_epoch; u32 rxq_index; u16 data_meta_off_min; u16 data_meta_off_max; u16 headroom_min; u16 headroom_max; u8 data_align_log2; u8 meta_align_log2; u16 adj_head_neg_max; u16 adj_head_pos_max; u16 adj_meta_neg_max; u16 adj_meta_pos_max; u32 metadata_bits; u32 verifier_caps; u64 layout_cookie; u64 desc_crc64; } __aligned(8); The descriptor contains no pointers. abi_version starts at 1. len allows forward extension. In-kernel storage is native-endian and naturally aligned; any export to userspace is serialized to the identically ordered little-endian uapi struct xdp_topology_desc_uapi_v1. metadata_bits is limited to BPF-visible metadata presence, such as RX hash availability, VLAN metadata presence, hardware timestamp presence, and driver-specific metadata blocks only if those blocks are reachable through existing verifier-approved data_meta access. layout_cookie is a stable numeric digest of the driver's queue layout contract, not a pointer. The canonical source of truth is xdp_rxq_info for queue identity and queue capabilities, and xdp_buff for packet/data/data_meta/headroom relationships. A driver may not publish cacheline placement, DMA ring addresses, or any non-BPF-observable hardware-private fact. 2. Publication and epoch update contract. The descriptor is replaced only at a queue quiesce point. For live queue update the driver: masks the queue IRQ; disables or pauses RX refill; calls napi_disable() or equivalent queue-drain primitive; waits until the current NAPI poll instance exits; computes the new descriptor; submits it to the kernel validator; allocates a new immutable descriptor object; and then publishes it with rcu_assign_pointer(rxq->topology_desc, new_desc) followed by queue re-enable. The queue_epoch increments on any change that can affect BPF-visible layout or helper legality, including firmware profile change, metadata enable or disable, MTU/headroom change, queue reset, XDP program attachment change that alters verifier caps, or driver path switch that changes data_meta placement. In-flight packets from the old epoch finish under the old descriptor because the queue is quiesced before publication. Netdev teardown or attachment teardown first clears the queue's variant pointer and attachment owner pointer, disables queues, invokes synchronize_net() to flush outstanding softirq readers, and retires descriptors and images through call_rcu(). No reader can observe a partially initialized descriptor because readers fetch only an RCU-protected immutable pointer. 3. Kernel-side validation and verifier clamp. During verification, the kernel computes for each program and attach context a specialization manifest containing: admissible ranges for data_meta offset and headroom-sensitive offsets; admissible helper adjustment ranges; metadata presence bits that are consumed by the program; and a per-insn may_specialize mask identifying load, compare, and constant-materialization sites whose operands may be constant-folded if a descriptor supplies a narrower fact. On publish, the validator intersects the driver descriptor with this manifest. A descriptor field is accepted only if it is a subset of the verifier-proven envelope. If a driver publishes an overly permissive range or a metadata bit not proven relevant and safe, that field is rejected and the queue falls back to the baseline image for that attachment. The effective specialization descriptor is therefore clamp(desc_from_driver, verifier_manifest), never union(desc, verifier). This closes the soundness gap for buggy or malicious drivers. 4. x86 JIT specialization rules. The x86 JIT may specialize only at instructions marked in may_specialize. Allowed transforms are: constant-folding of immediate values derived from the effective descriptor; deletion of bytecode-equivalent compare branches whose outcome is fully determined by descriptor facts; substitution of a verifier-permitted aligned load form for an already existing load when width is unchanged; and reordering of already present loads inside the same verifier-dominated safe region when doing so adds no memory touch and preserves Spectre-v1 hardening shape. Disallowed transforms are: adding any new load, store, helper call, or branch target; widening any memory access; hoisting a load across a verifier guard or helper boundary; changing the order or presence of hardened array-index masking or fence sequences when BPF Spectre mitigations are enabled; or specializing on non-observable driver-private facts. The JIT emits a compact audit trailer per variant containing the variant_gen, descriptor hash, and transformed-insn bitmap, and it can run a debug-time assert pass that checks every transformed instruction against the verifier manifest. 5. Attachment-owned variant owner and cache. Variant ownership lives in a dedicated attachment object, for example struct xdp_attach_variant_owner, referenced from the bpf_link for link-based XDP and from the per-netdev XDP attach object for legacy attach. The owner contains stable attach_id, a baseline image pointer, a fixed-size variant cache, and a queue-to-variant binding table. Shared bpf_prog and bpf_prog_aux remain unmodified except for immutable verifier manifest storage. A concrete bounded design is VARIANT_CACHE_MAX = 8 specialized images per attachment, selected by a fixed-order open-addressed cache keyed by {descriptor_hash, verifier_manifest_hash, jit_feature_mask}. Variant lookup probes at most 8 slots. Variant creation is not on the packet path; on cache miss the queue executes the baseline image and queues background creation in attach-control context. An entry is evictable only when refcnt == 0, it is not the current binding for any queue, and its retired bit is set; otherwise miss falls back to baseline rather than blocking. Specialized text is allocated with the existing BPF JIT text allocator, one text allocation per variant, and variant metadata lives in a dedicated kmem_cache. Retirement uses refcount drop plus call_rcu(). Tail-calls and bpf2bpf operate normally because the attachment-owned specialization affects only the entry program image; a tail-called callee executes its own baseline or its own attachment-scoped owner if separately attached in another hook. bpf_redirect, cpumap, and devmap behavior are unchanged because the descriptor contract is defined only for the current hook's entry context. 6. Dispatch integration without shared-program mutation. The indirection point is attachment-owned. For XDP multi-program dispatch, bpf_dispatcher first resolves the attachment slot as it already does; the slot target is a small attachment thunk owned by the variant owner, not the shared program image. That thunk reads the queue's RCU-published current variant binding and tail-jumps to either owner->baseline_image or owner->variant[slot]. For single-program attach, the netdev XDP attach object points directly to the same thunk. Concurrent attach/detach is safe because the dispatcher or attach object swaps one RCU-protected owner pointer, while queue bindings and variant pointers are independently RCU-protected and refcounted. No mutation of bpf_prog->bpf_func or shared per-prog JIT image is required. 7. NMI-safe attribution. Each CPU has a fixed-depth execution-tag stack, TAG_STACK_MAX = 4, stored in per-CPU aligned memory: struct xdp_exec_tag_v1 { u32 seq; u32 attach_id; u32 variant_gen; u16 variant_slot; u16 flags; u32 queue_epoch; }; At XDP entry the attachment thunk increments a per-CPU depth counter, selects the stack slot, writes seq odd, then WRITE_ONCE stores attach_id, variant_gen, variant_slot, flags, and queue_epoch, executes smp_wmb(), and writes seq even. On return it decrements depth. If depth exceeds TAG_STACK_MAX, the writer stores a sentinel overflow tag with flags.XDP_TAG_OVERFLOW = 1. NMI or perf overflow readers never write these structures. They read depth with READ_ONCE, sample the top slot, and retry at most ATTRIB_RETRY_MAX = 2 times if seq is odd or changes. Because the payload contains only numeric IDs, readers never dereference freed objects. Nested NMI or perf recursion cannot corrupt tags because readers are read-only and writers only run on hook entry or exit, not in NMI context. Perf, LBR, or PEBS sample augmentation appends a fixed little-endian record xdp_exec_tag_uapi_v1 so post-processing does not rely on address-to-symbol matching. 8. Bounded flow synopsis plane and userspace ABI. The optional flow synopsis runs only in NAPI softirq or threaded-NAPI context for queues whose owner has synopsis enabled. It is explicitly verifier-irrelevant: it is driver and core-kernel code, not a BPF map update. Each CPU has a fixed-size array FLOW_SYN_SLOTS = 64 of synopsis records and a monotonically wrapping cursor. Update is lock-free and per-CPU only. The writer uses this_cpu_ptr() and, on PREEMPT_RT or threaded NAPI, enters with migrate_disable() so CPU-local ownership is preserved for the bounded critical section. Each packet computes a compact flow key from already available fields, for example Toeplitz hash, L3/L4 protocol tuple class, packet length bucket, queue id, metadata bits, and variant_gen. The update probes at most PROBE_MAX = 4 slots using a compile-time fixed probe sequence derived from the flow hash. Each touched record update is wrapped with a per-record u32 seq version: seq odd, WRITE_ONCE fields, smp_wmb(), seq even. Readers that see mismatched seq values retry the record. No allocation, list walk, map lookup, or data-dependent loop is permitted. On CPU hotplug, affected queues are quiesced and rebound; the outgoing CPU's synopsis buffer remains publishable until synchronize_net() and then is retired. The userspace interface is a generic netlink family XDPV_GENL_FAMILY with commands GET_ATTACH_INFO, GET_QUEUE_DESC, and GET_FLOW_SYNOPSIS. Synopsis records use a fixed little-endian ABI: struct xdp_flow_synopsis_rec_v1 { __le16 abi_version; __le16 rec_len; __le32 cpu; __le32 attach_id; __le32 variant_gen; __le32 queue_epoch; __le32 flow_hash; __le16 l3_proto; __le16 l4_proto; __le16 len_bucket; __le16 tcp_flag_mask; __le32 packet_count; __le32 byte_count; __le32 metadata_bits; __le32 ml_feature_bits; __le32 seq; }; This is expressly a fixed-record feature plane for downstream IDS or ML classification, not a generic sketch API. Optionally, state transitions such as attach, fallback, eviction, or descriptor rejection may emit rate-limited pr_fmt lines carrying the same stable numeric tags for operator correlation, but pr_fmt is diagnostics only and never the control plane. 9. x86 microarchitectural benefit. By specializing only on validated descriptor facts, the x86 JIT can reduce dependent address arithmetic, shorten hot traces, and improve I-cache and uop-cache residency while preserving verifier and Spectre safety. The design also improves PMU interpretability because LBR and PEBS samples carry direct attachment and variant identity rather than requiring address-range inference.

Canonical data sources and invariants:
- Queue identity and queue capabilities come from xdp_rxq_info.
- Packet/data/data_meta/headroom relationships come from xdp_buff.
- Only BPF-observable or helper-legality facts may enter the descriptor.
- Descriptor objects are immutable after publication.
- Queue-epoch changes require queue quiesce before rcu_assign_pointer().
- Specialized x86 transforms are limited to may_specialize-marked instructions and must preserve verifier and Spectre hardening semantics.

2.5 Implementation Plan

1. Add a verifier-side specialization manifest. Extend the XDP verifier path to compute admissible descriptor ranges, metadata-bit dependencies, and a per-insn may_specialize mask, stored immutably with the verified program but not used to mutate shared JIT state. 2. Introduce xdp_topology_desc_v1 and validator helpers in net/XDP core. Provide helpers that derive canonical values from xdp_buff and xdp_rxq_info, validate driver submissions, and compute the effective descriptor as the intersection of driver facts and verifier facts. 3. Extend XDP attachment state with xdp_attach_variant_owner. For bpf_link attach, anchor the owner in the link object; for legacy netdev XDP attach, anchor the owner in the per-netdev attach state. Add stable attach_id allocation, queue_binding tables, bounded variant cache bookkeeping, refcounts, and RCU retirement. 4. Add the attachment thunk integration point to dispatcher and single-program attach paths. The thunk must be attachment-owned, RCU-updatable, and able to resolve queue-local variant bindings without touching shared bpf_prog->bpf_func. 5. Implement x86 JIT specialization in arch/x86/net/bpf_jit_comp.c. Consume the effective descriptor and verifier manifest, enforce the allowed-transform list, emit the audit trailer, and fall back to the baseline image if any transform precondition fails. Ensure the baseline image remains available at all times. 6. Implement queue publication/update hooks in driver support code and net core helpers. Require queue quiesce before descriptor publish, define epoch bump triggers, and wire live replacement, queue reset, and teardown to RCU-safe retirement. 7. Implement NMI-safe numeric attribution. Add per-CPU tag-stack data, entry/exit writer helpers in the attachment thunk, and perf/LBR/PEBS sample augmentation with fixed little-endian xdp_exec_tag_uapi_v1 payloads. 8. Implement the optional bounded flow synopsis plane. Add per-CPU fixed-slot storage, O(PROBE_MAX) update logic, seq-versioned record writes, PREEMPT_RT-safe migrate_disable() handling for threaded NAPI, CPU-hotplug rebinding, and generic netlink dump commands. 9. Add operator diagnostics. Emit rate-limited pr_fmt messages only on descriptor rejection, variant creation, fallback, eviction, and teardown with stable numeric identifiers so logs line up with perf and netlink output. 10. Document UAPI and invariants. Specify abi_version handling, little-endian serialization rules, alignment requirements, and the meaning of VARIANT_CACHE_MAX, TAG_STACK_MAX, FLOW_SYN_SLOTS, and PROBE_MAX.

2.6 Validation Plan

1. Verifier soundness tests: create selftests where drivers attempt to publish overly permissive data_meta bounds, helper adjust ranges, or metadata bits; confirm the validator rejects specialization and the baseline image executes. 2. JIT transform audit tests: compare baseline and specialized images for programs covering aligned loads, metadata reads, helper adjustments, Spectre-hardened bounds checks, tail-calls, and bpf2bpf; assert no new memory references, no widened loads, and no deleted hardening sequences. 3. Queue-epoch concurrency tests: repeatedly change MTU, headroom, metadata enablement, queue reset, and attachment state while traffic flows; confirm no packet executes under a partially updated descriptor and that old descriptors retire only after RCU grace. 4. Attach/detach lifecycle tests: stress concurrent bpf_link attach/detach, legacy XDP replace, multi-queue execution, and netdev teardown; confirm queue_binding, owner swap, and variant retirement remain race-free. 5. Dispatcher integration tests: run stacked XDP programs through bpf_dispatcher and single-program attach; verify the attachment thunk preserves stacking order and that variant selection is attachment-local. 6. NMI/perf attribution tests: trigger perf overflow, nested PMU interrupts, and recursive sampling while XDP traffic runs; confirm xdp_exec_tag_uapi_v1 records are numerically consistent, seq retries bound correctly, and overflow sentinel behavior is explicit. 7. Flow synopsis tests: verify O(PROBE_MAX) update cost with adversarial hash collisions, correctness of seq-versioned snapshot retries, PREEMPT_RT threaded-NAPI operation with migrate_disable(), and CPU-hotplug rebinding without stale-pointer exposure. 8. PMU and microarchitectural evaluation: on Xeon, measure cycles per packet, uop delivery, I-cache misses, branch mispredicts, and LBR/PEBS attribution quality for baseline versus specialized images under metadata-heavy XDP workloads and flow-based IDS feature extraction. 9. UAPI compatibility tests: validate generic netlink dumps, little-endian record decoding, abi_version negotiation, record-length forward compatibility, and pr_fmt correlation using stable numeric IDs. 10. Negative robustness tests: fuzz descriptor publication inputs, variant cache misses, tag-stack overflow, and synopsis buffer saturation; confirm deterministic fallback to baseline rather than silent corruption.

4. Risks & Mitigations

  • Risk: a buggy or malicious driver publishes an over-broad descriptor and causes unsound specialization. Mitigation: publish-time validator intersects driver facts with verifier-proven envelopes and disables specialization on any non-subset field.
  • Risk: live queue reconfiguration races with packet execution. Mitigation: require queue quiesce before rcu_assign_pointer(), use immutable descriptor objects, and retire old objects only after synchronize_net() or call_rcu() as appropriate.
  • Risk: attachment-owned variant ownership could be mistaken for shared-program mutation and invite prior-art attack. Mitigation: bind ownership to bpf_link or netdev XDP attach objects, keep shared bpf_prog images unchanged, and document the attachment thunk indirection point explicitly.
  • Risk: x86 specialization could accidentally weaken Spectre hardening or widen loads. Mitigation: use a verifier-produced may_specialize mask, enforce an allowlist of transforms, emit an audit trailer, and keep the baseline image permanently available for fallback.
  • Risk: NMI/perf recursion corrupts attribution state. Mitigation: store only numeric IDs in a fixed-depth per-CPU tag stack, use seq-versioned writes with READ_ONCE/WRITE_ONCE and bounded retries, and mark overflow explicitly with a sentinel flag.
  • Risk: the bounded flow synopsis is dismissed as a generic sketch. Mitigation: define exact record layout, little-endian UAPI, compile-time PROBE_MAX semantics, per-CPU execution restrictions, and the specific queue-epoch and variant correlation fields.
  • Risk: PREEMPT_RT and CPU hotplug break per-CPU synopsis assumptions. Mitigation: restrict updates to NAPI softirq or threaded-NAPI context, use migrate_disable() in threaded paths, quiesce and rebind queues on hotplug, and retire old buffers after grace periods.
  • Risk: cache pressure causes unpredictable latency. Mitigation: create specialized variants only off the packet path, bound cache size and probe count, and fall back to the baseline image instead of performing synchronous allocation or eviction in the hot path.
  • Risk: userspace tools misinterpret evolving ABIs. Mitigation: include abi_version and record length in every exported structure, serialize in little-endian form, and preserve forward compatibility through additive field extension only.
