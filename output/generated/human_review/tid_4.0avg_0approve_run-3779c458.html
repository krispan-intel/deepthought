2. Technical Invention Disclosure

2.1 Problem Statement

Linux RCU nocb offload already moves callback invocation away from protected CPUs, but the remaining callback-drain behavior is still governed by static heuristics, local backlog logic, polling modes, lazy-callback state, and administrator tuning rather than by package-scoped energy cost. On x86 servers, nocb drain bursts can become an expensive form of synchronization housekeeping when the package is in a high-cost uncore regime often correlated with coherence traffic, memory stalls, or L2/LLC miss pressure. Naively delaying drains is unsafe because callback retention can accumulate, reclaim can intensify, and catch-up bursts can damage latency on housekeeping CPUs. The kernel therefore lacks a practical mechanism that uses x86 RAPL telemetry at the same topology scope as the available energy counters while keeping telemetry acquisition, provider locking, and provider error handling entirely out of enqueue-side call_rcu paths, out of grace-period advancement, and out of expedited synchronization semantics.

2.2 Prior-Art Gap

Existing RCU nocb controls include rcu_nocb_poll polling mode, existing nocb batching and blimit-style drain limits, bypass-list admission and spillover heuristics, nocb_bypass_lim_per_jiffy-style throttles, lazy-callback deferral, call_rcu_hurry() overrides, and existing rcutree sysctls and wake heuristics. This invention does not change callback enqueue placement, bypass-list ordering, callback invocation order, grace-period start or completion rules, FQS cadence, expedited synchronize_rcu behavior, or the semantics of lazy callbacks and hurry operations. Prior energy-aware mechanisms such as EAS, cpufreq, thermal governors, powercap, and perf-based power accounting operate on task placement, frequency, or platform power limits, not on RCU nocb callback-drain cadence. Prior art also does not define the combined mechanism claimed here: a cached-RAPL provider with explicit freshness and wrap rules; a provable single-package or single-die admission algorithm for nocb drainers; a stable policy-word interface consumed only by rcuo threads; and a concrete safety clamp based on oldest callback age, backlog depth, and a lockless memory-pressure override.

2.3 Proposed Invention

Introduce a Package Energy Broker for NOCB (PEB-N) that exists entirely on housekeeping CPUs and has three explicit interfaces. First, a cached-RAPL provider exports per-package or per-die samples as {sample_ns, window_ns, delta_energy_nj, avg_power_uw, max_energy_range_uj, energy_unit_nj, flags}. Only the provider may touch arch/x86 RAPL helpers or powercap-owned state, only in process context, and it publishes samples with seqcount so readers never call into powercap internals. A sample is FRESH when sample_age_ns <= 4,000,000 and VALID when sample_age_ns <= 20,000,000 and window_ns is below wrap_guard_ns = (max_energy_range_uj * 1000) / max(peak_domain_power_uw, 1) / 4; otherwise the sample is marked STALE or INVALID and the broker writes baseline policy. Second, the broker runs every 2,000 us by default, clamped between 1,000 us and 10,000 us, and for each candidate domain computes domain_ready_sum, domain_oldest_age_us, per-member local_ready_max, and a normalized energy-per-ready-callback metric nepcb_q16 = (delta_energy_nj << 16) / max(domain_ready_sum, 1). It also samples two lockless VM signals in the same work item: a low-watermark predicate over populated zones and a direct-reclaim rate derived from PGSCAN_DIRECT deltas. Memory pressure asserts after two consecutive broker periods in which any populated zone is below low_wmark_pages(zone) plus a 2 percent safety margin or PGSCAN_DIRECT increases by at least 128 pages during the window; it clears only after eight consecutive clear periods. Third, the broker publishes a single u64 policy word per stable domain. Each rcuo callback-drain kthread READ_ONCE()s that word once after wakeup and before its local drain batch, never in enqueue or GP code, and uses it only to decide additional deferral, local callback quota, runtime budget, and a short partial-drain rearm. The feature is compile-time excluded when CONFIG_PREEMPT_RT is enabled and runtime-disabled if any participating rcuo thread is not SCHED_NORMAL, if its effective affinity intersects non-housekeeping or isolated CPUs, if the RAPL provider is stale or unsupported, or if the domain-stability proof is not satisfied. Callback order is unchanged because each kthread still drains its own lists in existing order; grace-period forward progress is unchanged because GP code never reads policy; and expedited sync is unchanged because FORCE_DRAIN and BASELINE both resolve to zero additional delay.

2.4 Architecture

Layer 1: mechanically checkable domain mapper and admission state machine.
- Effective affinity for each rcuo kthread is recomputed as task->cpus_ptr intersect cpu_online_mask intersect housekeeping_cpumask(HK_TYPE_RCU).
- Every CPU in that effective mask is mapped to a tuple {pkg = topology_physical_package_id(cpu), die = topology_die_id(cpu) if available else -1}.
- If the mask is empty, spans multiple tuples, intersects a non-housekeeping CPU, or the thread has a non-SCHED_NORMAL policy, the domain state is DISABLED and the broker publishes BASELINE.
- If the tuple or effective mask changes, or a CPU-hotplug seq counter changes, the domain enters PROBING.
- The domain becomes STABLE only after 3 consecutive broker periods observe the same effective mask and same tuple with FRESH provider data.
- CPU hotplug is detected immediately with cpuhp_setup_state_nocalls()-registered callbacks that bump a topology sequence. cpuset and sched_setaffinity changes are detected by the periodic recomputation of task->cpus_ptr on every broker tick. nohz_full and isolcpus are folded into the housekeeping mask; if a platform supports runtime housekeeping-mask rebuild, that event also bumps the topology sequence and forces DISABLED.

Layer 2: cached-RAPL provider.
- One provider worker per package or die runs only on housekeeping CPUs.
- Provider concurrency model: single writer under provider-local mutex; multiword sample published with seqcount; broker readers retry on seq mismatch; rcuo threads never read provider state.
- Provider may take only provider-local locks and existing power/rapl driver locks. It never runs while holding rq locks, RCU core locks, cpuset mutex, zone locks, or nocb list locks.
- Unsupported, virtualized, or permission-blocked RAPL domains set flags to INVALID; broker then writes BASELINE and performs no shaping.

Layer 3: domain metrics and memory-pressure override.
- Each nocb group exposes lockless ready-count and oldest-ready timestamp fields that are maintained in existing nocb processing context, not by consulting policy in enqueue fast paths.
- Broker computes domain_ready_sum as the sum of member ready counts and domain_oldest_age_us as the maximum oldest-ready age across members.
- Memory pressure is asserted when either: (a) any populated zone is below low_wmark_pages(zone) + 2 percent margin for 2 consecutive periods, or (b) PGSCAN_DIRECT delta is >= 128 pages in a broker window for 2 consecutive periods. It clears after 8 consecutive clear periods. No global locks or long critical sections are introduced because all reads occur in delayed_work context and use existing vmstat and watermark snapshots.

Layer 4: 64-bit policy word and thresholds.
- bit 0..1: mode (0 BASELINE, 1 CONSERVE, 2 CATCHUP, 3 FORCE_DRAIN)
- bit 2: memory_pressure flag
- bit 3: provider_stale flag
- bit 4..15: extra_deferral_us_div4
- bit 16..27: local_quota_callbacks
- bit 28..39: runtime_budget_us_div4
- bit 40..47: partial_rearm_us_div4
- bit 48..55: generation
- bit 56..63: reserved
- Broker clamps all encoded values to: extra deferral 0..2000 us, local quota 32..1024 callbacks, runtime budget 64..2048 us, partial rearm 0..500 us.
- Each domain maintains EWMA baseline energy cost base_nepcb_q16 over the last 64 non-pressured fresh windows.
- Enter CONSERVE after 3 consecutive fresh windows with nepcb_q16 >= base_nepcb_q16 * 5 / 4, domain_ready_sum >= 64, and domain_oldest_age_us < 4000. CONSERVE encodes deferral 256..2000 us, quota 32..256, runtime budget 64..512 us, rearm 64..256 us.
- Enter CATCHUP when nepcb_q16 <= base_nepcb_q16 * 7 / 8 for 2 windows, or when domain_oldest_age_us >= 6000, or when domain_ready_sum >= 512. CATCHUP uses deferral 0 us, quota 256..1024, runtime budget 512..2048 us, rearm 0..64 us.
- Enter FORCE_DRAIN immediately when memory pressure is asserted, when domain_oldest_age_us >= 8000, when domain_ready_sum >= 4096, or when any member local_ready count >= 1024. FORCE_DRAIN encodes zero extra deferral and maximum local quota and budget.
- BASELINE is published whenever the domain is unstable, unsupported, stale, PREEMPT_RT-disabled, manually RT-promoted, or otherwise outside the admissible envelope.

Layer 5: drain-side consumption and safety properties.
- Each rcuo thread READ_ONCE()s the policy once per wake cycle and applies only the encoded delay and bounds to its own local drain loop.
- No policy lookup occurs during callback enqueue, callback list linking, callback ordering, GP state transitions, or expedited sync paths.
- Existing wake sources across multiple nocb CPUs are unchanged; the invention does not add new enqueue-side cross-CPU wake fanout. A shared domain policy merely causes already woken member rcuo threads to drain more or less aggressively.
- Hard safety property: additional retention introduced by shaping is bounded. A callback may see at most 2000 us extra local deferral, and any domain whose oldest ready callback reaches 8000 us is forced to zero-deferral mode by the next 2000 us broker period. Therefore the extra retention attributable to shaping is bounded to approximately one max deferral plus one broker period beyond the force threshold, while additional retained callback count is capped by the 4096-domain and 1024-local force-drain thresholds plus arrivals during one broker interval.

2.5 Implementation Plan

Phase 1: add Kconfig and runtime gating. Introduce CONFIG_RCU_NOCB_RAPL_SHAPER depending on X86, RCU_NOCB_CPU, and !PREEMPT_RT; add boot parameter rcu_nocb_rapl_shaper=0/1 with default off for initial rollout. Runtime gate on provider availability, STABLE domain state, SCHED_NORMAL policy on participating rcuo threads, and full containment within housekeeping CPUs. Phase 2: implement the cached-RAPL provider. Reuse existing x86 RAPL or powercap-visible energy unit and max-energy-range information to compute delta_energy_nj, avg_power_uw, freshness, and wrap_guard. Publish samples with seqcount. Ensure the provider worker is pinned to housekeeping CPUs and never called from nocb drain code. Phase 3: add domain mapper and broker. Maintain a per-domain descriptor keyed by package or die tuple, topology_seq, and member rcuo thread list. Recompute effective affinity every broker tick; on any instability, publish BASELINE immediately and require three stable periods before reenabling shaping. Phase 4: wire in lockless nocb statistics and policy-word consumption. Expose ready-count and oldest-ready timestamp snapshots for each nocb group. In the rcuo drain loop, read the policy once after wake and before draining, then apply the encoded extra sleep, local callback quota, runtime budget, and partial-drain rearm. Do not alter callback order, GP sequencing, wake-source generation, or expedited sync behavior. Phase 5: add memory-pressure override and observability. Sample low-watermark state and PGSCAN_DIRECT deltas in broker context only; add tracepoints for provider freshness, domain-state transitions, policy-word generation, and force-drain reasons. Treat manual affinity changes, cpuset moves, CPU hotplug, stale provider windows, unsupported virtualization, and scheduler-policy changes as explicit fail-closed triggers. Phase 6: tune default thresholds for x86 server packages. Start with broker period 2000 us, max deferral 2000 us, age force threshold 8000 us, domain backlog force threshold 4096, local backlog force threshold 1024, and the stated hysteresis. Keep all values sysctl-tunable only for experimentation, but patent the mechanism independent of the exact constants.

4. Risks & Mitigations

  • Risk: low-load or bursty packages produce noisy energy-per-ready-callback estimates. Mitigation: require fresh samples, maintain a 64-window EWMA baseline, enter CONSERVE only after three consecutive above-baseline windows, and require a minimum ready backlog before using energy-driven shaping.
  • Risk: domain oscillation under cpuset churn, affinity changes, or CPU hotplug. Mitigation: use a DISABLED -> PROBING -> STABLE state machine, immediate fail-closed fallback on any topology change, and require three consecutive identical effective-affinity observations before reenabling shaping.
  • Risk: provider interaction could deadlock with scheduler, thermal, or powercap paths. Mitigation: isolate RAPL collection in housekeeping delayed_work, allow only provider-local and existing RAPL locks, publish samples via seqcount, and never call provider code while holding nocb, rq, cpuset, or mm locks.
  • Risk: callback retention could grow under CONSERVE mode and hurt reclaim or latency. Mitigation: clamp extra deferral to 2000 us, force zero-deferral drain on memory pressure, force drain when oldest age reaches 8000 us or backlog thresholds are crossed, and cap local and domain backlog before catch-up becomes mandatory.
  • Risk: latency-sensitive housekeeping setups or real-time tuning could regress. Mitigation: compile-time exclusion on PREEMPT_RT, runtime disablement for any participating rcuo thread not running as SCHED_NORMAL on housekeeping CPUs, and no changes to rcuo priority or affinity.
  • Risk: unavailable or virtualized RAPL support makes the policy unreliable. Mitigation: provider flags unsupported or untrusted domains, broker publishes BASELINE immediately, and the nocb drain logic continues with existing Linux behavior without any telemetry dependency.
