2. Technical Invention Disclosure

2.1 Problem Statement

AMX-heavy convolution, GEMM, and matrix-like reduction kernels often run as a sequence of short tile-compute slices separated by barriers, scalar epilogues, memory stalls, and userspace scheduling gaps. Utilization history alone does not encode that a CPU or physical core very recently executed AMX and is therefore unusually sensitive to both idle-state exit latency and abrupt inter-burst frequency collapse. The kernel needs a signal that is stronger than permission, narrower than generic utilization, and mechanically safe under x86 eager fpstate isolation rules. The signal must not be derived from stale live registers, must not depend on hardware exceptions as routine optimization control flow, and must never call cpufreq or cpuidle policy code from rq-locked or interrupt-sensitive contexts.

2.2 Prior-Art Gap

The gap is a missing bridge between exact AMX-save evidence and bounded Linux power QoS actions. Existing xstate code answers correctness, isolation, and permission questions, but it does not expose a governor-facing, expiring, lockless contract for 'AMX was materially present here very recently.' Existing schedutil/uclamp patterns aggregate runnable demand, not saved architectural XTILE evidence. Existing freq_qos and PM QoS mechanisms provide the actuators, but not the evidence model, state machine, or withdrawal rules. Historical x86 power-control work such as LongRun-style performance selection and embedded threshold-driven PMIC logic also fails to teach the claimed bridge because those systems are not keyed to per-task AMX materialization in Linux fpstate images. The topological void is therefore the combination of matrix-workload AMX evidence with Linux power-management QoS, not either mechanism alone.

2.3 Proposed Invention

Introduce an AMX power broker composed of four coupled contracts. First, a per-task evidence latch distinguishes transient PROBE signals from CONFIRMED AMX use, where CONFIRMED requires seeing XTILE state in a materialized in-memory fpstate image. Second, scheduler-adjacent code publishes a packed atomic64 per-CPU recency tuple containing monotonic time, hotplug epoch, and evidence stage so readers never observe a torn timestamp-plus-flag pair. Third, an unbound deferred broker aggregates tuples at the actual Linux coupling domains of logical CPU, physical core, and cpufreq policy, then applies OFF/SINGLE/GROUP floors through freq_qos minimum requests and per-CPU DEV_PM_QOS_RESUME_LATENCY requests on cpu_device objects. Fourth, an optional system-level, one-shot, bounded s2idle guard window reads the same evidence but never pins an mm or task reference and can defer suspend only once and only for a root-configured maximum interval.

2.4 Architecture

Control flow:

saved XTILE image in task fpstate
  -> set task_struct.thread.amx_observed = 1 (store-release, once per task lifetime until exec/exit)
  -> scheduler switch-in/switch-out of observed task on CPU N
  -> atomic64_set_release(per_cpu(amx_tuple, N), packed_tuple)
  -> queue delayed_work on unbound housekeeping workqueue
  -> cpufreq policy scan with cpufreq_cpu_get()/put()
  -> OFF / SINGLE / GROUP policy decision
  -> freq_qos_update_request(policy_ctx.soft_req / policy_ctx.hard_req)
  -> dev_pm_qos_update_request(cpu_ctx.resume_latency_req on get_cpu_device(cpu))
  -> optional read of atomic amx_s2idle_guard_until_us during s2idle prepare

Primary kernel objects:
- task_struct.thread.amx_observed: stable per-task execution-history latch
- atomic64_t per_cpu(amx_tuple): packed recency publication record
- struct amx_pm_policy_ctx: one per cpufreq_policy, owns work item, hrtimer, freq_qos requests, epoch snapshot
- struct amx_pm_cpu_ctx: one per logical CPU, owns dev_pm_qos_request for cpu_device
- atomic64_t amx_s2idle_guard_until_us: optional global one-shot suspend guard deadline

2.5 Implementation Plan

Phase 1: add task-local amx_observed/amx_probe_seen fields, implement the CONFIRMED predicate in the fpstate materialization path, and clear the fields on fork/exec/exit. Phase 2: add atomic64 per-CPU tuple publication in scheduler switch-in/switch-out paths plus hotplug-epoch invalidation. Phase 3: add unbound amx_pm_policy_ctx workers, hrtimer-based expiry, cpufreq_cpu_get()/put() policy pinning, and OFF/SINGLE/GROUP freq_qos floors. Phase 4: add per-CPU cpu_device DEV_PM_QOS_RESUME_LATENCY requests with SMT propagation. Phase 5: add optional s2idle one-shot guard sysfs control, default disabled. Phase 6: add tracing, selftests, and admin documentation for thresholds and backoff behavior.

2.6 Validation Plan

Validate on AMX-capable x86 systems using tiled GEMM and convolution microbenchmarks, bursty inference loops, and mixed scalar-plus-AMX workloads. Measure p50/p95/p99 wake-to-first-tile latency, inter-burst effective frequency, throughput per watt, package energy, and regression on non-AMX or permission-only tasks. Include negative tests proving that ARCH_REQ_XCOMP_PERM without actual XTILE save never triggers QoS, hotplug tests proving request withdrawal on policy rebuild, NOHZ full tests proving timer-driven expiry without scheduler ticks, SMT tests proving sibling-only latency caps, thermal tests proving clamp/backoff precedence, and suspend tests proving single-shot <=2 ms s2idle deferral with automatic withdrawal.

  3. The method of claim 1, further comprising aggregating the tuples at cpufreq-policy scope into an OFF, SINGLE, or GROUP state machine in which a single recently CONFIRMED core causes a first minimum frequency floor and multiple recently CONFIRMED CPUs or cores cause a stronger minimum frequency floor, both floors being rate-limited, expiring, and withdrawable.
  4. The method of claim 3, wherein each policy update pins cpufreq policy lifetime with cpufreq_cpu_get()/cpufreq_cpu_put(), owns one or more freq_qos_request objects in a policy context, and cancels timers and deferred work on policy exit or hotplug-induced rebuild to avoid use-after-free.

The method of claim 1, further comprising attaching per-CPU dev_pm_qos requests of class DEV_PM_QOS_RESUME_LATENCY to cpu_device objects for CPUs whose CONFIRMED tuples are within a bounded recency window, and withdrawing those requests when the windows expire, thereby constraining idle-state exit latency only for selected CPUs rather than system-wide.
  6. The method of claim 5, wherein a CONFIRMED tuple for one SMT sibling causes temporary propagation of the resume-latency cap to the other sibling of the same physical core for a shorter SMT propagation window.
  7. A system-level suspend-control method comprising maintaining an atomic AMX guard deadline derived from recent CONFIRMED tuples and deferring s2idle at most once and for at most a configured bounded interval, without holding a per-mm reference and without permitting indefinite denial of suspend.

4. Risks & Mitigations

  • Risk: false positives from permission-only enablement. Mitigation: only CONFIRMED saved-XTILE evidence can set amx_observed or drive QoS; PROBE is non-authoritative.
  • Risk: torn or stale recency reads on SMP. Mitigation: atomic64 packed tuple with acquire/release semantics and hotplug epoch invalidation.
  • Risk: cpufreq policy use-after-free during hotplug or governor transitions. Mitigation: cpufreq_cpu_get()/put() on every broker pass plus synchronous timer/work cancellation on policy exit.
  • Risk: accidental global idle suppression. Mitigation: use only per-cpu cpu_device DEV_PM_QOS_RESUME_LATENCY requests and never PM_QOS_CPU_DMA_LATENCY.
  • Risk: suspend DoS. Mitigation: optional feature, root-only sysfs enablement, hard 2 ms cap, and one-shot consumption per suspend attempt.
  • Risk: fighting thermal or autonomous hardware controls. Mitigation: clamp to effective policy max, no direct MSR manipulation, and back off after repeated thermal saturation.
  • Risk: overhead in scheduler-adjacent paths. Mitigation: one local atomic64 write only on relevant switch edges and no policy work under rq lock.

Kernel Hardliner REVISE ★★★★★ YC:0

⚠ Fatal Flaw: Per-mm recency/evidence is a correctness trap: AMX execution is per-thread, while an mm can be shared across threads/processes; this can misattribute AMX activity and apply QoS to unrelated runnable contexts unless the design is made strictly per-task (or per-mm with explicit, justified semantics and teardown synchronization).

  • Define the *exact* “materialized fpstate” hook points and locking: you must only inspect XTILEDATA after the task’s fpstate is fully synchronized to memory (e.g., after __fpu_sync_state()-class paths) and never from rq-locked/atomic contexts; spell out required locks (fpregs_lock()/fpu_lock) and preemption rules to avoid races with XRSTOR/XSAVE update paths.
  • The packed 64-bit recency word needs a concrete update protocol: use atomic64_cmpxchg() loops (or seqcount) so epoch+class+time are updated atomically and readers get a coherent snapshot; specify memory ordering (smp_store_release / smp_load_acquire semantics) rather than assuming x86 atomicity if any code is generic or if readers run on different CPUs.
  • Per-mm aggregation must have a teardown-safe lifetime model: mm exit/exec/fork requires explicit reset points and RCU/refcounting so broker readers cannot dereference freed mm data; if you keep per-mm state, justify why mm-scoped QoS is correct despite thread-level AMX usage and implement robust mm_notifier / mmu_notifier or mmu_gather-style lifetime rules.
  • dev_pm_qos per-CPU cpu_device requests must be CPU-hotplug correct: create/remove requests in CPUHP online/offline callbacks, handle get_cpu_device(cpu)==NULL, and ensure the broker never touches a request for an offline CPU (or a cpu_device that has been unregistered), otherwise you risk UAF and PM QoS list corruption.
  • freq_qos request lifetime must be tied to cpufreq_policy rebuilds: cpufreq policies can be re-created on hotplug/governor changes; you need a reattach strategy (policy notifier or cpufreq transition/policy notifiers) and strict cpufreq_cpu_get()/put() usage in sleepable context, otherwise the broker can leave stale requests or hit UAF on policy free.
