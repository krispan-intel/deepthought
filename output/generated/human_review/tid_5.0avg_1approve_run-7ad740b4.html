2. Technical Invention Disclosure

2.1 Problem Statement

On x86 systems with Intel HWP and EPP support, Linux exposes static or manually adjusted power preference knobs through intel_pstate and cpufreq, while perf_event and eBPF can observe phase changes at high rate. What is missing is a kernel-safe, hotplug-safe, policy-lifetime-safe mechanism that converts those observations into package-coherent HWP min/max/EPP updates without performing forbidden work in NMI or hardirq context, without caching stale cpufreq policy pointers, without racing sysfs or firmware authority, and without adding a second MSR writer that can conflict with intel_pstate, platform profile, thermal control, or out-of-band writers.

2.2 Prior-Art Gap

The topological voids show the missing bridge: X86_FEATURE_HWP_EPP plus energy-efficiency evaluation suggests a direct kernel path from hardware energy preference controls to measurable efficiency regimes; eBPF flow analysis plus distributed optimal power flow suggests solving package power as a constrained multi-policy allocation problem rather than independent governor decisions; and legacy preference features such as LONGRUN-like semantics highlight that the missing element is not another knob, but a coordinated contract that maps package-domain headroom and per-phase demand into coherent per-policy envelopes. Existing governors, thermal frameworks, and powercap mechanisms do not define package and die membership invariants, generation counters, rate-limited deferred actuation, or out-of-band writer detection for HWP request fields.

2.3 Proposed Invention

The invention introduces a Deferred Intent Governor embedded in the kernel as a package and die aware coordination layer above perf evidence collection and inside the authority boundary of intel_pstate actuation. Per-CPU perf and eBPF sources populate fixed evidence banks; a package coordinator worker snapshots an RCU-published domain graph, gathers race-safe telemetry, computes a filtered headroom scalar, solves a constrained allocation across all policies sharing the same power domain, converts that allocation into per-policy HWP min/max/EPP envelopes, and commits the envelopes only through a new intel_pstate helper that runs on each target CPU, rechecks policy lifetime generations, merges with immutable admin and firmware limits, writes only managed HWP fields, and immediately audits readback to detect unknown writers. If topology, telemetry freshness, authority, or readback verification fails, the controller suppresses envelopes and restores baseline state rather than guessing.

2.6 Validation Plan

Validate correctness with kernel selftests and fault injection across four axes: context safety, topology churn, authority conflict, and efficiency gain. Context safety tests verify no sleeping or allocation in perf and eBPF paths and bounded work per sample. Topology tests repeatedly hotplug CPUs, rebuild cpufreq policies, change platform profile, and suspend or resume while asserting that stale batches never apply. Authority tests inject user EPP writes, min or max changes, and external MSR modifications, then confirm suppression, readback detection, and SAFE rollback. Efficiency tests on Xeon platforms under package caps compare baseline intel_pstate behavior versus package-coherent envelopes for OLTP, microservices, and inference, measuring throughput per watt, p99 latency, package power variance, and oscillation rate. Stability is accepted only if EPP transitions remain within the configured dwell and hysteresis budget and no domain sees partial-apply across sibling policies.

4. Risks & Mitigations

  • Risk: external writers such as the msr driver or firmware can race managed HWP fields. Mitigation: readback auditing, external_writer_detected state, one-shot SAFE baseline reapply, and observe-only fallback.
  • Risk: hotplug or cpufreq policy rebuild can invalidate cached state. Mitigation: store only cpu ids and generation tokens persistently, rebuild domain tables under RCU, and reacquire live policies at batch build and commit time.
  • Risk: package telemetry may be missing or stale. Mitigation: internal snapshot bridge with freshness deadlines, baseline-only mode when stale, and policy-local fallback when powercap is absent.
  • Risk: oscillation under bursty workloads. Mitigation: low-pass headroom filtering, minimum dwell, EPP and perf hysteresis, and switch-cost penalty in the allocator objective.
  • Risk: hybrid-core asymmetry can make one EPP mapping invalid across classes. Mitigation: separate class-specific clamp tables and envelope translation for P-core and E-core policies within the same domain.
  • Risk: user-visible knob fights. Mitigation: explicit precedence table, immediate suppression on user writes, baseline refresh, policy_gen bump, and delayed reentry into adaptive mode.
