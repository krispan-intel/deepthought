2. Technical Invention Disclosure

2.1 Problem Statement

On modern x86 servers, interrupt placement is a constrained routing problem across CPUs, vectors, remapping entries, and isolation domains, not a simple per-IRQ affinity hint. A single NIC, NVMe controller, or accelerator can expose tens to hundreds of MSI-X vectors whose optimal destinations depend on NUMA distance, shared LLC slices, SMT pairing, vector allocator pressure, housekeeping versus isolated CPUs, and whether different IRQs collide inside the same short burst window. Existing balancing strategies mostly observe counts over coarse intervals and then call affinity APIs heuristically, which misses phase relationships such as co-firing, lagged bursts, and anti-correlation. The kernel also has to preserve strict x86 routing correctness. A new vector must have a well-defined owner on a well-defined CPU, reservations must not survive CPU hotplug or allocator churn incorrectly, userspace or driver affinity changes must not be overwritten spuriously, managed-interrupt constraints must be obeyed, and no old vector may be freed while hardware can still legally target it. Reprogramming order differs for MSI/MSI-X, IO-APIC, and VT-d interrupt remapping, and any failure path must restore working hardware state instead of leaving an IRQ masked or half-migrated. The technical problem is therefore to extract materially better routing decisions from fine-grained temporal structure while implementing a precise authority model, transport-specific no-mixed-delivery cutover, and a liveness-safe rollback protocol inside Linux/x86.

2.2 Prior-Art Gap

User-space irqbalance and common in-kernel affinity heuristics optimize mostly from rate, topology locality, or driver hints; they do not define a bounded hardirq-safe burst lattice with sealed epochs, lagged cross-correlation, or a formulation that turns IRQ coincidence into matrix multiply or convolution-style operations for AMX execution. Existing Linux vector-management flows such as irq_matrix allocation, assign_irq_vector, clear_irq_vector, and irq move cleanup handle necessary mechanics for a specific move, but they do not expose a first-class vector-lease object with old-vector image, new-vector lease, generation snapshots, rollback state, and quarantine rules that can be planned, revalidated, and either committed or reverted atomically from the optimizer. Prior art also lacks an explicit policy-arbitration layer spanning managed interrupts, effective affinity, cpusets, isolcpus, housekeeping masks, and concurrent irq_set_affinity or driver-directed updates. Known approaches rarely specify where invalidation generations live, which state transitions cancel a migration plan, how allocator ABA is prevented, or how hardware state is restored after a mid-cutover fault. The gap is therefore a kernel-resident protocol that jointly provides: sealed-epoch burst telemetry, AMX-accelerated scoring, vector-lease ownership semantics tied to real kernel locks and generation counters, and transport-specific cutover orders that guarantee no mixed old and new delivery state.

2.3 Proposed Invention

The invention introduces four cooperating kernel components. 1. Telemetry lattice. Each participating CPU owns a double-buffered per-CPU array of 512 telemetry slots per bank, with two-way set-associative hashing by IRQ number. Each slot stores irq number, local epoch tag, last-touched TSC base, overflow count, and a 64-bin ring of saturating u8 counters at 2 microsecond bin width, yielding a 128 microsecond horizon per slot. This structure bounds memory, avoids O(IRQs x bins) allocation for thousands of vectors, and focuses detailed telemetry on hot IRQs; cold or evicted IRQs still contribute to a coarse rate counter so no interrupt disappears from the optimizer's view. In hardirq context, the update path is: skip if in_nmi(); read active bank via __this_cpu_read(); obtain delta from rdtsc_ordered() only if the system advertises constant_tsc and nonstop_tsc; derive bin index by shift and mask; probe the two hashed slots; on hit or reusable slot, zero the slot lazily if its epoch tag differs, then increment the selected u8 bin with saturation; on nested hardirq depth greater than one or hash overflow, increment a per-CPU overflow counter only. The fast path performs no allocation, no lock acquisition, and no remote access. 2. Epoch sealing and AMX scoring. A housekeeping kthread seals the lattice every 1 millisecond or when overflow pressure exceeds a threshold. Sealing is done by smp_call_function_many() over the participating CPU mask; the callback disables local interrupts briefly, flips that CPU to the alternate bank, updates the bank epoch, and publishes the acknowledgement with smp_store_release(). The kthread waits with smp_load_acquire() for all acknowledgements, after which the previous bank is immutable and can be read without torn writes. The sealed slots are compacted into a dense matrix B where rows are active IRQs and columns encode CPU and time-bin position. Lagged overlap is then computed as Ck = B x shift_k(B)^T for one or more small lags k, which is mathematically equivalent to a short-window convolution or cross-correlation and maps directly onto AMX tile dot-product operations on quantized telemetry. The scorer combines the burst-overlap matrices with a topology matrix, SMT and LLC penalty matrix, vector-pressure penalty, and policy mask constraints to produce a cost for placing each IRQ on each eligible CPU. The planner can then choose a batch of non-conflicting moves by greedy descent or min-cost matching. 3. Policy arbitration. Before any move is planned, each IRQ receives an allowed destination mask computed as online CPUs intersect effective domain affinity intersect housekeeping-allowed CPUs intersect cpuset and isolcpus policy. Managed interrupts are either excluded entirely or moved only within the managed effective mask when the descriptor is marked for optimizer participation. Any userspace write through irq_set_affinity, any managed-affinity recomputation, or any driver-driven affinity update increments affinity_gen on the irq_desc and places that IRQ into a hold-down window during which the optimizer only observes and does not override. This prevents fights with irqbalance, explicit administrator writes, or device-driver queue placement. 4. Vector-lease transaction and commit engine. For each selected move, the optimizer creates a vector_lease handle containing irq number, source CPU and vector, destination CPU and candidate vector, source and destination vector lease sequence numbers, saved last-known-good hardware image, transport type, retry budget, and generation snapshots: desc_gen, affinity_gen, topology_gen, allocator_gen for source and destination CPUs, and domain_gen. Authority is split cleanly: irq_desc lock owns IRQ policy and routing intent, while vector_lock plus irq_matrix own per-CPU vector allocation. Under both locks, the destination vector is reserved and marked LEASED with owner irq and a new per-vector lease_seq. Free, leased, active, and quarantined vector states each advance the sequence number so a recycled vector cannot satisfy stale handles, eliminating ABA reuse. CPU-hotplug notifiers increment topology_gen and cancel any lease targeting an offlining CPU; allocator mutations increment allocator_gen under vector_lock; successful affinity changes increment affinity_gen under irq_desc lock; transport-domain reconfiguration such as MSI domain rebinding or remapping-mode changes increments domain_gen. Commit proceeds by transport-specific state machines that all enforce the same invariant: the source remains masked until the new hardware image is fully programmed and any required remapping invalidation has completed, and the old vector is not freed until quarantine ends. For MSI or MSI-X: reserve destination vector under locks and snapshot the saved MSI message; drop locks; mask the source at the device or MSI domain; drain in-flight handlers with synchronize_irq(); reacquire locks and revalidate all generations plus both vector lease_seq values; program the new MSI address and data for destination APIC and vector; issue a write barrier and any required posted-write readback; mark the new vector active; unmask the source; place the old vector in QUARANTINED state; after a cleanup epoch confirms no stale delivery path remains, free the old vector under vector_lock. On failure after masking, the rollback path rewrites the saved MSI message, flushes the posted write, unmasks the old route, marks the lease rolled back, and returns to the last-known-good state. For IO-APIC redirection entries: reserve the destination vector; snapshot the old RTE and any remapping state; mask the IO-APIC entry and read back the masked RTE; drain with synchronize_irq(); revalidate; program destination APIC and vector in the still-masked RTE; if interrupt remapping is active, update the corresponding IRTE and issue queued invalidation before unmask; write the final unmasked RTE; mark new active; quarantine old vector; later retire it only after cleanup. Rollback restores the saved RTE and saved IRTE, performs the required invalidation flush, then unmasks the original route. For VT-d interrupt-remapping with MSI source: reserve the destination vector; snapshot the saved IRTE and MSI message; mask the source; drain; revalidate; program the new IRTE target and vector; issue interrupt-entry cache invalidation and wait for completion; if the transport also carries a device MSI message, write that message after the IRTE is live; unmask; quarantine old vector; retire only after the quarantine rule is met. Rollback writes back the saved IRTE and MSI message, invalidates again, and unmasks the source on the old route. The liveness rule is explicit: masking is a transient commit state only. Each lease has a deadline; if the transaction cannot reach NEW_ACTIVE before the deadline, rollback to the last-known-good image is mandatory. After three failed attempts within a moving window, the IRQ enters cooldown and remains on the last-known-good route until the topology or policy generation changes again.

2.5 Implementation Plan

1. Add a Kconfig-gated optimizer subsystem under x86 interrupt management with an explicit build-time option for in-kernel AMX use; the AMX path is compiled only when the kernel build provides XTILECFG and XTILEDATA save and restore support for kernel threads, otherwise only AVX2 and scalar scorers are built. 2. Implement the telemetry lattice as per-CPU double-buffered slot arrays plus overflow counters. Wire updates into the hardirq entry path after IRQ number resolution but before expensive handler work. Gate fine-grained binning on constant_tsc and nonstop_tsc, and skip NMI context entirely. Expose read-only tracepoints and counters for observability, but keep control in the dedicated optimizer subsystem rather than in debug interfaces. 3. Implement epoch sealing with bank-flip IPIs and compact sealed data into a dense working set. Add the AMX scorer that executes only from a housekeeping kthread pinned to housekeeping CPUs. The AMX critical section uses migrate_disable(), preempt_disable(), kernel_fpu_begin(), tile operations, kernel_fpu_end(), preempt_enable(), and migrate_enable(), with no sleeping, no allocation, and no lock acquisition inside the tile region. 4. Implement policy arbitration and generation tracking. Add desc_gen to irq_desc state changes that affect commit safety, affinity_gen to successful affinity updates, topology_gen to CPU-hotplug and housekeeping-mask rebuild events, allocator_gen to vector_lock-protected irq_matrix mutations, and domain_gen to MSI-domain or interrupt-remapping reconfiguration. Add a hold-down timer for user or driver affinity writes and ensure managed interrupts are either excluded or constrained to their effective managed mask. 5. Implement the vector-lease manager. Introduce per-vector lease_seq values and vector states FREE, LEASED, ACTIVE, and QUARANTINED under vector_lock so a reserved vector has an auditable owner and stale handles cannot succeed after reuse. The lease object should store the last-known-good hardware image, retry budget, deadline, and rollback state so failures restore the exact old route. 6. Implement transport-specific commit engines for MSI/MSI-X, IO-APIC, and VT-d remapping. Use a common sequence of reserve, snapshot, mask, drain, revalidate, program, flush or invalidate, unmask, quarantine, retire, with transport-specific code responsible for the correct order of hardware writes and flushes. Ensure rollback always restores the saved hardware image and unmasks the IRQ before returning. 7. Integrate cooldown and retry logic. If a lease aborts due to generation mismatch, allocator pressure, or hardware-programming failure, retry only within a bounded budget and then pin the IRQ to the last-known-good route until a relevant generation changes. This converts fail-closed suppression into fail-safe restoration with explicit liveness guarantees.

2.6 Validation Plan

1. Functional correctness: create kernel selftests and fault-injection hooks that drive the lease state machine through success, generation-mismatch abort, hotplug abort, and mid-cutover programming failure. Verify that every failed path ends with the last-known-good hardware image restored and the IRQ unmasked. 2. No-mixed-delivery proof: instrument per-IRQ route state, vector ownership, and hardware programming points with tracepoints. Under synthetic interrupt storms, confirm that delivery is observed either on the old route before cutover or on the new route after unmask, never on both simultaneously after the transaction claims completion. 3. Policy arbitration: run managed MSI-X devices, userspace affinity writes, irqbalance activity, cpuset updates, and isolcpus or housekeeping changes concurrently. Validate that affinity_gen or topology_gen mismatches abort pending plans and that the optimizer enters hold-down instead of fighting external policy owners. 4. CPU hotplug and allocator ABA: repeatedly offline and online candidate CPUs while forcing heavy vector churn. Verify that topology_gen increments cancel destination leases, allocator_gen and lease_seq reject stale reservations, and no vector is committed after reuse on a different IRQ. 5. Telemetry safety and overhead: benchmark hardirq entry cost with and without telemetry enabled, verify no allocation or lock contention in the fast path, and confirm that NMI and nested hardirq cases do not corrupt slot state. Measure memory use from 1 to 256 CPUs with 512 slots per bank to confirm bounded footprint. 6. AMX safety and fallback: validate that the AMX scorer executes only on the opt-in kernel build and only from housekeeping kthreads. Force scheduling pressure, CPU migration attempts, and preemption points around the scorer to verify the migrate_disable and preempt_disable contract. On kernels without AMX enablement, confirm identical planner decisions within tolerance using the AVX2 or scalar path. 7. Performance efficacy: drive bursty NIC, NVMe, and mixed workload benchmarks and compare against static affinity and irqbalance. Measure p99 and p999 interrupt latency, softirq wakeup latency, LLC miss rate, and vector allocator pressure. The expected win is lower burst overlap and lower tail latency without increasing interrupt loss or routing failures.

4. Risks & Mitigations

  • Risk: In-kernel AMX support is not available in all Linux builds. Mitigation: make AMX an opt-in acceleration path behind a dedicated Kconfig and runtime feature check, with identical planner semantics on AVX2 or scalar fallback.
  • Risk: Hardirq telemetry could add unsafe or excessive overhead. Mitigation: use only preallocated __this_cpu arrays, rdtsc_ordered() on constant_tsc and nonstop_tsc systems, no allocations or locks, and skip NMI or nested-hardirq detailed updates in favor of overflow counters.
  • Risk: Torn reads of telemetry during snapshot. Mitigation: seal banks with synchronous bank-flip IPIs and release-acquire acknowledgement so the housekeeping thread reads only quiesced banks.
  • Risk: Vector reservation races or ABA reuse under allocator churn. Mitigation: define vector ownership under vector_lock, maintain per-vector lease_seq counters, and revalidate both sequence numbers and allocator_gen before programming hardware.
  • Risk: CPU hotplug invalidates destination assumptions. Mitigation: increment topology_gen on hotplug state changes, cancel leases targeting an offlining CPU, and keep the last-known-good route active until a fresh plan is built.
  • Risk: Concurrent irq_set_affinity or driver-directed updates cause policy fights. Mitigation: increment affinity_gen on every successful external affinity change, abort stale plans on mismatch, and enforce a hold-down window before the optimizer may act again.
  • Risk: Hardware could be left masked after a mid-cutover failure. Mitigation: store the exact last-known-good MSI message, RTE, or IRTE image in the lease and require rollback to restore and unmask that image before the transaction returns.
  • Risk: Immediate old-vector free could permit mixed delivery or lost interrupts. Mitigation: quarantine the old vector after successful cutover and free it only after cleanup confirms that no stale delivery path remains.
  • Risk: Memory footprint grows with CPU count. Mitigation: bound the design with fixed slots per CPU and hash-based hot-slot tracking instead of allocating per-IRQ full histograms, while preserving coarse counters for cold IRQs.
  • Risk: Prior-art overlap with generic affinity balancing or existing irq move flows. Mitigation: narrow the inventive core to sealed-epoch lagged cross-correlation formulated as matrix operations plus vector-lease ownership, generation revalidation, and quarantined old-vector retirement.
