1. Executive Summary

One-Liner

A new x86 kernel-internal bridge computes, after APIC/x2APIC and interrupt-remapping state are finalized but before MSI vector allocation is committed, a bounded snapshot of hard feasibility limits, advisory recommended vector count, eligible CPU envelope, and a fixed locality-preference enum so the generic Linux PCI MSI/MSI-X allocator and an optional driver-local Service-Class APIC Hint Table can make stable APIC-aware decisions without overriding IRQD_AFFINITY_MANAGED or userspace policy.
Novelty: The inventive step is the timing-locked, outputs-only snapshot itself and its bounded semantics. Existing Linux mechanisms expose only fragments of the needed information and at the wrong layer or wrong time: irq_calc_af
Feasibility: The proposal is implementable with ordinary kernel engineering and no new userspace ABI. The bridge can be declared as a new kernel-internal helper in include/linux/pci-msi-arch.h

2. Technical Invention Disclosure

2.1 Problem Statement

Linux PCI drivers request MSI or MSI-X vectors through pci_alloc_irq_vectors_affinity(), but the generic allocator must currently decide vector count and initial affinity layout without a single architecture-produced view of finalized x86 routing facts. On x86, the practical envelope depends on whether the machine settled on xAPIC or x2APIC, whether interrupt remapping is enabled, how many local-APIC vectors are already reserved, which CPUs are online, which CPUs are excluded by housekeeping or isolcpus policy, and which CPUs remain visible through the driver's effective cpuset. Those facts are spread across different subsystems and become visible at different times. The result is that generic code can make an optimistic early choice, only for x86 MSI-domain logic or vector allocation to correct it later. That late correction causes avoidable retries, unstable queue-to-CPU placement across machines, reduced determinism during CPU hotplug or resume, and poor handling of drivers with multiple service classes such as admin queues, latency-sensitive completions, and bulk throughput queues. The missing capability is not another allocator or another driver hint; it is a disciplined, kernel-internal bridge that presents the generic allocator with a bounded post-routing snapshot before allocation is committed.

2.2 Prior-Art Gap

Current Linux mechanisms cover only pieces of the problem. irq_calc_affinity_vectors() can compute how many vectors should be devoted to generic affinity sets, but it does not know finalized x86 APIC mode, interrupt-remapping outcome, model-specific local-APIC reservations, or current vector slack. irq_create_affinity_masks() can synthesize masks from a generic affinity descriptor, but it does not receive a precomputed x86 eligibility envelope or a compact topology preference enum tied to APIC locality. MSI-domain alloc callbacks on x86 can reject or reshape a request, but they do so late, after the generic path has already chosen a count and often after affinity preparation has started; they also do not export a reusable summary to the generic layer or to drivers. The x86 vector allocator and irq_matrix know real reservation pressure, but only in the act of allocating vectors; exposing raw live pressure directly to policy would create feedback loops, so the invention instead samples it once as an explicitly advisory recommendation. Existing driver use of irq_set_affinity_hint() is ad hoc, unguided by a route_epoch-tagged APIC snapshot, and rarely tracks safe teardown under hot-unplug. Therefore the prior art lacks each of the claimed fields at the required timing point: vector_floor and vector_ceiling as a single pre-allocation bounded contract, recommended_vectors as a one-shot advisory output rather than an allocation input loop, eligible_mask as an x86-derived hard envelope that already reflects cpu_online, housekeeping, and effective cpuset limits, and a fixed topology-preference enum that compresses APIC locality into a reusable allocator-visible form before allocation commit.

2.3 Proposed Invention

The invention introduces a new kernel-internal API surface and a bounded snapshot object. A new helper, for example arch_pci_msi_build_snapshot(dev, min_vecs, max_vecs, flags, affd, managed_request, &snap), is declared in include/linux/pci-msi-arch.h, implemented on x86 in arch/x86/kernel/apic/, and called by drivers/pci/msi/api.c after caller flags are sanitized but before irq_calc_affinity_vectors() or irq_create_affinity_masks() are invoked. The helper returns a struct arch_msi_alloc_snapshot containing: route_epoch; route mode facts such as xAPIC versus x2APIC and interrupt-remapping enabled or disabled; vector_floor; vector_ceiling; recommended_vectors; eligible_mask; an enum topology_pref with EXACT_CORE, LLC_LOCAL, NUMA_LOCAL, or SPREAD; and flags such as SNAPSHOT_HARD_CEILING, SNAPSHOT_LEGACY_ONLY, or SNAPSHOT_MANAGED_ENVELOPE_ONLY. The semantics are precise. vector_floor is not a new user contract; it mirrors the lowest count the generic layer should bother attempting for the chosen caller-admissible mode, and in the ordinary case equals the caller's min_vecs. vector_ceiling is a must-hold upper bound only when objective routing facts require it, such as when the relevant route is legacy-only, when the selected MSI mode is infeasible on the platform, or when interrupt-remapping or APIC-mode constraints make a larger request impossible. vector_ceiling is never reduced by transient irq_matrix pressure. recommended_vectors is always advisory; it is computed once per allocation attempt from a one-shot sample of reserved-vector slack or pressure plus the number of eligible CPUs and service-set geometry, and it must satisfy vector_floor <= recommended_vectors <= vector_ceiling whenever the mode is feasible. If pressure changes concurrently, the recommendation may become stale, but that is acceptable because it is not a must-hold invariant and is not recomputed inside the same allocation loop. eligible_mask is a hard envelope built by intersecting cpu_online_mask with the IRQ-eligible housekeeping mask and with the effective cpuset CPUs that govern the allocating context or device policy. If the caller supplied an affinity descriptor with an explicit CPU subset, that subset is intersected as well. topology_pref is then mapped to existing topology helpers within that envelope: EXACT_CORE maps to topology_sibling_cpumask(anchor_cpu) or the singleton anchor CPU when SMT is absent; LLC_LOCAL maps to cpu_llc_shared_mask(anchor_cpu); NUMA_LOCAL maps to cpumask_of_node(cpu_to_node(anchor_cpu)); and SPREAD maps to the full eligible_mask. anchor_cpu is selected from the device-local eligible CPUs using a stable local-spread rule. Recalculation is permitted only at allocation time and on coalesced topology-change events; runtime refresh is rate-limited by a per-device delayed_work so CPU hotplug bursts do not create workqueue storms. The timing rule is also explicit. Routing is considered finalized for snapshot purposes after boot or resume has completed APIC mode selection and the interrupt-remapping subsystem has either enabled a working remapping domain or conclusively fallen back. CPU hotplug does not change APIC mode, but it does change eligible CPUs, so it increments route_epoch. The same epoch increments on kexec re-entry, suspend-resume APIC or IOMMU reinitialization, and any transition that enables or disables interrupt remapping. The x86 MSI-domain allocation path revalidates the route_epoch just before commit. If unchanged, the allocation proceeds normally. If changed, the generic layer rebuilds the snapshot once and retries once; beyond that it falls back to existing generic decrement-and-retry behavior so no unbounded retry loop is introduced. Managed-affinity semantics are enforceable. The generic PCI layer passes managed_request into the helper from the same branch that will later result in IRQD_AFFINITY_MANAGED descriptors. When managed_request is true, the snapshot exposes only the hard eligible envelope and possibly a hard vector ceiling. The generic layer may use that envelope to clip initial placement, but it may not use advisory per-vector hints and may not publish runtime hint updates. For unmanaged IRQs, topology_pref and per-vector masks remain advisory and are always clipped to eligible_mask. Driver consumption is defined without changing caller semantics. The driver still passes its original min_vecs, max_vecs, flags, and optional affinity descriptor to pci_alloc_irq_vectors_affinity(). The generic layer internally chooses first_try = clamp(recommended_vectors, min_vecs, min(max_vecs, vector_ceiling)) when the snapshot is feasible. If that first try fails, the allocator retries using the existing backoff order down to min_vecs without expanding beyond the caller's original max_vecs and without changing the caller-admissible mode set. The driver therefore receives a normal return value and does not need to learn x86 internals. In the optional embodiment, after allocation returns nvecs, the driver instantiates a Service-Class APIC Hint Table sized to nvecs, not to the requested maximum. The Service-Class APIC Hint Table embodiment stores one entry per allocated vector, for example { service_class, cpumask hint_mask, u32 route_epoch, u32 irq_generation }. The table is allocated in probe or resume after nvecs is known. For each vector, the driver derives a hint_mask from topology_pref and the service class, clips it to eligible_mask, and publishes it only for unmanaged IRQs via irq_update_affinity_hint() or irq_set_affinity_hint(). Generation tracking prevents use-after-free publication: before freeing a vector the driver first clears the published hint with NULL, then increments irq_generation, cancels delayed_work, and only then frees the IRQs and cpumasks. Any asynchronous refresh work compares both route_epoch and irq_generation before publishing a new hint, so a stale worker cannot republish a hint for a freed IRQ.

2.4 Architecture

Core data structure:

struct arch_msi_alloc_snapshot {
    u32 route_epoch;
    u16 vector_floor;
    u16 vector_ceiling;
    u16 recommended_vectors;   /* advisory only */
    u16 reserved_vector_slack; /* advisory sample input */
    enum arch_msi_topology_pref {
        EXACT_CORE,
        LLC_LOCAL,
        NUMA_LOCAL,
        SPREAD,
    } topology_pref;
    unsigned long flags;       /* HARD_CEILING, LEGACY_ONLY, MANAGED_ENVELOPE_ONLY */
    cpumask_var_t eligible_mask;
};

Allocation path:

Driver
  -> pci_alloc_irq_vectors_affinity(dev, min, max, flags, affd)
     -> sanitize caller flags and mode order
     -> arch_pci_msi_build_snapshot(..., managed_request, &snap)
          reads:
            - finalized xAPIC/x2APIC mode
            - interrupt-remapping enabled/disabled result
            - local-APIC reserved-vector slack sample
            - cpu_online_mask
            - housekeeping IRQ-eligible mask
            - effective cpuset CPUs
            - LLC/core/NUMA topology helpers
     -> first_try = clamp(snap.recommended_vectors, min, min(max, snap.vector_ceiling))
     -> build affinity inputs
          EXACT_CORE => topology_sibling_cpumask(anchor)
          LLC_LOCAL  => cpu_llc_shared_mask(anchor)
          NUMA_LOCAL => cpumask_of_node(cpu_to_node(anchor))
          SPREAD     => snap.eligible_mask
     -> irq_calc_affinity_vectors() / irq_create_affinity_masks()
     -> msi_domain_alloc_irqs_descs_locked()
          -> x86 revalidate snap.route_epoch
          -> if changed: rebuild snapshot once, retry once
          -> else commit allocation
     -> descriptors created
          -> if IRQD_AFFINITY_MANAGED: hard envelope only, no runtime hints
          -> else optional Service-Class APIC Hint Table may publish advisory hints

Runtime hint refresh path:

CPU hotplug / cpuset change / resume / remap transition
  -> route_epoch++
  -> mod_delayed_work(system_unbound_wq, refresh_work)
  -> refresh_work checks route_epoch and irq_generation
  -> publish new unmanaged hints only if both still match

Formal invariants:
1. eligible_mask is always a subset of cpu_online_mask.
2. eligible_mask is always a subset of housekeeping-eligible CPUs and effective cpuset CPUs.
3. vector_ceiling is hard only for objective routing facts, never for transient pressure.
4. recommended_vectors is advisory only and sampled once per attempt.
5. vector_floor <= recommended_vectors <= vector_ceiling whenever snapshot feasibility is true.
6. Managed IRQs consume only the hard envelope and never receive runtime affinity-hint publication.

2.5 Implementation Plan

1. Add a new kernel-internal bridge header and snapshot struct in include/linux/pci-msi-arch.h, plus a generic no-op stub for non-x86 architectures. 2. Implement arch_pci_msi_build_snapshot() on x86 under arch/x86/kernel/apic/ so it latches route mode, interrupt-remapping state, route_epoch, eligible_mask, topology_pref, and a one-shot advisory pressure sample. The implementation should maintain a route_epoch incremented on APIC-mode reinit, interrupt-remapping state transition, CPU online or offline changes, and coalesced cpuset or housekeeping updates. 3. Integrate the helper into drivers/pci/msi/api.c immediately after caller flag sanitization and before irq_calc_affinity_vectors() or irq_create_affinity_masks() are called. The generic path uses vector_ceiling as a hard clamp, recommended_vectors as the first-try count, and eligible_mask as the clipping envelope. It must not expand the caller's flag set or max_vecs. 4. Add x86-side revalidation at MSI-domain allocation commit. If route_epoch changed between snapshot build and commit, rebuild once and retry once; otherwise continue into the existing allocation and backoff logic. This preserves bounded behavior and avoids allocation-feedback oscillation. 5. Add optional driver-side Service-Class APIC Hint Table support in a pilot multiqueue PCI driver. The driver allocates the table after actual vector count is returned, stores per-vector hint_mask, route_epoch, and irq_generation, publishes hints only for unmanaged IRQs, and clears hints before free. A single delayed_work item handles hotplug or cpuset refresh and is canceled synchronously during teardown. 6. Add tracepoints or debug counters for snapshot builds, route_epoch mismatches, first-try versus committed vector count, and skipped hint updates for managed IRQs so the behavior is observable during tuning and patent enablement.

2.6 Validation Plan

1. Boot and allocation matrix: validate on xAPIC and x2APIC systems, with interrupt remapping enabled and disabled, and with caller mode sets that allow MSI-X only, MSI only, or legacy fallback. Confirm that hard ceilings appear only for objective routing constraints and that recommended_vectors never exceeds vector_ceiling. 2. Managed versus unmanaged semantics: allocate vectors with managed affinity and with unmanaged affinity, verify that eligible_mask clipping occurs in both cases, and verify that runtime affinity-hint publication is skipped whenever descriptors are managed. 3. Concurrency and feedback stability: run parallel allocations from multiple devices while sampling vector pressure, confirm that recommended_vectors is computed once per attempt, that no self-reinforcing oscillation occurs, and that the generic backoff path still converges. 4. Topology correctness: force CPU hotplug, SMT sibling removal, cpuset narrowing, and housekeeping changes, then confirm that EXACT_CORE, LLC_LOCAL, NUMA_LOCAL, and SPREAD masks recompute correctly and remain subsets of the current eligible envelope. 5. Transition handling: exercise suspend and resume, kexec, and interrupt-remapping reinitialization paths; confirm route_epoch increments, one-shot retry occurs only when epochs differ, and stale snapshots are not committed. 6. Driver hint safety: for the optional Service-Class APIC Hint Table, verify teardown ordering by repeatedly probing and removing the driver under CPU-hotplug load, ensuring hints are cleared before free, irq_generation blocks stale work items, and no hint is republished for freed IRQs. 7. Performance demonstration: measure first-allocation success rate, vector-count stability across reboots and BIOS APIC settings, queue-to-CPU locality, and latency tail improvement for representative multiqueue NIC, NVMe, or accelerator workloads.

3. Patent Claims

  1. 1.   2. 2. The method of claim 1, wherein the bounded allocation snapshot carries a route_epoch, and an x86 MSI-domain allocation path revalidates the route_epoch immediately before allocation commit and, upon detecting change, rebuilds the snapshot at most once before falling back to an existing bounded retry path.
  3. 3. The method of claim 1, wherein the eligible CPU mask is formed by intersecting cpu_online_mask with a housekeeping-eligible CPU set and an effective cpuset CPU set, and wherein managed interrupts consume only the eligible CPU mask and hard vector limits while advisory per-vector hints are suppressed.
  4. 4. A system for advisory interrupt-locality publication in a Linux PCI driver, comprising a Service-Class APIC Hint Table instantiated after actual vector allocation, each table entry storing a service-class identifier, a clipped CPU hint mask, a route_epoch, and an IRQ generation value, wherein the driver publishes affinity hints only for unmanaged IRQs and updates or clears those hints only after revalidating both the route_epoch and the IRQ generation value so that stale workers cannot publish hints for freed interrupts.
  5. 5. The system of claim 4, wherein CPU hotplug, cpuset updates, suspend-resume reinitialization, kexec re-entry, or interrupt-remapping transitions increment the route_epoch and trigger coalesced delayed-work recomputation of advisory hint masks without overriding IRQD_AFFINITY_MANAGED routing.

4. Risks & Mitigations

  • Transient vector-pressure data could create allocator nondeterminism; mitigation: use pressure only to derive recommended_vectors, sample it once per allocation attempt, and prohibit it from changing vector_ceiling or the caller-admissible mode set.
  • Topology and policy masks can become stale during CPU hotplug or cpuset changes; mitigation: tag snapshots with route_epoch, increment the epoch on relevant transitions, revalidate once before commit, and coalesce runtime refresh through one delayed_work item per device.
  • Managed IRQ routing could be accidentally overridden by advisory hints; mitigation: pass an explicit managed_request precursor into the snapshot builder, allow only the hard eligible envelope in managed mode, and block irq_update_affinity_hint() or irq_set_affinity_hint() publication once descriptors are managed.
  • Runtime refresh could race with teardown and publish hints for freed IRQs; mitigation: store irq_generation per vector, clear published hints before free, cancel delayed_work synchronously, and require both route_epoch and irq_generation to match before every publication.
  • Prior-art overlap with existing affinity helpers could weaken patentability; mitigation: emphasize the missing timing point, the bounded semantics of hard versus advisory fields, the fixed locality enum, and the generation-tracked service-class hint publication path rather than generic affinity creation alone.
