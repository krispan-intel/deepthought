2. Technical Invention Disclosure

2.1 Problem Statement

On x86 servers with CXL Type-3 memory exposed through FS-DAX, the filesystem allocator sees a flat directly addressable capacity even when the underlying CXL topology is not flat: HDM decoder windows can map to different target sets, interleave geometries, switch hop counts, and CPU affinity domains, and HMAT or SLIT proximity data can rank those spans differently. Existing Linux filesystems know how to preserve their own block geometry, stripe alignment, and locality heuristics, but they do not receive a dax_device-scoped, allocator-ready, kernel-resolved representation of CXL topology that is safe to use under concurrency. Userspace can read sysfs or ndctl, but it cannot safely drive per-allocation kernel decisions or close races with hotplug and decoder reprogramming. A practical solution must therefore let XFS and ext4 choose among CXL-backed FS-DAX ranges using only mount-time translated metadata, must define complete invalidation rules for topology and rank changes, must map raw CXL spans to real filesystem block numbers without partially usable fragments, and must forbid live CXL or ACPI traversal, GFP_KERNEL allocation, or global topology locks in DAX faults and constrained allocator paths.

2.2 Prior-Art Gap

The gap is specific and not filled by known mechanisms. Userspace ndctl or sysfs policy can observe CXL topology, but it cannot produce a race-free, filesystem-private, fixed-format range map that concurrent kernel allocators can trust, and it cannot guarantee that a mount remains correct after decoder or target changes. NUMA mempolicy and mbind affect CPU-side memory placement for mappings, not on-media FS-DAX extent allocation over CXL address ranges. XFS AG selection, ext4 mballoc, stripe hints, flex_bg, and other allocator-local heuristics understand filesystem geometry, but they do not know decoder membership, target lists, interleave width, switch-hop distance, or x86 initiator affinity for a dax_device span. Generic cache-plus-generation-counter patterns also do not close this gap because they do not define the auditable bump events that matter for x86 CXL.mem, do not include HMAT/SLIT rank drift, and do not provide a block-aligned translation from raw HDM spans to allocatable AG or block-group segments. The inventive separation is the dax_device-scoped snapshot contract and its deterministic translation into allocator-consumable color spans that filesystems can use without ever dereferencing live CXL objects.

2.3 Proposed Invention

The invention introduces a two-stage kernel-internal contract. Stage 1 is a dax_device raw snapshot op, for example get_fs_topology_snapshot(dax_dev, out, out_len), which copies by value a versioned fixed-size record owned by the caller. The record contains no pointers and no CXL object references. A representative raw layout is: header {version, total_len, topo_gen, reason_mask, nr_spans, dax_align_shift, flags}; and up to a bounded number of span records, such as 256, each span containing {dax_off_bytes, len_bytes, phys_start, best_nid, best_distance, latency_rank, bandwidth_rank, switch_hops, interleave_ways, target_count, target_hash, flags}. topo_gen is monotonically incremented for every event that can change future placement semantics: decoder add or remove; decoder commit, reset, reprogram, base or size change; interleave-way, granularity, or target-position change; region add, remove, enable, disable, or resize; target list add, remove, reorder, or detach; memdev detach or reattach; CXL port or switch hotplug that changes reachable paths or hop count; and x86 locality-rank changes caused by CPU or memory hotplug that alter HMAT, SLIT, or SRAT-derived initiator affinity. Rank updates participate in the same topo_gen namespace; there is no separate untracked rank generation. Stage 2 is a mount-time filesystem translator that converts the raw snapshot into filesystem-private state. Translation is deterministic and geometry-safe: each raw span is split at every dax_direct_access discontinuity, every XFS AG boundary or ext4 block-group boundary, and every mixed-target seam if the raw span contains more than one latency or bandwidth class. After splitting, each fragment is rounded inward to filesystem allocation geometry with start = ALIGN_UP(dax_off_bytes, sb->s_blocksize) and end = ALIGN_DOWN(dax_off_bytes + len_bytes, sb->s_blocksize). Any fragment smaller than max(sb->s_blocksize, 1 << dax_align_shift) is discarded rather than exposed as partially usable capacity. No translated fragment is allowed to cross a raw span boundary, a mixed-latency seam, or an AG or block-group boundary, so dax_direct_access expectations remain intact. The translator then assigns each fragment to one of at most four canonical colors using a deterministic algorithm. First, the mount selects a stable home NUMA node: either an explicit mount policy node or, if none is specified, the lowest-numbered online CPU node among the minimum-distance nodes for the majority of mounted capacity. Second, each fragment computes distance_bin = 0 if its best_distance from the home node equals the minimum distance observed for any fragment in the mounted dax_device, else 1. Third, each fragment computes quality_bin = 0 if its latency_rank is in the fastest rank band for the device and its bandwidth_rank is in the best rank band for the device, else 1. Mixed-target fragments are split before this step; if a mixed fragment is smaller than one allocatable block and cannot be represented separately, it is conservatively assigned to the worse quality_bin. The final color_id is (distance_bin << 1) | quality_bin, yielding exactly four stable classes: 0 local-fast, 1 local-degraded, 2 remote-fast, 3 remote-degraded. Ties inside a class are broken deterministically by lower best_distance, then lower latency_rank, then higher bandwidth_rank, then fewer switch_hops, then lower dax_off_bytes. The filesystem stores an FS-private snapshot such as {topo_gen, nr_colors <= 4, nr_color_spans, color_desc[], color_span[]} where each color_desc records canonical semantics and preferred_nid, and each color_span records {fsb_start, fsb_len, color_id, ag_or_bg_first, ag_or_bg_last, flags}. Filesystems never dereference CXL regions, decoders, ports, or ACPI tables after this translation. XFS and ext4 keep their existing master free-space metadata as the source of truth; color pools are advisory secondary indexes only. Therefore colored placement cannot pin space, cannot create artificial ENOSPC, and can be abandoned at any point without corrupting allocator invariants. Allocation checks topology freshness with a single scalar comparison between the mounted snapshot topo_gen and the atomic dax_device topo_gen. On mismatch, the filesystem sets a stale flag, queues an asynchronous rebuild worker, bypasses all colored indexes for future allocations, and immediately reverts to ordinary uncolored allocation using the existing allocator. Existing extents are never relocated. DAX faults do not perform topology checks, CXL traversal, ACPI parsing, memory allocation, or lock acquisition beyond the existing DAX and iomap path; they operate only on already allocated extents.

2.4 Architecture

The architecture has five cooperating kernel components. (1) A CXL snapshot builder in CXL core owns event coverage and monotonic generation: it listens to region, decoder, target, memdev, port, switch, and x86 proximity-domain change events; under normal CXL locking it builds a new raw by-value snapshot and increments dax_device->fs_topo_gen with an atomic monotonic counter before publication. (2) A dax_device export layer publishes the raw snapshot through an internal op and exposes only scalar freshness state to filesystems; its contract is fixed-size, versioned, pointer-free, caller-owned, and auditable. (3) A filesystem mount translator allocates a private snapshot with GFP_KERNEL only at mount, remount, or background refresh worker time, computes canonical colors, rounds and splits spans to filesystem block geometry, and precomputes AG or block-group membership. (4) An allocator overlay keeps colored placement as a side index over existing free-space state. For XFS, each AG gets an advisory ag_color_mask plus per-color candidate summaries built from existing free-space btrees; stripe-unit and extent-alignment rules are applied before a colored candidate is accepted. For ext4, each block group gets a bg_color_mask and mballoc class summaries layered over the existing buddy allocator and flex_bg ordering. (5) A refresh and stale-handling controller uses RCU pointer swap to publish a new filesystem-private snapshot and rebuilds side indexes asynchronously; until rebuild_complete is true, allocations use only the uncolored master path. Locking is one-way: CXL locks are taken only inside the snapshot builder, never while a filesystem allocation lock is held. Filesystem allocation reads only RCU-protected snapshot memory and scalar atomics, uses only preallocated arrays, masks, and stack variables, and never takes CXL or ACPI global locks. DAX faults are outside this mechanism entirely and continue to use the normal iomap or DAX extent mapping path with no topology dereference.

2.5 Implementation Plan

Implement the mechanism in six steps. First, add the internal dax_device raw snapshot op and the raw fixed-format structs with explicit bounds, such as a maximum of 256 raw spans and a compact total size below 16 KB, plus a reason_mask that encodes the bump cause for debugging and auditability. Second, add CXL-core hooks that bump topo_gen on every covered event: region create, delete, enable, disable, resize, and resource reassignment; decoder add, remove, commit, reset, reprogram, base or size change, interleave-way change, granularity change, and target-position change; target list add, remove, reorder, attach, detach, and memdev offline or online; switch or port hotplug that changes the reachable path; and HMAT, SLIT, or SRAT affinity recomputation triggered by CPU or memory hotplug. Third, add a common FS-DAX helper that translates raw spans into filesystem-private color spans by splitting on AG or block-group boundaries, dax_direct_access discontinuities, and mixed-target seams, rounding inward to block geometry, and assigning one of four colors using the deterministic algorithm and stable tie-breakers. Fourth, integrate advisory side indexes into XFS and ext4. In XFS, the existing AG rotor and alignment logic run first, then the allocator tries colored candidates from the preferred color and one fallback color only, scanning at most four AGs total before abandoning colored placement and returning to ordinary AG selection. In ext4, mballoc keeps its normal order, but when coloring is enabled it first scans groups whose bg_color_mask matches the preferred color, then a secondary color, and after at most eight group probes or one failed buddy search it falls back to uncolored allocation. In both filesystems, master free-space ownership never moves into the color layer, so freeing and coalescing update the normal allocator first and the color summaries only opportunistically. Fifth, add stale semantics: on any allocation path entry, if mounted_topo_gen != READ_ONCE(dax_dev->fs_topo_gen), set SB_CXL_TOPO_STALE with cmpxchg, queue a worker, and bypass all colored indexes for new allocations until the worker installs a new snapshot and sets rebuild_complete. Frees that occur while stale update only the master allocator and may lazily repopulate color summaries after rebuild. Sixth, enforce fast-path constraints: color selection uses only existing inode policy, a cached home_nid, RCU snapshot reads, precomputed masks, and bounded loops; it may not call into CXL core, ACPI code, sysfs, ndctl, or any helper that can sleep, and it may not allocate memory with GFP_KERNEL in reclaim-sensitive paths. If any required advisory structure is absent, the allocator immediately takes the uncolored path.

2.6 Validation Plan

Validate the invention at three levels. For correctness, use cxl_test or QEMU-based CXL topologies to exercise decoder reprogramming, target add or remove, region resize, interleave changes, and simulated port or switch hotplug while a mounted XFS or ext4 FS-DAX filesystem performs concurrent allocation, free, mmap, and fault traffic; assert that topo_gen bumps for every covered event, that stale mode is entered exactly once per generation change, that existing extents remain accessible, and that no DAX fault path calls the snapshot op or takes CXL locks. For geometry safety, generate raw spans intentionally misaligned to filesystem block size, AG boundaries, and block-group boundaries; verify that inward rounding drops unusable tails, that no exported color span crosses a discontinuity, and that dax_direct_access can resolve every block in every published span. For allocator behavior, compare colored versus uncolored runs across multi-socket HMAT or SLIT configurations and mixed-latency targets, measure the bounded fallback path under stale snapshots and memory pressure, and confirm that fragmentation and ENOSPC behavior do not regress because master free-space metadata remains authoritative. For determinism, reboot or recreate identical topologies repeatedly and verify that color assignment, span ordering, and tie-break decisions are stable across runs and kernel builds. For constrained-context safety, run reclaim-stressed workloads and fault injection to confirm that colored placement never performs GFP_KERNEL allocation, ACPI table parsing, or CXL topology traversal in allocator fast paths.

4. Risks & Mitigations

  • Risk: an incomplete invalidation list would silently miscolor future allocations. Mitigation: centralize all CXL and x86 proximity bump events in one dax_device topo_gen owner, export a reason_mask for auditability, and add selftests that assert topo_gen changes for every enumerated event class.
  • Risk: hard color partitioning could increase fragmentation or trigger false ENOSPC. Mitigation: keep color pools as advisory secondary indexes only, leave master free-space metadata authoritative, and enforce bounded retries before immediate fallback to the ordinary uncolored allocator.
  • Risk: raw CXL spans may not align to filesystem block geometry or dax_direct_access boundaries. Mitigation: split on every discontinuity and AG or block-group boundary, round inward only, and discard sub-block tails rather than exposing partially allocatable extents.
  • Risk: HMAT may be absent or proximity ranks may change under hotplug. Mitigation: derive rank from HMAT when present, otherwise fall back to SLIT and SRAT distance plus hop count, and fold all rank recomputation into the same topo_gen invalidation domain.
  • Risk: allocator fast paths might accidentally sleep or take forbidden locks. Mitigation: require all translation and rebuild work to occur in mount or workqueue context, store only precomputed scalar metadata in fast paths, and make the colored path immediately abort to uncolored allocation if any advisory structure is unavailable.
  • Risk: deterministic four-color quantization could become too lossy on very complex topologies. Mitigation: preserve correctness by splitting mixed spans before quantization, order candidates inside each color by richer rank vectors, and define the color layer as a bounded policy class over precise range descriptors rather than as the only representation of topology.
  • Risk: snapshot rebuild could lag under memory pressure or heavy reconfiguration. Mitigation: stale mode is performance-degrading but safe, because new allocations revert to the master uncolored allocator until a new snapshot is published.
  • Risk: prior-art arguments may claim this is just caching topology. Mitigation: emphasize the specific combination of pointer-free dax_device export, complete x86 CXL invalidation semantics, deterministic four-class quantization, block-aligned AG or block-group translation, and advisory allocator integration that avoids live CXL dereference in filesystem hot paths.
