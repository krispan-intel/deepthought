2. Technical Invention Disclosure

2.1 Problem Statement

Linux block I/O has two separate scalar hot spots that become visible on plug-heavy, high-queue-depth NVMe workloads. First, blk-mq merge search spends locked CPU time evaluating many bio and request pairs that are permanently incompatible for simple structural reasons, yet the current path must still walk those pairs and perform branch-heavy scalar checks inline. Second, NVMe completion handling validates and retires large CQE bursts in hard-IRQ, threaded-IRQ, or poll context, where latency constraints discourage any rich batching structure and where a naive deferral scheme would immediately create races with request lifetime, tag reuse, reset, timeout, and queue teardown. Existing kernels therefore lack a safe way to move only the easy-negative or already-validated portion of the work into an off-lock process-context batch engine while preserving the exact observable semantics of first-match merge order, CQ head movement, request completion ownership, and reset behavior.

2.2 Prior-Art Gap

Current Linux merge helpers such as blk_mq_sched_try_merge(), blk_mq_attempt_bio_merge(), bio_attempt_back_merge(), and bio_attempt_front_merge() execute against live request state and do not define a bounded snapshot of merge-relevant invariants that can be evaluated off-lock and then revalidated before commit. Likewise, current NVMe completion paths such as nvme_process_cq(), nvme_poll_cq(), and nvme_handle_cqe() validate a live CQE and complete the request in the same owner domain, but they do not create a persistent host-owned CQE image with explicit release/acquire publication, do not arbitrate ownership across hard-IRQ, threaded-IRQ, and poll producers with a claim token, do not attach a queue epoch plus per-request completion cookie to a deferred snapshot, and do not define a normative RUNNING, QUIESCING, FLUSHING, and STOPPED state machine for controller reset, queue freeze, blk-mq quiesce, tagset shutdown, or PCI error recovery. Prior art around generic batching, barriers, or polling does not disclose this specific two-domain snapshot contract: a new ring data structure, a producer and consumer publication protocol, a direct-path overflow rule, and a revalidation rule set that confines acceleration to permanent negatives and already validated CQEs while leaving all accepting or side-effecting decisions to the baseline scalar code.

2.3 Proposed Invention

Introduce a per-hctx submission bridge and a per-NVMe-CQ completion bridge that share the same design rule: snapshot only information whose correctness window is explicitly closed, publish it with a formal ownership and memory-ordering contract, and require a final live scalar path to make any state-changing decision. Submission plane: while holding hctx->lock, the producer walks the same live request order used by the baseline merge probe and snapshots at most M candidate tuples of the form {bio pointer, request pointer, bio ref, request ref, request merge_cookie, queue epoch, hctx identity, op class, data direction, integrity-present bit, protection-information or crypto profile identifier or NONE, zoned-operation class, discard or write-zeroes or zone-append class, and any other merge attributes that are immutable or monotonic-non-healing during the current commit attempt}. The producer is forbidden to encode physical adjacency, current nr_segments, max-sector fit, scheduler position, plug ordering, zone write-lock progress, or any field that could change from not-mergeable to mergeable before commit. Off-lock, a process-context worker packs the immutable tuple fields into byte matrices and computes only permanent-negative bits in reject[i][j], using either a scalar or u64-lane implementation or, when enabled, AMX INT8 tile dot-product over 0 and 1 mismatch encodings. On re-entry, the commit code reacquires hctx->lock, walks the live request list in the original baseline order, and consults a reject bit only after revalidating request pointer identity, merge_cookie, queue epoch, and live membership. If any revalidation fails, the matrix entry is ignored and the baseline scalar merge helpers run as if no acceleration existed. A reject bit can only skip a pair that cannot become legal later in the same commit loop; every non-rejected pair still executes the existing scalar merge checks unchanged, so first-match and side-effect ordering are preserved. Completion plane: add a per-CQ ring of snapshot entries with a concrete layout such as {struct request *rq; struct nvme_completion cqe_copy; u32 slot_seq; u32 queue_epoch; u32 completion_cookie; u16 tag; u8 phase; u8 owner_domain;}. Add an owner token for the CQ, a per-CQ state enum RUNNING, QUIESCING, FLUSHING, STOPPED, and a per-tag or per-request cookie stored in struct nvme_request or an equivalent sidecar indexed by hardware tag. In nvme_process_cq(), threaded IRQ handling, or nvme_poll_cq(), a producer first tries to claim the owner token. If it loses, it does not publish and immediately executes the baseline direct completion path, which avoids multi-producer ring corruption. If it wins and the state is RUNNING, it validates each candidate CQE exactly as the baseline driver does: it checks the phase bit, performs dma_rmb() after phase observation on coherent DMA memory or dma_sync_single_for_cpu() plus dma_rmb() on non-coherent memory, validates tag bounds and status encoding, and rejects malformed CQEs back to the baseline error path instead of publishing them. Only after that validation does the producer map the tag to the live request, perform refcount_inc_not_zero() on the request, copy the full CQE and metadata into the ring slot, and publish the new producer index with smp_store_release(). Each CQE is claimed exactly once as either SNAPSHOT or DIRECT; dual claim is forbidden by the owner token and by the rule that only the publisher may advance prod. The producer may then advance the CQ head and ring the MMIO doorbell, using wmb() or io_wmb() before writel() and mmiowb() where posted-write serialization requires it. The novelty point here is decoupling CQ head advancement from request completion by substituting a validated host-owned snapshot; consumer processing is intentionally not on the critical path to doorbell advancement, but snapshot stores are on that path. A process-context consumer reads producer progress with smp_load_acquire(), batches snapshots, may optionally use AMX INT8 only in that process context to classify success, error, or other already validated categories, and then invokes the baseline scalar completion helpers. Before any deferred action, the consumer checks that rq, completion_cookie, and queue_epoch still match; timeout, abort, reset, or requeue paths invalidate the cookie or bump the epoch before tag reuse, so a stale deferred snapshot cannot complete a recycled request. Ring-full behavior is explicitly non-blocking: in hard-IRQ, threaded-IRQ, or poll context, if the ring has no free slot, the producer increments an overflow counter and immediately completes the CQE through the baseline direct path, bounded by the same IRQ or poll budget already enforced by the driver, without sleeping and without taking any new lock that could invert existing ordering. Reset and teardown are closed under concurrency by the state machine: RUNNING accepts snapshots; QUIESCING stops new publications and forces direct-path completion; FLUSHING drains the consumer-visible ring and invalidates cookies for any request being timed out, aborted, or requeued; STOPPED releases ownership and disables snapshotting until reinitialization zeroes the ring, bumps queue_epoch, and returns the queue to RUNNING.

Novelty boundary versus baseline
--------------------------------
1. Baseline nvme_process_cq() and nvme_poll_cq() directly retire CQEs in the live producer context; the invention inserts a validated per-CQ host snapshot ring with explicit owner arbitration and publish and consume protocol.
2. Baseline blk-mq merge code computes all merge legality inline under lock; the invention adds a bounded invariant-only negative filter that is revalidated before commit and can never turn a non-merge into a merge.
3. Baseline code has no need for deferred-completion cookies because it does not defer ownership past CQ processing; the invention adds cookies and queue epochs exactly to close that new deferred aliasing window.

Required happens-before edges
-----------------------------
A. Device DMA writes CQE bytes and status for slot n and makes the new phase value visible.
B. Producer observes the expected phase and executes dma_rmb() (plus dma_sync_single_for_cpu() first on non-coherent platforms) before reading the remaining CQE bytes.
C. Producer stores the snapshot fields into the ring slot.
D. Producer publishes prod_next with smp_store_release().
E. Consumer observes prod_next with smp_load_acquire() and only then processes the snapshot.
F. Producer performs wmb() or io_wmb() before the MMIO doorbell write so the host snapshot is globally visible before the device can reuse the CQ slot.
There is intentionally no required edge from consumer processing to doorbell update; that absence is the decoupling that removes request completion from the CQ-slot-reuse critical path while preserving correctness through the host-owned snapshot.

2.5 Implementation Plan

1. Submission-plane data model. Add a small per-request merge sidecar reachable from blk_mq_rq_to_pdu(rq) or an equivalent bridge-private allocation containing merge_cookie, originating hctx identifier, and any immutable merge-profile fields not already cheap to read live. Increment merge_cookie on request initialization and on any requeue or remap that can change request identity within the bridge. 2. Submission snapshot path. Wrap the existing merge probe entry so that, under hctx->lock, it snapshots at most M requests in the exact baseline traversal order, takes bio and request refs, records queue_epoch and merge_cookie, and immediately falls back to the pure scalar path if ref acquisition fails, the bound is exceeded, or the queue state is not RUNNING. 3. Reject-matrix engines. Implement a scalar or u64-lane permanent-negative classifier first; then add an optional AMX INT8 backend that runs only from a dedicated kthread or process-context work item. The AMX backend must check X86_FEATURE_AMX_INT8, XFEATURE_XTILEDATA enablement, kernel permission to use tile state, and invoke kernel_fpu_begin() and kernel_fpu_end() or the architecture-approved equivalent around tile configuration and execution. No hard-IRQ, softirq, tasklet, or poll fast path may enter AMX code. 4. Commit revalidation. On return to the live merge path, reacquire hctx->lock and require pointer identity, merge_cookie equality, queue_epoch equality, and list membership before consuming any reject bit. If any check fails, discard the accelerated result for that pair and continue through the baseline merge helper. 5. Completion-plane ring. Add a per-CQ SPSC-style ring sized to a power of two with fields rq, cqe_copy, slot_seq, queue_epoch, completion_cookie, tag, and phase. Add producer prod and consumer cons indices, owner_token, and state. The only publisher is the current owner-token holder; all other contexts must use direct completion and must not touch the ring. 6. Completion producer integration. Hook the validated snapshot step into nvme_process_cq(), the threaded IRQ variant, and nvme_poll_cq(). Enforce an existing driver budget or a bridge-specific max_snapshot_per_pass so producer work remains bounded. On malformed CQE, missing request, or refcount_inc_not_zero() failure, route to the baseline error or direct-completion path without publishing. 7. Cookie protocol. Store completion_cookie in struct nvme_request or a per-tag side array. Initialize it on request issue, copy it into the ring on snapshot, and invalidate or increment it before timeout completion, abort completion, reset-driven requeue, and final tag release. The consumer must compare both rq and completion_cookie before calling the baseline completion helper. 8. State machine and teardown. On reset, queue freeze, blk-mq quiesce, controller disable, or PCI error recovery, transition state with smp_store_release(RUNNING -> QUIESCING), synchronize against any current owner producer, stop new publications, drain the consumer in FLUSHING, invalidate cookies and bump queue_epoch, then move to STOPPED. Reinitialization zeroes prod and cons, clears owner_token, allocates a fresh queue_epoch, and returns to RUNNING. 9. Instrumentation and rollout. Expose only read-only counters such as snapshots_published, snapshots_revalidated, direct_fallback_overflow, cookie_mismatch_drops, malformed_cqe_drops, and amx_batches_executed. Gate the entire bridge behind a per-queue feature flag so deployment can begin with AMX disabled and the scalar classifier only.

2.6 Validation Plan

1. Semantic equivalence. Run paired A/B testing with the bridge disabled and enabled, asserting identical merge accept and reject outcomes, identical final request completion status, and identical externally visible ordering of first successful merge candidates and CQE retirement. 2. Concurrency coverage. Exercise hard-IRQ, threaded-IRQ, and nvme_poll_cq() configurations separately and under mixed controller fleets, verifying that only one owner publishes per CQ and that all losing contexts take the direct path without duplicate completions. 3. Reset and timeout races. Stress controller reset, queue freeze and unfreeze, blk-mq quiesce, timeout completion, abort, tag reuse, and PCI error recovery while injecting delayed consumer wakeups, confirming that queue_epoch and completion_cookie block stale deferred completions. 4. Ring-full behavior. Force ring saturation and confirm that direct-path fallback remains non-sleeping, respects the existing poll or IRQ budget, and does not deadlock or invert lock ordering. 5. Malformed CQE handling. Fault-inject bad phase bits, out-of-range tags, torn CQEs on non-coherent test platforms, and status-code anomalies; verify that invalid CQEs never enter the ring and instead follow the baseline error path. 6. Memory-ordering validation. Review the implementation against Linux memory-barriers and DMA-API guidance, then run lockdep, KCSAN, and architecture-specific stress on x86 and at least one non-coherent DMA platform to validate the coherent and non-coherent branches. 7. AMX safety. Verify that AMX code executes only in the designated process-context worker, that kernel_fpu_begin() and kernel_fpu_end() coverage is complete, that preemption and signal interactions do not leak tile state, and that feature gating cleanly falls back to the scalar classifier when AMX or XTILEDATA is unavailable.

4. Risks & Mitigations

  • Risk: a deferred CQ snapshot could alias a recycled tag or a request that has been timed out or requeued; mitigation: store rq plus completion_cookie plus queue_epoch in every ring entry, invalidate cookies on timeout, abort, reset, and requeue, and require the consumer to match all three before completion.
  • Risk: an invalid or malformed CQE could be published into the ring and later miscompleted; mitigation: validate phase, tag bounds, and status exactly as the baseline driver does before publication, and route malformed CQEs to the baseline error path instead of publishing them.
  • Risk: multi-context producers could corrupt the ring or double-complete a request; mitigation: use an explicit owner token per CQ, allow only the owner to publish, and force all non-owners to use the baseline direct path.
  • Risk: ring-full conditions could create an unbounded interrupt storm or a sleeping path in IRQ context; mitigation: define a formal non-blocking overflow rule that immediately falls back to baseline direct completion under the same existing IRQ or poll budget and without taking any new sleeping lock.
  • Risk: merge acceleration could incorrectly suppress a pair that later becomes mergeable; mitigation: limit the reject matrix to permanent-negative predicates only, forbid adjacency and segment-count style predicates, and always revalidate identity before consulting a reject bit.
  • Risk: AMX use in kernel code could leak tile state or violate atomic-context rules; mitigation: confine AMX to a dedicated process-context worker under kernel_fpu_begin() and kernel_fpu_end() or equivalent architecture wrappers, with feature gating and a complete scalar fallback.
  • Risk: reset, queue freeze, or PCI recovery could race with in-flight snapshots; mitigation: implement the explicit RUNNING, QUIESCING, FLUSHING, and STOPPED state machine, synchronize the producer owner before FLUSHING, drain the ring, invalidate cookies, and bump queue_epoch before returning to RUNNING.
