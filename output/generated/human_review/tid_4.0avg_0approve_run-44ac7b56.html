2. Technical Invention Disclosure

2.1 Problem Statement

Linux sched/fair makes many wakeup placements between CPUs that are already legal and already close according to existing load, capacity, and topology signals. In those near-tie cases, the current scheduler largely lacks a safe way to express that a task is in a coarse phase such as compute burst, cache-miss-heavy, branchy control, or power-cap-sensitive memory streaming, and it also lacks a bounded package-or-die pressure signal derived from actual x86 RAPL windows. Existing signals such as utilization, util_est, uclamp, thermal pressure, and energy_model/EAS each solve a different problem: runnable demand estimation, policy clamps, thermal capacity reduction, or energy-cost search. None of them provide a scheduler-safe bridge from per-task PMU phase signatures plus package-scoped energy windows into a hard-bounded tie-break that respects EEVDF fairness intent and never overrides hard placement rules. The resulting opportunity is largest on multi-die and hybrid x86 systems, where two CPUs can look nearly equal to the wakeup path while one belongs to a pressure-limited package or a topology class that is a systematically worse match for the task's current phase.

2.2 Prior-Art Gap

The gap versus known mechanisms is specific and narrow. Thermal pressure reduces effective CPU capacity after thermal throttling or similar capacity loss; it is not a learned phase hint and is not derived from a seqcount-latched RAPL contract. EAS/energy_model uses platform energy-cost tables during CPU selection; it is not a per-task PMI/NMI phase signature pipeline and does not claim a package-or-die cap-pressure scalar with validity, staleness, confidence, and generation semantics for lockless sched/fair readers. Prior RAPL-based scheduling papers generally optimize broader placement or throttling policy and do not specify an in-kernel single-writer snapshot publication contract, an RCU generation-safe domain map, or a near-tie-only bounded wakeup bias whose maximum influence is formally smaller than the tie gate. Intel Thread Director-class hardware classification also does not claim this Linux kernel software contract, the task cookie lifetime model, or the requirement that the classifier's output be consumed only as a symmetric bounded tie-break after cpuset, affinity, NUMA, asym_packing, SMT, uclamp, thermal pressure, and EAS have already had priority. The data-mining aspect is likewise narrower than generic ML: offline mining produces a finite threshold codebook that executes as fixed-point comparisons in kernel and can be audited, versioned, and bounded.

2.3 Proposed Invention

The invention is a scheduler-safe bridge with four contracts. 1. PMU signature contract. A perf event group samples per task on an instructions-based overflow period of 131072 retired instructions. The group uses architectural or generic x86 core events with stable cross-Xeon semantics: instructions, cpu-cycles, ref-cycles, cache-misses, and branch-misses. In PMI/NMI, the handler emits a fixed-width 64-bit signature word: [63:60] sig_ver, [59:56] caps, [55:48] sample_log2, [47:36] ipc_q12, [35:24] mpki_q12, [23:12] br_mpki_q12, [11:0] freq_q12. Normalization is fixed-point: ipc = instructions/max(cycles,1), mpki = 1000*cache_misses/max(instructions,1), br_mpki = 1000*branch_misses/max(instructions,1), freq = cycles/max(ref_cycles,1). Each field is clipped to a documented range and quantized to Q0.12. If an optional event is unsupported, the corresponding caps bit is cleared and the codebook falls back to a compatible lower-dimensional table. No model-specific uncore or SKU-fragile events are required in-kernel. 2. NMI-safe logging contract. The PMI/NMI path writes a sample to a per-CPU ring using a 64-bit reservation ticket and per-slot sequence numbers so nested interrupts on the same CPU cannot corrupt ordering. The record contains {ticket, task_cookie, exec_epoch, cpu, local_clock_ns, sig64}. The handler never takes task references, never allocates, and only resolves the current task to a stable non-reused cookie. Cookie-to-context resolution happens later under RCU in process context. 3. RAPL domain snapshot contract. A single writer thread per RAPL domain samples package or die energy every 2 ms by default using ktime_get_mono_fast_ns() as the timebase. For each sample, let dt_ns = now_ns - prev_ns. If dt_ns < 500000 ns or dt_ns > 20000000 ns, the sample is marked invalid and does not advance the filtered pressure. Let raw bits be B and modulus M = 2^B. Let dE_uj = ((raw_now - raw_prev) mod M) * energy_unit_uj. Let p_inst_uw = dE_uj * 1000000000 / dt_ns. Let p_ref_uw = pl1_uw if pl1_uw > 0 else pl2_uw if pl2_uw > 0 else 0. If p_ref_uw == 0 or the domain is unavailable, then confidence = 0 and cap_pressure = 0 by definition so the scheduler bias becomes zero. Otherwise define cap_inst = clamp((p_inst_uw/p_ref_uw - 0.85)/0.15, 0, 1). This yields a scalar in [0,1] that is explicitly not utilization: it measures proximity to the domain power cap, not runnable demand. The published filtered pressure is an EMA: cap_pressure_t = 0.75*cap_pressure_{t-1} + 0.25*cap_inst, stored as Q16 in [0,65535]. If PL2 is present and p_inst_uw exceeds PL2 by more than 5 percent, confidence is reduced because the sample is likely bursty or policy-shifted. Every published snapshot includes {seqcount, topo_generation, timestamp_ns, dt_ns, p_inst_uw, pl1_uw, pl2_uw, cap_pressure_q16, confidence_q16, flags}. Readers retry on seqcount until they observe a self-consistent tuple. 4. Scheduler integration contract. Active consumption happens only in the wakeup path, specifically in select_task_rq_fair() after the normal legality filters and after the existing wake_affine/idle-capacity logic has narrowed selection to two already-legal CPUs. There is no active use in periodic load balance, newidle balance, or nohz balance. Let BaseA and BaseB be the existing scheduler comparison values expressed in SCHED_CAPACITY_SCALE units and passed to the helper from the native wakeup code while it already holds or has computed the needed stable state. The phase helper never performs extra rq-load reads. The helper is enabled only if abs(BaseA - BaseB) <= TIE_DELTA, with TIE_DELTA = 32 and BIAS_MAX = 8. Let pref_i in {-1.0,-0.5,0,+0.5,+1.0} be the offline-mined phase preference for candidate i, based on the task phase class, CPU capacity bucket, and package-or-die locality class. Let conf_i = min(task_confidence, domain_confidence_i) in [0,1], and let stale_i = 0 if either task hint age > 4 ms or domain snapshot age > 8 ms or generation mismatches, else 1. Define bias_i = clamp(round(8 * stale_i * conf_i * pref_i) - round(4 * stale_i * conf_i * cap_pressure_i), -8, +8). Then Adj_i = Base_i + bias_i. Because bias_i is clamped, |Adj_i - Base_i| <= 8 for every candidate and the pairwise phase influence can never exceed 16. Hysteresis is explicit: if prev_cpu is one candidate, keep prev_cpu unless the alternate candidate exceeds it by at least HYST_DELTA = 4. This prevents ping-pong on repeated wakeups. The helper returns the existing scheduler choice unchanged whenever cpuset, affinity, NUMA placement, asym_packing priority, SMT policy, uclamp, thermal pressure, or EAS have already made the candidates non-equivalent or when the tie gate is not met. 5. Topology and hotplug contract. Domain enumeration prefers die domains when available and otherwise falls back to package domains. A phase_domain_map object contains an array of domain descriptors plus a per-CPU map {domain_id, topo_generation, capacity_bucket}. The map is rebuilt on CPU hotplug or topology regeneration and published with rcu_assign_pointer(). Old maps and old domain descriptors are freed only after synchronize_rcu(). Scheduler readers enter rcu_read_lock(), fetch the per-CPU map entry, read the domain snapshot with a seqcount retry loop, and reject any snapshot whose topo_generation does not match the current map generation. Readers therefore never dereference freed domain state and never mix generations. 6. Offline mining contract. The codebook is mined offline from traces gathered under shadow mode. Training labels are throughput-per-watt and latency outcomes for legal placements under controlled workloads including memory-streaming, cache-thrashing, branchy control, and Linpack-like compute bursts on homogeneous and heterogeneous x86 systems. The deployed kernel artifact is a small threshold table, for example 32 to 64 entries, versioned by sig_ver and topology class. No floating point, no runtime model loading, and no online learning are required.

Publication and lifetime rules:
- Scheduler readers take no locks, use seqcount retry for consistent u64+timestamp pairs, and tolerate 32-bit tearing by retrying until the snapshot is stable.
- RAPL writers are single-writer per domain; write_seqcount_begin/end serialize publication.
- CPU-to-domain maps are RCU-published and generation-tagged; mismatched generations force zero influence.
- If candidates are SMT siblings of the same core, the phase bias is forced to zero to avoid over-interpreting package-level pressure for per-thread SMT contention.

2.5 Implementation Plan

1. Add a CONFIG_SCHED_PHASE_HINTS feature with three modes: trace-only, shadow-score, and active. Trace-only logs signatures and RAPL snapshots only. Shadow-score computes bias but does not alter placement. Active enables the bounded wakeup tie-break. 2. Implement the per-CPU PMI/NMI ring. Use a fixed power-of-two slot count, a 64-bit reservation ticket, per-slot sequence numbers, and release/acquire ordering. On overflow, increment a drop counter and set a confidence-degraded bit in the task context. Do not take task references in NMI. 3. Introduce task cookies and phase_ctx objects. task_cookie is allocated once per task lifetime and invalidated on exec by incrementing exec_epoch. Store phase_ctx in an RCU-managed rhashtable keyed by cookie. Deferred workers resolve ring samples to phase_ctx and update a packed hint word containing {phase_class, confidence_q8, last_update_ns, sig_ver}. 4. Define the exact PMU contract in code and documentation. Use an instructions overflow leader with a fixed period of 131072. Group followers are cpu-cycles, ref-cycles, cache-misses, and branch-misses. Quantize to the documented 64-bit signature layout. If cache-misses or branch-misses are unavailable, clear caps bits and route to a smaller codebook instead of changing format. 5. Implement the RAPL writer as a freezable kthread or delayed work item pinned to a housekeeping CPU per domain. The writer may sleep and may call into powercap/intel_rapl internals, but it must not do so while holding rq locks, cpuset locks, cpuhp locks, topology mutexes, or the phase-domain-map mutex. Provider interaction is one-way: the writer calls energy-read helpers, computes pressure, and publishes via seqcount. No scheduler path ever calls the provider directly. 6. Build phase_domain_map generation logic. Enumerate die domains first; if no die domain exists for a CPU, map it to the package domain. Store per-CPU capacity_bucket derived from arch_scale_cpu_capacity() so the codebook can distinguish homogeneous versus hybrid buckets without overriding asym_packing. On hotplug or topology rebuild, allocate a new map, populate it completely, then publish with rcu_assign_pointer() and bump topo_generation. Free old maps after synchronize_rcu(). 7. Add the sched/fair hook only in the wakeup path. The helper should be called from select_task_rq_fair() only after the existing fair scheduler has already identified two legal candidates and computed the native base comparison values. Do not use it in load_balance_fair(), newidle_balance(), or periodic balancing. Pseudocode: if (!active_mode) return base_choice; if (!same_hard_constraints(cpuA, cpuB)) return base_choice; if (abs(BaseA - BaseB) > 32) return base_choice; read task_hint under READ_ONCE; read domA and domB via RCU + seqcount retry; if (invalid || stale || generation mismatch) return base_choice; biasA = clamp(round(8*confA*prefA) - round(4*confA*capA), -8, +8); biasB = clamp(round(8*confB*prefB) - round(4*confB*capB), -8, +8); AdjA = BaseA + biasA; AdjB = BaseB + biasB; if (cpuA == prev_cpu && AdjB < AdjA + 4) return cpuA; if (cpuB == prev_cpu && AdjA < AdjB + 4) return cpuB; return AdjB > AdjA ? cpuB : cpuA; 8. Handle corner cases explicitly. Use modular wrap arithmetic for RAPL energy. Use monotonic time only. Mark invalid any dt outside [0.5 ms, 20 ms]. If PL1 is zero or unknown, set confidence to zero; if only PL2 is known, allow a degraded-confidence fallback. If virtualization hides or emulates RAPL unreliably, keep trace-only PMU phase data but force scheduler bias to zero. If SMT sibling comparison is the only difference, force zero bias. If either signal stream is stale, invalid, or low confidence, consume nothing. 9. Keep the codebook small and auditable. Store it in read-only kernel memory, keyed by sig_ver and topology class. The codebook should output only coarse phase_class and per-bucket pref values, never a continuous utilization-like number.

1. Correctness and fairness: run active mode only after trace-only and shadow-score have shown stable behavior. Verify no changes to vruntime, EEVDF eligibility, task weight handling, or hard affinity semantics. Add scheduler tracepoints to confirm the helper is called only in the wakeup path and only when abs(BaseA-BaseB) <= 32. 2. Locking and lifetime: use lockdep, RCU stall detection, and hotplug stress to verify that scheduler readers never sleep, never take locks, and never dereference freed domain state. Repeatedly online/offline CPUs and rebuild topology while sampling RAPL and perf events. Confirm generation mismatch causes zero influence rather than mixed snapshots. 3. Snapshot integrity: inject artificial seqcount retries and torn 64-bit reads in test instrumentation to prove readers only consume consistent {timestamp, power, cap_pressure, generation} tuples. Validate that no scheduler path calls into powercap/intel_rapl directly. 4. Counter math: force small synthetic wrap intervals and altered power limits to validate modular energy arithmetic, dt anomaly rejection, PL1/PL2 runtime changes, and zero-confidence fallback for missing domains. Measure cap_pressure_q16 to confirm it always remains in [0,65535]. 5. Overhead: measure PMI cost, ring drops, worker backlog, RAPL writer CPU time, and scheduler delta on hackbench, schbench, tbench, kernel build, and netperf. Target less than 1 percent scheduler-path overhead in shadow mode and near-zero additional rq lock hold time because the helper performs no extra rq-state reads. 6. Workload outcomes: train and test on mixed workloads including STREAM, memcached, RocksDB or PostgreSQL OLTP, nginx or RPC microservices, Linpack/HPL bursts, and hybrid P/E systems with control threads plus compute threads. Compare baseline, shadow-score, and active modes for tail latency, throughput, and performance per watt. 7. Policy interactions: explicitly test cpusets, NUMA binding, uclamp_min/uclamp_max, asym_packing, SMT enabled/disabled, thermal pressure events, and EAS-enabled builds. Confirm the phase helper has zero effect whenever those mechanisms already produce a non-tie or disallow a candidate. 8. Ping-pong resistance: run wake-heavy microbenchmarks with alternating producer/consumer bursts and verify that the 4-unit hysteresis materially reduces migration oscillation relative to a no-hysteresis variant.

3. Patent Claims

  1. A Linux scheduler method in which per-task phase information is captured in perf PMI or NMI as a fixed-width normalized PMU signature using a same-CPU nested-safe reservation-ticket ring, resolved later through a non-reused task cookie and exec-epoch invalidation, and consumed only outside interrupt context.
  2. A power-domain publication method in which package or die x86 RAPL telemetry is sampled by a single sleepable writer, converted through explicit wrap-safe energy-delta and monotonic-time math into a bounded cap-pressure scalar in [0,1], and published to lockless scheduler readers as a seqcount-latched snapshot containing timestamp, power-limit metadata, confidence, and topology generation.
  3. A wakeup-placement method for sched/fair in which the scheduler applies the phase hint only when two already-legal candidate CPUs are within an explicit near-tie threshold in the same units as the scheduler's native comparison metric, and in which each candidate receives a symmetric bounded bias whose absolute value is clamped to a fixed maximum smaller than the near-tie threshold.
  4. The method of claim 3, wherein the scheduler retains the previous CPU unless an alternate candidate exceeds the previous CPU by a second explicit hysteresis threshold, thereby preventing wakeup ping-pong while preserving EEVDF vruntime, eligibility, and all hard placement constraints.
  5. A topology-validity method in which CPUs map to die domains when present and package domains otherwise, the CPU-to-domain map is RCU-published with a generation number on hotplug or topology rebuild, and scheduler readers reject mismatched generations without dereferencing freed domain state.
  6. A codebook-based classification method in which mined offline traces of PMU signatures and RAPL windows are compiled into a finite fixed-point threshold table versioned by signature format and topology class, allowing kernel inference without floating point, online training, or per-task energy attribution.

4. Risks & Mitigations

  • Risk: Prior-art overlap with generic RAPL-aware scheduling. Mitigation: claim the narrow combination of seqcount-latched domain snapshot semantics, generation-safe topology mapping, fixed-width PMI signature contract, and near-tie-only bounded sched/fair consumption rather than generic power-aware scheduling.
  • Risk: SKU fragility from model-specific PMU events. Mitigation: require only architectural or generic core events for the in-kernel signature and use capability bits plus codebook versioning for optional dimensions; keep uncore or model-specific detail offline only.
  • Risk: Lock inversion or scheduler interference through powercap/intel_rapl. Mitigation: isolate all provider interaction in a sleepable writer that never runs under rq, cpuhp, or topology locks, and forbid any scheduler path from calling provider APIs directly.
  • Risk: Stale or noisy telemetry causing incorrect placement. Mitigation: binary staleness gates at 4 ms for task hints and 8 ms for RAPL, confidence degradation on ring drops or dt anomalies, EMA smoothing on cap-pressure, and automatic zero influence on invalid inputs.
  • Risk: Wakeup migration oscillation. Mitigation: hard clamp bias to +/-8, require a 32-unit tie gate before activation, and add a 4-unit prev_cpu hysteresis so the feature can only perturb true near-ties.
