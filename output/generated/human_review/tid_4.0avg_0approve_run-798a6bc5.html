2. Technical Invention Disclosure

2.1 Problem Statement

On AMX-capable x86 systems, tiled compute often arrives as microbursts separated by sub-millisecond to few-millisecond gaps. During those gaps, cpuidle may select a deep C-state whose exit latency is disproportionate to the next burst, while Intel HWP as driven by intel_pstate may not raise its requested performance until generic utilization becomes visible again. Existing generic signals are a poor fit. Iowait says little about a thread that is about to resume tile compute. Scheduler utilization often rises only after the next burst has already begun. AMX permission and XFD first-use are also insufficient by themselves because a task can request AMX or incur a one-time first-use fault without entering a sustained tile-compute phase. A practical kernel solution therefore must: use only evidence that the kernel already encounters for AMX permissioning and xstate correctness; distinguish AMX permission, first-use, and definitive tile-state use; avoid reading stale or unmaterialized xstate; avoid leaking a CPU-local hint to unrelated tasks after context switches or migration; avoid new timers, IPIs, or PMU dependence; avoid new userspace ABI; reset cleanly across CPU hotplug and suspend or resume; and never override explicit user, governor, thermal, or power-limit constraints.

2.2 Prior-Art Gap

Known Linux mechanisms cover adjacent but materially different functions. Dev PM QoS and cpu-device latency QoS can constrain allowed idle exit latency, and cpuidle governors already honor those constraints. Intel_pstate and HWP can bias performance requests. Linux x86 AMX support already covers permissioning, XFD first-use, signal and context semantics, and save or restore correctness. There is also broader art around reacting to generic demand signals, iowait boosts, and heavy ISA behavior. However, none of those mechanisms discloses the claimed combination of constraints and evidence. In particular, the prior art does not teach: a two-stage AMX-specific evidence ladder in which XFD first-use is intentionally non-definitive and only a subsequently saved XTILEDATA in-use indication qualifies as definitive AMX activity; task-carried AMX recency windows used to reconstruct an effective per-CPU state only for the currently running task so that migrated or unrelated tasks do not inherit boost; one shared AMX-burst abstraction reused at both the per-CPU cpu_device PM QoS control point and the intel_pstate HWP request formation point; baseline-relative re-centering of the HWP adjustment on every existing update rather than a sticky boosted baseline; and a hard per-CPU token budget that limits AMX-caused extra HWP writes without any new periodic work. Generic recent-utilization timers, iowait boosts, or ISA-detection heuristics cannot safely provide the same coupled cpuidle and HWP alignment because they do not know that AMX permission and XFD are insufficient evidence, do not inspect saved XTILEDATA use, and do not carry AMX provenance with the task to prevent scheduler-migration contamination.

2.3 Proposed Invention

Maintain two tiny data structures: a task-carried AMX recency record and a per-CPU effective AMX power state record. The task record contains amx_probe_until_ns, amx_active_until_ns, amx_cooldown_until_ns, and amx_epoch. The per-CPU record contains effective_state, owner_epoch, qos_level_applied, hwp_tokens, and last_token_refill_ns. The effective state is always derived from the currently running task's carried record, never from a stale CPU timer alone. Evidence ladder and exact signals: 1. Eligibility gate: the task must have AMX permission or XCOMP enablement for XTILECFG and XTILEDATA. This gate alone causes no power action. 2. PROBE evidence: when the current task hits the existing AMX XFD first-use path, set current->amx_probe_until_ns = now + 250 microseconds if the eligibility gate is satisfied. This enters PROBE only. XFD does not create ACTIVE. 3. Definitive evidence: in the normal context-switch or FPU-save completion path, after the kernel has already materialized or updated the outgoing task's saved xstate header or equivalent xfeatures mask, test whether XFEATURE_MASK_XTILEDATA is set. If yes, set prev->amx_active_until_ns = now + 2 milliseconds, prev->amx_cooldown_until_ns = now + 6 milliseconds, prev->amx_probe_until_ns = 0, and increment prev->amx_epoch. XTILECFG without XTILEDATA does not create ACTIVE. 4. No forced save rule: if the current task's AMX state still lives only in registers and no normal save has occurred, the mechanism waits. It never initiates an XSAVE solely to obtain evidence. FSM and transition rules: - OFF: default state. Output: no additional PM QoS tightening, no HWP delta. - PROBE: entered only when the running task has unexpired amx_probe_until_ns and no definitive window. Output: still no PM QoS tightening and no HWP delta. Purpose: remember that AMX first-use occurred recently while awaiting definitive use. - ACTIVE: entered when the running task has unexpired amx_active_until_ns. Output: tighten the per-CPU cpu_device dev_pm_qos resume-latency request to 20 microseconds and apply an HWP desired_perf delta of +6 units. - COOLDOWN: entered when amx_active_until_ns has expired but amx_cooldown_until_ns remains unexpired. Output: tighten the per-CPU cpu_device dev_pm_qos resume-latency request to 60 microseconds and apply an HWP desired_perf delta of +2 units. - Any state to OFF: immediate when the running task has no unexpired AMX window, when a different task without AMX recency is switched in, when the CPU goes offline, when the system suspends, when AMX is disabled, or when intel_pstate is not in active HWP mode. - ACTIVE refresh: every new definitive evidence event resets the task's ACTIVE and COOLDOWN expiry windows. Anti-contamination and migration rules: - Definitive evidence is stamped onto the task, not only the source CPU. - On switch-out of an AMX task, the source CPU clears any effective owner unless the next task rehydrates the same epoch. - On switch-in, the destination CPU reconstructs its effective state solely from the incoming task's timestamps and epoch. If the incoming task lacks AMX recency, the CPU enters OFF immediately even if the previous task on that CPU was in ACTIVE or COOLDOWN. - This makes the AMX hint task-scoped for provenance but per-CPU for consumption. Cpuidle control point and contract: - At CPU online, create one dev_pm_qos request attached to that cpu_device. - Update that request only when the effective AMX state changes. OFF and PROBE map to the default unconstrained value. ACTIVE maps to a 20 microsecond resume-latency target. COOLDOWN maps to a 60 microsecond target. - Because cpu-device PM QoS uses minimum semantics, the invention can only tighten the allowed latency relative to the default. If another client is already tighter, nothing is loosened or overridden. - On CPU offline or suspend, remove or reset the request so no policy leaks across lifetime events. HWP control point and contract: - Consume the effective AMX state only inside the existing intel_pstate HWP request formation path. No self-scheduled MSR writes are allowed. - Start from the baseline request the governor and intel_pstate would already have produced for that update. This is the re-centering step. - Bias only desired_perf. Leave EPP, max_perf, and the baseline min_perf unchanged. - Compute desired_perf_biased = clamp(baseline_desired_perf + delta, lower = max(baseline_min_perf, user_min_limit), upper = min(baseline_max_perf, thermal_cap, powercap_cap, user_max_limit)), where delta is 0 in OFF or PROBE, +6 in ACTIVE, and +2 in COOLDOWN. - If the clamp collapses the delta, the invention has no HWP effect for that update. - If baseline logic was already going to issue an HWP MSR write, the biased desired_perf may piggyback on that write without spending budget. If baseline logic would not write, an AMX-caused extra write is permitted only if a per-CPU token bucket has available credit. - Token bucket: refill lazily only when intel_pstate already runs, at one token per 10 milliseconds, capped at three tokens. No timer or IPI is created. If no token is available, the baseline request is used and the AMX delta is deferred or dropped. PMU policy: - No PMU signal is required for any claim of the core invention. - In an optional dependent embodiment, a model-whitelisted AMX activity counter may extend ACTIVE by at most 250 microseconds only if the event is known reliable for that model and the kernel successfully acquires explicit exclusive ownership of the counter resource; otherwise PMU corroboration is disabled. The core mechanism remains fully functional without PMU input.

2.4 Architecture

The design has four stages: existing AMX evidence capture, task-carried recency stamping, per-CPU effective-state rehydration, and bounded reuse by cpuidle and HWP.

Transition table:
- OFF -> PROBE: running task hits AMX XFD first-use and probe_until not expired.
- PROBE -> ACTIVE: later switch-out or save path observes saved XTILEDATA in outgoing task.
- PROBE -> OFF: probe_until expires before definitive evidence.
- ACTIVE -> ACTIVE: new definitive evidence refreshes active_until and cooldown_until.
- ACTIVE -> COOLDOWN: active_until expires, cooldown_until still valid.
- COOLDOWN -> OFF: cooldown_until expires.
- ANY -> OFF immediately: switch to unrelated task without AMX recency, CPU offline, suspend, AMX disabled, or HWP path unavailable.

Cheapness and correctness constraints:
- The design never reads xcomp_bv as proof of use, because xcomp_bv expresses enabled layout, not dynamic use.
- The design never forces XSAVE just to inspect XTILEDATA.
- The design only tests the xfeatures mask after the existing save path has made that mask authoritative.
- The per-CPU state is an effective consumption cache, not the source of truth; the task-carried timestamps are the source of truth for provenance and migration correctness.

2.5 Implementation Plan

1. Add a compact AMX recency record to the task's x86 thread or FPU-adjacent metadata: probe_until_ns, active_until_ns, cooldown_until_ns, and epoch. 2. Add a per-CPU amx_pm_state structure: effective_state, owner_epoch, last_qos_state, hwp_tokens, last_token_refill_ns. 3. In the existing AMX XFD first-use handler, if the task has AMX permission for XTILECFG and XTILEDATA, set probe_until_ns = now + 250 microseconds. Do not set active_until_ns here. 4. In the existing FPU or xstate save completion path used on context switch-out, after the saved xfeatures mask is authoritative, test XFEATURE_MASK_XTILEDATA. If set, stamp active_until_ns = now + 2 milliseconds, cooldown_until_ns = now + 6 milliseconds, clear probe_until_ns, and increment epoch. If only XTILECFG is present, do nothing beyond any existing AMX bookkeeping. 5. In the schedule switch-in path, compute the incoming task's effective state from its timestamps: ACTIVE if now < active_until_ns, else COOLDOWN if now < cooldown_until_ns, else PROBE if now < probe_until_ns, else OFF. Write that state to the per-CPU record together with owner_epoch = task epoch. If the task has no recency, write OFF immediately. 6. Wire state transitions to one cpu_device dev_pm_qos request per CPU. Create the request in CPU hotplug online, update it only on state changes, and remove or reset it on CPU offline and suspend. 7. Extend the existing intel_pstate HWP request assembly path to snapshot the current CPU's effective AMX state with READ_ONCE, compute the baseline request first, then apply the desired_perf-only delta and final clamp ordering. Reuse existing locking or serialization in intel_pstate for the final request write path. 8. Implement the per-CPU token bucket in the HWP path itself. Refill tokens lazily from elapsed time since last_token_refill_ns. Allow piggyback on an already-needed baseline write without consuming a token. Only AMX-caused extra writes consume tokens. 9. Add suppression checks: feature disabled when CPU lacks AMX, when AMX is administratively disabled, when intel_pstate is not in active HWP mode, or during hotplug and suspend transitions. 10. Keep the fast path small. Non-AMX tasks should incur only a zero-metadata branch on switch-in; non-definitive AMX tasks should incur at most the existing XFD work plus timestamp stores; definitive evidence adds only one bit-test on an already-available xfeatures mask and a small metadata update.

2.6 Validation Plan

1. Functional evidence ladder test: run an AMX-enabled process that requests permission but never executes tile instructions; confirm the state remains OFF. Then execute one AMX first-use fault without subsequent XTILEDATA save; confirm PROBE appears briefly but no cpuidle tightening or HWP delta is emitted. Then run a real tiled microkernel and confirm ACTIVE is entered only after saved XTILEDATA is observed. 2. Migration contamination test: run an AMX microburst task on CPU A, migrate it to CPU B, and schedule an unrelated scalar workload on CPU A. Confirm CPU A drops to OFF immediately for the scalar task and CPU B rehydrates ACTIVE or COOLDOWN only for the migrated AMX task. 3. Cpuidle behavior test: with ftrace or tracepoints, verify that the cpu_device dev_pm_qos request changes only on AMX state transitions and that governors avoid states whose exit latency exceeds the ACTIVE or COOLDOWN target while leaving tighter third-party QoS requests untouched. 4. HWP composition test: compare baseline and biased IA32_HWP_REQUEST formation under user min or max constraints, thermal throttling, and powercap. Confirm desired_perf is the only field altered by the invention, that clamp ordering preserves all preexisting bounds, and that no write occurs when the clamp removes the delta. 5. Write-budget test: create a workload that would cause frequent AMX recency but little baseline HWP churn. Confirm the token bucket caps AMX-caused extra writes to three burst writes and then at most one additional write per 10 milliseconds thereafter, while piggyback writes remain allowed. 6. No-forced-XSAVE test: instrument the relevant xstate save path to confirm the mechanism never triggers an extra save solely for inspection and only reads xfeatures after the existing save path has already updated it. 7. Hotplug and suspend test: online and offline CPUs and cycle suspend or resume while AMX tasks are present. Confirm all per-CPU effective states reset to OFF and dev_pm_qos requests are removed or restored to default with no stale tightening. 8. Performance and energy study: benchmark bursty AMX GEMM or convolution loops with 200 microsecond to 3 millisecond gaps, measure tail latency to next-burst completion, idle-state residency, HWP request traces, package energy, and average power. Compare baseline, cpuidle-only, HWP-only, and combined invention modes to show that the shared AMX-specific abstraction delivers the intended alignment benefit. 9. Regression study on non-AMX workloads: run scheduler-intensive scalar and AVX workloads and confirm the added overhead is negligible and that no state transitions or PM actions are emitted absent AMX evidence.

4. Risks & Mitigations

  • Risk: saved xstate may be stale if inspected before the kernel has materialized it from registers. Mitigation: inspect XTILEDATA only after the normal save completion path has already made the saved xfeatures mask authoritative; never force XSAVE for this feature.
  • Risk: XFD first-use could create false positives if treated as definitive AMX activity. Mitigation: XFD only creates PROBE, which has no cpuidle or HWP output; ACTIVE requires later saved XTILEDATA evidence.
  • Risk: per-CPU boosting could leak to unrelated workloads after a context switch or task migration. Mitigation: make task-carried recency timestamps the source of truth and reconstruct the effective per-CPU state only from the currently running task; clear OFF immediately for unrelated tasks.
  • Risk: HWP bias may race with existing intel_pstate updates or policy changes. Mitigation: compute the AMX delta only inside the existing intel_pstate HWP request assembly path from a per-CPU snapshot, then apply final clamp ordering before the normal write decision.
  • Risk: excessive HWP writes could create overhead or be viewed as routine aggressive tuning. Mitigation: no self-scheduled writes, piggyback on baseline writes when possible, and gate any AMX-caused extra write with a lazy per-CPU token bucket.
  • Risk: PMU-based corroboration may conflict with perf, watchdogs, or model-specific counter availability. Mitigation: exclude PMU from the core invention entirely; if used in a dependent embodiment, require explicit exclusive ownership and a model whitelist with automatic fallback to the architectural path.
  • Risk: prior-art overlap with generic boost heuristics or ISA-aware throttling. Mitigation: emphasize the claim limitations requiring the AMX-specific evidence ladder, the saved XTILEDATA definitive gate, task-carried provenance to prevent migration leakage, baseline-relative HWP re-centering, and the hard write budget.
  • Risk: fixed timing windows may be suboptimal across platforms. Mitigation: describe the windows as internal kernel constants or model-tuned defaults while preserving the claim's relative structure of non-definitive PROBE, bounded ACTIVE, and bounded COOLDOWN.
