2. Technical Invention Disclosure

2.1 Problem Statement

On large Xeon-class Linux systems, interrupt affinity changes, managed-IRQ rebalancing, queue topology changes, and CPU hotplug can retire an old numeric x86 vector before software has an allocator-visible proof that the old route can no longer target that vector on the previous CPU. Existing arch/x86 cleanup logic is correctness-oriented and distributed across migration paths; it does not expose a typed, auditable distinction between numerically free and safe-to-reuse to irq_matrix. That gap blocks deterministic vector-capacity accounting, safe service-class placement, and reusable telemetry under dense MSI-X traffic from NICs, NVMe, and accelerators.

2.2 Prior-Art Gap

Linux already contains irq_move_cleanup, send_cleanup_vector, vector reservations, and route-specific masking, so the invention is not a claim to delayed reuse in the abstract. The gap is that current code does not force irq_matrix to observe a formal QUARANTINED versus REASSIGNABLE contract keyed by route taxonomy, proof bits, generation, and CPU hotplug epoch. Topological Void 1 highlights an unexplored connection between X86_FEATURE_APIC and apic_find_highest_irr; this draft narrows that connection to optional supplemental telemetry only, never sole authorization. Void 2 highlights a missing bridge between Intel application performance tooling and IRQ allocator state; this draft adds quarantine, retry, and band-pressure KPIs that can be correlated with perf, ftrace, BPF, VTune, Intel PCM, and Linpack-style mixed workload studies. Void 3 highlights confusion between APIC-delivered vectors and legacy PIC-style IRQ configuration; this draft resolves that with an explicit allow and deny matrix that excludes 8259 PIC, local-APIC system vectors, per-CPU vectors, and hypervisor or posted-interrupt paths.

2.3 Proposed Invention

Introduce a native Linux x86 vector-reuse contract in which an old vector first becomes QUARANTINED and only later becomes REASSIGNABLE after arch/x86 records route-specific proof of no stale targeting. Each CPU and vector slot gets metadata for state, generation, route type, proof bits, service band, old target CPU, and hotplug epoch, and new arch APIs publish this state to irq_matrix so allocation can only consume REASSIGNABLE slots. Authorization rules are type-specific: direct MSI or MSI-X requires mask, synchronize_irq, new message commit, and old-target cleanup completion; interrupt-remapped MSI or MSI-X also requires IRTE invalidation completion; IOAPIC edge requires mask, synchronize_irq, RTE commit, and old-target cleanup; IOAPIC level additionally requires level-quiesce evidence such as remote-IRR clear or controller-observed EOI completion. A bounded local-APIC IRR or ISR probe may be sampled only on the old target CPU in trusted native xAPIC or x2APIC contexts and is explicitly supplemental evidence, never sufficient by itself. Above that safety layer, service-class banding partitions only the allocatable irq_matrix set, keeps quarantined vectors charged to their originating band, and uses floor-based borrowing and repayment so critical NIC, NVMe, and accelerator queues can prefer higher allocatable vectors without touching reserved or sysvec ranges.

2.4 Architecture

State location and lifecycle: arch/x86 maintains a per-CPU per-vector x86_vector_slot_meta table beside existing vector ownership tables. Each slot carries state IN_USE, QUARANTINED, or REASSIGNABLE, plus generation, route_type, proof_mask, band_id, loan_owner, old_cpu, and hotplug_epoch. Locking and ordering: route reprogramming occurs under the existing IRQ descriptor and chip or domain serialization, then vector serialization updates the slot metadata, then irq_matrix publishes bitmap state; QUARANTINED and REASSIGNABLE publication uses store-release and allocator reads use load-acquire so a recycled vector never observes stale proof bits. New invariant: irq_matrix may allocate only slots marked REASSIGNABLE, never merely numerically free. Allow matrix: direct MSI, MSI-X, interrupt-remapped MSI or MSI-X, IOAPIC edge, and IOAPIC level are eligible only when their domains expose mandatory masking and quiesce hooks. Deny matrix: NMI, SMI, IPI, local-APIC timer, thermal, error, spurious, and other sysvec entries, per-CPU interrupts, legacy 8259 PIC routes, KVM posted interrupts, APICv or hypervisor-injected delivery, and any irqdomain lacking a trustworthy no-stale-targeting acknowledgement. Optional APIC drain probe envelope: trusted only on native bare metal where X86_FEATURE_APIC is present, the kernel owns LAPIC access, the callback executes on the old target CPU with local interrupts disabled, the candidate is an external APIC-delivered vector, and no virtualization or posted-interrupt interception is active; the callback reads the exact ISR and IRR bit for the candidate vector and may log apic_find_highest_irr as coarse telemetry. Band layout: for each CPU, build a sorted allocatable-vector list after subtracting all existing reserved, system, and per-CPU vectors; assign the highest max(8, ceil(20 percent)) entries to Critical, the next max(16, ceil(35 percent)) to Preferred, and the remainder to BestEffort. Borrowing is one-way from lower to higher service classes, lending never reduces the lender below its floor, loans are tagged per slot, and repayment occurs before fresh allocation so capacity is not permanently stranded. Isolation rule: nohz_full or isolated CPUs may consume Critical only for explicitly allowlisted IRQ personas, while housekeeping IRQ classes remain pinned to housekeeping CPU masks.

2.5 Implementation Plan

Phase 1 adds metadata and the allocator contract in arch/x86 vector management and kernel irq_matrix code: add slot metadata, QUARANTINED and REASSIGNABLE bitmaps, generation counters, and allocation helpers that refuse quarantined slots. Phase 2 wires typed route proofs into existing programming points. For direct MSI or MSI-X: mask at the device or irq_chip masking point, call synchronize_irq, compose and write the new MSI message, detach the old vector into QUARANTINED, unmask on the new target, and authorize reuse only after old-target move-cleanup completion for the same generation. For interrupt-remapped MSI or MSI-X: perform the same steps plus IRTE update and queued invalidation completion; only the tuple of mask or sync, IRTE commit, invalidation completion, and old-target cleanup upgrades the slot. For IOAPIC edge: mask the RTE, call synchronize_irq, commit the new RTE destination and vector, and authorize only after old-target cleanup completes. For IOAPIC level: mask the RTE, wait for controller-visible in-service completion and remote-IRR clear or equivalent level acknowledgement, call synchronize_irq, commit the new RTE, and authorize only after both level-quiesce evidence and old-target cleanup arrive. Managed IRQ migration reuses the same route-specific rules. CPU hotplug adds two x86 CPUHP hooks: CPUHP_AP_X86_VECTOR_QUIESCE publishes an offlining epoch, removes the CPU from eligible targets, retargets managed IRQs, freezes new loans, and drains pending move-cleanup work; CPUHP_AP_X86_VECTOR_DEAD runs after LAPIC shutdown and after route updates can no longer target that CPU, then flips any proof-complete quarantined slots for that epoch to REASSIGNABLE. Slots lacking full proof remain on a dead-CPU quarantine list and are not allocatable until the missing proof arrives or an explicit operator-visible timeout fires. The optional APIC drain probe is implemented as a rendezvous callback only in trusted bare-metal contexts and is ignored in virtualized x2APIC, posted-interrupt, or hypervisor-mediated environments. Telemetry is tracepoint-only and migration-only: vector_quarantine_enter, vector_quarantine_authorize, vector_quarantine_timeout, vector_band_borrow, vector_band_repay, and vector_reuse_retry.

2.6 Validation Plan

Validate in three layers. Functional safety: KUnit or kernel selftests that exercise direct MSI, remapped MSI, IOAPIC edge, IOAPIC level, managed IRQ affinity, and CPU hotplug races, proving that irq_matrix never reallocates a quarantined slot and that denied routes are rejected. Concurrency safety: stress migration and CPU offline or online under lockdep, hotplug stress, and high-rate IRQ injection while asserting monotonically increasing generation numbers, correct band charge accounting, and no reuse before proof bits plus epoch match. Performance and business relevance: on Xeon-class hosts with 100G or 200G NICs, NVMe, and accelerator queues, measure p95, p99, and p99.9 application latency, IRQ allocation retries, vector exhaustion events, quarantine duration distribution, and housekeeping interference under mixed networking, storage, and Linpack-style compute loads. Correlate the new tracepoints with perf, ftrace, BPF, VTune, and Intel PCM so operators can attribute latency spikes to vector pressure versus device or scheduler causes. Probe-specific validation hard-requires that the APIC IRR or ISR callback is ignored in guest, posted-interrupt, or intercepted-x2APIC modes and that successful probes never authorize reuse without the route-specific proof tuple.

4. Risks & Mitigations

  • Risk: route taxonomy misses a controller-specific edge case. Mitigation: default-deny any route without mandatory mask, synchronize, and no-stale-target acknowledgement hooks, and require explicit per-domain opt-in.
  • Risk: APIC drain probing is misconstrued as a correctness prerequisite. Mitigation: make the probe compile-time and runtime gated, supplemental only, tracepoint-visible, and never sufficient for authorization.
  • Risk: service-class banding starves lower classes or strands capacity. Mitigation: enforce per-band floors, one-way borrowing, tagged loans, first-repay semantics, and quarantine charge retention until proof-complete release.
  • Risk: CPU hotplug races cause premature release of quarantined vectors. Mitigation: bind release to generation plus hotplug epoch matching and split offline flow into quiesce and dead milestones with explicit targetability shutdown before allocator exposure.
  • Risk: commercial value is limited to a niche kernel deployment set. Mitigation: target mainstream high-core-count server personas such as NIC, NVMe, storage, AI inference, and mixed IO plus compute hosts where MSI-X density and CPU isolation are common.
  • Risk: prior-art conflict from existing Linux cleanup behavior. Mitigation: claim the allocator-visible contract, typed proof tuples, and isolation-safe band accounting as the core inventive combination, and treat APIC probing and telemetry as dependent features.
