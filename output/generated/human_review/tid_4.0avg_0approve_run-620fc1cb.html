2. Technical Invention Disclosure

2.1 Problem Statement

Linux NVMe polled completion is a hot path that currently inspects CQEs one at a time. In the baseline pattern, software reads cqe->status, tests the phase-validity predicate used by nvme_cqe_valid(), executes dma_rmb() before consuming any non-status field, decodes result/command_id/sq_head, completes one request, advances cq_head, toggles phase on wrap, and eventually writes the CQ head doorbell. This scalar loop is simple but branch-heavy and repeats control overhead even when the device has produced a long contiguous valid run. The optimization challenge is subtle: the only correctness-defining predicate for "CQE is present" is the phase bit in the 16-bit status field, where valid means ((le16_to_cpu(status) & 0x0001) == expected_phase). Error completions are still valid under that predicate; they are not filtered out by the classifier and must flow into the normal error-handling path after dma_rmb(). The device may also expose later CQEs before earlier ones become visible to the CPU, so any batched classifier must tolerate observing status matches for entry i+1 while entry i is still invalid. Therefore the optimization target is not arbitrary set membership; it is the longest consecutively valid prefix starting exactly at cq_head, with phase P before wrap and P^1 after wrap, under the existing single-consumer and doorbell-ordering rules.

2.2 Prior-Art Gap

Existing Linux NVMe code and known software optimization guidance leave a specific gap. Kernel code already has nvme_cqe_valid(), dma_rmb()-then-decode ordering, and scalar completion handling, but it does not materialize a status-only intermediate representation or compute the contiguous valid prefix with AVX-512 opmask extraction. Intel AVX-512 optimization literature teaches vector compares, masks, and prefix-friendly bit operations, but it does not describe a Linux-kernel-safe bridge for coherent DMA completion queues where only a 16-bit phase field may be speculatively read, no other CQE field may be touched before dma_rmb(), wrap changes the expected phase exactly once, and device visibility anomalies must be handled by shrinking the prefix during in-order re-validation. Generic snapshot/revalidate patterns are not enough because they do not define the NVMe-specific valid predicate, the two-segment wrap scan, the scratch-IR safety boundary, the single-consumer queue ownership rule, or the requirement that doorbell advancement be based on actual_consumed rather than the optimistic SIMD prefix. That missing bridge is the novelty gap.

2.3 Proposed Invention

The invention adds a bounded, poll-only classifier stage ahead of the existing scalar completion path. Step 1: assert the safety envelope. The queue must be a dedicated poll queue with one software consumer; if IRQ completion is armed for that queue, or if the caller is in hardirq, softirq, NMI, PREEMPT_RT-disallowed context, or any context where kernel_fpu_begin()/kernel_fpu_end() is not allowed or not worthwhile, the path falls back immediately to the existing scalar nvme_poll_cq(). Step 2: perform a short scalar pre-probe of the first 4 to 8 CQEs from cq_head using READ_ONCE(cqe->status) and the exact predicate phase_match(status, phase) := ((le16_to_cpu(status) & 0x0001) == phase). If the pre-probe finds a short prefix, stay scalar so SIMD never regresses small bursts. Step 3: snapshot only status words into a fixed per-queue scratch IR. With W = min(W_max, q_depth, remaining_budget) and W_max fixed at 32 in the primary embodiment, copy READ_ONCE(cq[idx].status) into status_ir[k] for k in [0, W), mapping idx = (cq_head + k) mod q_depth. No other CQE word is read here. Step 4: classify the scratch IR in at most two segments. Let tail = min(W, q_depth - cq_head). For k in [0, tail), expected_phase = P. For k in [tail, W), expected_phase = P^1. AVX-512 performs phase_bits = status_ir & 0x0001, compares phase_bits to a broadcast expected phase per segment, extracts the compare mask, and computes the first zero bit to obtain the segment prefix. Pseudocode: tail_prefix = prefix_len(mask(status_ir[0:tail], P)); if tail_prefix < tail then candidate_prefix = tail_prefix; else head_prefix = prefix_len(mask(status_ir[tail:W], P^1)); candidate_prefix = tail + head_prefix. Here prefix_len() returns the index of the first non-match or the full segment length if all match. Step 5: consume only by ordered live re-validation. For n from 0 to candidate_prefix-1, compute live_idx = (cq_head + n) mod q_depth and live_phase = P xor wrap_crossed(live_idx). Re-read live_status = READ_ONCE(cq[live_idx].status). If phase_match(live_status, live_phase) is false, stop immediately; do not inspect or complete later CQEs, even if the scratch IR showed them as matching. If true, execute dma_rmb(), then call the existing scalar completion decode/handle path, including normal treatment of success and error CQEs. Step 6: commit only actual_consumed entries. Advance cq_head, toggle phase on wrap exactly as the scalar path does, and ring the CQ head doorbell only for actual_consumed, not for candidate_prefix. If re-validation stopped early, no correction is needed because speculative entries were never committed. This makes the SIMD classifier advisory only and makes out-of-order device visibility safe by construction.

2.4 Architecture

Control flow and safety boundary:

  coherent DMA CQ memory
        |
        |  exact-width READ_ONCE(__le16 status) only
        v
  +---------------------------+
  | per-queue status_ir[32]   |  64B aligned, CPU-private scratch
  +---------------------------+
        |
        |  AVX-512BW mask on phase bit only
        v
  +---------------------------+
  | tail/head prefix finder   |  segment 0 expects phase P
  | - tail [head..end)        |  segment 1 expects phase P^1 after wrap
  | - wrapped [0..]           |  candidate_prefix <= min(32, q_depth, budget)
  +---------------------------+
        |
        |  advisory only; no CQE decode yet
        v
  +---------------------------+
  | ordered live re-validation|  READ_ONCE(status) -> phase match?
  | and existing scalar path  |  dma_rmb() -> read other CQE fields
  +---------------------------+
        |
        |  actual_consumed only
        v
  cq_head update / phase toggle / existing doorbell write

Exact validity predicate: valid-for-consumption means only that the phase tag matches expected phase, i.e. ((le16_to_cpu(status) & 0x0001) == expected_phase). This is intentionally identical to nvme_cqe_valid() semantics. Error CQEs remain valid and are completed normally after dma_rmb() and full status decode.

Memory-ordering model: the CQ resides in coherent DMA memory already used by Linux NVMe. The status field is 16-bit aligned within struct nvme_completion, so the same alignment assumption already relied upon by scalar code applies; on x86, an aligned 16-bit CPU load does not tear. READ_ONCE() prevents the compiler from widening, merging, or reusing status loads. No additional CPU barrier is required between the scratch copy and the SIMD compare because both operate only on CPU-private data and authorize nothing. The only correctness-critical ordering remains the existing live re-validation rule: after a fresh phase-match on the live CQE status, execute dma_rmb() before reading any other CQE field.

Concurrent-consumer rule: the fast path is enabled only for queues whose completion ownership is single-consumer, such as dedicated poll queues where IRQ completion is disabled or not armed for that queue. If poll and ISR could race on the same cq_head or doorbell, the AVX-512 path is not entered; the queue stays on the existing scalar path.

2.5 Implementation Plan

1. Add a build-time option such as CONFIG_NVME_PCI_AVX512_POLL, default off, depending on X86, 64BIT, and the existing kernel SIMD/xstate support. The implementation should use an out-of-line x86 helper (for example arch/x86/lib/nvme_avx512_prefix.S with a C prototype) or an existing arch/x86 SIMD wrapper, not ad-hoc compiler intrinsics in the core NVMe path. 2. Extend struct nvme_queue with a small aligned scratch buffer, preferably per-queue rather than on-stack, e.g. u16 *status_ir or a fixed __aligned(64) u16 status_ir[32]. This bounds memory to 64 bytes per queue and avoids large stack frames in the poll path. 3. Add a static-key runtime gate initialized at queue setup after checking CPU features and xstate policy: AVX-512F+BW availability, kernel willingness to use the required SIMD state, and queue eligibility as a poll-only single-consumer queue. 4. In nvme_poll_cq(), keep the scalar path as the canonical fallback. Before entering SIMD, check the context envelope: process-context poll only; no NMI, no hardirq/softirq use, no unsupported RT/raw-lock state, and only when kernel_fpu_begin()/kernel_fpu_end() is legal. If any check fails, branch directly to the unchanged scalar loop. 5. Add a scalar pre-probe heuristic. Example policy: read and test the first 4 CQE statuses; if fewer than 4 are consecutively valid, stay scalar. If 4 or more are valid and the queue depth is at least 16, acquire the kernel SIMD state once for the whole poll window and classify up to W_max=32 entries. This prevents kernel_fpu overhead from hurting tiny prefixes. 6. Implement the two-segment wrap algorithm exactly: limit = min(32, q_depth, remaining_budget); tail = min(limit, q_depth - cq_head); copy tail statuses starting at cq_head; if limit > tail, copy the wrapped statuses from index 0. Compare phase_bit(status_ir[0:tail]) against P and phase_bit(status_ir[tail:limit]) against P^1. Derive candidate_prefix from the first non-match in segment order. 7. Re-validate and consume in order. Pseudocode: actual = 0; phase = cq_phase; idx = cq_head; while actual < candidate_prefix { status = READ_ONCE(cq[idx].status); if (((le16_to_cpu(status) & 1) != phase)) break; dma_rmb(); existing nvme_handle_cqe(...); actual++; idx++; if (idx == q_depth) { idx = 0; phase ^= 1; } } Then update cq_head/cq_phase and ring the doorbell exactly as today using actual, not candidate_prefix. 8. Repeat the window until poll budget is exhausted. Since W_max is bounded, long valid runs are handled as repeated windows, avoiding oversized scratch buffers while still amortizing control overhead. 9. Document the microarchitectural cost model in code comments and commit text: for a 32-entry window, the fast path pays the same 32 exact-width status loads the scalar path already needs, plus 64B of scratch stores and a handful of vector ops (AND, compare, kmov, tzcnt). The major savings are removal of 32 branchy validity checks from the classification stage and reduced loop-control overhead. The SIMD path should therefore be gated so it is used only when the expected prefix is long enough to amortize kernel_fpu_begin()/kernel_fpu_end().

1. KUnit/unit tests for pure helpers: phase_match(), tail/head split math, wrap toggle logic, mask-to-prefix extraction, and corner cases including cq_head at q_depth-1, q_depth < 32, empty queue, full 32-entry window, and all-valid-across-wrap. 2. x86 selftests for the assembly/SIMD helper to prove that for every 16-bit status vector and expected phase, the AVX-512 result matches a scalar reference prefix finder for both non-wrap and wrap segments. 3. Fault-injection/reorder tests using a mock CQ producer or instrumented test harness that makes a later CQE phase-visible before an earlier one. Expected result: candidate_prefix may overestimate, but actual_consumed stops at the first failing live re-validation, and cq_head/doorbell advance only by actual_consumed. 4. Locking/context validation: verify the fast path is never entered in interrupt or disallowed RT contexts and that queues with possible mixed poll/IRQ ownership remain on the scalar path. 5. Functional NVMe regression on real hardware and emulation, covering success CQEs, error CQEs, queue wrap, deep queues, shallow queues, and repeated poll windows. 6. Performance evaluation with fio or io_uring polling workloads at multiple queue depths and prefix distributions. The acceptance criterion is no regression when prefixes are short because the scalar pre-probe blocks SIMD entry, and measurable CPU-per-IO reduction when prefixes are commonly 8+ entries. 7. Memory-ordering review against Documentation/core-api/wrappers/memory-barriers.rst: confirm that no non-status CQE field is read before dma_rmb() on any path and that scratch-IR classification cannot commit entries without fresh live status checks.

  5. 4. Risks & Mitigations

  • Risk: PCIe/device visibility may allow a later CQE's phase bit to become visible before an earlier CQE, causing the scratch classifier to overestimate the valid run. Mitigation: treat SIMD output as advisory only; re-read live status in order, stop at the first invalid CQE, and commit cq_head/doorbell using actual_consumed only.
  • Risk: kernel SIMD use may be illegal or too expensive in some contexts, especially interrupt, NMI, softirq, or PREEMPT_RT-sensitive paths. Mitigation: hard-gate the fast path to dedicated process-context poll queues and fall back immediately to the existing scalar path whenever kernel_fpu_begin()/kernel_fpu_end() is disallowed or not profitable.
  • Risk: copying status fields into scratch could erase the SIMD gain on short bursts. Mitigation: bound W_max at 32, keep scratch per-queue and cacheline-aligned, use a scalar pre-probe threshold, and invoke SIMD only when recent behavior or the current prefix suggests break-even.
  • Risk: incorrect memory-ordering claims could accidentally permit reads of non-status CQE fields before device writes are ordered. Mitigation: make the contract explicit that classification reads only status, scratch compare requires no DMA barrier because it authorizes nothing, and every consumed CQE still executes the existing live-status check followed by dma_rmb() before decode.
  • Risk: prior-art overlap with generic vector scans or generic batching. Mitigation: claim and describe the narrower bridge: exact 16-bit status-only IR from DMA memory, two-segment phase-aware wrap handling, advisory SIMD prefix extraction, ordered scalar proof, and unchanged Linux completion and doorbell semantics.
