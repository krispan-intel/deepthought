2. Technical Invention Disclosure

2.1 Problem Statement

Linux block-I/O request preparation is often bottlenecked by metadata handling rather than media bandwidth. The kernel must walk pointer-rich bio_vec streams, test contiguity, respect queue limits, account for bounce and highmem behavior, honor DMA and virtual boundary restrictions, handle integrity and zone constraints, and update segment accounting in a branch-heavy scalar loop before request submission. On modern x86 servers with fast NVMe and fragmented bios, this scalar metadata path can dominate CPU time. A related inefficiency exists in data writeback: some spans are entirely zero, but the kernel cannot safely replace a normal WRITE with WRITE ZEROES unless it can prove that the bytes are stable from folio scan to submission, that the writeback path is byte-preserving from folio to media, that REQ_NOUNMAP semantics survive every stacked device, and that completion and retry behavior remain semantics-equivalent to ordinary writes.

2.2 Prior-Art Gap

Known optimizations for this area typically use scalar unrolling, prefetch, wider batching, or generic SIMD comparisons, but they do not define a semantics-tight contract that separates copied metadata collection from authoritative merge decisions. They generally lack: an explicit ownership and invalidation model for any copied bio_vec snapshot; a one-sided proof system in which the fast path is allowed to prove only non-mergeability; mandatory re-entry into the unchanged scalar Linux predicate chain as the only source of truth for boundaries, bounce, DMA handling, and side effects; and a concrete legality model for in-kernel AVX-512 use, including may_use_simd(), kernel_fpu_begin/end, preemption and IRQ constraints, and anti-regression heuristics for AVX-512 downclock behavior. Likewise, existing zeroing pathways such as REQ_OP_WRITE_ZEROES, blkdev_issue_zeroout, filesystem hole and unwritten-extent handling, or simplistic zero-page scans do not provide an enforceable identity-writeback gate, a stable-bytes proof from scan through bio construction, or an end-to-end effective-target NOUNMAP and strict-completion contract across dm, md, and other stacking layers. The gap identified by the Intel AVX-512 optimization void is therefore not merely to use wider instructions, but to bind AVX-512 mask-first classification to Linux block-layer legality and writeback semantics in a way that remains correct under real kernel races and stacked-device behaviors.

2.3 Proposed Invention

The invention introduces a Bounded Extent Window, or BEW, with fixed lane count N and a formal proof contract. 1. Snapshot ownership and lifetime. At a point where a bio is fully constructed for submission and before lower-layer split or scatterlist emission, the submit path enters the queue and allocates a short-lived blk_bew_window on the stack or in per-CPU scratch. The window copies only immutable or already-authoritative fields needed for conservative reasoning: starting sector, length in bytes and sectors, offset, merge-relevant op class, selected bio and request flags, integrity and zone markers, and optionally a bounce-resolved page or physical identity token only when that token is already authoritative in the current path. The snapshot also stores a cookie composed of the bio pointer, bi_vcnt, starting sector, relevant bi_opf bits, and a request-queue limits generation value qlim_gen. The snapshot never escapes the current submit call, is never attached to the bio, and is discarded immediately after scalar revalidation. If any cookie component or qlim_gen changes before or during use, the BEW is invalidated and the code falls back to the pure baseline scalar path. 2. One-sided MUST-SPLIT proof. The prefilter computes a boundary mask where bit i corresponds to the boundary between lane i and lane i+1. A bit may be set only when the implementation can prove that the adjacent descriptors cannot legally co-reside in one emitted segment under the baseline Linux predicates. Proof sources include non-contiguous sectors; op-class or merge-flag mismatch; integrity or zone constraints; impossible offset or page continuity; lower-bound segment size overflow relative to queue limits; and DMA or virtual boundary impossibility when an authoritative physical token is available. If bounce, DMA, or physical identity information is not authoritative, the prefilter must leave the boundary as unknown rather than infer mergeability. An unset bit always means unknown, never safe-to-merge. 3. AVX-512 embodiment. On x86, the same lane arrays are optionally evaluated with AVX-512F/BW/VL compare, add, mask, shift, and mask-extract operations. Representative work includes vector computation of end sectors, adjacent-lane contiguity checks, equality-class masks for op and flag groups, lane-wise overflow tests against conservative queue-limit bounds, and kmov extraction of the MUST-SPLIT bitmask. The vector code does not emit requests, does not update accounting, and does not override a scalar merge predicate; it only produces one-sided evidence. If the vector backend is unavailable or disallowed, the identical lane format is executed with scalar integer and bitmask operations. 4. Scalar revalidation as sole authority. After prefiltering, the kernel walks the same descriptors in original order and re-invokes the unchanged scalar merge, queue-limit, bounce, DMA-boundary, integrity, and request-emission logic. MUST-SPLIT bits may prune impossible adjacency tests and shorten candidate runs, but the scalar logic remains the only authority for extending a segment, creating a new scatterlist element, splitting a bio, updating nr_phys_segments, and handling any side effect. The emitted result must therefore match baseline semantics exactly. 5. SIMD legality and profitability gate. The AVX-512 backend is entered only when all of the following are true: the global and per-queue policy is not off; the CPU advertises the required AVX-512 feature subset; may_use_simd() is true; the caller is in task context with IRQs enabled and not in NMI, hardirq, or softirq context; the path is not holding a lock class that forbids kernel FPU use; and the workload passes a conservative profitability filter. The profitability filter defaults to auto mode and requires at least two full windows of bio_vec entries, a queue type where metadata cost is material, and a per-CPU exponentially weighted moving average showing recent win over scalar. The implementation exposes a hard global boot parameter and a per-queue sysfs knob with off, auto, and force modes. Auto mode also disables itself for a cooling period after observed frequency-capacity drop or negative cycle delta on the current CPU model. 6. Optional identity-writeback zero substitution. A second embodiment reuses the same proof-only discipline for all-zero data-writeback spans, but only under a narrower contract. The integration point is a filesystem data-writeback hook, such as an iomap-style writepages stage or filesystem-specific equivalent, that runs on a locked folio or extent range before bio construction. The filesystem must provide an identity_writeback predicate for the exact span to be substituted. That predicate returns false unless the span is regular file data, fully backed by allocated blocks, fully covered by the current writeback operation, and guaranteed byte-preserving from folio bytes to lower block I/O. Explicit disqualifiers include metadata, journal, log, or allocator writes; fscrypt; fsverity; inline encryption; compression; reflink, copy-on-write, or dedupe; data-transforming checksums; dm-crypt; dm-integrity; RAID personalities that transform or reconstruct data; verity targets; and any other target or filesystem mode that can alter bytes or allocation semantics. 7. Stable-bytes and effective-target gates. The zero scan is valid only while the folio remains locked and under writeback control across scan through bio-build decision, with the filesystem's extent-mapping stability mechanism held for the covered range. If truncate, invalidation, partial-writeback change, redirty, or remap can occur for the span in that window, substitution is rejected. Even if the bytes are stably zero, the block layer must additionally prove effective-target support for REQ_OP_WRITE_ZEROES | REQ_NOUNMAP through the entire stack. The invention therefore requires an effective_write_zeroes capability descriptor propagated from leaves through dm and md. Required fields include nounmap_preserves_allocation, strict_completion, no_emulated_discard, max_write_zeroes_sectors, max_sectors, alignment, granularity, and boundary constraints. Substitution is allowed only when all layers preserve NOUNMAP allocation semantics, the range satisfies every size and alignment limit, and strict_completion guarantees full-range success-or-fail with retry and errno classification equivalent to a normal write for the covered span. Otherwise the kernel emits an ordinary WRITE bio and does not substitute.

2.5 Implementation Plan

1. Add a blk_bew_window helper and lane format shared by scalar and AVX-512 backends. The helper is invoked from the same submit context that already performs block-layer split and segment calculations, after the bio is fully built but before lower-layer request emission. Add a request-queue limits generation counter qlim_gen that increments when queue-limit state relevant to mergeability changes or when the queue is reconfigured, and store that value in each window cookie. 2. Implement the scalar BEW prefilter first. It computes only MUST-SPLIT proofs using copied lane metadata and never emits a merge decision. Wire the unchanged scalar merge and queue-limit helpers immediately after it so the result is identical to baseline. Add tracepoints and counters for windows processed, boundaries proven, cookie-invalidated fallbacks, and full fallbacks. 3. Add the x86 AVX-512 backend under a dedicated configuration option. The entry predicate is: policy not off, required AVX-512 feature subset present, may_use_simd() true, task context with IRQs enabled, no forbidden lock held, and profitability filter passed. Bracket only the vector compare region with kernel_fpu_begin/end. Provide a global boot parameter blk.avx512_prefilter=off|auto|force and a per-queue sysfs control queue/avx512_prefilter with the same modes. Auto mode initially enables only for allowlisted CPU models and only when bio_vec count exceeds a conservative threshold such as 2N; it disables itself temporarily after measured regression. 4. Keep physical and DMA-derived proofs optional and authoritative only. If the current submit path already has a reliable bounce-resolved page identity or DMA-relevant physical token, copy it into the window and allow MUST-SPLIT proofs from it. If not, leave those boundaries unknown and rely on scalar revalidation. This avoids making speculative physical assumptions from raw page metadata. 5. Introduce the optional identity-writeback embodiment as an explicit opt-in API, not a generic heuristic. Add a filesystem hook such as identity_writeback_ok(inode, folio, start, len) that returns true only for exact data spans whose bytes are known to reach the block layer unchanged. Initial enablement should be limited to filesystems and modes that can make that proof precisely, for example regular data extents in an opt-in non-transforming writeback path, and should exclude metadata, journal, fscrypt, fsverity, compression, reflink, COW, dedupe, inline encryption, and any data-transforming checksum mode. 6. Add blk_effective_write_zeroes_caps and require stacking layers to propagate it conservatively. dm and md may set nounmap_preserves_allocation and strict_completion only when every child path and target behavior preserve REQ_NOUNMAP allocation semantics and full-range WRITE ZEROES completion semantics. If any target can emulate with discard, deallocate blocks, transform data, or return non-equivalent completion behavior, the effective capability is cleared and substitution is unavailable. 7. Integrate the zero scan at the filesystem writeback hook while the folio lock, writeback state, and extent-map stability lock for the exact span are all held across scan through bio-build decision. The default scan may be scalar; an AVX-512 zero scan may be enabled only under the same legality gate as the segment prefilter and only when the filesystem hook runs in a permitted task context. If stability is lost or the exact span changes, abort substitution and emit a normal write. 8. Route completion of substituted WRITE ZEROES bios through the same writeback accounting object used by ordinary writes. Because substitution is gated on strict_completion, the completion path can preserve the same success, retry, and error propagation model for the span. If strict completion cannot be guaranteed by the effective target, substitution is not attempted.

2.6 Validation Plan

1. Differential correctness for segment mapping. Generate randomized bios spanning contiguous and fragmented bio_vec sequences, mixed offsets, zone and integrity cases, highmem and bounce candidates, and varying queue limits. Compare baseline scalar output versus BEW scalar-prefilter output versus AVX-512-prefilter output. The emitted boundaries, scatterlist layout, nr_phys_segments, bounce decisions, DMA-boundary handling, and error behavior must match exactly. 2. Race and invalidation tests. Inject qlim_gen changes between snapshot and revalidate, mutate merge-relevant bio fields through fault-injection hooks before submission, and verify that cookie mismatch forces immediate fallback to the pure baseline path. Validate that the window never survives beyond the submit call and cannot be dereferenced after the bio is completed. 3. Legality tests for SIMD use. Exercise task context, IRQ-disabled context, softirq, hardirq, NMI, direct reclaim, and generic writeback worker paths. Confirm that AVX-512 runs only when may_use_simd() and policy gates pass, and that all excluded contexts automatically use the scalar backend without functional change. 4. Anti-regression performance tests. Benchmark fragmented-read and fragmented-write submission on AVX-512-capable Xeon systems, on CPUs known to show AVX-512 downclock sensitivity, and on non-AVX-512 systems. Validate that off mode is neutral, auto mode only enables above the configured vector-worth threshold, and the cooling heuristic disables the vector path after measured regression. 5. Positive and negative identity-writeback tests. Positive cases should include only explicitly supported regular-data writeback paths on non-transforming filesystems and leaf or stacked targets that advertise effective strict WRITE ZEROES with NOUNMAP preservation. Negative cases must include metadata and journal writes, fscrypt, fsverity, inline encryption, compression, reflink/COW, dedupe, dm-crypt, dm-integrity, md RAID personalities that transform data, thin provisioning that can deallocate on zero, and any target lacking strict completion or NOUNMAP guarantees. Every negative case must fall back to ordinary WRITE. 6. Stable-bytes race tests. Force truncate, invalidate, remap, partial writeback, and redirty events against candidate all-zero folios while the feature is enabled. Verify that any loss of span stability aborts substitution before submission. Confirm that only the exact covered span may be substituted and that partial-last-block or partial-folio edge cases fall back unless fully proven safe. 7. WRITE ZEROES semantic tests. Fault-inject unsupported, emulated, partial, and transport-error behaviors in leaf and stacked targets. Confirm that substitution is rejected unless strict_completion and no_emulated_discard are true. Where substitution is allowed, verify that writeback error accounting, retry classification, and completion semantics match ordinary WRITE for the same span. 8. Prior-art differentiation package. Document that the fast path never proves mergeability, never bypasses the scalar Linux authority, and never substitutes zero-writeback without explicit identity and effective-target proofs. This establishes the novelty boundary against generic SIMD scans and existing zeroout pathways.

4. Risks & Mitigations

  • Risk: a copied metadata window diverges from live queue limits or bio state before submission. Mitigation: include qlim_gen and bio identity in the window cookie; invalidate on any mismatch and fall back to the pure scalar path.
  • Risk: AVX-512 causes latency regressions from frequency throttling or power cost. Mitigation: expose hard off control, default to auto mode, require conservative size thresholds and positive EWMA win, and cool off automatically after detected regression.
  • Risk: illegal kernel SIMD use in IRQ, reclaim, or lock-sensitive contexts. Mitigation: enter the vector backend only when may_use_simd() and execution-context gates pass and only around a tiny kernel_fpu_begin/end region; otherwise use the scalar backend.
  • Risk: false assumptions about bounce or DMA-relevant physical identity. Mitigation: allow physical or DMA-based MUST-SPLIT proofs only when the token is already authoritative in the current path; otherwise mark the boundary unknown.
  • Risk: all-zero writeback substitution changes semantics on transformed filesystems or targets. Mitigation: require an explicit filesystem identity-writeback predicate and enumerate hard disqualifiers including encryption, compression, checksumming transforms, reflink/COW, dm-crypt, dm-integrity, RAID transforms, and verity.
  • Risk: REQ_NOUNMAP is not actually preserved through stacking layers. Mitigation: require a conservatively propagated effective_write_zeroes capability and disable substitution unless every layer asserts nounmap_preserves_allocation.
  • Risk: WRITE ZEROES completion or retry behavior differs from ordinary WRITE. Mitigation: require strict_completion and no_emulated_discard in the effective capability; without those guarantees, do not substitute.
  • Risk: scanned folio bytes change between zero detection and I/O submission. Mitigation: hold folio lock, writeback state, and filesystem extent-map stability across scan through bio-build decision, and abort substitution on truncate, invalidate, redirty, remap, or partial-span mismatch.
