2. Technical Invention Disclosure

2.1 Problem Statement

Linux device backends attached to KVM guests, especially VFIO/iommufd and vhost/vDPA paths that own virtqueue pages, ring pages, doorbell pages, and transient DMA buffers, often retain long-lived page pins and IOMMU mappings after the guest CPU has effectively stopped touching the corresponding guest-physical range. Existing mmu_notifier and iommufd invalidation mechanisms protect correctness only after host mappings actually change, so they are reactive teardown signals rather than proactive, range-scoped evidence that a still-valid mapping is CPU-cold. x86 KVM already encounters EPT leaf state in TDP MMU paths and can sometimes observe accessed-bit transitions, but that telemetry is private, mode-dependent, and not bound to a driver-consumable identity or ordering contract. Without a correctness-safe bridge, Linux device drivers either keep pins and maps too long, increasing unreclaimable memory and IOMMU residency, or risk reclaim decisions based on stale observations that race with memslot moves, invalidations, or in-flight DMA.

2.2 Prior-Art Gap

Prior art covers reactive invalidation sequencing, EPT accessed/dirty harvesting for KVM's own MMU policy, and backend-specific DMA teardown logic, but it does not define a kernel-internal ABI that lets a backend register a precise guest-physical extent and later query a conservative CPU-access epoch that is machine-checkably bound to the same live struct kvm, memslot identity, memslot generation, and invalidation epoch. The missing element is an auditable concurrency contract: how the extent is registered, when hints are suppressed, what locks and SRCU domains protect readers and writers, how invalidate_start and invalidate_end are serialized through the canonical mmu_invalidate_seq, how fast paths avoid allocation and sleeping, and why no consumer can reclaim based on a stale extent after a slot move, split, unmap, dirty-log transition, or PFN replacement. That precise contract, not a generic heuristic, is the inventive gap.

2.3 Proposed Invention

KVM x86 exposes an internal service called an EPT epoch domain. A KVM-attached DMA owner registers one or more guest-physical extents that are page-aligned, wholly contained in a single live memslot, and annotated with the access class it cares about, typically CPU read/write touches that imply the guest is still actively driving a ring or buffer. For each registered extent, KVM stores a fixed-size descriptor keyed by struct kvm pointer, memslot id, memslot generation, base_gfn, npages, and access_mask. When existing TDP MMU code paths already inspect reliable EPT leaf state and observe a relevant CPU access, KVM advances a per-VM monotonic cpu_access_epoch and publishes the new epoch into the matching extent descriptor together with a snapshot of the current even mmu_invalidate_seq. Consumers may read that snapshot, but they are forbidden to shorten a pin or unmap merely because the epoch looks old. They may act only after they themselves stop and drain DMA for that queue or buffer, then immediately revalidate mmu_invalidate_seq, memslot generation, and PFN identity before unpinning or unmapping. If any check fails, the backend keeps the existing pin or map and falls back to baseline behavior.

Add suppression hooks for dirty logging, permission-forced write protection, unsupported tracking modes, and memslot generation changes. 6. Integrate backend-side DMA-quiesce plus revalidation logic before existing unpin or unmap calls. 7. Add tracepoints and internal stats for kernel developers only while keeping all telemetry non-user-visible in production paths.

2.6 Validation Plan

Correctness validation focuses on race freedom and fail-closed behavior: concurrent memslot move during candidate reclaim, invalidate_start between query and DMA drain, PFN replacement during long DMA fence, dirty-log enable while descriptors exist, and nested or unsupported modes where publication must remain UNKNOWN. Functional validation checks that stale hints never cause an unpin or unmap that baseline code would forbid. Performance validation checks that the registration and query path do not materially increase MMU lock hold time, notifier latency, or device completion latency. All tests compare against a baseline where the feature is compiled in but disabled, verifying zero behavior change except when the backend can prove DMA quiescence and the sequence-generation checks pass.

  2. The method of claim 1 wherein registration is rejected unless the extent is page-aligned, wholly contained within a single live memslot for the supplied generation, and requests only access classes observable without introducing new EPT faults or scanning paths, and wherein exhaustion of a preallocated descriptor budget returns an error that disables hinting for the requester without altering baseline mapping retention semantics.
  3. The method of claim 1 wherein invalidate_start and invalidate_end are represented by odd and even publications of the canonical mmu_invalidate_seq using release ordering, and wherein the querying backend uses acquire ordering on both an initial and a final sequence load to reject any snapshot that races with a concurrent memslot invalidation, split, move, unmap, or PFN replacement.
  4. The method of claim 1 wherein KVM suppresses publication or marks the extent non-actionable when dirty logging, access tracking without reliable hardware A/D observation, PFN write-protect, or unsupported execute-tracking modes make observed access state ambiguous for reclaim purposes.
  5. A non-user-visible KVM x86 subsystem comprising a per-VM extent table with fixed descriptors allocated outside notifier context, monotonic cpu_access_epoch state, and query helpers callable by VFIO, iommufd, vhost, or vDPA backends, the subsystem being configured so that notifier and MMU observation fast paths execute without sleeping or allocation and so that all failures fall back to retaining existing pins and IOMMU mappings.

4. Risks & Mitigations

  • Risk: a backend misinterprets a stale snapshot as permission to reclaim. Mitigation: mandate the DMA-quiesce plus seq, generation, and PFN revalidation protocol and document that the hint never bypasses existing unpin or unmap legality checks.
  • Risk: notifier fast paths regress latency through allocation or blocking. Mitigation: preallocate descriptors in sleepable registration context, hard-cap per-VM metadata, and fail registration with -ENOSPC rather than allocating in invalidate_start or invalidate_end.
  • Risk: dirty logging, access tracking, or write-protect modes distort access semantics. Mitigation: explicit suppression flags disable actionability whenever those modes are active or ambiguous.
  • Risk: NUMA or multi-socket contention on shared metadata. Mitigation: only a small per-extent descriptor and a per-VM epoch are touched in fast paths, and removing bucket counters or global aggregates from the core design avoids cacheline ping-pong across Xeon sockets.
  • Risk: telemetry could create a cross-tenant timing signal. Mitigation: keep all state per-VM and kernel-internal, expose only coarse monotonic epochs to in-kernel consumers, and exclude global aggregation or user ABI surfaces.
