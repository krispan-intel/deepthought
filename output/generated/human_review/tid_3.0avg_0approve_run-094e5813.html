2. Technical Invention Disclosure

2.1 Problem Statement

Linux KVM on x86 must decide, under heavy concurrency, which EPT-derived translations must be invalidated before a vCPU re-enters guest mode. The hard case is nested VMX with vmcs12 to vmcs02 translation, EPTP switching via nested state transitions or VMFUNC EPTP lists, memslot invalidation, MMU-notifier unmap or protect, SPTE write-protect or zap, large-page split or merge, private/shared memory conversion, and vCPU migration across pCPUs. Existing request-bit and generation-counter mechanisms tell a vCPU that some flush happened, but they do not encode which EPT root instance changed, the least-sufficient INVEPT class for the currently exposed EPTP set, or how to distinguish a recycled root from an older incarnation. Without that missing identity-and-ack boundary, Linux KVM must either over-flush with conservative all-context invalidations or rely on under-specified reasoning about generations, root reuse, and remote-vCPU progress. The unsolved problem is to give each producer event a precise EPT invalidation meaning, publish it with ABA-proof root identity, guarantee that safety-critical remote vCPUs reach a host checkpoint in bounded time, and let the consumer ACK only after the CPU that will actually execute VM-entry has executed the required INVEPT.

2.2 Prior-Art Gap

Current Linux KVM already has per-vCPU request bits, mmu_invalidate_seq or TLB-flush generations, and remote kick or IPI style flush assistance. Those mechanisms are useful baselines, but they primarily answer whether a vCPU must re-check state; they do not define a first-class ticket stream that binds a specific EPT root incarnation, a minimum INVEPT class, and an ACK rule that advances only after the actual invalidation instruction has executed on the CPU about to run the guest. That specific combination is not provided by broad generation counters alone.

2.3 Proposed Invention

1. Scope and carve-outs The invention governs only EPT-derived translation invalidation in Linux KVM on VMX hardware. The flush lattice is {none, single-context, all-context}. Orthogonal VPID, INVVPID, or PCID handling remains on existing KVM request paths. If a producer event requires both EPT invalidation and non-EPT invalidation, it emits the ledger ticket described here and also raises the existing non-EPT request; the consumer completes all pending INVEPT work before guest entry and then lets existing KVM logic perform any required INVVPID or VPID maintenance. 2. Core data structures - ept_root_token = { u64 root_id, u64 eptp, u64 last_seq, refcount_t refs, rcu_head rcu }. root_id is allocated from a per-VM atomic64 next_root_id counter when a shadow or nested EPT root becomes publishable. root_id never repeats within the VM lifetime. If next_root_id would wrap, the VM is forced into conservative VM-wide all-context mode and root recycling is disabled until VM teardown, making ABA impossible in practice. - invept_ticket = { u64 seq, u64 root_id_or_0, u64 eptp_or_0, u8 scope, u8 min_class, u16 cause_bits, u32 reserved }. scope is EPTP_GEN or VM_WIDE. min_class is NONE, SINGLE_CONTEXT, or ALL_CONTEXT. - invept_ledger = { raw_spinlock_t pub_lock, u64 tail_seq, u64 vmwide_last_seq, u32 ring_mask, invept_ticket ring[N], atomic64 next_root_id }. N is a power of two and is provisioned as N >= 2 * Bmax * Texit_max, where Bmax is the audited maximum ticket-publication burst rate for one VM and Texit_max is the bounded remote-exit window promised for safety-critical kicks. - vcpu_flush_state = { u64 vmwide_acked_seq, hot_root_cache[H] = { root_id, acked_seq }, u64 hotset_seq_seen }. The hot_root_cache covers the current effective EPTP and every EPTP-list entry that can become active without a host round-trip. 3. ABA-proofing and lifetime rules A ticket never carries a raw root pointer as its identity; it carries root_id and the contemporaneous EPTP value. The root object remains alive until refs drop to zero and an RCU grace period completes. Consumers reading the ring do not dereference a freed root through old tickets; they compare numeric root_id values and use the current hot-set snapshot to decide whether a flush is needed. Reusing the same EPTP physical address for a later root is harmless because the later root gets a new root_id. That root_id lifetime coupling is the non-reusable EPTP-generation identity. 4. Flush-decision lattice and exhaustive cause mapping Cause | Scope | Minimum class | Remote-exit requirement SPTE accessed or dirty aging clear within one root | per EPTP-generation | single-context | lazy allowed SPTE permission downgrade, write-protect for dirty logging, zapping a leaf, or large-page split or merge within one root | per EPTP-generation | single-context | safety-critical kick plus escalation Memslot change where affected roots are precisely enumerable | one ticket per affected root | single-context | safety-critical kick plus escalation Memslot change with unknown or large affected-root set, slot move, GPA alias change, or shared or private attribute transition | VM-wide | all-context | safety-critical kick plus escalation MMU-notifier invalidate_range, unmap, protect, or reclaim that may race many roots | VM-wide | all-context | safety-critical kick plus escalation Nested vmcs12 or host shadow change that introduces a brand-new never-entered root and does not modify any previously exposed root | per new root | none | none before first exposure Nested EPTP switch, EPTP-list rewrite, or modification of a previously exposed root with precise root identity | per affected root | single-context | safety-critical if guest could resume with stale entries Nested state transition where the precise affected root set is lost, or root retirement, free, or recycle | VM-wide | all-context | safety-critical kick plus escalation Consumer-detected sequence gap or overwrite ambiguity | VM-wide synthesized ticket | all-context | handled immediately on the entry CPU No producer in this invention requests an INVEPT class outside the lattice. Anything that additionally needs INVVPID, VPID, or PCID handling is explicitly carved out and serviced by existing KVM mechanisms in addition to, not instead of, the ledger. 5. Publish protocol and TSX scope Every producer funnels through ledger_publish(). Publication is serialized by pub_lock, but if RTM is enabled by kernel policy, cpu_has_rtm is true, no microcode mitigation forbids RTM, and the critical section is predicted to touch only the tail_seq cache line and one ring-slot cache line, the code may attempt xbegin or xend lock elision. RTM is limited to at most two retries. Any abort, disabled TSX state, contention hint, interrupt, or capacity overflow causes immediate fallback to raw_spin_lock on the same pub_lock. Both paths write identical bytes in identical order: fill ticket payload, optionally update vmwide_last_seq, optionally update root.last_seq, then store-release tail_seq as the final publication step. No INVEPT, INVVPID, VMREAD, VMWRITE, MMIO, sleeping lock, or MMU-notifier callback work is executed inside RTM. Correctness never depends on RTM commit; RTM is only a micro-optimization for host-memory metadata publication. 6. Remote-vCPU forward progress For any safety-critical cause, the publisher sets KVM_REQ_TLB_FLUSH on every vCPU that currently exposes the affected root in its hot set, or on all runnable vCPUs of the VM for VM-wide tickets, then issues kvm_vcpu_kick(). Kick alone is not treated as a sufficient bounded-exit guarantee. The design therefore defines an escalation threshold Texit_max: if a target vCPU remains guest-running beyond that window or if ring headroom approaches overwrite, KVM reuses its existing forced-exit machinery for remote TLB work, such as the remote IPI or reschedule-assisted path already used to pull a guest back to the host. The invention does not require a new interrupt type; it requires that safety-critical invalidations cannot rely forever on polite kicks. 7. Consumer execution point, active EPTP hot set, and ACK rule The consumer runs in the x86 KVM pre-VM-entry checkpoint centered on vcpu_enter_guest(), after request bits are observed and before the vendor VMX run hook reaches VMLAUNCH or VMRESUME. Preemption is disabled across the decision plus INVEPT execution window, and no ledger lock is held while executing INVEPT, preventing notifier-path lock inversion. INVEPT is never executed inside an RTM transaction. The consumer first snapshots the active EPTP hot set under seqcount-or-RCU rules: (a) the current effective vmcs02 EPTP, (b) every EPTP-list entry exposed for VMFUNC-based switching, and (c) both old and new roots during a nested vmcs12-to-vmcs02 transition until publication of the new hotset_seq completes. If hotset_seq changes during the scan, the snapshot is retried. ACK rules are explicit: - vmwide_acked_seq may advance only after the current CPU has executed any required INVEPT all-context that covers all VM-wide tickets up to the observed tail_seq. - A hot-root acked_seq may advance only after the current CPU has executed the required INVEPT for that root up to root.last_seq. - If the consumer detects seq mismatch in a ring slot, sees tail_seq minus acked_seq greater than N, or cannot reconstruct a continuous interval, it issues one INVEPT all-context and then store-release sets vmwide_acked_seq to the observed tail_seq and every hot-root acked_seq to its observed root.last_seq. This is safe because all-context invalidates all EPT-derived translations on the current logical processor, which is exactly the state that matters before the next guest entry on that CPU. - If a root is newly entering the hot set and has no ACK record or root.last_seq greater than acked_seq, the consumer performs at least one single-context INVEPT before exposing that root for guest execution unless a covering all-context has already executed. 8. Overwrite semantics The ring is overwrite-permitted, not overwrite-hidden. Producers may wrap ring slots once tail_seq advances, but any consumer that falls behind beyond N entries must observe that fact and conservatively collapse to all-context. Therefore no lost ticket can cause under-flush: information loss degrades only precision, never safety. 9. Same-vCPU EPTP switching Because the hot set includes the current effective EPTP plus guest-exposed EPTP-list entries, a vCPU cannot miss a per-root ticket simply because the root is not the currently active vmcs02 EPTP at scan time. Roots that are not in the hot set are safe to defer because they cannot become guest-selectable until a later host checkpoint republishes the hot set, at which point the root.last_seq versus acked_seq test forces the missing flush before exposure.

2.4 Architecture

Per-VM publish side

 MMU notifier / memslot / SPTE / nested-root event
                |
                v
        classify cause -> {scope, min_class, safety}
                |
         [RTM optional, max 2 retries]
                | abort or disabled
                +--------------------> raw_spin_lock(pub_lock)
                |
                v
  write ring[tail+1], update root.last_seq or vmwide_last_seq,
  store-release new tail_seq, unlock or commit
                |
                +--> if safety-critical: set KVM_REQ_TLB_FLUSH and kick or escalate remote vCPUs

Per-vCPU consume side before VM-entry

   vcpu_enter_guest() checkpoint
                |
                v
    snapshot hot set {effective EPTP + exposed EPTP-list roots}
                |
                v
   detect gap? ---- yes ---> INVEPT all-context ---> store-release ACKs ---> VM-entry
      |
      no
      v
  for each hot root with root.last_seq > acked_seq:
      scan continuous interval, join class for that root
      -> INVEPT single-context(root.eptp)
   if vmwide_last_seq > vmwide_acked_seq:
      -> INVEPT all-context
                |
                v
         store-release ACKs only after INVEPT retires

Key invariants
1. root_id is monotonic 64-bit and never reused within a VM lifetime.
2. tail_seq publication order is contiguous and serialized by pub_lock or its RTM elision.
3. INVEPT never runs inside RTM and never while holding pub_lock.
4. Gaps collapse to all-context; precision may be lost, safety may not.
5. A root cannot become guest-visible until its pending flush debt is covered by a root-specific ACK or a VM-wide ACK.

2.5 Implementation Plan

1. Add ept_root_token allocation to shadow and nested EPT root creation paths, tie root_id to root lifetime, and switch free paths to refcount plus call_rcu quarantine. 2. Add a ledger_publish() helper in the x86 KVM MMU or VMX layer with pub_lock, ring, cause enum, and a static-key-controlled RTM fast path. Set retry budget to 2 and make the spinlock fallback byte-for-byte identical in field ordering and memory barriers. 3. Instrument producer sites: SPTE permission downgrade or zap, large-page split or merge, memslot invalidate or move, MMU-notifier invalidate_range or unmap or protect, nested EPTP root install or remove, EPTP-list rewrite, private or shared memory transition, and root retirement. 4. Maintain per-root last_seq and per-VM vmwide_last_seq with store-release publication; compute N from audited Bmax and Texit_max, and expose counters for gap collapses, kick escalations, and RTM abort reasons. 5. Extend per-vCPU x86 state with hot_root_cache, vmwide_acked_seq, and hotset_seq. Populate the hot set from the current effective vmcs02 EPTP plus every guest-exposed EPTP-list entry; on nested transition publish old and new roots until the new hotset_seq becomes visible. 6. In vcpu_enter_guest() request handling, before the vendor run hook and with preemption disabled, snapshot the hot set, drain ledger obligations, execute single-context or all-context INVEPT as required, then store-release ACKs. Never execute INVEPT while holding pub_lock or inside RTM. 7. For safety-critical tickets, set KVM_REQ_TLB_FLUSH, kick affected vCPUs, and if request age exceeds Texit_max or ring headroom shrinks below a configured watermark, invoke the existing KVM forced-exit remote-flush path. 8. Preserve orthogonal INVVPID, VPID, and PCID handling by leaving current KVM request bits in place; when both planes are pending, run the INVEPT ledger step first and then existing non-EPT invalidation logic before VMLAUNCH or VMRESUME.

1. Unit-test the cause classifier so every producer event maps to the documented lattice and scope, with negative tests that no event silently falls outside {none, single-context, all-context}. 2. Model-check or litmus-test the publish and consume memory ordering so producer payload writes happen-before tail_seq or root.last_seq visibility, and ACK stores happen after INVEPT retirement. 3. Stress nested VMX with VMFUNC EPTP switching, memslot churn, MMU-notifier unmap or protect storms, large-page split or merge, dirty logging, private or shared memory transitions, and vCPU migration; verify that no guest resumes with a root whose root.last_seq exceeds its acked_seq unless covered by a later vmwide_acked_seq. 4. Force ring overwrite and artificial consumer lag to confirm deterministic gap detection and safe collapse to INVEPT all-context. 5. Run with TSX disabled in BIOS or kernel, TSX enabled with frequent abort injection, and TSX fully available; verify identical externally visible behavior and only latency differences. 6. Validate remote-progress escalation by stalling guest vCPUs, measuring Texit_max, and confirming that safety-critical invalidations trigger the existing forced-exit path before ring headroom exhaustion. 7. Benchmark against baseline KVM using nested guests, dirty logging, and mixed private or shared memory to show fewer unnecessary all-context invalidations, lower re-entry latency, and no correctness regressions.

4. Risks & Mitigations

  • TSX may be disabled, mitigated, or unstable on production hosts: make RTM default-off behind a static key, cap retries at two, and preserve identical spinlock semantics so the invention retains value without TSX.
  • Producer bursts may exceed the expected ring budget: size N from audited Bmax times Texit_max, surface telemetry for gap collapses, and treat any observed gap as an immediate all-context flush rather than a silent miss.
  • Polite kicks may not force a guest out quickly enough for safety-critical invalidations: define Texit_max and escalate to the existing KVM forced-exit remote-flush path when the bound is exceeded.
  • EPT root reuse bugs could reintroduce ABA: bind a monotonic root_id to root lifetime, never reuse it within the VM lifetime, and quarantine free through refcount plus RCU.
  • Hot-set growth from VMFUNC EPTP lists may exceed the fast-path cache: bound H for the common case and conservatively collapse to all-context when the guest exposes more roots than the cache tracks.
  • Interaction with INVVPID, VPID, or PCID maintenance can create ordering ambiguity: keep those invalidations on existing KVM request paths and serialize them with the ledger drain before VM-entry.
  • Incorrect memory ordering could allow stale ACK visibility: use store-release publication of tail_seq and ACK fields, load-acquire consumption, and litmus validation of the publish and consume protocol.
