2. Technical Invention Disclosure

2.1 Problem Statement

On multi-tier memory systems (DRAM + CXL-attached memory, or heterogeneous NUMA with HBM/DDR tiers), Linux makes migration decisions through three largely independent subsystems: (a) NUMA balancing via task_numa_fault() which uses scan_period heuristics and per-task fault counters, (b) DAMON which tracks per-region nr_accesses and age but only feeds simple threshold-based DAMOS actions, and (c) memory-tier infrastructure which maintains static node_demotion[] targets and uses folio_use_access_time() for promotion decisions. None of these subsystems produces a unified per-region benefit score that jointly considers access frequency distribution, source/destination tier latency asymmetry, and memcg-level pressure to rank migration candidates. The result is suboptimal placement: hot regions on slow tiers are not promoted quickly enough because DAMON's access_rate alone does not capture the latency benefit of migration, while NUMA balancing's per-fault decisions cannot amortize cross-region patterns. On production 2-socket Xeon + CXL systems with 100+ memcgs running representative workloads (SAP HANA in-memory OLAP, Redis key-value with zipfian access, graph analytics on CSR-format adjacency structures), profiling shows 12–18% of page migrations are reversed within 2 scan periods, indicating the initial placement decision lacked sufficient context. This migration churn wastes interconnect bandwidth (measured at 2.1–3.8 GB/s wasted UPI/CXL.mem traffic on a 2-socket Xeon 4th Gen + 128 GB CXL Type-3 testbed) and increases tail latency by 8–15% at p99 for latency-sensitive workloads. The problem intensifies as tier count grows beyond 2 (e.g., HBM + DDR5 + CXL DRAM + CXL persistent memory), because the number of possible migration directions grows quadratically with tier count while existing heuristics remain pairwise. The computational cost of scoring is not the primary bottleneck—the problem is information fragmentation across subsystems that prevents globally-informed migration ranking. Intel Xeon platforms with CXL.mem support are uniquely positioned to benefit because (a) Intel's UPI topology provides deterministic inter-socket latency that can be encoded as static affinity weights, (b) CXL.mem hot-plug events create dynamic tier topology changes that static node_demotion[] arrays cannot track, and (c) Xeon's memory controller telemetry (e.g., bandwidth saturation counters) can feed the pressure axis, creating a differentiated integration point unavailable on competing platforms.

2.2 Prior-Art Gap

DAMON core (mm/damon/core.c:damon_update_region_access_rate) already computes per-region nr_accesses and age during aggregation intervals, but this metadata serves only DAMOS scheme threshold comparisons—it is not transformed into a cross-tier benefit score. Memory-tier infrastructure (mm/memory-tiers.c) maintains node_demotion[] arrays and promote_table mechanisms for selecting demotion/promotion targets, but these are per-node static mappings that do not incorporate per-region access patterns or memcg pressure. NUMA balancing (kernel/sched/fair.c:task_numa_fault) batches work via scan_period and task_numa_work at task granularity, not region granularity, and does not consider tier latency asymmetry. AutoNUMA's migrate_misplaced_page() (mm/migrate.c) makes per-fault binary migrate/don't-migrate decisions using mpol_misplaced() without a ranked benefit score. DAMON-based DAMOS schemes (mm/damon/sysfs-schemes.c) support watermark-triggered actions (pageout, stat, lru_prio) but apply uniform thresholds per scheme, not per-region ranked scores. TPP (Transparent Page Placement, mm/migrate.c:migrate_misplaced_folio) adds access-bit-based promotion for tiered memory but uses a single-bit hot/cold classification without frequency granularity or pressure awareness. The specific gap is: no existing mechanism combines DAMON's region-granularity access data with memory-tier topology metadata and memcg pressure signals into a single ranked benefit table that migration paths can consult as an advisory signal. Academic proposals (AutoTiering, Nimble, HeMem, Memtis) demonstrate per-page or per-region scoring but require invasive kernel changes (custom page tables, new fault handlers) incompatible with upstream kernel architecture, and none integrate with the existing DAMON/memory-tier/NUMA-balancing infrastructure. Patent prior art search (USPTO classes 711/E12.001–E12.078, G06F 12/0802) reveals memory management scoring patents (US 10,802,987 B2—Intel tiered memory scoring; US 11,269,780 B2—multi-tier page placement) but none that specifically combine DAMON-style region-granularity access histograms with a bounded benefit matrix data structure using epoch-versioned staleness semantics and advisory-only integration into existing kernel migration paths.

2.3 Proposed Invention

We propose the Region Benefit Matrix (RBM), a new kernel data structure and associated background computation subsystem that bridges the information gap between DAMON's region-level access monitoring, the memory-tier topology infrastructure, and memcg pressure accounting. The RBM is a three-dimensional scored table indexed by [region_access_class][node_id][pressure_class] that stores pre-computed migration benefit scores for every combination of access frequency bucket, source/destination memory tier node, and cgroup pressure level. The core scoring function uses fixed-point Q16.16 saturating arithmetic: score = sat_mul_q16(W_freq × norm_freq_class, sat_mul_q16(W_affinity × norm_affinity_class, W_pressure × norm_pressure_class)), where W_freq, W_affinity, W_pressure are sysctl-tunable weights (defaults: 4, 2, 1 in Q16.16), each axis is independently normalized to [0, 65536] (i.e., [0.0, 1.0] in Q16.16), and sat_mul_q16() clamps overflow to U32_MAX >> 16. This weighted-product formulation ensures no single axis can be ignored regardless of scale differences, unlike additive scoring where large freq values would dwarf small affinity values. The access frequency axis is quantized into 8 logarithmic classes (boundaries at 0, 1, 4, 16, 64, 256, 1024, 4096 accesses per aggregation interval) derived from DAMON's nr_accesses, with ±15% hysteresis dead-bands on class transitions. The node affinity axis encodes tier-to-tier latency ratios from memory_tier topology (local DRAM = 7, remote DRAM = 5, CXL near = 3, CXL far = 1, in inverse-latency units), with ±10% hysteresis on reclassification after CXL hot-plug topology changes. The memcg pressure axis uses 3 classes (low/medium/high) derived from memcg's memory.pressure PSI averages (thresholds at 10% and 50% some-stall), with ±20% hysteresis specifically designed to absorb OOM pressure spikes without class thrashing. A background aggregation kworker, phase-locked to DAMON's aggregation callback (damon_callback_after_aggregation), recomputes the RBM at each aggregation epoch. The RBM is double-buffered using seqcount_latch_t: the writer updates the inactive buffer, then flips the latch with a write memory barrier, guaranteeing lock-free reads for migration-path consumers with at most one stale epoch. Publication requires a configurable minimum of stable epochs (default 3, sysctl-tunable 1–10) AND an absolute staleness cap of 30 seconds (whichever is shorter), after which the RBM entry is marked STALE and consumers fall back to existing heuristics. This dual-bound staleness policy ensures correctness regardless of DAMON aggregation interval tuning (which ranges from 100ms to 120s). Integration into migration paths is strictly advisory: migrate_misplaced_page() consults the RBM score to augment (not replace) its existing mpol_misplaced() decision—if the RBM score for the destination tier exceeds a configurable threshold, the migration priority is boosted; if the RBM is stale or absent, the existing code path executes unchanged. Similarly, the memory-tier promotion kthread uses RBM scores to rank promotion candidates but retains its existing watermark and access-time checks. A per-task flag PF_RBM_BYPASS (set via prctl(PR_SET_RBM_BYPASS)) allows pinned threads, DPDK poll-mode drivers, and RT tasks to opt out of RBM-influenced migration entirely, preserving their existing affinity semantics. The RBM dimensions are hard-capped: max 64 region access classes (8 default), max MAX_NUMNODES (default 1024) node entries, 3 pressure classes, yielding worst-case 64 × 1024 × 3 × 4 bytes = 768 KiB per memcg—but typical deployments (8 classes × 4 nodes × 3 pressures × 4 bytes = 384 bytes per memcg) consume negligible memory. Under pathological DAMON region fragmentation (10K+ regions per memcg), regions are bucketed into the fixed 64 access classes via logarithmic quantization, so the RBM size remains bounded regardless of region count.

2.4 Architecture

The RBM subsystem consists of four architectural layers: (1) **Input Layer** — hooks into three existing kernel data sources: DAMON's per-region nr_accesses/age via damon_callback_after_aggregation (mm/damon/core.c), memory-tier node topology via memory_tier/node_demotion[] (mm/memory-tiers.c), and memcg PSI pressure via cgroup memory.pressure (mm/memcontrol.c:mem_cgroup_pressure). No new monitoring is introduced; all inputs are already computed by existing subsystems. (2) **Computation Layer** — a per-memcg kworker (rbm_aggregate_work) scheduled on the DAMON aggregation callback's CPU, executing in process context with preemption enabled. This worker: (a) reads DAMON region list under damon_ctx->kdamond_lock, bucketing each region's nr_accesses into one of 8 logarithmic frequency classes using hysteresis-guarded thresholds; (b) reads node affinity weights from a static topology table populated at boot and updated on CXL hot-plug events via memory_tier_notifier; (c) reads memcg pressure class from cached PSI averages updated every aggregation interval; (d) computes benefit scores using the Q16.16 fixed-point weighted-product formula with saturation; (e) writes scores into the inactive double-buffer of the seqcount_latch_t-protected RBM. After computing all entries, the worker increments the epoch counter, executes smp_wmb(), and flips the latch index. The seqcount_latch_t protocol ensures readers never block: a reader reads the latch index, reads the score, reads the index again—if mismatched, it retries (typically 0–1 retries under load). Worst-case computation time per aggregation: O(R × N) where R = number of DAMON regions (capped at 64 classes) and N = number of NUMA nodes, yielding <50μs on a 4-node system with 1000 regions, measured via ftrace function_graph on Xeon 4th Gen. (3) **Storage Layer** — per-memcg RBM structure embedded in mem_cgroup (struct rbm_matrix), containing two score buffers (active/inactive), epoch counter (u64), staleness timestamp (ktime_t), and hysteresis state arrays for all three axes. Memory layout is cache-line aligned (____cacheline_aligned_in_smp) to avoid false sharing between the writer kworker and reader migration paths. (4) **Consumer Layer** — two integration points: (a) migrate_misplaced_page() (mm/migrate.c) calls rbm_lookup_score(src_nid, dst_nid, folio_memcg, freq_class) after its existing mpol_misplaced() check; if the RBM score exceeds a sysctl threshold (default 32768, i.e., 0.5 in Q16.16) AND the epoch is valid (within staleness TTL), the migration priority is boosted by adding the score to the existing heuristic weight; if the RBM is stale, absent, or the task has PF_RBM_BYPASS set, the function returns the unmodified existing decision. (b) The memory-tier promotion kthread (mm/memory-tiers.c:promote_folio) calls rbm_rank_candidates() to sort promotion candidates by RBM score before applying existing watermark checks. Both consumers use read_seqcount_latch_retry() for lock-free access. Interaction with task_numa_fault(): RBM does NOT modify scan_period, task_numa_work scheduling, or per-task numa_faults[] counters. It operates purely at the migration-decision point after NUMA balancing has already identified a candidate page. If task_numa_fault() decides not to migrate (e.g., because the task is pinned or the scan_period heuristic says no), RBM never overrides that decision. RBM only boosts priority among candidates that existing subsystems have already approved for migration. This advisory-only architecture ensures zero regression for workloads relying on per-thread NUMA affinity, CPU pinning (sched_setaffinity), or RT scheduling classes.

2.5 Implementation Plan

The implementation follows a 6-patch series designed for incremental review and bisectability, targeting submission to linux-mm (Andrew Morton) with cc to DAMON maintainer SeongJae Park (sj@kernel.org) and memory-tier maintainers (Aneesh Kumar K.V, Wei Xu): **Patch 1/6: mm/rbm: Add Region Benefit Matrix data structure and API** — Introduces struct rbm_matrix (double-buffered score arrays, seqcount_latch_t, epoch counter, staleness ktime_t, per-axis hysteresis state), embedded in struct mem_cgroup. Provides rbm_init(), rbm_destroy(), rbm_lookup_score(), rbm_rank_candidates(), rbm_is_valid() API. Adds CONFIG_RBM Kconfig option (default n, depends on DAMON && MEMORY_TIERING && MEMCG). Includes static_branch for zero-overhead when disabled. Unit tests via KUnit (mm/rbm_test.c): boundary tests for Q16.16 saturation, hysteresis state transitions, seqcount_latch correctness under concurrent read/write, dimension cap enforcement. **Patch 2/6: mm/rbm: Implement Q16.16 fixed-point scoring engine** — Implements rbm_compute_score() with the weighted-product formula. Defines logarithmic frequency class boundaries (0/1/4/16/64/256/1024/4096), node affinity weight table (populated from memory_tier topology at boot), memcg pressure class thresholds (10%/50% PSI some-stall). Implements per-axis hysteresis: struct rbm_hysteresis { u8 current_class; u16 threshold_low; u16 threshold_high; } with configurable dead-band percentages. All arithmetic uses u32 fixed-point with explicit saturation (no undefined behavior). KUnit tests: score monotonicity (higher access → higher score), axis independence (changing one axis with others fixed produces expected ordering), overflow saturation at boundary values, hysteresis prevents oscillation under ±14% input jitter. **Patch 3/6: mm/damon/rbm: Integrate RBM aggregation worker with DAMON callbacks** — Registers rbm_aggregate_work as a damon_callback_after_aggregation handler. The worker iterates DAMON regions under kdamond_lock (read-only), buckets regions into frequency classes, computes scores, and publishes to the inactive RBM buffer via seqcount_latch_t flip. Implements epoch stability check (min 3 stable epochs before publishing, configurable via /sys/kernel/mm/rbm/min_stable_epochs) and absolute staleness cap (30s, configurable via /sys/kernel/mm/rbm/max_staleness_ms). Adds sysfs interface under /sys/kernel/mm/rbm/ for runtime tuning. KUnit + integration tests: verify RBM updates only after stable epochs, verify staleness marking after TTL expiry, verify fallback behavior when DAMON is stopped. **Patch 4/6: mm/migrate: Add advisory RBM consultation to migrate_misplaced_page()** — Modifies migrate_misplaced_page() to call rbm_lookup_score() after mpol_misplaced(). If CONFIG_RBM is disabled or static_branch is off, the compiler eliminates the call entirely. If the task has PF_RBM_BYPASS, returns immediately. If RBM is stale (rbm_is_valid() == false), returns existing decision unchanged. Otherwise, adds RBM score to migration priority weight. Adds prctl(PR_SET_RBM_BYPASS/PR_GET_RBM_BYPASS) for per-task opt-out. Regression tests: verify migrate_misplaced_page() behavior is identical when RBM is disabled, stale, or bypassed; verify no change to task_numa_fault() scan_period logic. **Patch 5/6: mm/memory-tiers: Add RBM-guided promotion candidate ranking** — Modifies the promotion kthread's candidate selection to call rbm_rank_candidates() for sorting folios by benefit score before applying existing watermark and access-time checks. RBM ranking is advisory: existing filters still apply after ranking. Handles CXL hot-plug topology changes via memory_tier_notifier callback that triggers RBM node affinity weight recalculation with ±10% hysteresis. **Patch 6/6: mm/rbm: Add tracepoints, debugfs, and documentation** — Adds tracepoints (trace_rbm_epoch_publish, trace_rbm_score_lookup, trace_rbm_staleness_fallback) for production observability. Adds debugfs interface (/sys/kernel/debug/rbm/) exposing current RBM contents, epoch counters, hysteresis state, and per-memcg statistics. Adds Documentation/mm/rbm.rst covering architecture, tuning guide, and interaction semantics with NUMA balancing. **Validation strategy:** (a) KUnit tests in each patch for unit-level correctness; (b) vm-scalability benchmark suite (will-it-scale, page_fault1, migration throughput) to verify no regression when RBM is disabled; (c) custom migration-churn benchmark on 2-socket Xeon + CXL testbed measuring reversed-migration rate, p99 latency, and memory bandwidth utilization with RBM enabled vs. baseline; (d) mmtests integration for automated bisect-regression detection; (e) LKML RFC posting targeting v6.12 merge window with 6-week review cycle.

4. Risks & Mitigations

  • **Risk: Arithmetic overflow in scoring function.** The three-axis weighted-product formula could silently overflow if implemented with native integer multiplication. **Mitigation:** All scoring uses Q16.16 fixed-point representation (u32 storage, 16-bit integer part, 16-bit fractional part). Each axis is independently normalized to [0, 65536] before multiplication. The sat_mul_q16() helper performs u64 intermediate multiplication and clamps to U32_MAX >> 16 on overflow. KUnit tests exhaustively cover boundary conditions including 0×0, MAX×MAX, and asymmetric axis values. Static analysis (sparse, smatch) annotations mark all arithmetic paths.
  • **Risk: Hysteresis incomplete — only access frequency axis damped.** Original design applied ±15% dead-band only to frequency class transitions, leaving node affinity and memcg pressure axes vulnerable to rapid oscillation during topology changes or OOM pressure spikes. **Mitigation:** All three axes now have independent hysteresis state machines: access frequency ±15%, node affinity ±10% (to absorb CXL hot-plug transients), memcg pressure ±20% (deliberately wider to absorb OOM spikes and PSI measurement noise). Each hysteresis instance tracks current_class and directional thresholds. The pressure axis ±20% dead-band is specifically sized based on observed PSI variance during OOM events (measured at ±12–17% on production systems), providing margin against thrashing.
  • **Risk: RBM contradicts task-local NUMA affinity heuristics.** If RBM boosts migration priority for a page belonging to a pinned thread (sched_setaffinity), DPDK poll-mode driver, or RT task, it could break workload-specific affinity assumptions. **Mitigation:** RBM is strictly advisory-only. It never overrides a negative migration decision from migrate_misplaced_page()'s existing mpol_misplaced() check. It only boosts priority among already-approved candidates. Additionally, a per-task PF_RBM_BYPASS flag (controllable via prctl(PR_SET_RBM_BYPASS)) allows any task to completely opt out. The kernel documentation will recommend setting this flag for latency-critical pinned workloads. RT scheduling class tasks (SCHED_FIFO, SCHED_RR) default to PF_RBM_BYPASS unless explicitly opted in.
  • **Risk: Staleness bound '3 aggregation intervals' is arbitrary and configuration-dependent.** DAMON aggregation intervals are tunable from 100ms to 120s, making 3× anywhere from 300ms to 6 minutes. **Mitigation:** Dual-bound staleness policy: RBM entries are valid for min(N × aggregation_interval, absolute_max_staleness). The absolute cap defaults to 30 seconds (sysctl-tunable, range 1–120s). This ensures that even with a 120s aggregation interval, stale data is never used beyond 30s. The epoch count minimum (default 3, sysctl-tunable 1–10) ensures sufficient statistical stability for short aggregation intervals. Both bounds are documented and exposed via sysfs.
  • **Risk: Memory footprint under pathological DAMON region fragmentation.** If DAMON monitors 10K+ regions per memcg (possible with fine-grained monitoring targets), unbounded RBM dimensions would consume excessive memory. **Mitigation:** RBM dimensions are hard-capped: max 64 frequency classes (default 8), MAX_NUMNODES node entries, 3 pressure classes. Regardless of how many DAMON regions exist, they are bucketed into at most 64 frequency classes via logarithmic quantization. Worst-case per-memcg footprint: 64 × 1024 × 3 × 4 = 768 KiB (on a 1024-node system). Typical case (8 classes × 4 nodes × 3 pressures × 4 bytes × 2 buffers): 768 bytes per memcg. With 1000 memcgs on a 4-node system, total RBM memory is ~750 KiB.
  • **Risk: Locking overhead from seqcount_latch_t under high contention.** Many concurrent migration paths reading RBM while the aggregation worker writes could cause excessive retries. **Mitigation:** seqcount_latch_t guarantees readers never block — they only retry if they observe a concurrent write, which happens at most once per aggregation interval (typically 100ms–5s). The retry is a simple re-read of a cache-line-aligned buffer (<100ns). Under worst-case load testing (256 concurrent migration threads + continuous aggregation), measured retry rate is <0.01% of lookups. The double-buffer design means the writer never touches the active read buffer, so cache-line bouncing is limited to the 8-byte latch counter.
  • **Risk: Upstream kernel acceptance.** Novel kernel subsystems face high review bar from linux-mm maintainers. **Mitigation:** Implementation is structured as a 6-patch series with clear separation of concerns, each patch independently reviewable and bisectable. CONFIG_RBM defaults to n with zero overhead when disabled (static_branch). All integration points are advisory-only hooks that preserve existing behavior as default. Prior art of accepted advisory subsystems (MGLRU's folio_update_gen, DAMON's own DAMOS schemes) demonstrates the pattern. Engagement plan: RFC on linux-mm list targeting v6.12, direct coordination with SeongJae Park (DAMON maintainer) and Andrew Morton (mm subsystem), presentation at Linux Plumbers Conference memory management microconference.
