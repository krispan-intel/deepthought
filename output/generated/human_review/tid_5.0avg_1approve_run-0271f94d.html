2. Technical Invention Disclosure

2.1 Problem Statement

On high-core-count Intel systems, logically related filesystem work often executes on unrelated CPUs across submission, I/O completion, metadata follow-up, journal advancement, and optional parity or LDPC processing. The worst cases are inode-dense create/unlink storms, fsync-heavy builds, delayed-allocation conversion, block-group or AG accounting, and erasure-coded stripe repair or parity generation after asynchronous I/O. Because blk-mq completion can arrive in IRQ, softirq, ksoftirqd, or thread context depending on driver and configuration, a single object or stripe can bounce across LLC and NUMA domains. That increases remote cache misses, lock handoff latency, and cacheline migration, especially when cpusets shrink, CPUs hotplug offline, or housekeeping isolation excludes some CPUs. Existing mechanisms can pin threads or bias workqueues, but they do not give the filesystem a deterministic object-or-stripe-to-topology-domain identity together with a completion-safe bridge and a forward-progress-safe re-resolution rule when the preferred CPUs become unavailable.

2.2 Prior-Art Gap

Adjacent mechanisms are known, but each stops short of the claimed contract. CPU topology parsing and sibling masks expose package, die, core, and cache-sharing relationships; cpusets and task affinity constrain where a task may run; WQ_UNBOUND plus workqueue attrs can bias worker placement; irq_work can escape hardirq context; blk-mq and storage stacks already publish minimal completion state; and some subsystems serialize work per object or per stripe. What is missing is the combination of: (a) a filesystem-owned logical domain table whose keys are derived from topology and frozen as stable identities; (b) an explicit deterministic mapping from filesystem object identity or stripe identity to one such home domain; (c) a rule that hotplug, affinity, and cpuset changes do not remap the home domain but only affect the runtime execution CPU selected at enqueue time; (d) a strict split in which completion context may only publish compact deltas and request a bridge, while all sleeping lock acquisition and metadata mutation is reserved to a single-writer coordinator; and (e) concrete memory-ordering and replay semantics proving that concurrent completions cannot lose a wakeup. Narrowing the novelty to deterministic topology-domain binding plus publication-to-coordinator separation differentiates the invention from ordinary use of queue_work_on(), kthread_worker, or generic per-object work serialization.

2.3 Proposed Invention

At mount or superblock activation, the filesystem builds a logical domain table from cpu_possible_mask so the set of logical domains survives ordinary CPU hotplug. In the preferred x86 embodiment, each domain key is a canonical tuple of NUMA node ID, LLC identifier or canonical LLC-sharing-mask hash, package ID, and die ID as exposed by existing CPU topology and cache enumeration established from APIC/x2APIC parsing; if LLC identity is unavailable, the key degrades deterministically to package-and-node scope. The filesystem sorts unique keys to assign stable logical domain indexes. Each work-bearing entity receives a deterministic identity: for metadata buckets, a tuple such as fs UUID plus inode number bucket, block-group number, or AG number; for erasure-coded storage, a stripe ID computed from fs UUID, logical stripe number, and layout epoch. The mapping function is explicit and auditable, for example SipHash over that identity reduced modulo the number of logical domains. The home-domain index is therefore deterministic and stable across CPU hotplug and cpuset changes. Runtime execution does not cache a stale runnable mask. Instead, when an I/O completion or other producer event occurs, the completion path is limited to context-safe publication: it updates compact multi-writer fields such as pending event bits, status codes, or reference deltas using WRITE_ONCE and atomic operations, performs a release publication of a monotonically increasing publish generation, and sets a queued bit if clear. It then triggers a bridge using irq_work, which is safe in hardirq or softirq context. The irq_work callback performs no metadata mutation and only queues a lightweight resolver work item onto a general housekeeping-safe workqueue. The resolver runs in task context, takes cpus_read_lock() and RCU as needed, snapshots the current topology generation and live masks, and computes an allowed CPU set by intersecting the home-domain static mask with cpu_online_mask, workqueue-allowed CPUs, housekeeping CPUs, and, when the work item is attributed to a live owner task or cgroup, that owner's current effective affinity/cpuset mask. If the result is empty, the resolver walks a bounded deterministic fallback ladder: first sibling domains in the same NUMA node, then same package, then the filesystem service mask reserved for housekeeping-safe execution. Only a mount-option-controlled escape hatch may allow non-housekeeping execution; otherwise the item waits on a deferred list and is retried on CPU-online or cpuset-widen events. Once a non-empty mask exists, the resolver chooses a target CPU with cpumask_any_distribute() and dispatches the coordinator using queue_work_on(target_cpu, domain_wq, &coord_work). The coordinator is the sole writer for that object or stripe. It alone may acquire filesystem sleeping locks, mutate inode or extent state, advance journal or transaction state, finalize folio-facing actions, or run parity/LDPC follow-up. Single-writer-only fields include handled generation, replay cursor, and object-specific state-machine transitions; multi-writer completion-published fields are limited to pending bits, status deltas, and publish generation. After processing one batch, the coordinator compares handled generation to the current published generation and loops until stable; only then does it clear the queued bit and immediately recheck for a newly published generation, thereby preventing lost wakeups even when multiple completions arrive while the coordinator is running or just before it drops the queued flag. Remap invariants are explicit: CPU hotplug does not change home-domain index; cpuset or affinity changes do not change home-domain index; layout rebalancing changes stripe binding only when the layout epoch changes, which is an auditable filesystem event rather than an incidental scheduler event.

2.4 Architecture

Core structures and roles:
- fs_topology_map: RCU/seqcount-protected table of logical domains built from cpu_possible_mask.
- fs_domain: stable logical domain index, canonical topology key, static_domain_mask, fallback domain order, domain workqueue attrs.
- fs_work_item or fs_stripe_ctx: object/stripe identity, home_domain_idx, owner-task or owner-cgroup reference, pending_bits, publish_gen, handled_gen, queued flag, irq_work bridge, resolver work, coordinator work.

Deterministic identity and mapping:
- Metadata identity: (fs_uuid, inode_bucket) or (fs_uuid, block_group/AG).
- Stripe identity: SipHash(fs_uuid, logical_stripe_nr, layout_epoch).
- Home domain: identity_hash mod nr_logical_domains.
- Invariants: hotplug and cpuset changes do not alter home_domain_idx; a layout_epoch change is the only ordinary event that intentionally remaps a stripe.

  - never holds publish-path spin protection while taking sleeping locks
  - may run parity/LDPC only after all spinlocks are dropped
  - loops until handled_gen == publish_gen and pending_bits are drained

State machine and memory ordering:
- IDLE: queued=0, pending_bits=0, handled_gen==publish_gen.
- PUBLISHED: one or more producers set bits and release-publish a newer publish_gen.
- ROUTED: queued=1; bridge and/or resolver outstanding.
- RUNNING: coordinator is active as sole writer.
- REPLAY: coordinator observed publish_gen > handled_gen and continues without relinquishing single-writer ownership.
- Return to IDLE only after queued is cleared and a post-clear acquire check confirms no newer generation exists.

Context rules:
- Completion site may not sleep, may not acquire filesystem locks that can block, may not finalize folios or journal state, and may not invoke SIMD/FPU helpers.
- irq_work also performs no sleeping or metadata mutation.
- All metadata mutation, journal/extent operations, and optional codec work occur only in coordinator task context.

Locking rule to avoid ABBA:
- Completion publication uses only atomic fields and, if needed, a tiny publish-path spinlock local to the work item.
- Coordinator snapshots or clears publish fields before taking filesystem sleeping locks.
- Coordinator never takes journal, extent, inode, or transaction locks while holding publish-path spin protection, so it fits under the filesystem's pre-existing lock hierarchy rather than creating a cross-context cycle.

2.5 Implementation Plan

1. Domain discovery and freezing. During mount or superblock start, enumerate CPUs in cpu_possible_mask and collect canonical topology keys. Preferred x86 key order is (node_id, llc_id_or_llc_mask_hash, package_id, die_id). Sort unique keys and build fs_domain entries, each with a static_domain_mask and an ordered fallback list favoring same-NUMA then same-package neighbors. Publish the table under RCU with a topology sequence counter so resolver work can detect concurrent rebuilds. 2. Work-item identity and remap invariants. Add a small context structure for each metadata bucket or stripe. Store object identity, home_domain_idx, layout_epoch for stripes, owner task or cgroup reference if owner-local policy is desired, publish_gen, handled_gen, pending_bits, queued flag, irq_work, resolver work, and coordinator work. Define stripe ID as SipHash(fs_uuid, logical_stripe_nr, layout_epoch); define metadata IDs similarly from stable filesystem object numbers. Document invariants: hotplug and cpuset changes do not rewrite home_domain_idx; layout_epoch change intentionally remaps only when filesystem layout actually changes. 3. Completion publication path. In blk-mq or filesystem completion callbacks, permit only context-safe operations: update compact status or byte-count deltas with WRITE_ONCE, set event bits with atomic_or(), increment publish_gen with release semantics, and, if queued was previously clear, queue irq_work. No sleeping allocations, no folio final transitions, no journal handle acquisition, no blocking locks. The completion path may run in IRQ, softirq, ksoftirqd, or thread context and therefore must obey the strictest non-sleeping rules. 4. Bridge and resolver. Implement irq_work callback purely as a bridge that schedules resolver work to task context. In resolver work, take cpus_read_lock() and RCU, snapshot the current domain mask and applicable owner mask, and compute allowed CPUs as the intersection of domain mask, cpu_online_mask, workqueue mask, housekeeping mask, and live owner affinity/cpuset mask if present. Copy masks before dropping locks to avoid dereferencing freed data. If the home-domain intersection is empty, walk the deterministic fallback ladder. If all permitted masks are empty, place the item on a deferred list and register hotplug/cpuset callbacks to retry when a CPU becomes available; do not spin or livelock. 5. Coordinator and no-lost-wakeup proof. Dispatch coordinator work with queue_work_on(cpumask_any_distribute(allowed), domain_wq, &coord_work). The coordinator is the sole writer for handled_gen and object state transitions. It consumes pending_bits, performs metadata mutation, and then records handled_gen. Before relinquishing ownership, it clears queued and immediately performs an acquire check of publish_gen and pending_bits. If a producer published before queued was cleared, the generation mismatch is observed and the coordinator requeues itself; if a producer publishes after queued is cleared, that producer sees queued==0 and issues a new bridge. Therefore every publication is covered by either the in-flight coordinator replay loop or a newly scheduled run, preventing lost wakeups. 6. Hotplug and topology changes. On CPU-online/offline, rebuild only live masks and fallback reachability, not logical home-domain indexes. Because the domain table is based on cpu_possible_mask, ordinary hotplug merely deactivates or reactivates CPUs inside an existing logical domain. If firmware or virtualization changes topology so drastically that canonical domain keys change, bump topology sequence and rebuild on remount or an explicit administrative rebind event rather than silently remapping live stripes. 7. Cpuset, cgroup, and housekeeping policy. When work is owner-attributed, hold a safe reference to the owner task or cgroup and resolve its effective allowed CPUs at enqueue time, not once at submission. If owner constraints collapse to empty, relax only within a configured service policy: same NUMA and package first, then superblock service mask. By default, require housekeeping CPUs for all filesystem worker execution so nohz_full and isolation contracts are honored. Any escape to non-housekeeping CPUs must be an explicit mount or administrative policy bit. 8. SIMD/FPU and codec constraints. Optional parity or LDPC work is permitted only in coordinator task context, with no spinlocks held, no sleeping section inside kernel_fpu_begin()/kernel_fpu_end(), and bounded chunks sized so worker preemption latency remains acceptable. On architectures or configs where in-kernel FPU use is unavailable or disabled, switch to a scalar codec or pre-existing non-SIMD helper while keeping the same topology-domain scheduler contract. 9. Filesystem integration points. Hook the mechanism into metadata-heavy paths such as delayed allocation conversion, inode orphan cleanup, journal follow-up, per-AG accounting, or erasure-coded stripe repair. The coordinator must respect the host filesystem's existing lock ordering and transaction rules rather than inventing a parallel lock hierarchy. 10. Observability. Expose tracepoints for publish, route, fallback step chosen, queue CPU, replay count, and home-domain miss rate so operators can verify locality benefits and identify pathological cpuset or hotplug conditions.

2.6 Validation Plan

1. Correctness under mixed completion contexts: exercise blk-mq completions arriving in hardirq, softirq, ksoftirqd, and threaded modes; assert that completion callbacks never sleep, never take blocking filesystem locks, and only publish deltas plus bridge requests. 2. No-lost-wakeup stress: use fault injection to force bursts of concurrent completions against the same stripe or metadata bucket while delaying the coordinator. Verify monotonic publish_gen, handled_gen catch-up, and zero stranded pending_bits after quiescence. 3. CPU hotplug: offline and online CPUs within the home domain during heavy metadata activity. Confirm home_domain_idx is unchanged, selected CPUs move through the fallback ladder, and execution returns to the home domain when CPUs reappear. 4. Cpuset and cgroup churn: repeatedly shrink and widen owner cpusets while operations are in flight. Verify enqueue-time mask resolution honors current constraints, never dereferences stale masks, and never runs on disallowed CPUs. 5. Housekeeping and isolation: run with nohz_full or isolcpus deployments and confirm the default policy never dispatches metadata work onto non-housekeeping CPUs. 6. Locking and context safety: run lockdep, KCSAN, and relevant filesystem stress suites to confirm no new ABBA deadlocks, no sleeping-in-atomic warnings, and no data races around queued flag, publish_gen, or pending_bits. 7. Performance: compare baseline versus topology-domain scheduling on metadata-heavy benchmarks, fsync-heavy builds, inode scans, and erasure-coded repair. Measure p99/p999 latency, LLC miss rate, cross-node memory traffic, lock contention, and CPU migrations. 8. SIMD/FPU path validation: on supported x86 kernels, confirm codec execution occurs only in coordinator task context and never with spinlocks held; on unsupported configs, confirm automatic scalar fallback with identical correctness. 9. Failure and repair cases: inject I/O errors, journal aborts, and stripe checksum failures to verify the coordinator serializes recovery decisions and that fallback CPUs do not alter state-machine correctness. 10. Auditability: log home-domain index, chosen fallback stage, stripe ID, and layout epoch so remap behavior across rebalancing or repair is explainable and patent claims remain auditable.

4. Risks & Mitigations

  • Risk: prior-art overlap with generic workqueue affinity or per-object work serialization. Mitigation: narrow the claims to deterministic topology-domain identity, completion-publication-only semantics, enqueue-time live-mask resolution, and a generation-protected single-writer replay contract rather than claiming affinity or serialization broadly.
  • Risk: topology inputs may be incomplete on some platforms or virtual machines. Mitigation: define a canonical degradation path from LLC scope to package scope to NUMA scope, freeze the logical domain table from cpu_possible_mask, and treat x86 APIC/x2APIC as a preferred embodiment of the broader topology-derived-domain concept.
  • Risk: cpuset or affinity changes may empty the preferred mask and cause starvation. Mitigation: use a bounded fallback ladder ending in a prevalidated superblock service mask on housekeeping CPUs, and if no permitted CPU exists, park the item and retry on notifier events rather than livelocking.
  • Risk: concurrent completions could miss a wakeup or corrupt state. Mitigation: separate multi-writer published fields from single-writer consumed fields, use release/acquire publication of publish_gen, and clear queued only with a post-clear replay check so every generation is either consumed by the current coordinator or causes a new run.
  • Risk: deadlock with existing filesystem lock hierarchies. Mitigation: completion and bridge paths take no sleeping filesystem locks, and the coordinator drops any publish-path spin protection before taking journal, inode, extent, or transaction locks in the filesystem's established order.
  • Risk: SIMD or FPU use in kernel context can violate context rules or platform policy. Mitigation: permit codec execution only in coordinator task context with no spinlocks held and bounded non-sleeping kernel_fpu sections, and provide a scalar codec path whenever kernel FPU use is unavailable or administratively disabled.
  • Risk: remap behavior during repair or rebalance could become nondeterministic. Mitigation: define stripe identity to include an explicit layout epoch so remap occurs only on auditable layout transitions, not on incidental scheduler, cpuset, or hotplug events.
