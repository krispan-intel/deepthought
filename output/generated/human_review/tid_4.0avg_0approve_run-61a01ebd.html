2. Technical Invention Disclosure

2.1 Problem Statement

Linux memory management still makes most reclaim, aging, and huge-page decisions from conservative software proxies such as accessed bits, refault distance, workingset history, and MGLRU generation membership. Those signals are intentionally robust, but they can lag badly when a workload changes phase quickly, when hot data is touched sparsely but with high latency cost, or when one process mixes latency-critical and background regions inside the same memcg. x86 processors already expose PEBS records carrying precise retired data-address information for certain memory events, and analogous facilities exist on other architectures such as AMD IBS and Arm SPE. The unsolved kernel problem is not collecting samples for observability; it is converting those samples into reclaim-consumable signals without violating kernel safety or fairness constraints. Sampling occurs in PMI or NMI context where page-table walks, sleeping locks, allocations, GUP, and page pinning are forbidden. The target mm can exec, exit, or be reused before delayed processing. A PFN can be recycled after reclaim, a VMA can split or merge, and a pkey can change through pkey_mprotect(). If the resulting heat signal is unconstrained, it can distort MGLRU progress, override memcg fairness, leak access patterns, or create architecture-specific regressions under virtualization or PMU backpressure. The technical problem is therefore to transform precise sampled user-memory accesses into bounded, race-safe, privacy-safe, opt-in, and reclaim-safe heat signals that improve Linux memory-management choices while preserving exact baseline behavior whenever the sampling path is unavailable or low confidence.

2.2 Prior-Art Gap

The key prior-art gap is a missing kernel-safe conversion boundary between precise PMU data-address sampling and memory-management decisions. 1) DAMON distinction: DAMON periodically samples or monitors address regions and drives actions over coarse ranges. It is a region-scanner framework, not a precise retired-memory event pipeline, and it does not define NMI capture rules, an ABA-proof mm identity contract, or deferred no-fault VA-to-folio translation from sampled PMU records. 2) MGLRU and access-bit distinction: existing reclaim logic uses page-table recency proxies, refault heuristics, and generation aging, but it does not consume precise retired load-latency samples or define an external folio heat map tied to delayed address resolution. 3) perf mem and PEBS tooling distinction: perf and PEBS infrastructure expose addresses and event metadata for profiling and observability, but they stop at sample delivery; they do not define a kernel-only path that turns a sampled VA into a reclaim-safe folio signal under exact lock and fault constraints, nor do they prevent address aliasing by combining a boot-unique mm cookie with an exec generation. 4) Namespace distinction: memcg IDs are too coarse because they merge unrelated hot and cold regions and cannot represent distinct protected domains inside one process; raw VMA identity is too unstable because VMA split and merge churn would fragment history; anon_vma and address-range bucketing are either semantically weak or too volatile. vm_pkey is materially different because, when PKE is enabled through X86_CR4.PKE and the OS has OSPKE support, it provides a low-cardinality, per-mm domain tag that survives most VMA topology churn and can encode meaningful application intent such as protected data planes, model weights, scratch buffers, or tenant partitions. The invention also addresses the missing handling of pkey changes: historical heat is not reassigned speculatively across domains; instead, pkey change events trigger epoch advance and aggressive decay so new samples repopulate the correct bucket. 5) Policy-publication distinction: prior art does not define immutable per-reclaim snapshots for multi-field heat policy, leaving readers vulnerable to mixed-epoch settings on weakly ordered systems. The claimed system publishes policy by RCU pointer replacement, copies the snapshot once at reclaim entry, and never allows one invocation to observe torn controls.

2.3 Proposed Invention

The invention introduces a PEBS-first sampled-heat subsystem with a narrow provider interface and a strict five-stage kernel pipeline. Stage A, precise sample ingestion: an x86 provider programs PEBS only for event types that guarantee a precise retired user data address on supported microarchitectures, preferably retired load or load-latency classes; records lacking a precise user linear address are dropped immediately. In PMI or NMI context the handler performs only constant-time checks: it verifies user mode, current->mm presence, global enable, per-mm opt-in, and PEBS record validity. It then reads mm_cookie and mm_exec_epoch from current->mm with READ_ONCE, packages them with the sampled user VA, event class, latency bucket, CPU, NUMA node, and timestamp, and writes the record into a fixed-size per-CPU ring. No page-table walk, printk, allocation, sleeping lock, GUP, or page reference is allowed on this path. The ring is overwrite-oldest and NMI-safe: the producer reserves a slot by atomic head increment, writes the payload, and finally publishes a per-slot sequence number with store-release semantics. The consumer validates completion with load-acquire semantics; if head has advanced more than ring_size beyond tail, the consumer records overrun counters and skips directly to head - ring_size, thereby dropping oldest unresolved samples and bounding backlog. Stage B, mm anti-aliasing and liveness contract: each mm receives a boot-unique 64-bit mm_cookie at mm_init() or dup_mm(), allocated from a global atomic64 counter and inserted into an RCU-protected xarray registry. The cookie is never reused within the boot lifetime. Each mm also carries a 32-bit mm_exec_epoch incremented on successful exec image replacement. Fork creates a new mm and thus a new cookie; exec preserves the mm object but increments mm_exec_epoch; exit removes the cookie from the registry and defers final free through RCU. Delayed resolution never trusts a pointer captured in NMI. Instead it loads mm by cookie from the registry under rcu_read_lock(), checks that the current mm_exec_epoch still matches the sampled epoch, and obtains a live reference only through mmget_not_zero(). If lookup fails, the epoch mismatches, or mmget_not_zero() fails, the sample is dropped. This makes a reused mm_struct address or recycled PID irrelevant because identity is carried by the cookie plus exec epoch rather than by PID or pointer value. Stage C, deferred no-fault VA-to-folio resolution: records are drained from the per-CPU ring by irq_work into a per-node kthread running in process context. For each sample the resolver acquires mmap_read_lock(mm), locates the VMA with find_vma(), verifies that the address lies inside the VMA, reads vm_pkey from the current VMA, and rejects unsupported mappings such as VM_PFNMAP, VM_MIXEDMAP, VM_IO, secret-memory ranges, or mappings outside the per-mm allowed domain mask. It then performs an explicit page-table walk using pgd, p4d, pud, pmd, and pte helpers plus the relevant page-table lock. The resolver never invokes handle_mm_fault(), follow_page(), get_user_pages(), pin_user_pages(), or any faulting copy helper. It resolves only already-present entries. It drops the sample on any unstable state, including unmapped VMA, missing entry, non-present PTE, swap entry, migration entry, device-private or device-exclusive entry, hwpoisoned page, zero-page or special mapping if configured unsupported, pmd_trans_unstable(), THP split in progress, hugetlb mapping when hugepage support is not compiled for the subsystem, memory-offline PFN, or pfn_valid() failure. For a present regular page or THP head PFN, the resolver snapshots a per-PFN allocation generation alloc_gen from page_ext metadata while still under the relevant lock, then releases the walk. This yields a stable heat key without ever taking a folio pointer as durable identity. Stage D, external heat storage and pkey-domain aggregation: resolved samples update two bounded structures. First, a node-local folio heat table keyed by {mm_cookie, pfn, alloc_gen}. The value stores an exponentially decayed heat score, confidence, last-update time, sample count, and last event class. The table is implemented as a bounded rhashtable or xarray-backed shard with LRU eviction; because alloc_gen changes whenever a PFN is freed and later reallocated, PFN reuse cannot alias stale heat. The alloc_gen lives in page_ext or an equivalent external per-page extension so core folio layout remains untouched. On folio migration the migration path copies or re-inserts heat under the destination PFN with the same mm_cookie and alloc_gen-derived freshness; on THP split, the resolver either distributes a head-folio score across children according to subpage sample history or drops the sample if split is in progress; on reclaim or free, a page_ext generation bump invalidates old table entries naturally. Second, each mm maintains a 16-bucket vm_pkey heat vector plus a pkey_epoch. Aggregation is always scoped by mm_cookie; pkey value 3 in one mm is never compared with pkey value 3 in another. If pkey_mprotect() changes a VMA domain, the kernel increments pkey_epoch and right-shifts all bucket heat in that mm rather than attempting speculative reassignment of historical samples. Most mappings may remain in pkey 0, which becomes the default bucket; applications that care about finer domains opt in by assigning distinct pkeys. Stage E, bounded policy consumption and exact fallback: policy lives in an RCU-published struct sample_heat_policy containing thresholds, allowed event classes, overflow disable thresholds, minimum sample counts, pkey-domain masks, and per-consumer caps. Writers allocate a fully initialized replacement policy, publish it by rcu_assign_pointer(), and retire the old object with call_rcu(). Readers use rcu_dereference() and copy the snapshot once into a local stack struct at reclaim or THP decision entry, so one invocation never sees mixed settings. The policy may bias only four consumption points: 1) MGLRU aging retention for already-eligible folios within a lruvec; 2) MGLRU scan ordering within the same lruvec and memcg; 3) THP fault-allocation and khugepaged collapse candidate ranking, plus limited split deferral for hot anonymous THPs; and 4) compaction isolation preference to avoid isolating hot folios when cold alternatives exist. It may not change memcg charge accounting, reclaim targets, dirty-writeback rules, unevictable state, or absolute scan quotas. For each consumer the adjustment is monotone and capped, for example bias = confidence * min(raw_heat, max_heat_cap), with confidence forced to zero whenever ring overrun exceeds a threshold, resolver drop rate is too high, PMU precision flags are absent, sample rate falls below minimum, or the CPU is in unsupported guest or host PMU mode. Since quotas and eligible sets are computed by baseline logic before bias is applied, the subsystem only reorders victims inside an already-selected set. When confidence is zero, every bias term becomes zero and the code path is byte-for-byte equivalent to baseline behavior except for statistics counters. Security and privacy are enforced by requiring CAP_SYS_ADMIN or an equivalent admin policy, satisfying perf_event_paranoid restrictions, and requiring explicit per-mm opt-in through a dedicated prctl or cgroup-mediated enable. Raw sampled addresses never leave the kernel; exported observability is limited to aggregate heat counters and confidence levels. In virtualized environments, guest-mode samples are ignored unless the platform declares precise address attribution safe for that privilege domain; otherwise the subsystem disables itself and reverts to baseline.

2.4 Architecture

Core data path:

userspace thread with opt-in mm
  -> precise retired load produces PEBS record
  -> PMI/NMI handler on same CPU
  -> per-CPU overwrite-oldest sample ring
  -> irq_work wakeup
  -> NUMA-node resolver kthread
  -> RCU mm-cookie registry lookup
  -> mmget_not_zero + mmap_read_lock(mm)
  -> no-fault page-table walk
  -> {pfn, alloc_gen, vm_pkey} resolution
  -> bounded folio heat table + mm pkey heat vector
  -> immutable RCU policy snapshot copied at entry
  -> MGLRU aging / scan order + THP + compaction bias
  -> reclaim decision inside existing memcg and lruvec limits

Control and safety planes:

sysctl and admin policy
  -> global enable, sample rate, overflow thresholds
prctl or cgroup opt-in
  -> per-mm enable and allowed pkey mask
mm lifecycle hooks
  -> mm_cookie allocation, exec epoch bump, registry removal on exit
mprotect pkey hook
  -> pkey epoch bump and decay
page_ext metadata
  -> alloc_gen for PFN reuse protection

Key invariants:
1) NMI path performs no translation, allocation, locking that can sleep, GUP, or pinning.
2) Delayed resolver uses only present-page page-table walking and drops unstable mappings instead of faulting them in.
3) Folio identity is externalized as {mm_cookie, pfn, alloc_gen}; no long-lived folio pointer is stored.
4) vm_pkey is only an intra-mm domain label; it is never treated as a cross-mm tenant identifier.
5) Policy is immutable for one reclaim invocation because readers copy a single RCU snapshot.
6) Heat can reorder only already-eligible candidates and only within configured caps; memcg fairness and reclaim progress remain baseline-owned.
7) If precision, rate, overflow, virtualization, or resolver health checks fail, confidence drops to zero and all decisions revert exactly to baseline.

2.5 Implementation Plan

1. Add a new mm/sample_heat.c subsystem and mm/sample_heat.h interfaces. Implement global policy structs, the RCU-published policy pointer, per-node worker threads, the bounded folio heat table, and exported aggregate statistics. 2. Wire mm lifecycle hooks: assign mm_cookie in mm_init() and dup_mm() paths, insert the mm into an RCU xarray registry, bump mm_exec_epoch in the exec image replacement path, and remove the cookie from the registry on mm teardown using call_rcu(). Candidate touch points are kernel/fork.c, fs/exec.c, and mm teardown paths. 3. Add an x86 provider in arch/x86/events/intel/ or an adjacent PMU bridge file that subscribes only to PEBS record classes with precise user data addresses. The provider should validate privilege mode, honor perf_event_paranoid and admin gating, and enqueue compact records into a fixed per-CPU ring using atomic head reservation and per-slot sequence numbers. 4. Implement the resolver in mm/sample_heat.c. Drain samples through irq_work to a kthread, look up mm by cookie under RCU, call mmget_not_zero(), take mmap_read_lock(), find the VMA, read vm_pkey, and walk the page tables directly. Explicitly reject swap, migration, device, hwpoison, unstable THP, unsupported hugetlb, and non-present entries. Never call handle_mm_fault(), follow_page(), get_user_pages(), or pin_user_pages(). 5. Increment alloc_gen on page free and any other PFN-reuse boundary so stale heat entries cannot alias a recycled PFN. Provide migration and THP split hooks to preserve or safely invalidate heat state. 6. Add pkey-domain handling in the mprotect pkey path so pkey changes advance pkey_epoch and aggressively decay that mm's pkey vector rather than trying to relabel old heat. Require CPU and kernel support for OSPKE before enabling pkey-domain mode; otherwise bucket all samples into domain 0 and advertise that the pkey embodiment is unavailable. 7. Integrate bounded consumers: in mm/vmscan.c or the MGLRU-specific logic, apply a capped heat bonus only after baseline eligibility and memcg quotas are determined; in mm/huge_memory.c and mm/khugepaged.c, bias fault allocation, collapse ranking, and split deferral only when confidence exceeds threshold; in mm/compaction.c, prefer not to isolate hot folios when cold alternatives exist. 8. Add a secure control plane: sysctls under vm.sample_heat.*, a prctl or cgroup interface for per-mm opt-in, and admin-only readout of aggregate counters. Do not use debugfs as the primary control plane. 9. Provide alternate-provider stubs for AMD IBS and Arm SPE through a generic provider-ops layer, but keep the narrowest implementation and claims anchored on x86 PEBS precise retired-memory semantics.

2.6 Validation Plan

Validation is organized around correctness, safety, performance, privacy, and prior-art differentiation. Correctness: create kernel selftests that cover fork, exec, exit, VMA split and merge, pkey_mprotect(), THP split and collapse, migration, swap-out, swap-in, and reclaim-driven PFN reuse. Verify that stale samples are rejected on mm cookie miss, exec epoch mismatch, or alloc_gen mismatch, and that no sample survives after mm teardown. Safety: use lockdep, KCSAN, KFENCE, and stress-injected NMI storms to prove the NMI path never sleeps, allocates, or faults; measure ring overrun behavior and confirm overwrite-oldest semantics bound latency. Verify the resolver never calls faulting APIs by instrumentation and by running with deliberate non-present PTEs and migration entries. Performance: compare baseline Linux, baseline plus perf PEBS without memory-management consumption, baseline plus DAMON-based hotness, and the proposed subsystem on mixed anonymous and page-cache workloads such as Redis, RocksDB, graph analytics, sparse inference, JVM heaps, and memory-pressure microbenchmarks with pkey-separated hot and cold regions. Metrics include reclaim efficiency, refault rate, tail latency, THP collapse success, THP split rate, compaction success, CPU overhead, and PMU backpressure. Fairness and no-regression proof: confirm that memcg reclaim targets, charge accounting, and forward progress match baseline under disabled sampling, low sample density, high overrun rate, guest-mode PMU ambiguity, and unsupported CPUs; in each unhealthy case confirm confidence goes to zero and adjusted victim ordering becomes identical to baseline. Privacy and security: verify that unprivileged tasks cannot enable the subsystem, that perf_event_paranoid and capability checks are honored, that exported statistics contain no raw VAs or per-sample traces, and that one cgroup or container cannot inspect another mm's heat domains. Virtualization: test with KVM host and guest combinations, vPMU enabled and disabled, and confirm that ambiguous precise-address attribution disables the subsystem automatically. Prior-art differentiation: run a comparative study against DAMON region monitoring and classic MGLRU aging to show that the improvement comes specifically from PEBS precise retired-load-latency signals plus deferred no-fault translation, not from generic hotness scoring. Optional alternate-architecture validation can compare PEBS, AMD IBS, and Arm SPE provider fidelity using the same downstream resolver and policy engine to show that the bridge architecture is portable while the primary embodiment remains PEBS-specific.

4. Risks & Mitigations

  • Risk: NMI-path instability or latency spikes due to locking, logging, or allocation. Mitigation: the PEBS ingest path is limited to validity checks plus a constant-time per-CPU ring write using atomic head reservation and per-slot sequence publish; no sleeping locks, printk, allocations, translation, or page references are permitted.
  • Risk: stale samples alias a reused address space after exec or exit. Mitigation: sampled identity is the pair {mm_cookie, mm_exec_epoch}, lookup is performed through an RCU mm-cookie registry, and mmget_not_zero() must succeed before resolution; any lookup miss or epoch mismatch drops the sample.
  • Risk: PFN reuse causes old heat to attach to a newly allocated page. Mitigation: heat keys include alloc_gen sourced from external page_ext metadata that changes on PFN reuse; reclaim, free, migration, and split paths either advance generation or migrate state explicitly.
  • Risk: resolver deadlocks or faults in memory pressure. Mitigation: the resolver runs only in process context, takes mmap_read_lock(), performs an explicit no-fault page-table walk, and drops any sample that would require faulting, pinning, waiting on migration, or handling unstable THP state.
  • Risk: pkey-domain history becomes misleading after pkey_mprotect() or widespread default use of pkey 0. Mitigation: aggregation is always scoped by mm, pkey changes advance a pkey epoch and aggressively decay old buckets, and applications that want meaningful domain separation opt in by assigning distinct pkeys while default mappings simply accumulate in bucket 0.
  • Risk: reclaim fairness or forward progress is degraded by over-weighting hardware samples. Mitigation: heat is consumed only after baseline eligibility and quotas are fixed, all biases are capped, and confidence collapses to zero under unhealthy sampling so the algorithm reverts exactly to baseline MGLRU, THP, and compaction behavior.
  • Risk: raw address sampling leaks privacy or cross-tenant behavior. Mitigation: raw VAs never leave the kernel, only aggregate counters are exposed, enablement requires admin policy plus per-mm opt-in, and vm_pkey is treated strictly as an intra-mm domain tag rather than a cross-mm tenant identifier.
  • Risk: virtualization or PMU multiplexing makes precise-address attribution unreliable. Mitigation: the subsystem checks PMU capability and guest-host attribution state, ignores unsupported guest or host contexts, and forces confidence to zero with full baseline fallback when precise attribution is unavailable.
  • Risk: platform dependence makes the design seem tied to Intel only or vulnerable to prior-art broadening. Mitigation: the narrowest claims are explicitly PEBS-specific and require precise retired-memory semantics plus the deferred no-fault conversion boundary; AMD IBS and Arm SPE are described only as alternate provider embodiments behind the same downstream resolver and policy engine.
