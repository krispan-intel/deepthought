2. Technical Invention Disclosure

2.1 Problem Statement

On x86, PMU overflow interrupts arrive through the local APIC PMI path, typically delivered in NMI context via LVTPC delivery mode. The current Linux perf subsystem evaluates filters only after substantial per-sample work has been completed: perf_event_overflow() populates sample headers, perf_callchain() performs stack unwinding (the dominant cost at 2–6 µs per walk), the ring buffer is reserved via perf_output_begin(), and on PEBS-capable hardware the kernel parses variable-length PEBS records from the DS area via drain_pebs(). When most candidate samples are ultimately discarded by perf filters or BPF-derived predicates, the kernel pays the full interrupt cost in the most latency-sensitive code path on the system. On production systems running continuous fleet-wide profiling (e.g., 99Hz CPU profiling with cgroup-scoped collection on shared Kubernetes hosts), discard rates of 60–90% are common when profiling specific containers. Each wasted PMI costs 2–8 µs of NMI-blocked execution time, directly increasing P99 tail latency for co-located latency-sensitive workloads. The existing perf_bpf_filter infrastructure (perf_bpf_filter__parse, perf_bpf_filter__run) evaluates predicates against a fully materialized perf_sample_data structure, meaning sample header population, PEBS record parsing, and callchain collection have already occurred before any filtering decision. No kernel-internal representation exists that (a) is provably safe in NMI context with respect to reentrancy, lock-freedom, and RCU read-side constraints, (b) is specialized for x86 PMU metadata available before per-counter event iteration, and (c) can serve as an early admission gate to short-circuit the entire sample construction pipeline. FEASIBILITY: The approach requires ~1200 lines of kernel code changes across arch/x86/events/ and kernel/events/, touching intel_pmu_handle_irq(), the perf_event_open() syscall path, and the BPF verifier's perf-event program type. MARKET: Fleet-wide profiling infrastructure (Google-Wide Profiling, Meta Dynolog, AWS CodeGuru Profiler) deployed across millions of cores would benefit directly, reducing profiling overhead by 40–70% on filtered workloads. PROBLEM: The fundamental tension is that NMI context prohibits the synchronization primitives (spinlocks, RCU read locks, sleeping allocations) that generic BPF programs assume, yet the PMI handler must make a fast admit/reject decision using only data available before the expensive per-counter iteration loop begins.

2.2 Prior-Art Gap

Existing mechanisms fail to provide NMI-safe early-exit filtering for x86 PMU interrupts across five specific dimensions: (1) perf string filters (perf_event_attr filter expressions) and BPF attachment via perf_event_set_bpf_prog() both evaluate after full sample materialization—they are post-hoc filters, not admission gates. The perf_bpf_filter__run() function receives a populated perf_sample_data and cannot short-circuit PEBS drain or callchain walk. (2) Generic BPF programs attached to perf events may invoke helpers (bpf_map_lookup_elem, bpf_get_current_comm, bpf_perf_event_read) that are unsafe in NMI context due to lock acquisition (map bucket spinlocks), sleeping semantics (bpf_get_current_comm accesses task->comm with preemption assumptions), or RCU read-side assumptions (bpf_map_lookup_elem enters rcu_read_lock which is a no-op in NMI but may mask reentrancy bugs). (3) Intel Processor Trace (PT) provides hardware-level filtering by IP range (IA32_RTIT_ADDR{0,1}_{A,B}) and CR3, but cannot express task-level, cgroup-level, or event-attribute predicates—it filters WHERE code executes, not WHO is executing or WHICH perf event triggered. (4) ARM SPE (Statistical Profiling Extension) has built-in event filtering via PMSFCR_EL1/PMSEVFR_EL1 but is unavailable on x86 and limited to ARM-defined event types. (5) XRP (Zhong et al., OSDI 2022) demonstrates BPF-at-submission-boundary for NVMe storage, placing bounded predicates at the block I/O layer to avoid full kernel storage stack traversal—an exokernel-inspired admission gate pattern. However, XRP operates in softirq/process context with standard RCU guarantees, not NMI context, and its synchronization model (rcu_read_lock/rcu_read_unlock bracketing) is inapplicable to PMI handlers. (6) FnF-BFT (Gupta et al., 2023) uses bounded predicate evaluation to short-circuit BFT consensus message processing in the network receive fast path, demonstrating that O(1) predicate checks at protocol entry points can eliminate O(N) processing for filtered messages—but operates in network softirq context with preemptible RCU. (7) ExtFUSE (Bijlani et al., ATC 2019) places eBPF in the FUSE request path for passthrough decisions but requires bpf_prog_run_array() with preemption disabled, not NMI-safe execution. The topological void exists at the intersection of 'bounded-predicate admission gating' (established in storage and network domains) and 'NMI-context PMU interrupt handling' (established in x86 perf subsystem)—no prior work bridges these with a concrete synchronization model safe for hardware NMI delivery.

2.3 Proposed Invention

We propose the PMI Filter Image (PFI), a new kernel abstraction consisting of: (A) a restricted BPF program type (BPF_PROG_TYPE_PMI_FILTER) with a verifier enforcement policy that rejects all helper calls, map accesses, and memory loads outside a fixed-layout per-CPU PMI context structure; (B) a JIT compilation target producing an immutable machine-code image bounded to 256 bytes (4 icache lines × 64 bytes/line) that executes exclusively on architecturally-stable metadata; (C) a lifecycle state machine with transitions mapped to concrete kernel hooks; and (D) a NUMA-aware RCU-based image distribution mechanism that avoids all atomic operations in NMI context. The PFI is evaluated at a NEW early-exit check point inserted into intel_pmu_handle_irq(), positioned after the IA32_PERF_GLOBAL_STATUS readout and the counter-freeze writeback (wrmsr to IA32_PERF_GLOBAL_CTRL to zero, which halts all counters and eliminates the SKL150 errata race window where counter status bits could change during the PMI handler), but BEFORE the per-counter iteration loop (the for_each_bit_set loop over status bits that calls x86_pmu.handle_irq(idx) for each overflowed counter). This placement is critical: the filter runs after hardware state is frozen and known-good, but before any per-event callback, PEBS buffer drain, or sample construction begins. The input to the PFI is a read-only per-CPU structure (struct pmi_filter_ctx) containing: (1) frozen_status: the IA32_PERF_GLOBAL_STATUS value read by the PMI handler (64-bit bitmask of overflowed counters); (2) cached_pid/cached_tgid/cached_cgroup_id: per-CPU cached values refreshed at pmu::add() and perf_event_task_sched callbacks, stored in a dedicated cacheline to avoid false sharing; (3) event_config[MAX_COUNTERS]: array of perf_event_attr.config values for each active counter slot, cached at pmu::add() time. For PEBS-format events: when the PFI rejects all counters that triggered overflow, the PEBS buffer drain (drain_pebs/intel_pmu_drain_pebs_nhm) is skipped entirely by checking the filtered-out counter mask against the PEBS-enabled counter mask (IA32_PEBS_ENABLE). PEBS buffer space accounting is maintained by resetting the PEBS index register (IA32_DS_AREA buffer management fields) without parsing individual records, preserving the invariant that the DS area is always in a consistent state. When only some counters are filtered, PEBS drain proceeds normally for the remaining counters. The PFI lifecycle state machine maps to kernel hooks as follows: STAGED (image bytecode loaded via perf_event_open with new PFI_ATTACH_FLAG) → VERIFIED (BPF verifier pass completes, proving: no helpers, no maps, no unbounded loops, max 256 bytes JIT output, all loads within pmi_filter_ctx bounds) → MATERIALIZED (JIT compilation produces immutable image, allocated via kmalloc(256, GFP_KERNEL) in process context) → ACTIVE (image published to per-NUMA-node pointer arrays via rcu_assign_pointer(), triggered by pmu::add() when the event is scheduled onto a CPU) → RETIRED (pmu::del() or perf_event_release_kernel() marks image for deferred free via call_rcu(), monotonic generation counter incremented to prevent ABA). No transition requires GFP_KERNEL allocation or mutex acquisition in atomic context: STAGED→VERIFIED→MATERIALIZED occur in perf_event_open() syscall context (process context, may sleep), ACTIVE→RETIRED occurs in pmu::del() which runs with interrupts disabled but not in NMI (it is called from __perf_event_disable via IPI or schedule context). The NMI fast path performs only: (1) load per-CPU NUMA-local pointer to current PFI image (single cache-line read), (2) indirect call to image code (256-byte bounded execution), (3) test return value (accept/reject bitmask). Security analysis: the BPF verifier enforces that PFI programs cannot load arbitrary kernel memory (all loads are bounds-checked against the fixed pmi_filter_ctx layout, defeating KASLR probing), cannot perform timing-variable operations (no map lookups, no variable-iteration loops—the verifier proves O(1) execution), and the JIT backend applies standard Spectre-v1 mitigations (index masking via array_index_nospec equivalent in generated code, LFENCE barriers after indirect branches on affected microarchitectures as per existing BPF JIT hardening in arch/x86/net/bpf_jit_comp.c).

2.4 Architecture

The PFI architecture spans four kernel subsystems with precisely-defined interfaces: [Layer 1: Userspace API] perf_event_open() extended with PFI_ATTACH_FLAG in perf_event_attr.flags and a new PFI bytecode payload field, allowing userspace profiling tools (perf record, BPF-based profilers) to attach filter predicates at event creation time. The ioctl PERF_EVENT_IOC_SET_PFI provides runtime image replacement without event teardown. [Layer 2: Verifier/JIT] A new BPF program type BPF_PROG_TYPE_PMI_FILTER registered with restricted verifier ops: allowed_helpers=empty_set, max_insns=64 (maps to ≤256 bytes JIT output on x86-64 with typical 3–4 byte average instruction encoding), ctx_access limited to struct pmi_filter_ctx fields with BTF-based offset validation. The x86 JIT backend (arch/x86/net/bpf_jit_comp.c) applies Spectre-v1 mitigations (index masking, LFENCE insertion) and enforces the 256-byte output limit by failing compilation if exceeded. [Layer 3: Image Distribution] Canonical images stored in a global IDR (radix tree) keyed by predicate content hash (SHA-256 of verified bytecode), ensuring deduplication across events with identical filter expressions. Per-NUMA-node arrays of RCU-protected pointers (struct pfi_numa_slot { struct pfi_image __rcu *img; } indexed by image ID) distribute images to CPUs without cross-node cache-line traffic. Memory footprint: for N NUMA nodes and M unique predicates, total overhead is M × 256 bytes (images) + N × M × 8 bytes (pointers) = M × (256 + 8N) bytes. For a 128-CPU / 8-NUMA-node system with 100 unique predicates: 100 × (256 + 64) = 32,000 bytes = 31.25 KB—negligible compared to per-CPU perf_event structures. Image updates use rcu_assign_pointer() for publication and call_rcu() for deferred reclamation with a per-image monotonic generation counter to prevent ABA races (new image always gets generation = max_global_gen + 1, and free callbacks verify generation before reclaiming). CPU hotplug: cpu_dead callback nullifies per-CPU PFI pointers via rcu_assign_pointer(slot->img, NULL); cpu_online callback re-publishes images from the global IDR. [Layer 4: NMI Fast Path] Inside intel_pmu_handle_irq(), after rdmsrl(MSR_CORE_PERF_GLOBAL_STATUS, status) and wrmsr(MSR_CORE_PERF_GLOBAL_CTRL, 0) (counter freeze), a new code block: pfi_img = rcu_dereference_raw(this_cpu_pfi_slot->img); if (pfi_img) { filtered_mask = pfi_img->evaluate(pfi_ctx); status &= ~filtered_mask; } If status becomes zero (all counters filtered), the function returns early, skipping the per-counter loop, PEBS drain, and all sample construction. rcu_dereference_raw() is used (not rcu_dereference) because NMI context cannot hold rcu_read_lock(); safety is guaranteed by the grace-period ordering: call_rcu() in RETIRED transition ensures the image persists through any concurrent NMI. For PEBS: if filtered_mask overlaps with PEBS-enabled counters, the PEBS buffer index is reset (wrmsr to IA32_DS_BUFFER_MANAGEMENT fields) without record parsing, maintaining DS area consistency.

2.5 Implementation Plan

Phase 1 (Kernel Prototype, ~1200 LOC): (a) Define struct pmi_filter_ctx in include/linux/perf_event.h with fields: u64 frozen_status, u32 cached_pid, u32 cached_tgid, u64 cached_cgroup_id, u64 event_config[X86_PMC_IDX_MAX]. Add per-CPU cache population in pmu::add() (x86_pmu_add in arch/x86/events/core.c) and perf_event_task_sched_in(). (b) Register BPF_PROG_TYPE_PMI_FILTER in kernel/bpf/verifier.c with empty allowed_helpers list, max_insns=64, and ctx_access ops restricting loads to pmi_filter_ctx field offsets validated against BTF. (c) Extend perf_event_open() to accept PFI_ATTACH_FLAG: on flag set, invoke bpf_prog_load() for the embedded bytecode, JIT compile, enforce 256-byte output limit, allocate image via kmalloc(256, GFP_KERNEL), insert into global IDR with content-hash key. (d) Implement per-NUMA-node pointer arrays: struct pfi_numa_slot allocated at boot via alloc_percpu equivalent, rcu_assign_pointer publication in pmu::add(), call_rcu reclamation in pmu::del() with generation counter. (e) Insert early-exit check in intel_pmu_handle_irq() (arch/x86/events/intel/core.c) between status readout and per-counter loop. (f) PEBS integration: in the early-exit path, if filtered counters overlap PEBS-enable mask, reset DS buffer index via wrmsr without calling drain_pebs(). Phase 2 (Verifier Hardening): (a) Add Spectre-v1 mitigation pass in x86 JIT for PFI programs: insert array_index_nospec-equivalent masking for ctx field loads, LFENCE after indirect branches. (b) Verify PFI programs cannot encode variable-latency operations (guaranteed by empty helper set + no map access + verifier-proven bounded execution). (c) perf_event_paranoid enforcement: PFI attachment requires CAP_PERFMON (or perf_event_paranoid ≤ 1) to prevent unprivileged predicate injection. Phase 3 (Testing/Validation): perf_test_pfi selftests under tools/testing/selftests/perf_events/, fuzz testing via syzkaller perf_event_open coverage, performance benchmarks on SKL/ICL/SPR microarchitectures. Phase 4 (Upstream Submission): RFC patchset to linux-perf-users@ and bpf@ mailing lists.

2.6 Validation Plan

Validation proceeds across four tiers: (T1) Unit correctness: BPF verifier rejects programs exceeding 64 instructions, programs calling any helper, programs loading outside pmi_filter_ctx bounds; JIT produces ≤256 bytes for all valid programs; image lifecycle state transitions fire correctly with printk-based tracing. (T2) NMI safety: stress test with perf record -a -g -F 999 running concurrently with CPU hotplug operations (echo 0 > /sys/devices/system/cpu/cpuN/online), perf_event_open/close storms from multiple threads (testing ACTIVE→RETIRED transitions under NMI pressure), and lockdep/KCSAN annotations to detect any lock acquisition or data race in the NMI fast path. (T3) PEBS correctness: verify that PEBS buffer accounting remains consistent after PFI-elided events by comparing DS area buffer pointers before and after filtered PMIs (custom kprobe-based test), and ensure that non-filtered PEBS events on the same CPU still produce correct samples. Run perf_fuzzer (Weaver, 2024) with PFI-enabled events to detect PEBS corruption. (T4) Performance: measure PMI latency reduction on Intel Skylake (SKL), Ice Lake (ICL), and Sapphire Rapids (SPR) using rdtsc-bracketed measurements around intel_pmu_handle_irq(). Target: ≥60% reduction in mean PMI handling time when PFI rejects 80% of interrupts. Measure P99 tail latency improvement for a latency-sensitive benchmark (Redis SET/GET) under concurrent 99Hz profiling with cgroup-filtered PFI vs. baseline perf filter. Target: ≥30% reduction in P99 tail latency at 90% discard rate. (T5) Security: verify perf_event_paranoid enforcement blocks unprivileged PFI attachment; verify that PFI programs cannot distinguish kernel address ranges via timing (run KASLR probe test with 1000 crafted predicates and confirm no information leakage above random baseline); verify Spectre-v1 gadget scanner (tools/testing/selftests/bpf/spectre_v1_test) passes on all PFI JIT output.

4. Risks & Mitigations

  • RISK: Counter-freeze writeback (wrmsr to PERF_GLOBAL_CTRL) adds ~20-40ns to every PMI, even for non-filtered events. MITIGATION: The freeze writeback is already performed by existing intel_pmu_handle_irq() implementations on modern kernels (since commit 'perf/x86: freeze counters on PMI'); the PFI insertion point leverages this existing freeze rather than adding a new one. On microarchitectures without hardware counter-freeze (pre-Skylake), PFI is disabled and falls through to the standard path.
  • RISK: SKL150 errata—IA32_PERF_GLOBAL_STATUS may report stale values if read before counter-freeze completes. MITIGATION: The PFI evaluation point is strictly after the PERF_GLOBAL_CTRL←0 writeback, which serializes the status register on affected microarchitectures. The filter treats the frozen status as authoritative; if a counter overflow is missed due to the race window preceding freeze, it will trigger on the next PMI (no lost events, only deferred delivery).
  • RISK: PEBS buffer reset without record parsing could desynchronize the DS area if a concurrent PEBS record write completes between the status read and the buffer reset. MITIGATION: Counter-freeze (PERF_GLOBAL_CTRL←0) halts PEBS record generation before the PFI runs. The buffer index reset is performed atomically via wrmsr to IA32_PEBS_INDEX (or DS area management fields on older microarchitectures). A post-reset consistency check verifies buffer_base ≤ buffer_index ≤ buffer_max before returning.
  • RISK: Per-CPU cached pid/tgid/cgroup_id becomes stale if a context switch occurs between the cache refresh and the next PMI. MITIGATION: Cache refresh is hooked into both pmu::add() and perf_event_task_sched_in(), ensuring the cache is updated at every scheduling event. The maximum staleness window is one scheduler tick (~1-4ms at typical HZ=250-1000), which is acceptable for profiling use cases where samples are already attributed at statistical granularity. For cgroup_id, the value is stable across the task's cgroup membership (changes require explicit cgroup migration, which triggers sched callbacks).
  • RISK: RCU grace period stall could delay image reclamation, causing unbounded memory growth during rapid perf_event_open/close cycling. MITIGATION: Image deduplication via content-hash keying means M unique predicates produce at most M canonical images regardless of event count. The call_rcu() path is the same used by existing BPF program freeing (bpf_prog_free_deferred). A high-watermark check in perf_event_open() rejects PFI attachment if the global image count exceeds a configurable limit (default: 1024).
  • RISK: Unprivileged users could craft PFI predicates that leak KASLR information via timing side channels (e.g., branch prediction differences based on kernel address comparisons). MITIGATION: (1) PFI attachment requires CAP_PERFMON, same as BPF perf-event program attachment. (2) The verifier restricts all loads to the fixed-layout pmi_filter_ctx—no arbitrary kernel memory access is possible. (3) JIT applies constant-time index masking (array_index_nospec pattern) to all context field loads. (4) The 256-byte image bound and absence of variable-iteration loops ensure constant-time execution, eliminating timing-based information leakage.
  • RISK: Spectre-v1 gadgets in JIT-generated filter code could be exploited via speculative execution. MITIGATION: The x86 JIT backend for BPF_PROG_TYPE_PMI_FILTER applies the same Spectre-v1 hardening as existing BPF JIT (implemented in arch/x86/net/bpf_jit_comp.c): index masking after every array-style load, LFENCE insertion after indirect branches on affected microarchitectures. Additionally, PFI images contain no indirect jumps (verifier rejects them), eliminating Spectre-v2 attack surface.
  • RISK: CPU hotplug races—an NMI could dereference a PFI pointer on a CPU undergoing offline transition. MITIGATION: The cpu_dead callback nullifies the per-CPU PFI pointer via rcu_assign_pointer(slot->img, NULL) before the CPU is fully offline. Since rcu_assign_pointer includes a release barrier, any subsequent NMI on the dying CPU (possible during the teardown window) sees the NULL pointer and falls through to the standard unfiltered path. The cpu_online callback re-publishes images from the global IDR after the CPU is fully operational.
