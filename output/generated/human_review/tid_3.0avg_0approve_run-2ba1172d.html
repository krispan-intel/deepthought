2. Technical Invention Disclosure

2.1 Problem Statement

After dma_map_sg(), dma_map_sgtable(), dma_map_page_attrs(), or equivalent streaming DMA mapping, many Linux drivers still must derive the final hardware descriptor runs they will program into a submission ring. That step is not solved by page-level bio merging or by dma_map_sg()'s reduction in nents, because a mapped DMA extent can still require further splitting or folding according to descriptor-specific limits that are only meaningful at descriptor-programming time: maximum bytes per descriptor, no-crossing boundary rules such as 64 KiB or 1 MiB windows, required start-address alignment, required length granularity, optional final-tail exceptions, and device DMA address-mask limits. Existing drivers therefore embed bespoke post-map loops that mix contiguity testing, overflow handling, budget estimation, and hardware packing. The result is duplicated branch-heavy code, inconsistent treatment of 32-bit dma_addr_t and wrap hazards, and no uniform way to ask for the exact descriptor count required for an already-mapped request before ring reservation.

2.2 Prior-Art Gap

dma_map_sg() and related DMA APIs already coalesce adjacent entries when the DMA layer can present a single mapped segment, and blk/bio merge policies already merge requests before mapping. Those mechanisms are necessary but insufficient. They operate before final descriptor construction and do not provide a deterministic post-map partition under a formal descriptor contract. dma_set_max_seg_size() and dma_set_seg_boundary() influence mapping behavior, but they do not solve the full post-map problem because they do not yield an exact descriptor plan after IOMMU/SWIOTLB transformations, do not provide a count-then-emit API, and do not cover the full combination of start alignment, length granularity, tail policy, exact hardware descriptor budgeting, and explicit no-wrap behavior on narrow DMA address masks. scatterlist utilities iterate segments but do not standardize a reusable post-map folding result. The helper disclosed here fills that exact gap: given the already-mapped DMA stream and a formal legality contract, it returns the exact descriptor-safe run partition the driver will consume.

2.3 Proposed Invention

Introduce a three-part subsystem for post-map DMA run folding. First, a generic extent view converts the already-mapped request into an immutable sequence of DMA address/length pairs, regardless of whether the caller starts from a scatterlist, sg_table, or single mapped page. Second, a generic scalar planner/emitter walks that immutable sequence and computes the unique descriptor-safe partition implied by the constraint contract; it can either count runs only or emit them into caller-provided storage, and it is valid in atomic, softirq, or process context because it allocates nothing and uses no SIMD/FPU state. Third, an optional x86 AVX-512 classifier may be invoked only through a separate process-context API to precompute candidate break bits in 8-lane batches; the classifier never emits runs itself and never becomes semantically authoritative. The scalar emitter remains the single source of truth for boundaries, ordering, and error handling, ensuring bit-identical outputs with or without AVX-512.
