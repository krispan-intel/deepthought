1. Executive Summary

One-Liner

A Linux blk-mq/rq_qos extension evaluates a single non-sleepable, helper-free BPF program at a pre-tag block submission hook over a versioned pointer-free context containing only stable scalar queue and blkcg handles plus cached x86 stall-class scalars, and the program returns a bounded u16 policy_cookie that rq_qos resolves through one RCU-published composite bundle containing the ABI header, program pointer, per-hook cookie bounds, per-blkcg overlays, and preapproved atomic-safe action profiles.
Novelty: The inventive step is the specific selector/enforcer split for block I/O control: the BPF program is intentionally reduced to an exokernel-like naming plane that cannot call helpers, read maps, access pointers, or mutate

2. Technical Invention Disclosure

2.1 Problem Statement

Current Linux block control is fragmented across blkcg, io.latency, io.cost, ioprio, schedulers, and ad hoc rq_qos modules, none of which provides a kernel-stable per-submission programmable boundary that can combine request semantics, queue state, tenant identity, and x86 microarchitectural execution state while remaining safe for the blk-mq hot path. Existing experimentation often depends on unstable internal structs, raw kernel pointers, or user-space control loops that are too slow and too coarse for high-rate NVMe workloads. The practical need is strongest when identical I/O sizes should receive different admission treatment depending on whether the issuing CPU is frontend-bound, backend-memory-bound, core-bound, mispredict-heavy, or retiring efficiently, because the CPU-side cost of block submission and completion changes even when the storage request itself does not.

2.2 Prior-Art Gap

Prior art does not teach the specific block-layer boundary claimed here. tc, XDP, LSM, tracing, and cgroup-BPF programs may return small integers or verdicts, but they either expose richer mutable contexts, permit helper/map access, or distribute semantics across multiple kernel objects such that update and enforcement are not generation-atomic at a block-specific rq_qos boundary. Existing block controls such as blkcg weights, io.cost, io.latency, BFQ, kyber, and mq-deadline offer static or controller-specific scheduling heuristics, not a generic pointer-free BPF selector whose result is only a bounded policy_cookie resolved by rq_qos through one composite bundle containing {ABI header, prog pointer, cookie bounds, action profiles, and overlays}. perf and BPF observability integrations expose telemetry, but they do not provide an atomic-safe block enforcement plane tied to x86 stall classes with explicit no-mixed-generation semantics, stable handle rules, or queue-teardown ordering.

    Kernel Hardliner REVISE ★☆☆☆☆ YC:1
    • Locking/preemption context is underspecified for both proposed hooks (throttle(bio) and issue(request)). You must state exactly which contexts they run in (process vs softirq vs under hctx/queue spinlocks) and prove the BPF run + enforcement cannot sleep, cannot take mutexes, and is bounded enough to be acceptable under those locks. If any hook can run under a spinlock/irq-disabled region, you should explicitly wrap config access in rcu_read_lock() and avoid anything that can trigger slowpaths (e.g., static keys, ftrace, debug checks).
    • The ‘single RCU-published bundle prevents mixed generations’ claim needs a concrete publication protocol. Specify: (1) updater serialization (rq_qos mutex + blk_mq_freeze_queue()), (2) rcu_assign_pointer() ordering for the bundle pointer, (3) reader-side rcu_dereference() + rcu_read_lock() scope across bpf_prog invocation and action lookup, and (4) teardown via call_rcu() ensuring bpf_prog, overlay tables, and any execmem text remain valid until after the grace period. Right now it’s a slogan, not a verified concurrency story.
    • blkcg identity + overlay safety is not resolved. A ‘slot+generation overlay’ must explicitly handle blkcg offline/reuse: you need a stable key (e.g., cgroup_id plus a non-reused generation/serial) and a lifetime rule (css_tryget_online()/css_put() or equivalent) for any referenced overlay entry. Otherwise you risk either UAF (overlay points at freed blkcg state) or misattribution (slot reused for a different blkcg).
    • The optional ‘topology-local immutable JIT replicas’ is very risky with current BPF prog lifecycle and execmem model. If you intend multiple text copies, you must define how you keep semantics identical while preserving BPF metadata (prog ID, aux data, BTF, kallsyms, exception tables, JIT hardening, IBT/CET) and how you safely switch the entrypoint under RCU without violating bpf_prog reference counting. A safer v1 is: one prog, one JIT image; defer replicas until you can demonstrate a correct upstreamable execmem/text-poking story.
    • PMU/Topdown collection details are not kernel-feasible as written. Describe exactly how you create/read events in-kernel (perf_event_create_kernel_counter vs user-provided pinned events), how you handle multiplexing (time_enabled/time_running), CPU hotplug, and cgroup migration. Also justify the sampling period and scope (per-CPU vs per-queue vs per-cgroup): a ‘per-blkcg’ worker model can explode in overhead and contention on large fleets.
    Prior Art Shark REVISE ★★★★★ YC:2

    Security Guardian REVISE ★★★★★ YC:2
    • Side-channel risk: exposing Topdown-derived stall classes (even quantized) to a programmable policy in the submission path can leak co-tenant CPU activity. Gate attach/use behind strict capability checks and perf_event_paranoid policy, ensure signals are strictly cgroup-scoped (not systemwide), and consider coarse quantization + update-rate limiting + optional noise to reduce cross-tenant inference.
    • Blkcg identity/lifetime correctness: a “stable scalar blkcg handle” must be defined as a non-UAF, non-reuse-safe identifier (e.g., cgroup_id + css serial/generation). Document reuse semantics and ensure bundle overlay lookup defends against ID reuse (generation validation) to avoid misattribution and policy bypass after cgroup deletion/recreation.
    • RCU bundle teardown safety: specify exact publication/retirement rules so readers never call into freed bpf_prog or access freed action tables. Ensure refcounting/pinning of bpf_prog, execmem/JIT text (if any), and overlay tables; retire with call_rcu() only after all dependent objects are unreachable; integrate queue freeze/unfreeze ordering to prevent use-after-free during elevator/rq_qos transitions.
    • Speculation hardening must be complete: cookie bounds checking and array_index_nospec() must cover every index derived from the BPF return value (base table, per-blkcg overlays, per-hook namespaces). Also define behavior for out-of-range cookies (clamp vs reject vs default) and reserve cookie 0 semantics to avoid gadgety “error as index 0” ambiguity.
    • Compatibility/ABI guarantees: a fixed “64-byte pointer-free ctx” is brittle unless it includes explicit ctx_size/version fields and reserved padding with an append-only rule. Prefer a BTF-described ctx with size/version negotiation so old programs remain valid when fields are added; require verifier to reject programs that assume a larger ctx than provided.
