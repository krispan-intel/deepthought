2. Technical Invention Disclosure

2.1 Problem Statement

When CXL.mem is surfaced as system RAM on x86, the latency and bandwidth seen by a CPU depend not only on the NUMA node but on the specific CXL HDM decoder path, root-port attachment, host-bridge traversal, and interleave membership serving the target HPA range. Existing Linux scheduler placement is CPU-topology-aware and NUMA-aware, but it is not decoder-aware. On asymmetric multi-root-port systems, or on systems where firmware collapses multiple CXL paths into the same NUMA node or exposes non-discriminating HMAT data, a task may wake on a CPU that satisfies node-level policy yet is materially farther from the dominant decoder serving the mm's hot CXL-backed pages. The kernel needs a low-overhead, scheduler-safe way to translate decoder-granular CXL topology into a wakeup-time placement hint that never overrides affinity or policy, never fabricates latency numbers, never performs hot-path HPA translation in the scheduler, and remains valid for CLONE_VM address spaces whose threads may have partially divergent working sets.

2.2 Prior-Art Gap

Linux prior art already covers adjacent but different mechanisms: mempolicy constrains allocation, HMAT describes memory performance classes, memory tiers drive migration priority, and sched_numa_balancing learns preferred_nid at node granularity. Those mechanisms do not expose a scheduler-safe, wakeup-only discriminator for the case where multiple CXL HDM decoders map into the same NUMA nid and the same memory tier yet differ in effective CPU proximity because they sit behind different root ports or host bridges. The novelty anchor is therefore not generic NUMA preference; it is decoder-granular locality compiled into stable scheduler cookies and applied only as a post-policy tie-break in wakeup placement.

2.3 Proposed Invention

The invention introduces a generation-safe CXL locality hint path for Linux CFS wakeups on x86. A CXL mapping plane tracks HPA ranges backed by CXL regions and exports decoder_key metadata for each range, including multi-target interleave composition. Existing AutoNUMA follow-up events and folio migration completion events are sampled at bounded rates and appended to per-CPU batches keyed by mm_cookie and PFN. A fold worker resolves PFN/HPA to decoder_key, applies deduplication and decay, and updates small per-mm decoder counters. A compile worker projects decoder counters through the current decoder-to-scheduler locality map, aggregates weights per stable scheduler locality cookie, selects the top-K cookies, quantizes bias and confidence, and publishes an immutable mm-owned snapshot via RCU. Each fair-class task may cache a refcounted pointer to the current snapshot plus a per-task memory-dependence TTL expiry. During select_task_rq_fair() wakeup placement, if and only if normal scheduler logic has not already produced a unique winner and the candidate groups are equivalent within a fixed delta, the scheduler consults the cached snapshot and prefers the best-scoring cookie among the equivalent candidates. The hint cannot admit an otherwise illegal CPU and cannot override cpuset, p->cpus_ptr, cgroup, uclamp, asymmetric-capacity, RT/DL exclusion, wake_affine, select_idle_sibling, or a unique NUMA result.

2.4 Architecture

Control/data split:
1) Mapping plane: CXL topology notifier -> build RCU interval table of HPA ranges -> decoder_key entries.
2) Sampling plane: AutoNUMA follow-up or migration completion -> per-CPU batch append <mm_cookie, task_cookie?, pfn, epoch, weight>.
3) Compile plane: delayed worker -> resolve pfn/hpa to decoder_key -> decay and fold per-mm counters -> project to scheduler locality cookies -> publish immutable snapshot.

2.5 Implementation Plan

1) Add mm-owned hint state and task-cached snapshot pointers with refcount and RCU lifetime rules. 2) Add an RCU-protected HPA-range-to-decoder_key table built from CXL region and HDM decoder topology notifications. 3) Add per-CPU sample buffers and delayed fold workers for AutoNUMA follow-up and folio migration completion records. 4) Add the compile worker that applies deduplication, decay, decoder-to-cookie projection, top-K selection, confidence scoring, and immutable snapshot publication. 5) Add rebuild-time locality-cookie assignment for wakeup-relevant sched_groups and generation bumping on sched_domain rebuild/hotplug. 6) Add the wakeup-only advisory branch in select_task_rq_fair() guarded by static key, TTL, confidence, and generation checks. 7) Add tracepoints and debugfs counters for sample acceptance, fast rejects, generation mismatches, TTL skips, and winning-cookie selection. 8) Keep feature default-disabled behind a sched feature flag or static key until platform validation completes.

2.6 Validation Plan

Microbenchmarks: wakeup-heavy ping-pong, RPC worker pools, and epoll-based services whose hot pages are placed in CXL-backed memory behind different root ports but the same nid. Compare baseline CFS versus the feature on wakeup latency, remote decoder hit rate, and throughput. Correctness checks: verify zero cpuset or affinity violations, zero change to RT/DL behavior, and zero change on systems with no CXL memory. Reconfiguration tests: exercise CXL hot-add/hot-remove, decoder interleave changes, and CPU hotplug to confirm immediate generation-based fast reject and no stale dereference. Shared-mm tests: mixed CPU-bound and memory-bound sibling threads must show bias only for tasks with a live TTL. Observability: add a tracepoint carrying <pid, locality_cookie, confidence, ttl_valid, skipped_reason> and debugfs counters for no-CXL, null-cache, TTL-expired, generation-mismatch, non-equivalent-candidate, and hint-applied paths so decoder selection can be correlated with measured latency without adding scheduler hot-path instrumentation.

3. Patent Claims

  1. A method for operating a Linux fair scheduler on an x86 platform with CXL memory, comprising: maintaining an HPA-range-to-decoder mapping table for CXL HDM decoders; accumulating sampled residency records for an mm from AutoNUMA follow-up events and folio migration completion events; compiling the sampled residency into an immutable per-mm snapshot containing one or more stable scheduler locality cookies, bias values, confidence, and topology generations; caching a reference to the snapshot for a task; and, during a wakeup-path execution of select_task_rq_fair(), selecting among already legal and otherwise equivalent candidate CPUs using the snapshot only when a per-task memory-dependence TTL is valid and the topology generations match.
  2. The method of claim 1, wherein the stable scheduler locality cookie is assigned during sched_domain rebuild as a value tuple including a scheduler locality generation and a group sequence, and the method compares cookies by value without dereferencing sched_group pointers in the wakeup path.
  3. The method of claim 1, wherein the decoder mapping table stores a decoder_key that includes a CXL topology generation, a region cookie, a decoder cookie, interleave width, interleave granularity, and an ordered target vector hash for multi-target interleave.
  4. The method of claim 1, wherein multi-target interleave members are projected into scheduler locality cookies by summing member weights that map to the same cookie, thereby allowing several decoders to collapse into one scheduler locality entry deterministically.
  5. The method of claim 1, wherein the scheduler applies the snapshot only after cpuset, task affinity, cgroup, asymmetric-capacity, wake_affine, idle-sibling, and existing NUMA selection logic have either selected a unique winner or reduced the decision to equivalent candidates.
  6. The method of claim 1, wherein event-side residency sampling is bounded to per-CPU batch append operations and decoder resolution is deferred to a fold worker, thereby avoiding decoder lookup and per-mm atomic counter updates on a latency-sensitive fault or wakeup path.
  7. The method of claim 1, wherein the per-mm snapshot is owned by mm_struct, published by RCU, reclaimed by call_rcu after refcount release, and accessed in select_task_rq_fair() only through a task-cached pointer without dereferencing task->mm.
  8. The method of claim 1, wherein a task sharing an mm with other tasks uses a separate per-task TTL gate so a CPU-bound wakeup sharing the mm does not inherit locality bias generated by a different memory-active thread.
  9. A Linux kernel system comprising: a CXL topology control plane that assigns generationed decoder cookies and HPA range metadata; a scheduler locality control plane that assigns generationed locality cookies to sched_groups; an mm-owned snapshot compiler that projects decoder residency into top-K locality bias entries; and a wakeup-only CFS tie-breaker that consumes the compiled snapshot only when legality, equivalence, TTL, confidence, and generation checks pass.

4. Risks & Mitigations

  • Risk: stale scheduler-cookie or decoder-cookie mappings after sched_domain rebuild or CXL reconfiguration could steer wakeups incorrectly. Mitigation: use separate cxl_topology_gen and sched_locality_gen fields, rebuild cookies on each topology event, and ignore any snapshot whose generations do not match the current global values.
  • Risk: wakeup or AutoNUMA latency regression from extra accounting. Mitigation: make event-side work append-only and per-CPU, defer decoder resolution and aggregation to workers, and provide a static-key fast reject when no CXL memory is present or the feature is disabled.
  • Risk: use-after-free from task-cached snapshot pointers. Mitigation: mm owns the state, task caches hold explicit references, publication and dereference are under RCU, and reclamation is via call_rcu after refcount reaches zero.
  • Risk: migration-heavy workloads could be over-represented. Mitigation: deduplicate records within an epoch by folio and mm, cap per-mm accepted samples, and decay counters aggressively enough to follow current locality rather than historical churn.
  • Risk: decoder-locality bias could override scheduler fundamentals. Mitigation: restrict the feature to select_task_rq_fair() wakeups, require equivalent candidates, never widen candidate masks, and preserve the scheduler's existing deterministic order when the bias difference is below one quantization step.
  • Risk: shared-mm snapshots could mis-steer unrelated threads. Mitigation: add a per-task memory-dependence TTL gate set only by task-local qualifying memory activity; expired TTL means no bias even if the shared mm snapshot is valid.
  • Risk: prior-art overlap with generic NUMA-aware placement. Mitigation: anchor claims on decoder-granular locality, stable scheduler-cookie compilation, weighted interleave projection, and wakeup-only consumption after existing NUMA and affinity rules, specifically for the same-nid same-tier different-decoder case.
