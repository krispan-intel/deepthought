1. Executive Summary

One-Liner

A Linux blk-mq controller records sampled submit CPUs at request allocation and sampled completion CPUs at request end using tag-indexed telemetry slots, computes integer locality scores over LLC, package, NUMA, and APIC-cluster domains, and commits queue-map changes plus only-lawful IRQ affinity requests in rollback-capable epochs that honor managed-interrupt, hotplug, queue-reinit, and effective-affinity semantics.

2. Technical Invention Disclosure

2.1 Problem Statement

On multi-socket and multi-LLC x86 systems, Linux block I/O completions often run on CPUs that are remote from the CPU that allocated and submitted the request. The result is cross-LLC cacheline movement, extra coherency traffic, avoidable NUMA crossings, and interference with housekeeping or isolated CPU policy, all of which inflate p99 latency and waste CPU cycles. Existing static blk-mq queue mapping, irqbalance, and manual affinity pinning are each partial: they do not observe request-granular submit-versus-complete locality inside blk-mq, they do not serialize queue-map changes with queue freeze/quiesce and queue reinit, and they do not distinguish between deterministic IRQ steering, hint-only steering, and non-steerable legacy or managed IRQ classes using kernel-owned semantics. The practical problem is therefore a missing closed loop that can improve completion locality while remaining lawful under genirq, safe under cpuhp and tagset reinit, and respectful of x86 APIC topology limits.

2.2 Prior-Art Gap

Known Linux mechanisms expose ingredients but not the claimed bridge. blk-mq already computes queue maps from topology, housekeeping, and queue counts, but that mapping is largely static and not driven by observed submit-to-complete locality. Userspace irqbalance and BPF tracing can observe CPU placement or IRQ rates, but they lack a legal in-kernel actuation path tied to blk-mq queue lifecycle, cannot freeze/quiesce and refresh queue maps atomically with device state, and cannot compare requested versus effective IRQ affinity after irqdomain filtering inside the same control loop. Existing tracing approaches also tend to stamp richer per-request metadata or export events to user space, which is overhead-heavy and race-prone for a control system. The invention distinguishes itself by introducing a blk-mq-owned telemetry data structure keyed by rq->tag, an optional driver-to-blk-mq hctx-to-IRQ binding bridge, an explicit APIC capability class that affects permissible domain granularity, and a testable commit/rollback state machine that unifies queue-map refresh, lawful affinity requests, effective-mask readback, hotplug serialization, and rollback quarantine. That combination is the non-obvious bridge beyond 'measure locality and then pin things.'

2.3 Proposed Invention

The invention adds a blk-mq locality controller per request_queue or tag_set. The controller owns: (1) a topology snapshot with precomputed masks for online CPUs, housekeeping CPUs, cpuset-effective CPUs, queue-feasible CPUs, LLC domains, package domains, NUMA-local domains, and, when exported by x86 topology code, APIC-cluster domains; (2) a tag-indexed telemetry array per hctx, where each slot stores a compact sample record such as {submit_cpu, epoch_id, sample_bit}; (3) per-CPU, per-hctx locality counters for relations {same_cpu, same_llc, same_package, same_apic_cluster, remote_package, remote_numa}; (4) an hctx-to-IRQ binding table obtained through an explicit optional driver callback, so IRQ identity is learned outside the fast path; and (5) an epoch state machine. The control objective is explicit and integer-valued: maximize legal same-LLC and same-package completion ratios, penalize remote-package and remote-NUMA completions, and preserve fairness across hctxs while never violating online, housekeeping, cpuset, managed-IRQ, or queue-feasibility constraints. A representative score for each candidate domain is 3*(same_cpu%) + 2*(same_llc%) + 1*(same_package%) + 1*(same_apic_cluster%) - 2*(remote_package%) - 4*(remote_numa%), evaluated only when a minimum sample floor is reached. Candidate domains are ordered by lawfulness and granularity: exact CPU, APIC cluster when extapic/x2APIC-style cluster data is exported, LLC, package, then NUMA-local housekeeping set. The controller changes placement only if score improvement exceeds a hysteresis threshold, sample count exceeds a floor, p99 latency guardrails from existing blk-stat telemetry are not violated, and a minimum dwell time since the last commit has elapsed. The x86 APIC capability class is operationally defined, not rhetorical: CAP_LEGACY_LINT means legacy LINT/ExtINT or equivalent local-APIC pin routes are excluded from steering; CAP_TOPO_BASIC means only LLC/package/NUMA domains are trusted; CAP_EXTAPIC_CLUSTER means extended APIC-ID or equivalent x86 topology export allows sub-package APIC-cluster domains; and CAP_REMAP_CONSTRAINED means interrupt-remapping or virtualization consistently collapses requested masks on effective-mask readback, so IRQs are treated as hint-only and queue-only optimization dominates. IRQ classification is executable: deterministic requires irq_can_set_affinity(irq) to be true, the IRQ not to carry IRQD_AFFINITY_MANAGED or IRQD_MANAGED_SHUTDOWN semantics, the IRQ not to be per-CPU, the source not to be legacy MP_lintsrc_info/ExtINT/LINT derived, and the post-request effective mask to equal the requested mask. Hint-only covers managed MSI/MSI-X vectors or any vector whose requested mask is accepted but read back as a strict subset; excluded covers legacy LINT/ExtINT, per-CPU IRQs, torn-down IRQs, and any source outside the driver-exposed binding bridge. Partial acceptance does not cause flip-flopping: the IRQ is marked hint-only for a cooldown interval, the queue-map portion of the epoch may remain active if still beneficial, and full deterministic retry is suppressed until topology generation changes or the cooldown expires.

Key kernel touchpoints are explicit. Submit CPU is captured at blk_mq_get_request() or the first request-initialization point where rq->tag and rq->mq_hctx are stable. Completion CPU is captured at blk_mq_end_request() before tag release. IRQ identity is not looked up in the fast path; instead, a narrow driver binding callback populates hctx-to-IRQ tables during controller attach or queue reinit, which keeps the telemetry path bounded and makes the blk-mq/genirq bridge explicit. Queue-map actuation always precedes IRQ actuation so the freeze window stays short; any transient mismatch only reduces optimization quality, not correctness, because all requested masks are subsets of the same lawful completion domains.

2.5 Implementation Plan

1. Controller attachment and data structures: add a per-queue controller object allocated at blk-mq queue setup time. It contains a mutex for state serialization, an RCU-published active epoch pointer, topology generation counters, per-hctx sample-slot arrays indexed by rq->tag, per-CPU locality counters, and a dedicated freezable kthread_worker pinned to a housekeeping CPU. The sample slot is intentionally compact, for example 8 bytes: u16 submit_cpu, u16 epoch_id, u8 sampled, u8 reserved, u16 checksum or domain class. This avoids growing struct request and gives a clear ownership model. Slots are allocated per hctx based on queue depth and reallocated only in reinit paths. 2. Fast-path telemetry hooks: in blk_mq_get_request() or the first point where request allocation and hctx assignment are stable, evaluate a static-key-gated sampler. Default sampling is 1/64 requests; it may temporarily increase to 1/16 if the queue is under-sampled. For sampled requests, write submit_cpu = raw_smp_processor_id(), current epoch_id, and sampled=1 into slots[hctx][rq->tag]. In blk_mq_end_request(), read the slot before tag release, compute the relation between submit CPU and completion CPU using precomputed cpu_to_llc, cpu_to_pkg, cpu_to_numa, and cpu_to_apic_cluster tables, then update only per-CPU counters with this_cpu_inc() / this_cpu_add(). No sleeping, no dynamic allocation, and no broad locking occur here. 3. Optional hctx-to-IRQ bridge: define a narrow driver registration surface such as enumerate_completion_irqs(hctx, irqs[], max) so devices that can identify the MSI/MSI-X vector(s) serving each hctx expose that mapping at attach time and after reinit. Drivers that do not implement the bridge still benefit from queue-map optimization, but not deterministic IRQ actuation. This bridge is what makes the subsystem boundary explicit and testable. 4. APIC capability classification: at controller initialization and whenever topology generation changes, classify the platform into CAP_LEGACY_LINT, CAP_TOPO_BASIC, CAP_EXTAPIC_CLUSTER, or CAP_REMAP_CONSTRAINED. The classification uses kernel-exported x86 topology exposure, including x2APIC or X86_FEATURE_EXTAPIC-style extended APIC-ID availability when present, and runtime observation of whether effective affinity requests round-trip exactly. CAP_EXTAPIC_CLUSTER permits APIC-cluster candidate domains. CAP_LEGACY_LINT excludes any IRQ discovered as a legacy MP_lintsrc_info or equivalent LINT/ExtINT route from steering. CAP_REMAP_CONSTRAINED forces hint-only treatment even when irq_set_affinity() nominally succeeds. 5. Control algorithm: every 250 ms to 1 s, depending on queue depth, the worker snapshots counters under ctl_lock while holding cpus_read_lock() for stable online topology. It computes per-hctx candidate scores using only integer arithmetic and bitmap/cpumask operations. The controller changes a domain only when (a) at least 2048 sampled completions or an equivalent statistically meaningful floor are present for that hctx, (b) predicted score improves by at least 8%, (c) same-LLC or same-package rate improves materially, (d) blk-stat p99 does not regress by more than 4% over the previous dwell, and (e) at least 5 s have elapsed since the last accepted epoch. A fairness cap prevents any chosen domain from receiving more than 1.25x the mean completion load of sibling hctxs. The algorithm is intentionally integer/bitmap based so it can use cpumask_and(), bitmap_weight(), and word-at-a-time reductions; it does not require floating-point or kernel SIMD execution. 6. IRQ classification and actuation: for each hctx-bound IRQ, classify it before actuation. Deterministic requires irq_can_set_affinity(), absence of IRQD_AFFINITY_MANAGED and IRQD_MANAGED_SHUTDOWN semantics, non-per-CPU delivery, non-legacy source, non-quarantined state, and an exact post-request effective mask. Hint-only includes managed MSI/MSI-X or any vector whose requested mask collapses on readback. Excluded includes LINT/ExtINT/MP_lintsrc_info-derived routes, per-CPU interrupts, absent irq_data, and torn-down vectors. Actuation occurs only on the worker thread. The commit loop is bounded, e.g. at most 16 queue changes or 8 IRQ affinity operations per pass, followed by cond_resched(), so housekeeping CPUs are not trapped in long lock chains. 7. Commit sequence and locking: the worker enters PREPARE under ctl_lock and cpus_read_lock(). For queue-map commit, it executes freeze -> quiesce -> swap or refresh blk-mq queue-domain maps -> publish the new epoch pointer under RCU -> unquiesce -> unfreeze. Only after the queue-map phase completes does it enter COMMIT_IRQ. Each deterministic IRQ is then programmed in process context, effective affinity is read back, and the result is recorded in the epoch status. This ordering keeps the freeze window short and avoids blocking queue progress on affinity lock chains. Because all requested masks are subsets of legal completion domains, temporary old IRQ placement during COMMIT_IRQ is safe, only suboptimal. 8. Rollback and lifecycle handling: CPU hotplug callbacks and queue-reinit notifications never perform heavy work. They only increment topology_gen, mark the active epoch stale, and queue the worker. The worker then enters ROLLBACK or REBUILD. On rollback, the controller freezes and quiesces the queue, restores the last-known-safe queue map or a controller-disabled map, publishes a queue-only or disabled epoch, unfreezes, and clears any still-pending deterministic IRQ operations. Sample-slot arrays are RCU-swapped and freed only after queue drain plus synchronize_rcu(), preventing completions from touching freed memory. IRQ teardown is detected by failed irq_data lookup or device queue teardown callbacks and forces the affected IRQ into excluded state. Partial acceptance causes per-IRQ quarantine for a cooldown interval, preventing oscillation. 9. Concurrency proof summary: ctl_lock serializes solve/commit/rebuild; cpus_read_lock stabilizes online CPU masks during snapshot and commit; RCU protects readers in the fast path from epoch-plan replacement; queue freeze/quiesce prevents queue-map races with blk-mq dispatch; and bounded process-context batches with cond_resched() keep affinity setting lawful and scheduler-friendly.

2.6 Validation Plan

Validate on multi-socket x86 servers with NVMe and virtualized block devices under four classes of experiments. First, correctness and safety: exercise CPU hotplug, queue reinit, suspend/resume, device remove/add, MSI-X vector shrink/expand, and cpuset or isolcpus changes while fio workloads run; confirm no use-after-free in sample-slot arrays, no sleeping in atomic path, and no affinity writes from interrupt context. Second, control-loop efficacy: run mixed random-read, random-write, and database-like traces at multiple queue depths; measure same-CPU, same-LLC, same-package, and remote-NUMA completion ratios, along with p50/p99 latency and CPU-per-IO. Third, classification behavior: inject managed MSI/MSI-X, forced effective-mask collapse, and legacy or non-steerable IRQ cases; verify that deterministic, hint-only, and excluded classifications match expectations and that partial acceptance enters quarantine rather than oscillation. Fourth, overhead: with controller disabled, enabled but idle, and enabled under high load, measure IOPS regression, additional instructions per I/O, kthread CPU usage, memory consumed by sample slots, and scheduling latency of housekeeping CPUs. Success criteria are bounded fast-path overhead below a low single-digit percentage at default sampling, statistically significant reduction in remote-NUMA completions, p99 improvement on at least one multi-socket storage profile, and zero correctness regressions under hotplug and queue-reinit stress.

4. Risks & Mitigations

  • Risk: Fast-path overhead from telemetry could reduce peak IOPS. Mitigation: gate with a static key, sample at a bounded rate, keep per-request metadata in compact tag-indexed slots, and use only this_cpu_* counter updates in completion.
  • Risk: Queue-map and telemetry arrays could race with queue reinit or tagset changes. Mitigation: serialize lifecycle operations with ctl_lock, perform queue freeze/quiesce before map replacement, RCU-swap slot arrays, and free old arrays only after synchronize_rcu() and queue drain.
  • Risk: CPU hotplug could invalidate cpumasks during solve or commit. Mitigation: take topology snapshots and commit under cpus_read_lock(), use cpuhp callbacks only to mark epochs stale, and force rebuild before the next actuation step.
  • Risk: Affinity changes may block on internal IRQ locks or notifier chains. Mitigation: execute only on a dedicated housekeeping kthread_worker, cap the number of queue and IRQ operations per batch, and call cond_resched() between batches.
  • Risk: Managed MSI/MSI-X or interrupt remapping may make affinity non-deterministic and cause control-loop oscillation. Mitigation: classify such IRQs as hint-only or deterministic only after exact effective-mask readback, quarantine partial acceptances, and preserve a queue-only optimization path.
  • Risk: Prior-art overlap with irqbalance, static blk-mq mapping, or BPF telemetry may be asserted. Mitigation: narrow claims to the blk-mq-owned tag-indexed telemetry plus APIC-capability-aware IRQ lawfulness classification and epochal commit/rollback state machine, which those mechanisms do not provide.
