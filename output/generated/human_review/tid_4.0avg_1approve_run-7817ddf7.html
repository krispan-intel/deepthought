2. Technical Invention Disclosure

2.1 Problem Statement

Linux virtualization hosts still react to package-level power and thermal stress with coarse mechanisms that sit outside KVM, while KVM itself makes two expensive classes of decisions in isolation: TDP MMU mapping-size choices and vCPU halt-poll versus block behavior. The result is avoidable inefficiency and latency variance. Under package pressure, a host may keep polling too aggressively, wasting power and stealing cycles from runnable work, while simultaneously preserving or re-creating large mappings whose later split or zap costs amplify remote TLB shootdowns during reclaim, dirty logging, or mmu_notifier activity. Operators need a kernel-resident mechanism that converts package-scoped x86 RAPL and corroborating host pressure into bounded KVM behavior changes without doing MSR reads in hot paths, without taking new fast-path locks, without attributing package energy to individual VMs, and without weakening any existing KVM correctness gate for hugepages, dirty logging, nested virtualization, or wakeup latency.

2.2 Prior-Art Gap

Prior art covers the pieces in isolation: Intel RAPL telemetry and powercap; VMX enablement and architecture control bits such as CR4.VMXE; KVM halt-poll tunables and adaptive polling; TDP MMU large-page map, split, and promotion logic; generic power-aware scheduling; and cluster-level power allocation for virtualized workloads. The topological void is the missing kernel contract between package-scoped microarchitectural power pressure and KVM's internal decision points. Existing KVM heuristics do not publish an expiring, generation-tagged advisory word from a package broker into both memslot and vCPU state; they do not require pre-reserved invalidation budget before deferred SPTE churn; and they do not define mechanically checkable dominance rules that neutralize hints whenever dirty logging, page tracking, mmu_notifier invalidation, nested guest execution, NX hugepage workaround state, or reclaim and compaction pressure are present. Academic work on RISC-V or architecture-level virtualization control explains how virtualization is enabled, while HPC and virtual-lab power allocation work operates at scheduler or placement layers; neither teaches a bounded in-kernel KVM hint ABI driven by x86 RAPL epochs.

2.3 Proposed Invention

Introduce a package-pressure broker for KVM/x86. Each physical package owns one broker worker and one representative online CPU selected and refreshed by CPU hotplug callbacks. Every 50 ms the worker samples package energy and time, computes wrap-safe delta power as `delta = (curr - prev) & ((1ULL << rapl_width) - 1)`, converts it with the package energy unit, and derives three normalized inputs: `rapl_score` from proximity to package power limit or sustained high package power, `thermal_score` from `arch_scale_thermal_pressure(rep_cpu)`, and `host_score` from PSI plus vmstat reclaim and compaction deltas. The controller computes `pressure = 0.50*rapl_score + 0.30*thermal_score + 0.20*host_score`, enters `PRESSURE_CONSTRAINED` after three consecutive samples at or above 70, enters `PRESSURE_SEVERE` after three consecutive samples at or above 85, exits severe only after eight consecutive samples below 75, exits constrained only after eight consecutive samples below 55, and enforces a 500 ms minimum dwell. Confidence is a 0..127 score: base 32 when RAPL is continuous, +24 if thermal corroborates within 15 points, +24 if PSI or runnable pressure corroborates within 15 points, +16 if vmstat reclaim or compaction pressure agrees with the current state, and +31 reserved for consecutive-sample stability. The broker publishes two aligned 64-bit words. The memslot word layout is: bits 0..1 `hp_mode` where 0=neutral, 1=prefer_keep_2m, 2=prefer_hold_4k, 3=reserved; bits 2..8 `confidence`; bits 9..24 `expiry_epoch16`; bits 25..56 `slot_gen_low32` copied from the current memslot generation; bits 57..63 flags where bit 57 means deferred invalidation budget required and bits 58..60 encode source corroboration. The vCPU word layout is: bits 0..1 `idle_mode` where 0=neutral, 1=cap_poll_short, 2=cap_poll_medium, 3=skip_poll; bits 2..8 `confidence`; bits 9..24 `expiry_epoch16`; bits 25..40 `idle_gen16`; bits 41..48 `poll_cap_quanta` in 2 us units; bits 49..55 flags; bits 56..63 `pkg_id`. Epoch comparison is wrap-safe using signed 16-bit subtraction, so a hint is valid only if `(s16)(expiry_epoch16 - now_epoch16) > 0`. Publication uses `smp_store_release()` and consumption uses `smp_load_acquire()`, with the words naturally 8-byte aligned so x86_64 does not tear them. Dominance rules are explicit. A memslot hint is forced to neutral if `kvm_slot_dirty_track_enabled(slot)` is true, if page tracking for access or write-protect is active on that slot, if `mmu_invalidate_seq` is odd or changes during evaluation, if the slot generation no longer matches, if NX hugepage workaround state is active for the VM, if any vCPU is in nested guest mode, if recent vmstat deltas show direct reclaim or compaction activity, or if invalidation budget reservation fails. A vCPU idle hint is forced to neutral if the hint is expired, the vCPU has pending requests or events, `need_resched()` is set, the vCPU migrated to a different package than `pkg_id`, or confidence is below 32. Hugepage hints are consumed only in the existing target-level choice where both 2M and 4K are already legal. The fault path never initiates split or promotion because of the hint; it merely chooses among legal sizes. Any later split or promotion happens only in deferred worker context after an invalidation-token reservation succeeds. The invalidation budget is a per-VM atomic token bucket refilled every 500 ms to `min(256, 8 * online_vcpus)`. The broker charges one token per candidate 2M split, two tokens per promotion batch that requires zapping 4K SPTEs, and additional tokens proportional to the expected remote TLB flush fanout. Reservation uses atomic try-subtract before queuing deferred work; if the worker aborts before the first SPTE write it refunds the full reservation, otherwise it refunds only unused remainder after actual operation count. If reservation fails, hints remain advisory only and no invalidating work is queued.

2.4 Architecture

The architecture has six components. First, a per-package sampler worker runs in sleepable context only, so any powercap, thermal, or RAPL driver path that may sleep is isolated from KVM locks and fast paths; direct MSR fallback uses `rdmsrl_safe_on_cpu()` on the package representative CPU and is never called from fault, vCPU run, or halt-poll code. Second, CPU hotplug integration maintains a valid representative CPU per package; on offline notification the broker immediately marks confidence low, repicks another online CPU in the package, and publishes neutral hints until two consecutive good samples arrive. Third, the controller maintains `epoch16`, pressure state, dwell timers, and confidence. Fourth, publication writes per-memslot and per-vCPU words. Memslot words live in `kvm_memslot->arch.pkg_hp_hint`; vCPU words live in `vcpu->arch.pkg_idle_hint`; both are `__aligned(8)` and updated with release semantics after all derived fields are final. Fifth, consumption is narrow. The TDP MMU consult occurs only in the map or preserve path that already chooses a target level after all existing KVM legality checks have passed; if `hp_mode=prefer_hold_4k`, the code simply chooses 4K when 2M and 4K are both already allowed, and if `hp_mode=prefer_keep_2m`, it simply avoids selecting a smaller mapping when preserving an existing 2M mapping is already legal. No immediate split, promotion, zap, or MMU notifier action occurs in that fault path. Deferred split and promotion workers operate under the existing KVM SRCU domain plus `kvm->mmu_lock`, and before touching SPTEs they recheck that `mmu_invalidate_seq` is even and unchanged, the slot generation still matches, and the root is still active; they never cache a root pointer across unlocks. The lock order is sampling with no KVM MMU locks, then SRCU for VM enumeration, then `kvm->mmu_lock` for SPTE mutation; MMU work never calls back into RAPL, PSI, thermal, or hotplug paths. Sixth, vCPU idle consumption occurs at exactly two points: when entering the halt-poll loop to derive an effective poll cap, and immediately before block to decide whether optional polling should be skipped. `cap_poll_short` and `cap_poll_medium` can only reduce the existing poll ceiling, never increase it. `skip_poll` only bypasses the optional poll for that one idle entry and is ignored if a request, IRQ, timer event, or reschedule is pending. Because the word is re-read on each idle entry and checked again before schedule, a stale hint cannot keep a vCPU polling after host load changes and cannot force a block when wakeup conditions make polling preferable.

2.5 Implementation Plan

Phase 1 implements the broker skeleton: per-package state, CPU hotplug registration, representative CPU tracking, wrap-safe RAPL sampling, epoch accounting, and tracepoints for raw inputs, state transitions, and confidence. Phase 2 adds publication ABI fields in `kvm_memslot->arch` and `vcpu->arch`, with release/acquire semantics, expiry handling, and neutral-only shadow mode so real KVM decisions are logged but not changed. Phase 3 wires in the exact consume points: the TDP MMU target-level selector where both 2M and 4K are already legal, and the halt-poll poll-loop plus pre-block checks. Phase 4 adds the dominance predicates: dirty logging, page tracking, `mmu_invalidate_seq`, nested guest mode, NX hugepage workaround state, memslot generation churn, reclaim, and compaction. Phase 5 adds the deferred invalidation-token bucket and queues split or promotion work only when reservation succeeds, with charge and refund accounting plus root and generation rechecks before SPTE changes. Phase 6 tunes thresholds on multi-socket hosts and exposes read-only trace and debugfs observability for epochs, pressure state, confidence, token usage, and neutralization reasons. The feature is behind a KVM capability and package-level sysfs or module parameters so rollout can begin disabled or in observe-only mode.

2.6 Validation Plan

Validation covers correctness, stability, and value. Correctness validation runs KVM selftests and kvm-unit-tests with the feature disabled, observe-only, and active; exercises dirty logging, live migration pre-copy, page tracking, mmu_notifier invalidations from host memory pressure, nested virtualization enabled and actively used, NX hugepage workaround paths, CPU hotplug, and package representative CPU failover; and verifies that every such case forces neutral hints and preserves current behavior. Stability validation drives synthetic bursty workloads that alternately trigger RAPL caps, thermal pressure, PSI spikes, and reclaim or compaction, then checks that state transitions respect the 50 ms cadence, 500 ms dwell, deadband, and no-oscillation thresholds. Performance validation uses multi-socket Xeon hosts with noisy-neighbor mixes, memcached, OLTP, and VM consolidation workloads while collecting p95 and p99 guest latency, VM exit-to-entry time, halt-poll residency, guest wakeup latency, 2M mapping ratio, split and promotion counts, remote TLB shootdowns per second, package power, thermal throttling residency, and token-bucket exhaustion events. Success criteria are: no correctness regressions; no new fast-path locks or MSR reads; lower package power during pressured idle phases; fewer invalidation storms under reclaim or dirty logging; and bounded tail latency relative to baseline.

4. Risks & Mitigations

  • Risk: pressure-state oscillation under bursty noisy-neighbor workloads. Mitigation: fixed 50 ms cadence, three-sample entry, eight-sample exit, 500 ms minimum dwell, and confidence floor before non-neutral publication.
  • Risk: unsafe interaction with dirty logging, page tracking, mmu_notifier invalidation, or nested virtualization. Mitigation: explicit dominance predicates that force neutral before publication and again before deferred MMU work.
  • Risk: representative CPU hotplug or RAPL read failure produces stale or misleading pressure. Mitigation: CPU hotplug callbacks repick the package CPU, confidence drops immediately, and neutral hints are published until two consecutive valid samples complete.
  • Risk: invalidation storms from aggressive deferred split or promotion. Mitigation: per-VM token bucket with atomic reservation before queueing, exact charge and refund accounting, and no action when budget is exhausted.
  • Risk: wakeup latency regression from idle hints. Mitigation: idle hints only reduce or skip optional polling for one idle entry, are re-read at each consume point, and are ignored whenever requests, events, timers, or reschedule conditions are pending.
  • Risk: prior-art challenge that this is just combining known knobs. Mitigation: emphasize the precise single-word ABI, release/acquire semantics, generation plus expiry contract, mechanically checkable dominance rules, and pre-reserved invalidation budget.
