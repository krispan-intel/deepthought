2. Technical Invention Disclosure

2.1 Problem Statement

On x86 systems with Intel AMX, a task that carries non-init tile state has materially heavier user xstate than scalar-only tasks. Repeated cross-CPU or cross-LLC migration of such tasks can increase non-init XSAVE/XRSTOR traffic, disrupt tile-local reuse, and amplify tail latency in mixed AMX plus latency-sensitive workloads. Linux currently lacks a precise, low-overhead, architecture-defined way to tell the scheduler that a task's saved user fpstate is AMX state-carrying and to use that fact only as a bounded placement hint. Any viable design must avoid XSAVE-buffer parsing in scheduler hot paths, avoid kernel_fpu_begin()/kernel_fpu_end() in wakeup or pick paths, preserve rq locking and scheduler-class invariants, survive hotplug/cpuset/topology churn, remain correct across exec/fork/exit, sigreturn, ptrace xstate mutation, XFD feature disable/enable, and fpstate reset/discard, and account for security leakage from deterministic AMX-aware placement in shared environments.

2.2 Prior-Art Gap

Known scheduler work on AVX-512 'heavy task' handling, cache-hot packing, load-balance heuristics, NUMA locality, and accelerator-aware placement does not define an AMX-specific correctness contract around saved user xstate. Existing ideas generally classify tasks heuristically from instruction mix, frequency side effects, or recent CPU use; they do not define the architectural predicate for AMX non-init state, do not resolve the XTILECFG versus XTILEDATA corner cases, and do not specify which kernel path is authoritative when fpstate is copied, discarded, reset, restored from a signal frame, or rewritten by ptrace. They likewise do not define a scheduler-safe producer/consumer split in which x86 FPU/xstate code owns hint production while the scheduler consumes only immutable metadata and rq-local scalars. Nor do they publish per-root_domain placement masks with explicit topology/cpuset/hotplug/root-domain generation validation, TTL expiry, fallback semantics, SMT/core-sched-aware caps, or tenant-leak mitigation. The gap is therefore a kernel-credible AMX xstate contract plus bounded tie-break usage, not merely another packing heuristic.

2.3 Proposed Invention

Introduce an AMX state-carrying hint pipeline with exact producer rules, rq-local runnable accounting, and an asynchronously rebuilt root-domain hint plane. 1. Architectural predicate and invariant. - Add a scalar task field, e.g. thread.amx_state_carrying, and define it precisely: it is 1 iff the task's authoritative saved user fpstate, after normal xstate canonicalization and init-state elision handling, would restore XTILECFG or XTILEDATA in non-init architectural state on the next user return. - The predicate is based on xstate metadata that denotes non-init components, not on compacted-buffer presence. Compacted buffers may omit init components, so 'component present in buffer' is not sufficient. The authoritative test is whether the saved xstate marks XTILECFG non-init or XTILEDATA non-init after canonicalization. - XTILEDATA non-init alone implies state-carrying=true even if XTILECFG appears init in an intermediate image; that combination is treated as logically AMX state-carrying and the recompute path may normalize metadata so the cached bit never becomes stale-false. - The hint is not about transient hardware register contents and is never derived by scheduler code. 2. Authoritative producer update points. - Exec/new-mm path: when execve replaces the task image and x86 thread/FPU state is reinitialized, set amx_state_carrying=0. - Fork/clone path: after x86 FPU state copy for the child, recompute the child's bit from the copied saved fpstate; if the child starts from a clean/init fpstate representation, set 0. Parent remains unchanged. - Exit path: dequeue removes scheduler relevance; the field may be cleared before final free only for debug hygiene, not for functional correctness. - sigreturn/rt_sigreturn: after restoring user xstate from the signal frame, recompute the bit from the newly authoritative saved fpstate before returning to user mode. - ptrace regset write: on NT_X86_XSTATE or equivalent xstate write, recompute immediately from the written image; ptrace reads do not mutate the bit. - XFD AMX disable or any feature-mask contraction that invalidates/discards XTILE* user state: set 0 at the point the task's authoritative saved fpstate is reset or AMX components become architecturally unavailable. Re-enable alone does not set 1; only a later restore/save/write that creates non-init XTILE* state can do that. - FPU save/materialization boundary: when the outgoing task's live user AMX state is materialized into its saved fpstate during context-switch/FPU ownership handoff, recompute and publish the bit there. This is the only place where live register contents may indirectly affect the bit, and it occurs inside x86 FPU code, not scheduler code. - Any explicit fpstate reset, init, discard, or XRSTOR-to-init style path that canonicalizes XTILE* back to init clears the bit. - This makes x86 FPU/xstate code the sole producer and prevents stale-true/stale-false cases because every path that can create, copy, rewrite, or destroy authoritative saved XTILE* state also recomputes the scalar hint. 3. Exact runqueue accounting. - Add rq->amx_runnable_nr and a per-task latch p->sched_amx_on_rq. - Define membership narrowly: a task contributes to rq->amx_runnable_nr only while it is a FAIR runnable entity queued on that rq and p->thread.amx_state_carrying==1. - Under rq->lock, enqueue performs a 0->1 transition on p->sched_amx_on_rq and increments rq->amx_runnable_nr once; dequeue performs a 1->0 transition and decrements once. No transition, no counter change. - Migration: decrement on source detach under source rq->lock, increment on destination attach under destination rq->lock, following existing migration locking. The counter is rq-local only and need not be globally atomic across both locks. - Sleep/block, cgroup bandwidth throttle dequeue, class change away from FAIR, exec/exit dequeue, and affinity-driven dequeue all remove membership and decrement exactly once. - Unthrottle or wake re-enqueue increments exactly once. - A debug invariant uses WARN_ON_ONCE/SCHED_WARN_ON semantics to flag any attempt to decrement when p->sched_amx_on_rq==0 or rq->amx_runnable_nr==0; the design goal is no silent underflow and no double counting. 4. Asynchronous per-root_domain AMX hint plane. - Maintain an immutable RCU object per root_domain, e.g. struct sched_amx_plane { cpumask favored_mask; cpumask cold_mask; u64 topo_gen; u64 cpuset_gen; u64 hotplug_gen; u64 rd_gen; unsigned long expires_jiffies; u16 amx_core_cap; u32 tie_salt; }. - Generation meanings are explicit: topo_gen bumps when sched topology inputs change; cpuset_gen bumps when effective cpuset partitioning changes CPUs visible to a root_domain; hotplug_gen bumps on CPU online/offline transitions; rd_gen bumps whenever sched domains/root_domains are rebuilt or a root_domain is replaced. - Counters are stored as atomic64_t or equivalent monotonic scalars, bumped under the owning subsystem's existing serialization, read with READ_ONCE/atomic64_read, and snapshotted by the plane builder under cpus_read_lock so the published object is self-consistent. - A deferred worker in process context rebuilds the plane on meaningful AMX runnable deltas, on topology/cpuset/hotplug/root-domain generation changes, or on TTL expiry. No plane build occurs inside rq->lock or wakeup hot paths. - favored_mask contains CPUs in the smallest LLC-local set that already hosts most AMX runnable pressure but is still below cap; cold_mask contains CPUs in the same root_domain that currently have zero AMX runnable pressure and are available as expansion targets. When SMT or core scheduling is enabled, mask computation is done at whole-core granularity so sibling threads of a physical core share the same cap decision. - TTL is short, e.g. 4 ms or another admin-tunable small interval. If the plane is absent, expired, or any stored generation mismatches current generations, hot paths ignore both masks and fall back immediately to baseline scheduling behavior. 5. Bounded scheduler integration points and comparator. - The mechanism applies only to CFS placement decisions, not to RT/DL semantics and not to pick_next_task_fair vruntime ordering. - Integration points are explicit: (a) select_task_rq_fair() during wakeup placement after baseline allowed-CPU filtering and after affinity/cpuset/NUMA/asym/uclamp/thermal checks, (b) wake_affine()-style equal-rank decisions when baseline logic leaves prev_cpu and target_cpu tied, (c) select_idle_sibling()/select_idle_cpu() when several idle choices remain baseline-equal, and (d) find_idlest_group()/find_idlest_cpu() and load-balance candidate ranking only when baseline load metrics tie. - The AMX comparator is applied only among already-eligible CPUs that baseline code considers otherwise equal. A representative deterministic key for AMX tasks is: key(cpu) = (saturated(cpu), not_in_favored(cpu), spread_phase_and_not_in_cold(cpu), rq->amx_runnable_nr, cpu_index_mixed_with_tie_salt). Any earlier baseline difference short-circuits the AMX comparator. - saturated(cpu)=1 when that CPU or its physical core has reached amx_core_cap. Default cap is 1 AMX state-carrying task per physical core when SMT or core scheduling is enabled, otherwise 1 per logical CPU unless an admin explicitly raises it. - If all candidate CPUs are saturated, if the plane is invalid, or if the task is not AMX state-carrying, the scheduler uses the unmodified baseline tie-break. - Because the comparator never changes eligibility and never changes vruntime accounting, the mechanism cannot introduce starvation beyond baseline CFS behavior. Under sustained AMX load, saturation causes the bias to collapse toward baseline/spread behavior rather than intensifying packing, so fairness regressions are bounded. 6. Security model and anti-pathology rules. - Threat model: a co-resident observer could infer that another task or cgroup is AMX state-carrying from CPU-choice patterns, LLC contention, or wakeup timing. This is a placement-observable leak, not a direct data-exposure bug, but it matters for shared-hosting profiles. - Mitigations: default-off global enable behind a static key/sysctl; optional per-cgroup opt-in so only trusted workloads participate; deployment guidance to disable the feature on mutually untrusted shared hosts; optional dithering using tie_salt when several CPUs have identical comparator keys; and documentation of observables and trust assumptions. - Latency guardrails: if rd-wide AMX pressure crosses a threshold, if favored CPUs exceed cap occupancy, or if measured wakeup latency for protected workloads regresses past an admin threshold, the builder expands favored_mask, prefers cold_mask, or suppresses AMX bias for that root_domain until the next rebuild window. - SMT/core-sched interaction: when core scheduling is enabled, the mechanism operates at whole-core granularity and never uses AMX bias to choose a sibling thread in a way that violates existing core-sched cookie constraints; when SMT is enabled without trusted tenancy, policy should default to off.

2.4 Architecture

Control plane and data plane are deliberately separated.

ASCII overview:

[exec / fork-copy / sigreturn / ptrace xstate SET / XFD disable / fpstate reset]
                                |
                                v
                 [x86 FPU/xstate recompute amx_state_carrying]
                                |
                                v
                [enqueue/dequeue FAIR task under rq->lock]
                                |
              +-----------------+-----------------+
              |                                   |
              v                                   v
   [p->sched_amx_on_rq 0<->1 latch]      [rq->amx_runnable_nr +/- 1]
              |                                   |
              +-----------------+-----------------+
                                |
                                v
                 [deferred root_domain plane worker]
            snapshots: rq AMX counts + topo_gen + cpuset_gen
                       + hotplug_gen + rd_gen + SMT/core-sched mode
                                |
                                v
            [build favored_mask, cold_mask, amx_core_cap, tie_salt, TTL]
                                |
                                v
                     [RCU publish rd->sched_amx_plane]
                                |
                                v
  CFS equal-rank tie sites read only: {task bit, rq scalar, plane pointer if valid}
                                |
                                v
   select_task_rq_fair / wake_affine tie / select_idle_sibling / find_idlest_cpu

Data structures:
- Per task: thread.amx_state_carrying, sched_amx_on_rq.
- Per rq: amx_runnable_nr.
- Per root_domain: RCU pointer to sched_amx_plane.
- Global or subsystem-scoped monotonic generations: topology, cpuset, hotplug, root-domain rebuild.

Hot-path read contract:
1. Read p->thread.amx_state_carrying.
2. Read candidate rq->amx_runnable_nr.
3. RCU-dereference rd->sched_amx_plane, validate TTL and generations.
4. If any validation fails, ignore plane and use baseline tie-break.

This keeps all cpumask construction, generation synthesis, and pressure aggregation out of atomic scheduler paths.

2.5 Implementation Plan

Phase 1 - producer correctness in x86 FPU/xstate paths. - Add the per-task amx_state_carrying field and a small helper in x86 FPU/xstate code that recomputes it from authoritative saved fpstate metadata after canonicalization. - Wire that helper into exec reset, fork/clone fpstate copy, sigreturn restore, ptrace xstate write, XFD-disable/reset/discard paths, and the outgoing-user-state save/materialization boundary. - Add selftests or trace validation for stale-true/stale-false edge cases, especially XTILEDATA-non-init with XTILECFG appearing init in intermediate representations. Phase 2 - rq-local accounting and scheduler plumbing. - Add rq->amx_runnable_nr and p->sched_amx_on_rq. - Update CFS enqueue/dequeue/migration/throttle/class-change transitions so the counter reflects FAIR runnable membership only under rq->lock. - Add debug assertions for underflow and duplicate membership. Phase 3 - root_domain RCU plane. - Introduce sched_amx_plane, generation counters, and a deferred rebuild worker. - Hook rebuild triggers to sched-domain/root-domain rebuild, cpuset effective-mask changes, CPU hotplug, and meaningful AMX pressure delta/TTL expiry. - Compute favored_mask and cold_mask at LLC/core granularity with SMT/core-sched-aware caps. Phase 4 - bounded CFS tie-break integration. - Insert AMX comparator hooks only in equal-rank decision sites in kernel/sched/fair.c: wakeup CPU selection, idle sibling selection, and idlest-candidate ties. - Ensure baseline eligibility masks and ranking semantics run first and remain authoritative. - Add sysctl/static-key gating and optional cgroup policy surface for opt-in. Phase 5 - observability and rollout. - Add tracepoints and counters for hint transitions, plane publication, plane invalidation, tie-break invocations, same-LLC versus cross-LLC AMX migrations, and wakeup-latency impact. - Roll out in shadow mode first: compute hint and plane, log would-have-chosen CPU, but do not alter placement until measurements confirm no fairness or latency regressions.

2.6 Validation Plan

Hypotheses. 1. For AMX-heavy tasks, the mechanism reduces cross-LLC migrations of AMX state-carrying tasks versus baseline CFS. 2. For mixed workloads, non-init XTILEDATA save/restore events per unit of useful AMX work decline because fewer migrations force state movement. 3. For latency-sensitive non-AMX services co-located with AMX jobs, p95/p99 wakeup latency does not regress materially because bias saturates and falls back to baseline under pressure. 4. Under SMT or core scheduling, whole-core caps avoid sibling over-concentration. Instrumentation. - Add new tracepoints such as sched_amx_hint_transition, sched_amx_plane_publish, sched_amx_plane_reject(reason=expired|gen_mismatch|disabled), and sched_amx_tiebreak(task,candidate_cpu,chosen_cpu,reason). - Count same-LLC and cross-LLC migrations for AMX state-carrying tasks. - Count outgoing FPU save and restore events where saved user xstate is AMX non-init. - Track rq->amx_runnable_nr histograms, favored-mask occupancy, cold-mask usage, and latency-guard suppression events. Workloads. - Xeon AMX-capable platforms such as Sapphire Rapids or Emerald Rapids. - oneDNN or similar BF16/INT8 inference/training microbatches that repeatedly carry tile state. - Mixed deployments with latency-sensitive services such as nginx/envoy/memcached or RPC microservices pinned to overlapping root_domains/cpusets. - Stress cases: rapid cpuset repartitioning, CPU hotplug churn, signal-heavy workloads using sigreturn, ptrace xstate mutation, and XFD enable/disable transitions. Success criteria. - Reduced cross-LLC AMX migrations and reduced non-init XTILEDATA save/restore counts relative to baseline. - Stable or improved AMX throughput. - No statistically meaningful starvation signal, no fairness drift in CFS vruntime distributions, and no sustained p99 latency regression beyond an admin-set bound. - Zero observed stale-bit mismatches in trace validation: whenever authoritative saved fpstate is init, amx_state_carrying must be 0; whenever XTILECFG or XTILEDATA is non-init, it must be 1. Negative testing. - Verify that absent, expired, or generation-mismatched planes produce exact baseline CPU placement. - Verify that cpuset, affinity, NUMA, asymmetry, uclamp, thermal pressure, RT, and DL decisions dominate and are never overridden by AMX logic. - Verify that shared-host policy disables the feature and that tie dithering changes only exact-equality tie outcomes.

4. Risks & Mitigations

  • Risk: stale task hints across unusual xstate mutations. Mitigation: make x86 FPU/xstate code the sole producer, enumerate every authoritative update path, and trace-validate hint transitions against saved fpstate snapshots.
  • Risk: runqueue counter corruption under migration, throttle, or class changes. Mitigation: use a per-task sched_amx_on_rq latch, update only under rq->lock, and add underflow/double-entry debug assertions.
  • Risk: over-packing AMX tasks increases latency for co-located services. Mitigation: cap AMX occupancy per logical CPU or per physical core, expand favored masks when pressure rises, and suppress bias when latency guardrails trip.
  • Risk: deterministic CPU choice leaks which task or cgroup is AMX state-carrying. Mitigation: default-off global policy, optional per-cgroup opt-in, shared-host disable guidance, and optional dithering only within exact-equality ties.
  • Risk: stale favored/cold masks after topology, cpuset, or hotplug changes. Mitigation: carry monotonic topology/cpuset/hotplug/root-domain generations in each RCU plane and ignore any plane that is expired or generation-mismatched.
  • Risk: interference with SMT or core scheduling. Mitigation: whole-core saturation accounting, no override of core-sched cookie rules, and policy-default disable for untrusted SMT-sharing configurations.
