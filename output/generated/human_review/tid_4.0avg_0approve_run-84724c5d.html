2. Technical Invention Disclosure

2.1 Problem Statement

High-rate virtio-net receive paths repeatedly parse descriptor chains, resolve guest virtual or userspace virtual addresses to guest physical and host physical pages, and touch second-level translation structures for payload memory that is often scattered at 4 KiB granularity. On x86 virtualization stacks this creates extra EPT page walks, DTLB pressure, and page-walk traffic under multi-queue RX, especially for small and medium packets where metadata and translation overhead dominate. Existing virtio-net contracts let guests place RX header and payload bytes anywhere in the writable scatter-gather list, so a host backend cannot safely assume that payload pages are bounded, aligned, or reusable, cannot deterministically tell whether tail cleanup would stay within one writable segment, and cannot switch an optimization off at runtime without risking partial packet handling, stale pins, migration breakage, or IOMMU incoherency.

2.2 Prior-Art Gap

Known techniques optimize adjacent parts of the stack but do not define the claimed combination. Host THP, hugetlbfs, and KVM's opportunistic huge EPT mappings act on memory layout globally and do not provide a device ABI that constrains only RX payload bytes, encodes explicit header exclusion, carries advisory alignment and leaf-size hints, and supports runtime enable or disable with deterministic drain semantics. vhost-net zerocopy, generic GUP or FOLL_PIN reuse, and other page pinning schemes do not require the same co-existing elements of versioned virtio arena eligibility, per-packet all-or-nothing fallback, page identity plus compound metadata revalidation, bounded TTL and page budgets, seqcount plus SRCU invalidation generations, and a rule that no DMA or IOTLB mapping cache is retained. AF_XDP UMEM, netmap, and pre-registered hugepage regions likewise do not define this virtio-specific control plane and invalidation model for guest-driven RX payload placement.

2.3 Proposed Invention

The invention introduces a versioned virtio-net RX payload arena contract and a host-local acceleration policy that are useful only together. The device exposes feature bits such as VIRTIO_NET_F_RX_PAYLOAD_ARENA, an optional strict mode, and an optional query mode. Device configuration provides arena_version, arena_gpa_base, arena_size, arena_alignment_log2, host_leaf_hint_log2, header_exclusion_mode, header_exclusion_len, max_payload_sg_segments, max_payload_len, eligibility_flags, sanitization_mode, and arena_contract_generation. host_leaf_hint_log2 is explicitly advisory: it communicates a preferred leaf-size alignment target, such as 2 MiB, but never guarantees that EPT or any other second-level mapping will actually use that leaf size. A control virtqueue carries ENABLE, DISABLE, and QUERY commands on a per-queue basis so the host can begin or stop accepting arena-qualified packets without renegotiating the device feature or modifying the guest memory map. A packet is fast-path eligible only if all payload-bearing writable descriptors satisfy the negotiated contract: header bytes are entirely outside the payload arena according to header_exclusion_mode and header_exclusion_len; each payload descriptor lies wholly within the single configured arena range; descriptors are writable and non-overlapping; payload length and scatter-gather count are within negotiated limits; sanitization, if required, can be performed entirely inside the final writable payload segment; and no control or invalidation generation mismatch is observed. The host may then consult a short-term page-pin reuse cache keyed by an mm lifetime cookie, userspace page-aligned virtual address, head PFN, subpage index within the compound head, compound order, compound page count, writable attribute, and snapshot generations for contract and memory table state. Reuse is allowed only when current translation resolves back to the same page identity and compound metadata tuple. The host-side reader path runs under SRCU, samples a seqcount-protected policy snapshot, validates the current queue state and generation tuple, revalidates each cached entry's page identity, and proceeds only if the seqcount is unchanged after validation. Writers increment one or more generations and purge or disable reuse on mmu_notifier invalidate_range_start, mmu_notifier release, vhost_set_memory or memory-table swaps, queue reset, device reset, migration start, reclaim or OOM drain, and any vhost IOTLB or DMA translation update. In the base embodiment, the fast path is disabled whenever VIRTIO_F_ACCESS_PLATFORM, vIOMMU, or equivalent IOTLB-mediated translation is active, which ensures the design retains no DMA mapping cache and no IOTLB mapping cache. Tail sanitization, when enabled, zeros only bytes from the reported payload length up to the end of the same final writable payload segment and never crosses into another descriptor or into header space. Any failure or ambiguity causes deterministic all-or-nothing fallback for the entire packet to the baseline virtio-net RX path.

The architecture deliberately separates what the virtio contract states from what x86 EPT may or may not do. The guest-visible contract offers a bounded, aligned payload arena and explicit header exclusion. The host interprets alignment and host_leaf_hint_log2 only as an optimization hint that may improve the chance of favorable second-level leaf alignment when host memory layout permits, but every packet remains correct on 4 KiB EPT leaves or after THP split or merge because the fast path is admission-controlled and revalidated at use time.

2.5 Implementation Plan

Phase 1, define the virtio contract: add the RX payload arena feature bits, config fields, and control-virtqueue command formats in the virtio-net device model and guest driver; implement header_exclusion_mode and header_exclusion_len so the header is precisely delineated from payload, and expose QUERY status with reason codes for disabled states such as access-platform active, memory-pressure drain, migration, or generation mismatch. Phase 2, implement the host admission path in the virtio or vhost receive backend: parse RX descriptor chains, compute exact payload extents, reject packets whose payload crosses arena boundaries or violates writable, overlap, or SG-count rules, and enforce all-or-nothing per-packet fallback. At this phase, use normal pin acquisition with no reuse to validate the contract independently from the optimization. Phase 3, add the bounded reuse cache: store entries per queue and per VM with runtime-configurable ceilings such as TTL no greater than one NAPI poll epoch or 200 microseconds, packet reuse budget of 32 packets per entry, cumulative byte budget of 2 MiB per entry, per-queue page cap, and per-VM page cap. Each cache entry retains mm lifetime ownership via mmgrab and mmdrop, registers or references mmu_notifier state, and is keyed by mm cookie, uaddr page, PFN-based page identity, and compound metadata rather than by mm plus address alone. Phase 4, add the concurrency and invalidation model: protect reader snapshots with SRCU and seqcount; increment contract_gen on ENABLE, DISABLE, queue reset, and arena contract changes; increment memtable_gen on vhost_set_memory and other memory-table swaps or moves; increment mmu_gen on mmu_notifier invalidate_range_start and release; increment dma_gen on any vhost IOTLB or DMA translation update; increment pressure_gen on reclaim or OOM drain. Any mismatch purges or invalidates the relevant cache entries and forces the packet to baseline handling. Phase 5, implement same-segment tail sanitization and strict mode: after writing a packet of payload length L, identify the final writable payload segment containing byte L minus 1, and zero only from the end of valid payload to the end of that same segment. If sanitization would need to cross a segment boundary, mark the packet ineligible. Strict mode may drop such packets while non-strict mode routes them to the baseline path. In the base embodiment, fast-path admission is disabled entirely when VIRTIO_F_ACCESS_PLATFORM, vIOMMU, or equivalent IOTLB-managed translation is active; any alternate embodiment must still retain no DMA or IOTLB mapping cache and must invalidate fully on every relevant translation update.

2.6 Validation Plan

Correctness validation includes descriptor-fuzz and fault-injection tests that exercise every ineligible condition: payload outside arena, header overlap, non-writable segment, overlap, excessive SG count, sanitization crossing a segment boundary, expired TTL, exhausted page budget, seqcount mismatch, contract generation mismatch, mem-table swap, mmu_notifier invalidate_range_start, mmu_notifier release on mm teardown, THP split or merge changing compound metadata, queue reset, device reset, migration entry, reclaim drain, and VIRTIO_F_ACCESS_PLATFORM enablement. Every test must assert deterministic per-packet drain to baseline with no partial use of cached pins. Sanitization validation uses crafted scatter-gather layouts to prove that only bytes after the reported payload length and before the end of the same final writable segment are zeroed, that no byte in another descriptor is touched, and that packets requiring cross-segment zeroing are treated as ineligible. Separate tests cover mergeable buffers, checksum offload metadata, and zero-length or short payload cases. Concurrency validation uses stress workloads that race RX against memory invalidation sources: remap, munmap, mprotect, fork, exec, mm exit, live migration transitions, vhost_set_memory updates, and synthetic vhost IOTLB invalidations. Instrumentation must verify that generation changes force cache purge and that reader-side seqcount retries or fallback prevent use-after-free or stale-page reuse. Performance validation compares baseline virtio-net RX against the new mode on x86 hosts under both 4 KiB and hugepage-friendly memory layouts. Metrics include packets per second, host cycles per packet, EPT walk related PMU events, DTLB miss rate, LLC miss rate, softirq utilization, and CPU cost per queue. Additional runs confirm that when hints are absent or EPT remains at 4 KiB granularity, correctness is unchanged and only the expected optimization opportunity is lost. A final operational validation verifies that enabling migration, reclaim pressure, or access-platform mode disables the fast path cleanly and immediately.

4. Risks & Mitigations

  • Risk: overlap with known zerocopy or page-pinning prior art could make the idea appear to be only a cache of pinned pages. Mitigation: claim the mandatory combination of virtio arena ABI, explicit header exclusion, control-virtqueue runtime gating, bounded reuse keyed by page identity plus compound metadata, seqcount plus SRCU generation invalidation, same-segment sanitization, and deterministic all-or-nothing fallback.
  • Risk: long-lived FOLL_PIN reuse can interfere with migration, CMA, reclaim, or memory pressure. Mitigation: restrict reuse to short TTL and packet or byte budgets, enforce per-queue and per-VM page caps, purge on reclaim or OOM signals, and increment pressure generations that immediately disable or drain the fast path.
  • Risk: stale reuse after remap, THP split, or mm teardown could cause aliasing or use-after-free. Mitigation: include PFN-backed page identity and compound metadata in the key, revalidate the current translation before each reuse, register mmu_notifier callbacks, hold mm lifetime with mmgrab and mmdrop, and purge on invalidate_range_start and release.
  • Risk: DMA or IOTLB state changes under VIRTIO_F_ACCESS_PLATFORM or vIOMMU can invalidate assumptions. Mitigation: in the primary embodiment, disable the accelerated path whenever access-platform or IOTLB-managed DMA is active; never retain DMA or IOTLB mapping caches; if an alternate embodiment is pursued, require full invalidation on every translation update.
  • Risk: tail sanitization could corrupt adjacent guest buffers or violate header and payload separation. Mitigation: sanitize only within the same final writable payload segment after the reported payload length, treat any cross-segment sanitization requirement as ineligible, and use validation cases that prove no other descriptor is modified.
  • Risk: false negatives in eligibility checking may reduce performance benefit. Mitigation: bias toward conservative fallback, keep the baseline virtio-net path untouched, and expose QUERY reason codes so operators can distinguish disabled-for-safety states from genuine performance regressions.
  • Risk: x86-specific optimization language could overclaim guaranteed EPT behavior. Mitigation: describe alignment and leaf-size values only as host hints that may improve the chance of favorable second-level mappings, while stating explicitly that correctness and feature semantics do not depend on huge EPT leaves actually being present.
