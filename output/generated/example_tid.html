3. Technical Invention Disclosure

3.1 Problem Statement

Linux page-cache placement is largely filesystem-agnostic. On x86 systems with CXL Type-3 memory behind HDM decoders, generic demotion can move the wrong folios: hot metadata is displaced, journal-pinned dirty folios are churned, and cold contiguous extents remain in DRAM. The result is refault amplification, xa_lock pressure, and wasted DRAM in file-serving workloads.

3.2 Prior-Art Gap (Topological Void)

Existing NUMA balancing, DAMON-guided reclaim, and generic memory tiering reason about page heat, not filesystem extent boundaries, writeback epochs, or journal pinning. DAX bypasses the page cache entirely and does not solve buffered I/O placement. No mainstream mechanism translates filesystem allocation knowledge into CXL-aware folio residency decisions.

3.3 Proposed Invention

Introduce a cxl_folio_lease abstraction: a sidecar metadata structure indexed by (mapping, index) that stores lease state, lease epoch, extent heat bucket, and poison/quarantine bits. Lease states are HOT_DRAM, WARM_CXL, DIRTY_PINNED, REFAULT_PROMOTE, and POISON_QUARANTINE. The bridge consumes signals from workingset refault distance, filesystem extent maps, writeback completion, and CXL node latency class, then issues migration or retention decisions only from reclaim, readahead, and writeback worker context. The core novelty is a translation layer between file extent semantics and CXL memory-tier semantics, enforced by an explicit state machine rather than ad hoc reclaim heuristics.

3.4 Architecture Overview

[VFS read/write]\n -> [address_space / XArray i_pages]\n -> [cxl_folio_lease sidecar lookup]\n -> HOT_DRAM ---------> retain in DRAM\n -> WARM_CXL --queue-> demotion worker -> migrate_folio() -> CXL Type-3 node\n -> DIRTY_PINNED ----> defer until writeback completion\n -> REFAULT_PROMOTE -> promotion worker -> DRAM\n -> POISON_QUARANTINE -> split/isolate folio range\n\nFeedback edges:\n[workingset refault] -> lease epoch\n[writeback completion] -> lease transition\n[extent map / iomap] -> heat bucket and migration granularity

3.5 Implementation Plan

1. Add a per-mapping sidecar xarray or radix side structure storing lease metadata by folio index. 2. Hook lease updates into filemap refault, readahead completion, and writeback completion paths while keeping all heavy work in workqueues. 3. Derive extent-aligned migration granularity from ext4 extent status trees or iomap-backed filesystems such as XFS. 4. Feed boot-time CXL CDAT/HMAT latency classes into the lease policy so only specific CXL nodes are used for WARM_CXL state. 5. Use folio_lock plus xa_lock for state transitions, with mapping->invalidate_lock to exclude truncate/invalidate races. 6. Integrate poison handling by marking affected lease entries POISON_QUARANTINE and preventing repeated refault thrash.

3.6 Validation Plan

Benchmark mixed hot/cold file-serving, package-install, and VM image workloads with fio, fs_mark, and kernel build trees. Measure refault rate, DRAM consumption, median and tail open/read latency, writeback stalls, and migration volume. Validate correctness under truncate, hole-punch, reclaim, memory hotplug, and injected CXL poison. Compare against stock Linux memory tiering and against DRAM-only placement.

3.7 Draft Claims

  1. A Linux kernel method comprising maintaining lease metadata for file-cache folios in a sidecar structure keyed by address_space index, wherein the lease metadata encodes both filesystem extent information and x86 CXL latency-domain information, and wherein folio migration between DRAM and CXL Type-3 memory is driven from the encoded lease state.
  2. The method of claim 1, wherein lease transitions are limited to worker-context execution after evaluation of refault distance, writeback completion state, and filesystem extent contiguity, thereby excluding migration decisions from spinlock-held or interrupt-context paths.
  3. The method of claim 1, wherein dirty folios associated with active journal or writeback state are assigned a DIRTY_PINNED lease state that inhibits demotion to CXL memory until a post-writeback transition occurs.
  4. The method of claim 1, wherein CXL poison information is translated into a POISON_QUARANTINE lease state at folio or sub-folio granularity to suppress repeated refault and migration of corrupted cache ranges.

3.8 Risks and Mitigations

  • Risk: migration thrash on oscillating workloads; mitigation: add per-inode hysteresis and minimum lease residency epochs before a state change is eligible.
  • Risk: metadata overhead from sidecar tracking; mitigation: allocate lease entries lazily for multi-hit file-cache regions and compress state into compact bitfields.
  • Risk: truncate and invalidate races; mitigation: require mapping->invalidate_lock plus folio_lock before lease destruction or migration commit.

Even in a worker, you need an explicit deadlock-avoidance rule set (e.g., never take jbd2/ext4 journal locks or inode i_rwsem while holding folio_lock / xa_lock / LRU locks; rely on existing aops/writeback callbacks and stable helpers).
  • “Pinning journal-coupled dirty extents in DRAM” needs a reclaim-safe bound. Without a hard limit and a fallback policy, you risk unreclaimable DRAM under memory pressure (memcg pressure, global reclaim, memory.high). Specify: (1) a maximum DRAM budget, (2) what happens when exceeded (writeback-first, demote-clean-only, or temporary demotion), and (3) how this interacts with cgroup reclaim.
  • Account for mmap’d page-cache folios and pins: migrating file folios requires rmap/unmap + migration entries, and must gracefully fail for pinned pages (e.g., long-term GUP/FOLL_PIN). The draft should explicitly route all moves through existing migrate_folio/migrate_pages machinery and define when a folio is ineligible (writeback, pinned, under filesystem lock constraints).
  • Filesystem extent semantics are not stable: delalloc/unwritten extents, hole punch, truncate, reflink/COW (XFS), and extent splitting/merging can invalidate “extent topology” assumptions. You need a concrete invalidation/versioning mechanism (per-inode generation counter, extent-map change hooks, or conservative revalidation) so that stale sidecar decisions cannot misclassify dirty/journal-coupled regions.
  • Avoid fighting existing MM policy: workingset refault distance, MGLRU, and NUMA tiering/demotion already make placement decisions. Specify how your state machine composes with (or gates) mm/vmscan demotion and memory-tiering so you don’t create oscillation (ping-pong migration) or regress hot-path behavior via excessive scanning/marking.
